# SDLC router dead-end: CRITIQUE in_progress with no recorded verdict and no PR

## Problem

When `/do-plan-critique` runs but never persists a verdict, the SDLC router
dead-ends with `Blocked('no matching dispatch rule')` and the pipeline cannot
advance from CRITIQUE to BUILD without manual intervention.

The precise stuck state (reconstructed from the real issue-1643 run, PR #1664):

- `CRITIQUE` stage marker = `"in_progress"`
- `_verdicts.CRITIQUE` = `{}` (empty — the war-room cycles ran in-skill but the
  verdict-record step never fired)
- `meta["latest_critique_verdict"]` = `None`
- `meta["pr_number"]` = falsy (no PR yet)

In this state **every** dispatch rule and guard misses:

| Rule / Guard | Why it misses |
|---|---|
| row 2 `_rule_plan_not_critiqued` | excludes `critique_status == "in_progress"` (matches only None/pending/ready) |
| row 2b `_rule_critique_verdict_stale` | requires a `recorded_at` timestamp on the verdict — absent |
| rows 3 / 4a / 4b / 4c | all require a recorded critique verdict — absent |
| G1 `guard_g1_critique_loop` | requires NEEDS REVISION / MAJOR REWORK verdict text — absent |

Result: `decide_next_dispatch` falls through the entire `DISPATCH_RULES` list
and returns `Blocked('no matching dispatch rule', guard_id=None)`.

This is a **distinct variant** from the row-2b NEEDS-REVISION staleness case
(#1639 / PR #1659). Row 2b handles a *recorded-but-stale* verdict; this issue is
an *empty* verdict. The supervisor's manual recovery — re-running
`/do-plan-critique` — was the correct fix; the router should do it automatically.

## Why Previous Fixes Failed

PR #1659 (the prior fix for the "NEEDS REVISION dead-end") correctly solved the
*recorded-but-stale* verdict case via row 2b. It did not regress — it simply
never covered the case where the critique skill **fails to record any verdict at
all**. Row 2b's staleness check (`_critique_verdict_is_stale`) requires a
`recorded_at` timestamp, so an empty `_verdicts.CRITIQUE` returns "not stale" and
the row steps aside. The fix was scoped to the symptom that had been observed
(stale NEEDS REVISION), not the broader class (any non-terminal CRITIQUE with no
route forward). This plan closes the remaining hole in that class.

## Solution

Add a new dispatch rule — **row 2c** — that re-dispatches `/do-plan-critique`
when the critique stalled with no recorded verdict and no PR yet exists.

### New predicate `_rule_critique_in_progress_no_verdict`

```python
def _rule_critique_in_progress_no_verdict(stage_states, meta, context) -> bool:
    """CRITIQUE is in_progress but never recorded a verdict — re-dispatch critique.

    The #1668 dead-end: /do-plan-critique ran (CRITIQUE marker == "in_progress")
    but never persisted a verdict, so _verdicts.CRITIQUE is empty and
    latest_critique_verdict is None. With no PR yet, rows 2/2b/3/4* and G1 all
    miss, leaving Blocked('no matching dispatch rule'). Re-running the critique
    is the correct recovery (it is what the supervisor did manually).

    Narrowly gated so it cannot fire when:
      - a PR exists (defer to G3 / PR-stage rows 7-10b)
      - any critique verdict IS recorded (let rows 2b/3/4a handle it)
      - CRITIQUE is not in_progress (None/pending → row 2; completed/failed → other rows)
    """
    if meta.get("pr_number"):
        return False
    if stage_states.get("CRITIQUE") != STATUS_IN_PROGRESS:
        return False
    # A recorded verdict (in _verdicts or meta) means another row owns this state.
    if _latest_critique_verdict(stage_states, meta).strip():
        return False
    return True
```

### New `DispatchRule` row 2c

Inserted **after row 2b and before row 3** in `DISPATCH_RULES`:

```python
DispatchRule(
    row_id="2c",
    state_predicate=_rule_critique_in_progress_no_verdict,
    skill=SKILL_DO_PLAN_CRITIQUE,
    reason="Critique stalled with no recorded verdict — re-run critique",
),
```

Plus a `__doc__` assignment in the docstring block (required by the parity test):

```python
_rule_critique_in_progress_no_verdict.__doc__ = (
    "Critique in_progress, no verdict recorded (stalled) — re-critique"
)
```

### Placement rationale

- **After row 2b**: row 2b owns the *recorded-but-stale* verdict; row 2c owns the
  *empty* verdict. They are disjoint (2b requires `recorded_at`; 2c requires the
  verdict be absent), so order between them is immaterial for correctness, but
  keeping 2c immediately after 2b groups all critique-staleness recovery rows.
- **Before row 3**: row 3 and rows 4* all require a recorded verdict, so they
  never match the 2c state anyway; placing 2c before them is defensive ordering
  consistent with row 2b's placement.

### Loop safety (critical)

The new rule re-dispatches `/do-plan-critique` and is **not** bounded by G2
(`guard_g2_critique_cycle_cap` keys off `critique_cycle_count`, which is driven
by *recorded* verdicts and stays 0 in this empty-verdict state). It **is** bounded
by **G4** (`guard_g4_oscillation`): `same_stage_dispatch_count` increments each
time `/do-plan-critique` is dispatched without the stage snapshot changing, and
G4 blocks at `MAX_SAME_STAGE_DISPATCHES`, escalating to a human. This exactly
mirrors the bounded manual recovery (re-run once; if it keeps failing, a human
looks). No new guard is needed.

The happy path: the re-dispatched `/do-plan-critique` records a verdict on its
next run → `_verdicts.CRITIQUE` populates → row 2c stops matching → routing
advances via rows 2b/3/4a. The stage snapshot changes, so G4's
`same_stage_dispatch_count` self-clears.

**Interaction with G7 (`guard_g7_plan_revising`).** G7 runs in the `GUARDS` list
*before* any dispatch rule, so it can preempt row 2c. In the empty-verdict
dead-end this plan targets, `plan_revising` is expected **falsy** — the critique
skill never reached its lock-setting step (that is why no verdict was recorded) —
so G7's Gate 2 returns `None`, the guard steps aside, and row 2c owns the state.
If `plan_revising` IS set (e.g. the critique skill set the lock and then crashed
before recording a verdict), G7 **intentionally takes precedence**: its Gate 4
fires on `last_skill == /do-plan-critique` and routes to `/do-plan`, which is also
a valid recovery (a revision pass, then re-critique). The two paths are disjoint —
row 2c handles the common lock-unset case; G7 handles the rarer lock-leaked case —
so **no code change is needed for G7**; this paragraph only documents the ordering
so the builder does not later "fix" a G7 precedence that is correct.

## Rabbit Holes

- **Refactoring the critique-staleness rules into one mega-predicate.** Tempting
  to merge rows 2b and 2c. Don't — they have disjoint gating (recorded-stale vs.
  empty) and separate row IDs keep the dispatch trail readable. Keep them
  separate.
- **Fixing the root cause in `/do-plan-critique` (why it sometimes fails to
  record a verdict).** Out of scope. That is a separate, deeper investigation
  into the critique skill's verdict-persistence step. This plan makes the router
  *resilient* to the missing verdict; it does not prevent the verdict from going
  missing. (See No-Gos.)
- **Adding a new guard for loop-bounding.** Unnecessary — G4 already bounds it.
  Adding a G8 would duplicate G4's mechanism.

## Risks

- **Risk: row 2c masks the underlying `/do-plan-critique` verdict-record bug.**
  Mitigation: this is acceptable and intentional — the router's job is resilient
  routing, not diagnosing the critique skill. The re-dispatch is bounded by G4,
  so a persistently-broken critique skill still escalates to a human rather than
  silently looping. A follow-up issue should track the verdict-record bug in
  `/do-plan-critique` itself.
- **Risk: ordering interaction with row 2b.** Mitigation: predicates are disjoint
  (2b requires `recorded_at`; 2c requires verdict absent), proven GREEN by test 3.
- **Risk: row 2c fires for a `completed`/`failed` CRITIQUE.** Mitigation: the
  predicate gates on `CRITIQUE == STATUS_IN_PROGRESS` exactly, so completed and
  failed markers never match.

## Race Conditions

None. `decide_next_dispatch` is a pure, synchronous function over a snapshot of
stage_states + meta. The new rule adds no async operations and no shared mutable
state.

## No-Gos (Out of Scope)

- **NOT** fixing why `/do-plan-critique` sometimes fails to persist a verdict.
  That is a separate root-cause investigation; this plan only makes the router
  resilient to the symptom. A follow-up issue should be filed.
- **NOT** touching row 2b, the `_critique_verdict_is_stale` helper, or any
  existing rule/guard. The canonical row-2b NEEDS-REVISION path already works and
  must remain byte-for-byte unchanged.
- **NOT** adding a new guard. G4 already provides the loop bound.
- **NOT** modifying `sdlc-tool`, stage markers, or the PM/Dev persona dispatch
  tables — the fix is invisible to those surfaces (they already call
  `next-skill`).
