# chat_message_log on AgentSession (#1192)

## Problem

The agent regularly posts duplicated, paraphrased, or already-said-it content to Telegram because the message drafter (`bridge/message_drafter.py::draft_message`) composes from `user prompt + tool outputs` only — it has no first-class view of the conversation it is driving. It cannot see its own prior outbound messages within this session, including any sends made via `valor-telegram send` from inside the session via Bash. So it re-says things that were already posted.

**Current behavior:**
- `_build_draft_prompt` (bridge/message_drafter.py:1135) reads `session.message_text` (one field, the *triggering* message), `session.classification_type`, `session.branch_name`, `session.slug`, links via `get_links()`, and `session._get_history_list()` (lifecycle events from `session_events` — *not* chat history). There is no field that carries chat traffic.
- Real symptom from `PM: PBA` (chat_id=-5238229534) on 2026-04-28: same outbound text posted twice ~0.2s apart; long-form Endstate followed immediately by a paraphrase of itself; reply followed by self-narration of the same act.

**Desired outcome:**
- `AgentSession` carries a rolling, bounded log of inbound and outbound chat traffic for its own scope.
- Both Path A (`TelegramRelayOutputHandler.send`) and Path B (`valor-telegram send` invoked via Bash inside the agent) populate the log of whichever AgentSession owns the chat.
- `bridge/message_drafter.py` reads the log and includes it as first-class context in the drafter prompt.

## Spike Results

### spike-1: Field-size budget for 50 chat-log entries on AgentSession
- **Assumption**: 50 bounded entries × realistic content lengths fit within the existing AgentSession storage profile and 30-day Redis TTL window.
- **Method**: code-read.
- **Finding**: `AgentSession` already carries `session_events` (ListField, can be large) and `queued_steering_messages` (ListField). `Meta.ttl=2592000` (30 days). 50 × ~200 bytes/entry ≈ 10 KB upper bound — well within Redis hash comfort. Bound enforced on every append.
- **Confidence**: high.
- **Impact on plan**: Use `ListField(null=True)` for `chat_message_log`. Trim oldest on append. No separate Popoto model required.

### spike-2: Path B (`valor-telegram send`) attribution to its owning AgentSession
- **Assumption**: An agent invoking `valor-telegram send` via Bash inside its session can attribute the send to its own AgentSession id without changing the CLI surface.
- **Method**: code-read.
- **Finding**: `agent/sdk_client.py:1346-1369` injects both `VALOR_SESSION_ID` (bridge session_id string, e.g. `tg_project_-123456_789`) and `AGENT_SESSION_ID` (Popoto primary key, e.g. `agt_xxx`) into the spawned agent's environment. Only `AGENT_SESSION_ID` is a valid Popoto lookup key. `tools/valor_telegram.py::cmd_send` runs in that environment. Reading `os.environ["AGENT_SESSION_ID"]` at `cmd_send` time gives us the owning AgentSession. The synthetic `cli-{ts}` queue session id is unrelated to AgentSession identity and stays unchanged.
- **Confidence**: high.
- **Impact on plan**: Attribute Path B via `AGENT_SESSION_ID` env-var only (not `VALOR_SESSION_ID`, which holds a bridge `session_id` not a Popoto primary key). Deterministic, matches the #1191 pattern, no extra Redis round-trip. Add an `owner_agent_session_id` field to the relay payload so the relay records both the queue session_id (existing semantics) and the owning AgentSession (new).

## Data Flow

### Path A (Agent's natural final response — drafter-then-relay)
1. **Entry**: Agent session terminates → Stop hook → `bridge/message_drafter.py::draft_message(raw_response, session, medium="telegram")`.
2. **Drafter**: `_build_draft_prompt` reads `session.chat_message_log` (NEW), embeds it in the prompt under a "Recent chat in this thread" section.
3. **Output**: `TelegramRelayOutputHandler.send(chat_id, text, ..., session)` (`agent/output_handler.py:122`) writes payload to `telegram:outbox:{session_id}`. Existing `_record_sent_message` already runs when the relay confirms the send.
4. **Append**: At the relay's success path (`bridge/telegram_relay.py::_send_queued_message` after `msg_id` is assigned), append `{direction: "out", sender: "valor", content: text, message_id, ts}` to the *owning* `AgentSession.chat_message_log`. For Path A, owning session_id == queue session_id.
5. **Trim**: After append, slice to last 50 entries.

### Path B (`valor-telegram send` invoked by agent inside session)
1. **Entry**: Agent calls `valor-telegram send --chat ... "msg"` via Bash; `os.environ["AGENT_SESSION_ID"]` is set by the SDK harness.
2. **Queue**: `cmd_send` reads the env-var and adds `owner_agent_session_id` to the relay payload alongside the existing `chat_id`, `text`, `reply_to`, `session_id` (synthetic `cli-{ts}`).
3. **Relay**: `_send_queued_message` sends; on success uses `owner_agent_session_id` (tier-1) or the queue `session_id` if it is a real AgentSession primary key (tier-2) to identify the owning session and append the outbound entry to its `chat_message_log`. If neither resolves, skip silently. Trim to 50.

### Inbound (incoming Telegram message routed to a session)
1. **Entry**: `bridge/telegram_bridge.py` receives a message, resolves the target AgentSession (existing routing in `bridge/session_router.py` / fresh-session creation).
2. **Append**: Just before invoking the agent (or just before enqueueing the steering message for an active session), append `{direction: "in", sender: <name>, content: <text>, message_id: <tg_msg_id>, ts}` to the resolved session's `chat_message_log`. Trim to 50.
3. **Hand-off**: Existing routing continues unchanged — the new write is additive.

### Drafter Read
1. `_build_draft_prompt(text, artifacts, session)` checks `session.chat_message_log`. If non-empty, formats the last N entries (suggest last 20) into a concise transcript block.
2. The block is inserted into the prompt context section before the artifact section, above the `Original request:` line.
3. The drafter sees its own prior sends and can avoid duplication.

## Solution

### Key Elements

- **`AgentSession.chat_message_log`**: a bounded list of `{direction, sender, content, message_id, ts}` dicts. The session-scoped record of the conversation it is driving.
- **`agent_session.append_chat_log(direction, sender, content, message_id=None, ts=None)`**: a small helper method on `AgentSession` that appends an entry, trims to `CHAT_LOG_MAX_ENTRIES`, and saves. Encapsulates the bounded-write semantics so callers don't reimplement trimming.
- **Inbound write hook**: in `bridge/telegram_bridge.py` (or `bridge/session_router.py`, wherever the inbound message is matched to a session), call `session.append_chat_log("in", sender, content, message_id, ts)` before invoking / steering the agent.
- **Outbound write hook (single chokepoint)**: in `bridge/telegram_relay.py::_send_queued_message`, on a successful send, resolve the owning AgentSession using two tiers only — (tier-1) payload's `owner_agent_session_id` fetched via `AgentSession.query.get(owner_agent_session_id)`; (tier-2) if `session_id` on the payload is a real AgentSession id (no `cli-`/`local-` prefix), fetch via `AgentSession.query.filter(session_id=session_id)`. Both tiers use sync Popoto queries only (no `await`, no `get_active_session_for_chat`). If neither resolves, skip the append and log at debug. The relay must never crash on chat-log bookkeeping. Call `session.append_chat_log("out", "valor", text, msg_id, ts)`. Wrap in `try/except Exception as e: logger.warning(...)`.
- **Drafter read hook**: `_build_draft_prompt` formats the last ~20 entries of `session.chat_message_log` into a "Recent chat in this thread" section embedded in the prompt.
- **Path B attribution**: `tools/valor_telegram.py::cmd_send` reads `os.environ.get("AGENT_SESSION_ID")` ONLY (not `VALOR_SESSION_ID`, which is a bridge `session_id` string, not a Popoto primary key) and adds it to the relay payload as `owner_agent_session_id`. If `AGENT_SESSION_ID` is unset (CLI invocation outside any agent), omit the key from the payload entirely.

### Flow

User sends "what's the status?" → Inbound hook appends `{direction:"in", sender:"User", content:"what's the status?", ...}` → Agent runs → Mid-session, agent calls `valor-telegram send "Working on it"` → Path B writes `owner_agent_session_id` into payload → Relay sends, then appends `{direction:"out", sender:"valor", content:"Working on it", ...}` → Agent's session terminates → Drafter runs → `_build_draft_prompt` sees the prior "Working on it" → drafter avoids generating "I'm working on it" again → Drafter produces a non-redundant final reply → Path A relay sends it → Path A appends `{direction:"out", ..., content:<final reply>, ...}`.

### Technical Approach

- Add the field to `AgentSession` with `ListField(default=list)` (matches existing `tags`, `pm_sent_message_ids`). Define `CHAT_LOG_MAX_ENTRIES = 50` as a module constant in `models/agent_session.py`.
- Add `AgentSession.append_chat_log(direction, sender, content, message_id=None, ts=None)`. Implementation follows the optimistic-write pattern established by `record_pm_message`: use `self` directly (no re-fetch), append the new entry to the current list, trim to `CHAT_LOG_MAX_ENTRIES`, assign back, call `self.save()`. The `_heal_descriptor_pollution` mechanism (already generic) handles the new field on existing records. The `record_pm_message` precedent was chosen deliberately — it avoids the extra Redis round-trip that a re-fetch adds, and the race documented in Race Condition 1 is acceptable for v1 (same reasoning as `record_pm_message`'s own list-append).
- In `bridge/telegram_bridge.py` (or wherever the inbound is resolved to a session), call `append_chat_log("in", ...)` after the session is identified and before agent invocation.
- In `bridge/telegram_relay.py::_send_queued_message`, after `msg_id` is assigned and before `_record_sent_message`, resolve the owning AgentSession using **two tiers only** (no tier-3 / no `get_active_session_for_chat`):
  1. **Tier-1**: If payload has `owner_agent_session_id`, call `AgentSession.query.get(owner_agent_session_id)` (sync Popoto query).
  2. **Tier-2**: Else if `session_id` on the payload does NOT start with `cli-` or `local-`, call `list(AgentSession.query.filter(session_id=session_id))` and pick the newest — same pattern as `_record_sent_message`.
  3. If neither tier resolves a session, skip the append silently (log at debug). The relay must never crash on chat-log bookkeeping.
  Then call `session.append_chat_log("out", "valor", text, msg_id, ts)`. Wrap in `try/except Exception as e: logger.warning(...)` — the chat log is enrichment, never a critical path. Place a comment: "We append AFTER successful send, so the drafter never sees the current turn's text. That is the desired behavior."
- In `tools/valor_telegram.py::cmd_send`, add `owner = os.environ.get("AGENT_SESSION_ID")` and, only if `owner` is non-empty, set `payload["owner_agent_session_id"] = owner` before the `r.rpush`. Do NOT fall back to `VALOR_SESSION_ID` — that env var holds a bridge `session_id` string (e.g. `tg_project_-123456_789`), not a Popoto `agent_session_id` primary key (`agt_xxx`). The fallback would silently set an invalid owner and the relay's `AgentSession.query.get(...)` would find nothing.
- In `bridge/message_drafter.py::_build_draft_prompt`, after the existing `context_parts` block and before returning, format `session.chat_message_log` as:
  ```
  Recent chat in this thread (you have already said the "out" lines):
  [direction=in/out, sender=X, ts=...] content
  ...
  ```
  Cap at the last 20 entries (configurable; tunable later). Insert before the artifact section.

## Rabbit Holes

- **Cross-session chat awareness.** Multiple AgentSessions can have driven the same chat over time. Each sees only its own scope (per the issue: "session-scoped log"). Do NOT make this a chat-wide global log; that's the "read the room" issue (next), not this one.
- **Hydrating from Telegram history on session resume.** Tempting because reply-to revival creates a new session that lacks the prior chat. But: free-text retrieval of Telegram history is non-trivial, the user has flagged "fresh is fine, but flag for confirmation" (Open Question 3). Stay out of this until the question resolves; default to fresh.
- **Persona-prompt rewriting.** The drafter has many subtle persona/medium prompt rules. This work adds context. It does NOT rewrite voice. Resist the urge to refactor the drafter prompt structure.
- **Compressing/summarizing the chat log on append.** Tempting once entries get long. Don't — bounded raw text is debuggable; summary loses fidelity. If 50×raw becomes a bottleneck, adjust N first.
- **Merging `chat_message_log` with `session_events`.** They serve different purposes (lifecycle vs chat) and have different write paths. Keep separate (per issue's "Lean separate" stance).

## Risks

### Risk 1: Path B attribution drift
**Impact:** If the env-var path breaks (e.g., a future SDK change stops injecting `AGENT_SESSION_ID`), Path B sends would be unattributed and the chat log would silently miss outbound entries from `valor-telegram send`. The drafter would re-emerge into the duplication failure mode for that subset.
**Mitigation:** Two-tier sync resolution (`owner_agent_session_id` → real `session_id`) at the relay. If `AGENT_SESSION_ID` stops being injected, Path B sends become unattributed (no `owner_agent_session_id` in payload), and tier-2 (`session_id` on the payload is `cli-{ts}`, not a real AgentSession id) also fails — so Path B sends silently miss the log. A unit test pins the env-var injection in `agent/sdk_client.py`. If the env-var injection is lost, the duplication prevention degrades gracefully rather than crashing.

### Risk 2: Relay write contention or save races on the AgentSession record
**Impact:** Concurrent inbound and outbound writes to the same session's `chat_message_log` could lose entries due to read-modify-write races on the Popoto record.
**Mitigation:** `append_chat_log` uses optimistic-write semantics aligned with `record_pm_message` — it does NOT re-fetch before writing. This matches established precedent and avoids a spurious Redis round-trip. The residual race (two concurrent appends on the same record, one silently overwriting the other) is accepted for v1, same as `record_pm_message`. Given inbound writes are infrequent (one per user message) and the relay's outbound tempo is ~1 msg/sec ceiling, contention is rare. Documented as accepted residual risk for v1.

### Risk 3: Prompt size inflation
**Impact:** A 50-entry chat log of ~200-byte messages pushes the drafter prompt up by ~10 KB. The Haiku and OpenRouter calls have generous context windows, but every byte costs latency and tokens.
**Mitigation:** Cap the *displayed* slice in the drafter prompt to the last 20 entries (configurable constant). Storage stays at 50 (extra headroom for future read patterns). 20 × 200 ≈ 4 KB is well within typical drafter prompt sizes.

### Risk 4: Conflict with #1191
**Impact:** #1191 also modifies the Path B relay payload (`reply_to` defaulting). Concurrent merges could cause merge conflicts.
**Mitigation:** Both changes are additive payload keys (`owner_agent_session_id` here, `reply_to` default there). Merge conflicts are textual not semantic. Coordinate by landing whichever ships first; the second rebases trivially. No design coupling beyond touching the same file.

## Race Conditions

### Race 1: Concurrent inbound + outbound chat_log writes
**Location:** `models/agent_session.py::append_chat_log`, called from `bridge/telegram_bridge.py` (inbound, sync within bridge handler) and `bridge/telegram_relay.py::_send_queued_message` (outbound, async coroutine in relay loop).
**Trigger:** A user sends a message (inbound write) while the relay is concurrently posting an outbound message for the same session. Both read the AgentSession record, append to `chat_message_log`, and save. The later write wins; the earlier append is lost.
**Data prerequisite:** The current `AgentSession.chat_message_log` must reflect both writes after both complete.
**State prerequisite:** The session record's `chat_message_log` must be a true append-only log (modulo the bound).
**Mitigation:** `append_chat_log` uses optimistic-write (no re-fetch), matching the `record_pm_message` pattern. This means a concurrent write could overwrite an earlier append. Given the low concurrency ceiling (one user message per inbound write, ~1 relay outbound/sec), the probability of a lost entry is negligible in practice. If lost-update becomes observable, escalate to a Lua atomic-list approach. Documented as accepted residual risk for v1.

### Race 2: Drafter reads chat_log after Path A relay writes have not yet flushed
**Location:** Path A: drafter (`bridge/message_drafter.py::draft_message`) reads `session.chat_message_log`; the relay has not yet posted the *current* turn's outbound (it posts AFTER the drafter completes, since the drafter produces the text the relay will send).
**Trigger:** Within a single turn, the drafter does not need to see the *current* turn's outbound — it produces it. The drafter needs to see *previous* outbound, which is already in the log from prior turns / Path B sends earlier in *this* session.
**Data prerequisite:** Prior outbound entries from earlier in the session are durably in the log.
**State prerequisite:** No ambiguity about "did the current turn's text get logged before the drafter ran" — by construction it cannot have, because it is *what the drafter produces*.
**Mitigation:** None needed — this is the correct ordering. Document in code comment near the relay's append site: "We append AFTER successful send, so the drafter never sees the current turn's text. It sees prior turns and prior Path B sends. That is the desired behavior."

- Email-medium chat_message_log support. Out of scope for this issue; the field is generic but the write hooks are Telegram-only here. Email follows the same pattern in a follow-up if needed.

- Migrating `session_events` lifecycle entries into `chat_message_log` or vice versa. They stay separate.
- **Tier-3 relay fallback via `get_active_session_for_chat(chat_id)`**. Explicitly removed from scope. This async function cannot be called from the sync relay append helper without a running event loop. More importantly, it would attribute unowned traffic (manual human CLI sends) to whichever session happens to be active in that chat — crossing the Telegram-only and session-scoped boundaries simultaneously. The two-tier sync resolution (owner_agent_session_id + session_id) is sufficient for the agent-driven use cases this feature targets.

## Update System

No update system changes required — this feature is purely internal. No new dependencies, services, or config files. Existing AgentSession records pick up the new field automatically via Popoto's nullable-field semantics and `_heal_descriptor_pollution`.

## Agent Integration

No agent integration required — this is bridge-internal awareness. The agent does not gain a new tool; the drafter (which the agent does not directly invoke) gains better context. The agent's existing `valor-telegram send` Bash tool is unchanged at the user-visible surface; only its outbound payload gains an extra optional key.

The single agent-facing test that matters: pre-populate a session's `chat_message_log` with a `direction=out, content=X` entry (simulating what the relay would have written after Path B sent "X"), call `_build_draft_prompt` with that session, and assert the prompt string contains "X". This is a unit test at `tests/unit/test_chat_message_log_path_b.py` — it does not spawn a live agent or hit real Telegram.

## Open Questions

1. **Display cap in drafter prompt.** Plan proposes last 20 of 50 stored entries. Acceptable? If the duplication-prevention signal is mostly in the very recent past (last 3-5 messages), 20 is overkill and inflates prompt. Lower default to 10?
2. **Backfill on resume.** When a session resumes via reply-to (e.g., user replies after a 👍-triggered termination), the new session's chat_message_log starts fresh. The issue suggests "fresh is fine, but flag for confirmation." Confirm: fresh, OR hydrate the latest N messages from Telegram history into the resumed session's log?
3. **Sender field for outbound.** Plan uses literal string `"valor"` for all Path A and Path B outbound entries. Acceptable, or should we record the persona (`"pm"` / `"teammate"` / `"dev"`) so the drafter can see "I sent this as PM-Valor 30s ago"? (Mild lean: `"valor"` keeps the log audience-facing — what the user sees in the chat — and the persona is recorded elsewhere on the session.)

*Note: The former OQ2 (tier-3 fallback — human-CLI traffic attribution) has been resolved. Tier-3 is removed from scope entirely (see No-Gos). Manual CLI sends by humans outside an agent session remain unattributed, which is the correct behavior: the session-scoped log should only contain traffic the owning session generated or received.*
