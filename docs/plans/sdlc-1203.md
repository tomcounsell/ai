# RTR: Extend Pre-Send Pass to Path B (`valor-telegram send`)

## Problem

The Read-the-Room pre-send pass shipped in PR #1204 (issue #1193, plan
`docs/plans/completed/sdlc-1193.md` — migrated by commit `f40cdb48`) covers
**Path A only** — `agent/output_handler.py::TelegramRelayOutputHandler.send`.

**Path B** — `tools/valor_telegram.py::cmd_send` (the `valor-telegram send`
CLI registered in `pyproject.toml`) — writes directly to the Redis outbox
(`telegram:outbox:{session_id}`) and bypasses `output_handler.py` entirely.
The agent uses Path B for proactive sends that don't fit the
final-message-drafter mold: images, voice notes, file attachments,
mid-session updates from inside Bash. Humans also use Path B for ad-hoc
manual sends.

So the failure mode RTR was built to prevent — drafting a now-redundant
reply because the conversation moved on, or non-sequiturs landing into a
personal moment between two humans (the original 2026-04-28 PM:PBA
incident) — is half-fixed. An agent that switches from `_send_via_telegram`
(Path A) to `valor-telegram send --file ...` (Path B) gets zero
read-the-room protection.

**Current behavior:**
- Path A: `delivery_text` runs through the drafter, then through RTR (gated
  on `READ_THE_ROOM_ENABLED`), then to the outbox. Suppress/trim verdicts
  emit `rtr.suppressed` / `rtr.trimmed` / `rtr.bypassed` `session_events`
  and (for suppress with a reply anchor) queue a 👀 reaction.
- Path B (`cmd_send`): linkify + length-truncate, then straight to
  `telegram:outbox:{session_id}`. No drafter, no RTR, no event trail.

**Desired outcome:**
Agent-invoked Path B sends pass through RTR by default (gated by the same
`READ_THE_ROOM_ENABLED` flag), with the same fail-open semantics as Path A.
Human-invoked Path B sends do NOT pass through RTR — the human authored the
message intentionally.

The mechanism for distinguishing the two callers must work without
plumbing a new flag through every agent prompt. The default path (no flag
needed) must be: agent calls → RTR runs (when enabled); human calls → RTR
skipped.

## Solution

### Key Elements

- **Caller-type gate**: An auto-detection function that returns `True` when `cmd_send` is invoked from an agent (Bash tool inside an AgentSession), and `False` for human-invoked CLI runs. The gate is overridable via `--read-the-room` / `--no-read-the-room` flags.
- **Sync→async bridge**: A single `asyncio.run(read_the_room(...))` call inside the synchronous `cmd_send` function. Wrapped in try/except for the same fail-open contract as Path A.
- **Verdict handling**: Mirror the Path A logic — `send` (no-op, proceed), `trim` (substitute revised text, with `TRIM_TOO_SHORT_THRESHOLD` coercion to suppress), `suppress` (skip the outbox write, queue a 👀 reaction if a reply anchor exists).
- **Session reconstruction**: If `VALOR_SESSION_ID` is set, fetch the matching `AgentSession` from Popoto so RTR can append `session_events`. If absent or the lookup fails, pass `session=None` — RTR's `_append_event` already no-ops on `None`, and the `sdlc_slug` short-circuit becomes inert (which is the right answer for human sends anyway).

### Flow

**Caller-type detection (auto):**
- `VALOR_SESSION_ID` env var present **and** non-empty → agent context → RTR on (when `READ_THE_ROOM_ENABLED=true`)
- `VALOR_SESSION_ID` absent or empty → human context → RTR off
- `--read-the-room` flag → force on, regardless of env
- `--no-read-the-room` flag → force off, regardless of env
- `--read-the-room` and `--no-read-the-room` both set → argparse mutual-exclusion error

**Send flow inside `cmd_send` (post-linkify, pre-rpush):**

`text` (already linkified + length-truncated) → caller-type gate decision → if RTR enabled and `READ_THE_ROOM_ENABLED=true` → load `AgentSession` from `VALOR_SESSION_ID` (best-effort) → `asyncio.run(read_the_room(text, chat_id, session))` → branch on verdict:
- `send` → fall through to existing rpush.
- `trim` with revised text shorter than `TRIM_TOO_SHORT_THRESHOLD` → coerce to suppress.
- `trim` with revised text long enough → substitute `text = verdict.revised_text`, fall through to rpush.
- `suppress` → if `reply_to` set, queue a 👀 reaction payload (same shape as `_build_reaction_payload` in `agent/output_handler.py`); print a single-line confirmation to stdout (`Message suppressed by RTR (reason: ...)`). Skip the rpush.

### Technical Approach

**1. Caller-type detection (default rule).** Use `VALOR_SESSION_ID` as the auto-detection signal, NOT `CLAUDE_CODE_TASK_LIST_ID` (the parent plan's example). Reason: `VALOR_SESSION_ID` is the canonical session env var the worker injects via `agent/sdk_client.py:_extract_sdlc_env_vars`. It's the same signal the existing `cmd_send` already uses (lines 762-779 use `TELEGRAM_REPLY_TO` from the same injection block, and the `session_id` build for `cli-` prefix at line 763 is the manual-fallback case). `CLAUDE_CODE_TASK_LIST_ID` is set by the SDK client but its meaning is "we have a task list", not "we are inside an agent" — false positives are possible. `VALOR_SESSION_ID` is unambiguously agent-only.

**2. Override flags.** Add two argparse flags to the `send` subcommand:
- `--read-the-room` — store_true; force RTR on. (argparse `dest=read_the_room`.)
- `--no-read-the-room` — store_true; force RTR off. (argparse `dest=no_read_the_room`.)
- Add to a mutually-exclusive group inside argparse so users get a clean error.
- Document in the CLI epilog: "RTR auto-runs for agent-invoked sends when `READ_THE_ROOM_ENABLED=true`. Use `--read-the-room` to force on for human invocations or `--no-read-the-room` to opt out from inside an agent session."

**3. Sync→async bridge.** Use `asyncio.run(read_the_room(text, chat_id, session))`. The CLI entrypoint runs in its own process per invocation; there is no surrounding event loop to nest. This matches the existing pattern at `tools/valor_telegram.py:297` (`return asyncio.run(_fetch())`), so we are not introducing a new architectural choice. Wrap in try/except for fail-open: any exception (including `RuntimeError: asyncio.run() cannot be called from a running event loop`, which would only happen if a future caller embeds `cmd_send` in an async context — currently impossible) returns `verdict = RoomVerdict(action="send", reason="rtr_error")` and the original text passes through.

**4. Session reconstruction (best-effort).** When `VALOR_SESSION_ID` is set, look it up via the Popoto ORM:
```python
from models.agent_session import AgentSession
session = AgentSession.query.get(session_id) if session_id else None
```
On any error (session deleted, Popoto unavailable, malformed ID), set `session = None`. RTR's `_append_event` no-ops on `None`, so we lose the audit signal in `session_events` but the verdict still computes correctly.

**5. Suppress with reaction.** When the verdict is `suppress` and `reply_to` is set, queue a reaction payload to the same `telegram:outbox:{session_id}` key. The payload shape is documented in `tools/send_telegram.py` and matches `agent/output_handler.py::_build_reaction_payload`. Reuse the constant `RTR_SUPPRESS_EMOJI = "👀"` from `bridge/read_the_room.py`.

**6. Suppress without reaction anchor.** When verdict is `suppress` and `reply_to` is `None`, F4 from the parent plan applies: silent suppression breaks the I-heard-you contract. Fall through to send the original text and emit a `rtr.suppress_fallthrough` event (matching Path A behavior at `agent/output_handler.py:399-407`).

**7. CLI exit code.** A successful suppress is still a successful CLI invocation (exit 0). Print a one-line confirmation: `Message suppressed by RTR (reason: <reason>)`. The existing success print path becomes conditional: only print `Message queued (...)` if we actually rpush'd.

## Rabbit Holes

- **Don't refactor `cmd_send` into smaller functions.** It's already 145 lines with a single linear flow; extracting helpers for "validate inputs", "build payload", "RTR check", "rpush" would muddy the diff. Inline the RTR block.
- **Don't add an MCP-level wrapper.** The agent already invokes `valor-telegram send` via the Bash tool; piping through MCP adds a serialization layer for no benefit. The Bash → CLI → Redis path is already two layers of indirection.
- **Don't deprecate `tools/send_telegram.py`.** It's a separate cleanup; this plan covers `valor-telegram send` (the documented CLI in `pyproject.toml`). If a future plan unifies the two, RTR coverage moves with it.
- **Don't add per-call latency telemetry.** The `rtr.*` `session_events` already exist (when a session is supplied). Adding stdout timing in the CLI is cosmetic and bloats the user-facing print.

## Risks

### Risk 1: Mistaken caller-type detection
**Impact:** A human invocation is misidentified as an agent invocation (or vice versa) — RTR runs when it shouldn't, costing 0.5-3s of latency on a manual send; or RTR is skipped when it should run, defeating the purpose of this plan.
**Mitigation:** `VALOR_SESSION_ID` is unambiguously agent-only. The worker injects it via `agent/sdk_client.py:_extract_sdlc_env_vars`. Humans typing `valor-telegram send` in a terminal will never have it set unless they manually export it (which is itself an explicit opt-in). The mutex `--read-the-room` / `--no-read-the-room` flags give both populations an escape hatch.

### Risk 2: CLI latency for agent sends
**Impact:** Adding a Haiku call (≤3s SDK timeout) to every agent-invoked Path B send slows down image/voice-note/file delivery. Voice-note delivery in particular is latency-sensitive (the agent streams TTS, then immediately sends).
**Mitigation:** (a) `READ_THE_ROOM_ENABLED=false` is the default — agents don't pay the tax until the operator flips it on per-machine. (b) RTR's existing short-circuits already skip short outputs (< `SHORT_OUTPUT_THRESHOLD` = 200 chars), which covers most caption-bearing file sends. (c) The fail-open timeout is 3s; the median Haiku response is far faster. (d) After rollout, `rtr.bypassed`/`rtr.suppressed`/`rtr.trimmed` event counts on Path B will tell us whether Path B's value justifies the cost — same data path the parent plan's No-Gos called for.

### Risk 3: Sync→async bridge inside an event loop
**Impact:** If a future caller imports `cmd_send` and invokes it from inside an async context (currently no such caller exists, but defensible coding suggests planning for it), `asyncio.run()` will raise `RuntimeError: asyncio.run() cannot be called from a running event loop`.
**Mitigation:** (a) The outer try/except catches it and falls open to `verdict = send`, so the behavior degrades gracefully. (b) The existing `cmd_read` path at line 297 already uses `asyncio.run()` — this plan does not introduce a new architectural pattern, it follows the established one. (c) If the need ever arises to call `cmd_send` from inside an async context, the fix is to refactor RTR's call site into a small `_send_async()` helper and have the sync wrapper call `asyncio.run(_send_async())` — that refactor is out of scope here.

### Risk 4: Reaction payload shape drift
**Impact:** RTR Path A and Path B both queue a 👀 reaction on suppress. If the parent plan's reaction payload shape (the `_build_reaction_payload` helper) changes, Path B's hand-written copy could fall out of sync.
**Mitigation:** Import the helper from `agent/output_handler.py` rather than re-implementing it. The CLI module already imports across `tools/` and `bridge/`; adding `from agent.output_handler import _build_reaction_payload` is symmetric. If the helper is currently private (leading underscore), expose it as a module-level function (rename or re-export) — a single-line refactor in `output_handler.py` that the build can do.

## Race Conditions

### Race 1: RTR call vs. concurrent message arrival
**Location:** `tools/valor_telegram.py::cmd_send` between the snapshot fetch (inside `read_the_room`) and the rpush.
**Trigger:** Another participant posts a message AFTER RTR fetches the snapshot but BEFORE we rpush. RTR's verdict is based on a stale view of the room.
**Data prerequisite:** `TelegramMessage` records up to ~time-of-snapshot.
**State prerequisite:** None — the snapshot is read-only.
**Mitigation:** Same as the parent plan's Race 2 — accept the race. The post-RTR rpush is async with the relay's send anyway, so even a "perfect" snapshot would race the actual send. This is the documented and accepted posture in `bridge/read_the_room.py:9-13`.

### Race 2: Multiple Path B invocations in parallel from the same session
**Location:** Two `valor-telegram send` calls fired by the same agent in rapid succession.
**Trigger:** Agent emits two `valor-telegram send` Bash calls in adjacent tool blocks. RTR for call #1 fetches the snapshot and decides to send; call #2 starts before call #1's payload is recorded in `TelegramMessage`. Both RTR verdicts evaluate against the same pre-state.
**Data prerequisite:** `TelegramMessage` records would need to include call #1's outbound payload before call #2's snapshot fetch — they don't.
**State prerequisite:** None.
**Mitigation:** Same as the parent plan's Race 1 — accept the race; this is "first-second-of-a-double-send" symmetry with Path A. RTR's job is to catch sustained redundancy across the conversation, not to deduplicate within a single agent turn (that's the drafter's responsibility on Path A — and the agent doesn't have a drafter on Path B, which is one of the deferred decisions for a future plan).

## No-Gos (Out of Scope)

- **Deprecating `tools/send_telegram.py`.** That older sibling script is a separate consolidation; this plan covers `valor-telegram send` only.
- **Wiring RTR into `tools/send_message.py::_send_via_telegram`.** That helper is a different code path used by Path A's polymorphic `send_message` tool. It already routes through `output_handler.py`'s `TelegramRelayOutputHandler.send` (see imports in `tools/send_message.py:71-72`), so it's RTR-covered transitively. No additional work needed.
- **Drafter integration on Path B.** Path B sends are intentionally drafter-bypassing (the agent crafted them deliberately, e.g., a captioned image). Adding the drafter here defeats the purpose of having a separate path. RTR is the right level of intervention; the drafter is not.
- **Email Path B.** `valor-email send` writes to `email:outbox:{session_id}` and bypasses Path A's `_send_via_email_outbox` for the same reasons. Email RTR is independent — it would need an email-specific snapshot fetcher and a different system prompt. Defer indefinitely; email volume is far lower than Telegram and the redundancy failure mode is different.
- **A new caller-type signal beyond `VALOR_SESSION_ID`.** Don't bikeshed a perfect detection. If we ever ship a non-session-bearing agent-invoked CLI use case, that's the time to revisit.
- **Per-machine `RTR_FOR_BASH=true` env var.** The parent plan's issue body listed this as a candidate mechanism. We're not adding it. The combination of `READ_THE_ROOM_ENABLED` (machine-wide kill switch) + `--read-the-room`/`--no-read-the-room` (per-call override) + auto-detection (default sensible behavior) covers every case without a third axis.

## Open Questions

1. **Resolved:** `_build_reaction_payload` is a `@staticmethod` on `TelegramRelayOutputHandler` in `agent/output_handler.py:667` — class-private (leading underscore), only referenced by `react()`, `_rtr_queue_reaction()`, and `tests/unit/test_output_handler.py`. No external module imports it today. **Decision:** call it as the static method `TelegramRelayOutputHandler._build_reaction_payload(...)` from `cmd_send` — staticmethod is callable on the class without instantiation, leading underscore is a convention not a hard barrier, and we avoid an out-of-scope rename PR. If a future cleanup wants to drop the underscore + promote it to module-level, that's a trivial follow-up; not in scope here.

2. **Resolved:** **No.** Do not emit `rtr.bypassed` for the human-caller skip path. Two reasons: (a) The Path A `sdlc_session` bypass emits only because a session exists to anchor the event — `_append_event(session, ...)` requires a non-None session. Human invocations of `cmd_send` have no `VALOR_SESSION_ID` and thus no session, so there's nowhere to write the event. (b) The skip is structural (no session = no agent context = nothing to observe), not a runtime decision worth instrumenting. The `READ_THE_ROOM_ENABLED=false` short-circuit also doesn't emit an event — same reasoning. We keep parity-of-philosophy, not parity-of-mechanism.

3. **Resolved:** Use `--read-the-room` / `--no-read-the-room`. The existing `cmd_send` flags (`--reply-to`, `--voice-note`, `--cleanup-after-send`, `--chat`) are uniformly long-form descriptive multi-word names — there are zero abbreviated flags in the file. Self-documentation matches the file's convention; `--rtr` would be the lone outlier. The argparse `dest` resolves to `read_the_room` / `no_read_the_room` automatically.
