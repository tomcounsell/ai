# Frequent Update Polling Per Machine

## Problem

The `/update` command only executes on one machine instead of all running bridges.

**Current behavior:**
Sending `/update` in Telegram triggers the update on a single machine. Other machines either don't receive the Telegram message (MTProto delivers to one active session) or silently drop it due to the DM guard. The machines do not share a Redis server or filesystem, so there is no cross-machine signaling mechanism. The existing `com.valor.update` launchd plist runs only at 6 AM and 6 PM -- a 12-hour window means code pushes take up to half a day to propagate.

**Desired outcome:**
Every machine pulls latest code from main and restarts its bridge within 30 minutes of a push, without relying on Telegram message delivery. The manual `/update` Telegram command continues to work as a local override.

## Prior Art

- **PR #28**: Add running session check to /update command -- added graceful handling of in-flight sessions during update. This work is preserved; the restart-requested flag and `_check_restart_flag()` in `agent/job_queue.py` already implement graceful restart.
- **Issue #24**: Remote Update: Telegram command + auto-sync cron -- original design for the /update system. Established the architecture we're extending.
- **Issue #251**: Bug: Reflections never fires on secondary machines -- fixed deprecated launchctl API. Confirmed that launchd plists now work reliably on all machines.

## Data Flow

1. **Entry point**: launchd timer fires every 30 minutes, invoking `scripts/remote-update.sh`
2. **remote-update.sh**: Acquires lock (`data/update.lock`), calls `scripts/update/run.py --cron`
3. **run.py --cron**: Runs `git fetch` + `git pull`, checks if new commits arrived
4. **If new commits**: Syncs dependencies (if dep files changed), writes `data/restart-requested` flag
5. **Bridge job queue**: After each completed job (or when queue drains), `_check_restart_flag()` detects the flag, calls `_trigger_restart()` which sends SIGTERM
6. **launchd KeepAlive**: Restarts the bridge process with the new code
7. **Bridge startup**: `clear_restart_flag()` removes the stale flag on fresh start

## Architectural Impact

- **No new dependencies**: Reuses existing `remote-update.sh` and `scripts/update/run.py --cron`
- **Interface changes**: None -- the update orchestrator already supports `--cron` mode
- **Coupling**: No change -- the launchd plist is independent of the bridge process
- **Data ownership**: No change -- each machine owns its own `data/restart-requested` flag
- **Reversibility**: Trivial -- change the plist interval back or unload the plist

## Solution

### Key Elements

- **Updated launchd plist**: Change `com.valor.update` from 6AM+6PM calendar triggers to a 30-minute `StartInterval`
- **Plist generation/template**: Ensure the plist is generated or templated in the repo so `valor-service.sh install` propagates it
- **Install script update**: Ensure `valor-service.sh install` handles the updated plist (unload old calendar-based, load new interval-based)
- **Log rotation awareness**: More frequent runs means more log output; ensure update logs don't grow unbounded

### Flow

**launchd timer (30 min)** -> `remote-update.sh` -> `run.py --cron` -> git pull -> (if new commits) dep sync + write restart flag -> bridge picks up flag -> graceful SIGTERM -> launchd restarts bridge with new code

### Technical Approach

- Replace the `StartCalendarInterval` entries (6AM, 6PM) in the `com.valor.update` plist with `StartInterval` set to 1800 seconds (30 minutes)
- The plist template must live in the repo (currently it may be generated by `valor-service.sh`). Confirm location and update it.
- `valor-service.sh install` already handles unload/load for the update plist -- verify it handles the transition from calendar-based to interval-based cleanly
- The existing `--cron` mode of `run.py` already sets the restart flag instead of doing a direct restart, which is exactly what we want for frequent polling
- Add a brief log line at the start of each cron run so operators can verify polling frequency in `logs/update.log`

## Rabbit Holes

- Merging the update cron into reflections' 19-step sequence -- reflections runs daily at 6 AM, which is too infrequent. Keep them as separate launchd services with different intervals.
- Adding Redis pub/sub or any cross-machine signaling -- machines share no state; each polls independently.
- Making the polling interval dynamically configurable via Telegram -- static 30-minute interval is sufficient; manual `/update` covers urgent cases.
- Adding push-based webhooks from GitHub -- would require a publicly reachable endpoint on each machine, which is a different project entirely.

## Risks

### Risk 1: Frequent polling increases git fetch load
**Impact:** GitHub rate-limits or slows down git operations
**Mitigation:** 30-minute interval is modest (48 fetches/day/machine). GitHub's rate limit for authenticated git operations is well above this. The lockfile prevents overlapping runs.

### Risk 2: Bridge restart during active job
**Impact:** In-flight agent session is interrupted
**Mitigation:** Already handled -- `_check_restart_flag()` in `agent/job_queue.py` only triggers restart when no jobs are running across all projects. If jobs are active, it defers until the queue drains.

## Race Conditions

### Race 1: Concurrent update runs
**Location:** `scripts/remote-update.sh` lock acquisition
**Trigger:** launchd fires while a previous run is still executing (unlikely at 30-min interval, but possible if an update takes long)
**Data prerequisite:** `data/update.lock` directory must not exist
**State prerequisite:** No other update process running
**Mitigation:** Already handled -- `mkdir` is atomic on POSIX. The script skips if lock exists, with stale lock detection (>10 minutes) as a safety valve.

## No-Gos (Out of Scope)

- Cross-machine signaling (Redis pub/sub, shared filesystem, webhook endpoints)
- Changing the update orchestrator logic (`run.py`) itself
- Modifying the bridge's restart flag handling in `agent/job_queue.py`
- Making reflections run more frequently (separate concern)
- Push-based deployment (GitHub webhooks to each machine)

## Update System

This plan directly modifies the update system. The changes are:
- The `com.valor.update` launchd plist switches from `StartCalendarInterval` (6AM+6PM) to `StartInterval` (1800 seconds)
- `valor-service.sh install` must handle the plist transition (unload old, load new)
- Once merged, the next `/update` run on each machine will install the new plist via `install_service()`, which calls `valor-service.sh install`. After that, polling happens automatically every 30 minutes.

Migration: The first update after merge will install the new plist. No manual intervention needed -- the existing 6AM/6PM schedule will eventually trigger `remote-update.sh`, which pulls the new code and reinstalls the plist with the new interval.

## Agent Integration

No agent integration required -- this is a launchd scheduling change. The Telegram `/update` command continues to work unchanged as a manual override. No new MCP servers, no changes to `.mcp.json`, no bridge code changes.
