# Test-Suite Notify Isolation: Fixture Notifies Must Not Reach the Live Worker

## Problem

Running `pytest tests/` on a machine with a live worker lets test side-effects
reach production orchestration state. Two distinct leaks were observed live on
2026-07-17:

1. **Notify-channel leak (confirmed).** At 14:08:01 UTC the live worker logged
   `Received session notify: worker_key=test session_id=sess-1` →
   `[worker:test] Started session queue worker`, plus a
   `[worker:chat-reconcile-fail]` loop. Pytest fixtures published to the live
   `valor:sessions:new` notify channel and the production worker spun up queue
   loops for the fixture sessions. Had a fixture session still been `pending` at
   pop time, the live worker would have **executed a test fixture as a real
   session**.

2. **Worker-lifecycle SIGTERM leak (circumstantial).** Integration tests that
   exercise worker lifecycle (`tests/integration/test_watchdog_recovery.py`,
   `test_remote_update.py`, `test_crash_auto_resume.py`) appear to have SIGTERM'd
   the **launchd-managed live worker** twice (14:11:33, ~14:12:46; see #2146)
   when the suite ran unattended. Attribution is inventory+timing only, so this
   half is scoped as "verify + harden" rather than "fix a known line."

**Root cause (leak #1).** `agent/agent_session_queue.py::enqueue_agent_session`
publishes via `POPOTO_REDIS_DB.publish("valor:sessions:new", payload)` (line
414) and `_session_notify_listener` subscribes to the same hardcoded channel
(line 998). The autouse `redis_test_db` fixture (`tests/conftest.py`) repoints
`POPOTO_REDIS_DB` at a per-process test db (`db>=1`) — but **Redis pub/sub is
not database-scoped**. `PUBLISH`/`SUBSCRIBE` operate per Redis *server*, so a
fixture enqueue on db=1 still delivers on the one global `valor:sessions:new`
channel that the live worker (db=0) is subscribed to. Key isolation works;
notify isolation does not. That is the entire mechanism of leak #1.

**Desired outcome.** A full-suite run on a live machine cannot mutate live
orchestration state: fixture-session notifies are invisible to the live worker,
and worker-lifecycle tests provably signal only processes they themselves
spawned. The fix isolates the blast radius **without abandoning the no-mocks
real-integration philosophy** — the real Redis pub/sub path is still exercised,
just on a db-scoped channel name.

## Why Previous Fixes Failed

The `redis_test_db` fixture was designed to isolate *keys* (reads/writes/
flushdb), and it does so correctly. It never addressed pub/sub because pub/sub
is invisible to the db-SELECT mechanism — a channel is a server-global name, not
a keyspace entry. No prior fix targeted the notify path, so the leak persisted
behind an isolation layer that looked complete but was silent on pub/sub.

## Architectural Impact

- Introduces a single source of truth for the notify channel name derived from
  the active Redis db: production (db=0) keeps the exact wire name
  `valor:sessions:new`; any test db (`db>=1`) gets `valor:sessions:new:db{N}`.
- Both the publisher and the subscriber call the same helper against
  `POPOTO_REDIS_DB`, so they always agree by construction. Production behavior
  and the on-wire channel name are byte-identical to today.
- A test-spawned worker that connects on the claimed test db automatically
  derives the matching test channel, so intra-test worker↔notify flows still
  work end-to-end (real pub/sub, no mocks).
- Adds a reusable test guard asserting a target PID is never the launchd worker.

## Solution

### Design decision: db-derived notify-channel namespace (PRIMARY)

Considered three isolation mechanisms named in the issue:

| Option | Verdict |
|--------|---------|
| **Dedicated test Redis instance/port** (`REDIS_URL` override) | Rejected. Heavyweight: requires spinning a second `redis-server` in conftest and CI, and changes every subprocess's connection. Solves pub/sub but at high operational cost. |
| **Notify-listener filter on a `test-` worker_key convention** | Rejected as primary. Fragile — fixture worker_keys like `test` and `chat-reconcile-fail` are not reliably prefixed, so the filter would be a guessing game. Kept only as optional defense-in-depth (see below). |
| **db-derived channel namespace** | **CHOSEN.** The `redis_test_db` fixture already repoints `POPOTO_REDIS_DB` at a process-unique db `N`. Deriving the channel name from that db (`db=0`→`valor:sessions:new`, `db>=1`→`valor:sessions:new:db{N}`) makes fixture notifies land on a channel the live worker never subscribes to. Zero new env vars, zero new processes, inherits #2060's xdist-safe per-process uniqueness, and exercises the real pub/sub path (no mocks). |

**Implementation.** Add `notify_channel_for(client) -> str` in
`agent/agent_session_queue.py` (or a small shared module it already imports):

```python
def notify_channel_for(client) -> str:
    """Derive the session-notify channel from the client's Redis db.

    Production (db=0) uses the canonical `valor:sessions:new`. Any test db
    (db>=1, set by the redis_test_db fixture) gets a db-scoped suffix so
    fixture notifies never reach a live worker subscribed on db=0. Redis
    pub/sub is server-global, not db-scoped, so the channel NAME is the only
    lever available.
    """
    try:
        db = int(client.connection_pool.connection_kwargs.get("db", 0) or 0)
    except Exception:
        db = 0
    return "valor:sessions:new" if db == 0 else f"valor:sessions:new:db{db}"
```

- Publisher (`enqueue_agent_session`, line 414): replace the literal with
  `notify_channel_for(POPOTO_REDIS_DB)`.
- Subscriber (`_session_notify_listener`, line 998 subscribe + the NUMSUB
  self-check probes at lines ~895, ~1009 that reference the literal): replace
  each literal with `notify_channel_for(POPOTO_REDIS_DB)`, computed once per
  listener setup so the healthcheck probes the same channel it subscribed to.
- Under `redis_test_db`, `POPOTO_REDIS_DB` is the test client, so publish +
  subscribe both resolve `valor:sessions:new:db{N}` — a test-spawned worker on
  the same db joins that channel; the live worker (db=0) never does.

### Design decision: service isolation for worker-lifecycle tests

- **Audit** `test_watchdog_recovery.py`, `test_remote_update.py`,
  `test_crash_auto_resume.py` for any real `os.kill` / `pgrep`-then-kill /
  `proc.terminate()` whose target could resolve to the launchd worker PID
  (`pgrep -f "python -m worker"`). Confirm which one (if any) leaked the live
  SIGTERM.
- Add a shared test guard `assert_not_live_worker(pid)` (in
  `tests/integration/` conftest or a `tests/_worker_guard.py` helper) that
  raises if `pid` matches the registered launchd worker — resolved from the
  `register_worker_pid` record (`agent/session_health.py`, `worker:pid`
  heartbeat keys) and/or a live `pgrep -f "python -m worker"`. Route any
  real-kill path in the three tests through it, so a fabricated-PID test can
  never target the live worker even if a future refactor drops the probe mock.
- Where a test needs a worker to signal, it must **spawn its own** subprocess
  (the watchdog test already does via a self-spawned proc + `_pid_lookup_for`);
  generalize that so the pattern is the documented default.

### Design decision: AC#1 worker.log-scan assertion

New integration test `tests/integration/test_notify_isolation.py`:

- Capture the live `logs/worker.log` byte offset (skip/xfail cleanly if the file
  is absent, e.g. CI with no live worker — the assertion is vacuously satisfied
  there and is meaningful only on a live-worker machine).
- Enqueue a fixture session through the real `enqueue_agent_session` path on the
  claimed test db (autouse `redis_test_db` in effect), with a recognizable
  `test-notify-isolation-*` worker_key.
- Assert the tail of `worker.log` from the captured offset contains **zero** new
  `Received session notify: worker_key=test-notify-isolation` /
  `Started session queue worker` lines for that key — proving the live worker
  never saw the fixture notify.
- A companion assertion verifies the notify DID land on the db-scoped channel
  (subscribe a probe client on the test db to `valor:sessions:new:db{N}`,
  enqueue, assert one message received) — proving isolation, not mere silence.

### Optional defense-in-depth (include if cheap)

A guard in `_session_notify_listener` that drops (logs at DEBUG, does not spawn)
any notify whose payload carries an explicit `test_fixture: true` marker. With
channel namespacing the live worker never receives test notifies at all, so this
is belt-and-suspenders only; it costs one `if` and one payload field. Include if
it stays trivial; drop it rather than expand scope.

## Failure Path Test Strategy

- **Notify leak regression:** `test_notify_isolation.py` asserts zero live-worker
  loop starts from a fixture enqueue (AC#1). This is the direct failure-path
  test — it fails today (leak present) and passes after the channel change.
- **Wrong-channel regression:** the companion probe-client assertion fails if a
  future edit reverts the publisher to the literal channel while the subscriber
  stays db-scoped (or vice versa), catching a publish/subscribe channel skew.
- **Live-worker SIGTERM guard:** a unit test for `assert_not_live_worker` that
  feeds it a fabricated "live worker" PID (a self-spawned `python -m worker`-argv
  proc) and asserts it raises; and feeds it a non-worker PID and asserts it
  passes.
- **db=0 production parity:** a unit test asserting `notify_channel_for` returns
  exactly `valor:sessions:new` for a db=0 client, so production wire behavior is
  provably unchanged.

## Rabbit Holes

- **Do not** stand up a second Redis instance or port for tests — the db-derived
  channel name is sufficient and far cheaper. Explicitly out of scope.
- **Do not** rename the production channel or change its wire format — db=0 must
  keep emitting exactly `valor:sessions:new` for every non-test process.
- **Do not** attempt to make the live worker "test-aware" by parsing worker_key
  prefixes as the primary defense — that is the rejected fragile option.
- **Do not** broaden the service-isolation audit beyond the three named
  lifecycle test files unless the audit surfaces a fourth real-kill site.

## Risks

- **Channel skew:** if publisher and subscriber ever derive the channel from
  different clients, notifies silently vanish. Mitigation: both call the single
  helper against the same `POPOTO_REDIS_DB` symbol; the wrong-channel regression
  test catches skew.
- **Test-spawned worker on wrong db:** a lifecycle test that spawns a worker but
  connects it to db=0 would rejoin the production channel. Mitigation: spawned
  workers must inherit the claimed test db (via `redis_test_url` / derived from
  `POPOTO_REDIS_DB`), documented in `tests/README.md`.
- **worker.log absent in CI:** the AC#1 test must skip/xfail cleanly when there
  is no live worker log, so it never produces a false failure in CI. It is a
  live-machine guard by nature; CI validates the full suite separately.

## Race Conditions

- The notify healthcheck watchdog (`_notify_healthcheck_watchdog`) probes NUMSUB
  for the channel; it must probe the SAME db-derived channel the listener
  subscribed to, or it will spuriously report 0 subscribers and force a
  resubscribe loop. Mitigation: compute the channel once at listener setup and
  pass it to the watchdog, rather than each re-deriving a literal.
- Under xdist, each process claims a unique db (#2060) → a unique channel, so
  concurrent pytest processes cannot cross-deliver notifies. No new cross-worker
  race is introduced.

## No-Gos (Out of Scope)

- No mocking of Redis pub/sub — the real publish/subscribe path stays live.
- No change to production db=0 channel name or payload schema.
- No new Redis server/instance/port.
- No refactor of the `redis_test_db` claim mechanism itself.
- No fix for #2146's incident forensics beyond the verify-and-harden audit.

## Update System

No update-system changes required. The channel-derivation helper is internal to
`agent/agent_session_queue.py`; it introduces no new dependency, config file,
env var, or migration step. Existing installations pick it up on the next code
sync/worker restart with zero manual action — db=0 workers keep subscribing to
the identical `valor:sessions:new` channel, so a mixed-version fleet (some
machines updated, some not) still interoperates on the production channel.

## Agent Integration

No agent integration required. This is a worker/test-internal change: no new CLI
entry point in `pyproject.toml [project.scripts]`, and the bridge already
enqueues via `enqueue_agent_session`, whose external contract (channel name on
db=0) is unchanged. The agent reaches nothing new. Integration coverage is the
`test_notify_isolation.py` end-to-end enqueue→channel assertion, which exercises
the real path the bridge uses.
