### Technical Approach

- The 6 named punts dominate; the 13 sweep-extras are mostly narrative mentions of the trigger phrases inside critique tables, footnotes, or rationale sections rather than actual unfinished work. Each is classified during the audit but most produce no action.
- The launchd plist follows the `install_worker.sh` pattern (the canonical "load env vars from `.env` and bake into the plist" template): `launchctl bootstrap gui/$UID $HOME/Library/LaunchAgents/<plist>` with `KeepAlive=true` and `RunAtLoad=true`. The `valor-service.sh email-start` command remains the operational entry point; the plist is for boot-time auto-start on machines that opt in.
- The installer **must** inject `.env` values into the plist's `EnvironmentVariables` block using the same `dotenv_values()` Python block as `install_worker.sh` (lines 86–131 of that script). Without this, the launchd-loaded email bridge would block trying to `open()` the iCloud-synced `~/Desktop/Valor/.env` (macOS TCC denies launchd agents access to ~/Desktop). The plist also sets `VALOR_LAUNCHD=1` so `bridge/routing.py` skips its own iCloud-path reads.
- The bridge module itself **must** gate its own `load_dotenv` calls on `VALOR_LAUNCHD`. `bridge/email_bridge.py::main` (lines 1008-1012) currently calls `load_dotenv` unconditionally for both the repo `.env` and `~/Desktop/Valor/.env`. Under launchd that second call hits TCC and hangs, regardless of whether the installer baked env vars into the plist — the env injection only fixes the plist, not the bridge's own re-read at startup. Fix mirrors `bridge/telegram_bridge.py:42-48`: wrap the two `load_dotenv` calls in `if not os.environ.get("VALOR_LAUNCHD"):`.
- The installer **must** copy `~/Desktop/Valor/projects.json` to `config/projects.json` before bootstrap, mirroring `install_worker.sh:36-48` (`_copy_config_file`). Without this, `bridge/routing.py` falls through to `config/projects.json` under `VALOR_LAUNCHD=1` and that file may be stale or missing, breaking project routing on a freshly-installed machine.
- The installer self-gates on machine ownership via a small Python check that reads `~/Desktop/Valor/projects.json` (same source as `valor-service.sh has_bridge_role`) and exits 0 with a "skip" message if no project on this machine has an `email:` block. This prevents non-email machines from carrying a stale plist.
- Filed issues are already filed (#1335, #1336, #1337) — the build step references them rather than creating them, eliminating the placeholder-rewrite step from the original plan.
- Sweep regex source-of-truth is `validate_no_gos_justification.py::PUNT_PHRASES` (9 patterns, not the 5-pattern subset in the issue body). Both the audit-builder's sweep and the audit-validator's re-sweep use this canonical list.

## Risks

### Risk 1: Audit miscounts an item as "already done" when it isn't.
**Impact:** A real punt slips through the audit and stays unaddressed.
**Mitigation:** Each "already-done" classification cites a specific PR number and a `grep` command that confirms zero remaining matches. Reviewer can replay the grep.

### Risk 2: The new `com.valor.email-bridge.plist` conflicts with operator-installed launchd entries.
**Impact:** `launchctl bootstrap` fails with "service already loaded".
**Mitigation:** The installer runs `launchctl bootout` first (existing pattern in `install_worker.sh`). Idempotent.

### Risk 3: launchd-loaded email bridge hangs on iCloud-synced `~/Desktop/Valor/.env`.
**Impact:** macOS TCC blocks launchd agents from `open()`-ing `~/Desktop` files. Without `.env` injection, the email bridge would freeze on import-time `dotenv` reads, exactly as the worker did before its env-injection block landed.
**Mitigation:** The installer injects `.env` values into the plist's `EnvironmentVariables` dict at install time (mirroring `install_worker.sh` lines 86–131) and sets `VALOR_LAUNCHD=1` so `bridge/routing.py` skips its own iCloud reads. Verified by running `launchctl print gui/$UID/com.valor.email-bridge` after install — `EnvironmentVariables` must contain the expected keys.

### Risk 4: Plist installs on machines without email-configured projects, leaving a stale launchd entry.
**Impact:** Operator runs the installer on a Telegram-only machine, the plist boots an email bridge with no projects to serve, log noise builds, and `valor-service.sh status` may misreport.
**Mitigation:** The installer inspects `projects.json` (via the same `~/Desktop/Valor/projects.json` path that `has_bridge_role` uses) and exits 0 with a "skip" message if no project on this machine has an `email:` block. The check is a small Python snippet identical in shape to `has_bridge_role`.

### Risk 5: bridge/email_bridge.py's own load_dotenv calls hang under launchd.
**Impact:** Even with `VALOR_LAUNCHD=1` in the plist and env vars baked in by the installer, `bridge/email_bridge.py::main` (lines 1008-1012) calls `load_dotenv` unconditionally on both the repo `.env` and `~/Desktop/Valor/.env`. The second call hits TCC at `open()` and freezes the bridge process indefinitely.
**Mitigation:** Step 1 patches `email_bridge.py::main` to wrap both `load_dotenv` calls in `if not os.environ.get("VALOR_LAUNCHD"):`, mirroring `bridge/telegram_bridge.py:42-48` which already implements this pattern correctly. A new unit test `test_main_skips_dotenv_under_launchd` regression-guards the gate.
