## Solution

### Key Elements

- **Audit table** — one row per punt, columns: location, status (still-pending / already-done / external), action taken (fix / cite / file-issue), evidence (PR #, commit, or new file path).
- **launchd plist + installer** — `config/launchd/com.valor.email-bridge.plist` and `scripts/install_email_bridge.sh`, modelled on `install_worker.sh`.
- **Filed issues** — at minimum: `[EXTERNAL]` issue for index-staleness investigation (kill-is-terminal Fix E), tracked issue for overlay-drift audit (pm-workflow-announcement:334), and a tracked issue for the `/update` skill multi-machine overlay step (pm-workflow-announcement:361).
- **Audit report comment** on #1325 listing what got done vs. what got re-tracked, including the 13 sweep-extra hits.

### Flow

`grep` sweep → triage table → for each row { do-it | cite-evidence | file-issue } → post audit comment on #1325 with the full table.

### Technical Approach

- The 6 named punts dominate; the 13 sweep-extras are mostly narrative mentions of the trigger phrases inside critique tables, footnotes, or rationale sections rather than actual unfinished work. Each is classified during the audit but most produce no action.
- The launchd plist follows the `install_worker.sh` pattern: `launchctl bootstrap gui/$UID config/launchd/<plist>` with `KeepAlive=true` and `RunAtLoad=true`. The `valor-service.sh email-start` command remains the operational entry point; the plist is for boot-time auto-start.
- Filed issues use `/do-investigation-issue` for the index-staleness one (it is genuinely an investigation), and plain `gh issue create` for the others with the `[EXTERNAL]` justification embedded.

## Rabbit Holes

- **Re-running the new No-Gos validator on completed plans.** Tempting but wrong: the validator governs new and revised plans; rewriting completed plan files retroactively destroys the historical record.
- **Building a generic punt-detection reflection.** The 19 hits include many false positives (footnotes, narrative). A keyword-grep reflection would re-flag them weekly. The validator already prevents *new* punts; this audit is one-time.
- **Migrating the iCloud master overlay into the repo.** It's deliberately private. Out of scope; only the drift-check is.
- **Index-staleness diagnosis itself.** Filing the investigation issue is in scope; conducting the diagnosis is not — that's the issue's job.

## Risks

### Risk 1: Audit miscounts an item as "already done" when it isn't.
**Impact:** A real punt slips through the audit and stays unaddressed.
**Mitigation:** Each "already-done" classification cites a specific PR number and a `grep` command that confirms zero remaining matches. Reviewer can replay the grep.

### Risk 2: The new `com.valor.email-bridge.plist` conflicts with operator-installed launchd entries.
**Impact:** `launchctl bootstrap` fails with "service already loaded".
**Mitigation:** The installer runs `launchctl bootout` first (existing pattern in `install_worker.sh`). Idempotent.
