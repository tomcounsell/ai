# Inline `<private>` tag to exclude wrapped content from memory ingestion

## Problem

The subconscious memory system silently saves every Telegram message and Claude Code prompt that clears the trivial-pattern + min-length filters. A user pasting a one-off API key, a confidential client name, a home address, or a speculative comment ("hypothetically, if we fired Y…") has no per-message escape hatch — by the time they notice, the content is in Redis, in the bloom filter, and in the BM25 index. Post-hoc deletion via `python -m tools.memory_search forget --id …` is too late: the content has already been recallable in any session that ran in the meantime.

**Current behavior:**

Two ingestion call sites accept user input verbatim:

- `bridge/telegram_bridge.py:956` — `Memory.safe_save(content=text[:500], importance=HUMAN, source="human")`.
- `.claude/hooks/hook_utils/memory_bridge.py:740` (called from `.claude/hooks/user_prompt_submit.py:49` via `ingest()`) — `Memory.safe_save(content=stripped[:500], importance=6.0, source=SOURCE_HUMAN)`.

The same Telegram message text also lands in `TelegramMessage.content` (full untruncated, via `tools/telegram_history/__init__.py:store_message()` from `bridge/telegram_bridge.py:920`), in `AgentSession.message_text` (via `clean_text` at `bridge/telegram_bridge.py:1064`, `1465`), and in bridge log lines (`bridge/telegram_bridge.py:1007` logs `text[:50]`).

There is no existing private/redact infrastructure in the codebase.

**Desired outcome:**

The user wraps any portion of a Telegram message or Claude Code prompt in `<private>...</private>`. The wrapped content is **stripped before any persistent write** (Memory, TelegramMessage, AgentSession, bridge logs). The wrapped content **remains visible to the live agent in the current turn** — the user is masking *future* recall, not hiding from Claude in the moment. Stripping is forward-only; existing records are unaffected.

7. **AgentSession enqueue** uses `clean_text`/`augmented_text` → message_text in Redis. **SWITCH to `safe_clean_text`/`safe_augmented_text`** for the persisted `message_text` field; the bridge's live-agent prompt construction (the SDK invocation) keeps using `clean_text`/`text`.
8. **Live agent prompt construction** (worker spawning Claude SDK) — the agent receives the wrapped tags + content (the user is asking about it in real time). **DO NOT STRIP HERE.** The wrapped content is what the agent answers about in this turn (OQ1 resolution).

2. **UserPromptSubmit hook** (`.claude/hooks/user_prompt_submit.py:38`) receives `prompt = hook_input["prompt"]`.

## Solution

### Key Elements

- **`strip_private(text: str) -> str`**: A pure helper that removes `<private>...</private>` regions from a string. Non-greedy, single-level (no nesting), case-sensitive, idempotent. Cleans up adjacent whitespace so `the key is <private>X</private>, why?` → `the key is, why?` becomes `the key is , why?` → `the key is, why?` after collapse (single-space normalization on the seam only). Unmatched opening tags (`<private>` with no closing tag) are left as literal text.
- **Call-site preprocessing**: Apply `strip_private()` immediately before every persistent write of user-input text — Memory ingestion (Telegram + hook), TelegramMessage storage, AgentSession.message_text construction, and the bridge `text[:50]` log line.
- **Persona segment update**: Add a short directive to the global agent persona explaining that `<private>...</private>` tags mark do-not-quote-back content. This handles the Path C concern (agent response → memory_extraction → Memory).

### Flow

User types message with `<private>X</private>` → Bridge / hook receives full text →
`strip_private()` runs at every persistence boundary → Memory + TelegramMessage + logs hold the stripped version → Live agent prompt holds the original (tags + content) for this turn → Agent answers, may reference X in the current turn → Post-session extraction (if applicable) reads agent response; persona instructs the agent not to quote `X` back, reducing residual leakage → Future sessions never see `X` in recall.

### Technical Approach

1. **Create `agent/private_tag.py`** (chosen over `tools/private_tag.py` because the parser is import-cycle-safe in `agent/` and is conceptually agent-input preprocessing, alongside `agent/memory_*` modules):

   ```python
   import logging
   import re

   _PRIVATE_TAG_RE = re.compile(r"<private>(.*?)</private>", re.DOTALL)
   _MULTI_SPACE_RE = re.compile(r"[ \t]{2,}")

   logger = logging.getLogger(__name__)


   def strip_private(text: str) -> str:
       """Remove all <private>...</private> regions from text.

       Non-greedy, single-level (no nesting), case-sensitive.
       Unmatched <private> openers without a matching </private>
       are left as literal text. Idempotent.

       Collapses the multi-space residue left when a tagged region
       sat between two whitespace-bearing tokens, but ONLY when at
       least one tag was actually stripped (Risk 4 / C2 fix). On
       no-tag input, returns the input bit-identically — no
       whitespace changes whatsoever — so the function is a true
       no-op for the overwhelming majority of messages.

       Does NOT touch newlines (so multi-line prompt structure is
       preserved).

       Emits a DEBUG log line when stripping reduces the content to
       empty / whitespace-only (N3): operationally this means the
       downstream length filter in ``ingest()`` will drop the
       record, and a user diagnosing "why didn't my message land?"
       can grep ``private_tag.strip_to_empty`` in the logs.
       """
       if not text or not isinstance(text, str):
           return text or ""
       stripped, n = _PRIVATE_TAG_RE.subn("", text)
       if n == 0:
           # No tags matched — return input unchanged (no collapse).
           return text
       stripped = _MULTI_SPACE_RE.sub(" ", stripped)
       if not stripped.strip():
           logger.debug(
               "private_tag.strip_to_empty original_len=%d tags_stripped=%d",
               len(text),
               n,
           )
       return stripped
   ```

   - `store_message(content=safe_text, ...)` — line ~922 (inbound persistence).
   - `Memory.safe_save(content=safe_text[:500], ...)` — line ~959 (persistence).
   - `logger.info(f"{sender_name} in {chat_title or 'DM'}: {safe_text[:50]}...")` — line ~1007 (persistence/log).
   - `clean_text = clean_message(text, project)` — line ~1025. **Use the original `text`** so the live agent sees wrapped content this turn. Persistence happens later in the handler at the AgentSession.enqueue site.
   - **AgentSession.message_text — completed-resume path (line ~1433-1437, enqueue at line ~1465).** The `augmented_text` constructed by `_build_completed_resume_text(completed, clean_text, reply_chain_context=reply_chain_context)` is persisted to `AgentSession.message_text` via `dispatch_telegram_session(message_text=augmented_text, ...)` at line ~1465. **Both inputs leak**: (a) `clean_text` is built from unstripped `text`, and (b) `reply_chain_context` is `format_reply_chain(chain)` from prior messages that may themselves have contained `<private>` markers persisted before this PR shipped — or from the same session if a user typed `<private>` content earlier. **Fix**: compute `safe_clean_text = clean_message(safe_text, project)` and `safe_reply_chain_context = strip_private(reply_chain_context) if reply_chain_context else None`, then call `_build_completed_resume_text(completed, safe_clean_text, reply_chain_context=safe_reply_chain_context)` and pass the result as `augmented_text` to `dispatch_telegram_session`. Note: `_build_completed_resume_text` itself does **not** need modification — both inputs are pre-stripped.
   - **AgentSession.message_text — fresh-message path (line ~1947-1949, enqueue at line ~1465 via the prehydration path).** At line ~1947-1949 the bridge prepends `reply_chain_context` to `enqueued_message_text` to seed cross-turn context: `enqueued_message_text = f"{reply_chain_context}\n\nCURRENT MESSAGE:\n{enqueued_message_text}"`. The `reply_chain_context` is `format_reply_chain(chain)` (unstripped). **Fix**: replace with `safe_reply_chain_context = strip_private(reply_chain_context)` and splice that into `enqueued_message_text` instead. The `enqueued_message_text` itself must already be `safe_clean_text`-derived per the rule above.

Use `clean_text` for the live agent prompt construction (the SDK input the worker spawns); use `safe_clean_text` everywhere the bridge writes to `AgentSession.message_text` or its derivatives (`augmented_text`, `enqueued_message_text`). Whenever `reply_chain_context` is concatenated into a persisted field, route it through `strip_private()` first. Net: +2 variables (`safe_clean_text`, `safe_reply_chain_context`), ~+8 LOC.

The persona segment is the right layer; if empirical leak rate is non-zero post-ship, the right fix is a follow-up issue scoped to `agent/memory_extraction.py` and a possible outbound-strip pass — covered explicitly under Risk 3 and the No-Gos list. 4. **Persona segment update** (`config/personas/segments/`): Add a brief directive — file `config/personas/segments/private-tag.md` with a one-paragraph instruction:

   > When the user wraps content in `<private>...</private>` tags, treat that content as ephemeral: visible to you in the current turn for context, but do NOT quote it back verbatim in your response or include it in summaries. The user has marked it as do-not-persist; quoting it back risks re-introducing it into stored agent outputs.

   **Wiring (C3 decision — universal, defense-in-depth):** Append `"private-tag.md"` to `config/personas/segments/manifest.json["segments"]`. The manifest's existing docstring (`"All personas render all segments -- no skipping."`) is the established contract; `_assemble_segments()` in `agent/sdk_client.py:827` reads the manifest unconditionally for every persona. We deliberately do **not** add a session-type gate.

   **Why universal over Teammate-only:** the Operator critic raised C3 (segment loaded universally is dead weight in PM/Dev because they consume `safe_text` already). True for the Telegram inbound path, but two paths break that assumption: (a) PM/Dev sessions read the **outbound** Telegram log via `valor-telegram` and may see another agent's quote-back containing tags, and (b) any Claude Code session (PM, Dev, Teammate) directly reads the user's typed `<private>` content from the prompt because the hook's strip is for the **persisted** Memory, not the live prompt the agent sees (Path B step 5). A persona that doesn't know what `<private>` means is the cheapest possible leak source. The directive is one paragraph (~50 tokens), well below the cost of a custom gate. Trade-off accepted.

   No loader-code changes needed — the wire-up is exactly the one-line manifest append (OQ2 resolved).

### Decision: scope of stripping (resolves issue body's open question)

**Strip from Memory + TelegramMessage + AgentSession.message_text + bridge logs. Live agent prompt retains the original.** Rationale:

- The user's intent ("don't persist this") cannot be honored half-way. If `Memory` is stripped but `TelegramMessage.content` retains the secret, `valor-telegram read --search "sk-abc"` still returns it, and `valor-telegram` is itself a tool the agent can invoke. A bifurcated strip leaks through any persistent surface that stores the same input.
- The 200-line "broad" version is actually closer to 55 lines — there are only 4 write call sites in the bridge plus 2 in the hook, all reachable from one `safe_text` precomputation, plus a small `unsafe_text`/`clean_text` split for live-agent visibility.
- Trade-off accepted (per issue body): `valor-telegram read --search` will not find wrapped content in user scrollback. This is intentional — the user opted into that by tagging.
- **Live agent visibility (OQ1 resolved):** The live Telegram-path agent receives the original `clean_text` (built from unstripped `text`), satisfying the issue's third acceptance criterion ("the wrapped content IS still visible to the live agent in the current turn"). Persistence sites (Memory, TelegramMessage, AgentSession.message_text, logs) use `safe_text`/`safe_clean_text`. This adds one variable and ~5 LOC. The persona segment in Step 4 trains the agent to treat tags as do-not-quote-back so wrapped content does not re-enter Memory via post-session extraction.

### Decision: do we strip the agent persona's view of tags?

**No. The persona segment teaches the agent about the tag instead.** Rationale: stripping the tags from the agent's prompt would lose the signal that *the user marked this private*. The agent's correct behavior is to treat wrapped content as ephemeral context — that requires seeing the tag boundary. The persona segment makes this explicit.

## Rabbit Holes

- **Encrypting historical Memory records.** Out of scope (issue says forward-only). Tempting because "what about the secrets already in Redis?" but tackling that requires a key-management story we don't need.
- **Auto-detection of API keys / credit cards / addresses.** Out of scope (issue is opt-in only). Adding regex detection invites false positives and a long policy debate.
- **Per-user / per-chat "never save anything from this person" policy.** Different concern, separate issue.
- **Stripping inside the post-session Haiku extraction.** Tempting — but the Haiku reads the *agent's response*, not the user's prompt. The persona segment is the right layer; modifying `extract_observations_async` is the wrong layer.
- **Nested `<private>` tags.** The parser is single-level. Supporting nesting requires a real parser instead of a regex; the marginal value is zero (no realistic use case).
- **Case-insensitive matching (`<PRIVATE>`, `<Private>`).** Skipped. The tag is a deliberate marker; users learn the lowercase form. Case-insensitive matching invites accidental triggers in casual text.
- **Stripping from `bridge/response.py:clean_message()` directly.** Tempting because that's a "do all text cleanup in one place" instinct. Resisted because `clean_message` runs on `safe_text` already; pushing the strip into it couples a project-aware cleaner with a privacy-aware stripper, mixing concerns.

## Risks

### Risk 1: Wrapped content leaks via a fourth call site we didn't enumerate
**Impact:** A new persistence path lands in main between this plan and ship date, and we forget to apply `strip_private()` there. The user's secret leaks into that path.
**Mitigation:** A `grep -rn "Memory.safe_save\|store_message" .` audit check is added to the Verification table. Reviewers catch new ingestion sites in subsequent PRs. The persona segment provides defense in depth.

### Risk 2: Regex catastrophic backtracking on adversarial input
**Impact:** A user pasting a maliciously crafted string with many overlapping `<private>` openers stalls the bridge.
**Mitigation:** The regex `<private>(.*?)</private>` with non-greedy `.*?` is linear; `re.DOTALL` does not change complexity. Add a unit test with a 10KB input containing 100 `<private>` regions; assert it completes in <50ms.

### Risk 3: Persona segment is ignored by the agent ("but the user just told me about X, surely it's fine to summarize")
**Impact:** Agent quotes wrapped content in its response, which then enters Memory via `extract_observations_async`.
**Mitigation:** The persona segment is short and direct. If empirical leak rate is non-zero post-ship, we add a downstream defense in `extract_observations_async` (trim agent responses before Haiku) — a separate issue, not this plan. Risk accepted for v1.

### Risk 4: Multi-space collapse changes message semantics
**Impact:** A user who relies on intentional multi-spacing (rare) sees their formatting flattened.
**Mitigation:** The C2 fix lands in Step 1: `strip_private()` uses `re.subn` and gates the collapse on `n > 0` (at least one tag actually matched). On no-tag input the function returns the input bit-identically — verified by a unit test (Step 5) asserting `strip_private(x) is x` for no-tag input via `assert id(out) == id(input_text)` or equivalent string-identity check. Only after a tag was stripped does the seam-collapse run, and it leaves newlines and double-newlines untouched.

### Risk 5: Bloom-fingerprint inconsistency around deploy (informational)
**Impact:** The `Memory` model uses `ExistenceFilter(fingerprint_fn=lambda inst: inst.content or "")` (`models/memory.py:109`) — a bloom keyed on the `content` string. Pre-deploy, the `content` fingerprint is computed from the unstripped text; post-deploy, it's computed from the stripped text. So a user who repeatedly sends `"hello <private>secret</private>"` (a) creates one `content="hello <private>secret</private>"` bloom hit before deploy, then (b) creates a separate `content="hello"` bloom hit after deploy. The **same logical message** has two fingerprints across the deploy boundary — neither dedupes the other. Net effect: at most one extra Memory record per crossover pair for in-flight near-duplicates at deploy time. The bloom does NOT have TTL — fingerprints persist for the bloom's lifetime (until capacity rolls or the filter is rebuilt).
**Mitigation:** Forward-only is acceptable per the issue body. No backfill, no migration. Documented for the operator so they know not to chase a "duplicate Memory" anomaly in the first hour post-deploy. If the mismatch creates noise, run `python -m tools.memory_search forget` to drop one of the duplicates by hand. (Archaeologist C1.)

No race conditions identified — `strip_private()` is a pure synchronous function with no shared state. All call sites invoke it before passing the result to existing async I/O paths; the strip itself is in the calling thread's call stack.
