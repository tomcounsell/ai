# Inline `<private>` tag to exclude wrapped content from memory ingestion

## Problem

The subconscious memory system silently saves every Telegram message and Claude Code prompt that clears the trivial-pattern + min-length filters. A user pasting a one-off API key, a confidential client name, a home address, or a speculative comment ("hypothetically, if we fired Y…") has no per-message escape hatch — by the time they notice, the content is in Redis, in the bloom filter, and in the BM25 index. Post-hoc deletion via `python -m tools.memory_search forget --id …` is too late: the content has already been recallable in any session that ran in the meantime.

**Current behavior:**

Two ingestion call sites accept user input verbatim:

- `bridge/telegram_bridge.py:956` — `Memory.safe_save(content=text[:500], importance=HUMAN, source="human")`.
- `.claude/hooks/hook_utils/memory_bridge.py:740` (called from `.claude/hooks/user_prompt_submit.py:49` via `ingest()`) — `Memory.safe_save(content=stripped[:500], importance=6.0, source=SOURCE_HUMAN)`.

The same Telegram message text also lands in `TelegramMessage.content` (full untruncated, via `tools/telegram_history/__init__.py:store_message()` from `bridge/telegram_bridge.py:920`), in `AgentSession.message_text` (via `clean_text` at `bridge/telegram_bridge.py:1064`, `1465`), and in bridge log lines (`bridge/telegram_bridge.py:1007` logs `text[:50]`).

There is no existing private/redact infrastructure in the codebase.

**Desired outcome:**

The user wraps any portion of a Telegram message or Claude Code prompt in `<private>...</private>`. The wrapped content is **stripped before any persistent write** (Memory, TelegramMessage, AgentSession, bridge logs). The wrapped content **remains visible to the live agent in the current turn** — the user is masking *future* recall, not hiding from Claude in the moment. Stripping is forward-only; existing records are unaffected.

## Data Flow

Trace the path of a user input containing `<private>SECRET</private>` through the system:

### Path A: Telegram message

1. **Entry**: User sends Telegram message `the key is <private>sk-abc123</private>, why?` to a whitelisted chat.
2. **`bridge/telegram_bridge.py` event handler** receives the message; `text = message.message`.
3. **`store_message()` call (line 920)** writes `TelegramMessage.content = text` to Redis (full untruncated). **STRIP HERE** — call `strip_private(text)` before passing to `store_message`. Otherwise the secret persists in Redis history and surfaces in `valor-telegram read --search`.
4. **`Memory.safe_save()` call (line 956)** writes `content=text[:500]` to Memory. **STRIP HERE** — call `strip_private(text)` before truncation.
5. **`logger.info(...text[:50]...)` (line 1007)** logs the first 50 chars to `logs/bridge.log`. **STRIP HERE** — log `strip_private(text)[:50]` instead.
6. **`clean_message(text, project)` (line 1025)** produces `clean_text` used for routing and AgentSession.message_text. **STRIP HERE** — `clean_text = clean_message(strip_private(text), project)`.
7. **AgentSession enqueue** uses `clean_text` (or `augmented_text` derived from it) → message_text in Redis. Already covered by step 6.
8. **Live agent prompt construction** (worker spawning Claude SDK) — the agent receives the wrapped tags + content (the user is asking about it in real time). **DO NOT STRIP HERE.** The wrapped content is what the agent answers about in this turn.

### Path B: Claude Code user prompt

1. **Entry**: User types `the key is <private>sk-abc123</private>, why?` into Claude Code.
2. **UserPromptSubmit hook** (`.claude/hooks/user_prompt_submit.py:38`) receives `prompt = hook_input["prompt"]`.
3. **`memory_bridge.ingest(prompt, cwd=cwd)` (line 49)** — inside `ingest()`, `stripped = content.strip()` (line 714), then `Memory.safe_save(content=stripped[:500], ...)` (line 740). **STRIP HERE** — apply `strip_private()` to `content` at the top of `ingest()` *before* the length check.
4. **Prefetch path** (`prefetch()` in same module) — uses the original `prompt` for BM25 query against past memories. **STRIP HERE TOO** — strip the prompt before BM25 query, otherwise wrapped content seeds the lookup vector (low risk but cheap to fix).
5. **Live agent prompt** — the prompt with tags + content reaches Claude Code in the current turn. **DO NOT STRIP HERE.**

### Path C: Post-session memory extraction (out of direct scope, but called out)

`agent/memory_extraction.py:extract_observations_async` runs Haiku on the **agent's response_text** (not user input). If the agent quotes `<private>` content back in its response, the wrapped content can re-enter Memory through this path. The plan handles this defensively: the agent persona segment update (Step 4) instructs the agent to treat `<private>` content as do-not-quote-back. We **do not** modify `extract_observations_async` itself in this plan — that's a separate concern about agent behavior, not user-input ingestion. Documented in **No-Gos** below.

## Solution

### Key Elements

- **`strip_private(text: str) -> str`**: A pure helper that removes `<private>...</private>` regions from a string. Non-greedy, single-level (no nesting), case-sensitive, idempotent. Cleans up adjacent whitespace so `the key is <private>X</private>, why?` → `the key is, why?` becomes `the key is , why?` → `the key is, why?` after collapse (single-space normalization on the seam only). Unmatched opening tags (`<private>` with no closing tag) are left as literal text.
- **Call-site preprocessing**: Apply `strip_private()` immediately before every persistent write of user-input text — Memory ingestion (Telegram + hook), TelegramMessage storage, AgentSession.message_text construction, and the bridge `text[:50]` log line.
- **Persona segment update**: Add a short directive to the global agent persona explaining that `<private>...</private>` tags mark do-not-quote-back content. This handles the Path C concern (agent response → memory_extraction → Memory).

### Flow

User types message with `<private>X</private>` → Bridge / hook receives full text →
`strip_private()` runs at every persistence boundary → Memory + TelegramMessage + logs hold the stripped version → Live agent prompt holds the original (tags + content) for this turn → Agent answers, may reference X in the current turn → Post-session extraction (if applicable) reads agent response; persona instructs the agent not to quote `X` back, reducing residual leakage → Future sessions never see `X` in recall.

### Technical Approach

1. **Create `agent/private_tag.py`** (chosen over `tools/private_tag.py` because the parser is import-cycle-safe in `agent/` and is conceptually agent-input preprocessing, alongside `agent/memory_*` modules):

   ```python
   import re

   _PRIVATE_TAG_RE = re.compile(r"<private>(.*?)</private>", re.DOTALL)
   _MULTI_SPACE_RE = re.compile(r"[ \t]{2,}")

   def strip_private(text: str) -> str:
       """Remove all <private>...</private> regions from text.

       Non-greedy, single-level (no nesting), case-sensitive.
       Unmatched <private> openers without a matching </private>
       are left as literal text. Idempotent.

       Collapses the multi-space residue left when a tagged region
       sat between two whitespace-bearing tokens; does NOT touch
       newlines (so multi-line prompt structure is preserved).
       """
       if not text or not isinstance(text, str):
           return text or ""
       stripped = _PRIVATE_TAG_RE.sub("", text)
       stripped = _MULTI_SPACE_RE.sub(" ", stripped)
       return stripped
   ```

2. **Apply at all persistent-write call sites in `bridge/telegram_bridge.py`**:
   - Compute `safe_text = strip_private(text)` once after reading the message (at the top of the message handler, after `text = message.message`).
   - `store_message(content=safe_text, ...)` — line ~922.
   - `Memory.safe_save(content=safe_text[:500], ...)` — line ~959.
   - `logger.info(f"{sender_name} in {chat_title or 'DM'}: {safe_text[:50]}...")` — line ~1007.
   - `clean_text = clean_message(safe_text, project)` — line ~1025. This also flows into AgentSession.message_text via `clean_text` and `augmented_text`.
   - **Live agent prompt**: untouched. The agent receives the original `text` (or its `clean_text` derivative) when the worker spawns the SDK subprocess. Note: because `clean_text` now derives from `safe_text`, the live agent ALSO won't see the wrapped content. This is a deliberate scope decision — see **Decision** below.

3. **Apply at the hook ingestion path in `.claude/hooks/hook_utils/memory_bridge.py`**:
   - At the top of `ingest()` (after the `isinstance` guard, before the strip-and-length check), add: `content = strip_private(content)` (line ~712).
   - In `prefetch()` (line ~617), replace `stripped = _strip_pm_boilerplate(prompt).strip()` with `stripped = _strip_pm_boilerplate(strip_private(prompt)).strip()`. This prevents wrapped content from seeding the BM25 query against existing memories.
   - **Live agent prompt**: untouched. Claude Code receives the original `prompt` from the hook input; the hook only side-effects (memory ingest + prefetch return), it does not modify the prompt the agent sees.

4. **Persona segment update** (`config/personas/segments/`): Add a brief directive — file `config/personas/segments/private-tag.md` with a one-paragraph instruction:

   > When the user wraps content in `<private>...</private>` tags, treat that content as ephemeral: visible to you in the current turn for context, but do NOT quote it back verbatim in your response or include it in summaries. The user has marked it as do-not-persist; quoting it back risks re-introducing it into stored agent outputs.

   Wire this segment into the existing persona composition so it's loaded for all session types (PM, Dev, Teammate). Inspect `config/personas/segments/` to find the existing composition pattern (e.g., a registry list) and append.

### Decision: scope of stripping (resolves issue body's open question)

**Strip from Memory + TelegramMessage + AgentSession + bridge logs.** Rationale:

- The user's intent ("don't persist this") cannot be honored half-way. If `Memory` is stripped but `TelegramMessage.content` retains the secret, `valor-telegram read --search "sk-abc"` still returns it, and `valor-telegram` is itself a tool the agent can invoke. A bifurcated strip leaks through any persistent surface that stores the same input.
- The 200-line "broad" version is actually closer to 50 lines — there are only 4 write call sites in the bridge plus 2 in the hook, all reachable from one `safe_text` precomputation.
- Trade-off accepted (per issue body): `valor-telegram read --search` will not find wrapped content in user scrollback. This is intentional — the user opted into that by tagging.
- **Live agent visibility**: Because `clean_text` derives from `safe_text`, the **agent in the current Telegram turn will not see the wrapped content** either. This narrows the issue's "wrapped content IS still visible to the live agent" criterion to **the Claude Code path only** (where the prompt is unchanged before reaching Claude). This is a sharper interpretation than the issue body proposed; we accept it because (a) it's simpler, (b) Telegram users who genuinely need the agent to act on a secret in real time can simply not wrap it, and (c) the persona segment in Step 4 trains the agent to treat tags as do-not-quote-back regardless. **This decision is open for revision in critique** — see Open Questions.

### Decision: do we strip the agent persona's view of tags?

**No. The persona segment teaches the agent about the tag instead.** Rationale: stripping the tags from the agent's prompt would lose the signal that *the user marked this private*. The agent's correct behavior is to treat wrapped content as ephemeral context — that requires seeing the tag boundary. The persona segment makes this explicit.

## Rabbit Holes

- **Encrypting historical Memory records.** Out of scope (issue says forward-only). Tempting because "what about the secrets already in Redis?" but tackling that requires a key-management story we don't need.
- **Auto-detection of API keys / credit cards / addresses.** Out of scope (issue is opt-in only). Adding regex detection invites false positives and a long policy debate.
- **Per-user / per-chat "never save anything from this person" policy.** Different concern, separate issue.
- **Stripping inside the post-session Haiku extraction.** Tempting — but the Haiku reads the *agent's response*, not the user's prompt. The persona segment is the right layer; modifying `extract_observations_async` is the wrong layer.
- **Nested `<private>` tags.** The parser is single-level. Supporting nesting requires a real parser instead of a regex; the marginal value is zero (no realistic use case).
- **Case-insensitive matching (`<PRIVATE>`, `<Private>`).** Skipped. The tag is a deliberate marker; users learn the lowercase form. Case-insensitive matching invites accidental triggers in casual text.
- **Stripping from `bridge/response.py:clean_message()` directly.** Tempting because that's a "do all text cleanup in one place" instinct. Resisted because `clean_message` runs on `safe_text` already; pushing the strip into it couples a project-aware cleaner with a privacy-aware stripper, mixing concerns.

## Risks

### Risk 1: Wrapped content leaks via a fourth call site we didn't enumerate
**Impact:** A new persistence path lands in main between this plan and ship date, and we forget to apply `strip_private()` there. The user's secret leaks into that path.
**Mitigation:** A `grep -rn "Memory.safe_save\|store_message" .` audit check is added to the Verification table. Reviewers catch new ingestion sites in subsequent PRs. The persona segment provides defense in depth.

### Risk 2: Regex catastrophic backtracking on adversarial input
**Impact:** A user pasting a maliciously crafted string with many overlapping `<private>` openers stalls the bridge.
**Mitigation:** The regex `<private>(.*?)</private>` with non-greedy `.*?` is linear; `re.DOTALL` does not change complexity. Add a unit test with a 10KB input containing 100 `<private>` regions; assert it completes in <50ms.

### Risk 3: Persona segment is ignored by the agent ("but the user just told me about X, surely it's fine to summarize")
**Impact:** Agent quotes wrapped content in its response, which then enters Memory via `extract_observations_async`.
**Mitigation:** The persona segment is short and direct. If empirical leak rate is non-zero post-ship, we add a downstream defense in `extract_observations_async` (trim agent responses before Haiku) — a separate issue, not this plan. Risk accepted for v1.

### Risk 4: Multi-space collapse changes message semantics
**Impact:** A user who relies on intentional multi-spacing (rare) sees their formatting flattened.
**Mitigation:** The multi-space collapse only runs after a strip occurred (no, actually it runs unconditionally — fix in implementation: only collapse on the seam, gate the collapse on whether any tags were stripped). Test covers no-tag input as a no-op (no collapse).
