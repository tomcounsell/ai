# Read-the-Room Pre-Send Pass for the Telegram Drafter

## Problem

The agent posts redundant or stale messages into shared Telegram chats because the final-message drafter (`bridge/message_drafter.py`) has no awareness of the *room state* at send time. It composes from `(user prompt, tool outputs)` and commits unconditionally — even if another participant has just answered the same question or moved the conversation on.

**Real example from `PM: PBA` (`chat_id=-5238229534`), 2026-04-28 14:39–14:41:**

Three messages from Tom and Chris arrive in rapid succession discussing an unrelated apology ("Oops. Memory…. 😢" / "Now I'm polluting the conversation. Oops"). Mid-exchange, the agent posts "Logged 4 entries...", a now-irrelevant status update that lands as a non-sequitur into a personal moment between two humans.

This is structural, not a one-off. Two failure modes recur:

1. **Conversation-moved-on.** Agent drafts a reply; another participant answers the same question first; agent posts the now-redundant reply seconds later.
2. **Crossing-streams.** Agent finishes a turn and posts; meanwhile a participant has shifted topic; the reply lands as a non-sequitur.

**Current behavior:** `TelegramRelayOutputHandler.send()` (`agent/output_handler.py:162`) calls `draft_message()` (`bridge/message_drafter.py:1623`) to compose the reply, then writes the result straight to the Redis outbox. No check between the draft and the outbox write.

**Desired outcome:** Between the draft and the outbox write, a lightweight Haiku-tier LLM call inspects the recent chat snapshot + the drafted message and returns one of `{send, trim, suppress}`:

- **send** — message lands cleanly, post as-is.
- **trim** — message is partially redundant or partially stale; rewrite to a shorter form (e.g. a brief acknowledgment, a pointer).
- **suppress** — message is fully redundant or the conversation has moved on; do not write to the outbox.

Failures (LLM timeout, parse error) default to `send` — the pass is a guard, not a blocker. Suppress and trim verdicts emit a `session_events` entry so operators can answer "why didn't Valor reply?" without spelunking.

## Data Flow

**Path A (drafter → relay → Telegram), the primary path RTR sits on:**

1. **Entry point**: Worker invokes `TelegramRelayOutputHandler.send(chat_id, text, reply_to_msg_id, session)` after agent turn completes (`agent/output_handler.py:162`).
2. **Drafter**: `draft_message(text, session, medium="telegram")` returns `MessageDraft` with `text`, `was_drafted`, `artifacts`, etc. (`bridge/message_drafter.py:1623`).
3. **NEW — Read-the-Room pass**: `read_the_room(draft_text=delivery_text, chat_id=chat_id, session=session)` returns `RoomVerdict(action, revised_text, reason)`. Source of snapshot: `TelegramMessage.query.filter(chat_id=str(chat_id))` via `get_recent_messages` (last 10 messages or last 5 minutes, whichever is shorter — see Open Question Q2).
4. **Apply verdict**:
   - `send` → continue with `delivery_text` unchanged.
   - `trim` → replace `delivery_text` with `verdict.revised_text`.
   - `suppress` → append a `session_event` (`{type: "rtr.suppressed", reason, chat_id, draft_preview}`), skip outbox write, dual-write to file handler for audit, return.
5. **Outbox write**: `r.rpush("telegram:outbox:{session_id}", json.dumps(payload))` with the (possibly trimmed) text.
6. **Relay**: `bridge/telegram_relay.py` polls outbox, sends via Telethon.
7. **Output**: Telegram chat receives the message (or doesn't, on suppress).

**Path B (`valor-telegram send` from agent Bash):**

`tools/valor_telegram.py:691` `cmd_send` resolves chat → builds payload → writes directly to `telegram:outbox:{session_id}`. **Does NOT pass through `output_handler.py`.** RTR coverage on Path B is its own decision (see Open Question Q4 + No-Gos).

## Solution

### Key Elements

- **`bridge/read_the_room.py`**: A new sibling module exporting one async function `read_the_room(draft_text, chat_id, session) -> RoomVerdict` plus the `RoomVerdict` dataclass and the system prompt. Self-contained; no implicit imports from the drafter.
- **Haiku verdict call**: One Haiku tool_use call returning a flat structured result `{action: "send"|"trim"|"suppress", revised_text: str|None, reason: str}`. Reuses `anthropic_slot()` and the same model constant (`MODEL_FAST`) the drafter uses — no new client.
- **Snapshot fetcher**: Calls `tools.telegram_history.get_recent_messages(chat_id, limit=K)` and post-filters to entries within the time window. **Does NOT try to drop the in-flight draft** — see Implementation Note B2. Last-N-or-last-5-min, whichever is fewer.
- **Verdict application in `TelegramRelayOutputHandler.send`**: Inserted between the existing `draft_message` block and the outbox write. Same fail-open pattern (try/except, default to `send`).
- **Observability**: Suppress and trim verdicts append a `session_event` entry to `session.session_events` with `{type: "rtr.suppressed" | "rtr.trimmed", chat_id, draft_preview, revised_preview, reason, ts}`. Bypasses (Q3 / Q5) emit `{type: "rtr.bypassed", reason}`. Failures emit `{type: "rtr.failed", error}`.
- **Bypass flag**: `READ_THE_ROOM_ENABLED` env var (default `false` for first deploy — explicit opt-in). Plus a per-call short-circuit when **the session is SDLC-flagged** (`bool(session and getattr(session, "sdlc_slug", None))` — same check the drafter uses at `bridge/message_drafter.py:1672`) or session has a `bypass_read_the_room` attribute (handles SDLC pipeline status, urgent error reports — Open Question Q5).

### Flow

Worker turn completes → `TelegramRelayOutputHandler.send(chat_id, text, ...)` → `draft_message()` produces `MessageDraft` → **`read_the_room(draft.text, chat_id, session)`** → branch on verdict:

- `send` (or RTR disabled / bypassed / failed) → existing path: `r.rpush("telegram:outbox:{session_id}", payload)` → relay → Telegram.
- `trim` → swap `delivery_text = verdict.revised_text`, append `session_event`, then `rpush` as above.
- `suppress` → append `session_event`, dual-write to file handler for audit, **skip** `rpush`, return.

### Technical Approach

- **New module**: `bridge/read_the_room.py` with `RoomVerdict` dataclass and `read_the_room()` async function. Mirrors the drafter's structure: imports, constants, prompt, dataclass, function. Keeps `bridge/message_drafter.py` unchanged.
- **Snapshot retrieval**: `await asyncio.to_thread(get_recent_messages, str(chat_id), K)`. The Popoto query is sync; wrapping in `to_thread` keeps the event loop unblocked. Filter the returned messages to `now - max_age_seconds` cutoff. **Pass the snapshot through unfiltered** (no direction/sender filter) — the system prompt tells the model how to interpret entries authored by the agent. See Implementation Note B2.
- **Haiku call**: Reuse `anthropic_slot()` + `MODEL_FAST` + tool_use block with a flat schema `{action: enum, revised_text: string|null, reason: string}`. ~300 input tokens (10-message snapshot + 1 draft + system prompt) + ~50 output. Per-message latency budget: 600–900ms p50 based on Haiku 4.5 typical (validated in Risks below). **Concrete timeout wiring**: wrap the Anthropic call in `asyncio.wait_for(client.messages.create(...), timeout=3.0)` so a stalled connection fails open at the 3-second mark rather than hitting the SDK's default ~10-minute ceiling. See Implementation Note C2.
- **Fail-open**: Wrap the entire RTR call in try/except (catching `asyncio.TimeoutError`, `httpx.HTTPError`, `anthropic.APIError`, `ValueError` for bad tool_use shape, plus a bare `Exception` last-resort). On any exception, log at WARNING with the exception, append a `rtr.failed` session_event, return `RoomVerdict(action="send", revised_text=None, reason="rtr_error")`.
- **Drafter prompt unchanged**: This plan does NOT extend the drafter's prompt with the chat snapshot. (Issue's open question: "should the drafter also see the snapshot?" — see No-Gos. Adding that here doubles the cost and tangles two concerns. Ship RTR first; revisit drafter prompt enrichment in a follow-up if RTR alone proves insufficient.)
- **Path B coverage**: Out of scope for this plan (see No-Gos). `cmd_send` writes directly to the outbox and does not pass through `output_handler.py`. Adding RTR there requires a sync→async bridge (`asyncio.run`) and changes the CLI's latency profile for human-invoked sends. **Commitment**: file a follow-up issue titled "RTR: extend pre-send pass to Path B (`valor-telegram send`)" within 7 days of this plan merging, regardless of measured impact, so the deferral has a closed loop. See Implementation Note C3.
- **Threshold alignment with the drafter**: The drafter early-returns at `len(raw_response) < SHORT_OUTPUT_THRESHOLD` (200 chars, `bridge/message_drafter.py:1620`). RTR uses the **same constant** — short-circuit when `len(delivery_text) < SHORT_OUTPUT_THRESHOLD` — so the latency-bypass band matches the drafter's bypass band exactly. Avoids a 100–200 char "no drafter, yes RTR" zone where the saving from skipping the drafter is lost to the RTR call. See Implementation Note C1.

## Failure Path Test Strategy

### Exception Handling Coverage
- [ ] `bridge/read_the_room.py:read_the_room` — single try/except wrapping the Haiku call; test asserts that on `httpx.TimeoutException`, the function returns `RoomVerdict(action="send", reason="rtr_error")` AND a `rtr.failed` session_event is appended.
- [ ] `agent/output_handler.py:TelegramRelayOutputHandler.send` — RTR call wrapped in its own try/except; test asserts that on RTR raising, delivery proceeds with the original draft text (fail-open).

### Empty/Invalid Input Handling
- [ ] `read_the_room(draft_text="", ...)` — short-circuit return `send` without calling Haiku (no point evaluating an empty draft).
- [ ] `read_the_room(draft_text=..., chat_id=None)` — short-circuit return `send` (no chat to read; e.g. file-only delivery path).
- [ ] `get_recent_messages` returns `{"error": ...}` — treat as empty snapshot and return `send`. Do NOT raise.
- [ ] Empty snapshot (`messages=[]`) — return `send` (clean room, nothing to compare).

- [ ] Suppress verdict — assert no Redis `rpush` happened for the outbox key (use a fakeredis fixture); assert `session.session_events` contains an `rtr.suppressed` entry.

- [ ] Failure verdict — assert outbox payload contains the original draft text; assert `rtr.failed` event with the exception class name in `error`.
- [ ] When `READ_THE_ROOM_ENABLED=false`, assert `read_the_room` is never called (mock and assert_not_called).

## Rabbit Holes

- **Re-prompting the drafter with the snapshot.** Tempting because it would let the drafter compose more aware messages in the first place. But it doubles per-message cost (drafter sees snapshot AND RTR sees snapshot), tangles two concerns into one prompt, and makes regressions harder to attribute. Ship RTR as a clean guard layer first; revisit only if real traffic shows RTR firing too often.
- **Path B (`valor-telegram send`) coverage in this plan.** It would be a CLI-level change with sync→async bridging and CLI latency implications. Defer until we have data on Path A's effect.
- **Building a separate `RoomVerdictLog` Popoto model.** `session_events` already exists, is rendered in the dashboard, and supports the same shape. A second model duplicates infrastructure. Use `session_events` until volume or query needs justify a split.
- **Tuning K and window values upfront.** The right values depend on per-chat velocity. Ship with conservative defaults (K=10, 5-minute window) behind the env-var flag; tune from `rtr.suppressed`/`rtr.trimmed` event counts after the first week.
- **Snapshot caching across drafts.** Sharing one snapshot fetch across multiple sends within the same session feels efficient but risks staleness exactly in the window where it matters (rapid-fire turns). Re-fetch per send.
- **A "why didn't Valor reply?" CLI surface.** Useful but separable. The `session_events` entries are already inspectable via `valor-session inspect --id <id>`; build a dedicated CLI only if operators ask for one.

## Risks

### Risk 1: Per-message latency from the Haiku call
**Impact:** Adding ~600–900ms p50 to every outbound message could make the agent feel less responsive in chat. Worst case (Anthropic API blip): up to the request timeout (~30s) before fail-open kicks in.
**Mitigation:** (a) Set a hard `httpx` timeout of 3 seconds on the RTR call — fail-open at the timeout, not at the network default. (b) Reuse the existing `anthropic_slot()` semaphore so RTR shares the API budget with the drafter; no separate rate limiter. (c) Skip RTR for short non-SDLC outputs (`len(draft) < 100`) — these are usually emoji acks or one-liners where the saving doesn't justify the latency. (d) Roll out behind `READ_THE_ROOM_ENABLED=false` initially; flip in production after measuring real-traffic latency on a single chat.

### Risk 2: Haiku misjudges the room and suppresses a needed message
**Impact:** A user asks the agent something, the RTR pass decides "the conversation moved on" and suppresses, the user is left hanging. False suppression is harder to detect than false trim because there's no signal back to the human.
**Mitigation:** (a) The verdict prompt is conservative-by-default — when uncertain, return `send`. (b) Every `rtr.suppressed` event is logged to `session_events` with the draft preview; an operator can audit and tune the prompt. (c) Add a unit test that verifies a clean chat (snapshot has no recent messages on the same topic) always returns `send`. (d) After rollout, sample suppressed messages weekly and judge — if false-positive rate >10%, tighten the prompt or raise the suppression bar.

### Risk 3: Snapshot includes the agent's *own* prior turns under inconsistent attribution
**Impact:** Path A messages flow into `TelegramMessage` via two different recording sites: (a) the bridge handler path at `bridge/telegram_bridge.py:2381` writes `sender="Valor"` (which evaluates to `direction="out"` per `tools/telegram_history/__init__.py:389`), and (b) the relay's PM-direct path at `bridge/telegram_relay.py:598-608` writes `sender="system"` (which evaluates to `direction="in"`). A naive "drop sender==valor + out" filter misses the system-attributed entries entirely, so the RTR call may fail to see its own prior turns OR see them and treat them as inbound. Either way: false reasoning about the room.
**Mitigation:** **Do not filter the snapshot by direction or sender at all.** The verdict prompt explicitly enumerates the attribution shapes the agent's prior turns can take: "Snapshot entries with `sender` in `{valor, system}` ARE this agent's prior turns. Snapshot entries from any other sender are other participants. Treat agent-authored entries as context for 'what has already been said', not as competing input. If the candidate draft is substantially identical to an agent-authored entry within the last 60 seconds, that is redundancy — return `suppress` with `reason='self_duplicate'`. If the candidate draft adds new information, return `send`." Add a test (see Failure Path Test Strategy) seeding both `sender="Valor"` and `sender="system"` prior turns and asserting the verdict reasons correctly across both attribution shapes. See Implementation Note B2.

### Risk 4: Cost
**Impact:** One Haiku call per outbound message. At 50–100 outbound messages/day across all chats, this is ~3000/month — cheap (~$0.10/month at Haiku rates), but cost compounds if message volume grows.
**Mitigation:** (a) Skip RTR on short outputs and SDLC sessions (already covered above). (b) The `rtr.bypassed` event makes bypass rate observable; if RTR is firing on >90% of sends, the heuristic is wrong. (c) Document the per-call cost in `docs/features/read-the-room.md` so future work can revisit if volume changes.

## Race Conditions

### Race 1: Concurrent sends from the same session
**Location:** `agent/output_handler.py:TelegramRelayOutputHandler.send`
**Trigger:** A worker session emits two outputs in rapid succession (rare but possible — e.g., a tool result followed by a final message). RTR for send #1 reads the snapshot before send #1 lands in `TelegramMessage`. RTR for send #2 reads the snapshot before send #1 has been recorded as `direction=out`. Both verdicts evaluate against the same pre-state.
**Data prerequisite:** The snapshot must reflect the *most recent* `TelegramMessage` rows.
**State prerequisite:** None — RTR is read-only against `TelegramMessage`.
**Mitigation:** Accept the race. RTR is a best-effort guard, not a correctness primitive. The relay records outbound messages back to `TelegramMessage` (per `bridge/telegram_relay.py`), but recording lags the outbox write by tens of ms. If both sends are *redundant in the same way*, both will be sent — the second-line-of-defense is the relay itself, not RTR. Document this in the module docstring.

### Race 2: Snapshot fetch vs inbound message arriving mid-pass
**Location:** `bridge/read_the_room.py:read_the_room`
**Trigger:** A new inbound message lands in `TelegramMessage` after the snapshot fetch but before the verdict returns. RTR's view is stale by ~1s.
**Data prerequisite:** Snapshot is a point-in-time view.
**State prerequisite:** None.
**Mitigation:** Accept the race. The window is small (sub-second to ~3s in worst case) and the event we'd miss is at most the most recent inbound. The post-RTR outbox write is async with the relay's send anyway, so even a "perfect" snapshot would race the actual send. Document the freshness window in the module docstring; do not paper over it with locking.

The one observable surface is `session_events` — already inspectable via `valor-session inspect --id <id>` and the `/dashboard.json` endpoint. No new tool needed for visibility.

## Open Questions

1. **Snapshot K and window defaults.** Plan proposes K=10 messages, last 5 minutes (whichever shorter). Do you want different starting values, or are these acceptable for first rollout?
2. **Path B (`valor-telegram send`) coverage.** Plan defers Path B to a follow-up issue. Do you want Path B in scope for this plan, accepting the CLI latency cost? Or confirm: ship Path A first, measure, then decide.
3. **Bypass conventions.** Plan auto-bypasses for SDLC sessions (`session.session_type == "sdlc"`) and short outputs (`len(draft) < 100`). Want any other automatic bypass conditions (e.g., explicit "urgent" tag, error-report messages)? Or is "the agent or operator can flip the flag" enough?
4. **Drafter prompt enrichment with snapshot.** Plan defers; the drafter is unaware of the room. Confirm: ship RTR alone, or also augment the drafter prompt now (doubles per-message Haiku context cost, tangles two concerns).
5. **`session_events` shape.** Plan uses a flat schema `{type, chat_id, draft_preview, revised_preview, reason, ts}`. Acceptable, or do you want a dedicated `RoomVerdictLog` Popoto model from the start?
6. **Trimmed-message floor.** When Haiku trims to a very short form (e.g. a single emoji), should we suppress instead? The verdict prompt currently allows arbitrary `revised_text` length. Want a minimum-length policy (e.g., `len(revised_text) < 5` → treat as suppress)?
7. **Rollout cadence.** Plan ships behind `READ_THE_ROOM_ENABLED=false` by default. Do you want a separate per-chat flag, or is the global env-var sufficient?
