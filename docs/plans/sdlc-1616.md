# Failure-Loop Detector Loses the Real Error (Observability Bug)

## Problem

The `failure-loop-detector` reflection (`agent/sustainability.py`) auto-files a GitHub issue whenever ≥3 sessions fail in 4 hours with the same error "fingerprint." The intent is to surface real failure loops so they can be fixed. Instead, **every auto-filed issue is useless**: all failed/abandoned sessions collapse into a single degenerate cluster whose fingerprint is `06f5940a02173ba1` (verified locally as `SHA256("unknown:")[:16]`), and every issue reports `Sample error: Unknown error`.

This has already happened in production: issue #1406 (`[failure-loop] 9x same error fingerprint: 06f5940a02173ba1`, CLOSED) is a prior instance of this exact degenerate hash — proof the detector is firing but producing non-actionable noise.

**Current behavior:**
- `_compute_fingerprint(session)` reads error detail ONLY from `extra_context` keys (`http_status`/`status_code`/`exception_type`/`error_type`/`error_message`/`failed_reason`) plus a top-level `session.failed_reason` attribute.
- **None of those are ever written by production code.** Repo-wide search finds zero production writers; `failed_reason` is not a field on the `AgentSession` model at all (only a test mock sets it).
- Result: every failed/abandoned session yields `http_component="unknown"`, `error_message=""`, and therefore `raw="unknown:"` → the single degenerate hash. `_file_github_issue()` surfaces the same emptiness as `"Unknown error"` via its `or "Unknown error"` fallback.
- Whole failure classes are guaranteed to collapse — notably **abandoned** sessions (health-check no-progress path) and **granite/local CLI** failures (`local-*` IDs).

**Desired outcome:**
The real failure reason IS already persisted. `finalize_session(session, status, reason=...)` (`models/session_lifecycle.py:217`) records a human-readable reason at every failure site via `log_lifecycle_transition()` (`models/agent_session.py:1849`), which appends a `session_events` entry of shape `{"event_type": "lifecycle", "text": "{old}→{new}: {reason}"}`. The detector simply never reads it.

After the fix: when `extra_context` error keys are empty, `_compute_fingerprint()` derives its message component from the latest failure reason in `session_events`, producing distinct clusters per distinct reason, and `_file_github_issue()` shows the real reason in "Sample error". Auto-filed loop issues become actionable.

## Data Flow

1. **Entry point**: A session fails or is abandoned (e.g. health-check no-progress path, granite CLI failure).
2. **Reason recorded**: `finalize_session(session, status, reason="health check: 3 recovery attempts, never progressed (kind=...)")` → `log_lifecycle_transition(status, reason)` → `append_event("lifecycle", "{old}→{status}: {reason}")` → persisted into `session.session_events` as `{"event_type": "lifecycle", "text": "running→abandoned: health check: ..."}`.
3. **Detector scan** (hourly): `failure_loop_detector()` collects `failed`/`abandoned` sessions in the last 4h, calls `_compute_fingerprint(s)` per session, clusters by fingerprint.
4. **Fingerprint (BROKEN today)**: reads `extra_context` error keys (empty) + `session.failed_reason` (unset) → `raw="unknown:"` → all sessions collapse to one cluster.
5. **Fingerprint (FIXED)**: when `extra_context` error keys are empty, extract the latest lifecycle reason from `session_events` → `raw="unknown:{reason[:80]}"` → distinct clusters per distinct reason.
6. **Output**: `_file_github_issue(fingerprint, sessions, ids)` reads the sample session's reason the same way → "Sample error: {real reason}" instead of "Unknown error".

## Solution

### Key Elements

- **`_latest_failure_reason(session)` helper** (new, in `agent/sustainability.py`): Iterates `session.session_events` in reverse, finds the most recent `event_type=="lifecycle"` event, and extracts the reason — the substring after the first `": "` in the event's `text` (because the text is formatted `"{old}→{new}: {reason}"`). Returns `""` when there is no lifecycle event or the event carries no reason (text has no `": "`). Defensive against `session_events` being `None`, not a list, or containing non-dict entries — mirrors the existing `summary`/`result_text` property guards.
- **`_compute_fingerprint()` fallback**: After exhausting the existing `extra_context`/`failed_reason` chain, if `error_message` is still empty, set `error_message = _latest_failure_reason(session)`. The `http_component`/`raw` construction is otherwise unchanged, so the genuinely-empty case still degrades to `"unknown:"`.
- **`_file_github_issue()` fallback**: Insert `_latest_failure_reason(sample_session)` into the `error_message` resolution chain BEFORE the `"Unknown error"` literal, so "Sample error" shows the real reason whenever one exists in `session_events`.

### Flow

Failed/abandoned session (reason in `session_events`) → hourly `failure_loop_detector()` scan → `_compute_fingerprint()` finds empty `extra_context` → falls back to `_latest_failure_reason()` → distinct fingerprint per distinct reason → `_file_github_issue()` → actionable issue with the real reason in "Sample error".

### Technical Approach

- **Single source of truth for extraction**: both `_compute_fingerprint()` and `_file_github_issue()` call the same `_latest_failure_reason()` helper, so the fingerprint and the displayed sample can never diverge.
- **Reason parsing**: the lifecycle event text is `f"{old}→{new}" + (f": {context}" if context else "")`. Split on the first `": "` and take the remainder. If there is no `": "`, there is no reason → return `""`. Note the arrow `→` (U+2192) never contains `": "`, so a simple `split(": ", 1)` is safe; guard the no-reason case explicitly.
- **Truncation parity**: `_compute_fingerprint()` already truncates `error_message[:80]`; `_file_github_issue()` truncates `[:200]`. The helper returns the full reason and each caller keeps its own existing truncation, so behavior for populated `extra_context` is byte-for-byte unchanged.
- **Backward compatibility**: when `extra_context` DOES carry error keys (the future/intended path, and the path the existing tests exercise via `error_message`), the new fallback is never reached — `error_message` is already non-empty. The fix is purely additive for the empty case.
- **Ordering in `_compute_fingerprint`**: the `http_component` is still derived from `http_status`/`exc_type` first; the lifecycle reason only feeds the message component (`error_message`), not the http component. This preserves the existing clustering semantics for HTTP/exception failures and only enriches the message half for reason-only failures.

## Rabbit Holes

- **Backfilling `extra_context` error keys at failure sites**: tempting to "fix it properly" by making production write `extra_context["error_message"]` everywhere a session fails. That is a much larger, cross-cutting change touching every failure site and is NOT this issue — the reason is already captured in `session_events`. Read it; don't re-plumb the writers.
- **Parsing structured detail out of the reason string** (regex for http codes, exception class names embedded in the reason): out of scope. The message component is a plain truncated string; clustering on the raw reason text is sufficient and matches acceptance criteria.
- **Changing the clustering threshold, the 4-hour window, or the seen-fingerprint TTL**: unrelated to observability; leave untouched.
- **Touching `log_lifecycle_transition` or the `session_events` event shape**: the detector is a read-only consumer. Do not modify the producer side.

## Risks

### Risk 1: Reason strings vary slightly between otherwise-identical failures, fragmenting clusters
**Impact:** If reasons embed a volatile token (e.g. `kind=<unique-id>`), two "same" failures could get different fingerprints and never reach the ≥3 threshold, suppressing a real loop issue.
**Mitigation:** Accept for this fix — fragmentation is strictly better than today's total collapse, and the `[:80]` truncation already clips long tails. Note it in the issue body / docs as a known follow-up if observed; do not pre-optimize normalization now (rabbit hole).

### Risk 2: A bare `MagicMock` session double silently passes through the old path
**Impact:** A test double created with plain `MagicMock()` auto-creates a truthy `failed_reason` and truthy `extra_context.get(...)`, so the new fallback would never execute and the test would assert nothing meaningful.
**Mitigation:** The `_make_reason_only_session` helper explicitly sets `failed_reason = ""` and `extra_context = {}` (or a dict with no error keys), forcing the lifecycle-reason path. Validator confirms the helper construction, not just the assertion.

## Race Conditions

No race conditions identified. `_compute_fingerprint()`, `_latest_failure_reason()`, and `_file_github_issue()` are synchronous, single-threaded reads of already-finalized session objects.
