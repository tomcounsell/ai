# YouTube Transcription: Caption-First Fallback + Enrichment Fix

## Problem

When a user shares a YouTube link, Valor responds: *"Can't get the video to load through any of my tools."*

**Current behavior:**
The sole transcription path calls `transcribe_audio_file()` which requires `OPENAI_API_KEY`. That key is not in `.env`. Every YouTube link fails. Failure context strings generated by `process_youtube_urls_in_text()` are then silently discarded by `bridge/enrichment.py` (guarded by `if successful > 0`), so the agent receives the original message with no context — it can only guess what happened.

**Desired outcome:**
1. YouTube links with captions (the majority) → Valor reads the transcript and discusses the content
2. YouTube links without captions + no Whisper key → Valor tells the user what failed and suggests pasting the transcript manually
3. Failure context strings always reach the agent regardless of success count

## Prior Art

No prior issues or merged PRs address these failure paths. The feature was shipped with a single transcription path that silently requires `OPENAI_API_KEY`.

## Data Flow

Current broken path:
1. **User** sends Telegram message with YouTube URL
2. **bridge/telegram_bridge.py** sets `youtube_urls` field on `TelegramMessage`
3. **bridge/enrichment.py** calls `process_youtube_urls_in_text(enriched_text)`
4. **tools/link_analysis/__init__.py** → `process_youtube_url()` → `download_youtube_audio_async()` → `transcribe_audio_file()` → fails silently (no `OPENAI_API_KEY`)
5. `process_youtube_url()` returns `context = "[YouTube video, transcription failed: {url}]"`, `success = False`
6. **bridge/enrichment.py** line 112: `if successful > 0:` — condition is False, so `enriched_text` is NOT updated
7. **Agent** receives original message with no context

Fixed path:
1-3. Same as above
4. **tools/link_analysis/__init__.py** → `process_youtube_url()` → tries `YouTubeTranscriptApi.get_transcript(video_id)` first → succeeds for captioned videos
5. Falls back to `download_youtube_audio_async()` + `transcribe_audio_file()` only if captions unavailable
6. Returns rich failure context if both methods fail
7. **bridge/enrichment.py**: always applies `enriched_text` from `process_youtube_urls_in_text()` (no `if successful > 0` guard)
8. **Agent** receives transcript or actionable failure context

## Architectural Impact

- **New dependencies**: `youtube-transcript-api` added to `pyproject.toml`
- **Interface changes**: `process_youtube_url()` return dict unchanged; internal logic gains a new first step
- **Coupling**: No change — enrichment.py already calls into link_analysis
- **Reversibility**: Easy — the caption path is a new `try` block; removing it restores prior behavior

## Solution

### Key Elements

- **Caption-first transcription**: `process_youtube_url()` tries `YouTubeTranscriptApi.get_transcript(video_id)` before attempting audio download. No API key required. Works for any video with auto-generated or manual captions.
- **Whisper fallback**: Existing `download_youtube_audio_async()` + `transcribe_audio_file()` path remains unchanged, attempted only when captions unavailable.
- **Enrichment suppression fix**: `bridge/enrichment.py` line 112 — remove the `if successful > 0` guard. Always assign `enriched_text = yt_enriched`. The context strings (success or failure) should always reach the agent.
- **Actionable failure context**: When both methods fail, the context string names the specific failure reason and suggests a manual workaround.
- **Dependency**: Add `youtube-transcript-api>=0.6.0` to `pyproject.toml`.

### Flow

User sends YouTube link → enrichment.py calls process_youtube_url() → try caption API → success: append transcript to context → always write enriched_text → agent discusses content

Failure path: try caption API fails → try Whisper (if key present) → all fail → append actionable failure context → always write enriched_text → agent explains what broke

### Technical Approach

In `tools/link_analysis/__init__.py`, `process_youtube_url()`:
1. After extracting `video_id` and checking `video_info`, attempt `YouTubeTranscriptApi.get_transcript(video_id)` first
2. Join transcript segments into a single string
3. If captions succeed, set `result["transcript"]`, `result["success"] = True`, build context string, return
4. If `TranscriptsDisabled` / `NoTranscriptFound`, log and continue to Whisper path
5. Whisper path unchanged
6. If Whisper also fails, return a context string like: `[YouTube video: "{title}" — transcript unavailable (captions not found; Whisper API not configured). To discuss this video, paste the transcript or a summary directly into the chat.]`

In `bridge/enrichment.py` lines 110-123:
- Remove the `if successful > 0:` guard
- Always set `enriched_text = yt_enriched`
- Keep the `logger.info` logging conditioned on `successful > 0`
- Keep the `logger.warning` loop for error details

## Failure Path Test Strategy

### Exception Handling Coverage
- `transcribe_audio_file()` wraps its body in `except Exception` → returns `None` silently. Test that when Whisper returns 401 (no key), the caption path has already succeeded and the Whisper path is never reached.
- `YouTubeTranscriptApi.get_transcript()` raises `TranscriptsDisabled`, `NoTranscriptFound`, or `VideoUnavailable` — each should be caught and logged, falling through to Whisper path.

### Empty/Invalid Input Handling
- `process_youtube_url()` with a non-YouTube URL returns early with `success=False, error="Not a valid YouTube URL"`.
- `process_youtube_urls_in_text()` with text containing no YouTube URLs returns `(text, [])` unchanged.

### Error State Rendering
- When both caption and Whisper fail, the failure context string must propagate to the agent (verified by enrichment fix).
- Test that `bridge/enrichment.py` sets `enriched_text = yt_enriched` even when `successful == 0`.

## Test Impact

No existing tests affected — confirmed zero test coverage for `process_youtube_url()`, `process_youtube_urls_in_text()`, or the transcription pipeline. No existing assertions will break.

## Rabbit Holes

- **Transcript language selection**: `youtube-transcript-api` can return multiple languages. Don't implement language preference logic — just use the default (first available, usually auto-generated English). That's a separate concern.
- **Caching transcripts**: The existing audio cache (`data/media/youtube/`) is for Whisper. Don't extend caching to caption results — the API is fast and stateless.
- **Summarization for caption results**: Transcripts from captions can be long. The existing `summarize_transcript()` call (triggered at >2000 chars) handles this already — no changes needed.
- **yt-dlp video info preflight**: The existing `get_youtube_video_info()` call before download is useful for live-stream/duration checks. Keep it in place; the caption API call should go AFTER this check but BEFORE the audio download.

## Risks

### Risk 1: youtube-transcript-api not available on all machines
**Impact:** Import error crashes the transcription step
**Mitigation:** Wrap the import in a try/except; fall through to Whisper path if import fails. Add to `pyproject.toml` so `pip install -e .` installs it.

### Risk 2: YouTube changes caption API contract
**Impact:** `YouTubeTranscriptApi` raises unexpected exceptions
**Mitigation:** Broad except around the caption attempt logs and falls through to Whisper. No user-facing crash.

## Race Conditions

No race conditions identified — `process_youtube_url()` is already `async` and called sequentially per URL. The caption API call is synchronous but runs in an executor via `loop.run_in_executor()` (same pattern as existing audio download).

## No-Gos (Out of Scope)

- Do not remove the yt-dlp + Whisper path — valid fallback when `OPENAI_API_KEY` is present
- Do not add `OPENAI_API_KEY` to `.env` as a fix
- Do not implement language preference for caption selection
- Do not cache caption results
