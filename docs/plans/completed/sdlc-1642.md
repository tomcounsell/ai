# sdlc stage-query: derive `--repo` for PR-state gh calls + distinguishable UNKNOWN merge state

## Problem

The SDLC pipeline runs from "local PM sessions" whose cwd is usually `~/src/ai`, while the
work being shepherded lives in another repo (e.g. `tomcounsell/popoto`). `sdlc-tool next-skill`
reads live PR merge state via `tools/sdlc_stage_query.py` to decide whether the terminal
merge guard **G6** fires and dispatches `/do-merge`.

**Current behavior:**
When `GH_REPO` is unset (the normal local-session case), `_fetch_pr_merge_state` runs
`gh pr view <n>` with **no `--repo` flag**, so `gh` resolves the PR number against the wrong
repo (`tomcounsell/ai`). This is a silent **cross-repo PR-number collision**: for popoto PR #419
the call resolves AI #419 — a real, MERGED PR — and returns `{"mergeStateStatus":"UNKNOWN",...}`
with exit 0. `pr_merge_state` becomes `"UNKNOWN"`, G6 sees `!= "CLEAN"` and does not fire, and the
router returns `Blocked(reason="no matching dispatch rule")`. The Blocked reason carries no
merge-state context, so a wrong-repo UNKNOWN is **indistinguishable** from a genuinely-unmergeable
PR. In the field (popoto#408 / PR popoto#419, 2026-06-12) this cost a manual diagnosis cycle;
re-running with `GH_REPO=tomcounsell/popoto` exported made G6 fire correctly.

**Desired outcome:**
1. PR-state `gh` calls resolve against the correct target repo even when `GH_REPO` is unset, using
   the same resolution ladder PR #1545 established for `find_plan_path` (`SDLC_TARGET_REPO` →
   git-toplevel → fallback).
2. When merge state genuinely cannot be resolved (`UNKNOWN`/`None`) for a known PR, the router's
   blocked output names that explicitly instead of the generic "no matching dispatch rule" — so the
   next failure is self-diagnosing.

**Root cause pattern:** SDLC tooling repeatedly assumes the process is running *inside* the target
repo (cwd or `GH_REPO`). When invoked from a local `~/src/ai` PM session shepherding another repo,

durable fix is to route every repo-dependent operation through one shared resolver.

## Data Flow

1. **Entry point**: PM session runs `sdlc-tool next-skill --issue-number N` (cwd `~/src/ai`, `GH_REPO` unset).
2. **`query_enriched` → `_compute_meta`** (`sdlc_stage_query.py:270`): resolves `pr_number` (via session attr / `_pr_number` meta / `_lookup_pr_number` → `_gh_pr_list`).
3. **`_fetch_pr_merge_state(pr_number)`** (`:128`): `gh pr view <n>` **with no `--repo`** → resolves against `tomcounsell/ai` → returns `("UNKNOWN", ...)` for a colliding PR number. `_meta["pr_merge_state"] = "UNKNOWN"`.
4. **Router `decide()`** (`agent/sdlc_router.py`): `guard_g6_*` sees `pr_merge_state != "CLEAN"` → returns None → no dispatch rule matches → `Blocked(reason="no matching dispatch rule", guard_id=None)`.
5. **Output**: PM sees a generic block with no merge-state hint → manual diagnosis.

The fix inserts a **repo resolver** between steps 2/3 and the `gh` subprocess, and enriches the
Blocked reason at step 4 when `pr_number` is set but merge state is unresolved.

## Solution

### Key Elements

> **Critical env-var distinction (do not conflate these two):**
> - **`GH_REPO`** is an `owner/name` **slug** (e.g. `tomcounsell/popoto`). The `gh` CLI respects it natively, and it is the correct value to pass to `gh --repo`. It is injected for bridge-spawned sessions (`agent/sdk_client.py:1610`).
> - **`SDLC_TARGET_REPO`** is an **absolute filesystem PATH** to the target repo root (e.g. `/Users/.../src/popoto`), injected at `agent/sdk_client.py:1611-1612` from `project_working_dir`. It is consumed elsewhere as `Path(SDLC_TARGET_REPO)/"docs"/"plans"` (`tools/_sdlc_utils.py:223-225`). It is **NOT** a slug and must **NEVER** be passed to `gh --repo`. Its only role in this resolver is as the **`cwd`** for a `gh repo view` subprocess, which then extracts the slug.

- **`_resolve_target_repo()` helper**: returns an `owner/name` **slug** (str) or None, using this ladder:
  1. **`GH_REPO`** env (rung 0) — already a slug; return it directly at zero subprocess cost.
  2. **`SDLC_TARGET_REPO`** env (a filesystem path) — used as the **`cwd`** for `gh repo view --json nameWithOwner -q .nameWithOwner`; the command's *output* is the slug. The path itself is never returned.
  3. **git-toplevel** (`_git_toplevel()`) — used as the `cwd` for the same `gh repo view` call when neither env is set.
  4. **None** — every step falls through on failure so a missing env degrades to current behavior, not a crash.
- **`_fetch_pr_merge_state` gains a `repo: str | None = None` param**: when the caller passes a resolved slug, the `gh pr view` call includes `--repo <owner/name>`. When `repo` is None, behavior is unchanged (preserves the in-repo case where cwd already is the target). The function **never calls the resolver itself**.
- **`_gh_pr_list` gains a `repo: str | None = None` param**: same treatment so PR-number lookup also targets the right repo; also never calls the resolver itself.
- **Resolve-once**: `_resolve_target_repo()` is called **exactly once**, at the top of `_compute_meta`, and the resolved slug is threaded down into both `gh` helpers as the `repo=` argument.
- **Distinguishable UNKNOWN**: the router emits a merge-state-aware Blocked reason when `pr_number` is set and `pr_merge_state` is exactly in `{None, "UNKNOWN"}` (membership test, NOT `!= "CLEAN"` — DIRTY/BLOCKED must keep their own routing). The reason string includes the resolved target repo (or a `None`/cwd marker) so the operator can tell a resolution-failure from a genuinely-unmergeable PR. This is the minimum-bar guarantee independent of whether repo derivation succeeds.

### Flow

PM runs `sdlc-tool next-skill` → `_compute_meta` resolves target repo → `gh pr view --repo <target> <n>` returns true merge state → G6 sees `CLEAN` → dispatch `/do-merge`.

Failure flow: repo unresolved AND `gh` returns UNKNOWN → router Blocked reason names "PR #N merge state UNKNOWN — could not resolve mergeability (check GH_REPO / target repo)" → operator fixes env instead of guessing.

### Technical Approach

- Reuse `_git_toplevel()` from `tools/_sdlc_utils.py`. Add `_resolve_target_repo()` — prefer placing it in `tools/_sdlc_utils.py` next to `find_plan_path` so both env-gap fixes share one resolver, and import it into `sdlc_stage_query.py`.
- **Resolver implementation** (returns a slug or None — NEVER returns a path):
  ```python
  def _resolve_target_repo() -> str | None:
      # Rung 0: GH_REPO is already an owner/name slug — return it directly.
      if repo := os.environ.get("GH_REPO"):
          return repo
      # Rung 1: SDLC_TARGET_REPO is a FILESYSTEM PATH — use it as cwd, not as --repo.
      # Rung 2: else the git working-tree root, also used as cwd.
      cwd = os.environ.get("SDLC_TARGET_REPO") or _git_toplevel()
      if not cwd:
          return None
      try:
          proc = subprocess.run(
              ["gh", "repo", "view", "--json", "nameWithOwner", "-q", ".nameWithOwner"],
              cwd=cwd, capture_output=True, text=True, timeout=10,
          )
          if proc.returncode != 0:
              logger.warning(f"_resolve_target_repo: gh repo view failed (rc={proc.returncode}) in cwd={cwd}")
              return None
          slug = (proc.stdout or "").strip()
          return slug or None
      except Exception as e:
          logger.warning(f"_resolve_target_repo: gh repo view raised in cwd={cwd}: {e}")
          return None
  ```
  Note `SDLC_TARGET_REPO` and `_git_toplevel()` supply the **`cwd`** for the subprocess; the slug comes from the command's stdout. `gh repo view` is robust to non-`origin` remote names. `GH_REPO` short-circuits before any subprocess. An empty `SDLC_TARGET_REPO=""` is falsy and falls through to `_git_toplevel()` (never passes `cwd=""`).
- **Resolve-once**: call `_resolve_target_repo()` exactly once at the top of `_compute_meta`. Thread the result into both `_fetch_pr_merge_state(pr_number, repo=resolved)` and the `_gh_pr_list` path (via `_lookup_pr_number`) as a `repo` keyword. Neither helper calls the resolver internally.
- Thread the resolved slug into both `gh` invocations as `--repo <slug>` **only when non-None** (when None, omit `--repo` → current in-repo behavior).
- **Thread the resolved slug into meta** as `_resolved_target_repo` (a private meta key) in the `_compute_meta` return dict, so the router can read it at the Blocked construction site. Add it to `_default_meta()` as `None` too.
- Delete/replace the misleading comment at `sdlc_stage_query.py:185` — it asserts a contract that local sessions break.
- Router change: in `agent/sdlc_router.py`, where the final `Blocked("no matching dispatch rule")` is constructed (`:1037`), branch on `meta.get("pr_number")` being set while `meta.get("pr_merge_state") in (None, "UNKNOWN")` — an **exact membership test, NOT `!= "CLEAN"`** (DIRTY/BLOCKED must NOT trigger the UNKNOWN-specific message; they fall through to their own routing/blocking). Emit a specific reason that **includes the resolved target repo** read from `meta.get("_resolved_target_repo")` (rendered as the slug, or a `<none — using cwd>` marker when None) so a resolution-failure is visibly distinct from a genuinely-unmergeable PR. Do NOT change G6's firing conditions (G6 must still require exactly `"CLEAN"`).

## Rabbit Holes

- **Do not** rework `agent/sdk_client.py`'s `GH_REPO`/`SDLC_TARGET_REPO` injection — that path is correct for bridge-spawned sessions; the bug is strictly the local-session path where neither env is set. Touching sdk_client widens blast radius for no gain.
- **Do not** parse git remotes by hand (`git remote get-url` + regex) — `gh repo view --json nameWithOwner` already canonicalizes owner/name and handles SSH/HTTPS/forks. Hand-parsing is the tempting-but-wrong avenue.
- **Do not** make `gh pr view` failures loud/raise — silent-None-degrade is the existing, correct contract; only the *router's report* needs to get louder, and only when a PR number is known.

## Risks

### Risk 1: `gh repo view` adds a subprocess per `next-skill` call
**Impact:** Marginal latency on the hot router path.
**Mitigation:** Resolve once per `_compute_meta` and pass the result into both `gh` calls; `SDLC_TARGET_REPO` (set for bridge sessions) short-circuits before any subprocess.

### Risk 2: In-repo (cwd already is target) case regresses if resolver guesses wrong
**Impact:** A correct in-repo run starts passing `--repo` pointing elsewhere.
**Mitigation:** Resolver derives from the *current* git-toplevel, which IS the target in the in-repo case, so `--repo` matches the implicit default — no behavior change. Test covers it.
