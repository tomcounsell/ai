# Resume Re-Verification — evidence-cited completion claims after interruption

## Problem

Eng sessions are orchestrated by a PM persona that dispatches work to a
continuable dev subagent. In a recent production incident, a PM session whose
dispatch was interrupted first told the user *"nothing actually shipped this
session — neither the confirmation email nor the episode setup happened"*, then
after a resume asserted the opposite — *"the confirm email + episode setup were
already complete"* — citing no evidence for either claim. One statement was
false and the user had no way to know which.

**Current behavior:**
Completion verification is **forward-only**. Every gate fires before a session
first claims "done," but nothing requires a session that was interrupted or
resumed to re-derive prior completion state from live evidence before
re-asserting it.

- `.claude/commands/roles/_prime-rails.md` (§Completion Criteria, lines 38-52)
  and the engineer persona's Stage Artifact Verification
  (`config/personas/engineer.md:390-401`) both gate the *first* "done" claim.
  Neither applies to re-assertions after resume.
- The persona actively biases toward asserting from memory:
  `config/personas/segments/work-patterns.md:5` — *"When resuming a prior
  session, I do not announce it… I just respond to the current message
  naturally."*
- The only evidence gate that exists (`bridge/promise_gate.py`,
  `_EVIDENCE_PATTERNS` lines 263-273) runs on the agent→user delivery path.
  Dev→PM reports and a PM's internal "already complete" reasoning pass through
  no gate.
- There is no durable per-step record to consult. `PipelineLedger`
  (`agent/pipeline_ledger.py`, #2012) records which SDLC sub-skill was
  *dispatched* (written before invocation, `tools/sdlc_dispatch.py`), not which
  internal steps (send email, create record, run workflow) *completed*. After an
  interruption the orchestrator reconstructs state solely from a
  possibly-truncated transcript — exactly how the contradiction above happened.

**Desired outcome:**
A resumed or interrupted session must verify any claim about previously-completed
work against live evidence (git state, PR state, queue/DB state, sent-mail
record) before asserting it, and must cite the artifact it checked.
Contradictory completion claims across an interruption become structurally
discouraged: a completion claim without a fresh citation is a rule violation.

## Data Flow

How a completion claim reaches the user today, and where the new rule intercepts:

1. **Entry point**: the session runner (`agent/session_runner/`) resumes a PM
   or Dev turn after an interruption (worker restart, kill, mid-dispatch break).
   The prior transcript may be truncated; goal and rails are re-injected.
2. **Prompt assembly**: `agent/sdk_client.py::_assemble_segments` composes the
   persona from `config/personas/segments/manifest.json`
   (`identity → work-patterns → tools → private-tag`). The role prime command
   (`prime-pm-role.md` / `prime-dev-role.md`) instructs the session to read
   `_prime-rails.md` before acting. **This is where the new rule lives** — both
   surfaces are loaded on every resumed turn.
3. **Reasoning**: the PM reconstructs "what already shipped." Today it does so
   from memory/transcript. **The new rule forces a live-evidence re-derivation
   here** before any completion assertion.
4. **Output**: the completion claim flows Dev→PM (verbatim transcript relay) or
   PM→user (through `bridge/promise_gate.py`). The promise gate is unchanged —
   it stays an outbound-only backstop; the fix lands one layer earlier, in the
   reasoning step, so Dev→PM internal claims are covered too.

The fix is applied at step 3 (reasoning-time rule), not step 4 (transport gate),
because the contradiction originated in internal reasoning that never crossed the
outbound gate.

## Solution

### Key Elements

- **Resume Re-Verification rule (`_prime-rails.md`)**: a new, terse section
  applying to every session-runner role (PM, Dev, Teammate). On resume, any
  claim that prior work completed must be re-derived from live evidence and cite
  the checked artifact; "I remember doing it" is not evidence.
- **work-patterns.md reconciliation**: clarify that "don't announce the resume"
  governs *narration of the resume event*, not *silent re-verification of prior
  claims*. Not announcing ≠ asserting from memory.
- **engineer.md cross-reference**: one line in the Stage Artifact Verification
  section pointing at the rails rule so the persona's forward-only table is
  explicitly extended to re-assertions after resume.
- **Recorded Decision — no step-outcome ledger** (see below): the issue's open
  architectural question, resolved with written rationale rather than a
  foregone build.
- **Tests**: a deterministic unit test (CI gate) plus an LLM-judged integration
  eval that exercises the interrupted-then-resumed scenario.

### Recorded Decision: prompt-only enforcement, no step-outcome ledger

The issue asks whether side-effectful sub-steps (send email, create external
record, kick off workflow) should get durable outcome records — e.g. extending
`PipelineLedger` — so a resumed orchestrator has something authoritative to
consult. **Decision: no. Enforce via the re-verification rule alone.** Rationale:

1. **A step-outcome ledger reproduces the exact gap it claims to close.** A
   record written *before* the side effect is forward-only (the very failure
   mode of every existing gate). A record written *after* has an interruption
   window where the side effect happened but was not yet recorded. There is no
   way to write "email sent" atomically with the SMTP transaction, so a ledger
   is never more authoritative than the side effect's own live record — it just
   moves the trust boundary one layer down and adds a second thing that can
   disagree with reality.
2. **Live evidence already exists and is authoritative.** The side effects named
   in the incident all leave queryable ground truth: a sent email has a
   Gmail/IMAP sent-mail record (`valor-email read`), a pushed commit has git
   history, an open PR has GitHub state (`gh pr view`), a kicked-off workflow /
   session has a Redis/queue record. The re-verification rule points sessions at
   *these* sources — the real ones — not a shadow copy.
3. **Cost/benefit.** Instrumenting every side-effectful call site to write
   outcome records is invasive (matches the issue's "if judged too heavy" path)
   and buys nothing over querying the authoritative source directly.

**Consequence:** the ledger is explicitly out of scope (No-Gos). No follow-up
issue is filed because this is a decision *against* the mechanism, not a
deferral. The one conditional that would reopen it: a future side-effect class
with **no** queryable live-evidence source. If such a class appears, that
specific class — not a general ledger — gets a minimal outcome record, filed as
its own issue at that time.

### Flow

Resumed PM turn → rails + persona re-injected (Re-Verification rule present) →
PM reconstructs "what shipped" → **queries live evidence** (`gh pr view`,
`valor-email read`, git log, queue state) → asserts completion **with a cited
artifact**, or finds the artifact absent and **does the work** → grounded answer
to user (unchanged promise-gate backstop on the outbound edge).

### Technical Approach

- **`_prime-rails.md`** — add a `## Re-Verification on Resume` section. Keep it
  tight (rails load every turn; token cost matters — issue's Downstream
  constraint). Target ≤ 8 lines. Anchor phrasing: on resume treat prior-work
  claims as unverified until re-derived from live evidence; name the evidence
  sources (git/PR state, queue/DB record, sent-mail log); "I remember doing it"
  is not evidence; absent artifact ⇒ not done ⇒ do it; never state contradictory
  completion claims across an interruption without a fresh citation.
- **`config/personas/segments/work-patterns.md`** — extend the existing resume
  sentence (line 5) with a caveat: not announcing the resume does **not** mean
  asserting prior work from memory; before claiming any prior step completed,
  silently re-derive it from live evidence and cite the artifact (pointer to the
  rails rule). Preserve the "Do or do not / no mid-flight promises" ethos — the
  caveat is additive, not a revert of commit 271253a4.
- **`config/personas/engineer.md`** — in Stage Artifact Verification (lines
  390-401), add one sentence: these checks also apply when *re-asserting* a
  stage complete after a resume, not only before the first advance. Cross-link
  the rails rule.
- **No code changes.** `promise_gate.py`, `pipeline_ledger.py`, and
  `sdlc_dispatch.py` are read for context only and are not modified.

## Rabbit Holes

- **Building the step-outcome ledger.** Tempting because #2012 exists to extend,
  but it reproduces the interruption gap (see Recorded Decision). Do not.
- **Inlining the promise gate into every Dev→PM handoff.** The issue's own
  "Dropped" note: the gate is outbound transport; forcing it into internal
  handoffs is invasive. The rails rule is the right layer.
- **Rewriting the persona ethos.** "Do or do not / don't announce resuming" is
  deliberate (commit 271253a4). Reconcile with an additive caveat; do not revert.
- **A full deterministic simulation of resume/interruption in CI.** The behavior
  is LLM-mediated; a real behavioral test needs an LLM judge. Do not attempt to
  fake it with a scripted transcript that asserts on hardcoded strings — that
  tests nothing about model behavior.
- **Over-long rails text.** Every added line is paid on every headless turn.
  Resist elaboration; the rule must be dense.

## Risks

### Risk 1: Rails bloat degrades every turn
**Impact:** Rails load into every PM/Dev/Teammate turn; verbose additions raise
token cost system-wide and dilute other rails.
**Mitigation:** Cap the new section at ≤ 8 lines. The deterministic test asserts
the anchor phrases are present but does not mandate verbosity; keep it dense.

### Risk 2: The rule fires too broadly and slows every turn with redundant checks
**Impact:** If read as "re-verify everything always," sessions could waste turns
re-querying evidence for work done earlier in the *same* uninterrupted session.
**Mitigation:** Scope the rule explicitly to claims about *previously-completed*
work being re-asserted after a resume/interruption — not to first-time claims
within a live session (those keep their existing forward gates).

### Risk 3: LLM eval flakiness
**Impact:** A non-deterministic behavioral eval could flake in CI and erode
trust.
**Mitigation:** Make the deterministic unit test the required CI gate; run the
LLM eval as an integration test with a clear AI-judge rubric and a negative
control. Follow existing patterns in `tests/integration/test_memory_refusal_llm.py`.
