# Plan-doc commits on main: route plans through the feature PR

## Problem

Today the `/do-plan` and `/do-plan-critique` skills commit plan documents directly to `main` with no PR review. This was observed during cuttlefish issue #350's pipeline run, but the same convention is encoded in this repo's `.claude/skills-global/do-plan/SKILL.md` (lines 332-340 and 462-464) and is exercised continuously here too — the recent main log shows `Plan: skills-audit issue filing...`, `Plan(#1272): migrate ...`, `Plan(#1288): pin last_comment_id ...` all landing as direct commits.

Three distinct harms compound:

1. **Unilateral edits to canonical documents.** Plan docs at `docs/plans/*.md` record architectural rationale across multiple critique rounds. The agent has more write authority than any human contributor — substantive plan rewrites land on `main` with zero peer review (e.g., cuttlefish `565e4c8` rewrote a plan body during round-2 critique with no review).
2. **Rebase pressure on open PR branches.** When `/do-plan-critique` revises a plan mid-build, the commit lands on `main`. If a `session/{slug}` PR branch is open, GitHub flips its mergeable state to `CONFLICTING/DIRTY`, which (a) wastes a `/do-pr-review` cycle on a dirty tree and (b) forces an in-pipeline rebase. Cuttlefish PR #354 burned ~30 minutes on exactly this race.
3. **Post-merge plan deletion destroys forensic context.** This repo currently migrates plans to `docs/plans/completed/` via `scripts/migrate_completed_plan.py` (good), but `do-merge.md:542` ALSO `rm -f`s the plan from the working tree. If migration fails (it warns but does not block), the plan is just gone. Cuttlefish has the worse variant: it deletes outright with no migration.

**Current behavior:**
- `/do-plan` Step 2.5 explicitly switches to `main` before committing the plan: `git stash; git checkout main; git stash pop; git add docs/plans/{slug}.md && git commit; git push`. This is mandated at SKILL.md:332-340 and reaffirmed at 462-464.
- `/do-plan-critique` does not commit directly, but its NEEDS REVISION verdict re-invokes `/do-plan`, which re-commits to `main`.
- `/do-build` reads the plan from `origin/main:docs/plans/{slug}.md` (do-merge.md:538), so the build chain assumes plans live on main.
- `/do-merge` Step 4 (do-merge.md:530-549) calls `migrate_completed_plan.py` AND `rm -f` the working-tree plan. On migration failure, it logs a warning and continues — the plan is removed regardless of migration outcome.

**Desired outcome:**
- Plans live on the feature branch (`session/{slug}`) from the start. They ship as part of the same PR that ships the feature.
- Plan revisions during critique amend the plan-on-branch. They do not touch `main` and do not invalidate the open PR's mergeable state.
- Post-merge, the migration to `docs/plans/completed/{slug}.md` happens via `migrate_completed_plan.py`, but ONLY if migration succeeds; deletion-without-migration is removed.
- A guard hook prevents any future agent commit named `Plan: ...` or `Plan(#N): ...` from landing on `main`.

## Data Flow

Trace of a plan document's lifecycle today vs. proposed:

**Today:**
1. **Entry:** `/do-plan` is invoked with `{slug}` and an issue number.
2. **Plan write:** Plan template is filled in at `docs/plans/{slug}.md` on the current branch (whichever it is — could be main, could be a session branch, could be detached).
3. **Branch switch:** `git stash; git checkout main; git stash pop` — forced switch to `main` regardless of current branch.
4. **Commit + push:** `git add docs/plans/{slug}.md && git commit -m "Plan: ..."; git push`.
5. **Tracking issue update:** Issue body is prepended with a `**Plan:**` link pointing to `main` blob.
6. **Critique loop:** `/do-plan-critique` runs. NEEDS REVISION → re-invoke `/do-plan` Phase 4 → step 4 re-commits the revised plan to `main`.
7. **Build:** `/do-build` resolves `PLAN_PATH` and creates `session/{slug}` worktree from the current main (which now includes the plan). Build edits land on the session branch only.
8. **PR opens:** PR #N references plan via the issue link. Plan file is on main.
9. **Race window:** If `/do-plan-critique` runs again during build (say, a code review insight prompts a plan revision), the new plan commit lands on main. Open PR's mergeable state flips to CONFLICTING.
10. **Merge:** `/do-merge` migrates plan to `docs/plans/completed/{slug}.md` AND `rm -f`s `docs/plans/{slug}.md` regardless of migration success.

**Proposed:**
1. **Entry:** `/do-plan` invoked with `{slug}` and issue number.
2. **Branch ensure:** Switch to or create `session/{slug}` branch off current main. The session branch is the home of the plan doc from the first commit.
3. **Plan write:** Plan template filled at `docs/plans/{slug}.md` on `session/{slug}`.
4. **Commit + push:** `git add docs/plans/{slug}.md && git commit -m "plan(#N): {title}"; git push -u origin session/{slug}`.
5. **Tracking issue update:** Issue body links to plan blob on `session/{slug}` branch (not main). When the build PR merges via squash, the plan is preserved in the squash commit and migrated to `docs/plans/completed/`.
6. **Critique loop:** `/do-plan-critique` NEEDS REVISION → re-invoke `/do-plan` → revised plan amends or appends to the session branch. No main commit. Open PR (if exists) is unaffected.
7. **Build:** `/do-build` already runs in `.worktrees/{slug}/` checked out to `session/{slug}` — the plan is already there. (Today's build reads from `origin/main:docs/plans/{slug}.md` because the plan was on main; that read switches to `origin/session/{slug}:docs/plans/{slug}.md` or just the local working tree of the worktree.)
8. **PR opens:** PR includes the plan doc as one of its files.
9. **No race window:** Plan revisions during build land on the session branch — same place build commits go. No CONFLICTING flip.
10. **Merge:** `/do-merge` calls `migrate_completed_plan.py`. On success, the plan moves to `docs/plans/completed/{slug}.md` as part of the post-merge cleanup commit on main. On failure, the plan stays at `docs/plans/{slug}.md` as a visible warning — no silent loss.

## Solution

### Key Elements

- **`/do-plan` rewrite (Phase 2.5):** Drop `git checkout main`. Replace with "ensure on `session/{slug}`" — create the branch if missing, switch to it otherwise. Commit and push the plan there.
- **`/do-plan` rewrite (Branch Workflow section):** Replace the "MANDATORY: Plans are committed on `main`" mandate with the inverted rule: plans live on the session branch.
- **`/do-merge` cleanup hardening:** Remove the unconditional `rm -f` at do-merge.md:542. Migration script's exit code becomes the gate — if migration succeeds, the plan was already moved to `completed/`. If it fails, the plan stays in `docs/plans/` so the failure is visible.
- **Pre-push hook (or pre-commit):** Reject any commit on `main` whose message matches `^Plan(\(#\d+\))?:` AND that touches only `docs/plans/*.md`. Hook lives at `.claude/hooks/validators/no_plan_commits_on_main.py`.
- **Migration of in-flight plans:** Plans currently in `docs/plans/` that have an open `session/{slug}` branch get cherry-picked onto that branch (one commit per plan); plans without an open session branch stay on main as historical artifacts (they were already shipped or are legacy).

### Flow

User says "make a plan for issue 1304" → `/do-plan sdlc-1304` is dispatched → plan template is written into `.worktrees/sdlc-1304/docs/plans/sdlc-1304.md` on the `session/sdlc-1304` branch → committed and pushed to `origin session/sdlc-1304` → tracking issue gets a link to the session-branch blob → critique runs in war room (no commits) → if NEEDS REVISION, plan amends on session branch → eventually `/do-build` runs (already in worktree, plan is right there) → PR opens with plan + code → review → merge → migration moves plan to `docs/plans/completed/sdlc-1304.md` in a post-merge commit on main.

### Technical Approach

- **Skill edit (do-plan):** Lines 332-340 replace the `git stash; git checkout main; git stash pop; git add ...; git commit; git push` block with a `session/{slug}` branch-ensure block. Lines 462-464 ("Branch Workflow") rewrite to state plans live on the session branch.
- **Skill edit (do-merge):** Line 542 `rm -f "$PLAN_PATH"` is removed. The migration script already moves the file; the second deletion is what makes failures silent. Replace with a verification: assert `migrate_completed_plan.py` exit-code == 0; if not, surface a loud error and leave the plan in place.
- **Skill edit (do-plan-critique):** No git commits there today, but the ROUND_VERDICT.md or wherever revisions are described should reference "plan on session branch" not "plan on main."
- **Hook:** `.claude/hooks/validators/no_plan_commits_on_main.py` runs on `pre-push` (or `pre-commit` if pre-push isn't wired). Reads `git log origin/main..HEAD --oneline` for the push, rejects if any commit on `main` matches `^(Plan|plan)(\(#\d+\))?:` regex AND `git show --name-only` shows only `docs/plans/*.md` paths.
- **Migration script:** A one-shot `scripts/migrate_inflight_plans_to_branches.py` (run once by hand, not on every plan) inspects `docs/plans/*.md` against open `session/{slug}` branches; cherry-picks each plan onto its branch; leaves plans without a matching branch alone.
- **Tracking-issue link generator:** The "Plan link" in the issue body changes from `https://github.com/{owner}/{repo}/blob/main/docs/plans/{slug}.md` to `https://github.com/{owner}/{repo}/blob/session/{slug}/docs/plans/{slug}.md`. Auto-rewritten when the implementation PR merges, since GitHub auto-redirects deleted-branch blobs to a 404 — but the squash-merge creates a `docs/plans/completed/{slug}.md` blob on main, so the issue body's link should be re-pointed at merge time. Update `do-merge` Step 4 to rewrite the issue's plan-link line.

## Risks

### Risk 1: Build skill assumes plan on main
**Impact:** `/do-build` reads from `origin/main:docs/plans/{slug}.md` in some paths. If the plan is only on `session/{slug}`, the read fails and build aborts.
**Mitigation:** `/do-build` already runs inside `.worktrees/{slug}/` checked out to `session/{slug}` — the plan is in the working tree. Audit all `origin/main:` references to plans in skill files (`grep -rn "origin/main:.*docs/plans" .claude/`) and replace with working-tree reads. The do-merge.md:538 read at merge time still needs the plan from somewhere — but at merge time the squash-merge has already landed it on main, so reading from main there is fine.

### Risk 2: Hook false positives on legitimate plan-related commits
**Impact:** A human PM commits a plan touch-up directly on main as a deliberate exception, hook rejects.
**Mitigation:** Hook scope is narrow: matches commits whose message starts with `Plan:` or `Plan(#N):` AND touch ONLY `docs/plans/*.md`. Mixed commits (plan + other files) are exempt. A human override is `git push --no-verify`, which is the correct escape valve.

### Risk 3: Migration of in-flight plans is bumpy
**Impact:** Currently 30+ plans live on main with active session branches. Cherry-picking each onto its branch is tedious.
**Mitigation:** Don't migrate. The convention applies to NEW plans starting from this PR's merge. In-flight plans stay on main as historical artifacts — no harm. The hook only rejects commits with `Plan: ...` message format from this point forward; existing plan files on main are not touched.

### Risk 4: Squash-merge swallows the plan body in commit history
**Impact:** When the build PR squash-merges, the plan commit's body is folded into the squash commit message. Future archaeology via `git log -- docs/plans/` would show one squash commit, not the multi-round critique evolution.
**Mitigation:** The `docs/plans/completed/{slug}.md` file IS the forensic artifact — the post-merge migration commits it on main. Critique rounds amend the same file across multiple commits on the session branch; the squash-merge collapses those into one commit, but the file's content reflects the final state, which is what matters for archaeology. This is acceptable because the plan was always meant to be a living document, not a commit-history-driven log.

## Race Conditions

### Race 1: Concurrent plan revisions during build
**Location:** `/do-plan-critique` re-invocation during `/do-build` execution
**Trigger:** Build is mid-flight on `session/{slug}`, reviewer triggers a critique that produces NEEDS REVISION, `/do-plan` is re-dispatched.
**Data prerequisite:** The plan file at `docs/plans/{slug}.md` on the session branch must be writable.
**State prerequisite:** Build's worktree at `.worktrees/{slug}/` is on `session/{slug}` and may have uncommitted changes.
**Mitigation:** This plan does NOT solve the build/critique race — that's Issue 3. What this plan DOES do is move the race off main, so it no longer flips PR mergeable state. Concurrent edits to the plan file on the session branch are still possible; coordination is Issue 3's job. Today the issue is worse (race spans main + session); tomorrow it's contained to one branch.

### Race 2: Hook race during rebases
**Location:** `.claude/hooks/validators/no_plan_commits_on_main.py` (pre-push)
**Trigger:** Developer rebases a long-lived branch onto main, the rebase replays a plan commit that originally landed on a session branch, hook fires.
**Data prerequisite:** The plan commit's parent during replay is on main.
**Mitigation:** Hook checks the commit's REF (where it's being pushed to), not its parent. Pre-push only fires if the destination ref is `refs/heads/main`. Rebases that target session branches don't trigger.
