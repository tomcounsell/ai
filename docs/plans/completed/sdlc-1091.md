# Gate remote-update.sh worker kickstart on code-change detection

## Problem

`scripts/remote-update.sh` restarts the worker every 30 minutes whether or not anything changed. Long-running SDLC pipelines (build + test + review + docs + merge) get SIGTERM'd mid-flight every 30 min, and on skills-only machines local dev sessions (chat_id prefix `local-`) are abandoned by startup recovery on each restart — their context is irrecoverable.

**Current behavior:**

`scripts/remote-update.sh` (lines 140-157) runs
```bash
launchctl kickstart -k "gui/$(id -u)/$WORKER_LABEL"
```
whenever the worker plist is installed and loaded — with zero regard for whether `git pull` actually pulled new code or whether the pulled commits touched files the worker loads. Since `com.valor.update.plist` has `StartInterval=1800`, this fires every 30 min.

Confirmed cadence from `/Users/tomcounsell/src/ai/logs/worker.log` on 2026-04-21:
```
04:11:22 UTC worker INFO Starting standalone worker...
04:42:04 UTC worker INFO Starting standalone worker...   (+30m42s)
05:12:45 UTC worker INFO Starting standalone worker...   (+30m41s)
07:02:22 UTC worker INFO Starting standalone worker...   (+109m37s after pause)
08:01:58 UTC worker INFO Starting standalone worker...   (+59m36s)
```

**Desired outcome:**

`remote-update.sh` only kickstarts the worker when the post-pull HEAD differs from the pre-pull HEAD AND the diff between them touches worker-loaded code (`worker/`, `agent/`, `mcp_servers/`, `models/`, `tools/`, `bridge/`, `reflections/`, `scripts/`, `config/`, `.claude/`, `pyproject.toml`, `uv.lock`, or `com.valor.worker.plist`). On a quiet cycle with no new commits or only doc/test changes, the kickstart is skipped entirely. Long-running pipelines survive.

## Solution

### Key Elements

- **Pre-pull SHA capture**: record `BEFORE_SHA` before `git pull --ff-only`. Zero-cost, always safe.
- **Post-orchestrator SHA capture**: record `AFTER_SHA` AFTER `run.py --cron` returns (not immediately after `git pull`) — this captures commits made by `auto_bump_deps` during the orchestrator run. See Risk 5. If the pull failed (WARN path already exists), set `AFTER_SHA="$BEFORE_SHA"` so the gate evaluates false.
- **Worker-relevance path matcher** — SIGPIPE-safe: capture `git diff --name-only` output into a bash variable first, then feed to `grep -qE` via here-string (`<<<`). NO live-process pipe between `git diff` and `grep` (a pipeline would SIGPIPE under `set -euo pipefail` when `grep -q` short-circuits — see the BLOCKER discussion in Technical Approach). Include all paths the worker loads at runtime (full glob in Technical Approach).
- **Idempotent plist re-render**: keep the `sed` substitution that writes `$WORKER_DST` — it produces byte-identical content if `__PROJECT_DIR__` and `__HOME_DIR__` haven't changed, so it's free.
- **Kickstart skip log**: echo "[update] No worker-relevant changes detected — skipping restart" so the log is self-explanatory. (Matches the wording already used in the PR-under-review.)

### Flow

```
remote-update.sh fires (every 30 min via launchd)
    -> BEFORE_SHA=$(git rev-parse HEAD)
    -> git pull --ff-only
    -> run.py --cron --no-pull   (may auto-bump pyproject.toml + commit + push)
    -> AFTER_SHA=$(git rev-parse HEAD)      (AFTER orchestrator — catches auto-bump)
    -> render worker plist (idempotent; always)
    -> if NOT launchctl-loaded: bootstrap          (unchanged — first-run install)
       else:
         CHANGED = worker_code_changed   (SIGPIPE-safe; see function definition)
         if CHANGED: launchctl kickstart -k          (as today — code is stale)
         else: echo "No worker-relevant changes detected — skipping restart"
```

### Technical Approach

- Capture `BEFORE_SHA` as early as safe — right before the `git pull --ff-only` line. Use `git -C "$PROJECT_DIR" rev-parse HEAD` so the command is independent of cwd. Suppress stderr and default to empty string if `rev-parse` fails (e.g. freshly-cloned repo with no HEAD — `git pull` would also fail so the kickstart block wouldn't run either).
- `AFTER_SHA` is captured **after `run.py --cron` returns**, not immediately after `git pull`. The cron orchestrator runs `auto_bump_deps` which can itself commit to `pyproject.toml` and push during the same invocation (see `scripts/update/run.py:472-520`). Capturing `AFTER_SHA` only after `git pull` would miss those dep-bump commits and leave the worker running stale code. If the pull failed (the existing `else` branch prints a warning and continues with current code), set `AFTER_SHA="$BEFORE_SHA"` — nothing landed, skip the gate. Safer default: if we can't be sure, don't restart.

**BLOCKER FIX — SIGPIPE hazard under `set -euo pipefail`:**

The script runs with `set -euo pipefail` (line 8). The naive pattern `git diff "$BEFORE_SHA" "$AFTER_SHA" -- worker/ agent/ ... | grep -q .` is UNSAFE: when `grep -q` finds the first match it exits immediately with 0. The upstream `git diff` then receives `SIGPIPE` on its next write and dies with exit code 141 (128 + 13). Under `pipefail`, the pipeline's exit code becomes 141 (the rightmost non-zero), which inside an `if` context makes the `if` branch evaluate to **false** — meaning we'd falsely conclude "no worker-relevant changes" whenever the diff is larger than a single pipe buffer. The worker would then NEVER restart on real code changes. This is the exact opposite of the intended behavior and is production-blocking.

**Safe pattern — no pipe, no pipefail hazard:**

```bash
worker_code_changed() {
    # Unknown baseline (empty BEFORE_SHA) → fail-safe restart.
    [ -z "$BEFORE_SHA" ] && return 0
    # No new commits → no restart. Short-circuit avoids running diff at all.
    [ "$BEFORE_SHA" = "$AFTER_SHA" ] && return 1

    # Capture the diff output into a variable first — no pipeline, no SIGPIPE.
    # Disable pipefail briefly in case git itself writes to a closed fd somewhere
    # downstream, then restore it.
    local changed_paths
    local rc=0
    changed_paths=$(git -C "$PROJECT_DIR" diff --name-only "$BEFORE_SHA" "$AFTER_SHA" 2>/dev/null) || rc=$?

    # git diff failed (corrupted repo, missing object, force-push gap) → fail-safe restart.
    [ $rc -ne 0 ] && return 0

    # Now grep against a variable. No subprocess pipe, no SIGPIPE. `grep -qE` returns
    # 0 on first match or 1 on none — both are defined, pipefail-safe outcomes.
    # Using a heredoc-fed grep via <<< is pure bash string redirection; no pipe
    # between two processes, so SIGPIPE cannot occur.
    grep -qE '^(worker/|agent/|mcp_servers/|models/|tools/|bridge/|reflections/|scripts/|config/|\.claude/|com\.valor\.worker\.plist$|pyproject\.toml$|uv\.lock$)' <<< "$changed_paths"
}
```

Three critical properties of this implementation:

1. **No pipe between two live processes.** The diff output is stored in a variable (`changed_paths=$(...)`), then `grep` reads it via `<<<` (here-string, which bash implements as a temp file or pipe populated before grep starts — grep cannot SIGPIPE its own input because there's no upstream writer left).
2. **Explicit rc check on `git diff`.** We don't rely on `pipefail` to surface git-diff failures. Instead we capture `$?` directly with `|| rc=$?` and branch on it — fail-safe (`return 0` → restart) on any nonzero.
3. **`return 0`/`return 1` semantics match bash `if` convention.** `return 0` = success = "yes, code changed" — the `if worker_code_changed; then kickstart; fi` wrapper at the call site reads naturally.

The variable-buffer approach is safe for diff sizes this script will realistically see (30 min of commits; a worst-case 100-commit bulk-merge returns maybe 500 paths = a few KB of text — bash variables trivially hold megabytes).

## Failure Path Test Strategy

### Exception Handling Coverage

- [ ] No Python `except Exception: pass` blocks in this scope — the change is pure bash.
- [ ] Bash errors: `set -euo pipefail` is already in effect at the top of `remote-update.sh`. Unhandled `git` failures exit nonzero and the script aborts — preserving existing semantics. New `git rev-parse` / `git diff` calls use `|| echo ""` patterns (see tasks below) to gracefully degrade rather than abort the script on a transient git error.

### Empty/Invalid Input Handling

- [ ] If `BEFORE_SHA` is empty (new repo with no HEAD): `worker_code_changed` returns true → unconditional restart → preserves legacy behavior for unusual cases.
- [ ] If `AFTER_SHA` is empty (pull failed): set `AFTER_SHA="$BEFORE_SHA"` → gate evaluates false → skip restart. This is intentional and safer than restarting after a failed pull.
- [ ] If `git diff` itself fails (shouldn't happen but is possible on corrupted repo): return true from `worker_code_changed`. Fail-safe direction.

### Error State Rendering

- [ ] Each branch logs what it did: `[update] Worker plist loaded; restarting (code changed)` OR `[update] Worker plist loaded; no worker-relevant code changes — skipping kickstart` OR `[update] Worker plist missing — installing`. The existing logs (`ERROR: Failed to bootstrap ...`) are preserved.
- [ ] No silent swallowing — if `git diff` returns nonzero, the fail-safe log `[update] git diff failed; restarting worker conservatively` fires.

## Risks

### Risk 1: The path glob misses a file class that actually affects the worker

**Impact:** Worker keeps running stale code despite a pull that updated (say) `.claude/hooks/` (if hooks run inside the worker). Next `/update` manual pass or actual code change would fix it, but in the interim the worker runs stale.

**Mitigation:** Glob is intentionally broad — `worker/`, `agent/`, `mcp_servers/`, `models/`, `tools/`, `bridge/`, `reflections/`, `scripts/`, `config/`, `pyproject.toml`, `uv.lock`, `com.valor.worker.plist`. The pattern `.claude/hooks/` — the worker DOES invoke these via the CLI harness. Add `.claude/` to the glob list (no false-positive cost — if `.claude/` changed and we DON'T restart, the worker uses stale hooks). Better overshoot than undershoot.

### Risk 2: A malicious or buggy commit lands and the fix delays its rollout

**Impact:** If the fix keeps the worker running on a known-bad commit when a hotfix lands, the hotfix takes effect only on the next 30-min cron if the hotfix happens to touch a watched path. If the hotfix only touches (say) `docs/`, the worker never picks it up.

**Mitigation:** Hotfixes that matter always touch code in the worker-relevant glob, by definition — if they don't, they don't affect the worker. Documentation-only or test-only commits legitimately shouldn't restart the worker. For true emergencies, Tom can run `./scripts/valor-service.sh worker-restart` or `/update` manually; both paths are unchanged.

### Risk 3: `git diff` performance on a deep rewind (rebase from 100 commits behind)

**Impact:** Theoretical; `git diff --name-only` between two SHAs is an O(files) operation, not O(commits). Even a 100-commit pull completes in ms.

**Mitigation:** None needed; diff is fast. Not a real risk.

### Risk 4: Interaction with the existing `install_worker()` path in run.py

**Impact:** `run.py --cron` is called BEFORE the bash worker block. If `run.py` ever starts restarting the worker in cron mode (currently it doesn't — `do_service_restart=False` for cron config), we'd have two restart paths.

**Mitigation:** Verified today (`scripts/update/run.py:77-89`) that `UpdateConfig.cron()` sets `do_service_restart=False`. Plan does NOT change this. A contract test (`test_cron_config_does_not_restart_services`) could be added later but is out of scope for this plan — the existing `if config.do_service_restart` guard at `run.py:734` is sufficient.

### Risk 5: Auto-bump commits land after SHA capture and are missed by the gate

**Impact:** `run.py --cron` runs `auto_bump_deps` (see `scripts/update/run.py:472-520`). When PyPI has a newer pin for `anthropic`, `claude-agent-sdk`, etc., the orchestrator:
1. Rewrites `pyproject.toml` in place,
2. Runs smoke tests,
3. Commits with `git commit -m "Bump deps: anthropic 0.62.0->0.84.0"`,
4. Pushes.

This happens AFTER `git pull --ff-only` completes. If we capture `AFTER_SHA` right after `git pull` (the PR-under-review does this), the auto-bump commit lands in local HEAD but not in `AFTER_SHA` — so the gate says "no worker-relevant changes" even though `pyproject.toml` was just rewritten and the worker would run against the old pin until the NEXT cron cycle catches a different change. The practical effect: every dep-bump cycle takes 30-60 min to take effect instead of being picked up on the current run.

**Mitigation:** Capture `AFTER_SHA` AFTER `run.py --cron` returns, not after `git pull`. This is codified in the Technical Approach above and in Step 1 of the task list. A single-line placement decision, zero added complexity.

**Acknowledged residual risk:** if the auto-bump commit is pushed but NOT yet landed when the gate reads HEAD (race window: bash reads `git rev-parse HEAD` between `git commit` and the next `git fetch`), the local HEAD already reflects the commit — `rev-parse` reads the local `.git/HEAD`, not the remote. This is not actually a race because auto-bump's own `git commit` updates local HEAD synchronously before the Python function returns. Verified by reading `scripts/update/deps.py::auto_bump_deps` flow.

### Risk 6: Test precedent — the existing `TestRemoteUpdateScript` tests skip without `.venv`

**Impact:** The current tests (`test_already_up_to_date`, `test_no_restart_flag_when_up_to_date`) invoke `bash scripts/remote-update.sh` end-to-end and skip entirely when `.venv` is missing (e.g., CI without the venv bootstrapped, or a fresh worktree). ## Race Conditions

### Race 1: Simultaneous manual `/update` and cron-triggered `remote-update.sh`

**Location:** `scripts/remote-update.sh:50-68` (lockfile acquisition).
**Trigger:** Tom runs `/update` on the same machine the cron is about to fire on. Only one instance should run at a time.
**Data prerequisite:** `data/update.lock` must be respected by both callers.
**State prerequisite:** Lockfile is acquired atomically via `mkdir` (POSIX-atomic) — already handled.
**Mitigation:** No change — existing lockfile already prevents concurrent execution. The second caller exits with "Another update is already running. Skipping."

### Race 2: Pull succeeds but commit is force-pushed between BEFORE_SHA capture and AFTER_SHA capture

**Location:** `scripts/remote-update.sh` new logic.
**Trigger:** `BEFORE_SHA` captured at T0. Force push upstream at T1 < T2. `git pull --ff-only` at T2 FAILS (non-ff). `AFTER_SHA` read — equals `BEFORE_SHA` because pull didn't change anything.
**Data prerequisite:** Neither SHA should be stale or out of sync.
**State prerequisite:** If pull failed, we treat it as "nothing landed" (AFTER_SHA = BEFORE_SHA via fallback logic).
**Mitigation:** Explicit: when `git pull --ff-only` fails, the existing code-path already logs `WARN: git pull failed or had conflicts — continuing with current code`. The new logic sets `AFTER_SHA="$BEFORE_SHA"` in this case so `worker_code_changed` returns false — we don't restart on a failed pull.

  ```bash
  # Returns 0 ("true" / exit-success in bash convention, interpreted by `if` as true)
  # when the worker should be restarted. Returns 1 ("false" / exit-failure, interpreted
  # as false by `if`) when it is safe to skip the restart.
  worker_code_changed() {
      # Unknown baseline → fail-safe: restart.
      if [ -z "$BEFORE_SHA" ]; then
          return 0
      fi
      # No new commits → skip restart.
      if [ "$BEFORE_SHA" = "$AFTER_SHA" ]; then
          return 1
      fi
      # Capture diff output into a variable (no pipe, no SIGPIPE hazard under `pipefail`).
      local changed_paths
      local rc=0
      changed_paths=$(git -C "$PROJECT_DIR" diff --name-only "$BEFORE_SHA" "$AFTER_SHA" 2>/dev/null) || rc=$?
      # git diff failed → fail-safe: restart.
      if [ $rc -ne 0 ]; then
          return 0
      fi
      # Here-string feeds grep from a pre-populated buffer — no live upstream to SIGPIPE.
      grep -qE '^(worker/|agent/|mcp_servers/|models/|tools/|bridge/|reflections/|scripts/|config/|\.claude/|com\.valor\.worker\.plist$|pyproject\.toml$|uv\.lock$)' <<< "$changed_paths"
  }
  ```
