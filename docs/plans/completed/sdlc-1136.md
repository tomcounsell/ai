# TTS / `/do-debrief` — Reopened Acceptance Verification

## Problem

Issue #1136 was closed by PR #1183 which shipped the TTS module, `valor-tts` CLI,
`/do-debrief` skill, and bridge voice-note relay. The issue was **reopened**
because two acceptance criteria were checked off without ever being executed
end-to-end:

1. **Local Kokoro path** never had a proof artifact. The original `voices-v1.0.bin`
   download URL returned HTTP 404 on fresh installs, so the "Kokoro local"
   acceptance line was a green checkbox but a dead path. Commit `ab7b26f5` already
   fixed the URLs to the GitHub release tag `model-files-v1.0`. Local cache on
   this machine now has both files, but the local path was never exercised
   end-to-end with a checked-in proof artifact.
2. **End-to-end Telegram voice-message delivery** through the bridge was never run
   on a bridge-enabled machine — the merge machine had no bridge. The `/do-debrief`
   composite was checked off based on the file existing at ≤150 lines, not based
   on a voice bubble actually arriving in a chat.

A third concern raised in the reopen comment is whether the cloud fallback
silently no-ops when `OPENAI_API_KEY` is missing. Reading `tools/tts/__init__.py:286`
and `tools/tts/cli.py:57`, the missing-key path returns `{"error": "OPENAI_API_KEY
environment variable not set"}`, the CLI prints `Error: ...` to stderr and exits 1.
That branch is not silent. The concern is closed by reading the code; this plan
adds a regression test pinning the behavior so a future refactor can't regress it.

**Current behavior:** Issue #1136 is open. Two acceptance lines lack proof
artifacts. The original plan `docs/plans/tts-debrief.md` was deleted post-merge
(commit `cb4eb504`).

**Desired outcome:** Both unverified acceptance lines have **executable** proof
artifacts checked into the issue (file path + ffprobe output for local Kokoro;
Telegram message link for end-to-end delivery). The OPENAI_API_KEY-missing path
is pinned by a unit test. Issue #1136 closes.

### Key Elements

- **OPENAI_API_KEY-missing regression test**: pin the existing behavior at `tools/tts/__init__.py:286` and `tools/tts/cli.py:57` with a unit test that monkeypatches `OPENAI_API_KEY` away and asserts the error dict + exit-1 + stderr message.

### Flow

This is a verification flow, not a feature flow:

`Issue #1136 reopened` → Run local Kokoro CLI on this machine → Capture stdout + ffprobe → Attach to issue → Run E2E debrief on bridge machine → Capture Telegram link → Attach to issue → Add regression test → Open PR → Merge → Issue closes.

### Technical Approach

- **No code changes to `tools/tts/__init__.py` or `bridge/telegram_relay.py`.** The behavior under verification is the behavior already shipped.
- **One new unit test** at `tests/unit/test_tts_no_openai_key.py` (or appended to an existing TTS test file under `tools/tts/tests/`). It uses `monkeypatch.delenv("OPENAI_API_KEY", raising=False)` and `monkeypatch.setattr` to force-disable Kokoro, then asserts:
  - `synthesize("hi", "/tmp/foo.ogg", force_cloud=True)["error"] == "OPENAI_API_KEY environment variable not set"`
  - `cli.main(["--text", "hi", "--output", "/tmp/foo.ogg", "--force-cloud"])` returns `1` and prints `Error: OPENAI_API_KEY ...` to stderr (capture via `capsys`).
- **Proof artifacts** live as comments on issue #1136 (text outputs, ffprobe summary, Telegram message link). They are not committed to the repo — they're signals for closing the ticket.
- **Issue closure**: the PR for the regression test uses `Closes #1136` so the ticket auto-closes on merge after the artifacts are also posted.

## Risks

### Risk 1: Kokoro path still fails on this machine despite cached files
**Impact:** Local-Kokoro acceptance can't be artifact-verified from this machine.
**Mitigation:** First-step verification is `python -c "from tools.tts import synthesize; print(synthesize('hello', '/tmp/p.ogg'))"` — if it returns `backend=kokoro`, the path works. If it returns `backend=cloud`, read the `_kokoro_available_cache["reason"]` and fix that as a one-line subtask before continuing. (Likely candidates: stale `kokoro_onnx` package version, missing `soundfile`.)

### Risk 2: No bridge-enabled machine is available to run the E2E debrief verification
**Impact:** The end-to-end Telegram delivery acceptance can't be artifact-verified during this plan's execution window.
**Mitigation:** The regression test for the missing-key path can ship without it. Mark the E2E acceptance line as "verified by next bridge-enabled session" in the issue comment; reopen briefly if the next person on the bridge machine confirms the voice bubble arrives. Do NOT close the issue without that confirmation.

### Risk 3: Kokoro cache hits an ABI regression on this machine's CPU and the dynamic probe fails silently for 60s
**Impact:** First `valor-tts` invocation falls back to cloud; subsequent invocation within 60s also falls back; appears to be "Kokoro broken" when it's just a cold-cache mis-probe.
**Mitigation:** Run the verification command twice with a 90-second pause between. The second call clears the 60s availability cache. If both go to cloud, the dynamic probe is genuinely failing — read the WARN log line to diagnose.
