# SDLC Router: Verdict Normalization + Stale-Verdict Supersession (combined #1638/#1640/#1641)

## Problem

Three field-observed router defects (all surfaced during real `/do-sdlc` and popoto pipeline
runs in June 2026) each forced a manual operator override to unstick the pipeline. They look
like three disjoint bugs but share **one root pathology**: the SDLC router treats verdict
strings as raw text and trusts stage-status strings as proxies for facts they don't actually
assert. Specifically:

1. **Verdicts are matched as raw substrings** rather than against a canonical, normalized
   token — so a cosmetic spelling variant (`CHANGES_REQUESTED` vs `CHANGES REQUESTED`) silently
   desyncs dispatch (#1638).
2. **Stage statuses are trusted as evidence of artifacts that don't yet exist.** `PLAN="ready"`
   is read as "a plan exists" when it only means "the ISSUE stage completed" (#1640).
3. **A stale verdict is never superseded when the downstream stage that should clear it
   completes.** A pre-patch `CHANGES REQUESTED` keeps out-ranking a completed-but-unreviewed
   patch, looping `/do-patch` forever (#1641).

The same family already produced a fourth incident: the router **deadlocked on a stale
`NEEDS REVISION` verdict** by ignoring `revision_applied` (captured in agent memory under
"SDLC router NEEDS REVISION deadlock"). That history is the reason this plan fixes the
*pattern*, not three point-symptoms.

**Current behavior:**
- #1638: REVIEW skill records `CHANGES_REQUESTED`; `"CHANGES REQUESTED" in review_verdict_upper`
  is `False` (underscore preserved by `.upper()`); router returns "no matching dispatch rule"
  after REVIEW. PATCH never dispatches.
- #1640: Fresh issue, ISSUE completes → `complete_stage` sets `PLAN="ready"`
  ([`agent/pipeline_state.py:542-543`](agent/pipeline_state.py#L542-L543)) *before any plan is
  written*. `_rule_plan_not_critiqued` ([`agent/sdlc_router.py:622-630`](agent/sdlc_router.py#L622-L630))
  fires on `PLAN in (completed, "ready")` and dispatches `/do-plan-critique` with nothing to
  critique, instead of `/do-plan`.
- #1641: After a patch completes, both row 8 `_rule_review_has_findings` (registered at line 877)
  and row 8b `_rule_patch_applied_after_review` (registered at line 883) match simultaneously — the
  stale pre-patch `CHANGES REQUESTED` is still recorded, `PATCH=completed`. First-match ordering
  means row 8 wins, re-dispatching `/do-patch`: a no-op loop. Mandatory re-review never runs. The
  fix makes row 8 recognize that the verdict is **stale** — its `recorded_at` predates the
  `/do-patch` dispatch that was supposed to clear it — and step aside so row 8b runs the re-review.

**Desired outcome:**
- A single canonical verdict form is persisted at the `verdict record` write boundary, so the
  router can never receive a variant it doesn't recognize. Read-side normalization in the
  router is belt-and-suspenders for legacy records.
- The "plan not yet critiqued" rule requires *evidence a plan exists*, not just a status string.
- A completed-but-unreviewed patch always out-ranks a stale CHANGES-REQUESTED verdict — encoded
  as an explicit invariant, not an accident of list ordering.

## Why Previous Fixes Failed

| Prior Fix | What It Did | Why It Failed / Was Incomplete |
|-----------|-------------|-------------------------------|
| PR #1275 | Added `"ready"` to `_rule_plan_not_critiqued`'s `plan_status` set to catch `CRITIQUE="ready"` after a completed plan | Conflated two distinct meanings of `PLAN="ready"`: (a) post-ISSUE bootstrap, no plan doc yet; (b) plan completed, critique now ready. Fixed (b), broke (a) → #1640. Root cause: trusted a status string as a proxy for artifact existence. |
| (router design generally) | Matched verdicts via `"<LITERAL>" in verdict.upper()` substring checks scattered across G1/G3/G5/G6 and rows 8 | `.upper()` does not collapse underscores/whitespace, so cosmetic variants desync. No single normalization chokepoint → every literal is independently fragile (#1638). |
| (row ordering) | Relied on DISPATCH_RULES list order to express precedence between row 8 and 8b | Two predicates can be simultaneously true; "specific before general" was never encoded as an invariant, only as accidental adjacency. A stale verdict that should be superseded keeps winning (#1641). |

**Root cause pattern:** The router reads **un-normalized verdict text** and **un-qualified status strings** as ground truth, and **never supersedes a verdict when the stage that should clear it completes**. Fix the three levers — normalize at the write boundary, qualify the plan-existence check with real evidence, and encode the patch-supersedes-stale-verdict invariant — and all three acceptance sets fall out.

2. **Stage completion (entry):** `PipelineStateMachine.complete_stage` ([`agent/pipeline_state.py:504-545`](agent/pipeline_state.py#L504-L545)) sets the next stage `"ready"`. For ISSUE→PLAN this yields `PLAN="ready"` with no plan file. (No change here; #1640 is fixed at the read side with real evidence.)
3. **Meta computation:** `sdlc_stage_query._compute_meta` ([`tools/sdlc_stage_query.py:273-342`](tools/sdlc_stage_query.py#L273)) already calls `_find_plan_path(issue_number)` (line 318) for `revision_applied`. **`plan_exists` is computed here from the same resolved path — zero extra I/O.**
4. **Context build:** `sdlc_next_skill._build_context` ([`tools/sdlc_next_skill.py:75`](tools/sdlc_next_skill.py#L75)) assembles `proposed_skill` + `branch_exists`. (`plan_exists` rides in `_meta`, not `context`, since the path is already resolved in `_compute_meta`.)
5. **Dispatch decision:** `decide_next_dispatch` ([`agent/sdlc_router.py:981`](agent/sdlc_router.py#L981)) → guards G1–G7 → walk `DISPATCH_RULES`. 6. **Output:** `Dispatch(skill=...)` returned to the PM session, which invokes the chosen sub-skill.

## Architectural Impact

- **New dependencies:** None. Pure-Python string normalization; no new imports beyond `re` (already imported in the touched modules).
- **Interface changes:** A new module-level `normalize_verdict(text: str) -> str` helper (canonical home: `agent/sdlc_router.py`, imported by `tools/sdlc_verdict.py`). `_meta` gains one boolean key `plan_exists`. No public signature changes to `record_verdict`, `decide_next_dispatch`, or the CLI.
- **Coupling:** Slightly *decreases* fragility coupling — replaces N scattered raw-substring checks with one shared normalizer. `tools/sdlc_verdict` gains an import of `agent.sdlc_router.normalize_verdict`; this direction is already implied (verdict text is router-consumed) and introduces no cycle (`sdlc_router` does not import `sdlc_verdict`).
- **Data ownership:** Unchanged. `tools/sdlc_verdict.record_verdict` remains the single writer of `_verdicts`; this plan strengthens that invariant by making it the single *normalizer* too.
- **Reversibility:** High. Each of the three changes is independently revertable; normalization is idempotent and back-compatible (canonical form of an already-canonical string is itself).

## Solution

### Key Elements

- **`normalize_verdict(text)` canonical helper** — uppercase, replace `_` with space, collapse
  internal whitespace runs to a single space, strip ends. Idempotent. Lives in
  `agent/sdlc_router.py` (the router owns verdict vocabulary; the CRITIQUE_/SKILL_ constants
  already live there) and is imported by the write boundary.
- **Write-boundary normalization (#1638 primary fix)** — `record_verdict` normalizes the
  incoming `verdict` string before persisting it into `_verdicts[stage]["verdict"]`. The
  persisted value is always canonical. Optionally log (debug, non-blocking) when the input
  differed from the canonical form, for observability.
- **Read-side normalization (#1638 belt-and-suspenders)** — every router verdict comparison
  (G1, G3, G5, G6, `_rule_critique_*`, `_rule_review_has_findings`) routes its source text
  through `normalize_verdict()` instead of bare `.upper()`. This covers legacy `_verdicts`
  records written before this change and any future non-CLI writer.
- **Plan-existence evidence in `_meta` (#1640 fix)** — `_compute_meta` adds `plan_exists: bool`
  derived from the `plan_path` it already resolves (line 318). `_rule_plan_not_critiqued`
  accepts the `"ready"` status **only when `plan_exists` is true**; `STATUS_COMPLETED` continues
  to imply existence with no meta dependency (a plan can't be `completed` without a doc). This
  preserves the #1275 case verbatim.
- **Patch-supersedes-stale-verdict invariant (#1641 fix)** — add a **timestamp-staleness**
  early-exit to `_rule_review_has_findings`: the REVIEW verdict is *stale* (superseded) iff its
  `recorded_at` predates the latest `/do-patch` dispatch timestamp. When the verdict is stale,
  return `False` so row 8b (`_rule_patch_applied_after_review`) wins and re-review runs; when the
  verdict is **fresh** (`recorded_at >= latest /do-patch dispatch at`), the predicate returns
  `True` normally — a genuine post-patch `CHANGES REQUESTED` is a legitimate finding that must
  dispatch `/do-patch`. This is chosen over a bare `last_dispatched == /do-patch` boolean check
  because a TEST-fix patch *also* sets `last_dispatched == /do-patch` (`_rule_tests_failing`
  dispatches `/do-patch` too), so the boolean form would wrongly suppress a fresh post-patch
  finding. The timestamp comparison distinguishes "the verdict that triggered this patch" (stale,
  suppress) from "a new finding recorded after the patch landed" (fresh, dispatch). Row 8b is left
  as-is (it already gates correctly).

### Flow

**REVIEW records `CHANGES_REQUESTED`** → `record_verdict` normalizes → persists `CHANGES REQUESTED`
with `recorded_at=T0` → router row 8 matches canonical form → `/do-patch` dispatches at `T1 > T0` →
patch completes (`PATCH=completed`) → row 8 sees the verdict's `recorded_at (T0) < latest /do-patch
dispatch (T1)` → **stale** → row 8 early-exits → row 8b matches → `/do-pr-review` dispatches →
re-review records a *fresh* verdict at `T2 > T1`. If that fresh verdict is again `CHANGES REQUESTED`,
row 8 now sees `recorded_at (T2) >= latest /do-patch dispatch (T1)` → **fresh** → returns `True` →
`/do-patch` for the legitimate new finding. The loop terminates because each leg is keyed on a real
timestamp ordering, not a latching boolean.

**Fresh issue** → ISSUE completes → `PLAN="ready"`, no plan doc → `_compute_meta` sets
`plan_exists=False` → `_rule_plan_not_critiqued` rejects the `"ready"` case → row 1 / `_rule_no_plan`
or the `"ready"+plan_exists=False` path routes to `/do-plan` → plan written, `PLAN=completed` →
`_rule_plan_not_critiqued` matches (completed implies existence) → `/do-plan-critique`.

### Technical Approach

- **`normalize_verdict`** signature: `def normalize_verdict(text: str | None) -> str`. Returns
  `""` for falsy/non-str input (mirrors the graceful-failure contract). Implementation:
  `re.sub(r"\s+", " ", text.replace("_", " ")).strip().upper()`.
- **Constants stay as space-form** (`CHANGES REQUESTED`, `NEEDS REVISION`, `READY TO BUILD`,
  `MAJOR REWORK`, `APPROVED`) — they already match the canonical output of `normalize_verdict`,
  so `"<CONST>" in normalize_verdict(text)` is the new universal idiom. Add an `APPROVED`
  constant if one doesn't exist (G6 currently inlines the literal).
- **#1640 predicate** rewrite:
  ```python
  def _rule_plan_not_critiqued(stage_states, meta, context):
      plan_status = stage_states.get("PLAN")
      critique_status = stage_states.get("CRITIQUE")
      if critique_status not in (None, "pending", "ready"):
          return False
      if plan_status == STATUS_COMPLETED:
          return True                      # completed implies a plan doc exists (#1275 case intact)
      if plan_status == "ready":
          return bool(meta.get("plan_exists"))   # "ready" needs real evidence (#1640)
      return False
  ```
  Audit (acceptance criterion): grep confirms no *other* rule keys off `PLAN=="ready"` as a
  proxy for plan existence. (`_rule_no_plan` keys off `None`/`"pending"` — fine; it must still
  route the `"ready"+plan_exists=False` case to `/do-plan`. Verify this: with `PLAN="ready"`,
  `_rule_no_plan` returns `False`, so row 1 won't fire. Therefore the `"ready"+no-doc` case must
  be routed to `/do-plan` by *widening `_rule_no_plan`* to also accept `plan_status == "ready"
  and not plan_exists`, OR by a dedicated row. **Builder decision, captured as a spike-free design
  note:** widen `_rule_no_plan` to `plan_status in (None, "pending") or (plan_status == "ready"
  and not meta.get("plan_exists"))` so row 1 catches the bootstrap case and emits `/do-plan`.
  This is the cleanest single-rule home and keeps row 2 purely "plan exists, needs critique".)
- **#1641 timestamp-staleness early-exit** added near the top of `_rule_review_has_findings`,
  after the `pr_number` guard and after `review_verdict` is resolved (so a non-CHANGES-REQUESTED
  verdict is unaffected). It compares the REVIEW verdict's `recorded_at` against the latest
  `/do-patch` dispatch timestamp and suppresses the verdict **only when it is stale (pre-patch)**:

  - **Read the REVIEW verdict's `recorded_at`:** `_verdicts["REVIEW"]["recorded_at"]` is an ISO
    string written by `record_verdict` (`tools/sdlc_verdict.py:234,239`). Read it from
    `stage_states.get("_verdicts", {}).get("REVIEW", {})`. The text path already uses
    `_verdict_text`; add a sibling read for the timestamp from the same record dict.
  - **Read the latest `/do-patch` dispatch timestamp:** walk `stage_states["_sdlc_dispatches"]`
    (the FIFO list appended by `record_dispatch`, `agent/sdlc_router.py:1090-1113`; each entry is
    `{"skill": ..., "at": <isoformat>, "stage_snapshot": ...}`). Take the **last** entry whose
    `skill == SKILL_DO_PATCH` and read its `"at"` field. A small helper
    `_latest_dispatch_at(stage_states, skill) -> str | None` returning the most recent matching
    `"at"` (or `None`) keeps this readable.
  - **The comparison:** parse both ISO strings to aware `datetime` via
    `datetime.fromisoformat(...)` (Python 3.11+ parses the offset; `record_verdict` and
    `record_dispatch` both write `datetime.now(UTC).isoformat()`, so both are offset-aware).
    Suppress the verdict (`return False`, so row 8b wins) **only when**
    `verdict_recorded_at < latest_patch_dispatch_at`. Otherwise fall through to the normal
    `CHANGES REQUESTED` / `PARTIAL` / `REVIEW==failed` checks (fresh verdict → `True` → `/do-patch`).
  - **Edge cases (spell out, do not assume):**
    - (a) `recorded_at` missing/`None` on the verdict → treat as **NOT stale** (do not suppress) —
      the safe default is to surface the finding (`/do-patch`) rather than silently skip it.
    - (b) No prior `/do-patch` dispatch at all (`_latest_dispatch_at` returns `None`) → **NOT stale**;
      there is nothing the verdict could predate.
    - (c) Equal timestamps → **fresh** (tie-break: `<` is strict, so equal is NOT stale) — a finding
      is never silently dropped on a tie.
    - (d) Parse failure on either ISO string (`ValueError`) → **NOT stale** (same safe default as (a));
      wrap the `fromisoformat` calls so a malformed timestamp can never raise out of the predicate
      (the per-rule try/except at `decide_next_dispatch` line 1031 is a backstop, but the predicate
      should fail safe to "not stale" itself).

  Concretely (illustrative, not prescriptive on helper names):
  ```python
  def _rule_review_has_findings(stage_states, meta, context):
      if not meta.get("pr_number"):
          return False
      # ... resolve review_verdict (as today) ...
      if not review_verdict:
          return False
      # Timestamp-staleness supersession (#1641): if this verdict predates the
      # latest /do-patch dispatch, it is the verdict that triggered that patch —
      # it is stale; step aside so row 8b (_rule_patch_applied_after_review) runs.
      if _review_verdict_is_stale(stage_states):
          return False
      review_verdict_norm = normalize_verdict(review_verdict)
      if "CHANGES REQUESTED" in review_verdict_norm:
          return True
      # ... PARTIAL / REVIEW==failed unchanged ...
  ```
  where `_review_verdict_is_stale` reads `recorded_at` and `_latest_dispatch_at(..., SKILL_DO_PATCH)`,
  parses both to aware datetimes, and returns `recorded_at < latest_patch_at` — with every edge case
  above resolving to `False` (not stale).
- **G3** (line 363) and **G6** (line 575) verdict checks also switch to `normalize_verdict` for
  consistency, so the underscore variant can't desync the PR-lock redirect or the merge fast-path
  either (both are in-scope per #1638's "all router verdict literals" criterion).

## Failure Path Test Strategy

### Exception Handling Coverage
- [ ] `record_verdict` already returns `{}` on any failure (graceful-failure contract). - [ ] Router predicates are wrapped by `decide_next_dispatch`'s per-rule try/except (line 1031). ### Empty/Invalid Input Handling
- [ ] `normalize_verdict(None)` → `""`; `normalize_verdict("")` → `""`; `normalize_verdict("  ")` → `""` — unit tests for each.
- [ ] `_rule_plan_not_critiqued` with `meta={}` (no `plan_exists` key) and `PLAN="ready"` → `False` (defaults to "no evidence" → does not critique). Test this default-safe path.
- [ ] Verify the `"ready"+no-doc` case does not silently loop: with `plan_exists=False`, the router must emit `/do-plan` (row 1), not `Blocked` — assert the dispatch, guarding against an empty-output loop.

### Error State Rendering
- [ ] Router output is consumed by the PM session, not a user UI. ## Rabbit Holes

- **Don't rebuild the verdict vocabulary as an enum/state machine.** A `str` canonicalizer +
  `in` checks is sufficient and matches the existing idiom. Promoting verdicts to a typed enum
  touches every consumer and is a separate refactor.
- **Don't reorder all of `DISPATCH_RULES`.** #1641 is fixed with a single timestamp-staleness
  early-exit in `_rule_review_has_findings`; a global reorder risks the parity test and other
  inter-row precedence assumptions (e.g. #1639's branch). Surgical guard only.
- **Don't reach for a global per-issue pointer to express staleness.** `last_dispatched_skill`,
  `revision_applied`, and `pr_number` are all global pointers that cannot say *which* work cleared
  a verdict — that's the rows-4a/4b/4c point-patch trap. The timestamp predicate is the right tool;
  do not regress to a boolean pointer for the #1641 fix.
- **Don't fix #1639** (the sibling PLAN-branch ordering defect). It's a separate issue with its
  own acceptance criteria; bundling it expands the blast radius and the test surface.
- **Don't touch `agent/granite_container/`** — in-flight PR #1651 rewrites it; any edit here
  will conflict.
- **Don't add `plan_exists` to the `context` dict** when `_compute_meta` already resolves the
  plan path. Threading it through `_build_context` would re-walk `docs/plans/` a second time.

## Risks

### Risk 1: Read-side normalization changes a currently-passing verdict comparison
**Impact:** A verdict that today matches by luck (e.g. trailing whitespace tolerated by `in`)
could shift behavior.
**Mitigation:** `normalize_verdict` is a strict superset of `.upper()` for the canonical tokens
(it only collapses `_`/whitespace and upcases). The space-form constants are unchanged. Run the
full router suite serially (`-n0`) and diff dispatch outcomes against baseline for the existing
parametrized cases.

### Risk 2: `_rule_no_plan` widening accidentally swallows a legitimately-completed plan
**Impact:** If `plan_exists` is computed wrong (e.g. `find_plan_path` returns None for a real
plan due to a cwd/repo-root resolution miss), a completed plan could be misrouted back to
`/do-plan`.
**Mitigation:** The widening only triggers on `plan_status == "ready" AND meta["issue_number"] AND
not plan_exists` — never on `STATUS_COMPLETED`, and never when `issue_number` is None (B2b). A
completed plan is immune, and a session whose issue ref was dropped on retry falls back to the prior
status-only behavior rather than re-dispatching `/do-plan` for a real plan. Add a test for
`PLAN="ready"`+`plan_exists=True` → routes to `/do-plan-critique` (row 2), proving the evidence gate
opens correctly when the doc is present; and the B2b inverse-loop test (`issue_number=None` → not
re-dispatched).

### Risk 3: #1275 regression
**Impact:** Re-breaking the missed-critique dispatch #1275 fixed.
**Mitigation:** `STATUS_COMPLETED` path in `_rule_plan_not_critiqued` is preserved verbatim; the
existing `test_plan_completed_critique_ready_dispatches_critique` guards it. Treated as a
must-not-regress success criterion.

## Race Conditions

No new race conditions introduced. All three changes are synchronous, single-threaded
pure-function edits (router predicate logic + a string normalizer at a write that already uses
the `update_stage_states` cross-process-safe helper). The `_verdicts` single-writer invariant is
preserved — `record_verdict` remains the sole writer, now also the sole normalizer. `plan_exists`
is read at meta-compute time from a filesystem path that is already resolved in the same function
for `revision_applied`; no new read-after-write ordering dependency.
