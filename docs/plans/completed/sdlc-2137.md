# Uncommitted Work in Session Worktrees Has No Backstop

## Problem

Engineering sessions work in isolated git worktrees under `.worktrees/{slug}/`
on branch `session/{slug}` (see `docs/features/session-isolation.md`). In a
recent production incident, six modified uncommitted files in a
`session/dev-XXXX` worktree were destroyed — the reflog showed
`reset: moving to HEAD` — and the resumed session could not recover the work.
The delta was permanently lost.

**Current behavior:**

1. **Automated teardown discards dirty trees.** The session-completion `finally`
   block (`agent/session_executor.py:2419-2471`) force-removes any synthetic
   `dev-XXXXXXXX` worktree via `cleanup_after_merge` →
   `remove_worktree` → `git worktree remove --force`
   (`agent/worktree_manager.py:1189-1197`) on **every** session exit — normal
   return, exception, or cancellation. The unmerged-branch guard (#1646)
   protects only *committed* work; uncommitted working-tree edits (staged,
   unstaged, and untracked) are discarded with no stash or WIP-commit backstop.
   The same force-remove is reached by `_cleanup_stale_worktree`
   (`worktree_manager.py:827-863`) and `cleanup_after_merge`
   (`worktree_manager.py:1535-1537`).
2. **Nothing stops an agent from hard-resetting a dirty worktree.** The
   incident's reflog signature (`reset: moving to HEAD`) matches an agent-issued
   `git reset --hard`/`git stash` during a confused post-interruption recovery.
   No hook guards destructive git commands the way
   `.claude/hooks/validators/validate_no_raw_redis_delete.py` guards raw Redis
   writes.

The safeguards that DO exist protect other paths: `verify_worktree_branch`
refuses to switch branches on a dirty reused worktree
(`worktree_manager.py:342-365`); `ensure_clean_git_state` stashes rather than
resets but explicitly skips worktrees (`worktree_manager.py:1366-1369`) and has
no production callers.

**Desired outcome:**

Uncommitted session-worktree changes always survive in a recoverable ref (WIP
commit + durable named ref) before any force-remove, with the recovery pointer
logged; and destructive git commands (`git reset --hard`, `git clean -f`,
`git checkout -- .`, bare `git stash` without an explicit override) issued from
inside a dirty `.worktrees/` path are blocked by a PreToolUse hook validator.

## Data Flow

**Teardown preservation path:**
1. **Entry point**: session exit `finally` block (`session_executor.py`) OR
   `cleanup_after_merge` OR `_cleanup_stale_worktree`.
2. **`remove_worktree(repo_root, slug, ...)`**: before running
   `git worktree remove --force`, call the new
   `preserve_uncommitted_worktree_changes(repo_root, slug, worktree_dir)`.
3. **Preservation helper**: `git -C <worktree> status --porcelain` → if dirty,
   `git -C <worktree> add -A` + `git -C <worktree> commit --no-verify -m "WIP:
   auto-preserved before teardown ..."` onto `session/{slug}`, capture the
   commit SHA, then `git update-ref refs/session-wip/{slug} <sha>` in the common
   ref store (survives worktree removal and branch deletion), and log the
   pointer at WARNING.
4. **Force-remove proceeds**: `git worktree remove --force` runs as before; the
   WIP commit and `refs/session-wip/{slug}` ref persist.
5. **Output**: recovery pointer in the log
   (`[worktree-wip-preserved] slug=… ref=refs/session-wip/… sha=…`); recoverable
   via `git stash apply` equivalent / `git checkout refs/session-wip/{slug}`.

**Hook-validator path:**
1. **Entry point**: agent issues a Bash command; Claude Code fires PreToolUse
   hooks with `{tool_name, tool_input.command, cwd}` on stdin.
2. **`validate_no_destructive_git_in_worktree.py`**: parse command + cwd →
   `find_violation(command, cwd, is_dirty)`.
3. **Dirty detection**: if cwd resolves inside a `.worktrees/` path, run
   `git -C <cwd> status --porcelain`; non-empty ⇒ dirty.
4. **Decision**: destructive pattern + inside `.worktrees/` + dirty + no override
   ⇒ emit `{"decision": "block", "reason": ...}`. Otherwise exit 0 (allow).
5. **Output**: block message to the agent, or silent allow.

## Solution

### Key Elements

- **`preserve_uncommitted_worktree_changes(repo_root, slug, worktree_dir)`**:
  Detects a dirty worktree and preserves all uncommitted work (staged, unstaged,
  untracked) as a WIP commit on `session/{slug}` plus a durable named ref
  `refs/session-wip/{slug}`, logging the recovery pointer. Idempotent,
  non-blocking, fail-loud-not-fatal.
- **`remove_worktree` integration**: call the helper *before*
  `git worktree remove --force`. Preservation failure must NEVER block teardown —
  degrade to a loud `logger.error`/`logger.critical` and proceed.
- **`_cleanup_stale_worktree` integration**: same preservation call before its
  direct force-remove.
- **`validate_no_destructive_git_in_worktree.py`**: PreToolUse Bash hook blocking
  `git reset --hard`, `git clean -f[dx]`, `git checkout -- .` / `git checkout .`
  / `git restore .`, and bare `git stash`/`git stash push` (no pathspec) when cwd
  is inside a dirty `.worktrees/` path and no override token is present.
- **Override convention**: an inline `# allow-destructive-git` token (greppable,
  mirrors existing hook-override conventions) lets a human/agent deliberately run
  a destructive command.
- **Docs**: `docs/features/session-isolation.md` gains a
  "Uncommitted-work preservation" section documenting the semantics, the
  `refs/session-wip/*` recovery procedure, and the destructive-git guard.

### Flow

Session teardown → dirty worktree detected → WIP commit + `refs/session-wip/{slug}`
→ pointer logged → `git worktree remove --force` → work recoverable.

Agent runs `git reset --hard` in dirty worktree → PreToolUse hook fires → blocked
with recovery guidance → agent commits or uses override token instead.

### Technical Approach

- **Preservation mechanism (WIP commit + named ref, NOT `git stash`).** A plain
  `git stash` inside a worktree stores to the *per-worktree* `refs/stash`, which
  is destroyed with the worktree — useless as a teardown backstop. Instead:
  `git -C <worktree> add -A` (captures untracked + tracked), then
  `git -C <worktree> commit --no-verify --no-gpg-sign -m "WIP: auto-preserved
  before teardown [slug] [ISO-ts]"`, capture SHA via `git rev-parse HEAD`, then
  `git -C <repo_root> update-ref refs/session-wip/{slug} <sha>`. `update-ref`
  writes to the common ref store, so the ref survives both worktree removal and
  the unmerged-branch-guard branch deletion. `--no-verify` avoids pre-commit
  hooks hanging teardown; `--no-gpg-sign` avoids signing prompts.
- **Empty-dirty edge**: if `git status --porcelain` is empty, do nothing and
  return `{"preserved": False, "was_clean": True}`.
- **Non-blocking contract**: wrap the whole helper so any `subprocess` failure,
  timeout, or exception is caught, logged at ERROR/CRITICAL with the
  `[worktree-wip-preserve-failed]` tag, and returns a result dict — never raises
  into the teardown path. Teardown must remain idempotent and must never hang.
- **Validator dirty-detection**: `find_violation(command, cwd, is_dirty)` is the
  pure, unit-testable core (inject `is_dirty`). `_run_hook` computes `is_dirty`
  by resolving cwd, checking for a `.worktrees/` path segment, and running
  `git -C <cwd> status --porcelain` (guarded, fail-open). Block only when
  is_dirty is True — a `git reset --hard` on a clean tree loses nothing, so it is
  allowed.
- **Override token**: presence of `# allow-destructive-git` anywhere in the
  command string suppresses the block (matches the fail-open, human-in-command
  convention).
- **Command parsing**: match destructive git subcommands robustly across
  `cd .worktrees/x && git reset --hard` chains (reuse the cwd-and-cd inspection
  approach from `validate_no_uv_sync_in_worktree.py`).

## Rabbit Holes

- **Do NOT** try to make `ensure_clean_git_state` work on worktrees — it is

- **Do NOT** build a GC/retention daemon for `refs/session-wip/*` in this plan.
  Define the policy (manual reclamation; refs are cheap dangling-commit pointers)

  shell. Cover the four documented signatures + the `cd … &&` chain; fail open on

- **Do NOT** convert `git stash` usage elsewhere in the codebase — the validator
  only guards commands issued from inside a dirty `.worktrees/` cwd.

## Risks

### Risk 1: WIP commit pollutes the session branch history
**Impact:** A resumed session sees an extra `WIP: auto-preserved` commit on
`session/{slug}`.
**Mitigation:** This is the desired recovery affordance, not pollution — the
commit IS the preserved work. The durable `refs/session-wip/{slug}` ref gives a
stable pointer independent of branch state. Document the recovery/undo procedure
(`git reset --soft HEAD~1` to unstage the WIP if the session resumes and wants
its dirty tree back).

### Risk 2: Preservation hangs teardown (pre-commit hooks, signing prompts)
**Impact:** Session exit blocks.
**Mitigation:** `--no-verify --no-gpg-sign` + a subprocess timeout
(`settings.timeouts.git_subprocess_s`) on every git call; the outer try/except
degrades to loud logging and lets force-remove proceed.

### Risk 3: Validator false-positives block legitimate work
**Impact:** An agent legitimately needing `git reset --hard` in a worktree is
blocked.
**Mitigation:** Block only when the tree is *dirty* (clean-tree resets are
allowed) and provide the `# allow-destructive-git` override token; fail open on
any detection error.

## Race Conditions

### Race 1: Concurrent teardown and preservation
**Location:** `worktree_manager.remove_worktree`
**Trigger:** Two teardown paths racing on the same worktree.
**Data prerequisite:** The WIP commit and ref must be written before
`git worktree remove --force`.
**State prerequisite:** `remove_worktree` already holds the refuse-busy guard
(`worktree_busy_check`) that prevents removal while a non-terminal session
references the worktree, so preservation runs single-threaded per worktree in
practice.
**Mitigation:** `update-ref refs/session-wip/{slug}` is a last-writer-wins atomic
git operation; a second preservation on an already-clean tree (post-WIP-commit)
is a no-op. Idempotent by construction.
