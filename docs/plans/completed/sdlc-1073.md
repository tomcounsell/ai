# Drop `AgentSession` Delivery Fields and Remove `MESSAGE_DRAFTER_IN_HANDLER` Flag

## Problem

**Current behavior:**
Three `AgentSession` fields — `delivery_text`, `delivery_action`, `delivery_emoji` — remain declared in the Popoto model (`models/agent_session.py` L203-205) but no code reads or writes them. They are dead weight on every session hash.

A temporary feature flag — `MESSAGE_DRAFTER_IN_HANDLER` — guards the drafter-at-handler path in `TelegramRelayOutputHandler.send` (`agent/output_handler.py` L255) and `EmailOutputHandler.send` (`bridge/email_bridge.py` L306). The flag has a mirror in `config/settings.py::FeatureSettings.message_drafter_in_handler` (L186-195). It was introduced as an emergency rollback switch when PR #1072 (`#1035`) shipped the drafter refactor. The flag defaults to `True` and no production rollback has occurred.

**Desired outcome:**
- The three dead fields are deleted from the Popoto model and HDEL'd from existing Redis hashes.
- The feature flag and its four code touch-points are deleted; the drafter call becomes unconditional. The existing `try/except` error fallback (which protects against drafter failures) stays in place.
- Tests that exercised the flag's "off" branch are updated or deleted. Tests that asserted `_drafter_enabled is True` are deleted as trivial.
- Documentation no longer references the flag or the dead fields.

### Technical Approach

- **HDEL-only migration** is safe because nothing reads these fields. No backfill, no staged rollout. The migration script runs against local dev Redis as part of the build. Production Redis will be migrated by the PM's normal `/update` → `/do-deploy` path.
- **Drafter call becomes unconditional.** Previously, `if self._drafter_enabled:` wrapped the `try: draft = await draft_message(...); ...` block. After this change, the `try` block runs on every send with no gate. The inner `except Exception:` block already logs a warning and falls through to raw-text delivery — that behavior is preserved verbatim.
- **Self-draft steering + narration fallback + routing-field persistence** (lines 272-302 in `TelegramRelayOutputHandler.send`) are all nested inside the current `if self._drafter_enabled:` block. They move one indentation level out. No behavioral change — they already run only when the drafter succeeds.
- **Email handler has a simpler structure** (no steering, no narration gate — just draft-then-send). Removal is one indentation unwrap.
- **`FeatureSettings` may become empty.** Inspect the class after deletion. If empty, keep the class declaration (with a docstring noting it's a structural placeholder for future flags) — do NOT delete the class itself, because `config/settings.py` imports and compositions may reference it. Verify via `grep -rn "FeatureSettings" /Users/tomcounsell/src/ai`.
- **Migration idempotency.** `HDEL` on a missing field is a no-op. The script is safe to re-run.

## Rabbit Holes

- **Don't delete `FeatureSettings` itself if it becomes empty.** Keep the class as a structural placeholder. Deleting the class risks breaking Settings composition and import chains elsewhere.
- **Don't try to rewrite the drafter's internal fallback logic.** The `try/except Exception:` block that catches drafter failures is orthogonal to the flag and stays unchanged. Scope creep risk.
- **Don't rename or relocate the self-draft steering / narration fallback / routing-field-persistence helper methods** in `TelegramRelayOutputHandler`. They just get one indentation level removed from their call sites. Leave the method definitions alone.
- **Don't backfill or migrate the deleted fields.** There is no data to preserve. Pure HDEL.
- **Don't touch any test that references `determine_delivery_action`** (the function). It is a separate concept from the `delivery_action` schema field.
- **Don't expand scope to other `AgentSession` fields** that might also be dead weight. This PR is narrowly the three `delivery_*` fields and the one flag.

## Risks

### Risk 1: Production Redis has stale hash fields after deploy
**Impact:** Existing `AgentSession:*` records in production Redis still carry the three dead fields after the code ships. Memory waste; no functional breakage.
**Mitigation:** Migration script is part of the PR. Deploy order: merge → `/do-deploy` → run migration script on each bridge machine's local Redis. Migration is idempotent so re-running is safe. If we forget to run it, the harm is zero behavioral impact and a small bytes/record overhead that disappears naturally as old sessions expire (AgentSession records have TTLs).

### Risk 2: Dropping `FeatureSettings.message_drafter_in_handler` breaks env loading
**Impact:** If something else in `config/settings.py` references the field (a validator, a property, a factory), removal could crash Settings construction at startup.
**Mitigation:** Pre-change `grep -rn "message_drafter_in_handler" /Users/tomcounsell/src/ai` must show exactly the five expected locations (settings.py declaration, both handler reads, both test files). If any other reference appears, investigate before removing.

### Risk 3: A rollback becomes necessary after the flag is gone
**Impact:** If a production issue surfaces that would previously have been mitigated by setting `MESSAGE_DRAFTER_IN_HANDLER=0`, we now must `git revert` the PR instead.
**Mitigation:** Per the original plan's language, the two-week stability window is specifically to reduce this risk to near-zero. Open Question 1 below handles the timing call.
