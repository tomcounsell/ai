# PM sessions with distinct slugs serialize on project_key, blocking sibling SDLC parallelism

## Problem

When a parent PM spawns multiple sibling PM children—each with a distinct slug, branch, and
`.worktrees/{slug}/` directory—those children execute strictly sequentially because they all share
the same `worker_key = project_key`. The parallelism infrastructure (worktrees, branch isolation,
global 8-slot semaphore) is already in place but is bypassed by the routing layer.

**Current behavior:**

```python
# models/agent_session.py:395-396
if self.session_type == SessionType.PM:
    return self.project_key  # slug is ignored entirely
```

Three sibling PM sessions created within 21 seconds of each other ran in strict FIFO order:

```
sdlc-1215  ~9 min   (only started after nothing else was running)
sdlc-1206  ~17 min  (only started after sdlc-1215 completed)
sdlc-1214  ~20 min  (only started after sdlc-1206 completed)
```

Total wall time: ~46 min. With parallel execution, wall time = max(9, 17, 20) = ~20 min.

**Desired outcome:**

Slugged PM sessions execute concurrently (subject to the global `MAX_CONCURRENT_SESSIONS=8` cap),
matching the existing parallelism behaviour that dev sessions gained in PR #1087. Slugless PM
sessions continue to serialize per `project_key` — the original PR #828 invariant is preserved.

## Spike Results

### spike-1: What git state do sibling PM sessions share when at PLAN/ISSUE/CRITIQUE stages?

- **Assumption**: "PM sessions with slugs always operate in isolated worktrees, so their git state
  is always independent."
- **Method**: code-read
- **Finding**: FALSE. `resolve_branch_for_stage` maps PLAN, ISSUE, and CRITIQUE stages to
  `("main", False)` — no worktree, main checkout. When a PM session is dispatching `/do-plan` or
  `/do-plan-critique`, it runs in the main repo root (`working_dir = repo_root`), not in
  `.worktrees/{slug}/`. The worktree exists but the PLAN stage bypasses it entirely.
  Only BUILD, TEST, PATCH, REVIEW, and DOCS stages use the worktree.
- **Confidence**: high — confirmed by reading `agent/agent_session_queue.py:407` and
  `agent/session_executor.py:705-756`
- **Impact on plan**: The simple "change PM to fall through to slug" fix is NOT safe for PM
  sessions at PLAN/ISSUE/CRITIQUE stages. Two slugged PM siblings both at the PLAN stage would
  race on the main checkout's HEAD. The fix must preserve main-checkout serialization for these
  stages while enabling worktree-stage parallelism.

### spike-2: What Redis state is shared between sibling PM sessions?

- **Assumption**: "Pipeline state (stage_states) and sustainability keys are project-scoped and
  would be contaminated by concurrent PM sessions."
- **Method**: code-read
- **Finding**: `PipelineStateMachine` is keyed by `AgentSession` (session-scoped, stored in
  `session.stage_states` via Popoto). Sibling PM sessions each have their own `AgentSession`
  record and therefore their own `stage_states`. No shared pipeline state. Sustainability keys
  (`{project_key}:sustainability:queue_paused`, `{project_key}:worker:hibernating`) are
  project-scoped but read-only during pop — they would affect all sibling PM sessions equally
  (a circuit OPEN pauses all of them), which is correct behavior.
- **Confidence**: high — confirmed by `agent/pipeline_state.py:267-378` and
  `agent/session_pickup.py:183-207`
- **Impact on plan**: Redis state is not a blocker. Sibling PM sessions do not contaminate each
  other's pipeline state.

### spike-3: Does the cap-reservation strategy avoid PM↔dev deadlock under slug-based routing?

- **Assumption**: "Three parallel PM siblings each spawning a dev child could exhaust the 8-slot
  cap and deadlock (#1004 regression)."
- **Method**: code-read
- **Finding**: The global semaphore (`MAX_CONCURRENT_SESSIONS=8`) is acquired per-session at
  execution time in `_worker_loop`. PM sessions that are waiting for dev children do NOT hold the
  semaphore while waiting — the semaphore is released when the session's harness subprocess exits
  and the worker loop re-acquires it for the next pop. PM sessions re-enqueue themselves (via
  `valor-session create --role pm` from within the harness) rather than blocking the worker loop.
  The deadlock risk from #1004 was a different mechanism (PM waiting synchronously inside a worker
  slot). Current architecture: PM exits, re-enqueues, worker slot is freed.
- **Confidence**: high — confirmed by reading `agent/agent_session_queue.py:1203-1209` and
  `agent/session_executor.py:579-620`
- **Impact on plan**: No deadlock risk under the current re-enqueue pattern. Three parallel PM
  siblings each completing one turn and re-enqueueing dev children is safe.

## Data Flow

Session routing from enqueue to execution:

1. **Enqueue**: `valor-session create --role pm --slug sdlc-1228 --message "..."` calls
   `_push_agent_session()` with `session_type=PM` and `slug=sdlc-1228`. Inline `worker_key`
   computation (`agent/agent_session_queue.py:361`) currently sets `_wk = project_key`, ignoring
   slug.

2. **Notify**: pub/sub message `{"worker_key": "valor", "is_project_keyed": True}` wakes the
   project-keyed worker loop.

3. **Pop**: `_pop_agent_session(worker_key="valor", is_project_keyed=True)` queries
   `project_key=valor, status=pending`. Filters to sessions whose `s.worker_key == "valor"`.
   Dequeues the oldest eligible session.

4. **Execute**: `_execute_agent_session(session)` reads `session.slug` and `session.current_stage`.
   If stage is PLAN/ISSUE/CRITIQUE, `resolve_branch_for_stage` returns `("main", False)` — session
   runs in main checkout. If stage is BUILD/TEST/PATCH, returns `(f"session/{slug}", True)` —
   session runs in worktree.

5. **Bottleneck**: Steps 3-4 are single-threaded within the project-keyed worker loop. While
   sdlc-1215's PLAN stage runs in the main checkout, sdlc-1206 waits in the queue even though
   it could theoretically run concurrently if its stage was BUILD (worktree).

**After the fix**, the flow changes at step 1-3:

1. `_push_agent_session()` computes `_wk = slug` for a slugged PM session.
2. Notify: `{"worker_key": "sdlc-1228", "is_project_keyed": False}`.
3. `_pop_agent_session(worker_key="sdlc-1228", is_project_keyed=False)` queries
   `slug=sdlc-1228, status=pending` — isolated per slug.

However, a stage-awareness gate must prevent two slugged PM sessions from concurrently executing
turns that resolve to the main checkout.

## Solution

### Key Elements

- **Stage-aware `worker_key` for PM**: Slugged PM sessions return the slug as `worker_key` when
  the session's current SDLC stage is a worktree-compatible stage (BUILD/TEST/PATCH/REVIEW/DOCS).
  When the stage is a main-checkout stage (PLAN/ISSUE/CRITIQUE/MERGE/None), they fall back to
  `project_key`. Slugless PM sessions always return `project_key`.

- **Three inline sync points**: The `worker_key` logic is duplicated in three places and all three
  must be updated in lockstep: `models/agent_session.py:379-400` (the property), and two inline
  copies in `agent/agent_session_queue.py` (one at push, one at local-enqueue).

- **Test coverage**: Unit tests for the new property branches + an integration test that creates
  two slugged PM sessions at BUILD-compatible stages and asserts they get distinct `worker_key`s.

- **Documentation update**: `docs/features/bridge-worker-architecture.md` Decision Table must gain
  a new row for `pm | yes (worktree stage) | slug | Parallel-safe`.

### Flow

New PM session (slug=sdlc-1228, stage=BUILD) created
→ `worker_key` returns `"sdlc-1228"` (slug, because stage is worktree-compatible)
→ Slug-keyed worker loop starts, pops session, runs in `.worktrees/sdlc-1228/`
→ Sibling PM session (slug=sdlc-1214, stage=BUILD) gets its own worker loop
→ Both run concurrently

New PM session (slug=sdlc-1228, stage=PLAN) created
→ `worker_key` returns `"valor"` (project_key, because PLAN is a main-checkout stage)
→ Project-keyed worker loop pops and serializes (same as today for PLAN stages)

### Technical Approach

**Option A (Recommended): Stage-conditional `worker_key` property**

```python
# models/agent_session.py
@property
def worker_key(self) -> str:
    if self.session_type == SessionType.TEAMMATE:
        return self.chat_id or self.project_key
    if self.session_type == SessionType.PM:
        # Slugged PMs at worktree-compatible stages route by slug (parallel-safe).
        # Slugless PMs and PMs at main-checkout stages serialize by project_key.
        if self.slug and self._pm_stage_is_worktree_compatible():
            return self.slug
        return self.project_key
    # dev: isolated by slug if present
    if self.slug:
        return self.slug
    return self.project_key

def _pm_stage_is_worktree_compatible(self) -> bool:
    """Return True if the current stage uses a worktree (not the main checkout)."""
    stage = getattr(self, "current_stage", None)
    MAIN_CHECKOUT_STAGES = {"PLAN", "ISSUE", "CRITIQUE", "MERGE", None}
    return stage not in MAIN_CHECKOUT_STAGES
```

This matches the logic already in `resolve_branch_for_stage`. For slugless PMs and PMs at
PLAN/ISSUE/CRITIQUE/MERGE stages, behavior is identical to today.

**Sync point discipline**: Both inline duplications in `agent/agent_session_queue.py` must be
updated with matching logic. The existing `# KEEP IN SYNC` comment guides where.

**Why not Option B (always route slugged PM by slug)?** If we unconditionally use slug for all
slugged PMs regardless of stage, two siblings both at PLAN stage would race on the main checkout's
HEAD, causing git conflicts. Option A's stage gate prevents this.

**Why not Option C (opt-in flag)?** The issue's acceptance criteria specify automatic parallelism.
An opt-in flag would require callers to always pass it — adding friction and making the default
behavior wrong. Stage-conditional routing is self-contained and requires no caller changes.

## Failure Path Test Strategy

### Exception Handling Coverage

- `_pm_stage_is_worktree_compatible()` reads `current_stage` via `getattr(self, "current_stage",
  None)` — if `current_stage` is unset (e.g., a freshly-created PM session before any SDLC
  stage has been assigned), `None` is treated as a main-checkout stage and falls back to
  `project_key`. This is safe: a new PM with no stage should serialize until it reaches a
  worktree stage.
- If `MAIN_CHECKOUT_STAGES` lookup fails (should not happen — it's a frozenset literal), the
  `in` operator raises `TypeError`. Unit test: assert `_pm_stage_is_worktree_compatible` with
  a None stage returns False.

### Empty/Invalid Input Handling

- PM session with `slug=None` → falls through to `project_key` (unchanged behavior)
- PM session with `slug=""` (empty string) → same as slugless (falsy check)
- `current_stage=None` → treated as main-checkout stage → `project_key` (safe)
- `current_stage="UNKNOWN_STAGE"` → not in MAIN_CHECKOUT_STAGES → returns slug (treated as
  worktree-compatible). This is slightly optimistic but safe: an unknown stage is more likely a
  future worktree stage than a main-checkout stage.

### Error State Rendering

- `worker_key` is a pure property with no side effects. It cannot emit user-visible errors.
- The `is_project_keyed` discriminator is derived from `worker_key == project_key`. After the
  fix, slugged PMs at worktree stages will have `is_project_keyed = False`. Callers that use
  this discriminator (`session_pickup.py:214-219`, `agent_session_queue.py:372`) will switch to
  slug-based filters — which is correct because the session's slug field is set.

## Rabbit Holes

- **Unifying slugless PM and slugless dev routing**: Out of scope. The slug is the only reliable
  isolation signal for PM sessions.
- **Dynamic stage re-routing (hot-swap worker_key mid-session)**: When a PM session progresses
  from PLAN to BUILD, its `worker_key` will change. The existing session is already running on
  a project-keyed worker. After it completes and re-enqueues, the new turn is pushed with the
  updated slug-based `worker_key`. No mid-session re-routing needed — the re-enqueue handles it
  naturally.
- **Per-stage concurrency caps**: `MAX_CONCURRENT_PM_PLAN_SESSIONS=1` or similar. Unnecessary
  complexity — the stage gate already serializes main-checkout stages.
- **Detecting stage from the existing `current_stage` property**: `current_stage` is derived from
  `stage_states` which is per-session. There may be race conditions in reading it during a rapid
  multi-stage transition. The risk is bounded: if `current_stage` briefly shows the wrong stage,
  the worst case is a PM session briefly getting a project-keyed slot when it could have gotten
  a slug-keyed one — it serializes unnecessarily for one turn. This is acceptable.

## Risks

### Risk 1: Two slugged PM siblings both at PLAN stage collide on main checkout

**Impact:** Git conflict during `git push` from `do-plan` on `main` branch.
**Mitigation:** The stage gate in `_pm_stage_is_worktree_compatible()` returns `False` for PLAN,
ISSUE, and CRITIQUE stages — these sessions get `worker_key = project_key` and serialize through
the existing project-keyed worker loop. The collision cannot happen under the fixed logic.

### Risk 2: `current_stage` is stale or None when `worker_key` is read at enqueue time

**Impact:** A session that should be routed to a slug-keyed worker gets routed to the project-keyed
worker, causing unnecessary serialization for one turn.
**Mitigation:** This is a safe degradation: the session serializes instead of parallelizing for
one turn. It does not cause data corruption or incorrect behavior. The next time the session is
re-enqueued (after its current turn completes), `current_stage` will be set correctly.

### Risk 3: Inline duplicate out of sync with property

**Impact:** The enqueue path (pub/sub worker_key computation) routes the session to the wrong
worker loop, while the pop path uses the correct filter. Session is never popped.
**Mitigation:** The existing `# KEEP IN SYNC` comment mandates simultaneous updates. The PR must
update all three locations in one commit. A unit test that exercises the enqueue-path inline
computation will catch divergence.

### Risk 4: `_pop_agent_session` slug lookup does not find PM sessions

**Impact:** Slug-keyed worker loop wakes, calls `_pop_agent_session(worker_key=slug,
is_project_keyed=False)`, queries `slug=slug, status=pending` — this query will find the PM
session because `slug` is a `KeyField` and the PM session was created with `slug=...`.
**Mitigation:** Confirmed by `session_pickup.py:227`: the first query is `slug=worker_key`. PM
sessions with slugs will be found. No additional code change needed in the pop path.

## Race Conditions

### Race 1: Two slugged PM siblings both enter PLAN stage simultaneously

**Location:** `models/agent_session.py:379-400` (`worker_key` property)
**Trigger:** Parent PM spawns two children at the same time; both are at stage=None initially.
Both compute `worker_key = project_key` (correct — None stage → main-checkout). When they later
reach BUILD stage and re-enqueue, both correctly compute `worker_key = slug` and get distinct
worker loops.
**Data prerequisite:** `current_stage` must be set before `worker_key` is read for routing.
**State prerequisite:** Stage transitions are atomic per-session (PipelineStateMachine writes to
`session.stage_states` and saves). No inter-session coordination needed.
**Mitigation:** None needed — the stage gate naturally handles the None-stage case. Sessions
without a stage serialize; sessions with a worktree stage parallelize.

### Race 2: Worker key changes between enqueue notification and pop

**Location:** `agent/agent_session_queue.py:361-376` (push), `session_pickup.py:209-231` (pop)
**Trigger:** At enqueue time, stage=PLAN so `_wk = project_key`. By pop time, stage has advanced
to BUILD so `session.worker_key = slug`. The project-keyed worker pops and tries to execute —
but `session.worker_key` now returns a slug. The `s.worker_key == worker_key` filter at
`session_pickup.py:219` rejects this session, leaving it unpopped in the project-keyed queue.
**Mitigation:** This race is extremely unlikely (stage would have to advance between enqueue and
pop, typically < 1 second). If it occurs, the session sits in the project-keyed queue indefinitely
until the health check's fallback pop recovers it. Acceptable: no data corruption, just latency.
A future refactor could add a `worker_key` field to `AgentSession` and set it at enqueue time
(immutable for the session's lifetime). Out of scope for this plan.
