# Cleanup reflection: reap orphan claude/worker/MCP processes whose parent died

## Problem

When the bridge or worker dies ungracefully (panic, force-kill, SIGKILL, restart-without-graceful-shutdown), the child `claude` CLI process and its grandchildren (4+ MCP server subprocesses) become orphans: their PPID flips to 1 (launchd) and they keep running indefinitely. They hold file handles, consume RAM/CPU, and — for `claude` specifically — keep an Anthropic API session warm long after the parent that knew about it has gone.

**Current behavior:**

`agent/session_health.py` exposes `_cleanup_orphaned_claude_processes()` (lines 1722–1801) which:
- Is **only invoked once at worker startup** in `worker/__main__.py:296`. It is **not** wired into any reflection — the `agent-session-cleanup` reflection (`config/reflections.yaml:37`) calls `cleanup_corrupted_agent_sessions()` only.
- Matches **only** `claude_agent_sdk/_bundled/claude` (a single pgrep pattern).
- Does not touch worker (`python -m worker`) processes or MCP server subprocesses (`mcp_servers/*`).
- Does not walk the descendant tree — when it kills a `claude` PID, the MCP children attached to that `claude` are not reaped explicitly. They become orphans-of-orphans (PPID flips again to 1) and persist until the next worker startup.
- Has **no liveness gate**: it kills any matched process whose PPID==1, regardless of whether `AgentSession.last_heartbeat_at` says the corresponding session is healthy. (In practice this is safe today because it only runs at startup before any new sessions exist, but if we move it to a reflection it stops being safe.)
- Emits no metric — no count surfaced in the dashboard, no analytics rollup signal.

Between worker restarts (which can be hours or days apart), orphans accumulate. The fazm project documented `20+ orphan ACP bridges by Apr 30 2026` from the same failure mode. The Claude CLI itself has [Anthropic issue #51860](https://github.com/anthropics/claude-code/issues/51860) tracking orphaned `claude-code` CLI pinning 100% CPU after non-graceful parent exit — direct reinforcement of this issue's blast radius.

The current cleanup function leaks three classes of orphan:

1. **Worker-orphan claude:** worker dies → `claude` reparented to PID 1 → caught at next startup only.
2. **Bridge-orphan worker:** bridge dies but launchd `KeepAlive=true` respawns a new bridge → old worker (started by the dead bridge) is reparented to PID 1 → never reaped.
3. **Claude-orphan MCP subprocess:** `claude` killed cleanly → MCP child processes were children of `claude`, now reparented to PID 1 → never reaped.

**Desired outcome:**

A centralized reaper, run hourly from the existing reflection schedule, detects orphan `claude` / worker / MCP-server processes whose PPID==1 AND whose owning AgentSession has no fresh heartbeat, terminates them with their full descendant tree (SIGTERM + SIGKILL escalation), and emits a count to the dashboard / analytics. No per-subprocess watchdog code is added.

## Research

**Queries used:**
- `psutil reap orphan child processes PPID 1 reparented descendant tree SIGTERM SIGKILL pattern 2026`

**Key findings:**
- **psutil.Process pattern** ([psutil docs](https://psutil.readthedocs.io/)) — `proc.terminate()` (SIGTERM with PID-reuse check) → `proc.wait(timeout=N)` → `proc.kill()` (SIGKILL with PID-reuse check). PID-reuse protection inside psutil is the differentiator vs. raw `os.kill()`; macOS recycles PIDs in ~5 minutes. **Informs:** prefer `psutil.Process(pid).terminate()/.kill()` over `os.kill(pid, signal.SIGTERM)` for safety against PID reuse. Or, if staying on raw `os.kill`, follow PR #1236's pattern of single-shot drain (snapshot then clear) so a stale PID never lives across more than one tick.
- **Descendant tree reaping pattern** (Ray's Raylet, [Ray docs](https://docs.ray.io/en/latest/ray-core/user-spawn-processes.html)) — recursively walk `proc.children(recursive=True)` and SIGKILL each, step-by-step as parents die. Confirms the issue's preference to walk the descendant tree explicitly rather than trusting cascaded SIGTERM (MCP servers don't propagate signals to their own children reliably).
- **Anthropic claude-code issue #51860** ([anthropics/claude-code#51860](https://github.com/anthropics/claude-code/issues/51860)) — orphaned `claude-code` CLI from non-graceful parent exit pins 100% CPU indefinitely. Independently confirms claude CLI is a pathological orphan candidate, not just RAM/handle-bound.

1. **Entry point:** Reflection scheduler tick fires `agent-session-cleanup` (`config/reflections.yaml:37`, 1h interval) → calls `cleanup_corrupted_agent_sessions()` (`agent/agent_session_queue.py:86` re-export, real impl `agent/session_health.py:1607`).

3. **NEW pass appended:** Call `_reap_orphan_session_processes()` (renamed/extended from `_cleanup_orphaned_claude_processes`):
   - Use `psutil.process_iter(['pid', 'ppid', 'cmdline', 'create_time'])` to enumerate.
   - Match candidates by cmdline regex against three signatures: `claude_agent_sdk/_bundled/claude`, `python -m worker`, `mcp_servers/*` entry points.
   - Filter to candidates with `ppid == 1`.
   - **Liveness gate:** for each candidate, attempt to find an owning `AgentSession` by `claude_pid` field if persisted (see Solution / Open Questions), else fall back to "any AgentSession with `last_heartbeat_at` newer than 30 minutes is considered alive — skip kill." Without a PID-to-session mapping, the gate is conservative: if **any** session has a fresh heartbeat AND we cannot prove the orphan PID isn't owned by it, skip. (See Open Question 1.)
   - **Self-protection:** never kill `os.getpid()`, never kill the worker that's actively running this reflection.
   - For each surviving orphan: `psutil.Process(pid).terminate()`, walk `.children(recursive=True)` and `.terminate()` each, stage all PIDs on a single-shot `_pending_sigkill_orphans: set[int]` drained at top of next tick (matching PR #1236 pattern).
4. **Metric write:** Increment `{project_key}:session-health:orphan_process_reaped` Redis counter per kill.
5. **Logging:** `logger.info("[orphan-reap] Killed PID %d (cmd=%s, descendants=%d)", ...)`.
6. **Output:** Reflection findings list includes orphan-kill count; surfaces in `localhost:8500/dashboard.json` reflections panel and the analytics daily rollup.

## Solution

### Key Elements

- **`_reap_orphan_session_processes()`**: New function in `agent/session_health.py`. Scans process table for session-owned signatures with PPID==1 and stale liveness, kills them with descendant tree, returns count.
- **Heartbeat-gated liveness check**: Reuses existing `AgentSession.last_heartbeat_at`. Stale threshold: 30 minutes (well above the 5-minute health-check cadence, well below the 1-hour reflection interval).
- **Single-shot SIGKILL drain**: `_pending_sigkill_orphans: set[int]` (separate from PR #1236's `_pending_sigkill` to avoid name conflict and keep concerns distinct). Drained at the START of every cleanup reflection tick, exactly like PR #1236's pattern.
- **Reflection wiring**: Append the call to `cleanup_corrupted_agent_sessions()` so the existing `agent-session-cleanup` reflection covers both DB-row corruption AND OS-process orphans. Returns total count (corrupted_sessions + orphan_processes_killed) wrapped in a structured dict, OR keep `cleanup_corrupted_agent_sessions()` returning its existing int and have the reflection wrapper sum both. (Decision: keep cleanup_corrupted_agent_sessions's return signature unchanged; have the reflection's outer return aggregate. See Open Question 4.)
- **Metric**: Redis counter `{project_key}:session-health:orphan_process_reaped` per kill. Project key resolved from owning AgentSession if known, else `{worker_project_key}` fallback.
- **Kill switch**: `DISABLE_ORPHAN_PROCESS_REAP=1` env var (parity with PR #1236's `DISABLE_ORPHAN_REAP` and the existing `DISABLE_PROGRESS_KILL`).

### Flow

Worker starts → reflection scheduler → 1h tick → `cleanup_corrupted_agent_sessions()` (existing logic) → `_reap_orphan_session_processes()` (new) → process_iter → filter by signature + PPID==1 + heartbeat-stale + not-self → terminate + walk children + stage SIGKILL → next tick drains SIGKILL set → metric increments → reflection findings list reports count → dashboard.

### Technical Approach

- **Process enumeration:** `psutil.process_iter(['pid', 'ppid', 'cmdline', 'create_time'])` with try/except per-iteration (NoSuchProcess, AccessDenied) — never crash the reflection on a single bad PID.
- **Signature matching:** join cmdline list with spaces, regex match against three patterns (compiled once at module load):
  - `r"claude_agent_sdk/_bundled/claude\b"` (Claude CLI)
  - `r"python\s+-m\s+worker\b"` (worker process)
  - `r"mcp_servers/\w+\.py\b"` (MCP server entry points)
- **PPID==1 gate:** `proc.info['ppid'] == 1` — orphan signal.
- **Self-protection:** explicitly skip `os.getpid()`, skip the parent worker's PID (read from a Redis registry written at worker startup, OR fall back to "skip any process whose cmdline matches `python -m worker` AND has children" — alive workers have MCP/claude descendants while dead workers don't).
- **Heartbeat gate:** load all `AgentSession.query.filter(status="running")` (existing convention, ~1-3 rows per worker), check max `last_heartbeat_at`. If max heartbeat age < 30 minutes, conservatively skip claude/MCP candidates (we can't prove which session owns them without a PID field). Reap worker-process orphans regardless (worker has its own heartbeat semantics — see Open Question 2). The 30-min threshold ensures the gate doesn't false-positive against the 5-min health-check cadence.
- **Descendant walk:** `psutil.Process(pid).children(recursive=True)` BEFORE killing the parent — capture the tree first, then terminate parent, then iterate children. SIGTERM each, stage all on `_pending_sigkill_orphans`.
- **Two-tick escalation:** identical pattern to PR #1236 — top-of-tick drain attempts SIGKILL on `_pending_sigkill_orphans` snapshot, unconditionally clears the set regardless of outcome.
- **No psutil over raw os.kill:** use `psutil.Process(pid).terminate()` (PID-reuse safe) on the parent and root descendants. Inside the descendant walk, also use `psutil.Process.kill()` for SIGKILL escalation (PID-reuse safe). This is a deliberate upgrade from the existing `_cleanup_orphaned_claude_processes` (which uses raw `os.kill` and a 0.5s sleep loop).

## Failure Path Test Strategy

### Exception Handling Coverage
- [ ] `psutil.NoSuchProcess`, `psutil.AccessDenied`, `psutil.ZombieProcess` exceptions per-iteration: each must be logged at DEBUG and `continue` the loop. Test asserts that one bad PID does not abort the scan.
- [ ] `psutil.Process(pid).children(recursive=True)` raises if parent dies mid-walk: catch and continue with what we captured.
- [ ] Top-level `try/except Exception` wrapping the whole reaper: log at WARNING and return 0. Test asserts that a `psutil.process_iter` failure does not crash the reflection.

### Empty/Invalid Input Handling
- [ ] Empty process table (no orphans): function returns 0, no kills, no log spam.
- [ ] No AgentSession records at all (fresh install): heartbeat gate degrades safely. Decision: when `AgentSession.query.filter(status="running")` returns empty, treat as "no session is alive" and proceed with reaping (matches startup-only behavior of existing function).

## Rabbit Holes

- **Per-subprocess PPID watchdog (fazm pattern):** Tempting because it reacts in 5s instead of 60min, but distributes complexity into every spawn path (`agent/sdk_client.py`, every MCP server, `worker/__main__.py`). The user explicitly chose centralized in the issue. Defense-in-depth could be added later as a separate issue. **Do not pursue here.**
- **Cross-machine reaping:** Each machine only has visibility into its own process table. Do not attempt to reap orphans on other machines via Redis coordination.
- **Replacing `monitoring/bridge_watchdog.py`'s zombie killer:** That's a separate concern (etime-based, watchdog-scoped). Leave it alone — it's a defense-in-depth layer that runs every 60s and won't conflict.
- **Adding `claude_pid` to AgentSession in this plan:** Tempting for precise PID-to-session ownership, but the conservative heartbeat gate (skip if any session is alive) achieves the same safety. Adding the field is a real code change in `agent/sdk_client.py` (set the field in `on_sdk_started`) and `models/agent_session.py` — moves this from Small to Medium appetite. Defer to a follow-up issue if metrics show the conservative gate misses too many orphans.
- **Process ownership via `/tmp/`-style PID files:** The fazm `__watchdogStartPpid` pattern uses `/tmp/fazm-dev-status`. The Redis-backed AgentSession is already our equivalent — adding a parallel filesystem registry would duplicate state.

## Risks

### Risk 1: Killing a healthy `claude` process whose owning session has a stale heartbeat (false positive)
**Impact:** A long-running but legitimately-alive Claude session is SIGTERMed mid-API-call. The user sees a session abruptly end with no clear cause.
**Mitigation:** The 30-minute heartbeat threshold is 6× the health-check cadence (5min). A heartbeat older than 30min is itself a signal that the session is broken — issue #1246 documents tier-1 stdout-staleness reaper missing this. Adding `DISABLE_ORPHAN_PROCESS_REAP=1` kill switch lets ops disable while investigating. **Acceptance:** the false-positive rate is monitored via the Redis counter; if it exceeds N kills/day across all machines, escalate to adding `claude_pid` for precise ownership (Open Question 1).

### Risk 2: Killing the actively-running parent worker that's hosting this reflection (suicide)
**Impact:** Worker dies mid-reflection, launchd respawns it, every running session is recovered as `interrupted`. Cascading session loss.
**Mitigation:** Hard-coded `os.getpid()` skip. Additionally: skip any process matching `python -m worker` whose PPID is **not** 1 (a launchd-respawned worker has PPID==launchd, not orphaned). Test: spawn a fake worker process with PPID==1, run the reaper, assert it kills the fake but not the test runner.

### Risk 3: Double-kill race with `monitoring/bridge_watchdog.py`
**Impact:** Watchdog kills a 2h+ claude orphan; reaper finds the same PID and double-kills. SIGTERM on a dead PID is `ProcessLookupError`.
**Mitigation:** Both code paths swallow `ProcessLookupError` already (watchdog at line 285, reaper template at PR #1236's pattern). Test: mock `os.kill` to raise `ProcessLookupError`, assert reaper logs DEBUG and continues, returns count of 0 for that PID.

### Risk 4: PID reuse during the SIGKILL grace window
**Impact:** PID 23031 was an orphan claude; SIGTERM at tick T; by tick T+1h, macOS recycled PID 23031 to a brand-new process. Reaper SIGKILLs the wrong process.
**Mitigation:** Single-shot drain (snapshot+clear at top of next tick, exactly like PR #1236's `_pending_sigkill`). Use `psutil.Process(pid)` (PID-reuse-safe wrapper) instead of `os.kill`. Combined: a stale PID lives at most one tick in the SIGKILL set, AND psutil checks `create_time` on every operation. Two layers of defense.

## Race Conditions

### Race 1: Worker restarts (launchd respawn) mid-reflection
**Location:** `agent/session_health.py:cleanup_corrupted_agent_sessions` (calls the new reaper at the end)
**Trigger:** worker A is running the reflection loop; it crashes; launchd starts worker B with a new PID; worker B begins startup recovery (which calls `_cleanup_orphaned_claude_processes` at the existing line 296). Worker A's claude descendants may temporarily have PPID == launchd_pid (1) AND new worker B is not yet aware of them.
**Data prerequisite:** worker B's startup must run BEFORE worker B's first reflection tick — that's a 1-hour buffer, plenty of time. The existing startup call still fires (we keep it as a deprecation shim). No race in practice.
**State prerequisite:** `_pending_sigkill_orphans` is module-level state. When worker A dies, that state is lost — worker B starts with an empty set, no stale PIDs leak across restarts. This is correct.
**Mitigation:** Module-level state reset on worker startup. Document the invariant in the docstring.

### Race 2: AgentSession heartbeat updates concurrently with reaper scan
**Location:** Heartbeat is written by `_write_worker_heartbeat` (`agent/session_health.py`), reaper reads `AgentSession.query.filter(status="running")`.
**Trigger:** Reaper reads heartbeat, decides "stale (35min old), reap." Heartbeat updates 1ms later to fresh. Reaper proceeds with kill.
**Data prerequisite:** None — read-then-decide is the only safe pattern; locking would be overkill for a 1h-cadence reflection.
**State prerequisite:** A 30-minute threshold makes the read-write race practically irrelevant. A heartbeat that updates exactly at the 30-min boundary indicates the session was nearly dead anyway.
**Mitigation:** Document that the heartbeat gate is best-effort. The two-tick SIGTERM→SIGKILL escalation gives the session 60 minutes (one full reflection interval) to write a fresh heartbeat between the two tick steps; if it does, the SIGKILL drain at the top of the next tick still fires (drain is unconditional), but `psutil.Process(pid)` will raise `NoSuchProcess` on the original PID if the process actually self-recovered, and the safety holds. Re-checking heartbeat between SIGTERM and SIGKILL is over-engineering for the appetite.

## Open Questions

1. **Heartbeat gate granularity.** The conservative gate ("any running session with fresh heartbeat → skip ALL claude/MCP candidates") will leave orphans alive on systems running back-to-back sessions where the heartbeat is always fresh somewhere. Acceptable for v1, or do we need `claude_pid` on AgentSession (Medium-appetite expansion) from the start? Recommendation: ship v1 with the conservative gate and a metric; if the counter shows orphans accumulating between ticks, follow up with a `claude_pid` field issue.

2. **Worker-orphan reaping bypasses heartbeat gate.** A worker process can have PPID==1 only if its bridge died AND launchd respawn hasn't fired yet (rare and brief, ~seconds). Reaping it unconditionally is safe because launchd will re-respawn another. Confirm this is the intended bypass or whether worker-orphans should also gate on a per-worker heartbeat in Redis (`worker:{worker_key}:heartbeat`, written by `_write_worker_heartbeat`).

3. **Reflection entry: extend `agent-session-cleanup` vs. new `orphan-process-reap` entry.** Calling the reaper from inside `cleanup_corrupted_agent_sessions` keeps the reflection registry tidy (1 entry, 2 jobs) but couples failures (corrupt-record cleanup wrapped in try/except still inside the reflection callable). Splitting into a sibling entry in `config/reflections.yaml` gives independent scheduling. Recommendation: keep them in one entry — the appetite is small and the docstring already describes "extend `agent-session-cleanup`."

4. **Return-value contract.** `cleanup_corrupted_agent_sessions()` currently returns `int` (count of corrupted sessions). If we add the orphan count, do we return a sum (loses fidelity), a tuple (breaking change), or keep the int and emit the orphan count via metric only? Recommendation: keep the int unchanged, the orphan count is observable via the Redis counter and reflection-findings logging.
