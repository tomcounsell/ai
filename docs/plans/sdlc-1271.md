# Cleanup reflection: reap orphan claude/worker/MCP processes whose parent died

## Problem

When the bridge or worker dies ungracefully (panic, force-kill, SIGKILL, restart-without-graceful-shutdown), the child `claude` CLI process and its grandchildren (4+ MCP server subprocesses) become orphans: their PPID flips to 1 (launchd) and they keep running indefinitely. They hold file handles, consume RAM/CPU, and — for `claude` specifically — keep an Anthropic API session warm long after the parent that knew about it has gone.

**Current behavior:**

`agent/session_health.py` exposes `_cleanup_orphaned_claude_processes()` (lines 1722–1801) which:
- Is **only invoked once at worker startup** in `worker/__main__.py:296`. It is **not** wired into any reflection — the `agent-session-cleanup` reflection (`config/reflections.yaml:37`) calls `cleanup_corrupted_agent_sessions()` only.
- Matches **only** `claude_agent_sdk/_bundled/claude` (a single pgrep pattern).
- Does not touch worker (`python -m worker`) processes or MCP server subprocesses (`mcp_servers/*`).
- Does not walk the descendant tree — when it kills a `claude` PID, the MCP children attached to that `claude` are not reaped explicitly. They become orphans-of-orphans (PPID flips again to 1) and persist until the next worker startup.
- Has **no liveness gate**: it kills any matched process whose PPID==1, regardless of whether `AgentSession.last_heartbeat_at` says the corresponding session is healthy. (In practice this is safe today because it only runs at startup before any new sessions exist, but if we move it to a reflection it stops being safe.)
- Emits no metric — no count surfaced in the dashboard, no analytics rollup signal.

Between worker restarts (which can be hours or days apart), orphans accumulate. The fazm project documented `20+ orphan ACP bridges by Apr 30 2026` from the same failure mode. The Claude CLI itself has [Anthropic issue #51860](https://github.com/anthropics/claude-code/issues/51860) tracking orphaned `claude-code` CLI pinning 100% CPU after non-graceful parent exit — direct reinforcement of this issue's blast radius.

The current cleanup function leaks three classes of orphan, but only **two are addressable here** — the third is descoped (see "Desired outcome" rationale):

1. **Worker-orphan claude (in scope):** worker dies → `claude` reparented to PID 1 → caught at next startup only.
2. **Bridge-orphan worker (DESCOPED):** bridge dies but launchd `KeepAlive=true` respawns a new bridge → old worker (started by the dead bridge) is reparented to PID 1. Reaping these via cmdline match is unsafe because every live launchd-respawned worker also has PPID==1; see Blocker 1 in the critique table. Out of scope for this plan.
3. **Claude-orphan MCP subprocess (in scope):** `claude` killed cleanly → MCP child processes were children of `claude`, now reparented to PID 1 → never reaped.

**Desired outcome:**

A centralized reaper, run hourly from the existing reflection schedule, detects orphan `claude` and MCP-server processes whose PPID==1 AND whose owning AgentSession has no fresh heartbeat (gated per-PID via a new `claude_pid` field on AgentSession), terminates them with their full descendant tree (SIGTERM + SIGKILL escalation), and emits a count to the dashboard / analytics. No per-subprocess watchdog code is added.

**Worker process reaping is explicitly excluded from this plan.** On macOS `/sbin/launchd` IS PID 1, and `com.valor.worker.plist` runs the worker under launchd with `KeepAlive=true`. Every live, healthy worker has `ppid == 1` by design — a worker-signature + PPID==1 filter would match every running worker every reflection tick. The cmdline regex set therefore matches only `claude_agent_sdk/_bundled/claude` and `mcp_servers/*`, never the worker. As a positive-ID self-protection (defense-in-depth against future regex changes), the worker writes its own PID to a Redis key at startup and the reaper builds a skip-set from `worker:registered_pid:*` keys before scanning. Stranded sibling workers are out of scope: launchd's `KeepAlive` policy already restarts a crashed worker, and reaping a stranded-but-running worker creates higher blast-radius risk than the failure mode it would address.

## Research

**Queries used:**
- `psutil reap orphan child processes PPID 1 reparented descendant tree SIGTERM SIGKILL pattern 2026`

**Key findings:**
- **psutil.Process pattern** ([psutil docs](https://psutil.readthedocs.io/)) — `proc.terminate()` (SIGTERM with PID-reuse check) → `proc.wait(timeout=N)` → `proc.kill()` (SIGKILL with PID-reuse check). PID-reuse protection inside psutil is the differentiator vs. raw `os.kill()`; macOS recycles PIDs in ~5 minutes. **Informs:** prefer `psutil.Process(pid).terminate()/.kill()` over `os.kill(pid, signal.SIGTERM)` for safety against PID reuse. Or, if staying on raw `os.kill`, follow PR #1236's pattern of single-shot drain (snapshot then clear) so a stale PID never lives across more than one tick.
- **Descendant tree reaping pattern** (Ray's Raylet, [Ray docs](https://docs.ray.io/en/latest/ray-core/user-spawn-processes.html)) — recursively walk `proc.children(recursive=True)` and SIGKILL each, step-by-step as parents die. Confirms the issue's preference to walk the descendant tree explicitly rather than trusting cascaded SIGTERM (MCP servers don't propagate signals to their own children reliably).
- **Anthropic claude-code issue #51860** ([anthropics/claude-code#51860](https://github.com/anthropics/claude-code/issues/51860)) — orphaned `claude-code` CLI from non-graceful parent exit pins 100% CPU indefinitely. Independently confirms claude CLI is a pathological orphan candidate, not just RAM/handle-bound.

## Data Flow

1. **Entry point:** Reflection scheduler tick fires `agent-session-cleanup` (`config/reflections.yaml:37`, 1h interval) → calls `cleanup_corrupted_agent_sessions()` (`agent/agent_session_queue.py:86` re-export, real impl `agent/session_health.py:1607`).
2. **Existing pass (unchanged):** Filter Popoto descriptor phantoms → iterate AgentSession records → delete corrupted → `repair_indexes()`.
3. **NEW pass appended:** Call `_reap_orphan_session_processes()` (renamed/extended from `_cleanup_orphaned_claude_processes`):
   - Use `psutil.process_iter(['pid', 'ppid', 'cmdline', 'create_time'])` to enumerate.
   - Match candidates by cmdline regex against TWO signatures: `claude_agent_sdk/_bundled/claude` and `mcp_servers/*` entry points. Worker signature is intentionally NOT matched (see "Desired outcome" rationale).
   - Filter to candidates with `ppid == 1`.
   - **Per-PID liveness gate (NEW — replaces aggregate gate):** look up the owning AgentSession via `AgentSession.find_by_claude_pid(pid)` (a new helper backed by `claude_pid = IntField(null=True, indexed=True)` on the model, written by the existing `on_sdk_started(pid)` callback at `agent/sdk_client.py:2482`). If an owning session exists AND `(now - session.last_heartbeat_at) < 30min`, **skip** the kill (the session is alive; the PPID==1 is a transient reparenting during a worker handover, not a true orphan). If no owning session is found, OR the owning session is in a terminal state, OR its heartbeat is stale, proceed to kill. **MCP servers** typically don't have a direct `claude_pid` mapping; their parent claude's death reparents them to PID 1 directly, so the lookup falls through to "no owning session, kill" — which is correct because their parent claude is by definition gone. As a tier-2 belt-and-braces, when an MCP-signature candidate has a `proc.parent()` whose PID matches an owning session's `claude_pid`, the reaper inherits that session's heartbeat decision.
   - **Positive-ID self-protection (NEW — replaces negative PPID heuristic):** at module load the running worker writes `r.set("worker:registered_pid:{worker_key}", os.getpid(), ex=86400)` (24h TTL, refreshed by the existing `_write_worker_heartbeat` loop). The reaper reads all `worker:registered_pid:*` keys and builds a skip-set. Combined with the explicit `os.getpid()` skip, this defends against reaping the running worker even if a future code change accidentally re-adds the worker signature to the regex set. This is a **positive** identifier ("registered PID present in Redis means do not kill"), not a negative heuristic.
   - For each surviving orphan: `psutil.Process(pid)` is constructed once (capturing `create_time` at construction for PID-reuse safety), descendants captured via `.children(recursive=True)` BEFORE termination, parent and descendants `.terminate()`d. All PIDs are staged as `(pid, create_time)` tuples on `_pending_sigkill_orphans: set[tuple[int, float]]` and drained at the top of the next tick — when reconstructing `psutil.Process(pid)` for SIGKILL, the reaper compares `proc.create_time() == staged_create_time` and skips if mismatched (PID was recycled).
4. **Metric write (two counters, scoped by ownership knowledge):**
   - When the owning AgentSession is known (`find_by_claude_pid` hit): increment `{project_key}:session-health:orphan_process_reaped` (project resolved from owning session — accurate attribution).
   - When the owning session is unknown (the common case for true orphans whose parent claude is gone): increment the hostname-scoped counter `session-health:orphan_process_reaped:{worker_hostname}` (via `socket.gethostname()`). Avoids false attribution to whichever project's reflection tick happened to scan the orphan.
5. **Logging:** `logger.info("[orphan-reap] Killed PID %d (cmd=%s, owning_session=%s, descendants=%d)", ...)`.
6. **Output:** `cleanup_corrupted_agent_sessions()` returns a structured dict `{"corrupted": int, "orphans": int}` (replaces the previous int-only return). The reflection scheduler in `agent/reflection_scheduler.py` consumes the dict and surfaces both counts in the `findings` list visible at `localhost:8500/dashboard.json` reflections panel and the analytics daily rollup. Backward compat: any caller that needs the previous int reads `result["corrupted"]` explicitly.

## Solution

### Key Elements

- **`_reap_orphan_session_processes()`**: New function in `agent/session_health.py`. Scans process table for claude/MCP signatures with PPID==1, applies per-PID heartbeat gate (via `claude_pid` lookup) and registered-PID self-protection skip-set, kills survivors with their descendant tree, returns count.
- **`claude_pid` on AgentSession**: New field `claude_pid = IntField(null=True, indexed=True)` on `AgentSession`. Set by the existing `on_sdk_started(pid)` callback at `agent/sdk_client.py:2482` via an executor closure that captures the AgentSession ID. Cleared on session terminal-state transitions (delete or status=completed/failed/killed) to avoid stale lookups. New helper `AgentSession.find_by_claude_pid(pid)` performs an indexed lookup via the existing Popoto query API.
- **Per-PID heartbeat gate**: Replaces the rejected aggregate gate. For each candidate PID, look up the owning session via `find_by_claude_pid`. If the session exists and `(now - last_heartbeat_at) < 30min`, skip; otherwise kill. The 30-minute threshold is 6× the health-check cadence (5min). For MCP candidates without a direct mapping, walk `proc.parent()` once to find a claude PID and inherit that session's gate decision.
- **Registered-PID self-protection**: Worker writes `worker:registered_pid:{worker_key} = os.getpid()` (TTL 24h) at module load AND on every heartbeat tick. The reaper reads all `worker:registered_pid:*` keys (via `popoto`/Redis client) and adds the integer values to a skip-set, plus the explicit `os.getpid()` add. No reaped PID is ever in this set.
- **Create-time-stamped SIGKILL drain**: `_pending_sigkill_orphans: set[tuple[int, float]]` (PID + create_time tuple). Drained at the START of every cleanup reflection tick: for each tuple, construct `psutil.Process(pid)`, compare `proc.create_time() == staged_create_time`, skip if mismatched (PID was recycled by macOS), else `proc.kill()`. Always clear the set after drain regardless of outcome. This is a hardened variant of PR #1236's pattern, adapted for the 1-hour cadence (vs PR #1236's 5-min cadence which fits within macOS's PID-recycle window).
- **Reflection wiring**: Append the call to `cleanup_corrupted_agent_sessions()` so the existing `agent-session-cleanup` reflection covers both DB-row corruption AND OS-process orphans. **Decision (resolves Concern 5):** `cleanup_corrupted_agent_sessions()` returns a structured dict `{"corrupted": int, "orphans": int}`. The reflection scheduler's findings projection consumes the dict and surfaces both counts on the dashboard.
- **Metric (two-counter scheme, resolves Concern 4)**:
  - When owning session is known via `claude_pid`: `{project_key}:session-health:orphan_process_reaped`.
  - When owning session is NOT known: `session-health:orphan_process_reaped:{worker_hostname}` (hostname-scoped, no project attribution).
- **Kill switch**: `DISABLE_ORPHAN_PROCESS_REAP=1` env var (parity with PR #1236's `DISABLE_ORPHAN_REAP`).

### Flow

Worker starts → writes `worker:registered_pid:{worker_key}` → reflection scheduler → 1h tick → `cleanup_corrupted_agent_sessions()` (existing logic) → `_reap_orphan_session_processes()` (new):

1. Drain `_pending_sigkill_orphans`: for each `(pid, create_time)`, verify create_time match before SIGKILL. Clear set.
2. Build `skip_pids` from all `worker:registered_pid:*` keys + `os.getpid()`.
3. `psutil.process_iter` with try/except per iteration.
4. For each candidate (PPID==1, signature matches claude or mcp_servers, pid not in `skip_pids`): look up `AgentSession.find_by_claude_pid(pid)`. If session is alive (heartbeat <30min), skip. Otherwise capture descendants via `proc.children(recursive=True)`, SIGTERM parent and descendants via `psutil.Process.terminate()`, stage all `(pid, create_time)` tuples, increment appropriate counter.
5. Return `len(parent_kills)`.

The wrapper `cleanup_corrupted_agent_sessions()` returns `{"corrupted": int, "orphans": int}` → reflection scheduler emits structured findings → dashboard surfaces both counts.

### Technical Approach

- **Process enumeration:** `psutil.process_iter(['pid', 'ppid', 'cmdline', 'create_time'])` with try/except per-iteration (NoSuchProcess, AccessDenied, ZombieProcess) — never crash the reflection on a single bad PID.
- **Signature matching:** join cmdline list with spaces, regex match against TWO patterns (compiled once at module load):
  - `r"claude_agent_sdk/_bundled/claude\b"` (Claude CLI)
  - `r"mcp_servers/\w+\.py\b"` (MCP server entry points)
  - **Worker pattern intentionally excluded** (Blocker 1 fix).
- **PPID==1 gate:** `proc.info['ppid'] == 1` — orphan signal.
- **Self-protection (positive ID):** `skip_pids = {os.getpid()} | {int(v) for k, v in r.scan_iter("worker:registered_pid:*")}`. The reaper never kills any PID in this set, even if its cmdline somehow matches a future regex addition. This is the structural fix for Blocker 1.
- **Per-PID heartbeat gate (positive lookup, replaces aggregate):** for each candidate PID, `session = AgentSession.find_by_claude_pid(pid)`. Decision matrix:
  - `session` is None → no record, kill.
  - `session.status` in {`completed`, `failed`, `killed`, `cancelled`} → terminal, kill.
  - `session.last_heartbeat_at is None` → never heartbeated, kill.
  - `(now - session.last_heartbeat_at) < 30min` → alive, skip.
  - Otherwise (heartbeat older than 30min) → stale, kill.
  For MCP-signature candidates, attempt the lookup with `proc.parent().pid` first (the parent is usually claude); if that resolves a session, inherit its heartbeat decision; else fall through to the standard rules above.
- **Descendant walk:** `psutil.Process(pid).children(recursive=True)` BEFORE killing the parent — capture the tree first, then terminate parent, then iterate children. SIGTERM each, stage `(pid, create_time)` tuples on `_pending_sigkill_orphans`.
- **Two-tick escalation with create-time verification (Concern 3 fix):** at the top of each tick, snapshot the staged set, clear it, and for each `(pid, create_time)`: try to construct `psutil.Process(pid)`, compare `proc.create_time() == staged_create_time` (within `1e-3` epsilon for floating-point safety). If they match, `proc.kill()`. If they don't (PID recycled), log DEBUG and skip — do NOT kill the new PID. `psutil.NoSuchProcess` (the original process already exited) is also benign and skipped.
- **psutil over raw os.kill:** use `psutil.Process(pid).terminate()` (PID-reuse safe at construction time) on the parent and root descendants. Inside the descendant walk, also use `psutil.Process.kill()` for SIGKILL escalation. This is a deliberate upgrade from the existing `_cleanup_orphaned_claude_processes` (which uses raw `os.kill` and a 0.5s sleep loop).

## Failure Path Test Strategy

### Exception Handling Coverage
- [ ] `psutil.NoSuchProcess`, `psutil.AccessDenied`, `psutil.ZombieProcess` exceptions per-iteration: each must be logged at DEBUG and `continue` the loop. Test asserts that one bad PID does not abort the scan.
- [ ] `psutil.Process(pid).children(recursive=True)` raises if parent dies mid-walk: catch and continue with what we captured.
- [ ] Top-level `try/except Exception` wrapping the whole reaper: log at WARNING and return 0. Test asserts that a `psutil.process_iter` failure does not crash the reflection.

### Empty/Invalid Input Handling
- [ ] Empty process table (no orphans): function returns 0, no kills, no log spam.
- [ ] No AgentSession records at all (fresh install): heartbeat gate degrades safely. Decision: when `AgentSession.query.filter(status="running")` returns empty, treat as "no session is alive" and proceed with reaping (matches startup-only behavior of existing function).
- [ ] `cmdline` is None or empty list: skip without matching.

### Error State Rendering
- [ ] No user-visible UI for this — it's a reflection. The dashboard shows the count via `localhost:8500/dashboard.json` reflections panel; if the count is wrong (e.g., a kill failed silently), the test must catch it. Add an assertion that the returned count equals the number of PIDs that observably exited within 5 seconds of SIGTERM.

## Rabbit Holes

- **Per-subprocess PPID watchdog (fazm pattern):** Tempting because it reacts in 5s instead of 60min, but distributes complexity into every spawn path (`agent/sdk_client.py`, every MCP server, `worker/__main__.py`). The user explicitly chose centralized in the issue. Defense-in-depth could be added later as a separate issue. **Do not pursue here.**
- **Cross-machine reaping:** Each machine only has visibility into its own process table. Do not attempt to reap orphans on other machines via Redis coordination.
- **Replacing `monitoring/bridge_watchdog.py`'s zombie killer:** That's a separate concern (etime-based, watchdog-scoped). Leave it alone — it's a defense-in-depth layer that runs every 60s and won't conflict.
- **Reaping orphan workers via cmdline match:** Rejected because every live worker has PPID==1 under launchd `KeepAlive=true` (Blocker 1). Stranded sibling workers are out of scope; launchd's restart policy already handles crashed workers. If a future failure mode requires reaping stranded workers, do it via a registered-PID *negative* check (worker registered N min ago, no heartbeat update since) — not via cmdline match — and only in a separate issue.
- **Process ownership via `/tmp/`-style PID files:** The fazm `__watchdogStartPpid` pattern uses `/tmp/fazm-dev-status`. The Redis-backed registered-PID key (`worker:registered_pid:{worker_key}`) and the AgentSession `claude_pid` field are our equivalent — adding a parallel filesystem registry would duplicate state.

## Risks

### Risk 1: Killing a healthy `claude` process whose owning session has a stale heartbeat (false positive)
**Impact:** A long-running but legitimately-alive Claude session is SIGTERMed mid-API-call. The user sees a session abruptly end with no clear cause.
**Mitigation:** The 30-minute heartbeat threshold is 6× the health-check cadence (5min). With per-PID gating via `claude_pid`, the reaper kills only PIDs whose owning session is provably stale or terminal — no aggregate gate that depends on machine activity. `DISABLE_ORPHAN_PROCESS_REAP=1` kill switch lets ops disable while investigating. **Acceptance:** the per-project counter is the canonical signal; the hostname-scoped counter captures "true unowned orphans" — together they should add to a small steady-state number on a healthy machine.

### Risk 2: Killing the actively-running parent worker that's hosting this reflection (suicide)
**Impact:** Worker dies mid-reflection, launchd respawns it, every running session is recovered as `interrupted`. Cascading session loss.
**Mitigation (Blocker 1 fix):** Two layers.
- Layer 1 (regex exclusion): the cmdline regex set does NOT include the `python -m worker` pattern, so the worker is never even a candidate for reaping.
- Layer 2 (positive-ID skip-set): even if a future code change re-adds the worker pattern, the reaper builds a `skip_pids` set from `os.getpid()` plus all `worker:registered_pid:*` Redis keys before scanning. Any PID in that set is skipped unconditionally. The worker writes its registered PID at module load AND refreshes it on every heartbeat tick, with a 24h TTL — so even a wedged worker that hasn't heartbeated in 30min is still in the skip-set as long as TTL hasn't expired.
- Test (replaces old test #4 per Concern 6): spawn a registered worker (PID written to Redis), simulate it reparenting to PPID==1 (e.g., by mocking `psutil.Process.ppid()` to return 1) AND adding the worker pattern to the regex set, run the reaper, assert the registered worker survives.

### Risk 3: Double-kill race with `monitoring/bridge_watchdog.py`
**Impact:** Watchdog kills a 2h+ claude orphan; reaper finds the same PID and double-kills. SIGTERM on a dead PID is `ProcessLookupError`.
**Mitigation:** Both code paths swallow `ProcessLookupError` already (watchdog at line 285, reaper template at PR #1236's pattern). The `psutil.Process` wrapper raises `psutil.NoSuchProcess` instead of `ProcessLookupError`; the reaper catches both. Test: mock `psutil.Process.terminate` to raise `psutil.NoSuchProcess`, assert reaper logs DEBUG and continues, returns count of 0 for that PID.

### Risk 4: PID reuse during the SIGKILL grace window
**Impact:** PID 23031 was an orphan claude; SIGTERM at tick T; by tick T+1h, macOS recycled PID 23031 to a brand-new process. Reaper SIGKILLs the wrong process.
**Mitigation (Concern 3 fix):** Three layers.
- Layer 1: stage `(pid, create_time)` tuples instead of bare ints. At drain time, construct `psutil.Process(pid)` and compare `proc.create_time() == staged_create_time`. If they differ (PID was recycled), skip the SIGKILL — do NOT kill the new process at that PID.
- Layer 2: clear the staged set unconditionally after drain — a PID never lives across more than one tick.
- Layer 3: psutil's own internal PID-reuse check (it caches `_create_time` at construction and rechecks on signal calls) provides an additional guard if create_time mismatch is somehow within psutil's epsilon.

## Race Conditions

### Race 1: Worker restarts (launchd respawn) mid-reflection
**Location:** `agent/session_health.py:cleanup_corrupted_agent_sessions` (calls the new reaper at the end)
**Trigger:** worker A is running the reflection loop; it crashes; launchd starts worker B with a new PID; worker B begins startup recovery (which calls `_cleanup_orphaned_claude_processes` at the existing line 296). Worker A's claude descendants may temporarily have PPID == launchd_pid (1) AND new worker B is not yet aware of them.
**Data prerequisite:** worker B's startup must run BEFORE worker B's first reflection tick — that's a 1-hour buffer, plenty of time. The existing startup call still fires (we keep it as a deprecation shim). No race in practice.
**State prerequisite:** `_pending_sigkill_orphans` is module-level state. When worker A dies, that state is lost — worker B starts with an empty set, no stale PIDs leak across restarts. This is correct.
**Mitigation:** Module-level state reset on worker startup. Document the invariant in the docstring.

### Race 2: AgentSession heartbeat updates concurrently with reaper scan
**Location:** Heartbeat is written by `_write_worker_heartbeat` (`agent/session_health.py`), reaper reads `AgentSession.query.filter(status="running")`.
**Trigger:** Reaper reads heartbeat, decides "stale (35min old), reap." Heartbeat updates 1ms later to fresh. Reaper proceeds with kill.
**Data prerequisite:** None — read-then-decide is the only safe pattern; locking would be overkill for a 1h-cadence reflection.
**State prerequisite:** A 30-minute threshold makes the read-write race practically irrelevant. A heartbeat that updates exactly at the 30-min boundary indicates the session was nearly dead anyway.
**Mitigation:** Document that the heartbeat gate is best-effort. The two-tick SIGTERM→SIGKILL escalation gives the session 60 minutes (one full reflection interval) to write a fresh heartbeat between the two tick steps; if it does, the SIGKILL drain at the top of the next tick still fires (drain is unconditional), but `psutil.Process(pid)` will raise `NoSuchProcess` on the original PID if the process actually self-recovered, and the safety holds. Re-checking heartbeat between SIGTERM and SIGKILL is over-engineering for the appetite.
