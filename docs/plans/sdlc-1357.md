# Worktree-cleanup race: refuse busy removal + cwd-vanished watchdog

## Problem

When a PM (or dev) `AgentSession` is running with its cwd inside `.worktrees/{slug}/`, **another process** can call `remove_worktree(slug)` (e.g. `/do-merge` running `python scripts/post_merge_cleanup.py {slug}` after a sibling PR merged) and successfully delete the directory tree. On macOS the kernel does not signal the running subprocess: `getcwd(3)` returns ENOENT, the SDK harness hangs in `await proc.communicate()` forever, and the AgentSession row sits at `status=running` for hours while the tier-1 stdout-staleness reaper sees a still-alive heartbeat thread.

This was the trigger half of investigation #1246 — the wedge that blocked all PM work on `worker_key=valor` for 10.9 hours and required a manual `valor-session kill`.

**Current behavior:**

- `remove_worktree(repo_root, slug)` (`agent/worktree_manager.py:493-559`) calls `git worktree remove --force` unconditionally. PR #304 added a guard for the *calling* process's cwd (lines 519-534) but does nothing about *background SDK subprocesses* spawned with that cwd.
- `cleanup_after_merge` (`agent/worktree_manager.py:820-903`) and `scripts/post_merge_cleanup.py` (lines 31-88) inherit the same blind spot.
- `BackgroundTask._watchdog` (`agent/messenger.py:356-389`) ticks every 60s, logs `SDK heartbeat`, and never checks whether `working_dir` still exists on disk. The heartbeat keeps writing while the SDK subprocess is silently wedged on a dead cwd.

**Desired outcome:**

- `remove_worktree(slug)` refuses to delete the directory when any non-terminal `AgentSession.working_dir` references the worktree, returning `("blocked", session_id)` and a WARNING log. `force=True` overrides with a louder WARNING. Callers (`cleanup_after_merge`, `post_merge_cleanup.py`) propagate the block and exit non-zero with offending IDs.
- `BackgroundTask._watchdog` checks `os.path.isdir(working_dir)` each tick. On disappearance, logs `cwd_vanished session_id=...`, increments `{project_key}:session-health:cwd_vanished`, cancels the work task, and the existing `CancelledError` handler at lines 266-325 sends "I was interrupted" and finalizes cleanup.
- A new integration test asserts watchdog cancels a sleeping subprocess within ~75s of worktree removal.

## Prior Art

- **PR #304 / #301** — "Prevent shell CWD death when worktree is removed". Added the `if cwd is inside worktree, chdir to repo_root` guard in `remove_worktree` (lines 519-534). Protects the calling process only; does not protect the SDK subprocess that was spawned earlier with the same cwd.
- **PR #1236 / #1218** — "Orphan subprocess reap pass in session health check". Catches the *inverse* failure mode: row already terminal, subprocess still alive. This issue catches: row still running, cwd vanished underneath. Complementary, not overlapping.
- **PR #882 / #880** — "Add path-containment guard to `_cleanup_stale_worktree`". Different defense layer (pre-create cleanup), not the runtime busy guard.
- **`dcab49b1`** — "fix(session-executor): require worktree to exist on disk, not just in path string". Catches a missing worktree at session *entry*; this issue catches it *mid-run*.
- **#1287** — sibling investigation, concluded misdiagnosed; the `dcab49b1` guard already covers its named surface (different mechanism: local Claude Code `git checkout` in main checkout, not worktree GC race).

## Why Previous Fixes Failed

| Prior Fix | What It Did | Why It Failed / Was Incomplete |
|-----------|-------------|--------------------------------|
| PR #304 (#301) | `chdir` to repo_root before removing the worktree | Protects only the *calling* process. A long-running SDK subprocess (the actual session) was spawned earlier with `cwd=.worktrees/{slug}` and continues to hold that cwd vnode after the chdir. The kernel-level vnode is unlinked-but-referenced; the SDK keeps "running" but `getcwd(3)` returns ENOENT. |
| `dcab49b1` | Verify `working_dir.exists()` at session entry | Catches sessions that *start* with a missing worktree. Does nothing for sessions that *lose* their worktree mid-run. |
| PR #1236 (#1218) | Reap orphan SDK subprocess when row is terminal | Solves the case `row=terminal, subprocess=alive`. The current bug is the dual: `row=running, cwd=gone, subprocess=wedged`. Orthogonal. |

**Root cause pattern:** Each prior fix protected *one direction* of the row↔subprocess↔filesystem triangle. Nothing has yet protected the **filesystem→subprocess** direction at runtime. This plan adds two complementary guards: (a) the *source* (refuse-busy at `remove_worktree`) prevents the bad delete from happening; (b) the *detector* (cwd-vanished watchdog) catches it mid-run if it does happen anyway via a different path (manual `rm -rf`, OS-level cleanup, recovery scripts).

## Solution

### Key Elements

- **`worktree_busy_check`** (new helper in `agent/worktree_manager.py`): given `repo_root` and `slug`, query `AgentSession` records whose `working_dir` contains `.worktrees/{slug}` and `status not in TERMINAL_STATUSES`. Return the offending session id (`session_id`) and the agent_session_id, or `None` if clear.
- **`remove_worktree` busy guard**: at the top of the function (after slug validation, before the existing CWD-death chdir), call `worktree_busy_check`. If it returns a session, return `("blocked", session_id)` unless `force=True`. With `force=True`, log a WARNING `"force-removing worktree .worktrees/{slug} despite live session_id={session_id}"` and proceed.
- **`cleanup_after_merge` propagation**: when `remove_worktree` returns the `("blocked", ...)` tuple, surface that into the result dict (`result["blocked_by_session"] = session_id`, `result["worktree_removed"] = False`, append a `blocked: ...` line to `result["errors"]`).
- **`post_merge_cleanup.py` exit code**: when `result["blocked_by_session"]` is set, print `Error: worktree .worktrees/{slug} is in use by session_id={session_id} agent_session_id={agent_session_id}` to stderr and exit 2 (distinct from the existing exit 1 for generic errors). Exit 0 unchanged for successful or already-clean paths.
- **`BackgroundTask` cwd watchdog**: thread `working_dir: str | None` through the constructor (defaults to None for backward compat). In `_watchdog`, before each heartbeat sleep, check `working_dir is not None and not os.path.isdir(working_dir)`. On disappearance: log `[{session_id}] cwd_vanished session_id={session_id} working_dir={working_dir}`, increment Redis `{project_key}:session-health:cwd_vanished`, cancel `self._task`, and break the watchdog loop. The existing CancelledError handler in `_run_work` (lines 266-325) handles the rest.
- **`session_executor.py` integration**: at line 1693 (`task = BackgroundTask(messenger=messenger)`), pass `working_dir=session.working_dir` (the value already used to spawn the SDK subprocess earlier in the function).

### Flow

PM session running with `cwd=.worktrees/sdlc-1218/` → `/do-merge` runs `post_merge_cleanup.py sdlc-1218` for a sibling-merged PR → `cleanup_after_merge` → `remove_worktree` invokes `worktree_busy_check` → returns `("blocked", "0_1777566138117")` → cleanup_after_merge surfaces `blocked_by_session` → script prints session id and exits 2 → `/do-merge` operator sees the block and runs `valor-session kill --id 90c4117dbf06431c86ee4807d7a18bcd` first, then re-runs cleanup.

Independent failure path: someone manually does `rm -rf .worktrees/sdlc-1218/` while the session is live → `BackgroundTask._watchdog` ticks at the next 60s boundary → `os.path.isdir` returns False → log `cwd_vanished` → increment counter → cancel work task → `_run_work` CancelledError handler sends "I was interrupted and will resume automatically" → row flips to `failed` (or whatever the cancellation handler sets) → tier-1 reaper or recovery picks up.

### Technical Approach

- `worktree_busy_check(repo_root: Path, slug: str) -> tuple[str, str] | None` — lazy import of `models.agent_session.AgentSession` and `models.session_lifecycle.TERMINAL_STATUSES` to keep `worktree_manager.py`'s import-time graph unchanged. Implementation uses `AgentSession.query.all()` filtered with a Python comprehension on `working_dir` containment + status; no new Redis index. Plan validates per Section "Race Conditions" that this is acceptable scale (typical AgentSession count: tens, not thousands).
- `remove_worktree` keeps the existing `bool` return for the False/True paths; the new `("blocked", session_id)` path is the only union extension. Type annotation: `bool | tuple[Literal["blocked"], str]`.
- The `force=True` override is required for legitimate ops — e.g., `valor-session kill` followed by manual `worktree remove`, or a recovery script that has already verified the session is dead but the row hasn't flipped yet.
- `BackgroundTask`'s `working_dir` parameter is optional. **Must default to None** so callers we don't update (tests, future code) don't break.
- For the Redis counter: use `{project_key}:session-health:cwd_vanished` (project_key resolved via `messenger.session_id` → `AgentSession.query.get(session_id).project_key`). Fall back to bare `session-health:cwd_vanished` if the project_key lookup fails (shouldn't happen, but defensive — same fallback shape `agent/session_health.py:2106-2116` already uses for orphan reaps).
- Integration test uses `tempfile.TemporaryDirectory` + `subprocess.Popen([sys.executable, "-c", "import time; time.sleep(3600)"])` with `cwd=temp_worktree`. Spawn it inside an asyncio task that imitates `BackgroundTask` (or instantiates `BackgroundTask` directly with a mocked messenger). After ~5s of running, `shutil.rmtree(temp_worktree)`. Assert the task is cancelled within 75s. Watchdog interval will be temporarily monkeypatched to 5s for this test so we don't wait 60s + 60s = 120s in CI.

## Rabbit Holes

- **Generalizing `working_dir` containment to `os.path.commonpath` / resolved-symlink awareness.** Tempting because it would catch edge cases (symlinked worktrees, sub-worktrees-of-worktrees). But `_validate_slug` already rejects path traversal, and the `.worktrees/{slug}` substring match is what the current cleanup code already assumes. Don't widen the matching contract here.
- **Replacing the busy guard with a session-side lockfile.** Conceptually cleaner, but introduces a new failure mode (lockfile-not-cleaned-up after crash). The Popoto-row-based check has a single source of truth (`AgentSession.status`) that is already maintained by the worker.
- **Refactoring the heartbeat loop.** The watchdog already exists; we are adding ~5 lines. Resist the urge to "while we're here" extract a CwdMonitor class.
- **Auditing every other process that consumes cwd.** Out of scope per the issue. The watchdog covers them centrally.
- **Adding a "graceful kill" path that gives the SDK subprocess a chance to flush.** The existing `CancelledError` handler is the graceful kill; cancelling the asyncio task is the right primitive.

## Risks

### Risk 1: Popoto query in `worktree_busy_check` is slow on a busy worker
**Impact:** `remove_worktree` takes longer for every cleanup, including the common no-live-session path. If the AgentSession table has thousands of rows, the in-memory comprehension could add hundreds of milliseconds per call.
**Mitigation:** AgentSession count in production is tens, not thousands (worker reaps terminal sessions). If profiling shows >100ms median, swap the comprehension for a Redis SET membership index keyed by `worktree:{repo_root}:{slug}` populated by `AgentSession.save()` hooks. Don't pre-optimize.

### Risk 2: Stale `working_dir` field on a terminal session falsely refuses removal
**Impact:** A row that flipped to `complete` but still has `working_dir=.worktrees/sdlc-1218/` would not block (correct — `complete` is in `TERMINAL_STATUSES`). But if a bug somewhere leaves a row at `running` after the actual subprocess died, the busy guard will refuse forever and require `--force`.
**Mitigation:** This is the intended behavior. The busy guard is conservative on purpose — we'd rather refuse than delete a live worktree. The escape hatch is `force=True` (programmatic) or `--force` flag on `post_merge_cleanup.py` (operator). The `cwd_vanished` watchdog separately cancels the wedged session, which flips its row to a terminal state, which un-blocks future removal. The two layers are independent.

### Risk 3: Integration test flake on slow CI
**Impact:** The "watchdog cancels within 75s" assertion could flake under heavy CI load.
**Mitigation:** Monkeypatch `BackgroundTask`'s heartbeat_interval (currently hardcoded `60` at line 369) to 1s for the test. Assert cancellation within 10s. The test's job is to prove the *behavior*, not the production timing.

### Risk 4: `force=True` becomes the default after one operator gets blocked
**Impact:** Operators who hit the busy guard once may copy-paste `--force` into every future invocation, defeating the protection.
**Mitigation:** The WARNING log on `force=True` is grep-able. Document in `docs/sdlc/do-merge.md` that `--force` is for known-dead sessions only; the correct response to a busy guard is to investigate the session, kill it if dead, and re-run without `--force`. Add an entry to `docs/runbooks/` if recurring (out of scope here, file as follow-up if pattern appears).

### Risk 5: `working_dir` field is sometimes a path-like that doesn't contain `.worktrees/{slug}` literally
**Impact:** Sessions whose `working_dir` is set to a resolved/canonicalized path (e.g., via symlink) might evade the substring match.
**Mitigation:** All current callers pass paths produced by `get_or_create_worktree`, which builds them as `repo_root / WORKTREES_DIR / slug` and stringifies. We will normalize both sides via `os.path.normpath` before comparison, but we will NOT call `os.path.realpath` (that resolves symlinks and could amplify the match into unrelated dirs). Unit test covers `working_dir = ".worktrees/sdlc-1218"` and `working_dir = "/abs/path/to/.worktrees/sdlc-1218"` and `working_dir = "/abs/path/to/.worktrees/sdlc-1218/subdir"` (all match) versus `working_dir = "/abs/path/to/.worktrees/sdlc-1218-other"` (substring near-miss; the helper uses `os.path.commonpath`-style segment matching, not raw `in`).

### Risk 6: Watchdog races with shutdown
**Impact:** During worker shutdown, the watchdog may detect cwd_vanished after the work task is already cancelling, double-cancelling.
**Mitigation:** `task.cancel()` is idempotent on an already-cancelled task. The CancelledError handler's Redis-dedup key (`interrupted-sent:{session_id}`, lines 287-303) ensures the user sees only one message. No additional code needed.

## Race Conditions

### Race 1: Session transitions to terminal between busy-check and `git worktree remove`
**Location:** `agent/worktree_manager.py:remove_worktree`, between the busy-check call and the `subprocess.run([git, worktree, remove])` call.
**Trigger:** `worktree_busy_check` returns None (no live session). Before `git worktree remove` runs, a *new* AgentSession is created with `working_dir` pointing at this worktree (e.g., concurrent worker race spawning a sibling session).
**Data prerequisite:** `AgentSession` records are durable in Redis via Popoto.
**State prerequisite:** No two processes are concurrently provisioning a session for the same slug.
**Mitigation:** This race is acceptable because (a) `get_or_create_worktree` is the only legitimate spawn path, and it creates the worktree if missing — a deletion-then-spawn race produces a *fresh* worktree; (b) the `BackgroundTask` cwd-vanished watchdog catches the case where a session's worktree disappears mid-run regardless of who deleted it. The two layers compose: source-side guard + sink-side watchdog. We do NOT introduce a lock — the cost (lock-leak failure modes) outweighs the benefit (preventing a race that is already detected by the watchdog).

### Race 2: `working_dir` is mutated mid-busy-check
**Location:** `worktree_busy_check`, while iterating `AgentSession.query.all()`.
**Trigger:** A session's `working_dir` is set after the iterator started but before its row is read.
**Data prerequisite:** Popoto queries are point-in-time snapshots from Redis (no isolation level above READ_COMMITTED).
**State prerequisite:** Another worker is spawning sessions concurrently.
**Mitigation:** Acceptable. Worst case: a brand-new session's row is missed → `remove_worktree` proceeds → watchdog catches it within 60s. The guard is best-effort, not transactional.

### Race 3: Watchdog observes `cwd_vanished` between `git worktree remove` start and end
**Location:** `BackgroundTask._watchdog`, during `git worktree remove --force` execution.
**Trigger:** `git worktree remove` deletes the directory tree. The watchdog ticks during the few-millisecond window before `subprocess.run` returns to the caller.
**Data prerequisite:** None.
**State prerequisite:** The session whose worktree is being removed was *not* the caller of `remove_worktree` (caller's cwd is already at `repo_root` per PR #304's guard).
**Mitigation:** This is the *correct* behavior. The watchdog detects the disappearance, cancels the work task, sends "I was interrupted." The race is the feature.
