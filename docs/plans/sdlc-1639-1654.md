# SDLC router stale-critique dead-end (#1639) + war-room never finalizes (#1654)

## Problem

Two confirmed reliability defects break unattended SDLC pipeline runs. They were observed together in the same run (#1633 / #1644-#1647) and compound: a pipeline that returns `NEEDS REVISION` from critique cannot make forward progress without a human.

**#1639 — Router stale-critique dead-end.** After a plan is revised in response to a plain `NEEDS REVISION` critique verdict, the SDLC router has no route back to re-critique. The router keeps matching the stale cached verdict text and re-dispatching `/do-plan` forever.

**#1654 — War-room never finalizes.** `/do-plan-critique` spawns six background critics but has no explicit step that waits for them and records the verdict + completion marker. The critique stage silently stalls at `in_progress` unless a supervisor aggregates the findings inline.

**Current behavior:**
- #1639: state `{PLAN=completed, CRITIQUE=in_progress, _verdicts.CRITIQUE.verdict="NEEDS REVISION" recorded_at T0, _sdlc_dispatches has /do-plan at T1>T0, meta.revision_applied=true}` → `decide_next_dispatch()` returns `Dispatch(/do-plan, row_id=3)`. Row 3 (`_rule_critique_needs_revision`) matches on cached verdict text alone — no staleness check — so every dispatch is a no-op `/do-plan` until guard G4 blocks after 3 identical dispatches and halts the supervision loop.
- #1654: `/do-plan-critique` Step 3 spawns 6 critics `run_in_background:true`; Step 4 ("After all critics complete") has no wait/collect gate, so Step 5.5 (`sdlc-tool verdict record`) and the completion stage-marker are not reliably reached. The skill relies on the supervisor aggregating inline — that reliance is the bug.

**Desired outcome:**
- #1639: a stale critique verdict (recorded before the latest `/do-plan` dispatch) routes to `/do-plan-critique` for a fresh critique, instead of dead-ending on `/do-plan`.
- #1654: the critique skill blocks on all six critics, aggregates, records the verdict, and writes the completion marker on every path — no supervisor intervention required.

## Solution

### Key Elements

- **`_critique_verdict_is_stale(stage_states)`** (new, `agent/sdlc_router.py`): returns True when the CRITIQUE verdict's `recorded_at` predates the latest `/do-plan` dispatch. Exact structural mirror of `_review_verdict_is_stale`, swapping `REVIEW`→`CRITIQUE` and `SKILL_DO_PATCH`→`SKILL_DO_PLAN`.
- **Row 3 step-aside**: `_rule_critique_needs_revision` returns False when the critique verdict is stale, so a settled (revised) plan no longer matches the dead-end rule.
- **New row `2b` (`_rule_critique_verdict_stale`)**: dispatches `/do-plan-critique` when the critique verdict is stale, regardless of the `in_progress` marker. Placed *before* row 3 in `DISPATCH_RULES` so stale → re-critique wins over the stale-text → /do-plan match. Direct mirror of REVIEW row 8b.
- **Critique skill wait-and-collect**: a new explicit step in `do-plan-critique/SKILL.md` that blocks on all six background critics and retrieves their findings before aggregation, plus tightened Step 4/5.5 so verdict-record + completion-marker writes are mandatory and self-contained.

### Flow

**Dead-end state (revised plan, stale NEEDS REVISION)** → router queries → row `2b` (`_critique_verdict_is_stale` True) → dispatch `/do-plan-critique` → fresh critique runs → records new verdict + completion marker → pipeline advances (build / further revision) — no more dead-end.

**Critique run** → spawn 6 background critics → **wait-and-collect (new)** blocks until all six return → aggregate + dedupe → record verdict (`sdlc-tool verdict record`) → write completion marker on READY TO BUILD (mandatory) → skill finalizes.

### Technical Approach

**Fix 1 — `agent/sdlc_router.py` (mirror PR #1657 exactly):**

1. Add `_critique_verdict_is_stale(stage_states: dict) -> bool` immediately after `_review_verdict_is_stale` (~L748). Body is the structural twin:
   - `verdict_dict = stage_states.get("_verdicts", {}).get("CRITIQUE", {})`; if not a dict → False.
   - `recorded_at = verdict_dict.get("recorded_at")`; if falsy → False.
   - `latest_plan_at = _latest_dispatch_at(stage_states, SKILL_DO_PLAN)`; if None → False.
   - parse both with `datetime.fromisoformat` inside `try/except Exception: return False`.
   - return `verdict_dt < plan_dt` (strict `<`, so equal timestamps are *not* stale).
2. In `_rule_critique_needs_revision` (row 3, ~L656), add `if _critique_verdict_is_stale(stage_states): return False` before the existing `CRITIQUE_NEEDS_REVISION in verdict` check — mirrors the `_review_verdict_is_stale` step-aside at L815.
3. Add `_rule_critique_verdict_stale(stage_states, meta, context) -> bool`: returns `_critique_verdict_is_stale(stage_states)` AND there is a non-empty critique verdict text (guard against firing on no verdict). It must NOT require a particular CRITIQUE marker value (the dead-end marker is `in_progress`).
4. Insert a new `DispatchRule(row_id="2b", state_predicate=_rule_critique_verdict_stale, skill=SKILL_DO_PLAN_CRITIQUE, reason="Critique verdict is stale (plan revised since) — re-critique")` into `DISPATCH_RULES` **between row 2 and row 3**.
5. Update `tests/unit/test_sdlc_skill_md_parity.py::test_dispatch_rules_cover_expected_row_ids` `expected` set (L132) to include `"2b"`.

**Guard mutual-exclusivity (G1 vs new row 2b/row 3):** G1 fires only when `last_dispatched_skill == /do-plan-critique` (the critique just ran; plan unchanged). The #1639 dead-end has `last_dispatched_skill == /do-plan` (plan just revised). Guards run before `DISPATCH_RULES`, so when G1's precondition holds it short-circuits to `/do-plan` and the new rules never run; when the dead-end holds, G1 returns None and row 2b fires `/do-plan-critique`. The two conditions are disjoint on `last_dispatched_skill`, so no oscillation. The build task must add a unit test asserting both orderings.

**Fix 2 — `.claude/skills-global/do-plan-critique/SKILL.md` (skill-layer):**

1. Insert **Step 3.5: Wait and Collect** between Step 3 (spawn) and Step 4 (aggregate): an explicit instruction that the skill MUST block on all six background critic agents and retrieve every critic's findings before proceeding. State plainly that the skill does not yield to the supervisor for aggregation — it owns finalization end to end.
2. Tighten Step 4 + Step 5.5: make "after collecting all six critics, you MUST record the verdict via `sdlc-tool verdict record` AND (on READY TO BUILD) write the completion stage-marker" an unconditional, self-contained block. The verdict record and marker write are reached on every exit path, not only when a supervisor nudges.
3. Reinforce the Stage Marker note (L18-23): the completion-marker write on READY TO BUILD is mandatory and co-located with the verdict record so the two cannot desync.

**#1642 REVIEW-marker symptom (verdict/marker hardening):** Same failure class — `tools/sdlc_verdict.py::record_verdict` writes only `_verdicts`, and `do-pr-review/SKILL.md` writes the completion marker separately (L186-190); if the review skill exits before reaching it, REVIEW desyncs and router row 9 stalls. The focused, in-scope fix is the same skill-layer tightening applied to `do-pr-review/SKILL.md`: make the verdict-record + completion-marker writes a single mandatory, self-contained block on the approval path (mirror of the critique fix). No tools-layer change to `record_verdict` is made on this branch (co-writing the marker inside `record_verdict` would couple two single-writer responsibilities and is explicitly rejected — see Rabbit Holes). If the skill-layer tightening proves insufficient in review, the residual is filed as a separate issue (see No-Gos).

## Rabbit Holes

- **Adding a tools-layer co-write of the stage marker inside `record_verdict`.** Tempting (it would make verdict+marker atomic at the data layer), but it couples two deliberately-separate single-writer responsibilities (`sdlc_verdict.py` owns `_verdicts`; `sdlc_stage_marker.py` owns markers) and risks marking a stage complete on a non-terminal verdict. The skill-layer fix (mandatory co-located writes) solves the observed bug without this coupling. Do NOT refactor `record_verdict`.
- **Generalizing staleness into a single parametrized helper for both REVIEW and CRITIQUE.** A DRY refactor of `_review_verdict_is_stale` + `_critique_verdict_is_stale` into one parametrized function is tempting but expands the blast radius into the already-shipped #1657 REVIEW path. Keep them as two parallel helpers — the duplication is intentional and low-cost.
- **Re-architecting the war-room to run critics synchronously.** The background-spawn design is fine; the bug is the missing wait gate, not the concurrency model. Do not rewrite the critic dispatch.
- **artifact_hash-based staleness (the issue's original suggestion).** Comparing the current plan hash against the verdict's `artifact_hash` is a heavier, more failure-prone signal than the timestamp comparison #1657 already proved. Use timestamp staleness only.

## Risks

### Risk 1: New row 2b causes a critique↔plan oscillation
**Impact:** If row 2b fired while G1's condition also held, the pipeline could ping-pong between `/do-plan-critique` and `/do-plan`.
**Mitigation:** G1 and row 2b are disjoint on `last_dispatched_skill` (critique vs plan), and guards run before dispatch rows. A dedicated mutual-exclusivity unit test locks this in. Guard G4 (3-identical-dispatch backstop) remains as a final safety net.

### Risk 2: Over-suppression — a genuinely fresh NEEDS REVISION gets re-critiqued instead of revised
**Impact:** A plan that legitimately needs revision could be sent to re-critique, wasting an opus run.
**Mitigation:** Strict `<` staleness means a verdict recorded *after* the latest `/do-plan` is fresh and still routes via row 3 to `/do-plan`. The fresh-verdict test asserts this. Staleness only fires when a `/do-plan` dispatch is recorded *after* the verdict — i.e., the plan was demonstrably revised.

## Race Conditions

No race conditions identified. The router decision is a synchronous, single-threaded pure function over a `stage_states` snapshot. The #1654 critic concurrency is the *subject* of the fix (adding a wait barrier), not a new race the router introduces — the wait-and-collect step is precisely the synchronization that removes the existing timing gap.
