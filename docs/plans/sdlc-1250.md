# Expand session-liveness-check to detect stalled SDLC PRs (state-level idle detection)

Tracks #1250.

## Problem

SDLC pipelines can stall at the **state level** without any in-process wedge being detectable
by the existing `session-liveness-check`. The PM dispatch worked, the dev session ran and
completed, then everything went quiet — no active session is in flight, but the PR also stopped
advancing. Nothing in the current system notices this condition.

**Current behavior:**

Two SDLC sessions (PR #1237 for issue #1197, PR #1236 for issue #1218) sat with their last
commits approximately 8 hours old while the originating PM session had completed its turn and
gone dormant. No reflection was watching for this state-level stall. The CEO had to ask "did
anything happen overnight?" to surface the stall.

| Layer | "Stuck" signal | Existing detector |
|---|---|---|
| Process | row=`running`, SDK silent | `session-liveness-check` Tier 1 (#1226, #1246) |
| State | PR open, no commits in N hours, no active session | **(this issue — no detector)** |

**Desired outcome:**

A new reflection callable (`sdlc-progress-check`) runs every 30 minutes, detects SDLC slugs
whose PRs are open, whose branch commits are stale (no new commit in N hours), and for which
no AgentSession is currently `pending` or `running`. When all gates pass, it sends a single
deduplicated Telegram alert to the PM channel so the PM can resume or triage.

## Data Flow

1. **Entry point**: `ReflectionScheduler` fires `sdlc-progress-check` every 1800s (30 min).
2. **`run_sdlc_progress_check()`**: Calls `run_per_project_audit(_check_project_stalls, name="sdlc-progress-check")`.
3. **`_check_project_stalls(project)`** per project:
   - Calls `gh pr list --state open --json number,headRefName,isDraft,baseRefName --repo {repo}` to get open PRs with SDLC branches (`session/sdlc-*`).
   - For each SDLC PR, checks: not draft, issue still open (`gh issue view N --json state`), last commit timestamp via `git log --format=%ct -1 origin/session/sdlc-{N}`, no active AgentSession (`AgentSession.query.filter(slug=f"sdlc-{N}", status__in=[...])`).
   - If all stall gates pass, checks Redis dedup key `sdlc:stall:alert:{slug}:{last_commit_sha}` (TTL = cooldown_hours × 3600). If key absent: emit Telegram alert and set the Redis key.
4. **Telegram alert**: `subprocess.run(["valor-telegram", "send", "--chat", "Dev: Valor", message])` — same pattern as `_send_log_review_telegram` in `reflections/auditing.py`.
5. **Return**: `run_per_project_audit` aggregates findings and returns `{status, findings, summary}`.

## Solution

### Key Elements

- **`reflections/sdlc_progress.py`**: New module with `run_sdlc_progress_check()` as the top-level callable and `_check_project_stalls(project)` as the per-project audit body.
- **Stall heuristic**: Five gates, all must pass — (1) open PR, (2) not draft, (3) SDLC branch pattern (`session/sdlc-{N}`), (4) no new commit in N hours (default 4h, tunable), (5) no AgentSession with `slug=sdlc-{N}` in any non-terminal status. The set of blocking statuses comes directly from `models.session_lifecycle.NON_TERMINAL_STATUSES` (`pending`, `running`, `active`, `dormant`, `waiting_for_children`, `superseded`, `paused_circuit`, `paused`).
- **Dormant/paused session guard**: ALL non-terminal statuses block the stall alert. This aligns with the issue body's explicit requirement: *"Dormant/paused sessions waiting on humans are not flagged as stalled."* Rationale:
  - `dormant` sessions are waiting on a human reply — the human is the bottleneck, not a stalled SDLC.
  - `paused` and `paused_circuit` auto-recover via drip (`agent.sustainability.session_recovery_drip`) — they are not state-level stalls.
  - `superseded` is transitional and finalized immediately — never observed in steady-state.
  - `waiting_for_children` is awaiting child sessions — not stalled.
  - This is the conservative choice that minimizes false positives. The cost is missing a real stall when the dormant PM session itself is wedged — but that case is detected by the existing process-layer `session-liveness-check`. State-layer detection complements process-layer detection; it does not duplicate or replace it.
- **Dedup**: Raw Redis key `sdlc:stall:alert:{slug}:{last_commit_sha}` with TTL = `cooldown_hours * 3600`, written via `r.set(key, "1", nx=True, ex=ttl)`. The key namespace is custom (NOT a Popoto model), matching the established precedent in `reflections/docs_auditor.py:112`, `reflections/pm_audio_briefing/__init__.py:109`, and `agent/sustainability.py:97-98,115-116`. The CLAUDE.md "no raw Redis on Popoto-managed keys" rule does not apply — this is a non-model dedup namespace, not a key managed by an ORM. The sha suffix prevents re-alerting on the same stall within one cooldown window but allows re-alerting if new commits land and then go stale again.
- **`config/reflections.yaml`**: New entry `sdlc-progress-check` at 1800s interval, low priority, `execution_type: function`, `callable: reflections.sdlc_progress.run_sdlc_progress_check`, `enabled: true`.
- **Telegram alert format**: Single message per stalled slug — slug, PR URL, last commit age, last active session timestamp (if available).

### Flow

`ReflectionScheduler tick` → `run_sdlc_progress_check()` → per-project `_check_project_stalls(project)` → `gh pr list` → per-PR stall gates → Redis dedup check → `valor-telegram send` (if stall detected and not already alerted)

### Technical Approach

- New module `reflections/sdlc_progress.py` mirrors the structure of `reflections/docs_auditor.py`: top-level `run_*` callable + private helper(s).
- `_check_project_stalls(project)` returns `{status, findings, summary, duration}` to satisfy `run_per_project_audit`.
- **Project-to-repo resolution** (cross-project gh calls): All `gh` and `git` subprocess calls run with `cwd=project["working_directory"]`. `gh` natively reads `.git/config` and resolves the GitHub repo from the local clone — no `--repo` flag needed. This matches the established pattern in `reflections/maintenance.py:380-462` (`cwd=project_wd` for all gh and git subprocesses). If `project["working_directory"]` does not exist on disk, `load_local_projects()` already filters it out (see `reflections/utils.py:62-66`), so by the time `_check_project_stalls(project)` runs, the cwd is guaranteed valid.
- Parse SDLC slug from PR branch: `session/sdlc-{N}` → slug = `sdlc-{N}`, issue N = int(N). The branch pattern is the canonical signal — N is the issue number per the SDLC slug convention documented in `docs/features/session-isolation.md`. PR body `Closes #M` linkage is NOT used (it can mismatch the slug, and the slug is authoritative for SDLC pipeline tracking).
- **Active session check** (full implementation):
  ```python
  from agent.session_health import _filter_hydrated_sessions
  from models.session_lifecycle import NON_TERMINAL_STATUSES

  candidates = AgentSession.query.filter(slug=slug)
  hydrated = _filter_hydrated_sessions(candidates)
  active = [s for s in hydrated if getattr(s, "status", None) in NON_TERMINAL_STATUSES]
  if active:
      # Slug has at least one non-terminal session — NOT stalled. Skip.
      return
  ```
  Importing `NON_TERMINAL_STATUSES` directly avoids hand-listing statuses (which drifts when new statuses are added — see commit `353834a7` for an example of status-list drift). `_filter_hydrated_sessions` is the canonical phantom-row filter from `agent/session_health.py`, used by `agent/sustainability.py:160,163,299,358`.
- Commit recency: `git log --format=%ct -1 origin/session/sdlc-{N}` with `cwd=project["working_directory"]`. Parse stdout as float; compute age = `time.time() - last_commit_ts`. If `returncode != 0` (branch not found locally — e.g. project never fetched), skip the slug at DEBUG level.
- **Issue-state check**: For each candidate slug `sdlc-{N}`, run `gh issue view {N} --json state` with `cwd=project["working_directory"]`. Skip if state != "OPEN" (the issue was already closed; the PR is stale by design).
- Stall threshold default: **4 hours (14400s)**, tunable via env var `SDLC_STALL_THRESHOLD_HOURS`. Justification: the longest dev-session lifecycles observed in production telemetry (build → test → patch cycles) typically complete within 90 minutes; the original cited stalls (PR #1237, PR #1236) sat at ~8 hours of inactivity. 4h gives ≥2.5x buffer over the longest healthy session and ≤50% of the cited real-stall duration. Operators can tune via the env var without redeploy. Phase 5 success criteria includes a 30-day post-deploy review of the threshold against actual alert noise.
- Redis dedup key TTL: 6 hours (21600s), tunable via env var `SDLC_STALL_COOLDOWN_HOURS`. Set via `r.set(dedup_key, "1", nx=True, ex=ttl)`. The `nx=True` makes the dedup atomic — only the first detector tick within the cooldown writes the key and emits the alert.
- Alert message format:
  ```
  SDLC stall detected: {slug}
  PR: {pr_url}
  Last commit: {age_human} ago (branch: session/{slug})
  No active session since: {last_session_ts or "unknown"}
  Issue #{N} still open.
  ```
- **Telegram routing**: Default to the existing `Dev: Valor` chat (matches `_send_log_review_telegram` precedent in `reflections/auditing.py:280`). Per-project PM channel routing (resolving `project["telegram"]["groups"]["PM: {project_name}"]` via the `_resolve_chat_id` pattern in `reflections/pm_audio_briefing/delivery.py:43-66`) is deferred to a follow-up issue once we observe alert noise levels in v1. This is intentional: SDLC stalls are bridge-machine-internal anomalies, and the dev chat is the right initial audience. The `--chat "Dev: Valor"` form is what every other reflection uses; matching that minimizes routing surface area for v1.
- All subprocess failures (gh, git, valor-telegram) are logged at WARNING and do not raise — the reflection returns `{status: "ok"}` even if Telegram send fails.

## Failure Path Test Strategy

### Exception Handling Coverage

- `gh pr list` subprocess failure: log WARNING, return empty findings (no stall alerts). Test: mock `subprocess.run` to raise `FileNotFoundError` → no exception propagates, function returns `{status: "ok", findings: []}`.
- `git log` subprocess failure (branch not found): log DEBUG, skip slug. Test: mock returncode != 0 → slug is skipped, no alert sent.
- `AgentSession.query.filter` raises (Redis down): catch, log WARNING, return `{status: "error"}`. Test: mock to raise `ConnectionError` → caught, status="error".
- `valor-telegram send` failure: log WARNING, reflection still returns OK findings. Test: mock FileNotFoundError from subprocess → finding still emitted to log, no exception.

### Empty/Invalid Input Handling

- Project with no open PRs: `gh pr list` returns `[]` → no-op, function returns `{status: "ok", findings: []}`.
- PR branch not matching `session/sdlc-*` pattern: skipped by branch filter.
- `git log` returns empty string: skip slug (treat as "branch unknown, not stalled").

### Error State Rendering

- Telegram alert is the user-visible output. If `valor-telegram send` fails, the finding is logged to `worker.log` as a fallback — same pattern as log-review reflection.

## Risks

### Risk 1: False positives on legitimate long-thinking turns
**Impact:** Noise alerts when a dev session is running a very long tool call (e.g., 6+ hour test suite) but no new commits have been pushed yet.
**Mitigation:** The "no active session" gate catches this — if any non-terminal session has `slug=sdlc-{N}` (`pending`, `running`, `active`, `dormant`, `waiting_for_children`, `paused`, `paused_circuit`, `superseded`), no alert fires. Default threshold 4h gives substantial buffer over the longest healthy dev-session lifecycles observed in production (≤90 min).

### Risk 1b: Threshold too aggressive (false-positive storm)
**Impact:** If the 4h default catches legitimate "thinking quietly" pauses where a dormant PM has just woken and is composing a steering message, we get alert noise.
**Mitigation:** The `dormant`-blocks-alert behavior fully covers this — a dormant PM IS a non-terminal session with the slug, so no alert fires until the PM session itself becomes terminal. Threshold tunable via `SDLC_STALL_THRESHOLD_HOURS` env var. Plan includes a 30-day post-deploy review of the threshold against actual alert noise (in Success Criteria as an operational follow-up).

### Risk 2: `gh pr list` rate limits or auth failures
**Impact:** Reflection returns error status; stall alerts not sent.
**Mitigation:** All `gh` subprocess failures are caught and logged as WARNING. The scheduler records `last_status = "error"` on the Reflection record — operators can see auth issues in `worker.log`. No alert storm since failures are silent.

### Risk 3: Redis unavailable for dedup key check
**Impact:** May send duplicate stall alerts (if Redis is down, each tick re-sends).
**Mitigation:** Redis unavailability is caught and logged. The reflection exits early when Redis is down rather than spamming. Dedup key writes are best-effort — if the `setex` fails, the finding is still logged but the Telegram send is skipped to avoid potential spam.

## Race Conditions

### Race 1: Dev session just completed, stall check fires before PM session picks up
**Location:** `reflections/sdlc_progress.py:_check_project_stalls`
**Trigger:** Dev session transitions to `completed` (terminal), PM session has not yet been scheduled; stall check fires in the <30s window before PM wakes up.
**Data prerequisite:** AgentSession row for the dev session must be in terminal status before the stall check queries.
**State prerequisite:** The stall threshold is 4 hours — far longer than PM session wake-up latency (~seconds to minutes). This race cannot trigger a false alert given the conservative threshold.
**Mitigation:** 4-hour threshold eliminates this race entirely. No locking needed.
