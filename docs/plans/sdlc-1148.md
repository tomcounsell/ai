# PM Harness Sessions: Inject SESSION_TYPE + Load PM Persona

## Problem

**Real scenario (from issue #1148):** PM session `8ff90a3c86ab465a80438ab3d2d64e39` (project: valor, thread `tg_valor_-1003449100931_690`) received `MESSAGE: SDLC issue 1139`. Because its subprocess environment lacked `SESSION_TYPE=pm` and the PM persona was never loaded into the `claude -p` system prompt, the PM session did not know it was a PM session. It spawned a dev session with `message_text` that was the PM's own enrichment-header payload ("PROJECT: Valor AI\nFOCUS: ...") instead of the actual task. The dev session replied "the message appears to have been truncated" and stalled. No SDLC work was done for #1139.

**Current behavior:** `agent/session_executor.py:1324-1329` builds `_harness_env` with only `AGENT_SESSION_ID`, `CLAUDE_CODE_TASK_LIST_ID`, and (for PM/Teammate) `VALOR_PARENT_SESSION_ID`. As a result:

1. `os.environ.get("SESSION_TYPE")` is unset in the `claude -p` subprocess. `_is_pm_session()` in `agent/hooks/pre_tool_use.py:97-99` always returns `False` for harness-executed PM sessions, so `PM_BASH_ALLOWED_PREFIXES` and `_PM_BASH_FORBIDDEN_METACHARS` are never enforced. The PM session can run arbitrary Bash, including mutations.
2. The PM persona (loaded by `load_pm_system_prompt()` in `agent/sdk_client.py:898`) is never injected into the harness argv — `_HARNESS_COMMANDS["claude-cli"]` contains no `--system-prompt` / `--append-system-prompt`. The PM's SDLC pipeline rules (Rule 1–5), dev-session dispatch template, and pre-completion checklist are invisible to PM sessions.
3. `TELEGRAM_CHAT_ID` and `SENTRY_AUTH_TOKEN` (injected by the SDK-era `ValorAgent.env` at `sdk_client.py:1257-1285`) are also missing, so PM-side Telegram sends via `tools/send_telegram.py` break, and `sentry-cli` requires manual auth.

**Desired outcome:** PM harness sessions run with `SESSION_TYPE=pm` in the subprocess environment, the PM persona content is appended to `claude -p`'s system prompt, and `TELEGRAM_CHAT_ID` / `SENTRY_AUTH_TOKEN` are injected for PM/Teammate — restoring full parity with the SDK-era `ValorAgent.env` path. Hook enforcement for PM Bash restrictions fires correctly, the PM dev-dispatch template is present in context, and child dev sessions receive the intended task text (not enrichment headers).

## Spike Results

### spike-1: Can the PM persona fit in argv on macOS?
- **Assumption:** "The ~65k-char PM persona (issue's claim) fits safely in a single `--append-system-prompt <prompt>` argv slot."
- **Method:** code-read
- **Finding:** `getconf ARG_MAX` on this machine = 1,048,576 bytes (1 MB). Actual work-vault persona at `~/Desktop/Valor/personas/project-manager.md` is 7.3 KB; fallback persona at `config/personas/project-manager.md` is 22.4 KB. Even after `load_pm_system_prompt()` concatenates persona + project CLAUDE.md, total is well under 50 KB. Fits with 20x headroom. (Issue's 65k figure was inaccurate.)
- **Confidence:** high
- **Impact on plan:** Use `--append-system-prompt <content>` directly. No need for `--system-prompt-file` path or chunking.

### spike-2: `--system-prompt` (replace default) vs `--append-system-prompt` (append)?
- **Assumption:** "Either flag works; issue proposes `--system-prompt`."
- **Method:** code-read (Claude CLI `--help`)
- **Finding:** `--system-prompt` REPLACES Claude Code's default system prompt (tool instructions, CLAUDE.md auto-discovery, standard protocol). `--append-system-prompt` appends to the default. The PM persona is additive guidance that assumes Claude Code's default tool-handling stack is still in place — the PM persona's pipeline rules reference Bash, Read, Edit tool usage that the default prompt configures. **Using `--system-prompt` would break the baseline tool protocol.**
- **Confidence:** high
- **Impact on plan:** Use `--append-system-prompt` (REVISION to issue's sketch — issue proposed `--system-prompt`). Document this decision inline in the code change.

## Data Flow

1. **Entry point:** Telegram message arrives → bridge creates AgentSession with `session_type="pm"` → enqueued to Redis.
2. **Worker pickup:** `worker/__main__.py` picks up session → `agent/session_executor.py::_execute_agent_session()` is invoked.
3. **Turn input build:** `build_harness_turn_input()` (agent/sdk_client.py) prepends enrichment headers (PROJECT, FROM, SESSION_ID, SCOPE, MESSAGE) to the raw message.
4. **Env construction (the bug):** `_harness_env` is built at `session_executor.py:1324-1329` with only `AGENT_SESSION_ID`, `CLAUDE_CODE_TASK_LIST_ID`, and optionally `VALOR_PARENT_SESSION_ID`. **No `SESSION_TYPE`, no `TELEGRAM_CHAT_ID`, no `SENTRY_AUTH_TOKEN`, no persona.**
5. **Harness call:** `get_response_via_harness(message=..., env=_harness_env, ...)` → builds `claude -p` argv with `--verbose --output-format stream-json --include-partial-messages --permission-mode bypassPermissions` + `--model <alias>` + positional message. No `--append-system-prompt`.
6. **Subprocess runs:** Child `claude -p` runs. Auto-loads repo `CLAUDE.md` (working-dir scoped) but NOT the PM persona. `SESSION_TYPE` is unset in `os.environ`.
7. **Hooks fire inside child:** `pre_tool_use.py::_is_pm_session()` returns `False`. PM Bash allowlist is not enforced.
8. **PM session sends reply:** Without PM persona context, PM misunderstands its role. Calls `valor_session create --message "..."` with the wrong payload (forwards its own context headers instead of the task).
9. **Dev session stalls:** Receives garbage `message_text`, reports truncation, stalls. SDLC never progresses.

**Fix lives at step 4 and step 5** — extend `_harness_env` and pass persona through `get_response_via_harness` to `claude -p` argv.

## Solution

### Key Elements

- **Fix A — `_harness_env` parity with `ValorAgent.env`:** Extend the env dict in `agent/session_executor.py::_execute_agent_session()` to inject `SESSION_TYPE` (for all typed sessions), plus `TELEGRAM_CHAT_ID` and `SENTRY_AUTH_TOKEN` for PM/Teammate sessions. Mirror the exact logic from `sdk_client.py:1257-1285`.
- **Fix B — PM persona wiring:** Add `system_prompt: str | None = None` keyword-only parameter to `get_response_via_harness()`. When set, inject `--append-system-prompt <value>` into the harness argv. In `session_executor.py`, for PM sessions, call `load_pm_system_prompt(working_dir)` and pass the result to `get_response_via_harness`. Completion-runner callers (session_completion.py) keep default `None` — no signature break.
- **Fix C (optional hardening) — `valor_session create` enrichment-header guard:** In `tools/valor_session.py`'s `create` subcommand, reject `--message` values whose first 100 chars match the enrichment header pattern (starts with any of `SESSION_ID:`, `PROJECT:`, `FROM:`, `TASK_SCOPE:`, `SCOPE:`). Emit a clear error message explaining enrichment headers are injected automatically by the worker and only task content should be passed. Optional — lower priority than A and B.

### Flow

**Bug scenario today:**
Telegram PM message → worker → `_harness_env={AGENT_SESSION_ID, CLAUDE_CODE_TASK_LIST_ID}` → `claude -p` (no persona, no SESSION_TYPE) → PM confused → spawns dev with `message_text="PROJECT: Valor AI\nFOCUS..."` → dev stalls

**Fixed scenario:**
Telegram PM message → worker → `_harness_env={AGENT_SESSION_ID, CLAUDE_CODE_TASK_LIST_ID, SESSION_TYPE=pm, TELEGRAM_CHAT_ID=..., SENTRY_AUTH_TOKEN=...}` + `--append-system-prompt "<PM persona + CLAUDE.md>"` → `claude -p` (PM persona loaded, SESSION_TYPE set, hooks enforce Bash allowlist) → PM dispatches dev correctly → dev executes task → SDLC progresses

#### Step A — Extract `_resolve_sentry_auth_token()` helper

In `agent/sdk_client.py`, above `ValorAgent.env` (insert before line ~1200), add:

```python
def _resolve_sentry_auth_token() -> str | None:
    """Resolve Sentry auth token for PM/Teammate session env injection.

    Cascade: SENTRY_PERSONAL_TOKEN env var → SENTRY_AUTH_TOKEN env var →
    ~/Desktop/Valor/.env file read (only in terminal mode; launchd blocks
    ~/Desktop under TCC).
    """
    token = os.environ.get("SENTRY_PERSONAL_TOKEN") or os.environ.get("SENTRY_AUTH_TOKEN")
    if token:
        return token
    if os.environ.get("VALOR_LAUNCHD"):
        return None  # launchd cannot read ~/Desktop (macOS TCC)
    sentry_env = Path.home() / "Desktop" / "Valor" / ".env"
    if not sentry_env.exists():
        return None
    try:
        for line in sentry_env.read_text().splitlines():
            if line.startswith("SENTRY_PERSONAL_TOKEN="):
                return line.split("=", 1)[1]
    except (OSError, PermissionError):
        return None
    return None
```

Then refactor the existing cascade at `sdk_client.py:1272-1284` to call `_resolve_sentry_auth_token()`:

```python

    token = _resolve_sentry_auth_token()
    if token:
        env["SENTRY_AUTH_TOKEN"] = token
```

This is the only structural refactor — single source of truth for the Sentry cascade.

#### Step E — (Fix C) `valor_session create` enrichment-header guard

In `tools/valor_session.py`'s `create` subcommand handler, after argument parsing but before session creation:

```python
# Reject messages that look like forwarded enrichment headers (issue #1148,
# observed in PM session 8ff90a3c86ab465a80438ab3d2d64e39). Enrichment
# headers are injected automatically by the worker's build_harness_turn_input;
# passing them here means the caller is forwarding context instead of task text.
_ENRICHMENT_HEADER_RE = re.compile(r"^(?:SESSION_ID|PROJECT|FROM|TASK_SCOPE|SCOPE):", re.MULTILINE)
if args.message and _ENRICHMENT_HEADER_RE.match(args.message[:200]):
    print(
        "Error: --message looks like forwarded session context headers, not a task description.\n"
        "Enrichment headers (PROJECT:, FROM:, SESSION_ID:, TASK_SCOPE:, SCOPE:) are injected\n"
        "automatically by the worker. Pass only the task content.",
        file=sys.stderr,
    )
    sys.exit(2)
```

Note: case-sensitive to avoid false positives on normal prose (e.g., "scope: ..." in lowercase). Regex uses `^` anchor plus 200-char prefix window so a buried `MESSAGE: ...\nSESSION_ID: ...` construct (where `MESSAGE:` legitimately appears in the body) does NOT trip the guard — only enrichment-header-LED messages do.

## Failure Path Test Strategy

### Exception Handling Coverage

- [ ] `load_pm_system_prompt(working_dir)` can raise (both work-vault and fallback missing — rare but possible on a fresh machine). Test that `_execute_agent_session` catches, logs `[pm-persona-missing]`, and does not crash. Assert the subprocess still spawns (session continues degraded) and `_pm_system_prompt` is `None` at the harness-call site.
- [ ] `_resolve_sentry_auth_token()` can fail on `~/Desktop/Valor/.env` file read (TCC-blocked under launchd). Test with monkey-patched `Path.home` that points to a non-existent dir: assert `None` is returned without raising. Also test with `VALOR_LAUNCHD=1`: assert the file-read path is skipped entirely.
- [ ] `get_response_via_harness` with `system_prompt=""` (empty string, falsy) — assert `--append-system-prompt` is NOT injected. Only truthy non-empty strings trigger injection.
- [ ] `get_response_via_harness` with `system_prompt=None` — same as empty: no flag injection.

## Risks

### Risk 1: Persona size growth breaks argv length

**Impact:** If the PM persona grows past `ARG_MAX` (1 MB on macOS; typically 128 KB on Linux), `claude -p` subprocess spawn fails with `E2BIG`, PM session turns silently 500.

**Mitigation:** Defensive length check in `get_response_via_harness` before argv assembly: if `len(system_prompt) > 512_000`, log a warning and omit `--append-system-prompt`. Session proceeds degraded but functional. Actual persona today is ~25 KB — 20x headroom. Covered by failure-path test.

### Risk 2: `--append-system-prompt` semantics differ from SDK `system_prompt` option

**Impact:** The SDK path sets a `system_prompt` on the Agent SDK options object; the CLI `--append-system-prompt` appends to the default. Behavior may drift subtly (e.g., tool instruction ordering, message delimiters).

**Mitigation:** spike-2 confirmed `--append-system-prompt` is semantically correct (preserves default Claude Code tool-handling). If drift is observed post-ship, the CLI also supports `--system-prompt-file` and `--system-prompt` — trivial to adjust. Integration test captures the exact argv so regressions are visible.

### Risk 3: `load_pm_system_prompt` fails silently, PM runs without persona, bug recurs invisibly

**Impact:** If `load_pm_system_prompt` raises (e.g., persona file missing on a fresh machine), the `except: log-and-continue` path keeps the session alive but back in the broken state from before this fix.

**Mitigation:** Emit a structured log line with a distinctive prefix (`[pm-persona-missing]`) so the dashboard / log analysis can alert on it. Include a startup-time validation check in `/do-docs` or the `doctor` tool that fails loudly if the persona file is missing — catches config problems at deploy time, not at first PM turn. (Doctor addition is optional; primary defense is the log.)

### Risk 4: Completion-runner accidentally passes the PM persona to drafter turns

**Impact:** `agent/session_completion.py::_deliver_pipeline_completion` calls `get_response_via_harness(..., model="opus")` for the Pass-1/Pass-2 summary drafter. If some future refactor accidentally passes `system_prompt=<pm_persona>` through, drafter turns would get the PM orchestration rules baked into their system prompt, corrupting the summary.

**Mitigation:** Keep the new parameter keyword-only with `None` default. Add an inline comment at each `get_response_via_harness` call site in `session_completion.py` explaining "drafter turns intentionally omit system_prompt — content is isolated from parent PM persona." Caught at review time; not a runtime concern.
