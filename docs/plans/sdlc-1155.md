# Self-Healing SDLC Merge Gate

## Problem

A PM session that reaches the MERGE stage on an approved, mergeable PR should be able to finish the pipeline without pinging a human. Today, seven distinct gate conditions halt the pipeline with a "report and stop" output — each one is a well-understood failure mode with a well-understood fix. The PM has no documented pattern for self-resolution, so each one escalates to a human.

**Current behavior:**

1. **Cold Redis state** — `.claude/commands/do-merge.md:9-64` reads `stage_states` via `PipelineStateMachine.get_display_progress()`. If the per-machine Redis is cold (fresh machine, eviction, or a cleared Popoto session), every stage returns `"pending"` and the gate prints `WARNING: No pipeline state found ... Treat all stages as unconfirmed` — escalating instead of deriving completion from durable artifacts (PR branch plan file, PR existence, CI rollup, review comment, docs in the diff).

2. **Stale review comments** — The Structured Review Comment Check (`.claude/commands/do-merge.md:117-140`) uses `jq '[.[] | select(.body | startswith("## Review:"))] | last'`. `last` after GitHub's default ASC ordering = newest, but there is no filter against the latest commit SHA's committer date. A stale "Approved" from before a `git push --force-with-lease` passes; a stale "Changes Requested" that was later re-approved via a non-structured comment still blocks.

3. **Lockfile drift caught late** — `.githooks/pre-commit:24-56` runs ruff format, ruff check --fix, and a secret scan. No `uv lock --locked` step. Drift between `pyproject.toml` and `uv.lock` only surfaces at the merge gate (`do-merge.md:218-236`), where it blocks the PM. The fix (`uv lock && git commit`) is trivial but happens at the wrong layer.

4. **Baseline never decays** — `data/main_test_baseline.json` accumulates `real`/`flaky`/`hung`/`import_error` entries (schema v2 shipped in #1084). The `baseline_gate.py` already produces `baseline_keys_no_longer_failing` as an advisory, but nothing consumes it. Tests that have passed on `main` for many merges stay in the baseline forever, masking if a *new* regression re-enters the same test. No quarantine hint is emitted when a flaky re-occurrence pattern is repeat.

5. **PM has no documented gate-recovery behavior** — PM Rule 5 (`config/personas/project-manager.md:42-55`) mandates MERGE before sign-off. PR #1010 ensures the PM cannot exit with an open PR. But there is no documented mapping of "gate output says X" → "PM dispatches Y." A senior dev reads the gate report, fixes the specific blocker, and re-runs. The current persona leaves this implicit, so the PM tends to report the failure and wait.

6. **No merge-troubleshooting playbook** — `docs/sdlc/` has per-stage addenda but no `merge-troubleshooting.md`. Common blockers with well-known fixes (conflict rebase, unchecked plan checkoffs, G4 oscillation, stale review, lockfile drift, flake false regression) are undocumented. The PM cannot cite a playbook because there is none.

7. **Merge-guard regex self-blocks on diagnostic messages** — `.claude/hooks/validators/validate_merge_guard.py:40` matches `_MERGE_CMD_RE = re.compile(r"\bgh\s+pr\s+merge\b")` against any Bash command text. The exemption at line 85-87 only covers commands whose `strip()` prefix is `echo ` / `echo\t` / `printf `. Observed live during triage of this very issue: a `git commit -m "..."` whose body referenced the guard's trigger substring self-blocked. The PM enters a self-avoidance loop around its own tooling.

**Desired outcome:**

A PM session can drive any approved, mergeable PR from REVIEW → DOCS → MERGE to a merged state without human intervention. The seven friction points are either eliminated at the source (cold-Redis fallback, commit-SHA filter, `uv lock` in pre-commit, baseline decay + quarantine hint, guard exemption) or documented as senior-dev playbooks the PM can follow (gate-recovery rule, merge-troubleshooting doc). Human intervention is reserved for genuinely unique architect-level judgement — not gate mechanics.

## Research

**Queries used:**
- `uv lock --locked pre-commit hook exit code 2026`
- `GitHub API issue comments filter by commit date force-push stale review 2026`
- `pytest flaky test retry --lf last-failed distinguish regression 2026`

**Key findings:**

- `uv lock --locked` is **read-only** and exits non-zero if the lockfile is out of sync with `pyproject.toml`. Never modifies files. Confirmed by the official uv docs ([docs.astral.sh/uv/guides/integration/pre-commit](https://docs.astral.sh/uv/guides/integration/pre-commit/)) and the `astral-sh/uv-pre-commit` repo. **Informs item 3**: the pre-commit phase should call `uv lock --locked` — no mutations, safe to run on every commit.
- GitHub's REST API returns issue comments in `created_at` ASC by default. Force-push does NOT delete comments or reviews automatically; GitHub only dismisses "stale approvals" if the branch protection "Dismiss stale pull request approvals when new commits are pushed" is enabled ([community discussion 12876](https://github.com/orgs/community/discussions/12876)). We cannot rely on this branch-protection setting because our PRs are not all behind one. **Informs item 2**: filter out any `## Review:` comment whose `created_at` predates the PR's latest commit `committer_date`, read via `gh api repos/{o}/{r}/pulls/{n}/commits` and taking the last entry.
- pytest has `--lf`/`--last-failed` built in via the cache plugin, plus external plugins (`pytest-rerunfailures`, `pytest-retry`, `flaky`). We already run a single retry via `/do-test`'s flaky filter (PR #484), so no additional retry tooling is needed at the merge gate. **Informs the narrowing of item 4**: the retry layer stays at `/do-test`; the merge gate consumes the `baseline_keys_no_longer_failing` advisory for decay, not a new retry pass.

## Data Flow

This plan touches two distinct runtime paths. Tracing each end-to-end:

### Path A: PM session reaches MERGE stage (cold Redis)

1. **Entry**: PM session's SDLC router invokes `/do-merge {pr_number}`.
2. **Pre-Merge Pipeline Check** (`.claude/commands/do-merge.md:9-64`): reads `stage_states` from Redis via `PipelineStateMachine.get_display_progress()`.
   - **Today**: returns `{}` on cold Redis → prints escalation warning → PM stops.
   - **After item 1**: `{}` on cold Redis → `derive_from_durable_signals()` → returns `{PLAN: 'completed', BUILD: 'completed', TEST: 'completed', REVIEW: 'completed', DOCS: 'completed'}` if the durable artifacts exist → gate proceeds.
3. **Prerequisites Check** (lines 74-115): query TEST/REVIEW/DOCS. Same as above.
4. **Structured Review Comment Check** (lines 117-140): read `gh api .../comments` → filter `## Review:` → take most recent.
   - **After item 2**: before taking "most recent", drop any comment older than the PR's latest commit `committer_date`.
5. **Plan Completion Gate** (lines 144-214): read plan from `origin/$BRANCH` (hotfix `e2c26b77` — preserve).
6. **Lockfile Sync Check** (lines 218-236): `uv lock --locked`.
   - **After item 3**: this is now a belt-and-suspenders check. The first-line-of-defense is the pre-commit hook. The merge-gate check stays (cheap, safe).
7. **Full Suite Gate** (lines 239-328): `pytest tests/` → `scripts/baseline_gate.py` → verdict JSON with `new_blocking_regressions`, `new_flaky_occurrences`, `preexisting_failures`, `baseline_keys_no_longer_failing`.
   - **After item 4**: post-merge reset path updates `recent_passes_by_test` counters; entries with N consecutive passes are dropped. Flaky occurrences that repeat M times emit a quarantine hint to stderr.
8. **Gate verdict** → if all pass, `touch data/merge_authorized_{pr}` → `gh pr merge {pr}` → cleanup.

### Path B: Bash command with "gh pr merge" in an argument

1. **Entry**: Claude Code invokes a `Bash` tool call (e.g. `git commit -m "…describes merge flow…"` or `gh issue edit --body "…mentions /do-merge…"`).
2. **PreToolUse hook**: `.claude/hooks/validators/validate_merge_guard.py` runs.
3. **Today**: `_MERGE_CMD_RE.search(command)` matches because the heredoc body contains "gh pr merge" → exemption at line 85-87 only fires for bare `echo`/`printf` prefixes → block message emitted → Bash call rejected.
4. **After item 7**: new classifier `_extract_executed_commands(command)` tokenises the Bash string into a sequence of "actual executed commands" (first token after start, `&&`, `||`, `;`, `|`, newline) and applies `_MERGE_CMD_RE` only to those positions. Heredoc bodies and quoted argument strings are excluded. The guard still blocks a real `gh pr merge` direct call; it stops false-positive-ing on diagnostic text.

## Solution

### Key Elements

- **Item 1 — Durable-Signal Fallback**: When `stage_states` is cold, derive PLAN/BUILD/TEST/REVIEW/DOCS completion from durable sources (plan file on PR branch, PR existence, CI rollup, most-recent `## Review:` comment, `docs/` files in PR diff). Only kicks in when `get_display_progress()` returns empty. Preserves the `e2c26b77` PR-branch-first pattern.
- **Item 2 — Commit-SHA Review Filter**: Drop `## Review:` comments whose `created_at` predates the PR's latest commit `committer_date`. Eliminates stale "Approved" (before force-push) and stale "Changes Requested" (after re-approval) false verdicts.
- **Item 3 — `uv lock --locked` Pre-Commit Phase**: New phase in `.githooks/pre-commit` between ruff and secret-scan. Runs `uv lock --locked` and blocks commit with a clear fix hint if it exits non-zero. No mutations. Skips gracefully if `uv` is not installed.
- **Item 4 — Baseline Decay + Quarantine Hint**: Consume the existing `baseline_keys_no_longer_failing` advisory from `baseline_gate.py`. On every clean merge, increment a per-test `recent_pass_count` in a new `_decay_tracker` field of `main_test_baseline.json`; at `recent_pass_count >= 5` the entry is aged out. When a PR's `new_flaky_occurrences` list includes a test that has appeared in it 3+ consecutive runs, emit a `QUARANTINE HINT` line to stderr.
- **Item 5 — PM Gate-Recovery Rule**: Add "Gate-Recovery Behavior" section to `config/personas/project-manager.md` after Rule 5. Maps each gate blocker category to a remediation skill. Requires the PM to read `docs/sdlc/merge-troubleshooting.md` (item 6) and dispatch the mapped skill, then re-dispatch `/do-merge`. Converges within G4's 3-dispatch window.
- **Item 6 — Merge-Troubleshooting Playbook**: New `docs/sdlc/merge-troubleshooting.md` with six sections: merge conflict, unchecked plan checkboxes, G4 oscillation, stale review, lockfile drift, flake false regression. Each section: symptom → `gh`/`git`/`python` commands → expected outcome → link back to the relevant gate section in `.claude/commands/do-merge.md`.
- **Item 7 — Merge-Guard Regex Exemption**: Replace the bare-prefix check with a command tokeniser that identifies which positions in the Bash string are *actual* executed commands (first token, after `&&`/`||`/`;`/`|`/newline) vs. positions inside heredocs, single-quoted strings, double-quoted strings, or backtick substitutions. Apply `_MERGE_CMD_RE` only at actual-command positions. A direct `gh pr merge` call is still blocked; diagnostic text mentioning the command is not.

### Flow

**PM session facing a merge failure** → Reads `/do-merge` gate output → Consults `## Gate-Recovery Behavior` section of PM persona → Maps blocker category to remediation skill → Consults `docs/sdlc/merge-troubleshooting.md` for exact commands → Dispatches remediation → Re-dispatches `/do-merge` → **Merged PR, no human intervention**.

### Technical Approach

- **Additive only**: no existing behavior is rewritten. Every new code path is a *fallback* or *extension*, and the primary paths stay identical. The `e2c26b77` and `1d67d81e` hotfixes are preserved verbatim.
- **Item 1's derivation lives in `agent/pipeline_state.py`**, not in the shell script. The shell block at `.claude/commands/do-merge.md:9-64` becomes a two-line change: after the existing `PipelineStateMachine(session).get_display_progress()` call, if the result is empty, call `PipelineStateMachine.derive_from_durable_signals(session)` and use that instead. Pushing the logic into Python keeps it unit-testable via direct function import (follows the pattern established by `scripts/baseline_gate.py`).
- **Item 2's filter logic** is a handful of lines in the shell block — `LATEST_COMMIT_DATE=$(gh api .../pulls/$PR/commits --jq '.[-1].commit.committer.date')` → filter the `## Review:` comments with a `jq` comparison `select(.created_at > $latest_date)` before taking `last`. No Python changes needed.
- **Item 3's pre-commit phase** uses `uv lock --locked` directly (read-only; safe in pre-commit). If `uv` is not on `$PATH`, skip with a one-line warning (machines without uv still get a working hook). When the check fails, the error message includes the exact fix command.
- **Item 4's decay tracker** lives inside `main_test_baseline.json` as an optional top-level `_decay_tracker: {test_id: {recent_pass_count: N}}` field. The post-merge reset block (`do-merge.md:262-279`) increments counters for each baseline entry that did NOT fail in the PR run; entries with `recent_pass_count >= DECAY_THRESHOLD` (default 5) are removed. The quarantine hint is emitted by a new function `scripts/baseline_gate.py::format_quarantine_hints()` reading a companion counter `_flake_tracker: {test_id: {consecutive_flake_runs: N}}`.
- **Item 5's persona section** does not change any Rule. It appends a new `## Gate-Recovery Behavior` section documenting the blocker → remediation mapping. Cross-links to `docs/sdlc/merge-troubleshooting.md`.
- **Item 6's playbook** is a pure doc file. Each section: symptom (one line) → diagnostic commands (copy-pasteable) → remediation commands (copy-pasteable) → verification step → cross-links. Matches the house style of existing `docs/sdlc/do-*.md` files.
- **Item 7's tokeniser** is the only non-trivial piece. Uses a small hand-rolled state machine (existing pattern — `agent/sdlc_router.decide_next_dispatch` uses the same style). Handles: single-quoted strings, double-quoted strings, heredocs (`<<EOF`/`<<'EOF'`/`<<-EOF`), backticks, escaped chars, command separators (`&&`/`||`/`;`/`|`/newline). Returns a list of `(start, end)` positions that are actual command locations. `_MERGE_CMD_RE` matches only at those positions. The module-level `_is_authorized` check remains unchanged — authorization is orthogonal to detection.

## Failure Path Test Strategy

### Exception Handling Coverage
- [ ] `agent/pipeline_state.py::derive_from_durable_signals` — any subprocess failure (`gh api`, `git show`) returns `{}` (cold-Redis equivalent) rather than raising. Test: mock `subprocess.run` to raise; assert empty dict returned and a warning is logged.
- [ ] `scripts/baseline_gate.py::format_quarantine_hints` — malformed `_flake_tracker` returns empty hint list, does not crash. Test: pass `_flake_tracker=None` and `_flake_tracker={"foo": "not-a-dict"}`; assert empty list, no exception.
- [ ] `.claude/hooks/validators/validate_merge_guard.py::_extract_executed_commands` — any tokenizer exception falls back to the old bare-prefix behavior (fail-closed: still blocks direct `gh pr merge`). Test: feed a pathological input (mismatched quotes); assert the guard still blocks a bare `gh pr merge 42` afterward.
- [ ] No `except Exception: pass` blocks will be added in this plan. Any try/except has a logged warning.

### Empty/Invalid Input Handling
- [ ] `derive_from_durable_signals` with no PR, no plan file, no review → returns all-`pending` (item 10b in the SDLC router already handles this case by dispatching `/do-merge` anyway).
- [ ] `uv lock --locked` pre-commit phase with no `pyproject.toml` → skips with a warning.
- [ ] Commit-SHA filter with zero comments → returns empty (matches current behavior: "No '## Review:' comment found" → GATES_FAILED).
- [ ] Baseline decay with missing `_decay_tracker` field → treats as fresh (counter starts at 0).

## Rabbit Holes

- **Rewriting `PipelineStateMachine` to use durable-signals as primary**: TEMPTING but OUT OF SCOPE. Durable signals are a fallback for cold-Redis cases; Redis remains the fast path. #704's primary-signal contract is preserved. Do not conflate "fallback" with "primary."
- **Dismissing stale reviews via the GitHub branch-protection setting**: we cannot rely on a setting across all PRs, and the fix at the `/do-merge` layer is cheaper and portable. No action on branch protection.
- **Adding pytest-retry or pytest-rerunfailures**: `/do-test`'s existing retry is sufficient. Adding another retry layer at the gate duplicates work without closing gaps.
- **Writing a general-purpose shell lexer**: item 7's tokeniser needs only enough to identify command positions — full POSIX shell lexing is out of scope. The tokeniser is allowed to be conservative (when in doubt, treat as a quoted context = no match = no block). Conservative = fail-open on the guard, which is the WRONG direction. So: when in doubt, treat as an actual command position (fail-closed). This is stated explicitly in the implementation notes.

**Correction on previous rabbit-hole note**: The tokeniser's default-ambiguous policy is **fail-closed (treat as actual command)**. A false positive (wrong block on a quoted reference) is annoying; a false negative (missed block on a real merge) defeats the guard. When in doubt, block.

- **Centralising `main_test_baseline.json` in git**: explicitly out of scope per `docs/features/merge-gate-baseline.md` ("Data ownership"). Per-machine is deliberate.
- **Making the PM auto-decide "is this a unique architect-level question?"**: item 5's scope is "documented recovery for known gate-failure categories." LLM judgement about novelty is not added here — the G4 guard + explicit 3-dispatch cap handles the escalation boundary.

## Risks

### Risk 1: Durable-signal fallback disagrees with a stage that actually hasn't run
**Impact:** A stage is marked `completed` in the derived view when it truly wasn't executed (e.g., DOCS derivation says "completed" because a random `docs/` file was touched, but `/do-docs` never ran and the plan's `## Documentation` checkbox is not ticked).
**Mitigation:** Item 1's DOCS derivation uses TWO signals in AND: (a) at least one `docs/` file in the PR diff OR (b) the plan's `## Documentation` section has all checkboxes ticked. We take the stricter interpretation on the narrow fallback path. On any ambiguous case, fall back to `pending` (which triggers `/do-docs` dispatch via existing router logic — no silent skip).

### Risk 2: Commit-SHA filter drops a legitimate review on a no-op push
**Impact:** A dev pushes a trivial comment fix or a branch-reset that writes a new committer date but does not change the diff; the filter then drops a previously valid APPROVED review.
**Mitigation:** This is the correct conservative behavior — any push resets the commit date, and the semantic "reviewer saw exactly the commits being merged" is preserved. If the dev wants to preserve the review, they can re-request review in the GitHub UI. This matches GitHub's own "Dismiss stale approvals" behavior. Document the tradeoff in `docs/features/self-healing-merge-gate.md`.

### Risk 3: `uv lock --locked` in pre-commit slows down every commit
**Impact:** Developer productivity loss if the check is slow.
**Mitigation:** `uv lock --locked` is a read-only consistency check — it does NOT solve the lockfile, only compares. On a typical repo it runs in <200ms. Measure on our repo during build; if >1s, add a cache key based on `pyproject.toml` mtime. Skip the check entirely for commits that do not modify `pyproject.toml` (check with `git diff --cached --name-only`).

### Risk 4: Baseline decay ages out a genuinely-fixed-then-reintroduced failure
**Impact:** A regression test was failing on main, someone fixed it, it passed for 5 merges (decays out), then a new commit reintroduces the failure — without the baseline entry, the gate correctly blocks the merge. This is the INTENDED behavior; the risk is only if the reintroduction should have been caught earlier. But "earlier" would require a different tool (CI smoke suite), not the baseline.
**Mitigation:** None needed — the failure *will* be caught at merge, just not pre-merge-on-main. `DECAY_THRESHOLD = 5` is conservative. Make it tunable in `data/main_test_baseline.json` top-level `_decay_threshold: N` so per-project tuning is possible without a code change.

### Risk 5: Merge-guard tokeniser has a parse bug that lets a real merge through
**Impact:** The guard's entire purpose defeated on one class of bug.
**Mitigation:** Fail-closed default (any tokeniser exception → old bare-prefix behavior, which still blocks direct merges). Seven unit tests covering all observed false-positive patterns AND the direct-merge case. Regression test (tests/unit/test_validate_merge_guard.py::test_tokenizer_failure_fails_closed) explicitly feeds a pathological string and asserts a real `gh pr merge 42` still blocks afterward.

### Risk 6: PM persona Gate-Recovery section cascades into loop behavior
**Impact:** PM reads the recovery rule, dispatches remediation, `/do-merge` fails again, loops indefinitely.
**Mitigation:** G4 oscillation guard (already live at `.claude/skills/sdlc/SKILL.md:151`) caps same-skill dispatch at 3 without state change. Any gate-recovery loop that exceeds this converges to human escalation. Unit test (tests/unit/test_pm_persona_guards.py) asserts the persona section explicitly references the G4 cap.

## Race Conditions

No race conditions identified — all operations are synchronous and file/process-based. The `main_test_baseline.json` read-then-write in item 4 runs only inside `/do-merge`'s post-merge block, which holds the `data/merge_authorized_{pr}` file as an effective lock. Only one `/do-merge` runs per PR at a time (enforced by the authorization file's presence/absence contract). Redis reads in items 1 and the existing state machine are best-effort fallback paths — a concurrent write to `stage_states` after our read does not matter; we never write back from the derivation function.

## Open Questions

1. **Default thresholds for baseline decay (N=5) and flake quarantine (M=3)**: These are my conservative picks. Should they be lower (N=3, M=2) to decay faster and quarantine sooner? Higher values reduce noise but slow the feedback loop. My recommendation is N=5, M=3, configurable via `_decay_threshold` and `_flake_threshold` top-level fields in the baseline JSON.

2. **Should the tokeniser in item 7 also cover `git commit --amend -m "..."` and `git notes add -m "..."`?**: The issue lists the specific commands we know about. A truly general tokeniser handles these automatically. My current design IS general (tokenises any shell command), so these are covered — but should any non-`gh`/`git`/`echo`/`printf` command that mentions `gh pr merge` in an argument string also be exempted? My answer is yes: the tokeniser is pattern-agnostic. Confirming the approach.

3. **Do we want the pre-commit phase (item 3) to auto-apply the fix (`uv lock`) instead of only blocking?**: The issue specifies "block the commit" — I kept that. But ruff's phase auto-fixes and re-stages. Should `uv lock` follow the same pattern (run `uv lock`, stage the updated lockfile, re-run the commit)? My recommendation is NO: `uv lock` without `--locked` has network cost (resolves deps), which is not acceptable in a pre-commit hook. Keep as read-only block with a fix hint.

4. **Should the merge-troubleshooting playbook (item 6) include a section on "Redis was cleared mid-session"?**: This is covered by item 1's fallback, but if the PM hits a weirder variant (partial state: some stages recorded, some missing), the playbook might need a section. My inclination is yes, add a seventh section "Partial Pipeline State." Confirm whether to include.

5. **Is the durable-signal fallback's DOCS derivation too strict?** The current design requires EITHER (a) `docs/` file in PR diff OR (b) all `## Documentation` plan checkboxes ticked. If neither, treat as pending → router dispatches `/do-docs`. Is there a cleaner signal I am missing (e.g., the presence of a `status: docs_complete` frontmatter flag from `/do-docs`)? Checking whether the frontmatter flag is a reliable signal to include alongside the OR conditions.
