# Daily PM Audio Briefing Reflection

## Problem

Every morning a PM has to scan the dashboard, GitHub issue list, and yesterday's merges to know what shipped and what's queued. Voice messages are lower-friction at the start of the day — the PM can listen on the commute or during first coffee. The construction logic for a 30-second executive brief already exists in the `/do-debrief` skill; the missing piece is a scheduler that fires once per day, per project, into the right Telegram group, with content shaped by per-project preferences.

A single global reflection won't work because **different projects care about different content angles** — research projects care about new findings, product projects care about user-facing PRs and bug burn-down, internal-tooling projects care about reliability fixes and dashboard health. The reflection must accept per-project preferences for what to surface vs. drop, and that preference set must be editable as feedback comes in.

**Current behavior:**
- `/do-debrief` exists but is user-invoked only — no scheduler hook.
- No reflection produces audio output today; existing reflections post text to GitHub or trigger PM sessions.
- PMs read text dashboards / issue lists every morning, manually.

**Desired outcome:**
- Each project listed in `projects.json` with a `pm_briefing` block runs a daily reflection at the configured local-time slot.
- The reflection collects the previous day's merges + open `bug` / `upvote` issues, applies the project's content-angle preferences, runs them through the `/do-debrief` construction phases, and queues a voice message to the project's PM Telegram group(s).
- Numbers (issue/PR numbers) **never** appear in the spoken transcript — they go in a written follow-up text message that the same reflection posts after the voice note.
- Per-project content-angle preferences are editable: a PM can hand-edit `projects.json` (v1) to surface or suppress categories, and that preference set persists.

## Solution

### Key Elements

- **`reflections/pm_audio_briefing/__init__.py`** — defines `run()` async callable. Single entry point for the scheduler; iterates `load_local_projects()`, filters those with `pm_briefing.enabled: true`, and per project decides "is now the project's local schedule slot?" before fanning out.
- **`reflections/pm_audio_briefing/collector.py`** — pure-Python collector. Pulls raw signals (yesterday's merges from `git log`, open issues with configured labels via `gh`, recent PRs) using the project's `working_directory` and `github.org/repo`. Filters by `angles.include` / `angles.exclude`.
- **`reflections/pm_audio_briefing/builder.py`** — composes the brief transcript. Implements the deterministic phases of `/do-debrief` (Collect already done by collector → Categorize → Pass A draft → Pass B cut). Pass A enforces the "no numbers in audio" rule by stripping or substantively-rephrasing any multi-digit identifier. Returns `(audio_transcript, written_followup_markdown)`.
- **`reflections/pm_audio_briefing/delivery.py`** — synthesizes audio with `valor-tts` and delivers via `valor-telegram` (subprocess shell-out — same pattern as `/do-debrief`'s SKILL.md). Posts the written follow-up as a separate text message after the voice-note send returns success.
- **One `Reflection` record per project** — `Reflection.get_or_create(name=f"pm-audio-briefing-{project_key}")`. Lifecycle: `mark_started` at fan-out per project, `mark_completed(duration, error)` after delivery (or per-project failure isolation — one project failing must not break the others).
- **`config/reflections.yaml` entry** — single registry entry `pm-audio-briefing` with `interval: 60` (the scheduler tick is 60s; the callable internally checks each project's schedule slot and skips when not due). `execution_type: function`, `callable: reflections.pm_audio_briefing.run`, `priority: low`.
- **`projects.json` schema extension** — optional per-project `pm_briefing` block (see Resolved Open Questions for shape).

### Flow

`Scheduler tick (60s)` → `pm_audio_briefing.run()` → `load_local_projects()` → per project:
- If `pm_briefing.enabled` is false or absent → skip.
- If now (in project's tz) does not match `schedule` (within the 60s window) → skip silently.
- If yesterday already produced a successful run for this `pm-audio-briefing-{project_key}` (check `Reflection.ran_at` against today's slot) → skip (idempotency).
- Else → `collector.collect(project)` → `builder.build(raw_signals, angles)` → review-gate check (auto-confirm v1, see Resolved Open Questions) → `delivery.send(transcript, followup, groups)` → `mark_completed`.

### Technical Approach

**Resolved Open Questions (from issue):**

1. **Schedule placement (issue Q1):** **Per-project, in `projects.json`.** The reflections registry declares one entry; the per-project `schedule` field lives in `projects.<key>.pm_briefing.schedule`. Rationale: schedule is project-content, not infrastructure-content; vault-synced `projects.json` is the right home. The reflections registry stays generic (one entry, fires every minute, internally fans out).
2. **Empty-day handling (issue Q2):** **Per-project default with override.** Default behavior is to send the configured `fallback_message` (predictable cadence). Projects can set `skip_when_empty: true` to silence. Per the issue's wording, the default is "send the fallback".
3. **Multiple PM groups per project (issue Q3):** **One construction, fan out delivery.** Build the transcript and the written follow-up once per project, then iterate `pm_briefing.groups` and post both messages to each. Avoids LLM/TTS cost duplication and keeps cross-group consistency.
4. **Review gate vs auto-confirm (issue Q4):** **v1 ships auto-confirm.** A daily auto-reflection cannot block on a human, and option (b) "react 👍 to publish" introduces a 5-minute uncertainty window in the chat that competes with the cadence promise. v1 ships auto-confirm with strong observable safeguards: (i) the transcript is logged to `logs/reflections/pm-audio-briefing-{project_key}-{date}.txt` BEFORE TTS so it's auditable post-hoc; (ii) Pass B's word-count gate (55-80 words) is a hard guard — out-of-band transcripts are dropped with an error in the Reflection state record; (iii) the "no numbers in audio" rule is enforced by a regex pre-TTS check that fails the run if any 3+ digit number survives Pass A. Option (b) is documented in No-Gos as a v2 enhancement.
5. **Construction logic reuse (issue Q5):** **Hybrid.** Collect + Categorize + the deterministic structural Pass B cut are pure Python in `builder.py`. The Pass A draft (the LLM transformation that turns raw items into spoken prose with default-and-confirm phrasing and no numbers) calls Anthropic via `agent.sdk_client.get_anthropic_client()` directly inside the callable — no full PM session, no agent execution_type. This is cheaper, deterministic in shape, and matches the existing function-mode reflections. The user-invocable `/do-debrief` skill stays unchanged; the markdown skill and the Python builder share the same construction principles by virtue of both being authored against `.claude/skills/do-debrief/SKILL.md` (which is the source of truth for the rules).
6. **Numbers-in-followup format (issue Q6):** **Bullet list, sectioned by category.** The written follow-up mirrors the audio's structure with three sections — `## Decisions`, `## Heads-up`, `## FYIs` — each with bullets that include the issue/PR number, link, and a one-line description. Example: `- [#1197](https://github.com/tomcounsell/ai/issues/1197) — daily audio briefing wiring`.
7. **Cross-machine ownership (issue Q7):** **Existing single-machine-ownership invariant covers this.** `reflections/utils.py:load_local_projects()` already filters projects to those whose `machine` field matches `scutil --get ComputerName`. The callable iterates that filtered list, so a non-owning machine simply has zero projects to brief and the run is a no-op. No new plumbing.
8. **Feedback capture for content angles (issue Q8):** **v1 ships hand-edited `projects.json`.** PMs edit the `pm_briefing.angles.include` / `exclude` lists directly in the vault. The plan reserves `/do-briefing-feedback` (natural-language feedback skill) as a v2 enhancement, documented in No-Gos.

**Implementation approach:**

- **Module layout:** `reflections/pm_audio_briefing/` (package with `__init__.py`, `collector.py`, `builder.py`, `delivery.py`, plus `tests/`). The callable invoked by the scheduler is `reflections.pm_audio_briefing.run`. Package layout (vs. single file) makes the Pass A LLM call, the deterministic Pass B, and the side-effecting delivery testable in isolation.
- **Schedule slot detection:** "Now (in project's tz) is within the 60s window of `schedule`" — concretely, the scheduler ticks every 60s; the callable computes the project's local time and tests `local_time.hour == schedule_hour and local_time.minute == schedule_minute`. Combined with idempotency (don't fire twice for the same project on the same date), this gives at-most-once daily delivery without needing a separate cron.
- **Per-project failure isolation:** The top-level `run()` wraps each project's processing in `try/except` and writes per-project errors to that project's `Reflection` record; one project's failure never aborts the others. The scheduler-level `mark_completed` records aggregate success/failure (success if ≥1 project succeeded, error otherwise).
- **Pass A "no numbers" enforcement:** Two-layer guard. Layer 1: the LLM prompt explicitly forbids reciting any multi-digit identifier. Layer 2: a regex post-check (`re.search(r"\d{3,}", transcript)`) fails the run before TTS if any 3-digit-or-longer number survives. The run is recorded as `error` in the Reflection state, the audio is NOT sent, and the written follow-up is NOT sent (don't ship a half-brief).
- **Idempotency anchor:** The `Reflection.ran_at` field is a unix timestamp of the last run. The callable computes `last_run_date = date.fromtimestamp(reflection.ran_at, tz=project_tz)` and skips if it equals today's date in the project's timezone.
- **Subprocess vs import for TTS / telegram:** Shell out to `valor-tts` and `valor-telegram` via `subprocess.run` — matches the `/do-debrief` SKILL.md pattern, avoids cross-coupling the reflections package to the CLIs' internal modules, and gets the cleanup-after-send semantics for free.

**Concrete `pm_briefing` schema:**

```json
"pm_briefing": {
  "enabled": true,
  "schedule": "08:30",
  "timezone": "America/Los_Angeles",
  "groups": ["PM: Valor"],
  "angles": {
    "include": ["merges", "open-bugs", "upvote-queue", "dashboard-health"],
    "exclude": ["plan-only-commits", "lockfile-bumps"]
  },
  "skip_when_empty": false,
  "fallback_message": "Nothing shipped yesterday — three things queued for today."
}
```

`angles.include` enumerates well-known categories the collector knows how to gather. v1 categories: `merges`, `open-bugs`, `upvote-queue`, `dashboard-health`, `experiments`, `plan-only-commits`, `lockfile-bumps`. Categories not in `include` are not collected; categories in `exclude` are filtered after collection (collector ergonomics — `exclude` is the "subtle filter" lane that runs over already-collected items).

## Risks

### Risk 1: `valor-tts` produces a transcript that violates the "no numbers" rule despite the LLM prompt forbidding it
**Impact:** Audio brief reads "issue eleven ninety-seven" or "PR three sixty-three" aloud, violating Tom's audio-briefs-no-numbers rule and degrading listener experience.
**Mitigation:** Layered guard. Layer 1 — explicit prompt instruction. Layer 2 — pre-TTS regex check (`\d{3,}`) hard-fails the run before synthesis. Layer 3 — the daily test suite includes a regression test feeding a known-numerous payload through builder.py and asserting the guard fires. Layer 4 — Reflection record's `last_error` surfaces the failure on the dashboard within minutes.

### Risk 2: Daily fan-out posts duplicate briefs after a worker restart mid-execution
**Impact:** PM gets two voice notes 30 seconds apart.
**Mitigation:** Idempotency by `(project_key, date_in_project_tz)`. Check `Reflection.ran_at` against today's date in the project's tz before doing any work for that project. The check is cheap (Redis `get_or_create`) and runs first in each per-project loop body.

### Risk 3: Per-project schedule slot misses fire when worker is paused or hibernating
**Impact:** A briefing scheduled for 08:30 in a project's tz misses if the worker was hibernating between 08:30:00 and 08:30:59.
**Mitigation:** Widen the slot match — match if `now.hour == schedule.hour and now.minute >= schedule.minute and now.minute < schedule.minute + 5` AND idempotency-by-date hasn't fired today. Combined, this gives a 5-minute window for the daily shot without risking duplicates.

### Risk 4: `gh` CLI rate limits during morning fan-out across many projects
**Impact:** `gh issue list` calls fail; collector returns partial signals; brief is wrong.
**Mitigation:** v1 limits the per-project signal pull to 4 `gh` calls (yesterday's PRs, open bugs, upvote queue, recent commits). If the machine has, say, 5 projects all firing at 08:30, that's 20 calls — well below `gh`'s 5000/hr authenticated limit. Document the math in Update System. If a `gh` call fails, the per-project run records error and skips; other projects continue.

### Risk 5: `pm_briefing` config drift between vault and machines (single-machine-ownership invariant)
**Impact:** A project's briefing fires from the wrong machine, or doesn't fire at all because the owning machine doesn't see the config.
**Mitigation:** No new invariant — re-use the existing `projects.<key>.machine` field. `reflections/utils.py:load_local_projects()` already does the filter. Add an explicit assertion in the callable's setup logging: "loaded N projects with pm_briefing.enabled=true on this machine ($COMPUTERNAME)". This makes drift visible in `logs/worker.log`.

## Race Conditions

### Race 1: Two daily ticks land within the same schedule minute
**Location:** `reflections/pm_audio_briefing/__init__.py::run()` per-project loop.
**Trigger:** Scheduler ticks at T=08:30:00 and again at T=08:30:58 — both inside the schedule slot for a project at 08:30.
**Data prerequisite:** `Reflection.ran_at` for `pm-audio-briefing-{project_key}` must be set BEFORE the first tick releases the lock, so the second tick observes today's date and skips.
**State prerequisite:** `mark_started()` writes the start timestamp atomically in Redis via `model.save()`. Subsequent ticks read it via `get_or_create` and compute `date.fromtimestamp(reflection.ran_at, tz=project_tz) == today_in_tz` to short-circuit.
**Mitigation:** Per-project lock-by-record. The per-project loop acquires the lock by reading Reflection, computing date equality, and (only if not already today) calling `mark_started()` immediately. The rest of the work (collect/build/deliver) happens after the lock is held. If two ticks race the read, one will lose the date-equality check after the other writes. Belt-and-suspenders: a Redis SETNX-style flag is overkill for v1 because daily cadence + 60s tick + per-project work taking >60s is almost impossible (the actual collect/build/deliver loop is sub-30s).

### Race 2: Worker crashes mid-delivery — voice-note sent, written follow-up not sent
**Location:** `reflections/pm_audio_briefing/delivery.py::send()`.
**Trigger:** Worker crash (segfault, OOM, kill) between voice-note success and written-follow-up dispatch.
**Data prerequisite:** None — both messages are independent.
**State prerequisite:** None.
**Mitigation:** Idempotency by date. On worker restart, the next tick sees `Reflection.ran_at == today` and skips. The PM gets the voice-note but no written follow-up that day. Acceptable v1 tradeoff — voice-note is the primary brief; written follow-up is a backstop. v2 enhancement: split the Reflection state into two phases (`audio_sent_at`, `followup_sent_at`) and resume from the right phase on restart. Documented in No-Gos as v2.

## Open Questions

The eight open questions raised in the issue are resolved in the Technical Approach section. The three remaining items below are now resolved as well:

1. **TTS voice for PM briefings.**
   **Resolved:** Expose `pm_briefing.voice` as an optional per-project field. Default to `am_michael` when absent (matches `KOKORO_DEFAULT_VOICE` in `tools/tts/__init__.py:68` and the `/do-debrief` default at `.claude/skills/do-debrief/SKILL.md:21`). The callable passes `--voice <value>` to `valor-tts` only when the field is set; otherwise lets `valor-tts` apply its own default. Config schema gains an optional `voice: "am_michael"` line in the example block.

2. **Schedule-slot widen window.**
   **Resolved:** 5-minute slack is acceptable. A daily executive brief is not a real-time signal — the listener cares that it arrives in the morning, not which exact minute. The widen window protects against worker hibernation / circuit-pause windows (which can pause processing for 60-180s). Combined with the per-date idempotency anchor, the 5-minute window cannot produce duplicates. Implementation per Risk 3: `now.minute >= schedule.minute and now.minute < schedule.minute + 5` AND `last_run_date != today_in_project_tz`.

3. **Per-project disabled-projects logging.**
   **Resolved:** Startup-only (one log line on the first tick after process start), matching the convention in `reflections/maintenance.py` and `reflections/auditing.py` where projects are inspected silently per tick. Implementation: a module-level `_startup_logged` flag (set False at import, flipped True after first call) gates a single `logger.info(f"pm-audio-briefing: loaded {N} projects with pm_briefing.enabled=true on $COMPUTERNAME (skipped {M} without config)")`. Subsequent ticks log nothing about disabled-project enumeration; only per-project errors surface in the log.
