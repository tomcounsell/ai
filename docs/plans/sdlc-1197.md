# Daily PM Audio Briefing Reflection

## Problem

Every morning a PM has to scan the dashboard, GitHub issue list, and yesterday's merges to know what shipped and what's queued. Voice messages are lower-friction at the start of the day — the PM can listen on the commute or during first coffee. The construction logic for a 30-second executive brief already exists in the `/do-debrief` skill; the missing piece is a scheduler that fires once per day, per project, into the right Telegram group, with content shaped by per-project preferences.

A single global reflection won't work because **different projects care about different content angles** — research projects care about new findings, product projects care about user-facing PRs and bug burn-down, internal-tooling projects care about reliability fixes and dashboard health. The reflection must accept per-project preferences for what to surface vs. drop, and that preference set must be editable as feedback comes in.

**Current behavior:**
- `/do-debrief` exists but is user-invoked only — no scheduler hook.
- No reflection produces audio output today; existing reflections post text to GitHub or trigger PM sessions.
- PMs read text dashboards / issue lists every morning, manually.

**Desired outcome:**
- Each project listed in `projects.json` with a `pm_briefing` block runs a daily reflection at the configured local-time slot.
- The reflection collects the previous day's merges + open `bug` / `upvote` issues, applies the project's content-angle preferences, runs them through the `/do-debrief` construction phases, and queues a voice message to the project's PM Telegram group(s).
- Numbers (issue/PR numbers) **never** appear in the spoken transcript — they go in a written follow-up text message that the same reflection posts after the voice note.
- Per-project content-angle preferences are editable: a PM can hand-edit `projects.json` (v1) to surface or suppress categories, and that preference set persists.

## Solution

### Key Elements

- **`reflections/pm_audio_briefing/__init__.py`** — defines `run()` async callable. Single entry point for the scheduler; iterates `load_local_projects()`, filters those with `pm_briefing.enabled: true`, and per project decides "is now the project's local schedule slot?" before fanning out.
- **`reflections/pm_audio_briefing/collector.py`** — pure-Python collector. Pulls raw signals (yesterday's merges from `git log`, open issues with configured labels via `gh`) using the project's `working_directory` and `github.org/repo`. v1 ships **3 categories only** (`merges`, `open-bugs`, `upvote-queue`) per the issue's minimum spec; additional categories are added on demand as one function + one test (see C9 fix). Implementation: `_COLLECTORS = {"merges": _collect_merges, "open-bugs": _collect_open_bugs, "upvote-queue": _collect_upvote_queue}`. Unknown categories in `angles.include` are logged-warned and skipped, not raised. Filters by `angles.include` / `angles.exclude`.
- **`reflections/pm_audio_briefing/builder.py`** — composes the brief transcript. Implements the deterministic phases of `/do-debrief` (Collect already done by collector → Categorize → Pass A draft → Pass B cut). Pass A's prompt forbids forward-looking commitments (no "we will", "I'll push", "pushing to", "unless you want it sooner") since auto-confirm removes the human safety check (see C5 fix). Post-LLM regex `\b(?:issue|pr|#)\s*\d{2,}\b` (case-insensitive) catches "issue 1197", "#1197", "PR 1197" but allows legitimate "$500", "250 users", and product version numbers (see C4 fix). Pass B is a deterministic word-count cut (target 55-80 words) that also verifies the first sentence is a decision/heads-up, not setup (see N5). Returns `(audio_transcript, written_followup_markdown)`.
- **`reflections/pm_audio_briefing/delivery.py`** — synthesizes audio via the **direct Python `tools.tts.synthesize()` call** (no subprocess) and queues the voice-note via the **Redis outbox payload pattern from `tools/valor_telegram.py:780-816`** (no subprocess). After the voice-note enqueue succeeds, posts the written follow-up via the same outbox pattern (a separate payload). Subprocess is reserved for `gh` only (see C7 fix).
- **One `Reflection` record per project** — `Reflection.get_or_create(name=f"pm-audio-briefing-{project_key}")`. Lifecycle: `mark_started` at fan-out per project, `mark_completed(duration, error)` after delivery (or per-project failure isolation — one project failing must not break the others). The dashboard does NOT auto-discover Redis reflection records; it iterates `config/reflections.yaml` only. To make per-project records render, this plan also extends `ui/data/reflections.py` with prefix-merge logic (see B2 fix in Critique Results).
- **`config/reflections.yaml` entry** — single registry entry `pm-audio-briefing` with `interval: 300` (5 minutes). The 5-minute window comfortably envelopes the per-project fanout (collect + Anthropic + TTS + Telegram-relay enqueue) even with 5+ projects. `execution_type: function`, `callable: reflections.pm_audio_briefing.run`, `priority: low`. The callable internally checks each project's schedule slot and skips when not due.
- **`projects.json` schema extension** — optional per-project `pm_briefing` block (see Resolved Open Questions for shape).

### Flow

`Scheduler tick (every 5 minutes — `interval: 300`)` → `pm_audio_briefing.run()` → `load_local_projects()` → per project:
- If `pm_briefing.enabled` is false or absent → skip.
- If `project.machine` ≠ `scutil --get ComputerName` → skip (B3 explicit machine filter).
- If now (in project's tz) does not match `schedule` within the 5-minute slot widen window → skip silently.
- If today already produced a **successful** run for `pm-audio-briefing-{project_key}` (check `Reflection.ran_at_date == today AND Reflection.last_status == "success"`) → skip (idempotency, see C2). On `last_status == "error"`, allow the run to retry.
- Acquire a per-project per-day Redis CAS lock: `redis.set(f"pm-briefing-lock:{project_key}:{today.isoformat()}", str(time.time()), nx=True, ex=90000)`. If `nx` returns False, another tick beat us → skip. TTL 90000s = 25h so stale locks expire next day. (See C3.)
- Else → `collector.collect(project)` → `builder.build(raw_signals, angles)` → review-gate check (auto-confirm v1, see Resolved Open Questions) → `delivery.send(transcript, followup, groups)` → `mark_completed`.

### Technical Approach

**Resolved Open Questions (from issue):**

1. **Schedule placement (issue Q1):** **Per-project, in `projects.json`.** The reflections registry declares one entry; the per-project `schedule` field lives in `projects.<key>.pm_briefing.schedule`. Rationale: schedule is project-content, not infrastructure-content; vault-synced `projects.json` is the right home. The reflections registry stays generic (one entry, fires every minute, internally fans out).
2. **Empty-day handling (issue Q2):** **Per-project default with override.** Default behavior is to send the configured `fallback_message` (predictable cadence). Projects can set `skip_when_empty: true` to silence. Per the issue's wording, the default is "send the fallback".
3. **Multiple PM groups per project (issue Q3):** **One construction, fan out delivery.** Build the transcript and the written follow-up once per project, then iterate `pm_briefing.groups` and post both messages to each. Avoids LLM/TTS cost duplication and keeps cross-group consistency.
4. **Review gate vs auto-confirm (issue Q4):** **v1 ships auto-confirm.** A daily auto-reflection cannot block on a human, and option (b) "react 👍 to publish" introduces a 5-minute uncertainty window in the chat that competes with the cadence promise. v1 ships auto-confirm with strong observable safeguards: (i) the transcript is logged to `logs/reflections/pm-audio-briefing-{project_key}-{date}.txt` BEFORE TTS so it's auditable post-hoc; (ii) Pass B's word-count gate (55-80 words) is a hard guard — out-of-band transcripts are dropped with an error in the Reflection state record; (iii) the "no numbers in audio" rule is enforced by a regex pre-TTS check that fails the run if any 3+ digit number survives Pass A. Option (b) is documented in No-Gos as a v2 enhancement.
5. **Construction logic reuse (issue Q5):** **Hybrid.** Collect + Categorize + the deterministic structural Pass B cut are pure Python in `builder.py`. The Pass A draft (the LLM transformation that turns raw items into spoken prose with default-and-confirm phrasing and no numbers) calls Anthropic directly inside the callable — no full PM session, no agent execution_type. **Use the same pattern as `reflections/utils.py:186-194`**: `import anthropic; client = anthropic.Anthropic(); response = client.messages.create(model=HAIKU, max_tokens=2048, messages=[...])`. Wrap the import in `try/except ImportError` and the call in `try/except Exception` exactly like `run_llm_reflection`. Pull `HAIKU` from `config.models`. **Do NOT reference `agent.sdk_client.get_anthropic_client()` — it does not exist.** This is cheaper, deterministic in shape, and matches the existing function-mode reflections. The user-invocable `/do-debrief` skill stays unchanged; the markdown skill and the Python builder share the same construction principles by virtue of both being authored against `.claude/skills/do-debrief/SKILL.md` (which is the source of truth for the rules).
6. **Numbers-in-followup format (issue Q6):** **Bullet list, sectioned by category.** The written follow-up mirrors the audio's structure with three sections — `## Decisions`, `## Heads-up`, `## FYIs` — each with bullets that include the issue/PR number, link, and a one-line description. Example: `- [#1197](https://github.com/tomcounsell/ai/issues/1197) — daily audio briefing wiring`.
7. **Cross-machine ownership (issue Q7):** **Filter explicitly inside the callable.** `reflections/utils.py:load_local_projects()` filters by `wd.exists()` only — it does NOT filter by ComputerName (verified at `reflections/utils.py:37-65`). On a multi-machine setup with a shared checkout, both machines would otherwise fire the briefing. The callable must add an explicit filter at the top of its per-project loop:

```python
import subprocess
this_machine = subprocess.check_output(["scutil", "--get", "ComputerName"], text=True).strip()
# inside per-project loop:
if (project.get("machine") or "").strip() != this_machine:
    continue  # logged at DEBUG once per startup; accepted projects logged at INFO once per startup
```

This is a per-callable invariant, not a change to `load_local_projects()`. A separate hardening item (fixing the helper itself) would be an independent hotfix and is out of scope for this plan.
8. **Feedback capture for content angles (issue Q8):** **v1 ships hand-edited `projects.json`.** PMs edit the `pm_briefing.angles.include` / `exclude` lists directly in the vault. The plan reserves `/do-briefing-feedback` (natural-language feedback skill) as a v2 enhancement, documented in No-Gos.

**Implementation approach:**

- **Module layout:** `reflections/pm_audio_briefing/` (package with `__init__.py`, `collector.py`, `builder.py`, `delivery.py`, plus `tests/`). The callable invoked by the scheduler is `reflections.pm_audio_briefing.run`. Package layout (vs. single file) makes the Pass A LLM call, the deterministic Pass B, and the side-effecting delivery testable in isolation.
- **Schedule slot detection:** "Now (in project's tz) is within the 5-minute slot widen window of `schedule`" — concretely, the scheduler ticks every 5 minutes (`interval: 300`); the callable computes the project's local time and tests `local_time.hour == schedule_hour and local_time.minute >= schedule_minute and local_time.minute < schedule_minute + 5`. The 5-minute widen survives worker hibernation / circuit-pause windows; combined with idempotency (don't fire twice for the same project on the same date), this gives at-most-once daily delivery without needing a separate cron.
- **Registry-interval semantics:** `interval: 300` means "this entry is eligible to fire on every scheduler tick if at least 300s have elapsed since last run." It is NOT a precise scheduler — it's a coarse gate. The per-project schedule-slot detection inside the callable is the real "fire today / not today" gate. (See N2.)
- **Machine-ownership filter:** Per B3 (Resolved Open Questions item 7) the callable does its own `scutil --get ComputerName` filter at the top of the per-project loop. Skipped projects are logged at DEBUG once per startup; accepted projects logged at INFO once per startup (see Resolved Open Question 3).
- **Per-project failure isolation:** The top-level `run()` wraps each project's processing in `try/except` and writes per-project errors to that project's `Reflection` record; one project's failure never aborts the others. The scheduler-level `mark_completed` records aggregate success/failure (success if ≥1 project succeeded, error otherwise). Failures do NOT lock the daily slot — the next scheduler tick that lands inside the 5-minute slot will retry on `last_status == "error"`. (See C2.)
- **Pass A "no numbers" enforcement:** Two-layer guard. Layer 1: the LLM prompt explicitly forbids reciting issue numbers, PR numbers, and `#`-prefixed identifiers, AND forbids forward-looking commitments ("we will", "I'll push", "pushing to", "unless you want it sooner"). Layer 2: a tightened regex post-check `re.search(r"\b(?:issue|pr|#)\s*\d{2,}\b", transcript, re.IGNORECASE)` catches "issue 1197", "#1197", "PR 1197" but allows "$500", "250 users", and product version numbers like "v3.5.2". Failure of the regex check fails the run before TTS — the run is recorded as `error` in the Reflection state, the audio is NOT sent, and the written follow-up is NOT sent (don't ship a half-brief). (See C4 + C5.)
- **Idempotency anchor:** The `Reflection.ran_at` field is a unix timestamp of the last run, paired with `last_status`. The callable computes `last_run_date = date.fromtimestamp(reflection.ran_at, tz=project_tz)` and skips ONLY if `last_run_date == today_in_tz AND last_status == "success"`. Belt-and-suspenders: a Redis SETNX lock keyed by `(project_key, today.isoformat())` (TTL 25h) prevents two ticks landing in the same minute from both proceeding past the lock. (See C2 + C3.)
- **Direct Python imports vs subprocess for TTS/Telegram:** Use `from tools.tts import synthesize` and replicate the Redis-outbox payload pattern from `tools/valor_telegram.py:780-816` directly (`payload = {...}; r.rpush(f"telegram:outbox:{session_id}", json.dumps(payload))`). This avoids 100-300ms × 2-3 calls per project per day of subprocess overhead. Subprocess is retained ONLY for `gh` calls (no Python wrapper available). (See C7.)
- **DRY_RUN env hatch:** When `DRY_RUN=1` is set, the callable runs collect → build but skips `delivery.send()` and writes the would-be transcript + follow-up to `logs/reflections/pm-audio-briefing-{project_key}-{date}.txt` for inspection. (See N3.)

**Concrete `pm_briefing` schema:**

```json
"pm_briefing": {
  "enabled": true,
  "schedule": "08:30",
  "timezone": "America/Los_Angeles",
  "target_groups": ["PM: Valor"],
  "angles": {
    "include": ["merges", "open-bugs", "upvote-queue"],
    "exclude": ["plan-only-commits", "lockfile-bumps"]
  },
  "skip_when_empty": false,
  "fallback_message": "Nothing shipped yesterday — three things queued for today."
}
```

`angles.include` enumerates well-known categories the collector knows how to gather. **v1 ships 3 categories** (`merges`, `open-bugs`, `upvote-queue`) per the issue's minimum spec. New categories are added on demand as one function in `_COLLECTORS` plus one test (see C9). Categories not in `include` are not collected; categories in `exclude` are filtered after collection (collector ergonomics — `exclude` is the "subtle filter" lane that runs over already-collected items). Unknown categories in `include` are log-warned and skipped, not raised.

**Field naming note (N1):** the briefing's chat-targets field is `target_groups` (list of group names), NOT `groups` — `groups` already exists as a dict shape under `projects.<key>.telegram.groups` and reusing the name here would create a collision in any code that walks the project config generically.

## Failure Path Test Strategy

### Exception Handling Coverage

- [ ] The new `reflections/pm_audio_briefing/__init__.py` `run()` wraps each per-project block in `try/except Exception`. Each except block must record the error to the project's Reflection state via `mark_completed(duration, error=str(e))`, NOT swallow silently. Test: simulate a `RuntimeError` in `collector.collect()` for project A; assert project B still completes and project A's Reflection record has `last_status == "error"` and `last_error` is non-None.
- [ ] `delivery.send()` raising from `tools.tts.synthesize` (e.g. backend unavailable) must propagate as an exception and the written follow-up Redis payload must NOT be enqueued. Test: monkeypatch `tools.tts.synthesize` to raise; mock `r.rpush`; assert delivery raises and `r.rpush` was called zero times.
- [ ] The TTS pre-flight regex check (the "no numbers" guard) raises `BriefingNumbersDetectedError` on a transcript matching `\b(?:issue|pr|#)\s*\d{2,}\b`. Test: feed a transcript containing "issue 1197" through the guard; assert the exception is raised, the run records `error`, and no audio is synthesized.
- [ ] The Redis SETNX lock prevents duplicate fan-out: simulate two concurrent calls to `run()` for the same project and date; assert only one calls `delivery.send()`.

### Empty/Invalid Input Handling

- [ ] Project with `pm_briefing` block but `enabled: false` → callable skips silently, no Reflection record update.
- [ ] Project with `pm_briefing.enabled: true` but `target_groups: []` → callable raises a config error during a startup config-validation pass; the daily run records the error once and skips for that project until config is fixed.
- [ ] Project with `machine` ≠ this machine's `scutil --get ComputerName` → callable skips silently (B3 explicit filter), no Reflection record update.
- [ ] Project with empty `angles.include` → no signals collected; if `skip_when_empty: false`, the fallback message is delivered; if `skip_when_empty: true`, the run is recorded as `success` with a noop note.
- [ ] Yesterday produced zero merges and zero matching issues → empty-day handling fires per the project's `skip_when_empty` setting.
- [ ] `tools.tts.synthesize` returns silence (zero-byte OGG) → delivery short-circuits and records error; written follow-up NOT enqueued (we don't ship one half of a brief).

### Error State Rendering

- [ ] On any per-project failure, the Reflection record's `last_error` shows in the dashboard at `/dashboard.json` (existing reflections dashboard surface) — verify the dashboard renderer doesn't blow up on per-project naming (`pm-audio-briefing-{project_key}`).
- [ ] Audio synthesis failure must surface as a written-only failure notice in the PM group: post a single text message saying "Daily briefing failed: TTS unavailable" rather than silently dropping. (Tradeoff: one noisy failure beats a silent one — PMs notice when the brief stops arriving.)

## Rabbit Holes

- **Building a generic "scheduled audio brief" framework.** Tempting because the construction logic could be reused for other personas (Dev briefings, Teammate briefings). v1 ships PM-only; if a second persona needs this, refactor on demand.
- **Replacing `/do-debrief`'s markdown skill with a Python module the skill imports.** Tempting because it would unify construction logic. The cost is a bigger refactor — the markdown skill is intentionally human-authored prose with phrasing rules that don't translate to Python directly. The plan keeps both surfaces and shares the rules document (`.claude/skills/do-debrief/SKILL.md`) as the contract.

## Risks

### Risk 1: Pass A produces a transcript that violates the "no numbers" rule despite the LLM prompt forbidding it
**Impact:** Audio brief reads "issue eleven ninety-seven" or "PR three sixty-three" aloud, violating Tom's audio-briefs-no-numbers rule and degrading listener experience.
**Mitigation:** Layered guard. Layer 1 — explicit prompt instruction (forbids issue/PR numbers AND forward-looking commitments per C5). Layer 2 — pre-TTS regex check `\b(?:issue|pr|#)\s*\d{2,}\b` (case-insensitive) hard-fails the run before synthesis. The tightened pattern catches "issue 1197", "#1197", "PR 1197" but allows legitimate "$500", "250 users", "v3.5.2" (per C4). Layer 3 — the daily test suite includes a regression test feeding a known-numerous payload through builder.py and asserting the guard fires. Layer 4 — Reflection record's `last_error` surfaces the failure on the dashboard within minutes (now actually rendered per B2's prefix-merge).

### Risk 2: Daily fan-out posts duplicate briefs after a worker restart mid-execution
**Impact:** PM gets two voice notes 30 seconds apart.
**Mitigation:** Two layers per C2 + C3. (a) Idempotency by `(project_key, date_in_project_tz, last_status == "success")`. Check Reflection record before doing any work; on `last_status == "error"` allow retry (don't lock the daily slot to a failure). (b) Redis SETNX lock keyed by `(project_key, today.isoformat())` with TTL 90000s — atomic gate against same-instant racers. Both are kept; SETNX is the within-tick atomicity primitive, the Reflection record is the cross-restart durable signal.

### Risk 3: Per-project schedule slot misses fire when worker is paused or hibernating
**Impact:** A briefing scheduled for 08:30 in a project's tz misses if the worker was hibernating during the slot.
**Mitigation:** Two-layer. (a) The registry entry's `interval: 300` (5 minutes) means the scheduler is eligible to fire `pm_audio_briefing.run()` every 5 minutes — comfortably more often than the per-project slot widen. The 5-min interval also gives the per-project fanout (collect + Anthropic + TTS + Telegram-relay enqueue × N projects) plenty of headroom before the next tick, avoiding the "appeared stuck (exceeded 2x interval)" reset. (b) The callable's per-project slot match is `now.hour == schedule.hour and now.minute >= schedule.minute and now.minute < schedule.minute + 5` AND idempotency-by-date `+success` hasn't fired today. Combined, this gives a 5-minute window for the daily shot without risking duplicates.

### Risk 4: `gh` CLI rate limits during morning fan-out across many projects
**Impact:** `gh issue list` calls fail; collector returns partial signals; brief is wrong.
**Mitigation:** v1 ships only 3 categories (`merges`, `open-bugs`, `upvote-queue`), so the per-project signal pull is **at most 3 `gh` calls per project per morning, per machine**. If the owning machine has, say, 5 projects all firing in the same morning, that's **15 `gh` calls per machine per morning, regardless of repo overlap** (each project is filtered to its own repo via `working_directory` + `github.org/repo`) — well below `gh`'s 5000/hr authenticated limit. Document the math in Update System. If a `gh` call fails, the per-project run records error and the next 5-minute tick retries (failures don't lock the daily slot, per C2); other projects continue. (See N4.)

### Risk 5: `pm_briefing` config drift between vault and machines (single-machine-ownership invariant)
**Impact:** A project's briefing fires from the wrong machine, or doesn't fire at all because the owning machine doesn't see the config.
**Mitigation:** Re-use the existing `projects.<key>.machine` field — but **the callable must do the filter itself** (B3): `reflections/utils.py:load_local_projects()` filters by `wd.exists()` only, not by ComputerName. Inside the callable, `this_machine = scutil --get ComputerName` is captured once, and each project is gated on `project.get("machine") == this_machine` before any work. Startup-only logging records `loaded N projects with pm_briefing.enabled=true on this machine ($COMPUTERNAME) (skipped M without config or owned by other machines)`. This makes drift visible in `logs/worker.log`.

## Race Conditions

### Race 1: Two ticks land within the same per-project schedule slot
**Location:** `reflections/pm_audio_briefing/__init__.py::run()` per-project loop.
**Trigger:** Scheduler ticks at T=08:30:00 and again at T=08:34:58 — both inside the 5-minute schedule slot for a project at 08:30. (With `interval: 300` this is an edge case but theoretically possible if the first tick is in-flight when the second arrives.)
**Data prerequisite:** `Reflection.ran_at` for `pm-audio-briefing-{project_key}` must be set BEFORE the first tick completes, so the second tick observes today's date and skips.
**State prerequisite:** `mark_started()` writes the start timestamp atomically in Redis via `model.save()`. Subsequent ticks read it via `get_or_create` and compute `date.fromtimestamp(reflection.ran_at, tz=project_tz) == today_in_tz AND last_status == "success"` to short-circuit.
**Mitigation (atomic, per C3):** Redis SETNX lock keyed by `(project_key, today_in_tz.isoformat())` is the authoritative gate. `redis.set(f"pm-briefing-lock:{project_key}:{today.isoformat()}", str(time.time()), nx=True, ex=90000)` — TTL 90000s (25h) so locks expire next day. Whichever tick wins the SETNX proceeds; the other sees `nx=False` and skips. The Reflection-record idempotency check runs FIRST (cheap pre-filter for "already done today on success"); the SETNX runs SECOND as the atomic gate against same-instant racers. Both are kept — the Reflection record is the cross-restart durable signal; the SETNX lock is the within-tick atomicity primitive.

### Race 2: Worker crashes mid-delivery — voice-note sent, written follow-up not sent
**Location:** `reflections/pm_audio_briefing/delivery.py::send()`.
**Trigger:** Worker crash (segfault, OOM, kill) between voice-note success and written-follow-up dispatch.
**Data prerequisite:** None — both messages are independent.
**State prerequisite:** None.
**Mitigation:** Idempotency by date. On worker restart, the next tick sees `Reflection.ran_at == today` and skips. The PM gets the voice-note but no written follow-up that day. Acceptable v1 tradeoff — voice-note is the primary brief; written follow-up is a backstop. v2 enhancement: split the Reflection state into two phases (`audio_sent_at`, `followup_sent_at`) and resume from the right phase on restart. Documented in No-Gos as v2.

## Open Questions

The eight open questions raised in the issue are resolved in the Technical Approach section. The three remaining items below are now resolved as well:

1. **TTS voice for PM briefings.**
   **Resolved:** Expose `pm_briefing.voice` as an optional per-project field. Default to `am_michael` when absent (matches `KOKORO_DEFAULT_VOICE` in `tools/tts/__init__.py:68` and the `/do-debrief` default at `.claude/skills/do-debrief/SKILL.md:21`). The callable passes `voice=<value>` to `tools.tts.synthesize()` only when the field is set; otherwise lets the synthesizer apply its own default. Config schema gains an optional `voice: "am_michael"` line in the example block.

2. **Schedule-slot widen window.**
   **Resolved:** 5-minute slack is acceptable. A daily executive brief is not a real-time signal — the listener cares that it arrives in the morning, not which exact minute. The widen window protects against worker hibernation / circuit-pause windows (which can pause processing for 60-180s). Combined with the per-date+success idempotency anchor and the SETNX lock, the 5-minute window cannot produce duplicates. Implementation per Risk 3: `now.minute >= schedule.minute and now.minute < schedule.minute + 5` AND `(last_run_date != today_in_project_tz OR last_status != "success")`.

3. **Per-project disabled-projects logging.**
   **Resolved:** Startup-only (one log line on the first tick after process start), matching the convention in `reflections/maintenance.py` and `reflections/auditing.py` where projects are inspected silently per tick. Implementation: a module-level `_startup_logged` flag (set False at import, flipped True after first call) gates a single `logger.info(f"pm-audio-briefing: loaded {N} projects with pm_briefing.enabled=true on $COMPUTERNAME (skipped {M} without config)")`. Subsequent ticks log nothing about disabled-project enumeration; only per-project errors surface in the log.
