# Schedulable check-in primitive (issue #2139)

**Issue:** [#2139](https://github.com/tomcounsell/ai/issues/2139)
**Slug:** sdlc-2139
**Appetite:** Small — one new CLI subcommand + a promise-gate template line. Additive only.

## Problem

The [promise gate](../features/promise-gate.md) (`bridge/promise_gate.py`) blocks
unevidenced forward promises ("I'll report back when it lands") unless the draft cites a
verifiable autonomous-delivery mechanism (`session_id`, `schedule_id`, PR URL). But **no
runtime primitive exists for a session to create such a mechanism**. Detection and
fulfillment are permanently disconnected: the referenced `reflections_create` MCP tool was
never implemented, and the one runtime self-scheduling path
(`tools/agent_session_scheduler.py schedule --after`) is hard-wired to issue-bound `/sdlc`
with a `dashboard_only` sink. When async work legitimately outlives a turn, a session
either blocks in-turn or says nothing — the gate's escape hatch is unreachable.

## Solution

Add a `checkin` subcommand to `tools/agent_session_scheduler.py` that registers a one-shot
future Eng session with an arbitrary prompt, delivered to the originating chat, at time T,
returning a citable `schedule_id` that satisfies the promise gate's
`_SCHEDULED_DELIVERY_PATTERNS`.

The durable future-fire substrate already exists and is proven: an `AgentSession` created
with `scheduled_at=T` is skipped by `agent/session_pickup.py` until T passes, then popped
and run as a normal Eng session whose output routes to its `chat_id`. The `checkin` command
reuses that substrate — it is `schedule --after` generalized to (a) an arbitrary prompt
instead of `/sdlc {issue_url}`, and (b) delivery to the originating chat via the session's
`chat_id`.

### Behavior

- Input: `--prompt <text>` (required, arbitrary) and exactly one of `--at <ISO>` /
  `--in <duration>` (e.g. `30m`, `2h`, `1d` — parsed by the existing
  `reflection_schedule.parse_every_duration`).
- Optional `--chat-id` (default: `CHAT_ID` env / originating chat), `--project`,
  `--priority`.
- Creates an `AgentSession` (`session_type=eng`, `status=pending`, `scheduled_at=T`,
  `chat_id=<originating chat>`, `message_text=<prompt>`, `sender_name="System (Check-in)"`)
  via `AgentSession.create` (ORM only).
- Registers a companion `Reflection` row (`output_sink="telegram:<chat_id>"`,
  `execution_type="agent"`, `at:<ISO>`, `auto_delete_after_run=True`) for dashboard
  visibility — the generalized form of the existing `_register_scheduled_reflection`.
  (This row is dashboard-only surfacing; the `ReflectionScheduler` never ticks it because it
  is not in the YAML registry — execution is driven solely by the scheduled `AgentSession`.)
- Returns JSON including `schedule_id` (a 32-char hex token) and a ready-to-cite
  `citation` string `schedule_id=<hex>` that the session can paste into its reply so the
  promise gate allows the forward promise.

### Abuse limits (reuse existing)

- Depth cap: `MAX_SCHEDULING_DEPTH` (3) via `_get_scheduling_depth()`.
- Rate limit: `MAX_SCHEDULED_PER_HOUR` (30) via `_check_rate_limit()`.
- New `CHECKIN_MAX_LEAD_SECONDS` (default 7 days, env-overridable, provisional/tunable) caps
  absurdly-far schedules. Past `--at` times are rejected with a clear error.

### Time-based-with-reschedule (the deterministic default)

The primitive is one-shot by design (the issue's simple, deterministic default). "Wake at T,
check, reschedule if not done" is emergent, not a code feature: the woken session inspects
the work and, if still incomplete, calls `checkin` again. No new session state, no
completion-triggered wakeups, no parent-attached children (#1633), no nudge-loop content
inspection (#1058).

### Promise-gate wiring

Update `_RECOVERY_TEMPLATE` in `bridge/promise_gate.py` to name the primitive as the
sanctioned deferral path (a third option `(c)`), and the LLM system prompt / docs to point
sessions at it. The template still names no bypass mechanism (anti-leak tests unaffected —
they only forbid `PROMISE_GATE_ENABLED` / `--no-promise-gate` / `VALOR_OPERATOR_MODE`).

## Success Criteria

- [ ] A session can schedule a one-shot future check-in with an arbitrary prompt that
  delivers its result to the originating chat (`checkin --prompt ... --in 30m`).
- [ ] The call returns a `schedule_id` matching `promise_gate._SCHEDULED_DELIVERY_PATTERNS`
  (`schedule_id=<hex>`), verified by asserting the gate ALLOWs a forward-deferral draft that
  cites it.
- [ ] The promise-gate recovery template names the primitive as the sanctioned deferral path.
- [ ] Depth cap (3) and rate limit (30/hr) from the existing scheduler apply to `checkin`.
- [ ] Integration test: schedule a near-future check-in, verify the `AgentSession` is created
  with the future `scheduled_at`, correct `chat_id`, and arbitrary prompt, and that
  `session_pickup` defers it until due then makes it eligible.

## No-Gos

- **No nudge-loop content inspection** (#1058) — the router stays content-blind; fulfillment
  lives entirely in the scheduling primitive. Justification: reintroducing content markers in
  the router was explicitly removed and the issue forbids it.
- **No parent-attached child sessions** (#1633) — the check-in is a standalone scheduled
  session, not a child. Justification: parent-attached children are disabled; a scheduled
  session achieves the outcome without them.
- **No new lifecycle state** — reuses `pending` + `scheduled_at`. Justification: a new
  "awaiting async result" state is heavyweight and overlaps disabled child machinery.
- **No MCP tool this increment** — CLI subcommand is sufficient (sessions have Bash); the MCP
  `reflections_create` tool is the heavier alternative and is deferred. Justification: the
  issue sanctions CLI-only for the first increment.

## Update System

No update-system changes required. The `checkin` subcommand is added to an already-installed
module (`tools/agent_session_scheduler.py`, invoked via `python -m`); no new dependency, no
new config file, no launchd plist, no propagation step. `CHECKIN_MAX_LEAD_SECONDS` is an
optional env override with a safe default, so no `.env` change is required on any machine.

## Agent Integration

The primitive is reachable via the agent's Bash tool as
`python -m tools.agent_session_scheduler checkin ...` — the same invocation surface as every
other `agent_session_scheduler` subcommand. No new `pyproject.toml [project.scripts]` entry is
required (the module is already a `python -m` entry point). The bridge does not need to import
it directly. Integration coverage: a test invokes the subcommand path and asserts the
`AgentSession` + `schedule_id` contract, and a promise-gate test asserts the emitted citation
is accepted by the gate.

## Failure Path Test Strategy

- Past `--at` → error (no session created).
- `--at` beyond `CHECKIN_MAX_LEAD_SECONDS` → error.
- Neither / both of `--at` and `--in` → argparse error.
- Malformed `--in` / `--at` → clear ValueError-backed error, no session created.
- Depth cap reached → error, no session created.
- Rate limit exceeded → error, no session created.
- Reflection-row write failure → non-fatal; the `AgentSession` (execution path) still wins and
  the command returns success.
