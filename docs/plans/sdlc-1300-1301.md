# /do-pr-review: Bot Identity + Conflict-State No-Op (Bundle #1300 + #1301)

## Problem

Two related defects in `/do-pr-review` surfaced on cuttlefish PR [#354](https://github.com/yudame/cuttlefish/pull/354) on 2026-05-06:

**#1300 — Reviews post under operator's GitHub identity.** Three agent-authored reviews (two `CHANGES_REQUESTED`, one `APPROVED`) were attributed to `tomcounsell`. The `APPROVED` verdict can satisfy a "1 approving review required" branch-protection rule. There is no forensic signal distinguishing agent reviews from human reviews on the wire.

**#1301 — Conflict-state preflight wastes a review pass.** When `mergeable=CONFLICTING / mergeStateStatus=DIRTY`, the skill posted a formal `CHANGES_REQUESTED` review whose body was "rebase first." The PR review history accumulates pseudo-reviews that aren't real code-review verdicts.

These are bundled because they touch the same files (`SKILL.md`, `sub-skills/post-review.md`, `sub-skills/checkout.md`) and a single PR avoids two rebase cycles on the same skill.

**Current behavior:**
- `gh pr review` runs under the operator's credential. Body has no machine-readable agent marker.
- `post-review.md` decision tree (`§3`) routes `BLOCKED_ON_CONFLICT` to `gh pr comment`, but the actual production run on PR #354 produced a formal `CHANGES_REQUESTED` review — meaning the short-circuit guard isn't bulletproof. A higher-level path (the SKILL.md `Tier 1` block at lines 348–372) still calls `gh pr review --request-changes` whenever `REVIEW_BODY` is composed, with no preflight check.

**Desired outcome:**
- Pipeline-driven reviews post under a non-human identity, with a machine-readable `<!-- SDLC-AGENT-REVIEW v1 -->` preface in every body. Local developer use is unchanged (developer's `gh` credential, no marker).
- On `mergeable=CONFLICTING` or `mergeStateStatus=DIRTY`, the skill emits the `BLOCKED_ON_CONFLICT` OUTCOME without ever calling `gh pr review` / `gh api .../reviews`. A short `gh pr comment` is retained for inline visibility on the PR.

## Why Previous Fixes Failed

**Root cause pattern (#1301 only):** PR #1122 added the §2b block in `post-review.md` and a preflight-aware decision tree in §3, but SKILL.md retained a parallel Tier 1/2/3 decision tree (§6, lines 346–427) that does not consult `$PREFLIGHT_VERDICT`. Agents reading SKILL.md top-to-bottom hit `§6` and execute `gh pr review --request-changes` before they reach `post-review.md §3`. **The fix is duplication elimination, not new logic** — delete the Tier 1/2/3 block from SKILL.md and make `post-review.md §3` the single source of truth.

## Research

**Queries used:**
- (skipped — internal infrastructure decision; codebase context sufficient)

**Key findings:**
- `gh` CLI documents `GH_TOKEN` env var as overriding the stored credential per-invocation. Setting it for a single subprocess (the review-post step) is a clean, well-supported pattern — no `gh auth switch` dance needed.
- GitHub branch protection's "approving reviews from someone other than the last pusher" rule does NOT exclude specific accounts directly, but **CODEOWNERS** combined with "Require review from Code Owners" can scope the gate to a human-owners team that excludes the bot. Alternative: use a GitHub Ruleset with `bypass_actors` configured to omit the bot from approver-counting.
- `<!-- HTML comment -->` markers in PR review bodies are stable across the GitHub UI and API (`reviews[].body`); they survive edits and are queryable via `jq`.

## Architectural Impact

- **New dependency:** A repo secret / vault entry for `SDLC_AGENT_GH_TOKEN` (PAT scoped to the bot account, repo-scoped). Loaded by the bridge / sdk_client and exported into the agent's env when `/do-pr-review` runs in pipeline context.
- **Interface changes:** None to the OUTCOME contract. Review body grows a single-line marker preface — does not change verdict parsing.
- **Coupling:** Reduces — collapses two parallel decision trees (SKILL.md §6 and post-review.md §3) into one.
- **Reversibility:** Trivial. Revert the skill commits and unset the env var; reviews resume posting under the operator credential.

## Solution

### Key Elements

- **Bot identity (#1300)**: Service-account PAT loaded from `SDLC_AGENT_GH_TOKEN` env var. The skill exports `GH_TOKEN` for the *single* `gh pr review` / `gh pr comment` subprocess that posts the review. All other `gh` calls (read-only `gh pr view`, `gh pr diff`, `gh issue view`) continue under the operator's `gh auth` credential — they are reads, identity is irrelevant.
- **Signed marker (hybrid)**: Every agent-posted review body starts with `<!-- SDLC-AGENT-REVIEW v1 sha=<HEAD_SHA> -->` on its own line. Marker is emitted whenever the review is composed in agent context (env var `CLAUDE_AGENT_REVIEW=1`, set by `sdk_client.py` when the skill runs in the SDLC pipeline). Local developer runs (no env var) emit no marker — review body is unchanged.
- **Branch-protection exclusion**: Document in the new `## Review Identity` section how to configure GitHub Rulesets so the bot's approval does NOT satisfy the "N approving reviews" gate. Two supported patterns: (a) CODEOWNERS gating, (b) Ruleset with `required_review_thread_resolution` + `bypass_actors` excluding the bot.
- **Single decision tree (#1301)**: Delete the Tier 1/2/3 block in `SKILL.md §6` (lines 346–427). All review-post logic lives in `post-review.md §3`. The first action in §3 becomes a hard guard: `if PREFLIGHT_VERDICT in {BLOCKED_ON_CONFLICT, PR_CLOSED}: gh pr comment ONLY; never gh pr review`.
- **Preflight comment retained**: For visibility on the PR, the §2b block still posts a short `gh pr comment` citing `mergeStateStatus`. This is informational, not a review.
- **Historical posture**: Existing reviews on cuttlefish PR #354 are left untouched. The cutover is documented in the new `## Review Identity` section. Backfilling markers retroactively is out of scope.

### Flow

Pipeline review → preflight (`gh pr view`) → `PREFLIGHT_VERDICT` set → single `post-review.md §3` decision tree:

- `PREFLIGHT_VERDICT in {BLOCKED_ON_CONFLICT, PR_CLOSED}` → compose §2b/§2c body (no marker — informational comment, not a review) → `gh pr comment` (under operator credential — comments don't count as reviews) → emit OUTCOME with `next_skill: null`.
- Otherwise → run code review → compose review body with `<!-- SDLC-AGENT-REVIEW v1 sha=... -->` preface (if `CLAUDE_AGENT_REVIEW=1`) → export `GH_TOKEN=$SDLC_AGENT_GH_TOKEN` for the single `gh pr review` subprocess → post `--approve` / `--request-changes` → unset `GH_TOKEN` → emit OUTCOME.

Local developer flow: `CLAUDE_AGENT_REVIEW` unset, `SDLC_AGENT_GH_TOKEN` may be unset. Skill behaves exactly as today — operator credential, no marker.

## Failure Path Test Strategy

### Exception Handling Coverage
- [ ] No `except Exception: pass` blocks in scope. The skill is a Bash-driven workflow — failures are surfaced via non-zero exit codes from `gh`.
- [ ] If `SDLC_AGENT_GH_TOKEN` is unset but `CLAUDE_AGENT_REVIEW=1`, the skill MUST fail loudly with a clear error message ("agent context but bot token missing — refusing to post under operator identity") rather than silently fall back.

### Empty/Invalid Input Handling
- [ ] If `mergeable` returns `UNKNOWN` even after the existing 2s retry, treat as `CONFLICTING` for safety (don't post a review under uncertain mergeability). Test asserts this conservative path.
- [ ] If `gh pr view` itself fails (network, auth), the skill exits with `status:"fail"` OUTCOME and `next_skill:null`. No partial review posted.

- [ ] When `gh pr review` returns non-zero (token revoked, repo permissions changed), the skill must emit a `PUSH_FAILED`-style OUTCOME (`status:"fail"`) and NOT retry as the operator. Test verifies the fallback is not silent.

## Rabbit Holes

- **GitHub App route.** Tempting because it gives a real `*-bot[bot]` identity and avoids PAT rotation. Costs: app registration, per-repo install, JWT signing, app-token short-TTL refresh logic. Defer — service-account PAT covers all the requirements at a fraction of the complexity.
- **Retroactive marker backfill.** Editing existing reviews on cuttlefish PR #354 to add the marker. Skip — `gh api PATCH /repos/.../reviews/<id>` does not support body edits via the standard surface, and the reviews are auditable as-is via timestamp + author + content.
- **Splitting the bundle into two PRs.** Both fixes touch `post-review.md §3` (the single decision tree). Splitting forces a rebase cycle on the same lines.
- **General `gh` write-path inventory.** Issue #1300's recon notes a wider audit (issue comments, label changes, `git push`). That is Issue 5's territory and should not be pulled into this work.

## Risks

### Risk 1: Bot account locked out / token revoked
**Impact:** Pipeline reviews silently post nothing or fall back to operator credential, undoing the fix.
**Mitigation:** Hard fail when `CLAUDE_AGENT_REVIEW=1` and `gh pr review` exits non-zero. Emit `status:"fail"` OUTCOME with explicit `failure_reason`. The PM session sees the failure and surfaces it; no silent operator-credential fallback.

### Risk 2: Bot's `APPROVED` still satisfies branch protection on a repo without CODEOWNERS configured
**Impact:** The fix at the skill layer doesn't fully prevent branch-protection bypass — config has to land per-repo.
**Mitigation:** Document the CODEOWNERS / Ruleset patterns in `## Review Identity`. The acceptance criteria for #1300 require *one* representative repo (yudame/cuttlefish) configured. Other repos remain at-risk until similarly configured — that's a runbook task, not a skill defect.

### Risk 3: Tier 1/2/3 deletion in SKILL.md breaks an existing call site
**Impact:** Some other skill or doc may link to the Tier 1/2/3 line numbers.
**Mitigation:** `grep -rn "Tier 1\|Tier 2\|Tier 3" .claude/ docs/` during build. Update or remove dangling references. The replacement pointer ("see post-review.md §3") preserves discoverability.
