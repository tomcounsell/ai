# session-liveness-check: confirm why a 4-day-stale running session evaded recovery, then gate the sticky own-progress fields (only if warranted)

## Problem

A `teammate` session in the Cyndra project (`agent_session_id=74b3c8e84a92460d8eb1780a303a3056`, thread `tg_cyndra_-1003794218389_535`) sat in `status=running` for **~4 days** and was never recovered by `session-liveness-check`, despite the check running every 5 minutes (~1,100 ticks). It was only cleared by a manual `valor_session kill`. The `claude_pid` (19007) was already dead; `last_heartbeat_at` froze at +74s; `sdk_ever_output` was `False`; `claude_session_uuid` was set (auth succeeded before death).

A stuck `running` record on a Telegram thread blocks fresh messages on that thread from spawning new work (they resume a dead session) and silently consumes a worker-tracking slot.

**Current behavior:**
A running session that authenticated (set `claude_session_uuid`) but never produced a tool/turn event, whose owning worker task is no longer making progress, can evade `session-liveness-check` recovery indefinitely. There is a *demonstrable code gap* — the own-progress fields in `_has_progress` are ungated — but it is **not yet confirmed** that this gap is the operative cause of the observed zombie.

**Desired outcome:**
The root cause is **confirmed first** (which recovery branch the zombie fell into; why the heartbeat froze at +74s while status stayed `running`). Only then is a behavioral fix prescribed, scoped precisely to the confirmed cause, with an explicit test proving a legitimately-running long-thinking session is **not** recovered (no false-positive regression of the kind #1226/#1356 fought).

## Data Flow

This traces how a `running` session reaches (or evades) recovery, so the investigation knows every branch and skip-point.

1. **Entry point:** `_agent_session_health_check()` runs every 5 min (scheduled reflection `session-liveness-check`). It calls `AgentSession.query.filter(status="running")` and passes results through `_filter_hydrated_sessions` (phantom guard, drops orphan `$IndexF` members).
2. **Per running entry — pre-branch guards (any `continue` here skips recovery entirely):**
   a. **Terminal-status guard** (L1535): if `getattr(entry,"status",None)` is in `_TERMINAL_STATUSES`, skip (stale running-index entry, #1006).
   b. **Delivery guard** (L1549): if `response_delivered_at` is set, finalize as `completed` and `continue` (never reaches branch logic).
3. **Branch decision** (L1577-1622):
   - `worker = _active_workers.get(entry.worker_key)`; `worker_alive = worker is not None and not worker.done()`.
   - `running_seconds = now - started_at` (None if no `started_at`).
   - **Branch 1 (`not worker_alive`):** recover if `started_at is None` OR `running_seconds > AGENT_SESSION_HEALTH_MIN_RUNNING` (300s). **Does NOT consult `_has_progress`.**
   - **Branch 2 (`worker_alive`):** recover only if `running_seconds > 300s` AND `not _has_progress(entry)` (and Tier-2 reprieve gates also fail).
4. **`_has_progress(entry)` (L677):** when `sdk_ever_output is False` — sub-check B (heartbeat fast-path, GATED by `NO_OUTPUT_BUDGET_SECONDS` since #1356) → **own-progress fields (UNGATED): `turn_count>0` / non-empty `log_path` / truthy `claude_session_uuid` → return True** → child-progress check → False.
5. **Recovery** via `_apply_recovery_transition` (L1112): enforces `MAX_RECOVERY_ATTEMPTS=2`, OOM defer, delivery finalize, kill-switch. Requeues to `pending` (and, per #1537, should kill `claude_pid`).
6. **Parallel path — startup recovery** (`_recover_interrupted_agent_sessions_startup`, L439): on worker boot, scans `running` sessions; for a *bridge* (non-`local`) teammate session it takes the `else` branch (L629) and re-queues to `pending`. The observed zombie is a bridge teammate session, so **this path should also have caught it on the next worker restart** — unless no restart happened in 4 days, or a guard skipped it.

**The contradiction the investigation must resolve:** Steps 3 (branch 1) and 6 (startup recovery) are two independent recovery paths that should have fired *without* consulting `_has_progress`. For the own-progress gap (step 4) to be the operative cause, the zombie must have been in branch 2 the whole time — i.e., the original worker process stayed alive for 4 days with the task hung un-done and `_active_workers` still holding a not-done future. That is the hypothesis to confirm or refute first.

## Risks

### Risk 1: Gating the sticky fields without confirming the branch reintroduces false-positives
**Impact:** A legitimately-running long-thinking session (the #1226/#1356 regression class) gets recovered mid-work, losing progress and producing a confusing requeue. This is the requester's explicit #1 worry.
**Mitigation:** The investigation gate (spike-1/spike-2) is a hard precondition — no behavioral change ships until the branch is confirmed. The gate, if built, reuses the queue-layer heartbeat (which spike-3 confirms stays fresh during long thinking) so a live session always reads fresh. The existing `test_fresh_heartbeat_no_stdout_yet_is_progress` plus the new fresh-vs-stale own-progress PAIR (`test_own_progress_gate_fresh_heartbeat_is_progress` / `test_own_progress_gate_stale_heartbeat_is_zombie`) are the regression sentinels: the FRESH sibling must stay green, the STALE sibling must flip to recovered.

### Risk 2: The real defect is in worker-liveness (branch 1), and a sticky-field gate is a false fix
**Impact:** Shipping the gate "fixes" nothing for the observed zombie (it was in branch 1) while adding false-positive surface area. Future zombies recur.
**Mitigation:** spike-1/spike-2 explicitly test the branch-1-should-have-fired hypothesis (in-memory worker restart empties `_active_workers` → branch 1). If branch 1 is implicated, the plan's behavioral fix pivots to the worker-liveness / startup-recovery path and the sticky-field gate is dropped or deferred behind its own confirmed evidence. The Open Questions encode this fork.

### Risk 3: Recovery requeues without killing the dead/zombie subprocess (#1537 regression)
**Impact:** A recovered session orphans its worker slot or (if `claude_pid` were somehow alive) leaves a competing harness.
**Mitigation:** Recovery flows through `_apply_recovery_transition`, which already inherits #1537's kill-then-requeue behavior. Confirm in the investigation that the shared helper kills `claude_pid` when set; if the zombie path bypasses it, add the kill (Open Question 5). For the observed case `claude_pid` is already dead, so this is belt-and-suspenders.

### Risk 4: A heartbeat-freshness gate window too tight kills sessions during a transient save outage
**Impact:** If Redis briefly rejects the 60s heartbeat save (swallowed by the loop), a too-tight gate could read the session as stale and recover a live session.
**Mitigation:** spike-3 sizes the gate window to tolerate multiple missed 60s saves (e.g., mirror sub-check B's 1800s running-budget rather than a sub-90s heartbeat window, so transient save failures cannot cross the threshold). The gate is conservative by construction.

## Race Conditions

### Race 1: Recovery transition vs. a late-arriving worker completion
**Location:** `agent/session_health.py` branch logic (L1577-1660) + `_apply_recovery_transition` (L1112)
**Trigger:** The health check decides to recover at the same moment the (long-thought-dead) worker task actually finishes and tries to finalize the session.
**Data prerequisite:** `entry.status` must still be `running` when the recovery transition executes.
**State prerequisite:** The CAS / `expected_status="running"` guard in the lifecycle helpers must hold.
**Mitigation:** Recovery uses the existing CAS-on-status pattern (`update_session(..., expected_status="running")` / `StatusConflictError` handling already present in the startup-recovery and delivery paths). The fix adds no new mutation outside that guarded helper, so the existing CAS protects this race. The investigation must confirm the zombie's recovery path goes through the CAS helper and not a raw save.
