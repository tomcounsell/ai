# Pre-check locked-elsewhere branch in verify_worktree_branch

## Problem

`agent/worktree_manager.py::verify_worktree_branch` attempts `git checkout <expected_branch>` on a clean-but-mismatched worktree without first verifying that `expected_branch` is available for checkout. When the expected branch is already checked out by another worktree (e.g., `main` held by the canonical repo at `/Users/valorengels/src/ai`), git rejects the checkout with:

```
fatal: 'main' is already used by worktree at '/Users/valorengels/src/ai'
```

The function then surfaces this as a `WorktreeBranchMismatchError` whose cause is the raw git error. The executor logs the failure and refuses to launch the harness. This caused 9 dev-session failures in a 4-hour window on 2026-05-21 (issue #1406 failure-loop fingerprint `06f5940a02173ba1`).

## Desired Outcome

When `verify_worktree_branch` detects that `expected_branch` is already locked by another worktree, it raises `WorktreeBranchMismatchError` *before* attempting the checkout, with a structured, actionable message naming the holding worktree path. This gives the executor (and any future caller) a clean signal rather than a raw git stderr leak, and makes downstream remediation (e.g., resolving the slug→branch mismatch upstream) clearer.

## Why Previous Fixes Failed

No prior fixes attempted this specific case. The #1377 author covered the dirty-working-tree refusal but treated `git checkout` as always-succeeds on a clean tree — overlooking that git's worktree model can reject the checkout on lock contention even when the local tree is pristine.

## Data Flow

Entry: `session_executor.py:917` calls `verify_worktree_branch(working_dir, branch_name)` for dev sessions whose `working_dir` lives under `.worktrees/`.

Inside `verify_worktree_branch`:
1. `git -C <wt> rev-parse --abbrev-ref HEAD` → `actual_branch`.
2. If `actual_branch == expected_branch`, return (no-op).
3. `git -C <wt> status --porcelain` → `dirty` list.
4. If dirty, raise (existing behavior, unchanged by this plan).
5. **[NEW]** Resolve main repo root from inside the worktree via `git -C <wt> rev-parse --git-common-dir`, then `_find_worktree_for_branch(main_repo_root, expected_branch)`.
6. **[NEW]** If a holding worktree path is found AND it is not the current `worktree_path` itself, raise `WorktreeBranchMismatchError` with `cause=f"cannot checkout '{expected_branch}': already used by worktree at '{holding_path}'"`.
7. Otherwise, run `git checkout <expected_branch>` as before (existing path).

Exit: executor catches `WorktreeBranchMismatchError`, logs at ERROR with the structured message, marks the session failed via the normal failure pipe.

## Solution

### Technical Approach

Edit `agent/worktree_manager.py::verify_worktree_branch` to insert a pre-checkout guard between step 4 (cleanliness check) and the existing `git checkout` call (currently line 161).

Steps:
1. After the dirty check passes (clean worktree, branch mismatch), resolve the main repo root from inside the current worktree using `git -C <worktree_path> rev-parse --git-common-dir` and taking its parent. (Cannot import / reuse `resolve_main_repo_root` directly because that helper accepts a `file_path` and re-resolves; doing the subprocess inline keeps the verify path self-contained and avoids an import cycle.)
2. Call existing `_find_worktree_for_branch(main_repo_root, expected_branch)`.
3. If it returns a path AND that path is not the current `worktree_path` (compared via `Path(...).resolve()`), raise `WorktreeBranchMismatchError` with a structured `cause` message including the holding worktree path. The exception's existing `expected_branch` / `actual_branch` fields stay populated for parity with other failure modes.
4. If `_find_worktree_for_branch` returns `None`, or returns the current worktree's own path (defensive), fall through to the existing `git checkout` call.

The new branch lives entirely inside `verify_worktree_branch` — no public API changes, no new helper functions, no executor changes. `WorktreeBranchMismatchError` already supports a `cause=` kwarg.

### Edge Cases

- **`git rev-parse --git-common-dir` failure**: Wrap in the same `subprocess.CalledProcessError` handler shape used elsewhere in the function; on failure, raise `WorktreeBranchMismatchError` with `cause=f"git rev-parse --git-common-dir failed: ..."`. Fail-loud — silent fallback would re-introduce the bug.
- **`_find_worktree_for_branch` returns the current worktree's own path**: Should never happen for a mismatched branch (current worktree is on `actual_branch`, not `expected_branch`), but if it does, treat as "not locked" and proceed to checkout. Defensive only.
- **`expected_branch` doesn't exist as a ref at all**: `_find_worktree_for_branch` returns `None`, we fall through to `git checkout`, which fails with its existing "pathspec did not match" error and is wrapped by the existing handler. No regression.

## Race Conditions

None. `verify_worktree_branch` runs inside a single executor's `_handle_dev_session` path; the worktree lock state can in principle change between the `_find_worktree_for_branch` check and the subsequent checkout, but a race here would degrade to the existing failure mode (raw git error captured as cause) rather than a new defect. The fix tightens the common case; the unfixable race is acceptable.

## Rabbit Holes

- **Don't refactor `verify_worktree_branch` to use `resolve_main_repo_root`**. That helper does its own argument-shape gymnastics (`file_path` accepted, parent computed); inlining `git rev-parse --git-common-dir` is shorter and avoids a cross-function dependency that doesn't reduce duplication meaningfully here.
- **Don't try to auto-recover from the locked-elsewhere case** (e.g., by switching the main repo to a different branch, or by removing the sibling worktree). The root cause is upstream — the executor resolved the expected branch as `main` when the worktree was actually on `session/{slug}`. Fixing that lives in the executor / branch-resolution path, not in this guard. This plan deliberately scopes to the structured-error fix only.
- **Don't widen the exception type or change its fields.** Adding a new `holding_path` attribute to `WorktreeBranchMismatchError` would be tidier but creates an unnecessary surface change for one call-site. The `cause` string carries the path.
