# Harden bare `launchctl bootstrap` call sites against transient errno 5

## Problem

A `/update` run on machine "Valor the Bald" aborted mid-service-install with:

```
Bootstrap failed: 5: Input/output error
Try re-running the command as root for richer errors.
```

**Current behavior:**
`scripts/valor-service.sh` runs under `set -e` and contains three **bare**
`launchctl bootstrap` calls with no error tolerance and no kickstart fallback.
`errno 5` (`Input/output error`) is the known macOS launchd race where the
service label is still registered/draining in the `gui/<uid>/` domain when
`bootstrap` runs (e.g. immediately after a `bootout`, or a stale half-load from
a prior crash). Under `set -e`, a single transient errno 5 aborts the entire
`valor-service.sh install`/`restart`/`worker-start` invocation, which `/update`
calls via `scripts/update/service.py`. The `install_*.sh` helper scripts share
the identical bare-bootstrap shape and are latently exposed to the same abort.

**Desired outcome:**
Every shell `launchctl bootstrap` call site is idempotent and fail-soft:
bootout-before-bootstrap, then on bootstrap failure fall back to
`launchctl kickstart -k`, wrapped so `set -e` never aborts the whole install.
A transient errno 5 downgrades to a best-effort kickstart and the install/restart
completes instead of hard-failing. This matches the pattern already established
in `scripts/remote-update.sh` (L247-281) and
`scripts/update/service.py::install_log_rotate_agent` (L804-811).

## Data Flow

1. **Entry point**: `/update` skill invokes `scripts/remote-update.sh` →
   `scripts/update/run.py` → `scripts/update/service.py::install_service` /
   `restart_service`.
2. **service.py** shells out to `scripts/valor-service.sh install` / `restart` /
   `worker-start` (and, on some paths, the `install_*.sh` helpers).
3. **valor-service.sh** (under `set -e`) reaches a bare `launchctl bootstrap`.
   If launchd returns errno 5, the bare call's non-zero exit trips `set -e` and
   the whole script aborts — no further `[update]` output, hard mid-install stop.
4. **Output**: `/update` run terminates with `Bootstrap failed: 5`. Services are
   left in an indeterminate load state.

**After the fix**: at step 3 the bootstrap is wrapped so its failure is caught,
a `kickstart -k` fallback runs, and control continues to the script's own
verification (`is_worker_running` / `launchctl list` grep) — the install
completes or reports a *distinct* failure without a bare `set -e` abort.

## Why Previous Fixes Failed

| Prior Fix | What It Did | Why It Failed / Was Incomplete |
|-----------|-------------|-------------------------------|
| PR #1421 (#1407) | Switched worker/watchdog from `load` to `bootstrap` and added a defensive `bootout` before `bootstrap` | Left the `bootstrap` call **bare** — no rc-check, no `kickstart -k` fallback, no `set -e` guard. A transient errno 5 (label still draining) still aborts the whole script. |

**Root cause pattern:** the `bootout`-before-`bootstrap` shape prevents "service
already bootstrapped" (errno for an *already-registered* label) but does NOT
prevent errno 5 — the label can still be mid-*drain* in `gui/<uid>/` when
`bootstrap` runs. The only robust remedy is to tolerate the bootstrap failure and
fall back to `kickstart -k`, which is what `remote-update.sh` / `service.py`
already do and what this plan applies uniformly.

## Solution

### Key Elements

- **`bootstrap_or_kickstart()` helper (valor-service.sh)**: a single shell
  function that performs `bootout` (tolerant) → `bootstrap` → on non-zero, falls
  back to `kickstart -k`; wrapped so `set -e` never aborts. All three
  valor-service.sh sites call it.
- **Inline hardened block (install_*.sh)**: each standalone installer's bare
  `launchctl bootstrap` is replaced with a `bootout`-then-`bootstrap`-rc-check-then-`kickstart -k`
  block wrapped in `if`/`||` so a bootstrap failure degrades to kickstart instead
  of aborting the installer.

### Flow

`/update` → `valor-service.sh restart` → reaches a service load site →
`bootstrap_or_kickstart LABEL PLIST` → bootout → bootstrap → (errno 5) →
kickstart -k → continue → script's own `is_worker_running` verify → install
completes.

### Technical Approach

- **Helper signature** in `valor-service.sh`:
  ```bash
  # Idempotent, fail-soft service (re)load. Bootout-before-bootstrap, then on
  # bootstrap failure fall back to `kickstart -k`. Never aborts under `set -e`.
  # Args: $1 = full launchd label, $2 = plist path.
  bootstrap_or_kickstart() {
      local label="$1" plist="$2" uid
      uid="$(id -u)"
      launchctl bootout "gui/$uid/$label" 2>/dev/null || true
      if launchctl bootstrap "gui/$uid" "$plist"; then
          return 0
      fi
      # bootstrap failed (e.g. errno 5: label still registered/draining in
      # gui/<uid>/). Best-effort kickstart -k gets it running without aborting.
      echo "  bootstrap failed for $label; falling back to kickstart -k"
      launchctl kickstart -k "gui/$uid/$label" 2>/dev/null || true
      return 0
  }
  ```
  - Each valor-service.sh site derives its label. `bootstrap_plist_idempotent()`
    already has `$label`; bridge install uses the bridge label (derive from
    `$PLIST_PATH` basename or existing `$BRIDGE_LABEL`/service-prefix var — read the
    file to use whatever the script already defines); `worker-start` uses
    `$WORKER_PLIST_NAME`.
  - The helper `return 0`s even on kickstart failure so `set -e` never trips; the
    script's own downstream verify (`is_worker_running`, `launchctl list` grep, or
    the bridge's post-load echo) remains the authority on whether the service is up.
    This preserves the existing failure-reporting behavior (e.g. `worker-start`'s
    `return 1` on `! is_worker_running`).
- **install_*.sh inline block** (each installer, replacing the bare `bootstrap`):
  ```bash
  echo "Loading $LABEL..."
  launchctl bootout "gui/$(id -u)/$LABEL" 2>/dev/null || true
  if ! launchctl bootstrap "gui/$(id -u)" "$PLIST_DST"; then
      echo "  bootstrap failed for $LABEL; falling back to kickstart -k"
      launchctl kickstart -k "gui/$(id -u)/$LABEL" 2>/dev/null || true
  fi
  ```
  - The conditional `bootout` blocks that already precede these installs
    (`if launchctl list | grep -q "$LABEL"; then ... bootout ...`) stay as-is; the
    new unconditional `bootout` immediately before `bootstrap` is idempotent and
    harmless (the earlier conditional bootout is a no-op if the label is gone).
    Prefer the minimal, local edit: keep the existing conditional bootout, and only
    replace the *bare bootstrap line* with the rc-checked bootstrap + kickstart
    fallback. (An unconditional bootout at the bootstrap site is optional; do not
    add a second bootout if the script already booted out the same label two lines
    above — avoid redundant churn. Read each script and choose the minimal edit.)
  - `install_worker.sh` has **two** sites: the worker `bootstrap` (~167) and the
    watchdog `bootstrap` (~205, already has a `bootout ... || true` immediately
    before it). Harden both.
  - For installers under `set -euo pipefail` (`install_nightly_tests.sh`,
    `install_sdlc_reflection.sh`), the `if ! launchctl bootstrap ...` guard is
    exactly what prevents the abort — same shape, same effect.
- **Do NOT touch** the Python helpers in `scripts/update/service.py`
  (`install_worker`, `install_reflection_worker`, `install_nightly_tests`) — they
  use `run_cmd` (captures output, returns bool, logs WARN) and already degrade
  gracefully. This is confirmed in the issue's recon.

## Rabbit Holes

- **Refactoring the `install_*.sh` scripts into a shared sourced library.** Tempting
  (they duplicate the pattern) but out of scope and risky — these are standalone
  installers invoked independently; introducing a shared lib changes their
  invocation contract. Keep the inline per-script edit.
- **Adding retry loops / sleeps around bootstrap.** The established repo pattern is
  a single `kickstart -k` fallback, not a retry loop. Do not invent a new
  resilience mechanism; mirror `remote-update.sh` / `service.py` exactly.
- **Touching the Python `service.py` install helpers.** Explicitly out of scope per
  the issue — they already degrade via `run_cmd`.
- **"Fixing" the earlier conditional `bootout` blocks.** They are correct; only the
  bare `bootstrap` line is the defect. Don't rewrite the surrounding install logic.

## Risks

### Risk 1: A masked genuine failure — kickstart silently "succeeds" but the service isn't actually healthy
**Impact:** `/update` completes but a service is degraded, and the operator doesn't
notice because the bootstrap failure was swallowed.
**Mitigation:** The helper echoes a distinct fallback line, and the scripts' own
downstream verification is preserved: `worker-start` still `return 1`s on
`! is_worker_running`; the bridge/installer success echoes remain. We only tolerate
the *bootstrap-to-kickstart* transition, not the final health check. This matches
`remote-update.sh`, which keeps its `RESTART FAILED` distinct line + non-zero exit.

### Risk 2: Label derivation for the bridge site is wrong (bridge install has no local `$label` var like the others)
**Impact:** helper called with the wrong label → kickstart no-ops on a nonexistent
label; bridge left down.
**Mitigation:** During build, read the bridge-install section and use whatever label
variable the script already defines (service-label prefix / `$BRIDGE_LABEL` /
basename of `$PLIST_PATH`). Verify with a grep that the label passed matches the
`Label` key in the rendered plist. The bridge already has `stop_bridge` before the
load, so bootout is effectively covered; the kickstart fallback is the only new bit.

## Race Conditions

### Race 1: Label still draining in gui/<uid>/ when bootstrap runs (errno 5)
**Location:** every `launchctl bootstrap` call site listed above.
**Trigger:** a `bootout` (or a prior crash's half-load) leaves the label
registered/draining in `gui/<uid>/`; `bootstrap` runs before the drain completes
and launchd returns errno 5.
**Data prerequisite:** none (state, not data).
**State prerequisite:** the label must be fully unregistered from `gui/<uid>/`
before `bootstrap` can succeed.
**Mitigation:** this IS the bug being fixed. We do not try to *win* the race
(no sleep/retry) — we tolerate losing it: on bootstrap failure, `kickstart -k`
force-restarts the (already-registered) service, which is the correct action
precisely when the label is still registered. Fail-soft under `set -e` guarantees
the race never aborts the install.
