# Bound Tier-1 Sub-Check B with a No-Output Running-Time Budget

## Problem

`agent/session_health.py:683-693` (Tier-1 sub-check B) reads
`AgentSession.last_heartbeat_at` as evidence that a session whose SDK has not yet
produced any structured output is still alive. That field is written every 60s
by `_heartbeat_loop` in `agent/session_executor.py:1712-1729`. The loop is keyed
on the asyncio task wrapper (`while not task._task.done()`), **not** on the SDK
subprocess making forward progress.

Any wedge that lets the asyncio task keep awaiting without raising or returning
— including the cwd-disappearance case observed in investigation #1246 — produces
an indefinitely fresh `last_heartbeat_at`. The session passes Tier 1 forever.
The Tier-2 escalation guard at `_tier2_reprieve_signal` (lines 770-775) is gated
behind `_has_progress` returning False first, so it never fires. The session
sits in `running` until a human notices.

**Current behavior:**

PM session `90c4117dbf06431c86ee4807d7a18bcd` ran with `status=running` for ~10.9
hours with `sdk_ever_output=False`. Tier-1 sub-check B passed on every single one
of ~130 health-check ticks because `last_heartbeat_at` was always within the 90s
freshness window. Tier 2 was never consulted.

**Desired outcome:**

A session with `sdk_ever_output=False` whose `running_seconds` exceeds the no-output
budget (`MAX_NO_OUTPUT_REPRIEVES * HEARTBEAT_WRITE_INTERVAL` = 1200s = 20 min) makes
sub-check B return False on the next health-check tick, regardless of how fresh
`last_heartbeat_at` is. Tier 1 then declines, the existing Tier 2 escalation guard
fires after `MAX_NO_OUTPUT_REPRIEVES` ticks, and `_apply_recovery_transition`
recovers the session.

Sessions in their startup window (`running_seconds < STARTUP_GRACE_SECONDS`)
remain unaffected — fresh heartbeat alone passes Tier 1, preserving pre-#1226
behavior for ad-hoc Telegram traffic before the SDK emits its first turn.

## Why Previous Fixes Failed

| Prior Fix | What It Did | Why It Failed / Was Incomplete |
|-----------|-------------|-------------------------------|
| PR #1039 (#1036) | Introduced `last_heartbeat_at` as a queue-layer Tier-1 signal. | Coupled to the asyncio task wrapper, not SDK forward progress. Any wedge that keeps the task awaiting refreshes the heartbeat indefinitely. |
| PR #1243 (#1226) | Promoted per-turn `last_tool_use_at`/`last_turn_at` to Tier-1 sub-check A; added Tier-2 reprieve cap. | Sub-check B (the legacy heartbeat path) was kept unchanged for backward compatibility. The reprieve cap (Tier 2) is unreachable until sub-check B declines, so a session that never produces output never reaches Tier 2. |

**Root cause pattern:** Both fixes correctly identified that "subprocess-exists" is not the
same as "subprocess is making progress". #1226 fixed sub-check A (per-turn signals) but left
sub-check B with the same tick-driven heartbeat that #1036 introduced. The hand-off from Tier 1
to Tier 2 was assumed to be reachable via the per-turn signals — but for sessions that never
auth or never emit a first turn, sub-check B holds Tier 1 open forever.

## Solution

### Key Elements

- **`STARTUP_GRACE_SECONDS` constant** (new): the running-time threshold below which a fresh
  `last_heartbeat_at` alone is sufficient evidence of progress. Set to `HEARTBEAT_FRESHNESS_WINDOW`
  (90s) so a session in its very first health-check tick still passes Tier 1.
- **No-output budget computation**: derived inline as
  `MAX_NO_OUTPUT_REPRIEVES * HEARTBEAT_WRITE_INTERVAL` (= 20 * 60 = 1200s = 20 min). Documented
  alongside `STARTUP_GRACE_SECONDS`. Not a new module-level constant — keeping it as an inline
  computation keeps the relationship to existing constants visible.
- **Sub-check B gate**: before returning True on a fresh heartbeat (when `sdk_ever_output=False`),
  the function reads `entry.started_at`. If `started_at` is None or `running_seconds` is within
  `STARTUP_GRACE_SECONDS`, the existing fast-path stands. If `running_seconds` exceeds the
  no-output budget, sub-check B returns False (not True), and `_has_progress` proceeds to the
  remaining own-progress fields and child-progress check. Tier 1 declines if all signals are
  absent.
- **Telemetry counter**: `session-health:tier1_falloff:no_output_budget_exceeded`, INCR'd
  exactly once on the new path (when sub-check B would have passed but the budget gate denied it).
  Mirrors existing counters under `session-health:tier2_*` and `session-health:recovery:*`.

### Flow

Health-check tick fires → `_has_progress(entry)` runs sub-check A → no per-turn signal fresh
→ sub-check B inspects `last_heartbeat_at` → fresh → **NEW GATE**: read `started_at`,
compute `running_seconds`, compare to no-output budget → if budget exceeded, INCR telemetry
counter and skip the sub-check-B return → continue to own-progress fields → all empty →
return False → caller evaluates `_tier2_reprieve_signal` → after `MAX_NO_OUTPUT_REPRIEVES`
consecutive declines, Tier 2 returns None → `_apply_recovery_transition` recovers the session.

### Technical Approach

1. Add `STARTUP_GRACE_SECONDS` constant near the existing constants block (line ~228), with
   a docstring explaining the relationship to `HEARTBEAT_FRESHNESS_WINDOW` and the no-output
   budget. Default value: `HEARTBEAT_FRESHNESS_WINDOW` (90s). Env-tunable via
   `STARTUP_GRACE_SECONDS` for parity with other tunables in the file.

2. Modify sub-check B body (lines 688-693) to compute `running_seconds` from
   `entry.started_at` and gate the fresh-heartbeat fast-path:
   - If `started_at` is None → preserve existing fast-path (sessions predating `started_at`
     introduction).
   - If `running_seconds < STARTUP_GRACE_SECONDS` → preserve fast-path.
   - If `running_seconds >= STARTUP_GRACE_SECONDS` AND
     `running_seconds > MAX_NO_OUTPUT_REPRIEVES * HEARTBEAT_WRITE_INTERVAL` →
     INCR telemetry counter, do NOT return True from sub-check B; fall through to own-progress
     fields.
   - In the band between `STARTUP_GRACE_SECONDS` and the no-output budget, fresh heartbeat
     still passes (no behavior change in that band — preserves backward compatibility for
     sessions in their normal startup-to-first-turn window).

3. Use `tools.metrics_writer.incr_counter()` (or whatever the canonical metrics helper is in
   this repo — to be confirmed at build time by reading existing telemetry sites in
   `session_health.py`) to increment `session-health:tier1_falloff:no_output_budget_exceeded`.
   Wrap in a try/except that logs at WARNING and returns control — telemetry must never
   crash the health-check loop.

4. Add unit tests in `tests/unit/test_health_check_recovery_finalization.py` (existing home of
   sub-check B tests) under a new `TestSubCheckBNoOutputBudget` class. Cover:
   - Fresh heartbeat + `started_at` None → True (legacy path preserved).
   - Fresh heartbeat + `running_seconds=30s` → True (startup grace).
   - Fresh heartbeat + `running_seconds=89s` → True (just inside grace).
   - Fresh heartbeat + `running_seconds=300s` → True (in the normal band, before budget).
   - Fresh heartbeat + `running_seconds=1201s` + `sdk_ever_output=False` → False (budget exceeded).
   - Fresh heartbeat + `running_seconds=4h` + `sdk_ever_output=False` → False (the 11h-wedge pattern).
   - Fresh heartbeat + `running_seconds=4h` + `sdk_ever_output=True` (e.g. `last_tool_use_at` set
     stale) → False from sub-check A (existing semantics — not affected by this change).
   - The Tier-2 hand-off integration: with the above 4h-no-output entry, calling
     `_tier2_reprieve_signal` returns "alive" until `reprieve_count >= 20`, then None.

5. Update existing tests in `TestHasProgressDualHeartbeat` and `TestStdoutStaleRetired` that
   currently assume fresh heartbeat alone is sufficient — set `started_at=_ago(30)` (or similar)
   on the entry so the new gate doesn't change their outcome. Audit dispositions in the
   "Test Impact" section below.

## Rabbit Holes

- **Replacing `_heartbeat_loop` to track SDK forward progress.** The right long-term answer
  is to make `last_heartbeat_at` reflect SDK subprocess liveness rather than asyncio task
  wrapper liveness. That's disproportionate to a once-observed defect and explicitly out of
  scope per the issue body. **Avoid.**
- **Surveying every site that writes `last_heartbeat_at`.** The two writers
  (`_heartbeat_loop` and the T+0 write in `session_executor.py`) are both coupled to the
  asyncio wrapper. The gate in `_has_progress` neutralizes both without touching either.
  **Avoid.**
- **Tightening the no-output budget below 1200s.** Sessions that legitimately spend 5-10
  minutes on a large initial prompt (digesting context, fetching memories, running heavy
  tools) before the first turn would get false-killed. The 1200s budget already aligns with
  `MAX_NO_OUTPUT_REPRIEVES` and gives Tier 2 ~30 min total before recovery (20 min budget +
  20 ticks * 90s reprieve window ≈ 30 min more). **Avoid.**
- **Adding a new field to `AgentSession` for "no-output start time".** `started_at` is
  already canonical and present. **Avoid.**
- **Refactoring constants into a frozen dataclass or settings module.** Out of scope; the
  existing module-level constants pattern is the project convention. **Avoid.**

## Risks

### Risk 1: Existing tests assume fresh heartbeat alone is sufficient
**Impact:** The build introduces a subtle regression where a test that previously passed
fails because `started_at` was implicitly None and the new gate's "legacy path preserved"
branch is misread.
**Mitigation:** The legacy path explicitly preserves `started_at=None` semantics — fresh
heartbeat returns True. Test Impact section above audits each existing test. Build phase
includes an explicit grep for `last_heartbeat_at=_ago` + nearby `started_at` to confirm no
regressions.

### Risk 2: False-positive kills on legitimate slow-startup sessions
**Impact:** A session that genuinely needs 25+ minutes to digest a huge initial prompt before
emitting its first turn (extreme case — not seen in production telemetry) gets killed by the
new gate after 20 minutes.
**Mitigation:** The budget is env-tunable (`STARTUP_GRACE_SECONDS` and the existing
`SDK_PROGRESS_FRESHNESS_WINDOW_SECS` together control the budget). If the false-positive
appears in production, operators bump the env var without a code change. Plan also documents
the ratio: any single tunable bump must respect `STARTUP_GRACE_SECONDS <
MAX_NO_OUTPUT_REPRIEVES * HEARTBEAT_WRITE_INTERVAL`.

### Risk 3: Telemetry counter helper API drift
**Impact:** The metrics helper used in other parts of `session_health.py` may have a
different signature than expected at build time, causing the counter call to crash.
**Mitigation:** Build phase reads the existing counter call sites in `session_health.py`
(grep for `INCR\|incr_counter\|tier2_reprieve_total`) before writing the new call. The
counter call is wrapped in try/except WARNING per Failure Path Test Strategy.
