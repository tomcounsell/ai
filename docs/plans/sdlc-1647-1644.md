# Granite container: prime/work separation + mandatory user-facing wrap-up

Combined fix for two production bugs that share one root cause in
`agent/granite_container/`. Both were observed live in session
`ec1e7c6ede0b4cc491247067dea676f6` (2026-06-12). They ship as **one PR**
because the fix to either in isolation leaves the other half-broken.

- **#1644** — The PM's `[/dev]` instruction (emitted during its *prime*
  turn) is never relayed to the Dev. The Dev does the whole task anyway by
  self-starting from the raw user MESSAGE embedded in its prime
  `$ARGUMENTS`. The PM routing layer is dead weight on this path.
- **#1647** — The same session completed with **zero** user-facing
  messages. The PM never produced a `[/user]`/`[/complete]`, yet the
  container exited `completed` and the human got only a 👏 emoji. A
  high-quality Dev completion summary existed but was never harvested
  (it lived inside the Dev's prime turn, which the operator never parses
  — the same root cause as #1644).

## Problem

### Shared root cause

The container conflates **persona installation** (prime) with **work
initiation** and **routing**. Concretely, in `agent/granite_container/container.py`:

1. `_prime_session()` (line ~525) writes `f"{slash_cmd} {self.user_message}"`
   to **both** the PM and the Dev PTYs. The raw user MESSAGE rides along in
   `$ARGUMENTS` for the Dev too, so the Dev can — and does — start the whole
   task during its prime turn, directly contradicting its own persona text
   ("Wait for the PM to send the first instruction... you do not start work
   on your own", `.claude/commands/granite-poc/prime-dev-role.md:16`).
2. The **startup loop** (lines ~599–637) only runs `_handle_startup()` →
   `parse_startup_frame()`, which looks for *startup events*
   (trust-folder, update notice). It **never** runs `classify_pm_prefix()`
   against the PM's prime-turn buffer. So the PM's prime-turn `[/dev]`
   emission is invisible — `response=None`, "both idle, breaking" — and the
   instruction is never relayed.
3. Because the Dev already self-started and finished, both PTYs are idle when
   the steady-state loop begins. Turn 0 re-reads the PM's **stale** prime
   buffer (still containing the prime-turn `[/dev]`), and either re-dispatches
   a task the Dev already finished or trips `PM_COMPLIANCE_NUDGE`
   (`container.py:167`) — the "Re-send your reply starting with exactly one of
   [/dev], [/user], [/complete]" prompt. That prompt makes the PM **re-send
   the stale `[/dev]`** instead of producing the missing user-facing wrap-up.
4. The container reaches `pm_complete`/`pm_max_turns`/exit with **no
   `[/user]`/`[/complete]` ever classified**, and exits "successfully".

### Why the human got only an emoji (#1647, deeper finding)

The completion emoji in `agent/session_executor.py:2000` is keyed on
`messenger.has_communicated()`. But `has_communicated()` returns
`len(self.messages_sent) > 0` (`agent/messenger.py:176`), and the granite
container delivers `[/user]`/`[/complete]` payloads through the
**BridgeAdapter** `send_cb` path (`bridge_adapter.py::_deliver_sync`), which
does **not** route through `messenger.send()`. So `messages_sent` never grows
on the granite path. **Even a perfectly routed `[/user]` would leave
`communicated=False`** and select `REACTION_SUCCESS` (the bare emoji) instead
of a content reply. This is a structural always-False, not just a
this-session fluke — it must be fixed for the guard in #1647 to mean anything.

### Minor swallowed error (#1647)

`agent/branch_manager.py::return_to_main` (line ~472) unconditionally runs
`git checkout main`. Inside a worktree, main is already checked out by the
primary working tree, so this **can never succeed** — it logs
`ERROR Failed to switch to main: fatal: 'main' is already used by worktree`
and is then swallowed. Noise that masks real failures.

## Solution

Appetite: **Medium**. Four coordinated changes, all rooted in the
prime/work/routing separation. The throughline: **the prime turn installs a
persona and nothing else; work is initiated exactly once, by the operator
relaying the PM's first instruction; and a session cannot reach `completed`
without a user-facing message having been routed.**

### S1 — Dev prime must not carry the user MESSAGE (#1644)

Stop the Dev from self-starting by removing the raw user MESSAGE from the
Dev's prime `$ARGUMENTS`. The PM is the only role that needs the user task
verbatim (it decides routing); the Dev should receive its instruction *only*
via the operator relay.

- In `container.py`, split prime sending so the **Dev** prime is sent
  **without** `self.user_message` appended — just `DEV_PRIME_SLASH_CMD`.
  The PM prime still carries `self.user_message`.
- Update `.claude/commands/granite-poc/prime-dev-role.md`: the `$ARGUMENTS`
  / "What the user said" section becomes a statement that no task is present
  yet and the Dev must wait for the operator's first instruction. This makes
  the persona text and the runtime behavior agree (today they contradict).

### S2 — Operator relays the PM's first instruction; prime is not the work trigger (#1644)

The PM legitimately emits `[/dev]` as its first substantive output. The
operator must **classify the PM's prime-turn buffer** and relay that first
instruction to the Dev, instead of treating prime-turn output as invisible.

- After both primes complete and before/at the start of the startup-settle
  break, run `classify_pm_prefix()` on the PM's prime-turn buffer. If it
  yields a `dev`/`user`/`complete` destination, route it through the **same**
  steady-state routing code (extract → write to Dev, or deliver to user, or
  complete) rather than discarding it.
- Refactor the steady-state routing block (`container.py` ~lines 694–874)
  into a reusable `_route_pm_classification(classification, pm_buf, result)`
  helper so the prime-turn relay and the steady-state loop share one code
  path (no duplicated routing logic — NO LEGACY rule).
- This resolves the "PM emits `[/dev]` during prime" question (#1644 Q3):
  the PM emitting `[/dev]` during prime is **fine**; the bug is the operator
  not relaying it. We make the operator relay it.

- **Stale-PM-buffer race guard (resolves BLOCKER 3).** Verified fact:
  `pty_driver.write()` resets `_turn_text=""` (`pty_driver.py:415`) only on a
  write to *that* PTY. The prime-relay's `dev` branch writes to the **Dev**
  PTY (resetting the Dev buffer) but **not** to the PM PTY — so the
  steady-state loop's first `_cycle_idle(self._pm_pty)` (`container.py:642`)
  would re-read the PM's still-stale prime-turn buffer (still containing the
  same `[/dev]`) and re-classify → **double Dev dispatch**. Reusing the
  content-floor discipline does NOT fix this: the stale buffer is already
  above the floor. The concrete fix is a **full dev cycle that ends in a PM
  write**, which resets the PM `_turn_text`:

  When `_route_pm_classification` routes a `dev` destination (whether on the
  prime-turn path or in steady state), it runs the **complete** cycle:
  write Dev → `_cycle_idle` the Dev → `summarize_for_pm(dev_buf)` →
  **`self._pm_pty.write(summary)`**. That final PM write resets the PM's
  `_turn_text` to `""`, so the next steady-state `_cycle_idle(self._pm_pty)`
  blocks for **fresh** PM output (its acknowledgement of the summary) instead
  of re-reading the stale prime-turn `[/dev]`. Because the prime-relay path
  calls the *same* helper, it inherits this PM-resetting write — the prime
  `[/dev]` is consumed exactly once.

  Belt-and-suspenders for the non-dev prime cases (`[/user]`/`[/complete]`
  during prime, which do not write back to PM): after the prime-turn relay
  returns, set a one-shot `self._prime_relayed = True`. The steady-state loop
  skips re-classifying the PM buffer on its very first iteration *only if*
  `self._prime_relayed` is set AND the PM `_turn_text` has not changed since
  the relay (compare against a stored `self._prime_pm_buf_hash =
  hash(pm_buf)`); on a hash match it issues a fresh-output wait
  (`_cycle_idle`) before classifying. This guarantees the identical buffer is
  never classified twice regardless of which destination the prime turn
  produced. The regression test asserts **exactly one** Dev dispatch:
  `dev_pty.write.call_count == 1` for the relayed instruction.

### S3 — `communicated` reflects granite deliveries (#1647, prerequisite)

The `communicated` flag must flip when the granite path actually delivers a
`[/user]`/`[/complete]` payload, so the completion emoji and the S4 guard
mean what they say.

**Chosen mechanism (MANDATED — no longer an open question):** Option (b),
threading a `user_facing_routed` boolean back out of the container via the
`agent_session`, because Option (a) is *dead on arrival*. Verified against the
code: `BridgeAdapter.__init__` (`bridge_adapter.py:131-190`, constructed at
`session_executor.py:1707`) holds **no** messenger reference — its ctor
receives `agent_session, project_key, transport, pool, resolve_callbacks,
delivery_timeout_s, session_env, pm_system_prompt, pm_model` and nothing else.
The `messenger` is built separately at `session_executor.py:1355` and handed
only to `BackgroundTask` at `session_executor.py:1740`. So "have the adapter
mark the shared messenger" cannot work without a constructor-signature change
the plan never scoped. We therefore use the carrier the adapter *already*
holds: `self._agent_session`.

Concrete wiring (exact path, end to end):

1. **`_deliver_sync` returns `bool`** (folds in concern C1). Today it returns
   `None` whether delivery succeeds, times out, or raises (it swallows all
   exceptions at `bridge_adapter.py:420-435`). Change its signature to
   `-> bool`: return `True` only on a confirmed successful delivery (the sync
   send_cb returned without raising at line 388, or the same-loop
   `loop.create_task` schedule at line 414, or `future.result(timeout=...)`
   returned without raising at line 419); return `False` on every
   failure/timeout/no-loop branch (lines 401, 423, 435) — the existing
   `_record_delivery_failure` calls stay.
2. **The user/complete callbacks propagate the bool.** `_make_user_callback`
   and `_make_complete_callback` (lines 322-357) currently return
   `Callable[[str], None]`. Change their inner `_on_user`/`_on_complete` to
   capture the `_deliver_sync` return and, **on `True` only**, set a flag on
   the adapter: `self._user_facing_routed = True`. Add
   `self._user_facing_routed: bool = False` to `__init__`. (The container's
   `on_user_payload`/`on_complete_payload` hooks remain `-> None` — the flag
   lives on the adapter, not the container.)
3. **The adapter publishes the flag onto `agent_session` after the run.** In
   `_publish_exit_summary(result)` (called at `bridge_adapter.py:246`, on the
   asyncio thread after `Container.run` returns), set
   `self._agent_session.user_facing_routed = self._user_facing_routed or
   result.user_facing_routed` and persist it with the existing partial-save
   pattern (`save(update_fields=["user_facing_routed", "updated_at"])`). The
   OR-in of `result.user_facing_routed` covers the S2 prime-relay
   `[/user]`/`[/complete]` and the S4 wrap-up delivery, which set the flag on
   the `ContainerResult` inside the container thread (see S4). `agent_session`
   gets a new `user_facing_routed = Field(default=False)` on the
   `AgentSession` model.
4. **The executor reads it in the emoji branch.** At `session_executor.py:2000`
   the branch is currently `elif messenger.has_communicated():`. Change it to
   `elif messenger.has_communicated() or getattr(agent_session,
   "user_facing_routed", False):`. This is the *one* flag the executor already
   consults plus the new granite carrier OR'd in — no new parallel signal path
   feeding `has_communicated()` itself, and `messages_sent` is left untouched
   (the granite path never calls `messenger.send()`, so mutating
   `messages_sent` would be a lie about which transport delivered).

This keeps the invariant ("one decision point the executor consults") while
using only carriers that already exist at each hop: the adapter's
`agent_session` ref, the `ContainerResult` it already returns, and the
executor's existing `agent_session` local.

### S4 — A granite session cannot reach `completed` with zero user-facing messages (#1647)

This is the core guard. When the container is about to exit "successfully"
(`pm_complete`, `pm_user`, `pm_max_turns`, or normal end) **and** nothing
user-facing was ever routed, the operator must drive the PM to produce a
`[/user]`/`[/complete]` wrap-up — **seeded with the Dev's final report** —
before exiting. Only after a genuine attempt (bounded retries) may it fall
back to a hardcoded operator-sent terminal message (NOT a bare emoji — see
the terminal-fallback bullet, which folds in concern C2).

- Track `user_facing_routed: bool` on `ContainerResult` (default `False`;
  set `True` whenever a `[/user]` payload, or a `[/complete]` *with a
  non-empty body*, is routed — including via the S2 prime relay and the S4
  wrap-up delivery). This is the carrier S3's `_publish_exit_summary` OR's
  onto `agent_session`.

- **Dev-final-report seed capture (resolves BLOCKER 2).** The plan's original
  `summarize_for_pm(dev_buf)` seed had no accessible source at wrap-up: in
  the *primary* failure scenario (Dev self-starts during prime, completes,
  steady-state sees PM already idle and exits) the loop-local `dev_buf`
  (`container.py:835`) is **never assigned**, and `pty_driver.write()` resets
  `_turn_text=""` (`pty_driver.py:415`), so the Dev's report can be gone from
  the live screen too. We capture the seed deterministically:

  - **New instance field `self._last_dev_report: str | None = None`** on
    `Container`, set at two capture points:
    1. **Primary — steady-state harvest:** in the steady-state `dev`
       branch, immediately after `summary = summarize_for_pm(dev_buf)`
       succeeds (`container.py:846`), assign `self._last_dev_report = summary`.
       This captures every completed Dev turn's summary, so multi-turn runs
       carry the latest.
    2. **Prime-path harvest (covers the primary #1647 scenario):** S1 stops
       the Dev from self-starting, so post-S1 the Dev produces no prime-turn
       work. But for defense-in-depth and to harvest a Dev that *did* respond
       to the S2-relayed instruction during what is effectively turn 0, the
       wrap-up guard does a **fresh `_cycle_idle(self._dev_pty)` read at
       wrap-up time** (option (c) in the critique) and runs
       `summarize_for_pm` on that buffer when `self._last_dev_report is None`
       and the Dev buffer is non-empty. This reads the Dev's *current* screen
       rather than relying on a loop-local that was never assigned.
  - **Mandatory canned fallback (resolves the empty-seed NameError/empty
    string).** When neither capture point yields content
    (`self._last_dev_report` is `None`/blank AND the fresh Dev read is blank),
    seed the wrap-up prompt with the literal constant
    `DEV_REPORT_UNAVAILABLE = "The developer did not produce a captured
    report."` so `PM_WRAPUP_PROMPT` stays grammatical and never interpolates
    an empty string or raises `NameError`.

- On a successful-shaped exit with `user_facing_routed == False`: before
  finalizing, write `PM_WRAPUP_PROMPT` (a module constant, distinct from
  `PM_COMPLIANCE_NUDGE`) to the PM, interpolating the Dev-report seed above.
  Re-read the PM via `_cycle_idle(self._pm_pty)`, classify, and route through
  `_route_pm_classification`. Cap at `MAX_WRAPUP_ATTEMPTS = 1` (see Open
  Questions — a refusing PM refuses again; a second attempt only adds
  latency). If after the capped attempt(s) still nothing user-facing was
  routed:
  1. **Send a hardcoded operator terminal message on the user channel,
     bypassing the PM** (resolves concern C2 — the human must never be left
     with only an emoji, which IS the exact #1647 symptom). The operator
     calls the adapter's user-delivery path directly with the constant
     `OPERATOR_TERMINAL_MESSAGE = "Your request was completed; a summary
     could not be generated."` This delivery sets `user_facing_routed = True`
     on success (so the emoji becomes `REACTION_COMPLETE`, and the human has a
     real message). The terminal message is delivered via the same
     `on_user_payload` callback the routing path uses, so it inherits the
     `_deliver_sync` bool + flag wiring from S3.
  2. Set `exit_reason = "pm_no_user_message"` (a distinct reason
     `_maybe_publish_exit_anomaly` surfaces) so the anomaly is logged even
     though the human got the canned terminal message.

- **`PM_WRAPUP_PROMPT` content spec (folds in concern C3).** The template must
  instruct the PM to **pass through the concrete specifics from the Dev
  report — files changed, commands run, outcomes — not paraphrase into a
  vacuous "Task completed."** Shape: "The developer has finished. Here is
  their final report:\n\n«seed»\n\nSend your `[/user]` or `[/complete]`
  summary to the human now. Include the specific outcomes from the report
  above — which files changed and what was done — not a generic
  acknowledgement." This keeps the wrap-up substantive rather than empty.

### S5 — Replace the harmful routing-prefix retry, and make `[/complete]` require a body (#1647 Q2, Q3)

- **Q2:** The bare `PM_COMPLIANCE_NUDGE` is the prompt that pushed the PM to
  re-send a stale `[/dev]`. Where the operator detects the PM is being asked
  to wrap up (S4 path), use the **wrap-up prompt** (S4), not the generic
  compliance nudge. The generic nudge stays only for a genuine
  *no-prefix-at-all* miss in mid-loop. This narrows the nudge's blast radius
  so it can no longer re-dispatch finished work.
- **Q3:** Make `[/complete]` with an empty body **not** count as a
  user-facing message. "Empty body" is defined precisely as
  `not classification.payload.strip()` (C7) — the same predicate the existing
  empty-`[/dev]` guard at `container.py:768` uses. In the routing path
  (`_route_pm_classification`), a `complete` destination with
  `not classification.payload.strip()` is treated like a compliance miss → it
  does NOT set `user_facing_routed` and does NOT deliver; it falls through to
  the S4 wrap-up rather than silently terminating. A `[/complete]` whose
  `payload.strip()` is non-empty delivers it and sets
  `result.user_facing_routed = True`.

### S6 — Fix the worktree `return_to_main` error (#1647 minor)

- In `branch_manager.py::return_to_main`, detect when `working_dir` is a
  **linked worktree** (`git rev-parse --is-inside-work-tree` +
  `git rev-parse --git-common-dir` differing from `--git-dir`, or
  `git rev-parse --show-toplevel` against the main worktree). When it is, do
  **not** attempt `git checkout main` — it can never succeed inside a
  worktree. Log at debug/info ("worktree: skipping checkout main") instead of
  ERROR, and return `True` (the no-op is success). Non-worktree behavior is
  unchanged.

## Technical Approach

- **Routing helper extraction (S2):** lift the four routing branches
  (`complete`, `user`, `dev`-empty, `dev`-extract) from `Container.run`'s
  steady-state `for` loop into `_route_pm_classification(self, classification,
  pm_buf, turn_index, result) -> RouteOutcome`, where `RouteOutcome` carries
  `(should_break, exit_reason)`. The startup phase calls it once on the prime
  buffer; the loop calls it per turn. This avoids duplicating the routing
  logic and keeps a single source of truth.
- **`communicated` carrier (S3) — DECIDED, Option (b):** the adapter does
  NOT hold the messenger (`bridge_adapter.py:131-190` ctor takes no
  messenger; the messenger is built at `session_executor.py:1355` and handed
  only to `BackgroundTask` at `:1740`), so the "mark the shared messenger"
  option is impossible without an unscoped ctor change. Instead:
  `_deliver_sync` returns `bool` (concern C1); `_make_user_callback` /
  `_make_complete_callback` set `self._user_facing_routed = True` only when it
  returns `True`; `_publish_exit_summary` writes
  `agent_session.user_facing_routed = self._user_facing_routed or
  result.user_facing_routed` (partial-saved); a new
  `AgentSession.user_facing_routed = Field(default=False)` field carries
  it; and the executor's emoji branch at `session_executor.py:2000` becomes
  `elif messenger.has_communicated() or getattr(agent_session,
  "user_facing_routed", False):`. `messages_sent` is left untouched (granite
  never calls `messenger.send()`). Full path written in S3.
- **Wrap-up + terminal constants (S4/S5):** new module constants in
  `container.py`:
  - `PM_WRAPUP_PROMPT` — template interpolating the Dev-report seed,
    instructing the PM to pass through concrete specifics (files/outcomes),
    not paraphrase (C3).
  - `MAX_WRAPUP_ATTEMPTS = 1` (Open Question #2 closed — a refusing PM
    refuses again; a retry only adds latency).
  - `DEV_REPORT_UNAVAILABLE = "The developer did not produce a captured
    report."` — the mandatory canned seed fallback for the empty case
    (BLOCKER 2).
  - `OPERATOR_TERMINAL_MESSAGE = "Your request was completed; a summary
    could not be generated."` — the hardcoded operator-sent user-channel
    message delivered (via the `on_user_payload` callback, bypassing the PM)
    when the bounded wrap-up still yields nothing, so the human is never left
    with only an emoji (C2). On successful delivery it sets
    `result.user_facing_routed = True`.
  - New `exit_reason = "pm_no_user_message"` added to the anomaly allowlist in
    `bridge_adapter.py::_maybe_publish_exit_anomaly` — the tuple at
    `bridge_adapter.py:277` becomes `("pm_hang", "dev_hang",
    "startup_unresolved", "pm_no_user_message")`. `pm_max_turns` is
    **deliberately NOT** added (concern C9) — a max-turns exit that delivered
    a wrap-up or terminal message is not an anomaly.
- **Dev-report seed capture (S4, BLOCKER 2):** new instance field
  `Container._last_dev_report: str | None`, assigned after each successful
  steady-state `summarize_for_pm` (`container.py:846`); at wrap-up, when it is
  `None`/blank, a fresh `_cycle_idle(self._dev_pty)` read + `summarize_for_pm`
  supplies the seed; if still blank, `DEV_REPORT_UNAVAILABLE` is used. Never
  references the loop-local `dev_buf` (which is unassigned on the primary
  failure path).
- **Worktree detection (S6, concern C8):** run `git rev-parse
  --git-common-dir` and `git rev-parse --git-dir`, then compare them
  **after normalizing both with `os.path.realpath()`** (the common-dir is
  often returned as a relative `.git` in the main worktree while the git-dir
  is absolute, which would otherwise false-positive the main repo as a linked
  worktree). If the normalized paths differ → linked worktree → skip
  `checkout main`, log at info, return `True`. If either `rev-parse`
  **fails** (non-git cwd — e.g. the sandbox tempdir, where the command exits
  non-zero), catch the error and fall back to the current checkout behavior
  with **no ERROR log** (it is a legitimate non-git caller, not a failure).

## Race Conditions

- **Double-route of the prime-turn `[/dev]` (the BLOCKER-3 race).**
  `pty_driver.write()` resets `_turn_text=""` only on a write to *that* PTY
  (`pty_driver.py:415`); the prime-relay's Dev write does NOT reset the PM
  buffer, so a naive steady-state loop would re-read the stale prime `[/dev]`
  and dispatch twice. Mitigation (concrete, per S2's race guard): the `dev`
  routing branch always ends in `self._pm_pty.write(summary)`, which resets
  the PM `_turn_text` so the next `_cycle_idle(self._pm_pty)` waits for fresh
  PM output. For non-dev prime cases the one-shot `self._prime_relayed` flag +
  `self._prime_pm_buf_hash` comparison forces a fresh-output wait before the
  first steady-state classification when the PM buffer is byte-identical to
  the relayed one. The regression test asserts **exactly one** Dev dispatch
  (`dev_pty.write.call_count == 1` for the relayed instruction) — not merely
  "a Dev dispatch happened".
- The S4 wrap-up write happens on the pexpect worker thread; the
  `_deliver_sync` it triggers schedules onto the captured loop via
  `run_coroutine_threadsafe` and blocks — same pattern already hardened in
  #1572 (ADV-5). No new cross-thread primitive introduced.

## Failure Path Test Strategy

- **PM never wraps up:** simulate a PM that only ever emits `[/dev]`; assert
  the operator issues `PM_WRAPUP_PROMPT`, and on continued silence delivers
  `OPERATOR_TERMINAL_MESSAGE` on the user channel (so the human is never left
  with only an emoji — C2), exits with `pm_no_user_message`, the anomaly
  event is published, and `user_facing_routed` is True.
- **Empty Dev-report seed (BLOCKER 2):** simulate the primary scenario where
  no `dev_buf` was ever assigned in the loop and the Dev screen is blank;
  assert the wrap-up prompt is seeded with `DEV_REPORT_UNAVAILABLE` (no
  `NameError`, no empty interpolation).
- **Wrap-up loop bound:** assert the wrap-up retries are capped at
  `MAX_WRAPUP_ATTEMPTS` (= 1) and do not spin.
- **Delivery failure (depends on C1's bool return):** a failing `send_cb`
  makes `_deliver_sync` return `False`; assert `user_facing_routed` stays
  `False`, a `delivery_failure` session_event is recorded, the container does
  not crash, and the emoji falls back to `REACTION_SUCCESS`. This test only
  means anything because `_deliver_sync` now distinguishes delivered from
  attempted (C1).
- **Worktree return_to_main (C8):** simulate a linked-worktree `working_dir`
  (rev-parse common-dir ≠ git-dir after realpath); assert no ERROR log and
  `True` return. Simulate a non-git cwd (rev-parse raises); assert it falls
  back to the checkout path with no ERROR log. Simulate a normal repo
  (common-dir == git-dir) and assert the original checkout path still runs.
