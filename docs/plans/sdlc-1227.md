# PM session first-turn cold start: eliminate 15–20 min TTFT

## Problem

Every newly spawned PM session pays a 15–20 minute time-to-first-token (TTFT) on its first turn. The worker logs the pattern verbatim:

```
[harness] Using --model opus for session_id=0_1777540282099
[harness] Appending 74769-char system prompt for session_id=0_1777540282099
SDK heartbeat: running 60s, communicated=False
SDK heartbeat: running 120s, communicated=False
…
SDK heartbeat: running 840s, communicated=False
```

The same pattern reproduced in Apr 28 logs (`tg_valor_-5051653062_9413`): 1020 s of `communicated=False`, worker restart, then `--resume` finishes in 59 s. Once API-side cache exists, subsequent turns are fast — the *first* turn is the bottleneck.

Three compounding factors cause this:

1. The full persona + project CLAUDE.md (~74,769 chars / ~18,750 tokens) is concatenated and passed as a CLI argument to `claude` via `--append-system-prompt` (`agent/sdk_client.py:2125`).
2. PM sessions default to `--model opus` (`agent/session_executor.py:34-54`).
3. **No prompt-caching infrastructure exists anywhere in the codebase** — `cache_control` / `ephemeral` markers are never set; no cache-hit metrics are collected; each new `claude -p` invocation pays full cold-start cost even if a previous session just processed an identical prefix.

**Current behavior:** A fresh PM session is dormant for 15–20 minutes before the first observable token or tool call. Users perceive the system as hung.

**Desired outcome:** A new PM session produces its first observable output (token or tool call) in **under 90 seconds** on a typical `claude -p` invocation. The 15–20 minute cold-start regime is eliminated.

**Root cause pattern:** The system prompt was built for *correctness* (PM has the right context) without considering *latency* (cold-start TTFT) or *caching* (each invocation is its own cache key). The fix needs to either (a) annotate the existing prompt for caching so the second invocation is hot, (b) shrink the prompt's required first-turn surface, or (c) use a cheaper model for the first turn — preferably some combination, evaluated in cost-of-implementation order.

## Data Flow

1. **Entry point:** Worker picks up an `AgentSession` with `session_type="pm"`.
2. **`agent/session_executor.py:1417-1424`:** PM-branch loads system prompt via `load_pm_system_prompt(working_directory)`.
3. **`agent/sdk_client.py:998-1026`:** Reads persona base (`config/personas/project-manager.md` or `~/Desktop/Valor/personas/project-manager.md`) + appends `{working_directory}/CLAUDE.md` if it exists. Returns concatenated string (~74K chars).
4. **`agent/sdk_client.py:2105-2129`:** Builds `harness_cmd`, sets `--model opus`, validates `len(system_prompt) <= 512_000`, appends `--append-system-prompt {system_prompt}` to argv.
5. **Subprocess spawn:** `claude -p` is exec'd. The CLI reads stdin/argv and constructs the API request to Anthropic with the appended system prompt.
6. **Anthropic API:** Receives request with full prefix. With no `cache_control` markers → full cold-start cost (~15–20 min on a 18K-token prefix to Opus during cluster load). With markers → ~5–60s on a cache hit.
7. **Output:** First stdout chunk reaches the worker → `communicated=True` → SDK heartbeat stops complaining.

The bottleneck is step 6. Steps 1–5 are sub-second.

## Solution

### Direction Evaluation Order (per issue mandate: A → B → C → D)

The four directions are not mutually exclusive; the plan evaluates them in cost-of-implementation order and stops once the 90-second criterion is met.

| # | Direction | Implementation Cost | Expected Win | Decision Trigger |
|---|-----------|--------------------|--------------|------------------|
| A | Prompt caching at the harness/SDK layer | Low (1–2 files, no new deps) | High if confirmed by spike-1 | Spike-1 confirms `--append-system-prompt` is NOT auto-cached |
| B | Shrink prompt via lazy loading (split persona + reference into discoverable parts) | Medium (refactor `load_pm_system_prompt`, update tests, add lazy-load tool) | Medium — saves 50–60% of tokens but doesn't fix cold-start uniformly | A insufficient OR cache TTL insufficient |
| C | Cheaper model for first turn (Sonnet/Haiku for routing, then escalate to Opus) | Medium (per-turn model selection in `_resolve_session_model` + handoff plumbing) | Medium — 5–10x faster TTFT on Sonnet at the cost of routing quality | A+B insufficient |
| D | Pre-warming hot Opus context | High (long-running `claude` process, IPC, lifecycle mgmt) | High but operationally complex | A+B+C insufficient — last resort |

**Default plan path:** Spike → A → measure → if insufficient, B → measure → if insufficient, C → measure. D is documented as a "do not implement unless A+B+C all fail" outcome (see Rabbit Holes).

### Key Elements

- **`agent/prompt_cache.py` (new, conditional)** — Owns the cache-annotated system-prompt builder if Direction A's spike requires explicit `cache_control` markers. Exposes `build_cached_pm_system_prompt(working_directory) -> CachedSystemPrompt` returning a dataclass with `{persona_part: str, project_claude_md_part: Optional[str]}` so the harness invocation site can pass cache-marker hints to whichever transport (CLI flag or direct SDK call) is used.
- **`load_pm_system_prompt()` refactor (Direction B, conditional)** — If A is insufficient, split the function into `load_pm_first_turn_prompt()` (identity + role + critical rules, ~5K chars) and a discoverable lazy-load surface for the rest of CLAUDE.md (e.g., a tool the PM can call to fetch sections on demand, or a `tools/load_project_claude_md.py` CLI). The first-turn prompt must include enough context that the PM can route the first request without halting on missing info.
- **Per-turn model selection (Direction C, conditional)** — Extend `_resolve_session_model` (`agent/session_executor.py:34-54`) and the harness invocation to support a "first-turn model" override. PM session starts with `--model sonnet` for the routing turn, then `--resume` switches to `opus` for sustained reasoning. This requires that `--resume` accepts a different model than the original invocation (verify in spike).
- **Measurement harness** — A small instrumentation that logs `claude -p` spawn → first-stdout-byte elapsed time per session, written to `logs/cold_start_metrics.jsonl` so we have an empirical before/after for each direction. This lands in Task 1 and is used as the verification gate for every subsequent direction.
- **Cache-hit metric** — If Direction A surfaces `cache_creation_input_tokens` / `cache_read_input_tokens` (Anthropic API response fields), log them per session to the same JSONL. If the harness CLI does not surface them, we accept TTFT as the sole metric.

### Flow

PM session created → worker dispatches → harness loads system prompt → **measurement instrumentation logs spawn time** → `claude -p` invokes → API responds → **first stdout chunk logs TTFT** → PM does its work → session ends. The flow is unchanged; instrumentation is additive. If Direction C lands, the flow gains a "first turn on Sonnet, resume on Opus" branch in the harness invocation.

### Technical Approach

1. **Spike-first** — Implement minimal instrumentation (Task 1) that records TTFT for every PM session in `logs/cold_start_metrics.jsonl`. Run it for 24 hours of normal worker activity to get a baseline distribution. This data will validate every subsequent direction's claim of improvement.
2. **Spike-1: empirical cache check** — Run two back-to-back PM sessions on the same `working_directory` (same `load_pm_system_prompt()` output) within 5 minutes. Compare TTFT. If second is dramatically faster than first → CLI auto-caches → Direction A is partial (we just need to ensure cache TTL doesn't expire between PMs in production). If both are slow → CLI does NOT auto-cache → Direction A means manual annotation.
3. **Direction A — manual cache annotation** (only if spike-1 says CLI does not auto-cache):
   - Investigate whether the `claude` CLI accepts a flag or env var that toggles `cache_control` on `--append-system-prompt` content. If yes → use it.
   - If not → bypass the CLI for the first turn: use `agent/anthropic_client.py`'s direct SDK pathway to send the first message with explicit `cache_control: {"type": "ephemeral"}` on the system prompt block, capture the response, then hand control to `claude -p --resume` for subsequent turns. The first-turn output goes through the same `OutputHandler` protocol so user-visible behavior is unchanged.
   - Critically: the bypass path must NOT regress #1148 (PM persona overlay must still be present in the system prompt). The cached prefix must be the same ~74K char string the CLI would otherwise pass.
4. **Measurement gate** — After Direction A lands, sample 10 fresh PM sessions and verify median TTFT < 90s. If yes → done; A meets criteria. If no → proceed to Direction B.
5. **Direction B — lazy load (conditional)** — Split `load_pm_system_prompt()` into a 5–10K-char first-turn prompt and a discoverable surface for the rest. The first-turn prompt must be self-sufficient for routing (PM can pick subskill, slug, message). Reference material (Quick Commands table, env config) is fetchable on demand.
6. **Direction C — cheaper first-turn model (conditional)** — Add a `first_turn_model` field to `_resolve_session_model` returning `("sonnet", "opus")` for PM sessions. The harness invokes with `--model sonnet` initially, then on resume switches to `--model opus`. Verify that `claude -p --resume --model opus` actually upgrades the model on subsequent turns (this may require investigation; if not supported, the implementation is harder and Direction C is downgraded).
7. **Direction D is OUT OF SCOPE** — pre-warming a hot Opus context is documented as a Rabbit Hole. Operational complexity exceeds the appetite.

### Why Direction A first

The issue body explicitly orders directions A → B → C → D. A has the smallest blast radius (cache-marker annotation is a string-construction change), the largest known potential win (a 5-minute prompt cache TTL turns a 15-minute cold start into a sub-minute warm start for any subsequent session within the window), and is the most reversible (revert one file). B/C/D require deeper structural changes and should only be reached if A is insufficient.

## Rabbit Holes

- **Direction D (pre-warming hot Opus context)** — Maintaining a long-running `claude` process the worker hands new PM sessions to. Sounds elegant; in practice it's a giant lifecycle-management problem (process death, memory leaks, persona drift if the process is hot for too long, IPC protocol design). The 5-minute Anthropic prompt cache TTL gives us most of the benefit "for free" via Direction A without any of the operational risk. Do NOT implement D in this plan.
- **Building our own caching layer** — Anthropic provides server-side prompt caching with a documented TTL. Building a local Redis-backed prompt-prefix cache that intercepts and short-circuits API calls is reinventing the wheel and breaks correctness (we cannot cache model output without invalidation rules). Use Anthropic's caching only.
- **Aggressively shrinking CLAUDE.md** — Tempting because the file is large. But CLAUDE.md is the single source of truth for project rules and is referenced by every developer (human and agent). Mutilating it for prompt-size optimization breaks the developer experience. Direction B (lazy loading) is the right tool — leave CLAUDE.md intact and let the PM discover sections on demand.
- **Switching PM default model to Sonnet permanently** — Sonnet may handle routing well but is weaker on sustained planning reasoning, which is the PM's core competency. Direction C (Sonnet first-turn, Opus resume) preserves Opus quality where it matters; a permanent Sonnet downgrade does not.
- **Tuning the 512KB ARG_MAX guard** — The guard at `agent/sdk_client.py:2118` is correct as-is and protects against macOS ARG_MAX. Do not raise the limit; the right answer is to shrink the prompt or move it off argv (e.g., via stdin or a temp file). Stdin/temp-file delivery is its own rabbit hole — the harness CLI's accepted input shapes are constrained, and changing them is not in scope.

## Risks

### Risk 1: Anthropic prompt cache does NOT survive between independent `claude -p` invocations

**Impact:** Direction A's win is limited to within-session turn 2+ (not the cold-start turn 1, which is the issue). We'd need Direction B/C to actually solve the issue.

**Mitigation:** Spike-2 resolves this empirically before we commit to Direction A as the sole fix. If cache does not survive cross-process, the plan path is A (best-effort, win on resume) + B (shrink to make first turn cheaper).

### Risk 2: `claude -p` CLI does not expose `cache_control` annotation

**Impact:** Direction A becomes "bypass the CLI for the first turn", which adds a new direct-SDK code path that must integrate with `OutputHandler` and `--resume`. More complex, more risk surface.

**Mitigation:** The bypass path is well-bounded — first turn only, then hand off to `--resume`. The `OutputHandler` protocol exists explicitly for transport-agnostic delivery. If integration proves harder than expected, fall back to Direction B.

### Risk 3: Refactoring `load_pm_system_prompt()` breaks #1148 (PM persona overlay missing)

**Impact:** Regression on the persona overlay loading; PM sessions act like generic Claude without the project-manager persona.

**Mitigation:** The test contract at `tests/unit/test_sdk_client.py:308-432` asserts the persona overlay is present. Any refactor that changes the return type must port that assertion forward. A new explicit test asserts the persona content is in the cached prefix.

### Risk 4: TTFT measurement instrumentation distorts measurement

**Impact:** Adding instrumentation that itself takes 100s of ms could mask the real win.

**Mitigation:** Instrumentation is a single `time.monotonic()` call at spawn and at first-stdout-byte. JSONL append is async/best-effort. Overhead is sub-millisecond.

### Risk 5: Direction C's Sonnet-then-Opus handoff loses context

**Impact:** Sonnet's first-turn output (routing decision) doesn't transfer cleanly to Opus on `--resume`, and Opus has to re-read the conversation, defeating the win.

**Mitigation:** Test Direction C with realistic PM tasks before committing. If `--resume` truly upgrades the model AND retains conversation, ship it. If not, downgrade the direction or drop it.

## Race Conditions

- **Race 1: Two PM sessions spawn simultaneously, both miss the cache** — If both PM sessions are first-turn cold and the Anthropic cache hasn't been populated yet, both pay full cost. This is acceptable — caching is best-effort. The next session within 5 minutes is hot.
- **Race 2: `--resume` from cached first-turn (Direction A bypass path)** — If the direct-SDK first-turn writes a session but `claude -p --resume` cannot find the session ID (because it was never registered with the CLI's session registry), the resume fails. **Mitigation:** verify in spike that the CLI accepts externally-created session IDs OR have the bypass path fall through to a fresh `claude -p` for subsequent turns (still benefits from Anthropic-side cache hit, just not from `--resume`).
- **Race 3: TTFT log file write contention** — Multiple workers writing to `logs/cold_start_metrics.jsonl` simultaneously. **Mitigation:** use `O_APPEND` open mode, which is atomic on POSIX for line-sized writes (each line is < PIPE_BUF = 4KB). Single-line JSON entries, no partial writes.

## No-Gos (Out of Scope)

- **Pre-warming hot Opus context** (Direction D). Operational complexity exceeds appetite. Documented in Rabbit Holes.
- **Aggressively shrinking CLAUDE.md** content. CLAUDE.md is owned by the project and should not be mutilated for prompt-size optimization. Direction B is the right tool.
- **Switching PM default model permanently to Sonnet**. PM needs Opus for sustained reasoning.
- **Building a local prompt-cache layer**. Use Anthropic's server-side caching only.
- **Raising or removing the 512KB ARG_MAX guard** at `agent/sdk_client.py:2118`. This guard is correct and protects against macOS limits.
- **Migrating off `claude -p` to a different harness**. The harness abstraction is correct; the issue is prompt construction, not transport.
- **Adding new Python dependencies** (constraint from issue).
- **Touching the dev-session or teammate-session prompt loading paths**. This issue is PM-specific. Other session types do not pay the same TTFT cost (smaller prompts, different defaults).

- **The bridge** does not need to import or call any new code — it continues to enqueue PM sessions and the worker continues to spawn them; only the spawn-side prompt construction changes.
- **No new tools** in `tools/` need to be exposed to the agent. (Direction B's lazy-load surface, IF implemented, would be a new tool — but that is conditional and would be designed at that time, with its own integration test.)

If Direction B lands and adds a lazy-load tool: a new CLI entry point like `valor-load-claude-md = "tools.load_claude_md:main"` may be added. That is conditional and the design happens at the Direction-B decision point.
