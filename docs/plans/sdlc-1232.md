# Unify Memory Writer project_key Resolution into a Single Helper

## Problem

Five separate Memory writer callsites each resolve `project_key` with their own ad-hoc fallback logic. Each chain is similar in shape but diverges in detail — different cwd sources, different env var names, different defaults — which means any drift in one callsite is invisible to the others. PR #820 (closed #811) flipped the global constant from `"dm"` to `"default"` but the bridge writers were unaffected, which is exactly how the `dm`-namespace leak that spawned #1173 persisted for weeks.

**Current behavior (post-#1173 merge):**

| Callsite | Resolution chain |
|---|---|
| `bridge/telegram_bridge.py` | `project.get("_key")` or `None` → skip; no fallback to a named default (patched in #1173) |
| `agent/memory_hook.py` | `VALOR_PROJECT_KEY` env → `config.memory_defaults.DEFAULT_PROJECT_KEY` |
| `tools/memory_search/__init__.py::_resolve_project_key` | arg → `VALOR_PROJECT_KEY` env → `DEFAULT_PROJECT_KEY` |
| `.claude/hooks/hook_utils/memory_bridge.py::_get_project_key` | `VALOR_PROJECT_KEY` env → projects.json cwd match → cwd basename → `DEFAULT_PROJECT_KEY` |
| `agent/memory_extraction.py` (extract + post_merge_extract) | arg → `VALOR_PROJECT_KEY` env → `DEFAULT_PROJECT_KEY` |

The chains are not identical. `memory_bridge.py` is the richest (env → projects.json match → cwd basename → default), while `memory_hook.py` and `memory_extraction.py` skip the cwd-match branch entirely. If projects.json is extended with new resolution hints, each callsite must be updated independently.

**Desired outcome:**

A single `resolve_project_key(cwd, env, projects_path)` helper — housed in `config/` or `tools/memory_search/` — that every Memory writer imports. The helper encapsulates the full priority chain and returns `str | None`. Writers handle `None` consistently: skip the write (the bridge already does this post-#1173). The `_warn_if_legacy_namespace` guard in `Memory.safe_save` (introduced in #1173) becomes an effective regression detector because all callsites share one implementation.

## Data Flow

The new helper sits between callers and `Memory.safe_save`. Each writer already has all needed inputs; the unification just centralizes the decision logic.

1. **Entry**: A Memory write event occurs (user message ingested, session observation extracted, post-merge learning distilled, knowledge chunk indexed).
2. **Caller** (`bridge/`, `agent/memory_hook.py`, `agent/memory_extraction.py`, `tools/memory_search/`, `.claude/hooks/`): collects available context (cwd, env, project object), calls `resolve_project_key(...)`.
3. **`resolve_project_key()`**: Priority: explicit `project_key` arg → `VALOR_PROJECT_KEY` env var → projects.json cwd match → cwd basename → `None`. Returns `str | None`.
4. **Caller**: if result is `None`, skips the `Memory.safe_save(...)` call. If result is a string, passes it as `project_key=` to `safe_save`.
5. **`Memory.safe_save()`**: saves the record. `_warn_if_legacy_namespace()` fires if `project_key in ("dm", "default")` — audit trail for future regressions.

## Architectural Impact

- **New dependency**: A new module `config/project_key_resolver.py` (or `tools/memory_search/resolver.py`). All Memory writer callsites import from it. This is a leaf module with no imports from the writers — the dependency graph is a fan-in, not a cycle.
- **Interface changes**: `memory_bridge._get_project_key()` becomes a thin wrapper (or is replaced by a direct import). `tools/memory_search/__init__.py::_resolve_project_key()` is replaced. `agent/memory_hook.py` and `agent/memory_extraction.py` inline logic is extracted.
- **Coupling**: Decreases. Five separate resolution chains collapse into one. Future callers get the canonical resolver for free.
- **Data ownership**: No change — `Memory` model still owns the write. The resolver is a pure function (no side effects, no I/O except projects.json reads).
- **Reversibility**: Easy to undo — each callsite can revert to its old inline logic independently if needed.

## Solution

### Key Elements

- **`resolve_project_key(cwd, env, projects_path)` helper**: Pure function. Priority: explicit `project_key` arg → `VALOR_PROJECT_KEY` env → projects.json `working_directory` prefix match → cwd basename → `None`. Returns `str | None`. Housed in `config/project_key_resolver.py`.
- **Callsite migration**: Each of the 5 writers replaces its inline logic with a call to the new helper. Bridge callsite (post-#1234) already returns `None` on unresolved; migration confirms it passes through `resolve_project_key`.
- **Test matrix**: Unit tests cover every branch of the priority chain, including `None` return when all inputs are absent or empty.

### Flow

`Memory.safe_save(project_key=X)` caller →  
  `resolve_project_key(cwd=..., env=os.environ, projects_path=...)` →  
  `str | None` →  
  if `None`: log + skip →  
  if string: `Memory.safe_save(project_key=string)` →  
  `_warn_if_legacy_namespace(string)` (if `"dm"` or `"default"`)

### Technical Approach

- **Module location**: `config/project_key_resolver.py`. Placing it under `config/` means it has no imports from `agent/`, `tools/`, `bridge/`, or `models/` — avoiding circular imports. The existing `config/memory_defaults.py` is its only dependency (for `DEFAULT_PROJECT_KEY`).
- **Function signature**: `resolve_project_key(cwd: str | None = None, env: dict | None = None, projects_path: Path | None = None, project_key: str | None = None) -> str | None`. Accepts all available context; callers pass what they have.
- **Priority chain**: (1) `project_key` arg if non-None/non-empty → return immediately. (2) `env.get("VALOR_PROJECT_KEY")` if truthy. (3) projects.json working_directory prefix match against `cwd`. (4) `Path(cwd).name` if cwd is non-empty. (5) `None` — callers must handle this by skipping the write.
- **Removing `DEFAULT_PROJECT_KEY` as a fallback**: The helper deliberately does NOT fall back to `DEFAULT_PROJECT_KEY`. Any writer that previously fell through to `"default"` will now receive `None` if it has no cwd and no env var set. Writers must skip the write on `None`. This is the correct behavior — a write with an unknown partition key should not silently succeed.
- **Backward compatibility for tools/memory_search CLI**: The CLI takes an explicit `--project <key>` flag. This passes through as the `project_key` arg and bypasses all resolution. No behavior change.
- **Thin wrappers vs. direct import**: `memory_bridge._get_project_key()` becomes a thin wrapper calling `resolve_project_key(cwd=cwd)`. Existing mock targets in tests continue to work without mass test rewrites.
- **`memory_extraction.py` and `memory_hook.py`**: Both have `cwd` available (passed from the SDK hook system or the caller). Thread `cwd` through if not already present.

## Failure Path Test Strategy

### Exception Handling Coverage

- `resolve_project_key` wraps the projects.json file read in try/except (same pattern as current `_get_project_key`). Test must assert that a corrupt/missing projects.json falls through gracefully to the cwd basename branch (not an exception).
- `Memory.safe_save` `_warn_if_legacy_namespace` is wrapped in try/except — this is existing behavior from #1234; the test in `test_memory_guard.py` must confirm it never crashes the writer.

### Empty/Invalid Input Handling

- `resolve_project_key(cwd=None, env={})` → `None` (no fallback to `"default"`).
- `resolve_project_key(cwd="", env={})` → `None`.
- `resolve_project_key(cwd="/tmp/unknown", env={})` → `"unknown"` (cwd basename).
- `resolve_project_key(project_key="")` → treat empty string as absent, continue resolution chain.
- Writers receiving `None` must log a `WARNING` (not `DEBUG`) and skip — not raise. Use: `logger.warning("[memory] write skipped: resolve_project_key returned None (cwd=%r, env_key=%r)", cwd, os.environ.get("VALOR_PROJECT_KEY"))`. The WARNING level is necessary because a silent None-skip is harder to detect in production than the original `"dm"` leak (which at least produced visible records). The existing `_warn_if_legacy_namespace` guard detects `"dm"`/`"default"` namespace leaks but NOT None-skips — the WARNING covers this gap.

### Error State Rendering

- No user-visible output — Memory writes are silent. Error path is: `None` returned → debug log → write skipped. No user notification.

## Rabbit Holes

- **Making the resolver async**: The projects.json read is synchronous. Callers that are async (e.g., `memory_extraction.py`) should call it synchronously — file I/O is cheap and the file is small. Do NOT convert to async.
- **Caching the projects.json parse**: The file is read at most a few times per session. Do not introduce a cache — it adds invalidation complexity for zero observable benefit.
- **Returning a richer object** (e.g., a dataclass with `project_key`, `resolved_via`, `confidence`): The extra metadata is unused by callers. `str | None` is sufficient.
- **Unifying Memory READ callsites**: Recall/retrieval also uses `project_key`. This issue is scoped to WRITE paths only. Read-path unification is a separate concern.
- **Moving projects.json parsing to a shared utility**: `memory_bridge.py`, `scope_resolver.py`, and the new helper all parse projects.json. A shared loader is appealing but out of scope — that's a separate refactor.

## Risks

### Risk 1: `None` return breaks a callsite that didn't previously handle it

**Impact:** A writer that previously fell through to `DEFAULT_PROJECT_KEY="default"` now receives `None` and must skip the write. If the skip logic is missing, the writer will raise `TypeError` or pass `None` as `project_key` to Redis.
**Mitigation:** Migrate each callsite in a single PR. Add defensive guards at each write site: `if project_key is None: logger.debug(...); return`. Test each callsite explicitly for the `None` path.

### Risk 2: Sequencing with PR #1234

**Impact:** If this plan ships before #1234 merges, the bridge still has `dm` fallbacks, and the `_warn_if_legacy_namespace` guard doesn't exist in `models/memory.py`. The regression detector is absent.
**Mitigation:** Track #1234 as a prerequisite. If starting before #1234 merges, work from the `session/sdlc-1173` branch. The final PR for #1232 must be rebased onto main after #1234 merges.

### Risk 3: Circular import via `config/project_key_resolver.py`

**Impact:** If `project_key_resolver.py` imports from `models/` or `agent/`, it creates a cycle because `agent/memory_hook.py` imports from both.
**Mitigation:** The new module imports only `pathlib`, `os`, `json`, and `config/memory_defaults.py`. No model or agent imports. This is enforced by a lint rule (or a comment in the module header) that the file has no non-stdlib/non-config imports.

No race conditions identified. The resolver is a pure function reading an immutable file (projects.json) and environment variables. No shared mutable state. All operations are synchronous.
