# MERGE-stage dev session silently crashes on stale slug worktree (issue #1377)

## Problem

A dev session created for the MERGE stage of an SDLC pipeline silently dies before producing any output when the PM reuses the same slug as an earlier BUILD/RESEARCH session.

**Current behavior:**
1. A BUILD dev session creates `.worktrees/{slug}/` checked out to branch `session/{slug}`.
2. The PM later dispatches a MERGE session with the same slug via `valor-session create --role dev --slug {slug}`.
3. `valor-session` calls `get_or_create_worktree(repo_root, slug)` which returns the existing worktree path — still checked out to `session/{slug}`.
4. The session executor reads `session.working_dir = .worktrees/{slug}/` and calls `resolve_branch_for_stage(slug, stage)`. When `current_stage` is not populated on the AgentSession, `stage=None` and the function returns `("main", False)` — `needs_wt=False` so worktree provisioning is skipped.
5. Claude Code starts inside the worktree (still on `session/{slug}`) while the executor logs `branch=main`. Branch context mismatch produces zero agent output. The SDK heartbeat reports `communicated=False` for 6+ minutes until startup-recovery kills the session.

**Desired outcome:**
- A reused worktree whose HEAD branch does not match the expected branch is detected before the Claude Code subprocess is launched. The mismatch is either corrected (checkout to the expected branch) or raised with a clear error so the executor can fall back to a clean checkout.
- MERGE-stage dev sessions never silently crash because of branch/worktree mismatch.
- No regression to existing worktree isolation guarantees (`#887` main-checkout protection still enforced).

**Root cause pattern:** Guards in `worktree_manager.py` check existence and busy-ness, but never check **branch correctness**. The slug → branch invariant (`worktree at .worktrees/{slug}/ is checked out to session/{slug}`) is assumed but never enforced after the worktree is first created.

## Data Flow

1. **Entry point:** PM session invokes `valor-session create --role dev --slug {slug} --message "..."` for MERGE stage.
2. **`tools/valor_session.py:433-441`:** Resolves `repo_root` from project config, calls `get_or_create_worktree(repo_root, slug)`, sets `working_dir = wt_path`. This step reuses the existing `.worktrees/{slug}/` directory (BUILD left it there on `session/{slug}`).
3. **`agent_session_queue._push_agent_session`:** Enqueues the AgentSession with `working_dir=.worktrees/{slug}/`, `slug={slug}`, `session_type=dev`.
4. **`session_executor.py:750`:** Worker picks up the session, sets `working_dir = Path(session.working_dir)`.
5. **`session_executor.py:793-808`:** Reads `slug = session.slug`, looks up `stage` from `get_authoritative_session(...).current_stage`. **If MERGE stage was not recorded on the AgentSession**, `stage=None`.
6. **`session_executor.py:808`:** `resolve_branch_for_stage(slug, None)` returns `("main", False)` — branch resolution thinks we want `main` with no new worktree.
7. **`session_executor.py:822`:** Worktree provisioning is skipped because `needs_wt=False`. `working_dir` stays at `.worktrees/{slug}/`.
8. **`session_executor.py:862-879`:** Main-checkout protection guard checks `WORKTREES_DIR in str(working_dir)` — passes (path IS under `.worktrees/`).
9. **`session_executor.py:898-901`:** Logs `branch=main, cwd=.worktrees/{slug}/`.
10. **Claude Code launch:** Subprocess starts with `cwd=.worktrees/{slug}/`. Git inside that directory reports HEAD=`session/{slug}`. Branch/context mismatch between what the harness/system prompt was told (`branch=main`) and reality (`session/{slug}`) causes the agent to produce no output. `communicated=False` for 6+ min, killed by startup-recovery.

**Critical hand-off:** The boundary where the bug occurs is **step 6 → step 7**: `resolve_branch_for_stage` returns a branch name with `needs_wt=False`, and the executor trusts the caller-provided `working_dir` without verifying the worktree's HEAD matches the resolved branch.

## Solution

### Key Elements

- **Branch-mismatch guard in `worktree_manager.py`:** A new helper `verify_worktree_branch(worktree_path, expected_branch)` (or equivalent) that runs `git -C <path> rev-parse --abbrev-ref HEAD` and compares against `expected_branch`. On mismatch: either checkout to `expected_branch` if the worktree is clean, or raise a clear error if it has uncommitted state.
- **Invocation site in `session_executor.py`:** After `resolve_branch_for_stage` and any worktree provisioning, but BEFORE launching the Claude Code subprocess, call the verification helper. The exact placement is just after the main-checkout protection guard (line ~879).
- **MERGE-stage handling:** When the resolved branch is `main` (because `current_stage` was None or MERGE) AND the working_dir is an existing `.worktrees/{slug}/` checkout on `session/{slug}`, the executor must either (a) checkout `main` inside the worktree, or (b) redirect cwd to the repo root on `main`. Decision below.

### Flow

PM dispatches MERGE dev session → valor-session creates AgentSession reusing `.worktrees/{slug}/` → worker picks up session → executor resolves branch → **NEW: verify HEAD matches expected branch** → mismatch detected → **fail fast with clear error** OR **auto-recover by checking out expected branch** → harness launches with correct context.

### Technical Approach

Two complementary fixes, **both required**:

1. **Add `verify_worktree_branch(worktree_path: Path, expected_branch: str) -> None` to `agent/worktree_manager.py`.** This is the missing primitive.
   - Run `git -C <worktree_path> rev-parse --abbrev-ref HEAD` to read current branch.
   - If it matches `expected_branch`: return silently.
   - If it differs: check `git -C <worktree_path> status --porcelain`. If clean, run `git -C <worktree_path> checkout <expected_branch>` and log at INFO. If dirty, raise `WorktreeBranchMismatchError` with both branch names and the dirty file list.
   - Rationale for auto-checkout-on-clean: a worktree freshly handed off between stages should be clean, and silent recovery is preferable to a crash when the recovery is safe and obvious. Raising on dirty state preserves uncommitted work for human inspection.

2. **Invoke `verify_worktree_branch` in `agent/session_executor.py` after line 879** (after the main-checkout protection guard, before the Claude Code launch). This catches every dev session, not just MERGE.
   - For dev sessions with a slug: assert the worktree HEAD matches `branch_name` (the resolved branch from `resolve_branch_for_stage`).
   - For MERGE specifically, this means: if `branch_name=="main"` and the worktree is on `session/{slug}`, the guard fires and either auto-checks-out main (clean case) or raises (dirty case).

3. **Optional follow-up: fix the upstream signal.** The deeper root cause is that `current_stage` is sometimes None on the AgentSession when the executor runs `resolve_branch_for_stage`. That belongs to a separate issue (do NOT fix here) — the branch-mismatch guard is a defense-in-depth check that catches this AND any future regression where stage inference fails. Filed as a [SEPARATE-SLUG] entry below.

**Why not Option B from the issue (MERGE sessions don't inherit the slug worktree):** Option B requires changing `valor-session create` to not call `get_or_create_worktree` for MERGE stage, AND threading stage awareness down through the CLI. That's larger surface area than the guard, and it loses the ability to MERGE from a worktree if a future workflow needs it. The guard is the smaller, more conservative fix.

## Rabbit Holes

- **Refactoring `resolve_branch_for_stage` to eagerly require a non-None stage.** Tempting but out of scope — it would require auditing every dispatch site and changing the contract. The guard is sufficient defense-in-depth.
- **Adding stage-aware logic inside `valor-session create`.** Same trap — pulls stage awareness into the CLI, which the architecture deliberately keeps stage-agnostic.
- **Reworking worktree provisioning for all stages to redirect MERGE to repo root.** Option B from the issue. Larger surface area; reject in favor of the guard.
- **Adding a watchdog for `communicated=False` sessions.** Already exists (startup-recovery kills them after 6+ min). The bug isn't that the silent crash isn't detected — it's that it happens in the first place.

## Risks

### Risk 1: Auto-checkout silently changes worktree state
**Impact:** A human or another process expecting the worktree on `session/{slug}` finds it on `main`. Could cause confusion or, worst case, a follow-on git operation lands on the wrong branch.
**Mitigation:** Only auto-checkout when the worktree is clean (no uncommitted changes). Log at INFO with both branch names so post-mortems can trace the transition. Add a structured event so the dashboard can surface auto-checkout occurrences for review.

### Risk 2: Guard introduces a new failure mode where the session refuses to start
**Impact:** A dev session that previously silently crashed now loudly fails with `WorktreeBranchMismatchError`. From the user's perspective this is BETTER, but it changes the failure visibility.
**Mitigation:** Loud failure with a clear error message IS the desired behavior — silent crashes are the bug. Document this in the worktree feature doc so operators recognize the new error mode. Verify the error propagates to AgentSession.last_error and surfaces in the dashboard.

### Risk 3: `git rev-parse --abbrev-ref HEAD` fails on a detached HEAD or corrupted worktree
**Impact:** Guard raises a confusing git error instead of a clear branch-mismatch error.
**Mitigation:** Wrap the subprocess call, catch CalledProcessError, and translate to a `WorktreeBranchMismatchError` with a descriptive message ("could not read HEAD of worktree at {path}: {git stderr}").

## Race Conditions

No race conditions identified — `verify_worktree_branch` runs synchronously in the executor before the harness launches. Worktrees are single-writer at the file-system level (a separate concurrent session creating the same worktree is already gated by the refuse-busy guard from PR #1367). The check-then-act pattern (verify then checkout) is safe because the executor holds the worktree until the session terminates.
