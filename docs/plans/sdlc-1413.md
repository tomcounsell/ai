# Tool-Failure Circuit Breaker (sdlc-1413)

## Scope Boundary — Two-Part Fix

On 2026-06-03, 24 auto-filed duplicate "thrashing" issues were consolidated into #1413 (see issue comment `4610199770`), splitting the root-cause work into two complementary parts that live in **disjoint files**:

- **Part 1 — Tool-failure circuit breaker** *(this plan)*: a mid-session, deterministic interrupt in `agent/health_check.py` that flags a runaway session before it burns dozens of failed calls. Scoped and detailed below.
- **Part 2 — Stop the auto-filer flood** *(delegated to `docs/plans/sdlc-1414-thrash-detector-and-confidence-gate-fixes.md`, already `Ready`)*: fixes the reflections detector that *generated* the 25 duplicate issues — the false-positive thrash ratio (`reflections/session_intelligence.py:65-77`) and the leaky high-confidence gate (`reflections/utils.py:411-425`).

These two plans do not share any source files (`agent/` vs. `reflections/`), so they build independently with no merge conflict. Part 1 is built first per operator direction; Part 2 proceeds via its own plan. This plan is **not** expanded to cover the reflections changes — doing so would duplicate the existing sdlc-1414 plan.

## Problem

Reflections filed issue #1413 after observing a session with 73 tool calls and only 2 successful turns (97% failure ratio). The existing post-hoc thrash detector in `reflections/session_intelligence.py:65-77` (`THRASH_RATIO_THRESHOLD = 0.5`) only fires *after* the session ends — it files a GitHub issue but does not interrupt the runaway session. The existing Haiku-based watchdog (`agent/health_check.py`) runs every 20 tool calls but judges holistic "is the agent making progress" rather than counting consecutive failures, so a session can burn dozens of failed tool calls between checks.

There is no mid-session, deterministic interrupt that says: "you have failed N tool calls in a row — stop and reassess before the next one."

## Desired Outcome

A check inside the existing `watchdog_hook` tracks consecutive tool-call failures per session. When the counter reaches a configurable threshold (default: 5), the hook flags the session via the existing `AgentSession.watchdog_unhealthy` mechanism with a reason naming the most recent failing tools, e.g. `"5 consecutive tool failures (Bash, Bash, Edit, Read, Bash) — strategy reassessment required"`. The output router (`agent/session_executor.py:1060` → `agent/output_router.py:141`) already reads `watchdog_unhealthy` and stops auto-continuing; that surfaces the failure to the operator (or to the PM session) for strategy reassessment.

Any successful tool call resets the counter and clears the recent-failure ring buffer.

**Note on the shared signal field:** the Haiku watchdog and this breaker both write the single `AgentSession.watchdog_unhealthy` field (last-writer-wins). The field *latches* on first fire — `clear_unhealthy` is not called in production — so a re-fire only refreshes the reason text; it does not generate an additional pause. The re-fire policy (below) therefore matters for *reason freshness / dashboards*, not for producing repeated nudge-loop pauses.

## Definitions

| Term | Definition |
|------|-----------|
| Failed tool call | A PostToolUse event whose `tool_response` is a dict with `is_error == True`, OR `tool_response` is a string starting with `"Error: "` (defensive: rare SDK paths). All other shapes (`None`, lists, scalars, dicts without `is_error`) are treated as success. Bash exit codes are NOT inspected — the SDK marks failed Bash via `is_error`. |
| Consecutive failure counter | Per-session in-memory counter, keyed by the same `session_id` already resolved by `watchdog_hook` (`AGENT_SESSION_ID` env, falling back to Claude Code's `session_id`), incremented on each failed tool call and reset to 0 on each successful tool call. |
| Recent failure ring | `collections.deque(maxlen=5)` per session holding the tool names of the last failures (for inclusion in the unhealthy reason string). Cleared on success. |

| Strategy reassessment | `agent/session_executor.py:1060` reads `watchdog_unhealthy` and routes via `agent/output_router.py:141`, stopping auto-continue — the session's next output is delivered to Telegram for human/PM review instead of being silently continued. No new mechanism required. |

## Solution

The check lives **inside `watchdog_hook` (agent/health_check.py:519)**, immediately after the `_tool_counts[session_id] += 1` line at agent/health_check.py:541-542. Wiring inside `watchdog_hook` rather than a separate function at the `post_tool_use_hook` layer ensures:

- Single source of truth for `session_id` resolution (we reuse the one `watchdog_hook` already computed).
- The check runs alongside the existing counter — no new env reads, no duplicated session lookup.
- `post_tool_use.py` stays a thin delegator; all health logic lives in `health_check.py`.

Steps performed inline in `watchdog_hook`, post-`_tool_counts` increment:

1. Read `tool_response` from `input_data` defensively (`input_data.get("tool_response")`).
2. Classify:
   - failure if `isinstance(tool_response, dict) and tool_response.get("is_error") is True`
   - failure if `isinstance(tool_response, str) and tool_response.startswith("Error: ")`
   - success otherwise
3. On success: `_consecutive_failures[session_id] = 0`; `_recent_failures[session_id].clear()` (if present).
4. On failure: increment `_consecutive_failures[session_id]`; append `tool_name` to `_recent_failures[session_id]` (auto-created as `deque(maxlen=5)`). If counter reaches `CONSECUTIVE_FAILURE_THRESHOLD`, call `_set_unhealthy(session_id, reason)` where `reason = f"{count} consecutive tool failures ({', '.join(recent_tools)}) — strategy reassessment required"`, then reset counter and clear ring (so it re-fires every N failures, not just once).
5. The entire block is wrapped in `try/except` that logs at debug (never blocks the hook).

No changes to `AgentSession` schema. No changes to nudge loop. No changes to `post_tool_use.py`, bridge, or worker. The existing `watchdog_unhealthy` signal already does what's needed downstream.

## Race Condition Analysis

- `_consecutive_failures` and `_recent_failures` are module-level dicts. Hooks run in the worker process, single-threaded per session — no concurrent access within a session. Cross-session writes are independent keys, no contention.
- Both dicts are process-local and reset on worker restart. That is acceptable: a restart clears all in-flight session state anyway; the counter is a heuristic, not durable state.
- `_set_unhealthy` writes to Redis via Popoto. The nudge loop reads asynchronously. If the unhealthy flag is set between two nudge cycles, the operator sees it on the next cycle — no race that loses signal.

## Failure Path Test Strategy

- **Threshold reached:** unit test asserts `_set_unhealthy` is called with a reason matching `r"5 consecutive tool failures \(.*\) — strategy reassessment required"` after exactly 5 failures, and that the counter resets so a 6th failure does *not* re-trigger immediately.
- **Tool names in reason:** unit test feeds 5 failures with tool names `["Bash", "Bash", "Edit", "Read", "Bash"]` and asserts the reason contains exactly those names in order.
- **Ring buffer maxlen:** unit test feeds 7 failures with distinct tool names and asserts the reason contains only the last 5.
- **Counter reset on success:** unit test asserts 4 failures + 1 success + 4 failures does NOT trigger; ring is also empty after the success.
- **Malformed `tool_response`:** unit test passes `None`, `{}`, `[]`, `123`, and a dict without `is_error`; hook must not raise and must not increment the failure counter.
- **String error path:** unit test passes `tool_response="Error: file not found"` — counts as failure.
- **Missing session_id:** unit test ensures the hook gracefully no-ops if neither `AGENT_SESSION_ID` env nor `input_data["session_id"]` is resolvable.
- **Distinguishable from the Haiku watchdog:** the existing Haiku watchdog and the circuit breaker write the same single, latching `watchdog_unhealthy` field (last-writer-wins; `clear_unhealthy` is not called in production). The breaker's reason string ("N consecutive tool failures …") is distinct so dashboards can attribute which writer set the flag. The test asserts the reason format, not that two reasons coexist.

## No-Gos

- Do NOT add a new field to `AgentSession`. Reuse `watchdog_unhealthy`.
- Do NOT modify the bridge, worker, or nudge loop. The signal channel already exists.
- Do NOT replace or alter the Haiku-based watchdog. The two run independently.
- Do NOT make this a hard PreToolUse block. Blocking the next tool call denies the agent the chance to recover via its own reasoning; the soft signal (unhealthy flag → nudge loop pauses) is sufficient.
- Do NOT persist the counter or ring to Redis. In-memory per-process is sufficient and matches the existing `_tool_counts` pattern.
- Do NOT count Skill or ToolSearch failures separately — a tool-result error is a tool-result error regardless of which tool emitted it.
- Do NOT add the new check to `post_tool_use.py`. Wire it inside `watchdog_hook` post-`_tool_counts` increment for a single source of truth.

## Rabbit Holes

- **Defining "failed" beyond `is_error`:** Tempting to inspect Bash exit codes, Edit "old_string not found", Read missing files. The SDK already surfaces these via `is_error`. Going deeper means duplicating tool-specific knowledge in the hook. Stop at `is_error`.
- **Per-tool thresholds:** Tempting to say "5 Read failures vs. 5 Bash failures are different." Single global threshold is simpler. Defer per-tool tuning until data shows it's needed.
- **Strategy injection via steering message:** Tempting to inject a "Stop and reassess your approach" steering message via `queued_steering_messages`. Out of scope.
- **Persisting consecutive-failure history per session:** Tempting to add a `consecutive_failure_streak` field to `AgentSession` for dashboards. Defer.

## Resolved Decisions

1. **Re-fire policy:** The breaker re-fires every 5 *additional* consecutive failures (counter resets to 0 after firing). This ensures the operator is alerted to repeated runaways rather than only the first, at the cost of slightly noisier dashboards. Acceptable trade — `watchdog_unhealthy` is the channel of record, and the nudge loop pauses on each set regardless.
2. **Tool names in reason:** Include failing tool names via `deque(maxlen=5)` — aids operator triage without bloating Redis.
