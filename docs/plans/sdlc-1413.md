# Tool-Failure Circuit Breaker (sdlc-1413)

## Problem

Reflections filed issue #1413 after observing a session with 73 tool calls and only 2 successful turns (97% failure ratio). The existing post-hoc thrash detector in `reflections/session_intelligence.py:65-77` (`THRASH_RATIO_THRESHOLD = 0.5`) only fires *after* the session ends — it files a GitHub issue but does not interrupt the runaway session. The existing Haiku-based watchdog (`agent/health_check.py`) runs every 20 tool calls but judges holistic "is the agent making progress" rather than counting consecutive failures, so a session can burn dozens of failed tool calls between checks.

There is no mid-session, deterministic interrupt that says: "you have failed N tool calls in a row — stop and reassess before the next one."

## Desired Outcome

A PostToolUse hook tracks consecutive tool-call failures per session. When the counter reaches a configurable threshold (default: 5), the hook flags the session via the existing `AgentSession.watchdog_unhealthy` mechanism with a reason like `"5 consecutive tool failures — strategy reassessment required"`. The nudge loop in `agent_session_queue.py` already reads `watchdog_unhealthy` and stops auto-continuing; that surfaces the failure to the operator (or to the PM session) for strategy reassessment.

Any successful tool call resets the counter to 0.

## Definitions

| Term | Definition |
|------|-----------|
| Failed tool call | A PostToolUse event whose `tool_response` is a dict with `is_error == True`, OR `tool_response` is a string containing the standard SDK error prefix (`"Error: "` at position 0). Bash exit codes are NOT inspected directly — the SDK marks failed Bash via `is_error`. |
| Consecutive failure counter | Per-session in-memory counter, keyed by `VALOR_SESSION_ID` (falling back to Claude Code's `session_id`), incremented on each failed tool call and reset to 0 on each successful tool call. |
| Circuit breaker threshold | The number of consecutive failures that triggers the unhealthy flag. Configurable via constant `CONSECUTIVE_FAILURE_THRESHOLD` in `agent/health_check.py`, default 5. |
| Strategy reassessment | The nudge loop in `agent_session_queue.py` detects `watchdog_unhealthy` and stops auto-continuing — the session's next output is delivered to Telegram for human/PM review instead of being silently continued. No new mechanism required. |

## Solution

A new module-level function `circuit_breaker_check(input_data)` in `agent/health_check.py` that:

1. Reads `tool_response` from the PostToolUse input.
2. Determines failure: `isinstance(tool_response, dict) and tool_response.get("is_error") is True`. Strings starting with `"Error: "` are also treated as failure (defensive: some SDK paths return strings).
3. Resolves `session_id` from `VALOR_SESSION_ID` env var (falls back to `input_data["session_id"]`, matching existing `_tool_counts` keying convention at `agent/health_check.py:34`).
4. Maintains a module-level `_consecutive_failures: dict[str, int] = {}` counter.
5. On success: `_consecutive_failures[session_id] = 0`.
6. On failure: increment. If counter reaches `CONSECUTIVE_FAILURE_THRESHOLD` (default 5), call `_set_unhealthy(session_id, "N consecutive tool failures — strategy reassessment required")` and reset the counter to 0 (so it re-fires every N failures, not just once).

Wire-up: `agent/hooks/post_tool_use.py::post_tool_use_hook` calls `circuit_breaker_check(input_data)` after the liveness write and before the existing `watchdog_hook` call. Failures inside `circuit_breaker_check` are caught and logged at debug (never block the hook).

No changes to `AgentSession` schema. No changes to nudge loop. No changes to bridge or worker. The existing `watchdog_unhealthy` signal already does what's needed downstream.

## Data Flow

```
Tool call completes
  -> PostToolUseHookInput delivered to post_tool_use_hook
    -> liveness write (existing)
    -> circuit_breaker_check(input_data)        [NEW]
        -> inspect tool_response for is_error
        -> update _consecutive_failures[session_id]
        -> if threshold reached: _set_unhealthy(session_id, reason)
    -> watchdog_hook (existing, fires every 20 calls)
    -> Skill stage completion (existing)

Next nudge cycle in agent_session_queue.py:
  -> reads AgentSession.watchdog_unhealthy
  -> if non-null: stop auto-continue, surface output to operator/PM
```

## Race Condition Analysis

- `_consecutive_failures` is a module-level dict. Hooks run in the worker process, single-threaded per session — no concurrent access within a session. Cross-session writes are independent keys, no contention.
- The dict is process-local and resets on worker restart. That is acceptable: a restart clears all in-flight session state anyway; the counter is a heuristic, not durable state.
- `_set_unhealthy` writes to Redis via Popoto. The nudge loop reads asynchronously. If the unhealthy flag is set between two nudge cycles, the operator sees it on the next cycle — no race that loses signal.

## Failure Path Test Strategy

- **Threshold reached:** unit test asserts `_set_unhealthy` is called with the expected reason after exactly 5 failures, and that the counter resets so a 6th failure does *not* re-trigger immediately.
- **Counter reset on success:** unit test asserts that 4 failures + 1 success + 4 failures does NOT trigger the breaker (counter reset clears state).
- **Malformed `tool_response`:** unit test passes `None`, an empty dict, and a non-dict non-string value; hook must not raise.
- **Missing session_id:** unit test ensures the hook gracefully no-ops if neither `VALOR_SESSION_ID` env nor `input_data["session_id"]` is resolvable.
- **No false positives on the Haiku watchdog:** integration check — the existing watchdog and the circuit breaker write to the same `watchdog_unhealthy` field; the breaker's reason string is distinct so dashboards can distinguish them.

## No-Gos

- Do NOT add a new field to `AgentSession`. Reuse `watchdog_unhealthy`.
- Do NOT modify the bridge, worker, or nudge loop. The signal channel already exists.
- Do NOT replace or alter the Haiku-based watchdog. The two run independently.
- Do NOT make this a hard PreToolUse block. Blocking the next tool call denies the agent the chance to recover via its own reasoning; the soft signal (unhealthy flag → nudge loop pauses) is sufficient.
- Do NOT persist the counter to Redis. In-memory per-process is sufficient and matches the existing `_tool_counts` pattern.
- Do NOT count Skill or ToolSearch failures separately — a tool-result error is a tool-result error regardless of which tool emitted it.

## Rabbit Holes

- **Defining "failed" beyond `is_error`:** Tempting to inspect Bash exit codes, Edit "old_string not found", Read missing files. The SDK already surfaces these via `is_error`. Going deeper means duplicating tool-specific knowledge in the hook. Stop at `is_error`.
- **Per-tool thresholds:** Tempting to say "5 Read failures vs. 5 Bash failures are different." Single global threshold is simpler and matches the issue's heuristic. Defer per-tool tuning until data shows it's needed.
- **Strategy injection via steering message:** Tempting to inject a "Stop and reassess your approach" steering message via `queued_steering_messages`. Out of scope — the unhealthy flag + operator visibility is the contract for this issue. Steering-message injection is a separate, larger design.
- **Persisting consecutive-failure history per session:** Tempting to add a `consecutive_failure_streak` field to `AgentSession` for dashboards. Defer — the existing post-hoc thrash detector already records the final ratio; adding a streak field is a separate dashboards concern.

## Open Questions

1. Should the breaker re-fire every 5 *additional* consecutive failures (current plan: yes, reset to 0 after firing) or only once per session (cleaner signal, but the operator might miss a second runaway)?
2. Should the unhealthy reason string include the most recent failed tool name(s) for operator triage, or is the bare count enough? (Current plan: bare count + generic message; the activity stream already has the tool history.)
