# Worker Watchdog: Active Recovery via launchctl kickstart

## Problem

The worker process died at `2026-05-06 08:37:23 UTC` and stayed dead for 7+ hours. Three layers of supposed protection — launchd `KeepAlive=true`, the 120s `worker-watchdog` health loop, and reasonable expectation that "the system restarts itself" — every layer logged the failure but none recovered. `monitoring/worker_watchdog.py:147` detects the dead worker on every tick and logs `Worker not running — launchd handles restart` without taking any action.

**Current behavior:**
- `worker_watchdog.py:main()` returns immediately on `status == "down"` after a single log line. No `launchctl` invocation. No verification.
- `logs/worker_watchdog.log` shows every line duplicated at the same microsecond — logger misconfiguration: `logging.basicConfig()` at line 33 attaches a `StreamHandler` to root, then line 38 attaches a second handler to the named logger (which propagates to root by default). Any root-bound `StandardOutPath`/`StandardErrorPath` capture also doubles up.
- The existing `recover()` function handles only the *stale-heartbeat* case (kills the worker so launchd restarts it). The *worker-missing-entirely* branch has no parallel recovery.

**Desired outcome:**
- When the worker is missing for >2 consecutive ticks, the watchdog actively kicks launchd via `launchctl kickstart -k gui/<uid>/com.valor.worker`.
- Verifies a live PID returns within 10s; escalates to `launchctl enable` + retry on failure (handles sticky-disable from `worker-disable`).
- Critical-level escalation writes a Redis key (`worker:watchdog:critical:{hostname}`) so a downstream feature can surface the failure.
- Each watchdog log line appears exactly once.

## Solution

### Key Elements

- **`_kickstart_worker()`**: Invokes `launchctl kickstart -k gui/<uid>/com.valor.worker` via subprocess; returns success/failure.
- **`_verify_worker_alive(grace_seconds=10)`**: Polls `_get_worker_pid()` until non-None or grace expires.
- **`_enable_worker()`**: Invokes `launchctl enable gui/<uid>/com.valor.worker` for the sticky-disable case.
- **`_record_critical_status(reason)`**: Writes `worker:watchdog:critical:{hostname}` Redis key (TTL 1h) via the existing Popoto/Redis client used elsewhere in `monitoring/`. Best-effort — never raises.
- **Down-state counter**: A small file (`data/worker_watchdog_down_ticks`) holding an integer; incremented on each `down` tick, deleted when worker is healthy. Survives watchdog process restarts (each tick is a fresh launchd invocation).

### Flow

`down` tick observed → increment `data/worker_watchdog_down_ticks` → branch on level:

- **L1 (count == 1)**: Log `Worker missing — giving launchd one tick to restart` and return.
- **L2 (count >= 2)**: Log warning. Run `_kickstart_worker()`, then `_verify_worker_alive(10)`. If PID appears, log success and reset counter.
- **L3 (L2 verify failed)**: Log warning. Run `_enable_worker()` then `_kickstart_worker()` then `_verify_worker_alive(10)`. If PID appears, log success and reset counter.
- **L4 (L3 verify failed, count >= 3)**: Log CRITICAL. Call `_record_critical_status("kickstart+enable both failed after N ticks")`. Counter persists; subsequent ticks repeat L4 (idempotent — Redis SET overwrites with fresh TTL).

Healthy tick → if `data/worker_watchdog_down_ticks` exists, delete it.

### Technical Approach

- Branch in `main()` at the existing `if status["status"] == "down":` site. Replace the single log line with the level-based handler. Keep `main()` shape: still a single tick function, no class abstraction, no state machine.
- Service label is read once: `WORKER_LAUNCHD_LABEL = "com.valor.worker"` (matches `scripts/install_worker.sh` `SERVICE_LABEL_PREFIX:=com.valor`). UID resolved at call time via `os.getuid()`. Honor `SERVICE_LABEL_PREFIX` env var for parity with installer.
- All `launchctl` shell-outs use `subprocess.run(..., timeout=10, capture_output=True, text=True)`. Log returncode and stderr on failure.
- **Logger fix**: remove the `logging.basicConfig(...)` call at line 33. Configure the named logger explicitly: set level, attach the rotating file handler, set `logger.propagate = False`. Add a `StreamHandler(sys.stdout)` to the same named logger so launchd's `StandardOutPath` still captures one copy. Net effect: each line written once to the file and once to stdout (which launchd routes to the same file via the plist) — but the dedup is at the logger level, so the file gets exactly one line.
  - Verify the plist redirects `StandardOutPath` and `StandardErrorPath` to the *same* file (`logs/worker_watchdog.log`). If they are the same path AND the named logger writes via its own handler, propagation to root prints to stdout/stderr, which launchd appends to the same file → duplication. Fix is `logger.propagate = False` + remove `basicConfig` + drop the `StreamHandler` (let launchd capture stdout/stderr from the underlying Python interpreter only).
- Redis key write uses the same client pattern as `monitoring/crash_tracker.py` (existing reference). Wrap in `try/except Exception: pass` — watchdog must never crash on Redis unavailability.

## Risks

### Risk 1: kickstart kills a worker mid-task
**Impact:** A worker processing an AgentSession gets SIGKILLed (kickstart `-k`), losing in-flight work.
**Mitigation:** L2 fires only when `_get_worker_pid()` returns None for 2 consecutive ticks (≥240s). At that point the worker is already dead — kickstart cannot kill what is not running. The existing `recover()` path (stale-heartbeat) is untouched.

### Risk 2: Sticky-disable persists past L3
**Impact:** If the user ran `worker-disable` deliberately, L3's `launchctl enable` will overrule the operator's intent.
**Mitigation:** Honor the existing `data/worker-disabled` flag (used by `worker-disable`/`worker-enable` commands per CLAUDE.md). Watchdog short-circuits at the very top of `main()` if the flag exists — log `Worker disabled by operator — skipping check` and return. (Verify the exact filename during build by reading `scripts/valor-service.sh`.)

### Risk 3: Redis client unavailable
**Impact:** Critical key never written; downstream surface never alerts.
**Mitigation:** Best-effort write. CRITICAL log line is the primary signal; Redis is the secondary surface. Test asserts watchdog continues on Redis failure.

### Risk 4: Logger fix introduces silent log loss
**Impact:** Removing `basicConfig` could mute stderr output that ops scripts grep for.
**Mitigation:** Verify `logs/worker_watchdog.log` content before/after via the integration test. The plist's `StandardErrorPath` still captures any uncaught exceptions printed by the Python interpreter.

## Race Conditions

### Race 1: Two watchdog ticks overlap during slow kickstart
**Location:** `monitoring/worker_watchdog.py:main()` — launchd fires every 120s; a kickstart + 10s verify takes ~12s, leaving ample headroom. But if a tick hangs (e.g. `pgrep` slow), launchd may start a second instance.
**Trigger:** First tick blocked >120s, second tick begins, both increment the down-counter file.
**Data prerequisite:** The down-tick counter file must reflect a single source of truth.
**State prerequisite:** Counter increments must be atomic.
**Mitigation:** Read-modify-write the counter under a `flock` on a sidecar lock file (`data/worker_watchdog.lock`). Skip the tick if lock is held — log and return. Same pattern is used in `bridge_watchdog`.
