# Default `--reply-to` on agent-invoked `valor-telegram send`

## Problem

When the agent invokes `valor-telegram send` from inside an active AgentSession (mid-session status updates, intermediate replies, self-narration), the message is queued without `reply_to`. The resulting Telegram message dangles from the topic root rather than threading off the user message that triggered the agent turn.

If a third party in the chat then replies directly to one of those dangling messages, the bridge cannot reliably reassemble the reply into the originating conversation thread. The conversation owner (the AgentSession) loses context and may treat the reply as a new conversation.

**Current behavior:**
- **Path A (`agent/output_handler.py:TelegramRelayOutputHandler.send`)** — the natural "final response" path — already threads correctly because the worker passes `session.telegram_message_id` as `reply_to_msg_id` (`agent/session_executor.py:903,1013,1050`).
- **Path B (`tools/valor_telegram.py:cmd_send`, line 764)** — the mid-session CLI path — sets `reply_to = getattr(args, "reply_to", None)` and writes `None` to the outbox payload when the agent does not pass `--reply-to` explicitly. Agents calling this from Bash mid-session typically don't pass it.

**Desired outcome:**
When the agent invokes `valor-telegram send` from inside a session, `cmd_send` defaults `reply_to` to the session's triggering user message (already exposed as the `TELEGRAM_REPLY_TO` env var). Mid-session sends thread off the same root as the agent's final response, producing a single visible reply chain per turn. Explicit `--reply-to` continues to override the default. Sends from outside an AgentSession context (ad-hoc operator use, no env var set) preserve current behavior.

## Data Flow

End-to-end trace of the *fixed* path:

1. **Entry point**: User sends a Telegram message → bridge resolves chat/project → enqueues an `AgentSession` with `initial_telegram_message["telegram_message_id"] = <triggering_msg_id>`.
2. **`agent/sdk_client.py:_extract_sdlc_env_vars`** (line 515): worker resolves the AgentSession, reads `session.telegram_message_id` (computed property at `models/agent_session.py:891`), sets `env["TELEGRAM_REPLY_TO"] = str(tg_msg_id)`.
3. **Subprocess launch** (`agent/sdk_client.py:_build_env`): the env dict is passed to `claude -p` so the agent's Bash environment carries `TELEGRAM_REPLY_TO`, `TELEGRAM_CHAT_ID`, `AGENT_SESSION_ID`, `VALOR_SESSION_ID`, etc.
4. **Agent invokes Bash**: `valor-telegram send --chat "Dev: Valor" "interim status"` (no `--reply-to`).
5. **`tools/valor_telegram.py:cmd_send`** (line 764, *post-fix*): `reply_to = getattr(args, "reply_to", None)`; if `None`, fall back to `os.environ.get("TELEGRAM_REPLY_TO")`. Build payload `{"chat_id", "reply_to", "text", "session_id", "timestamp"}` and `RPUSH telegram:outbox:cli-{ts}`.
6. **`bridge/telegram_relay.py`**: drains the outbox key, calls Telethon `client.send_message(chat_id, text, reply_to=reply_to, ...)`.
7. **Output**: Telegram message appears as a direct reply to the user's triggering message. Same visible thread as the agent's eventual final response (Path A also uses `session.telegram_message_id`).

## Solution

### Key Elements

- **`tools/valor_telegram.py:cmd_send` (env fallback)**: When `args.reply_to` is `None`, fall back to `os.environ.get("TELEGRAM_REPLY_TO")` (parsed as int). Explicit `--reply-to` always wins.
- **No new env var**: `TELEGRAM_REPLY_TO` already exists, already populated by `agent/sdk_client.py:585-590` from `session.telegram_message_id`. Reuse it.
- **No CLI surface change**: `--reply-to` argparse stays as-is. Help text gets a one-line addendum noting the env-var default for in-session use.
- **No call-site changes**: Agents continue to invoke `valor-telegram send` exactly as today. Default behavior just becomes correct.

### Flow

PM session running in worker → agent shells out: `valor-telegram send --chat "Dev: Valor" "interim status"` → `cmd_send` reads `args.reply_to` (None) → falls back to `TELEGRAM_REPLY_TO` env (`= triggering_msg_id`) → outbox payload has `reply_to = triggering_msg_id` → relay sends with `reply_to_msg_id` set → message threads off the user's triggering message in Telegram.

### Technical Approach

- **Approach A wins (fix in CLI).** Recon confirmed the env-var infrastructure already exists. Approach B (inject `--reply-to` at every wrapper) requires identifying every persona, hook, and skill that exposes the CLI to the agent — Approach A is one localized change.
- **Default = user's triggering message_id**, not "last agent message in this turn." Reasoning:
  - Mirrors Path A (`TelegramRelayOutputHandler.send` uses `session.telegram_message_id`). Consistency between paths means a single visible reply chain per turn.
  - "Last agent message" requires post-send writeback of the bot's outbound `message_id` into AgentSession state, plus a read on the next send. Heavy for a marginal UX gain.
  - Users see one tidy thread off their original message rather than a chain-of-replies that visually nests deeper with each interim send.
- **No forum-vs-basic-group branching.** Path A doesn't branch; Telethon validates `reply_to_msg_id` and degrades gracefully if the message doesn't exist in a basic group. Branching would add code with no observable benefit.
- **Parsing safety:** The env var is set via `str(tg_msg_id)` (`agent/sdk_client.py:590`). On read, attempt `int(env_val)` inside a try/except — invalid values silently fall through to `None` (preserves "no reply" behavior rather than crashing). Empty string `""` from a stripped env is also treated as `None`.
- **Help text:** Update the `--reply-to` argparse help string to one line: `"Message ID to reply to. Defaults to $TELEGRAM_REPLY_TO when set (in-session use)."`

## Rabbit Holes

- **Don't track "last agent message in this turn."** Tempting to fan interim sends into a chain off the previous bot message. Requires AgentSession state writeback after every send + read on next send. Gain (visually slightly deeper threading) does not justify the state-management cost.
- **Don't branch on chat type (forum vs basic group).** Path A doesn't, Telethon handles invalid `reply_to_msg_id` gracefully. Adding chat-type detection in the CLI introduces a Telegram API dependency that the CLI doesn't have today.
- **Don't introduce a `VALOR_AGENT_SESSION_ID` env var or have `cmd_send` look up an AgentSession.** The issue text suggested this; recon showed it's unnecessary. `TELEGRAM_REPLY_TO` is pre-resolved by `sdk_client.py` and already carries the message_id directly. Adding a CLI-side AgentSession lookup would couple the CLI to the Popoto ORM and Redis, making it heavier and harder to test.
- **Don't add a "thread debugging" mode or session-aware `--reply-to=auto` sentinel.** The env-var fallback IS the auto behavior. Adding an explicit sentinel would force agents to remember a magic flag, defeating the point.

## Risks

### Risk 1: Env var leaks into ad-hoc operator CLI use
**Impact:** If a human operator runs `valor-telegram send` from a shell that has `TELEGRAM_REPLY_TO` exported (e.g., a debug session, sourced env file), their messages will silently thread off whatever message_id is in the env. Unexpected.
**Mitigation:** Acceptable. `TELEGRAM_REPLY_TO` is only ever set by `agent/sdk_client.py` for spawned agent subprocesses; it does not appear in `~/Desktop/Valor/.env` or `.env.example`. A human shell that has it set has explicitly chosen to (and probably wants the threading). Document this in the help text of `--reply-to`.

### Risk 2: Stale env var from parent agent leaks into nested subprocess
**Impact:** If an agent shells out to a subprocess that re-invokes `valor-telegram send` after the parent's `telegram_message_id` is no longer current (e.g., a long-lived subprocess), the reply could thread off a stale message_id. Practically: the message would still thread off the *user's original triggering message*, which is exactly what we want even across subprocess depth.
**Mitigation:** Not a real risk in practice. `telegram_message_id` is the user's triggering message and remains the correct thread root for the entire turn, regardless of subprocess depth. Document the env-var contract in `docs/features/pm-dev-session-architecture.md` (under PM session env vars).

### Risk 3: Telethon rejects a reply_to_msg_id that no longer exists
**Impact:** If the triggering user message has been deleted between the agent receiving it and sending the reply, Telethon may raise `MessageReplyHeaderInvalidError` or silently send without threading.
**Mitigation:** Out of scope. Same risk affects Path A today (`output_handler.py` passes the same value). The relay handles Telethon errors via existing retry/dead-letter mechanisms; this change does not introduce a new error mode.

## Race Conditions

No race conditions identified. The fix is a synchronous, single-process read of an env var inside `cmd_send`, which writes once to the Redis outbox and exits. The env var is set once at subprocess launch (`agent/sdk_client.py:_build_env`) and is read-only thereafter.
