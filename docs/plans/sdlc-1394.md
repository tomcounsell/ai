# PM pipeline governance: watchdog resilience and plan-doc lifecycle (issue #1394)

## Problem

Two governance failures compound to make autonomous SDLC pipeline runs unreliable, both observed during the cuttlefish #350 full-pipeline run.

**Problem 1 — Watchdog kills the PM mid-pipeline.**
The stream watchdog (`monitoring/session_watchdog.py`, 600s threshold) kills the PM session when `/do-test` runs a long pytest suite or `/do-deploy` waits for a Render auto-deploy. The PM has no mechanism to emit liveness events during these blocking waits. Two separate PM sessions were needed to complete the cuttlefish #350 pipeline, and human intervention was required for final deploy verification.

**Problem 2 — Plan docs commit directly to `main` and are later deleted.**
`do-plan` is designed to commit plan documents directly to `main` (`.claude/skills-global/do-plan/SKILL.md` lines 332-340 and 462-464). These commits create rebase pressure on open PR branches — when critique revises a plan mid-build, the PR's mergeable state flips to CONFLICTING. At end-of-pipeline, the plan doc is deleted via `rm -f` in `do-merge.md:542` as part of pipeline cleanup, destroying the forensic record of what was planned vs. what shipped. The `docs/plans/completed/` migration directory already exists and is partially used; the deletion happens regardless of whether migration succeeds.

**Desired outcome:**
1. PM sessions survive long-running sub-stages without watchdog kills.
2. Plan documents follow a predictable lifecycle: created on a `session/{slug}` branch, never committed directly to `main`, archived (not deleted) on pipeline completion.

This issue consolidates #1303 (PM watchdog) and #1304 (plan-doc lifecycle). The existing plan at `docs/plans/sdlc-1304.md` covers Problem 2 in detail and its research, data flow, and solution analysis are carried forward here.

## Research

Both problems are purely internal. No external libraries or APIs required.

For Problem 1: the watchdog threshold is configurable in `monitoring/session_watchdog.py`. The simplest fix is a per-stage threshold override for PM sessions running `do-test` or `do-deploy`. A complementary approach is having long-running sub-skills emit periodic liveness tokens via stdout — the watchdog resets its timer on any output token. The PM's `current_stage` field is available on the `AgentSession` record and readable by the watchdog.

For Problem 2: the mechanical changes are well-understood from sdlc-1304.md. The skill markdown files are the source of truth; editing them is the integration. `migrate_completed_plan.py` already exists and works; the only missing piece is removing the unconditional `rm -f` and making migration success the gate.

## Data Flow

### Problem 1 — Watchdog resilience

**Today:**
1. PM dispatches `/do-test` or `/do-deploy`.
2. Dev session (or sub-skill) runs a blocking operation for 600+ seconds.
3. Watchdog fires: no output tokens for 600s → PM session killed.
4. Worker creates a new PM session; pipeline must recover state.

**Proposed:**
1. PM dispatches `/do-test` or `/do-deploy`.
2. Sub-skill emits a liveness token (`…still running (Xs elapsed)`) every 60s via stdout.
3. Watchdog sees output tokens; does not fire.
4. Sub-skill completes; PM continues pipeline.

Alternative (per-stage threshold):
1. Watchdog reads `session.current_stage` on each check.
2. If `current_stage in ("TEST", "DEPLOY")`, watchdog uses 3600s threshold instead of 600s.
3. All other stages use 600s.

Both approaches are complementary; implement both for defense-in-depth.

### Problem 2 — Plan-doc lifecycle

Trace from `docs/plans/sdlc-1304.md` (Data Flow section), abbreviated:

**Today:**
1. `/do-plan` forces `git checkout main`, commits plan, pushes to main.
2. Critique may re-commit to main mid-build, flipping PR to CONFLICTING.
3. `/do-merge` calls `migrate_completed_plan.py` AND `rm -f "$PLAN_PATH"` regardless of migration success.

**Proposed:**
1. `/do-plan` ensures `session/{slug}` branch, commits plan there, pushes to `origin session/{slug}`.
2. Critique revisions amend on the session branch. No main commit. PR mergeable state unaffected.
3. `/do-build` reads plan from working tree (already on `session/{slug}` in the worktree).
4. `/do-merge` calls `migrate_completed_plan.py`; on success, plan moves to `docs/plans/completed/{slug}.md` via a cleanup commit on main after merge. On failure, plan stays in `docs/plans/{slug}.md` as a visible warning. `rm -f` is removed.
5. A pre-push hook rejects `Plan: ...` commits destined for `main`.

## No-Gos

- **Migrating existing in-flight plans from main to session branches.** Convention applies forward only. Plans currently on main stay there as historical artifacts. [SEPARATE-SLUG #1304] A one-shot migration script was scoped in the prior plan but is not in scope here — the forward-only convention is the safe path.
- **Replacing `/do-plan` with a fully PR-based workflow** (issue #1304 Solution sketch 2). Adds another PR per feature for marginal value. The session-branch commit approach achieves the same isolation with less ceremony. Not done here because it adds complexity without benefit.
- **Designing a critique-amends-plan locking protocol** (race between concurrent build and plan-revision). This plan moves the race off main. [SEPARATE-SLUG #1304] The concurrent-edit lock belongs to a separate issue — the race condition is narrowed but not eliminated by this plan, and solving it requires a different surface (worktree file locks or plan-revision guards).
- **Extending watchdog resilience to non-PM sessions.** Dev sessions have shorter expected run times. The extended threshold and liveness tokens apply to PM-orchestrated long-running stages only. Non-PM session watchdog tuning is a separate concern with no observed failure mode.
- **Renaming `docs/plans/completed/` to `docs/plans/done/`.** Both directories exist; use `completed/` (more populated). Renaming would require migrating all existing archived plans — not worth the disruption. Not in scope.
- **Unifying `.claude/skills/` and `.claude/skills-global/`.** Hard-link sync behavior is opaque; touching it risks breaking skill dispatch. Not in scope for this governance fix.

## Rabbit Holes

- **Building a generic "agent commits to protected branches" guard.** The narrow scope is `Plan: ...` commits. Don't rebuild branch protection.
- **Auto-rewriting old plans' tracking-issue links.** Existing closed issues link to main blobs. Leave them alone — those plans are archived and links resolve fine.
- **Dynamic watchdog thresholds based on test suite size.** The per-stage static map is sufficient. A dynamic threshold requires introspecting the test suite, which adds complexity with marginal benefit.
- **Streaming aggregate output from `do-test` into the PM session transcript.** Out of scope. The PM receives a completion signal; test output stays in the dev session's log.
- **Monitoring Render deploy progress via API.** The `/do-deploy` skill currently polls a URL or waits. Integrating Render's API for progress streaming is a separate enhancement.
