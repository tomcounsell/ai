# Bridge Media Persist — Loud Failure + Worker Self-Heal (sdlc-1330)

## Problem

A Telegram voice message arrived at the bridge, was successfully downloaded to disk by Telethon, but never reached the agent as a transcription. The agent replied *"No file attached on my end — just text came through."* The file was orphaned: present on disk, absent from the `TelegramMessage` record, never transcribed.

Concrete incident: 2026-05-08 11:18:15 UTC, DM chat `179144806`, Telegram msg `9730`, file `bridge/data/media/voice_20260508_111815_9730.ogg` (29 KB, on disk), but `media_local_path=None` on the record. Worker enrichment emitted the warning `[enrichment] has_media=True but media_local_path is unset; skipping AI enrichment (legacy record?)`. The smoking gun: a `POPOTO.Query ERROR: one or more redis keys points to missing objects` fired 60 ms before the download started.

**Current behavior:**
- `bridge/telegram_bridge.py:1256` calls `TelegramMessage.query.get(stored_msg_id)`.
- During a transient Popoto stale-index condition, `query.get()` returns `None`.
- The `if _record is not None:` branch is skipped **silently** — no log, no retry, no fallback.
- The file remains on disk; the record has `media_local_path=None`; worker skips enrichment; the `[media]` placeholder is all the agent sees.
- A second silent path exists: `_download_media_with_retry` can return `(None, None)` when `download_media()` returns `None` because `filepath.exists()` is False after `client.download_media()` (`bridge/media.py:316-318`) — outcome `no_path`, no error string. The persist block becomes a no-op with no warning.

**Desired outcome:**
A voice (or image, or document) message that arrives during a transient Popoto inconsistency still gets transcribed/described and the agent receives the content. The previously-silent failure modes log a `WARNING` with enough metadata to correlate (stored_msg_id, chat_id, message_id, downloaded path).

## Solution

### Key Elements

- **Loud bridge persist**: After `_record = TelegramMessage.query.get(stored_msg_id)`, branch on `None` and emit a `WARNING` with `stored_msg_id`, `chat_id`, `message.id`, and the downloaded path. Branch on the dual-`None` case (`_local_path is None` AND `_download_error is None`) and emit a `WARNING` with the same metadata.
- **Single bounded retry on `None`**: When `query.get(stored_msg_id)` returns `None`, do ONE immediate re-query (no `keys(clean=True)` — heavy, mutates the index, and per memory `feedback_never_raw_delete_popoto.md` we don't run index-clean on hot paths). If the re-query also returns `None`, log and fall through.
- **Worker-side self-heal**: In `bridge/enrichment.py`, when the existing warning `[enrichment] has_media=True but media_local_path is unset` would fire AND `media_download_error` is also unset, search `bridge/data/media/` for `*_{message_id}.*`. If exactly one file matches, adopt it (treat as if it were `media_local_path`) and proceed to AI enrichment. If zero or multiple match, log and fall through to current skip behavior.

### Flow

Bridge intake → Telethon download succeeds → file on disk → `_record = query.get(stored_msg_id)` returns `None` → log WARNING → single re-query → still `None` → log second WARNING and proceed (file stays on disk, record has `media_local_path=None`) → AgentSession enqueued → worker enrichment loads record → sees `has_media=True, media_local_path=None, media_download_error=None` → searches `bridge/data/media/*_{message_id}.*` → finds the orphaned file → runs Whisper/vision → agent receives transcription.

### Technical Approach

1. **`bridge/telegram_bridge.py:1253-1266` — make silent branches loud.**
   - After `_record = TelegramMessage.query.get(stored_msg_id)`, if `_record is None`: do ONE immediate retry of `TelegramMessage.query.get(stored_msg_id)`. If still `None`, `logger.warning` with `stored_msg_id`, `chat_id=event.chat_id`, `message.id`, `local_path=_local_path`, `download_error=_download_error`. Do NOT call `keys(clean=True)`.
   - After the persist block, if `_local_path is None` AND `_download_error is None` AND `message.media` is truthy: `logger.warning` indicating the download returned no path with no error (the `no_path` outcome from `_download_media_with_retry`). This catches the second silent path even when the record IS present.

2. **`bridge/enrichment.py:73-91` — worker-side self-heal on missing `media_local_path`.**
   - In the existing `if not media_local_path:` branch, when both `media_local_path` and `media_download_error` are falsy, attempt a file-search fallback before the existing "legacy record?" log.
   - Compute `message_id` from `getattr(telegram_message, "message_id", None)`. If unset, skip (current behavior).
   - Search `bridge/data/media/*_{message_id}.*` via `Path("bridge/data/media").glob(f"*_{message_id}.*")`. If exactly one match: log INFO `[enrichment] self-heal: recovered orphan media file <path> for message_id={message_id}`, set local `media_local_path = str(path.resolve())`, and fall through to the existing AI enrichment branch at line 92.
   - If zero or multiple matches: fall through to the existing "legacy record?" warning (current behavior).
   - The `bridge/data/media/` path must be resolved from the same location the bridge writes to (`bridge.media.MEDIA_DIR`). Import that constant rather than hard-coding the string.

3. **Logging hygiene.** All new warnings use structured key=value format (`stored_msg_id=N chat_id=N message_id=N local_path=...`) consistent with existing `[media]` and `[enrichment]` log conventions.

## Risks

### Risk 1: File-search self-heal adopts the wrong file
**Impact:** If a glob match returns a stale file from a previous session with the same `message_id` (theoretically possible across very long-running deployments if `bridge/data/media/` is never cleaned), the agent gets the wrong content.
**Mitigation:** `message.id` is unique per chat per Telegram session; the existing filename includes a timestamp prefix. Adopt the file only when exactly one match is found; log the path so audit is possible. Two-match collisions log and skip — that surfaces the issue rather than silently misrouting.

### Risk 2: Re-query on transient Popoto inconsistency still returns `None`
**Impact:** The bridge persist block still no-ops; the record never gets `media_local_path` written. We rely entirely on worker self-heal for recovery.
**Mitigation:** Worker self-heal is the durable safety net; the bridge re-query is best-effort. Both layers must work — we don't trust either alone.

### Risk 3: Worker self-heal masks a real download failure
**Impact:** If the file was actually NOT downloaded but a stale file from a previous incident is sitting in `bridge/data/media/` with the same `message_id`, the worker could adopt it.
**Mitigation:** Same as Risk 1 — message IDs don't collide within a Telegram session lifetime. The self-heal only fires when `has_media=True` AND `media_local_path=None` AND `media_download_error=None` — the third clause means the download wrapper genuinely returned `(None, None)`, not a known error. Self-heal logs the adopted path at INFO so we can audit if a regression appears.

## Race Conditions

### Race 1: Worker reads `bridge/data/media/` before bridge finishes the download
**Location:** `bridge/enrichment.py` self-heal glob vs `bridge/media.py:314` `client.download_media`
**Trigger:** Worker enrichment fires for an AgentSession while the bridge is still downloading the media file.
**Data prerequisite:** The file at `bridge/data/media/{prefix}_{timestamp}_{message.id}{ext}` must be fully written before glob.
**State prerequisite:** Bridge intake completes `download_media` before enqueuing the AgentSession.
**Mitigation:** The bridge already awaits `_download_media_with_retry` BEFORE the AgentSession is enqueued for worker processing (`bridge/telegram_bridge.py:1241` runs before the `dispatch_telegram_session` call further down). By the time the worker picks up the session, the file is either on disk or the download wrapper has returned. No additional locking needed.

## Open Questions

1. Should the worker self-heal also handle the case where `media_local_path` is set but the file is missing on disk (e.g., file deleted between bridge and worker)? Current plan does NOT — that path is already handled by the existing `path.exists() and os.access(path, os.R_OK)` check at `enrichment.py:94`. If the file is gone, we don't try to recover. Acceptable?
2. The bridge re-query strategy is a single immediate retry. If the Popoto inconsistency takes longer than that to settle (which would be unusual), the re-query also returns `None`. The worker self-heal covers this — but should the bridge re-query also try a small backoff (e.g., 50ms, 100ms)? The current plan says no (keep the hot path fast); the worker is the safety net. Confirm?
