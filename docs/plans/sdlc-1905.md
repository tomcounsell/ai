# Dead telemetry counter: D0 never-started gate makes no_output_budget_exceeded unreachable

## Problem

The D0 never-started gate (commit `2efb58ce`, issue #1724) sits at the top of
`_has_progress` sub-check B in `agent/session_health.py`. It returns `False` for
any session with zero SDK output (`sdk_ever_output=False`) once
`running_seconds > NEVER_STARTED_GRACE_SECS (120) + NEVER_STARTED_CONFIRM_MARGIN_SECS (30) = 150s`.
That early return preempts the entire #1356 grace/budget band that follows it.

**Current behavior:**
Two code paths downstream of the D0 gate are dead — but only *conditionally*
dead under the current code, because of a cross-clock inconsistency the removal
must fix first (see below):

1. The budget-band leg `if running_seconds <= NO_OUTPUT_BUDGET_SECONDS: return True`
   (line 1593). To reach it, a non-legacy session must pass the D0 gate — which
   requires `running_seconds <= 150` — and then the `running_seconds < STARTUP_GRACE_SECONDS (300)`
   leg (line 1585) already returns `True` for every such survivor. No session
   with `running_seconds` in `(300, 1800]` ever reaches this leg's condition.
2. The telemetry INCR block
   `{project_key}:session-health:tier1_falloff:no_output_budget_exceeded`
   (lines 1606-1617) fires only when `running_seconds > 1800` AND
   `sdk_ever_output=False` — a strict subset of the population the D0 gate
   already returned `False` for at 150s. The counter can never increment.

`tests/unit/test_health_check_recovery_finalization.py::test_d0_gate_preempts_budget_exceeded_counter`
(added in PR #1903) already pins that the counter never fires.

**Cross-clock hazard that must be fixed before the deadness proof holds
(critique blocker, #1905):**
The deadness argument above assumes the D0 gate and sub-check B's
`running_seconds` share one clock. They do NOT, today. The D0 gate at line 1560
calls `_never_started_past_grace(entry)` with **no `now` argument**, so the
function derives its own `now` from `datetime.now(tz=UTC)` (the worker's local
wall clock — line 1385). Sub-check B's own `running_seconds` (line 1584) is
computed from `now_utc = _trusted_utc_now()` — Redis TIME, the C2/#1817 trusted
clock that exists **precisely because a worker's local clock can lag Redis
TIME**. Under >150s of skew (local clock behind Redis), the D0 gate can compute
a local `running_seconds <= 150` and return `False` (not-yet-past-grace) while
sub-check B's trusted `running_seconds` already exceeds 1800s — reaching the
supposedly-dead budget leg / INCR block for a real, non-legacy session. The
central safety invariant ("the leg is unreachable") is therefore false until
both checks share `now_utc`.

**Fix that makes the leg provably dead:** thread the trusted clock into the D0
gate — change line 1560 to `_never_started_past_grace(entry, now=now_utc)`. The
function already accepts an optional `now` (line 1343); the two peer call sites
(lines 4432, 4445) already thread `now`; the 1560 site is the outlier. Once both
checks share `now_utc`, `running_seconds <= 150` (D0-gate survivor) ⟹
`running_seconds < 300 = STARTUP_GRACE_SECONDS` holds **unconditionally** — the
`running_seconds < STARTUP_GRACE_SECONDS` leg returns `True` for every survivor,
so the band leg and INCR block are truly unreachable and safe to delete. This
clock-consistency fix is sequenced **before** the dead-code removal, and a
divergent-clock regression test pins the invariant before the leg is deleted.

**Desired outcome:**
The D0 gate call site is made clock-consistent (`now=now_utc`), then the dead
budget-band leg and the dead INCR block are removed (NO LEGACY CODE TOLERANCE). The `_has_progress` docstring and the module-level
constant comments document the D0 gate (150s) as the authoritative bound for
never-started sessions; the #1356 1800s band is subsumed. The four live feature
docs that still describe the counter are corrected. The constant
`NO_OUTPUT_BUDGET_SECONDS` is retained (still used by the #1614 own-progress
heartbeat gate and the Tier-2 reprieve cap).

  this is what MAKES the band leg provably dead**: change
  `_never_started_past_grace(entry)` to `_never_started_past_grace(entry, now=now_utc)`
  so the D0 gate and sub-check B's `running_seconds` (line 1584) share the single

  **before** the removal — without it, a >150s local-vs-Redis skew lets a
  never-started session reach the band leg / INCR block, so the leg is not
  actually dead. The peer call sites (lines 4432, 4445) already thread `now`;

  it does (`now_utc`), and the 1560 site now passes it.
- **`_has_progress` sub-check B (`agent/session_health.py`)**: After the clock fix,
  remove the (now-provably-dead) budget-band `return True` leg (line 1593) and the
  dead `no_output_budget_exceeded` INCR block (lines 1599-1617). Keep the D0 gate,
  the legacy-None fast-path (line 1580), and the

- **Tests**: Add a divergent-clock regression test (proving that with the shared

### Technical Approach

- **Sequencing is load-bearing.** The clock-consistency fix (thread `now=now_utc`
  into the line-1560 D0 gate) lands first and is pinned by a divergent-clock
  regression test; only then are the band leg and INCR block deleted. Deleting the
  leg before the clock fix would remove the code that (incorrectly, under skew)
  handles the reachable case, changing behavior for a real never-started session.
- The constant `NO_OUTPUT_BUDGET_SECONDS` (line 376) is **retained** — it is still
  read at line 1638 (#1614 own-progress heartbeat gate) and underpins the Tier-2
  reprieve cap via `MAX_NO_OUTPUT_REPRIEVES`. Removing it is out of scope and would
  break live behavior.
- The `running_seconds` computation (lines 1573-1584) is **retained** — the
  `started_ref` legacy-None branch (return True, line 1580) and the
  `running_seconds < STARTUP_GRACE_SECONDS` leg (line 1585) remain reachable and
  load-bearing for the `running_seconds <= 150` survivors of the D0 gate and for
  legacy/phantom records.
- After removal, sub-check B for a never-started session past 150s returns `False`
  at the D0 gate exactly as it does today. No observable behavior changes for any
  reachable input; this is a dead-branch prune plus documentation truth-up.
- Do NOT touch `docs/plans/completed/sdlc-1356.md` — it is an archived historical

## Risks

### Risk 1: A reachable input path is misjudged as dead and its removal changes behavior.
**Impact:** A never-started session's liveness verdict silently flips, risking a false kill or a missed recovery.
**Root cause the critique caught:** under the *unfixed* code the leg was NOT dead — a >150s local-vs-Redis clock skew reached it for a real session, because the D0 gate used the worker's local clock while sub-check B used the trusted Redis clock. This is why removal without the clock fix would change behavior.
**Mitigation:** (1) The clock-consistency fix (thread `now=now_utc` into the D0 gate) lands FIRST, making the leg provably dead, and is pinned by the new `test_divergent_clock_d0_gate_fires_before_removed_leg` regression test written and green BEFORE the leg is deleted. (2) The full `TestSubCheckBNoOutputBudget` suite (minus the two deleted counter tests, plus the new divergent-clock test) is retained unchanged and must stay green — it covers every reachable branch (legacy-None, 30s, 299s, 600s, 1500s, 1801s, 4h, sub-check A fall-through, clock skew, created_at fallback). Green suite = no observable behavior change. Verification rows assert the D0 gate threads `now_utc`, the divergent-clock test is present, and the counter key string is absent from the source.

### Risk 2: A live doc or the retained constant comment is left describing the removed counter.
**Impact:** Operators reading `docs/features/*` or the module comments chase a metric that no longer exists.
**Mitigation:** Grep for `no_output_budget_exceeded` across live (non-archived, non-test) source and docs post-change; the only remaining hits must be the retained-constant comments (reworded) and the archived `docs/plans/completed/sdlc-1356.md`. Verification row enforces this.

## Race Conditions

No race conditions identified. `_has_progress` is a synchronous, single-threaded
predicate over one `AgentSession` record. The removed INCR was the only shared-state
write in the affected branch; deleting it removes a write, introducing no new
concurrent access. The retained reads use the already-trusted `_trusted_utc_now()`
clock source (#1817) — and the clock-consistency fix extends that trusted clock
to the D0 gate (`now=now_utc`), so the gate and sub-check B's `running_seconds`
now derive from one monotonic source, eliminating the cross-clock skew hazard the
critique identified.
