# Remove stale `MAX_CONCURRENT_DEV_SESSIONS` from `.env.example` (W5)

## Problem

Item W5 from the Worker Lifecycle & Session Management integration audit (umbrella issue #1019) flagged a supposed env-var name mismatch: `MAX_CONCURRENT_DEV_SESSIONS` "missing" from `.env.example` while `MAX_CONCURRENT_SESSIONS` was present.

The investigation inverts that framing. `MAX_CONCURRENT_DEV_SESSIONS` is not *missing* from `.env.example` — it is *present but dead*. The worker code no longer reads it. The audit ran before PR #1029 finished cleaning up its own artifacts; what remains in `.env.example` is a stale four-line block that will mislead any operator who sets the variable expecting it to do something.

**Current behavior:**
- `.env.example` lines 49–52 document an optional `MAX_CONCURRENT_DEV_SESSIONS=1` env var with the note "Must be <= MAX_CONCURRENT_SESSIONS. Default 1 preserves sequential behavior."
- `worker/__main__.py:179` only reads `MAX_CONCURRENT_SESSIONS` (default 8). There is no reader for `MAX_CONCURRENT_DEV_SESSIONS` anywhere in runtime code (`agent/`, `worker/`, `config/`, `ui/`, `tests/`, `docs/features/`).
- An operator setting `MAX_CONCURRENT_DEV_SESSIONS=3` in `~/Desktop/Valor/.env` gets no behavior change and no warning. The variable is silently ignored.

**Desired outcome:**
- `.env.example` documents only the env vars the worker actually reads.
- The stale four-line `MAX_CONCURRENT_DEV_SESSIONS` block is removed.
- Item W5 on umbrella issue #1019 is closed out with a reference to the cleanup PR.
