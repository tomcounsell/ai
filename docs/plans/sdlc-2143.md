# Worker hang detection latency + resume-to-pending pickup stall

## Problem

Two related worker-reliability defects, investigated together under one slug.

**#2143 — worker main loop hangs, watchdog detection ~10× its threshold.**
On 2026-07-17 the worker hung twice in one day (both needed SIGKILL). The
watchdog log read *"heartbeat is 1789s old (threshold 180s)"* — a ~30-minute
machine-wide outage that should have been bounded to a few minutes.

**#2165 — a session resumed to `pending` sits unpicked for 8–16 minutes.**
During supervised SDLC runs with frequent `valor-session resume`, a
terminal→pending resume is not popped by the per-key queue loop; the 5-min
health check reports the worker as live so the orphaned-pending rescue never
fires. A manual `worker-restart` recovers instantly.

**Current behavior:**
- The watchdog's launchd `StartInterval` is **300s** while its staleness
  threshold is 180s (and the module docstring wrongly claims 120s). Best-case
  detection is already ~300–480s; a healthy tick logs nothing, so "watchdog
  not running" is indistinguishable from "watchdog saw healthy."
- When the external watchdog SIGTERMs a wedged worker, **no thread dump is
  produced** — the SIGTERM handler only logs and sets an asyncio.Event a wedged
  loop never processes. The only thread dump today is on the deadman's own
  `_self_kill` path.
- `resume_session` transitions terminal→pending but never publishes the
  session-notify nor calls `_ensure_worker` — unlike `create`. Pickup then
  depends on a fragile 1.5s poll or the 5-min health scan whose liveness test
  is blind to a parked/busy loop.

**Desired outcome:**
- Watchdog detection latency bounded near ~2× its threshold, with every check
  tick logged so silence is unambiguous.
- A wedged worker that survives a watchdog SIGTERM leaves an all-threads stack
  dump in `logs/worker_error.log` so the next occurrence is diagnosable.
- A resumed-to-pending session is picked up within ~1s (one notify hop),
  without a worker restart.

## Why Previous Fixes Failed

| Prior Fix | What It Did | Why It Was Incomplete |
|-----------|-------------|-----------------------|
| PR #1823 (#1815) | Dead-man's switch on the on-loop beacon | Detects only a synchronously frozen event loop. A *parked single per-key loop* keeps the beacon fresh (loop-tick still runs) → deadman writes green forever, watchdog sees green → #2165 is invisible to it. A freeze that also blocks the off-loop thread → deadman can't self-kill → #2143's 30-min hang. |
| PR #2164 (#2146) | SIGKILL-failure escalation + kill-list | Improves what happens *after* detection; does nothing about the 300s detection latency or the missing forensic dump. |

**Root cause pattern:** the liveness machinery equates "the asyncio Task object is not `.done()`" (health rescue) and "the on-loop beacon is fresh" (deadman) with "the worker is doing useful work." Both are false for a parked/wedged loop. And the *external* watchdog — the one backstop that can catch a whole-process freeze — runs too infrequently (300s), logs nothing on healthy ticks, and captures no forensics on the kill.

With no notify, pickup falls to either: (a) an existing idle `_worker_loop`'s 1.5s `DRAIN_TIMEOUT` re-poll (`agent_session_queue.py:2073`), or (b) the 5-min health pending-scan (`session_health.py:3703`), whose `worker_alive` test (line 3715-3716) treats a parked/busy loop as alive → no-op `event.set()` nudge → `continue`, never starting a fresh popper.
4. `worker-restart` re-scans `status=pending` at startup and `_ensure_worker`s every pending worker_key (`worker/__main__.py:881-887`) after rebuilding the index → instant pickup.

**#2143 watchdog detection (the gap):**
1. Off-loop heartbeat thread writes `data/last_worker_connected` mtime each cycle (`session_health.py:4235`).
2. launchd fires `worker_watchdog.py` every **300s**; `check()` reads the file mtime (`worker_watchdog.py:182`); if age > 180s → status "stale" → escalation ladder.
3. Detection latency = up to 300s (poll gap) after the heartbeat crosses 180s; healthy ticks log nothing.

## Solution

### Key Elements

- **Watchdog cadence + observability**: reduce the launchd `StartInterval` to a value ≤ half the 180s threshold (target 90s) so worst-case detection is ≤ ~2× threshold; log every check tick at INFO (including healthy) so a silent window unambiguously means "not running"; correct the module docstring to the real value.
- **Watchdog SIGTERM forensics**: install `faulthandler.register(signal.SIGTERM, all_threads=True, chain=True)` at worker startup so an external watchdog SIGTERM synchronously dumps all thread stacks to `logs/worker_error.log` (stderr) even when the event loop is wedged — then chain to the existing graceful handler.
- **Resume publishes like create (#2165)**: after `resume_session` transitions to pending, publish the same session-notify and call `_ensure_worker(worker_key)` so a resumed session is picked up within one notify hop, restart-free.

### Flow

`valor-session resume` → transition to pending → **publish notify + ensure_worker** → notify listener/idle loop pops within ~1s → session runs.

Watchdog: launchd every ~90s → `check()` → **log tick (healthy or stale)** → stale → escalation ladder (unchanged) → if the worker survives SIGTERM, **thread dump already in worker_error.log** from the faulthandler handler.

### Technical Approach

- **Watchdog StartInterval**: change `scripts/install_worker.sh` (the `<integer>300</integer>` under the watchdog plist) to `90`; also update `scripts/valor-service.sh` if it writes the same plist, and `scripts/lib/launchctl.sh`'s documented "300" comment. Fix the `monitoring/worker_watchdog.py:4` docstring. Make the healthy-tick log line INFO (not debug) in `main()` so it lands in `logs/worker_watchdog.log`. Existing installations pick up the new interval on the next `/update` (which re-installs the plist) — see Update System.
- **faulthandler on SIGTERM**: in `worker/__main__.py` `main()` (before `asyncio.run`), call `faulthandler.enable()` and `faulthandler.register(signal.SIGTERM, all_threads=True, chain=True)`. `chain=True` re-raises into Python's existing SIGTERM handler so the graceful-shutdown asyncio handler still runs on a *healthy* worker; on a *wedged* worker the C-level dump fires regardless. Direct the dump to the already-configured stderr (→ `logs/worker_error.log`).
- **Resume notify**: extract the create-path's notify publish into a reusable helper (or call the existing `_push_agent_session` notify segment) and invoke it from `resume_session` after the successful `transition_status`. Also call `_ensure_worker(session.worker_key, is_project_keyed=session.is_project_keyed)` — but ONLY when running inside the worker process's event loop; `resume_session` is also called from the CLI (a separate process with no worker loop and no running asyncio loop), where `_ensure_worker`/`create_task` must NOT be attempted. Gate on `asyncio.get_running_loop()` availability (try/except RuntimeError) so the CLI path publishes the notify (cross-process, via Redis) and the in-worker path additionally ensures a local loop. The notify is the cross-process mechanism that wakes the live worker regardless of which process issued the resume.

## Rabbit Holes

- **Do NOT try to root-cause the #2143 native-level freeze in this plan.** The evidence (alive PID + stale heartbeat + deadman not firing) points to a GIL-holding native/syscall block that needs a live thread dump to diagnose. This plan *enables* that forensic capture; it does not speculatively "fix" an unexplained freeze.
- **Do NOT rework the per-key serialization / head-of-line-blocking** (#2165 occurrence-1, a genuinely wedged `exec_task` holding its worker_key). That is per-key serialization working as designed and belongs to the #2143 freeze investigation, not the resume-notify fix.
- **Do NOT convert the watchdog from launchd StartInterval to a persistent loop.** A persistent daemon reintroduces the "watchdog itself can hang" failure mode #1767 deliberately avoided with a one-shot-per-tick design. Just shorten the interval.
- **Do NOT touch the deadman thresholds** (`WORKER_DEADMAN_*`). Out of scope; tuning them risks false-positive SIGKILLs.

## Risks

### Risk 1: Shorter StartInterval increases launchd wakeups on a battery-powered Air
**Impact:** Marginally more frequent watchdog process spawns (every ~90s vs 300s).
**Mitigation:** The watchdog `check()` is cheap (one stat + a Redis read). 90s is still well within normal launchd cadence; the bridge-watchdog already runs at 60s. Net battery impact negligible.

### Risk 2: `faulthandler.register(SIGTERM, chain=True)` interferes with graceful shutdown
**Impact:** If chaining misbehaves, a healthy worker might dump stacks on every normal SIGTERM (log noise) or skip graceful drain.
**Mitigation:** `chain=True` explicitly re-invokes the previously-registered Python SIGTERM handler after dumping, so the existing graceful-shutdown handler still runs. Verified by a unit test that patches the handler chain. The dump only adds a stack trace to `worker_error.log` on shutdown — acceptable, low-frequency noise that is itself useful signal.

### Risk 3: `_ensure_worker`/`create_task` called from the CLI process (no running loop) raises
**Impact:** `resume` from the CLI would crash.
**Mitigation:** Gate the `_ensure_worker` call on `asyncio.get_running_loop()` (try/except `RuntimeError`); the CLI path publishes the cross-process notify only. Explicit test for the no-loop path.

## Race Conditions

### Race 1: resume notify published before the pending transition is index-visible
**Location:** `tools/valor_session.py:837` (transition) → new notify publish.
**Trigger:** The worker pops on receiving the notify before the `status=pending` write is visible to `_pop_agent_session`.
**Data prerequisite:** the `transition_status(..., "pending")` save (and its index update) must complete before the notify is published.
**State prerequisite:** the pending record is queryable/poppable when the woken loop polls.
**Mitigation:** Publish the notify strictly AFTER `transition_status` returns successfully (transition is synchronous and updates the Popoto index before returning). The woken loop's pop is itself retried on the 1.5s drain fallback, so even a momentary visibility lag self-heals within one poll. This mirrors the create path, which publishes after the record is saved.

### Race 2: double pickup (notify + health scan both start a worker)
**Location:** `_ensure_worker` (`agent_session_queue.py:1601`).
**Trigger:** notify-driven `_ensure_worker` races the health-scan `_ensure_worker`.
**Data prerequisite:** none.
**State prerequisite:** `_active_workers`/`_starting_workers` guards must be idempotent.
**Mitigation:** `_ensure_worker` already has the two-guard idempotency mechanism (`_active_workers[key].done()` check + `_starting_workers` set) — a duplicate spawn is a no-op. No new race introduced.
