# Worker Media Enrichment: Bridge-Side Download, Worker-Side AI

## Problem

After the bridge/worker split, `bridge/enrichment.py` still gates its media branch on a module-level Telegram client (`_telegram_client`) that is only ever populated inside the bridge process (`bridge/telegram_bridge.py:1000-1002`). The worker process always sees `None`, the media branch silently no-ops, and every photo/voice/audio/document message reaches the agent stripped of its description. The agent honestly responds "I can't see image content" because no description was prepended. Reproduced 2026-05-06 04:45 UTC, msg 600 in Agent Builders Chat.

The deferred-enrichment pattern was introduced specifically to keep media downloads off the synchronous intake path. Reverting that was the original Option 3 in the issue. We don't want to do that.

**Revised:** Issue's Option 1 (worker initializes its own Telethon client against the same session file) carries a real SQLite-locking risk. Telethon stores session metadata in a SQLite database; two simultaneous `TelegramClient` instances on the same `.session` file race on the SQLite write lock, and the auth_key / DC state can desync. Telethon docs explicitly recommend one client per session. Verified by reading Telethon source (sessions/sqlite.py) — connections are not pooled across processes, and concurrent writes have caused `database is locked` reports in the wild. We will *not* take Option 1.

**Pre-requisites:** None. Bridge and worker run on the same machine and share `data/` via the filesystem.

**Dropped:**
- Option 1 (worker initializes its own Telethon client) — SQLite-shared-session risk, see Revised.
- Option 3 (fully revert deferred enrichment) — overcorrects; YouTube/links/reply-chain enrichment paths still benefit from being off the intake critical path.

## Spike Results

### spike-1: Is `process_incoming_media` separable into download + AI halves?

- **Assumption**: The Telethon-dependent half (`download_media`, `client.get_messages`) can be split from the AI-dependent half (`describe_image`, `transcribe_voice`, `extract_document_text`).
- **Method**: code-read.
- **Finding**: Yes. `bridge/media.py:404` is a single `download_media(client, message, ...)` call; the rest of the function operates on `downloaded: Path` and uses no Telethon objects. The split is a clean cut.
- **Confidence**: high.
- **Impact**: Plan introduces a new `process_downloaded_media(path: Path, media_type: str) -> tuple[str, list[Path]]` that takes the AI half. Bridge calls `download_media` + persists path. Worker calls `process_downloaded_media`. `process_incoming_media` is kept as a thin wrapper that calls both, used by any path that still has the client (none after this change, but harmless to retain for now).

### spike-2: Where in the bridge's intake path does the download fit?

- **Assumption**: Bridge can `await download_media(...)` after persisting the `TelegramMessage` and before enqueueing the `AgentSession`, without measurably degrading intake.
- **Method**: code-read of `bridge/telegram_bridge.py` `handler()` and timing estimation.
- **Finding**: The handler already does `await` work (Redis writes, classification). Adding one more `await download_media(...)` adds ~200ms-1s for typical media (network-bound). Acceptable. The download must complete before enqueue so the worker sees `media_local_path` populated.
- **Confidence**: medium (estimate based on Telethon RPC latencies; will validate with an actual photo in build).
- **Impact**: Insert download synchronously in the bridge handler. If download fails, persist `TelegramMessage` with `has_media=True, media_local_path=None, media_download_error="..."` so the worker can log a `WARNING` and proceed with caption-only.

### spike-3: Storage location and lifecycle for downloaded media files

- **Assumption**: Existing `download_media` already writes under a stable path; we just need to capture it.
- **Method**: code-read of `bridge/media.py:download_media`.
- **Finding**: `download_media` writes to `media/incoming/{prefix}_{timestamp}_{name}`. Files persist on disk indefinitely today. The worker must read them before they're cleaned up.
- **Confidence**: high.
- **Impact**: No new cleanup logic in this plan. Files stay where `download_media` puts them. Future cleanup is out of scope (separate issue).

### spike-4: Does `enrich_message` need any other Telethon-dependent path fixed?

- **Assumption**: Media is the only branch that uses `telegram_client`.
- **Method**: `grep -n "telegram_client" bridge/enrichment.py`.
- **Finding**: Two branches use `telegram_client`: media (line 77) and reply chain (line 158). Reply chain calls `fetch_reply_chain(client, ...)` which does `client.get_messages` over Telethon RPC. **This is also broken in the worker** for the same reason.
- **Confidence**: high.
- **Impact**: Plan must address reply chain too, or explicitly defer it. We address it: bridge fetches the reply chain synchronously at intake (same pattern as media), persists fetched messages on `TelegramMessage.reply_chain_json`, worker reads from there.

## Data Flow

**Today (broken):**
1. Telegram → bridge handler → persist `TelegramMessage(has_media=True)` → enqueue `AgentSession`.
2. Worker pops session → calls `enrich_message(telegram_client=None, ...)` → media branch skipped → agent gets bare caption.

**After this plan:**
1. Telegram → bridge handler → call `download_media(client, msg)` synchronously → persist `TelegramMessage(has_media=True, media_local_path="media/incoming/photo_...jpg")` → if `reply_to_msg_id`, also fetch reply chain via `fetch_reply_chain(client, ...)` → persist `reply_chain_json` → enqueue `AgentSession`.
2. Worker pops session → calls `enrich_message(...)` (no longer needs `telegram_client`) → media branch reads `media_local_path` from `TelegramMessage` → calls `process_downloaded_media(path, media_type)` → AI describe/transcribe/extract → reply-chain branch reads `reply_chain_json` → builds enriched text → agent gets full enrichment.

## Solution

**Chosen option: refined Option 2 (split download/AI).**

The bridge owns Telegram RPC; the worker owns AI work. We split `process_incoming_media` along that line. Bridge runs the cheap, network-bound, Telethon-dependent half synchronously at intake (~sub-second). Worker runs the slow, AI-bound half (vision, Whisper, document extraction) where it has been all along. Net change to intake latency: ~200ms-1s for media messages; zero for text-only.

### Concrete changes

1. **`models/telegram.py`** — add two fields:
   - `media_local_path: Field(null=True)` — string path to the downloaded file (relative to repo root) or `None` if download failed.
   - `media_download_error: Field(null=True)` — string error if download failed (so worker can log and proceed).
   - `reply_chain_json: Field(null=True)` — JSON-encoded list of reply-chain message dicts pre-fetched by the bridge.

2. **`bridge/media.py`** — refactor:
   - Extract `process_downloaded_media(path: Path, media_type: str) -> tuple[str, list[Path]]` containing the post-download AI logic (describe/transcribe/extract). No Telethon dependency.
   - Keep `process_incoming_media(client, message)` as a thin wrapper that calls `download_media` + `process_downloaded_media`. Existing callers continue to work.
   - Fix the stale comment at `bridge/media.py:436` (`# Use Ollama LLaVA` → `# Use Claude Haiku 4.5`).

3. **`bridge/telegram_bridge.py`** — in the message handler:
   - After `TelegramMessage` is persisted but before `AgentSession` is enqueued, if `has_media`, call `download_media(client, event.message, prefix=media_type)` and update the `TelegramMessage` with `media_local_path` (or `media_download_error` on failure). Wrap in try/except: a download failure must not block enqueue.
   - If `reply_to_msg_id` is present, call `fetch_reply_chain(client, chat_id, reply_to_msg_id)` and persist `reply_chain_json` on the `TelegramMessage`. Same try/except.

4. **`bridge/enrichment.py`** — rewrite:
   - Drop `_telegram_client`, `set_telegram_client`, `get_telegram_client`. Their last call sites are removed.
   - `enrich_message(...)` no longer takes `telegram_client`. Signature change.
   - Media branch: read `media_local_path` from the caller's already-loaded `TelegramMessage` (passed in as a new arg), call `process_downloaded_media(Path(media_local_path), media_type)`. If `media_local_path is None and media_download_error is not None`, log `WARNING` and skip (no silent fallthrough).
   - Reply-chain branch: read `reply_chain_json` from `TelegramMessage`, deserialize, call `format_reply_chain(...)` directly (no Telethon fetch).
   - Summary line: distinguish `media=yes` (description added), `media=no` (no media on message), `media=skipped:download_failed` (media present but no usable path), `media=skipped:client_unavailable` (defensive — should not occur after this fix).

5. **`agent/session_executor.py`** — at the call site (~line 1380):
   - Stop importing `get_telegram_client`. Pass the loaded `TelegramMessage` (already fetched at line ~1404 for the agent_session_id back-reference) into `enrich_message`. Reorder fetches if needed so the message is loaded before enrichment.
   - Remove the `telegram_client=tg_client` argument.

6. **`bridge/telegram_bridge.py`** — drop the `from bridge.enrichment import set_telegram_client; set_telegram_client(client)` block at lines 1000-1002. Dead code after the rewrite.

### Race condition analysis

- **Bridge writes `media_local_path` after `TelegramMessage` save but before enqueue.** Worker only sees the session after enqueue, so `media_local_path` is durably visible by the time the worker reads. Single-writer (bridge) for these fields; no cross-process race.
- **Disk file lifecycle.** `download_media` writes to `media/incoming/`. The worker reads later (could be seconds to hours, depending on queue depth). Files are not cleaned up by anything in scope of this plan — out-of-scope to introduce cleanup. Pre-existing behaviour.
- **Bridge crash mid-download.** `TelegramMessage` is persisted before download starts (handler order: persist → download → enqueue). If the bridge crashes during download, the session is never enqueued; on bridge restart, the message is in Redis without a session — pre-existing failure mode, unchanged.

### Why not Option 1

Option 1 (worker initializes its own Telethon client) requires either (a) sharing the SQLite session file with the bridge, which Telethon does not officially support and which has caused `database is locked` errors in the wild, or (b) a second Telethon login (separate session file), which adds an interactive setup step we have to plumb through `/setup` and `scripts/update`. Both options expand the surface area of "things that can break the bridge". We avoid it.

If the bridge and worker ever need to live on different hosts, we revisit — but that's a separate architectural decision (not "fix this regression").

## Implementation Notes (Critique Revision Pass — 2026-05-07)

Critique verdict: READY TO BUILD with concerns. The five concerns below are addressed pre-build so the builder has unambiguous direction.

### 1. Bridge-side Telethon RPC timeouts (HARD REQUIREMENT)

Bare `try/except` does not catch hangs — Telethon RPCs can stall indefinitely on network/DC issues, blocking intake.

- Wrap `download_media(client, event.message, prefix=media_type)` in `asyncio.wait_for(..., timeout=10.0)`. On `asyncio.TimeoutError`, persist `media_download_error="timeout after 10s"` and proceed to enqueue.
- Wrap `fetch_reply_chain(client, chat_id, reply_to_msg_id)` in `asyncio.wait_for(..., timeout=5.0)`. On timeout, persist `reply_chain_error="timeout after 5s"` (new nullable field — add alongside the others in `models/telegram.py`) and proceed.
- Both timeouts must be logged at WARNING with chat_id + msg_id for correlation.

### 2. Empty `TelegramMessage` fallback in `enrich_message`

`enrich_message` will be called from contexts where `telegram_message_key` may be missing or the lookup may return `None` (legacy sessions, manual test sessions, races). Spec the fallback explicitly:

- If `telegram_message_key is None` or `TelegramMessage.query.get(...)` returns `None`: log at INFO `[enrichment] no TelegramMessage record; skipping media/reply-chain branches`, set summary `media=skipped:no_record, reply_chain=skipped:no_record`, return the bare `message_text` unchanged.
- Do **not** raise. Do **not** treat absence as an error — it is a normal path for non-Telegram sessions and pre-migration records.

### 3. Worker filesystem access to `media/incoming/`

The bridge writes the file; the worker reads it. Single-machine assumption holds today, but we still verify access at the worker boundary rather than crashing inside `process_downloaded_media`.

- Persist `media_local_path` as an **absolute path** (resolve via `Path(downloaded).resolve()` in the bridge handler before saving). Avoids cwd-relative interpretation drift between bridge and worker processes.
- In `enrich_message` media branch: before calling `process_downloaded_media`, check `path.exists() and os.access(path, os.R_OK)`. On failure, log WARNING `[enrichment] media file at {path} not readable; skipping AI enrichment`, set summary `media=skipped:file_unreadable`, return bare caption.
- Document this contract in `docs/features/media-enrichment.md` so a future cross-host split surfaces the file-share requirement.

### 4. p95 latency measurement mechanism

The success criterion "Bridge handler latency for a photo intake stays under 2s p95" needs a concrete observation point. Two acceptable paths:

- **Preferred:** emit a structured log line `[bridge] intake_duration_ms={ms} has_media={bool} has_reply={bool} chat_id={id}` at the end of the handler (after enqueue). The metric is then derivable from log scraping or future Prometheus export. Builder adds the timing capture (`t0 = time.monotonic()` at handler entry, log at exit).
- **Or:** rephrase the success criterion to "Manual check: send 5 photos through the bridge, observe handler completion times in `logs/bridge.log` are all under 2s." Still requires the timing log line.

Builder MUST implement the timing log. The success criterion is then verifiable; without it, it is aspirational.

### 5. Reply-chain pre-fetch scope split

Issue #1297 is media-only. Reply-chain pre-fetch was added during planning because spike-4 found the same `_telegram_client` defect on that branch. Two acceptable resolutions:

- **Recommended:** split reply-chain work into a follow-up issue. Keep this plan media-only. Update **Step by Step Tasks** during build to drop reply-chain steps; update **Success Criteria** to drop reply-chain assertions.
