# SDLC Session-Resolution Skew: Writes Honor Env-Var Session, Reads Honor Issue Number

## Problem

During the #1668 SDLC run, the forked CRITIQUE and REVIEW stage subagents (spawned by `/do-sdlc`) wrote their verdicts, stage markers, and dispatch-history entries to a *different* `AgentSession` than the authoritative `sdlc-local-{issue}` session that the router resolves by `--issue-number`. The router's view showed `_verdicts.CRITIQUE` empty and `last_dispatched_skill` stranded at `/do-plan-critique`, tripping guard G3 in a loop. The pipeline only completed because a human manually reconstructed the authoritative session state.

The same divergence is the upstream cause of #1672: `/do-plan-critique` records its terminal verdict (Step 5.5 is now mandatory on every exit path since #1654), but because the verdict write targets the wrong session, the router reading `sdlc-local-{N}` sees CRITIQUE stuck at `in_progress` with no verdict.

**Current behavior:**
- READ path (`sdlc-tool next-skill --issue-number N`) resolves the session by issue number → deterministic `sdlc-local-{N}` (or the bridge PM session whose `issue_url` ends in `/issues/N`).
- WRITE path (`sdlc-tool verdict|dispatch|stage-marker|meta-set --issue-number N`) calls `find_session(..., issue_number=N, ensure=True)`, which checks `VALOR_SESSION_ID` / `AGENT_SESSION_ID` env vars **before** the `issue_number` lookup. A forked stage subagent that inherited a parent's `VALOR_SESSION_ID` writes to the *parent's* session, not `sdlc-local-{N}`.
- Result: reads and writes resolve to different sessions; SDLC state silently fragments.

**Desired outcome:**
Every `sdlc-tool` write issued with an explicit `--issue-number N` lands on the **same** session the router reads for that issue number. Session resolution is deterministic and convergent across the read path and all write paths. The bridge case (no `--issue-number`, env-var session is authoritative) is preserved byte-for-byte.

## Solution

### Key Elements

- **Issue-number-first write resolution**: When a write subcommand passes an explicit `issue_number >= 1`, `find_session()` attempts issue-based resolution (`find_session_by_issue`) **before** falling back to the env-var session ID. Only if issue-based resolution finds nothing does it fall back to `VALOR_SESSION_ID` / `AGENT_SESSION_ID`, then to the auto-ensure path.
- **Symmetric fix in `ensure_session()`**: The env-var short-circuit in `tools/sdlc_session_ensure.py` is reordered so that an explicit `issue_number` resolving to an existing issue-scoped session wins over the inherited env-var session. The bridge dedup (#1147) is preserved: when issue-based resolution returns nothing, the env-var short-circuit still reuses the bridge session.
- **Convergence guarantee**: After the change, for any given `issue_number`, the session selected by the READ path (`find_session_by_issue`) and the WRITE path (`find_session(..., ensure=True)`) is identical, because both now consult `find_session_by_issue` first.
- **Skill-doc correction**: Update the `/do-plan-critique` SKILL.md note (~line 275) and the matching `/do-pr-review` / `/do-build` notes so the env-var fallback is documented as a *last resort*, subordinate to `--issue-number`, rather than a co-equal resolution path.

### Flow

`sdlc-tool verdict record --issue-number N` (forked subagent, inherited `VALOR_SESSION_ID=parent`) → `find_session(session_id=None, issue_number=N, ensure=True)` → **issue-based pass first** → `find_session_by_issue(N)` finds `sdlc-local-N` → write lands on `sdlc-local-N` → router reading `--issue-number N` sees the verdict. (Previously: env-var pass first → write lands on `parent` → router sees nothing.)

Bridge case (real PM session, `VALOR_SESSION_ID=pm-bridge`, write *without* `--issue-number`): `find_session(session_id=None, issue_number=None, ensure=True)` → no issue number → env-var pass → resolves `pm-bridge`. Unchanged.

Bridge case *with* `--issue-number N` where the bridge PM session's `issue_url` ends in `/issues/N`: issue-based pass finds the same bridge PM session (via the `issue_url` pass in `find_session_by_issue`) → identical result, no behavior change.

### Technical Approach

- In `tools/_sdlc_utils.py::find_session`, restructure resolution order to:
  1. Explicit `session_id` argument (highest precedence — callers that pass a concrete id mean it).
  2. **If `issue_number is not None and issue_number >= 1`: try `find_session_by_issue(issue_number)`; return it if found.** (NEW: moved ahead of env-var resolution.)
  3. Env-var session ID (`VALOR_SESSION_ID` / `AGENT_SESSION_ID`) — fallback when no explicit id and no issue match.
  4. Auto-ensure (writes only, gated on `issue_number >= 1` or env-var presence) — unchanged.
- The `session_id` *argument* still wins over issue-based resolution (step 1 before step 2). Only the *env-var* session loses to an explicit issue number. This is the minimal, surgical reordering — it does not touch the explicit-argument contract.
- In `tools/sdlc_session_ensure.py::ensure_session`, move the `find_session_by_issue(issue_number)` lookup ahead of the env-var short-circuit when `issue_number >= 1`, so an existing issue-scoped session is returned before the env session. Keep the env-var short-circuit as the fallback (bridge dedup preserved when no issue session exists).
- No change to the four write CLIs themselves — they already pass `--issue-number` through; the fix is entirely in the two shared resolvers.
- Update SKILL.md notes (`do-plan-critique`, `do-pr-review`, `do-build`) describing the fallback ordering so the docs match the new precedence.

## Rabbit Holes

- **Rewriting the resolver to a single unified `resolve_session(issue, session_id, env)` helper across read and write paths.** Tempting for elegance, but the read and write paths have legitimately different `ensure` semantics. Keep the surgical reorder; don't refactor the whole resolution surface.
- **Stripping `VALOR_SESSION_ID` from forked-subagent environments at the harness level.** This would also fix the symptom but is a much larger, riskier change to how subagents are spawned, and it would break the legitimate bridge case where the env var IS the right session. Out of scope — fix the precedence, not the environment.
- **Auditing every `sdlc-tool` caller for `--issue-number` presence.** The CLIs already pass it through; the fix is in the resolver. Don't go spelunking through every skill's bash blocks beyond the three SKILL.md doc-notes that explicitly describe the fallback ordering.
- **Building a cleanup utility to reconcile already-skewed sessions.** Per the prevention-over-cleanup principle: the fix prevents future skew; historical skewed records from the #1668 run were already manually repaired. No cleanup script.

## Risks

### Risk 1: Breaking the bridge dedup (#1147) by reordering `ensure_session`
**Impact:** If issue-based resolution runs before the env-var short-circuit unconditionally, a bridge PM session could be bypassed and a duplicate `sdlc-local-N` created.
**Mitigation:** The bridge PM session carries `issue_url` ending in `/issues/N`, so `find_session_by_issue(N)` resolves *to the bridge session itself* (via the `issue_url` pass) — not a new session. Add a test with a bridge-shaped PM session (issue_url set, env var set) asserting `ensure_session(N)` returns that same session and does NOT create a duplicate. Gate the new pass on `issue_number >= 1` so no-issue-number bridge writes keep the env short-circuit.

### Risk 2: A caller relies on env-var-wins for an issue-scoped write
**Impact:** Some skill might intentionally set `VALOR_SESSION_ID` to redirect an issue-numbered write to a non-issue session.
**Mitigation:** Grep confirms no such caller exists — the env-var fallback is documented only as a *last resort* when `--issue-number` is unknown (SKILL.md notes say "If `$ISSUE_NUMBER` is unknown, omit the flag"). The new precedence matches the documented intent. The `session_id` *argument* still overrides issue resolution for callers that genuinely need an explicit target.

## Race Conditions

No race conditions identified. Session resolution is synchronous and read-mostly: each `sdlc-tool` subcommand is a short-lived subprocess that resolves one session and performs one write. The auto-ensure path is idempotent (keyed on the deterministic `sdlc-local-{N}` id with an existing-by-id check before create), so concurrent ensures of the same issue converge on one record rather than racing. No shared mutable in-process state is introduced.
