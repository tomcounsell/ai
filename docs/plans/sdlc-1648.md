# Dashboard telemetry parity for granite PTY sessions

## Problem

During live observation of a real production session (`ec1e7c6ede0b4cc491247067dea676f6`, 2026-06-12), every question the dashboard exists to answer — "is it making progress?", "what is it doing right now?", "did it succeed?" — required bypassing the dashboard and hand-reading `~/.claude/projects/*/*.jsonl` transcripts plus `logs/worker.log`. The dashboard actively misled: it showed 102 seconds of evidence, then went silent, while five more minutes of real work happened, and finally rendered an `exception` exit as a `completed` success with a 👏 reaction — for a session that never sent the human a single message and whose work product was force-deleted.

**Current behavior:**
The granite PTY container (production since #1572) drives PM + Dev interactive `claude` TUIs over pseudo-terminals, bypassing `agent/sdk_client.py` — the sole writer of per-session telemetry. As a result:

- `turn_count`, `tool_call_count`, all token/cost counters → `0`
- `current_tool_name`, `last_tool_use_at`, `recent_thinking_excerpt`, `last_stdout_at` → `null`
- `harness_pid`, `process_alive` → `null` (two live `claude` TUI PIDs the entire session)
- `events` → opaque rows; the `exit_summary`/`exit_anomaly` dicts use a `type` key with no `text`, so `ui/data/sdlc.py::_parse_history` renders them as raw `str(entry)` dumps
- `status` → `completed` even on `exit_reason: 'exception'`, because `bridge_adapter.run()` returns `""` (no raised error) and the executor maps status as `"completed" if not task.error`
- `_maybe_publish_exit_anomaly` early-returns on `exception` — no ERROR log, no Sentry, no `exit_anomaly` event

The only working field, `last_turn_at`, works only because of the granite-specific `_bump_last_turn_at` `on_turn` shim — whose docstring explicitly notes the rest of the harness-path signals are dead on this path.

**Desired outcome:**
A granite session's dashboard card carries the same fidelity an operator gets from the SDK path — live counters, current activity, token totals, real PIDs, a first-class `exit_reason`, anomaly treatment for `exception` exits, routing-protocol events in the feed, and a visible flag when a session "completed" without ever talking to the human. An operator should never again have to drop to raw JSONL to answer the three core questions.

## Data Flow

**Current (hollow) flow:**

1. **Entry point**: Worker executes a granite session → `BridgeAdapter.run()` → `Container.run()` in `asyncio.to_thread`.
2. **Container loop**: spawns 2 `claude` TUIs over PTYs; each TUI writes a complete JSONL transcript under `~/.claude/projects/{cwd-slug}/{uuid}.jsonl`. The loop accumulates `TurnRecord`s and byte counts into an in-memory `ContainerResult`.
3. **Exit**: `BridgeAdapter._publish_exit_summary` writes one `exit_summary` dict (with `exit_reason`, `turns`, `compliance_misses`) into `session_events`. `_maybe_publish_exit_anomaly` skips `exception`. `run()` returns `""`.
4. **Status**: executor sets `status = "completed"` (no `task.error`). `ContainerResult` is **discarded** — turn/tool/token counts, PIDs, current activity never reach the `AgentSession` record.
5. **Output**: `ui/app.py::_session_to_json` reads the `AgentSession`'s default-`0`/`null` fields → hollow card.

**Target (instrumented) flow:**

1. **Entry point** (unchanged): `BridgeAdapter.run()` → `Container.run()`.
2. **Container loop**: each PTY's transcript UUID + child PID are captured at spawn time and exposed on the result. A lightweight **transcript tailer** (running in the worker's event loop, NOT inside the PTY thread) periodically reads the PM + Dev JSONL files and computes `turn_count`, `tool_call_count`, token totals, `current_tool_name`, `last_tool_use_at`, `recent_thinking_excerpt`, persisting them onto the `AgentSession` via partial saves (mirroring the `_bump_last_turn_at` pattern).
3. **Exit**: `_publish_exit_summary` now also writes `exit_reason` onto a first-class `AgentSession.exit_reason` field and writes granite identity (PTY slot, both PIDs, both transcript paths) onto the record. `_maybe_publish_exit_anomaly` includes `exception`.
4. **Status / reaction**: a non-clean `exit_reason` renders as a warning state regardless of lifecycle status; the success reaction is gated on `exit_reason` + `user_facing_routed`.
5. **Output**: `_session_to_json` emits the new first-class fields; the card and modal render counters, identity links, the `exit_reason` warning, and a `communicated=False` chip.

## Architectural Impact

- **New dependencies**: none (no new pip packages). Reads JSONL with stdlib `json`.
- **Interface changes**:
  - `ContainerResult` gains `pm_pid`, `dev_pid`, `pm_transcript_path`, `dev_transcript_path`, `dev_resume_uuid` (additive fields, defaults preserve existing callers).
  - `PTYDriver` gains a public `pid` property (`self._child.pid` or None).
  - `AgentSession` gains `exit_reason` (and granite-identity fields, see Solution) — additive nullable fields.
  - New module `agent/granite_container/transcript_tailer.py` with a pure-ish `read_transcript_telemetry(path) -> TranscriptTelemetry` function.
- **Coupling**: the tailer introduces a *read-only* coupling from the granite path to Claude Code's JSONL format. This format is already parsed in `agent/sdk_client.py` (the SDK message stream is the same shape), so the coupling already exists elsewhere — we are not inventing a new dependency on an undocumented format, but we ARE depending on an on-disk layout. Mitigated by fail-silent parsing (see Risks).
- **Data ownership**: telemetry fields stay owned by `AgentSession`; the granite path becomes a second writer alongside `sdk_client`. The two never write the same session (a session is either SDK-path or granite-path), so no write contention.
- **Reversibility**: high. Every change is additive; reverting the tailer leaves granite sessions exactly as hollow as today, with no schema cleanup needed (nullable fields).

## Solution

### Key Elements

- **Transcript tailer** (`agent/granite_container/transcript_tailer.py`): pure function `read_transcript_telemetry(path)` that parses a Claude Code JSONL transcript and returns a `TranscriptTelemetry` dataclass — `turn_count`, `tool_call_count`, `total_input_tokens`, `total_output_tokens`, `total_cache_read_tokens`, `current_tool_name`, `last_tool_use_at`, `recent_thinking_excerpt`. Reads tolerant of partial/in-flight lines (the file is being appended live).
- **Tailer driver in `BridgeAdapter`**: a periodic async task (started in `run()`, cancelled on container exit) that reads the PM + Dev transcripts and persists aggregated telemetry onto the `AgentSession` via partial saves. Runs in the event loop, never blocks the PTY thread.
- **Transcript-path + PID capture**: `PTYDriver` exposes `pid`; `Container` records each PTY's transcript path (derived from cwd-slug + the resume UUID it already captures) and PIDs onto `ContainerResult`. `BridgeAdapter` writes them onto the `AgentSession` as granite-identity fields.
- **First-class `exit_reason`**: a nullable `AgentSession.exit_reason` field, written by `_publish_exit_summary`. The dashboard renders any non-clean reason as a warning chip even when `status == completed`.
- **`exception` as anomaly**: add `exception` to the `_maybe_publish_exit_anomaly` allowlist (ERROR log → Sentry, `exit_anomaly` event).
- **Routing-protocol events**: the container's existing `[/user]`/`[/complete]`/compliance-retry/classifier decisions are appended to `session_events` as typed entries with a `text` + `event_type` so `_parse_history` renders them as readable feed rows (instead of opaque dumps).
- **`communicated=False` completion chip**: the dashboard reads `user_facing_routed` (already on the record post-#1647) and renders a visible "no user message sent" chip on a completed session where it is False. The success reaction is gated on `exit_reason ∈ clean-set AND user_facing_routed`.

### Flow

Operator opens dashboard → sees a running granite session card → card shows live `turn_count` / `tool_call_count` climbing, `current_tool_name` ("Edit"), token totals, and "PTY slot 1 · PM pid 12345 · Dev pid 12346" with two transcript-path links → operator clicks a transcript path to jump to raw JSONL in one step → session ends on `exception` → card flips to a warning state ("exit: exception") with a red "no user message" chip, an `exit_anomaly` event in the feed, and an ERROR captured in Sentry — instead of a green success.

### Technical Approach

- **Reuse the cwd-slug + resume-UUID the container already has.** Claude Code names transcripts `~/.claude/projects/{slugified-cwd}/{session-uuid}.jsonl`. The container already captures `last_resume_uuid()` for the PM (`container.py:834`); extend to capture the Dev UUID and the cwd-slug so both transcript paths are derivable. The exact slug algorithm is the one verifiable unknown → **spike-1**.
- **Tail in the event loop, persist incrementally.** Model the persistence on the proven `_bump_last_turn_at` pattern (`save(update_fields=[...])`, fail-silent). Cadence is a tradeoff between freshness and Redis write volume → **spike-2** (confirm a sane interval; default ~5s). The tailer reads both transcripts each tick, sums counters, picks the most recent `current_tool_name` / thinking excerpt across the pair, and writes the aggregate.
- **`exit_reason` first-class, status untouched.** Do NOT change the executor's `status` mapping (it correctly maps non-clean granite exits to `failed` per the granite doc, but the adapter swallows the exit_reason before it reaches that code — see spike-3). Instead, persist `exit_reason` onto the record and let the *dashboard* render the warning. This avoids destabilizing the lifecycle state machine. Confirm the actual status-mapping path for granite → **spike-3**.
- **Routing events**: append typed `session_events` from the existing callback sites in `BridgeAdapter` (`_on_user`, `_on_complete`, `_record_delivery_failure`) and from the container's compliance-miss / classifier-decision points, each with `event_type` + a short human `text` so `_parse_history` renders them.
- **Reaction gating**: the executor already selects `REACTION_COMPLETE` vs `REACTION_SUCCESS` via `user_facing_routed` (session_executor.py:2001-2014). Extend that branch to fall through to `REACTION_ERROR` (or a neutral reaction) when `exit_reason` is non-clean. Keep this change minimal and well-tested — it is the highest-risk edit.

## Failure Path Test Strategy

### Exception Handling Coverage
- [ ] The transcript tailer MUST fail-silent on malformed/partial JSONL lines (the file is appended live; a half-written line is normal). Add a test feeding a transcript with a truncated final line and assert the tailer returns telemetry for the complete lines and logs at most a `debug`/`warning` — never raises.
- [ ] The tailer's periodic task MUST never crash the container run. Add a test where the transcript path does not exist (race: tailer ticks before the TUI writes its first line) and assert the tick is a no-op, not an exception.
- [ ] `_maybe_publish_exit_anomaly` and `_publish_exit_summary` already wrap saves in `_append_session_event` (fail-silent); add an assertion that an `exception` exit produces both an ERROR log and an `exit_anomaly` event.
- [ ] Existing `except Exception` blocks in `bridge_adapter.py` (`_append_session_event`, `_bump_last_turn_at`, `_deliver_sync`) — each already has fail-silent behavior; new tailer code follows the same contract and gets the same "raises → logged, run continues" test.

### Empty/Invalid Input Handling
- [ ] `read_transcript_telemetry` on an empty file → returns a zero/`None`-filled `TranscriptTelemetry`, not an exception.
- [ ] `read_transcript_telemetry` on a file with only non-telemetry lines (e.g., only user lines, no `usage`/`tool_use`) → counters `0`, `current_tool_name=None`.
- [ ] Tailer with both transcript paths `None` (PIDs/UUIDs never captured, e.g. spawn race) → no-op tick.

### Error State Rendering
- [ ] The `exit_reason` warning chip renders on a `completed`-status card when `exit_reason == "exception"` (test `_session_to_json` output + template assertion).
- [ ] The `communicated=False` chip renders when `user_facing_routed` is False on a completed session, and does NOT render when True.
- [ ] An `exit_anomaly` event renders as a readable feed row (not a `str(dict)` dump) — assert `_parse_history` produces a `PipelineEvent` with a human `text`.

## Rabbit Holes

- **Full token-cost parity with sdk_client.** sdk_client has elaborate per-model cost math. Do not reimplement it. Sum `usage` token fields from the JSONL and reuse any existing cost helper; if no trivial helper exists, populate token counts and leave `total_cost_usd` best-effort. Cost precision is not what the operator needed at 3am.
- **Live PTY-buffer scraping for `current_tool_name`.** Tempting to read the 73KB PTY buffer directly. Don't — the JSONL transcript already carries structured `tool_use` blocks with timestamps. Scraping ANSI-laden PTY text is a parser tar-pit.
- **Reworking the lifecycle status state machine.** The executor's `completed`/`failed` mapping is load-bearing across the whole system. Render the `exit_reason` warning in the dashboard layer; do not retrofit a new terminal status for granite exceptions.
- **A general-purpose transcript-recording subsystem.** That is epic #1536. Build only the dashboard-facing read here; resist designing the durable recording store.
- **Inotify/file-watching for the tailer.** A simple periodic re-read is sufficient for a 6h session at ~5s cadence. File-watch APIs across platforms are a portability rabbit hole.

## Risks

### Risk 1: Claude Code JSONL on-disk format/layout drifts
**Impact:** The tailer silently produces wrong/empty telemetry if Claude Code changes its transcript schema or the `~/.claude/projects/{slug}/` layout.
**Mitigation:** Fail-silent parsing that degrades to "hollow as today" rather than crashing. spike-1 pins the current slug algorithm and field names. A unit test fixture captures a real transcript snippet so a format change surfaces as a test failure, not a silent production regression.

### Risk 2: Tailer write volume / Redis pressure
**Impact:** A too-aggressive tick interval multiplies partial saves across every concurrent granite session.
**Mitigation:** spike-2 sets a conservative default (~5s) and only writes when a counter actually changed (diff-gate the save, like the stage-snapshot dedup in `_parse_history`). Bounded by the PTY pool size, so concurrency is already capped.

### Risk 3: Reaction-gating regression
**Impact:** Over-gating could suppress the legitimate success reaction for healthy sessions; under-gating leaves the misleading 👏 on failures.
**Mitigation:** This is the single highest-risk edit. It gets explicit table-driven tests for every `(exit_reason, user_facing_routed)` combination (see Test Impact). Keep the change to the existing reaction branch (session_executor.py:2001-2014); do not introduce a parallel path.

## Race Conditions

### Race 1: Tailer ticks before the TUI has written its transcript
**Location:** `agent/granite_container/transcript_tailer.py` (new) + the tailer driver in `bridge_adapter.py::run()`.
**Trigger:** The periodic tailer fires after the PTY spawns but before Claude Code has created/written its `{uuid}.jsonl` (or before the resume UUID is known).
**Data prerequisite:** The transcript path must be resolvable (UUID captured) before the tailer reads it.
**State prerequisite:** The JSONL file must exist and contain at least one complete line.
**Mitigation:** The tailer treats a missing file / unresolved path / empty file as a no-op tick (returns zero-telemetry, persists nothing). No exception, no partial-write of bogus zeros over real data — only write when telemetry is non-empty and changed.

### Race 2: Final tick vs. exit-summary write
**Location:** `bridge_adapter.py::run()` — tailer task cancellation vs. `_publish_exit_summary`.
**Trigger:** The container exits while a tailer tick is mid-read; the final telemetry and the exit summary race to save the same record.
**Data prerequisite:** The last telemetry snapshot should reflect the final transcript state.
**State prerequisite:** `exit_reason` and final counters must both land.
**Mitigation:** Cancel-and-await the tailer task before `_publish_exit_summary` runs, then do one final synchronous tailer read so the final counters are captured deterministically after the loop ends. Partial saves use disjoint `update_fields` sets (telemetry fields vs. `exit_reason`/identity), so even an interleave does not clobber.
