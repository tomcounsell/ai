# Dashboard telemetry parity for granite PTY sessions

## Problem

During live observation of a real production session (`ec1e7c6ede0b4cc491247067dea676f6`, 2026-06-12), every question the dashboard exists to answer — "is it making progress?", "what is it doing right now?", "did it succeed?" — required bypassing the dashboard and hand-reading `~/.claude/projects/*/*.jsonl` transcripts plus `logs/worker.log`. The dashboard actively misled: it showed 102 seconds of evidence, then went silent, while five more minutes of real work happened, and finally rendered an `exception` exit as a `completed` success with a 👏 reaction — for a session that never sent the human a single message and whose work product was force-deleted.

**Current behavior:**
The granite PTY container (production since #1572) drives PM + Dev interactive `claude` TUIs over pseudo-terminals, bypassing `agent/sdk_client.py` — the sole writer of per-session telemetry. As a result:

- `turn_count`, `tool_call_count`, all token/cost counters → `0`
- `current_tool_name`, `last_tool_use_at`, `recent_thinking_excerpt`, `last_stdout_at` → `null`
- `harness_pid`, `process_alive` → `null` (two live `claude` TUI PIDs the entire session)
- `events` → opaque rows; the `exit_summary`/`exit_anomaly` dicts use a `type` key with no `text`, so `ui/data/sdlc.py::_parse_history` renders them as raw `str(entry)` dumps
- `status` → `completed` even on `exit_reason: 'exception'`. This is verified, not hypothesized (see Spike Results / spike-3): `BridgeAdapter.run()` returns `""` and raises no error, so the executor's `final_status = "completed" if not task.error else "failed"` evaluates `task.error` as falsy and lands on `completed`. The granite `exit_reason` never reaches that mapping. The plan does NOT change this mapping — it surfaces `exit_reason` as a first-class field and renders the warning in the dashboard layer.
- `_maybe_publish_exit_anomaly` early-returns on `exception` — no ERROR log, no Sentry, no `exit_anomaly` event

The only working field, `last_turn_at`, works only because of the granite-specific `_bump_last_turn_at` `on_turn` shim — whose docstring explicitly notes the rest of the harness-path signals are dead on this path.

**Desired outcome:**
A granite session's dashboard card carries the same fidelity an operator gets from the SDK path — live counters, current activity, token totals, real PIDs, a first-class `exit_reason`, anomaly treatment for `exception` exits, routing-protocol events in the feed, and a visible flag when a session "completed" without ever talking to the human. An operator should never again have to drop to raw JSONL to answer the three core questions.

## Spike Results

The critique correctly flagged that spike-1 (transcript path/slug) and spike-3 (executor status path) were load-bearing and unresolved. All three spikes are now resolved against baseline HEAD and the findings are embedded below.

### spike-1: Transcript path derivation (slug algorithm + UUID source)
- **Assumption**: "The container can derive `~/.claude/projects/{cwd-slug}/{uuid}.jsonl` for both PM and Dev transcripts."
- **Method**: code-read + `claude --help` empirical verification on this machine.
- **Result (RESOLVED)**: Two findings, both verified empirically:
  1. **Slug algorithm**: Claude Code derives the per-project directory under `~/.claude/projects/` by replacing **every `/` in the absolute cwd with `-`** (leading slash → leading dash; all other characters — dots, hyphens, underscores — preserved verbatim). Verified examples on this machine: `/Users/tomcounsell/src/ai` → `-Users-tomcounsell-src-ai`; `/Users/tomcounsell/src/ai/.worktrees/sdlc-1648` → `-Users-tomcounsell-src-ai-.worktrees-sdlc-1648`. The transformation is a single `cwd.replace("/", "-")`.
  2. **UUID is set by US, not scraped**: `claude --session-id <uuid>` is a real flag ("Use a specific session ID for the conversation (must be a valid UUID)") and **works with the interactive TUI** (confirmed via `claude --help` exit 0). This eliminates the entire scrape-the-`--resume`-hint mechanism. The container generates two `uuid4()` values itself, passes one to each PTY spawn via `--session-id`, and the transcript filename is then **deterministically known at spawn time**: `~/.claude/projects/{slug(cwd)}/{that-uuid}.jsonl`. No discovery scan, no "most-recent file," no `isalive()`-gated `finally`.
- **Confidence**: high.
- **Impact if false (n/a — resolved)**: had `--session-id` not existed, the fallback would have been a startup discovery scan asserting exactly one new `{uuid}.jsonl` per PTY; the flag makes that unnecessary.

### spike-2: Tailer cadence
- **Assumption**: "A ~5s periodic re-read is a sane default cadence; byte-offset incremental reads keep it O(Δ) not O(n)."
- **Method**: code-read of the existing `_bump_last_turn_at` save cadence + PTY pool concurrency bound.
- **Result (RESOLVED)**: ~5s is confirmed sane. Concurrency is hard-bounded by `GRANITE_PTY_POOL_SIZE` (the pool semaphore), so total write volume is `pool_size` sessions × 1 diff-gated save per 5s — trivial Redis load. **The tailer MUST use byte-offset incremental reads, not a full re-read** (critique concern): tailer state is `(byte_offset, running_totals)`; per tick it `seek(byte_offset)`, reads to EOF, folds deltas into the totals, then `byte_offset = file.tell()`. This makes each tick O(Δbytes-since-last-tick), not O(file). Resolves Open Question #1 — periodic ~5s wins over the `on_turn` hook because `on_turn` only fires once per PM turn (counters would freeze during a multi-minute Dev edit burst), defeating the "what is it doing now?" question.
- **Confidence**: high.

### spike-3: Executor status mapping for granite exceptions
- **Assumption (the contradiction the critique flagged)**: Problem §said the executor maps a granite `exception` to `completed`; the old Technical Approach §line 145 claimed it "correctly maps non-clean exits to `failed`." These cannot both be true.
- **Method**: code-read traced end-to-end against baseline HEAD.
- **Result (RESOLVED — Problem § was correct; old line 145 was WRONG and has been struck)**: Traced: `BridgeAdapter.run()` (`bridge_adapter.py:200-254`) returns `""` and raises no error on any `exit_reason`, including `exception` (the exception is caught inside `Container.run()` at `container.py:828`, which sets `result.exit_reason="exception"` and returns a normal `ContainerResult` — it does NOT re-raise). Therefore in `session_executor.py:1860`, `task.error` is falsy and `final_status = "completed" if not task.error else "failed"` yields **`completed`**. The granite `exit_reason` is discarded by `run()` and never reaches the status mapping. **Conclusion**: the executor does NOT map granite exceptions to `failed`; it maps them to `completed`. The plan's approach is therefore correct and unchanged: persist `exit_reason` as a first-class field and render the warning in the dashboard — do NOT retrofit the status mapping (Rabbit Hole). The old line-145 self-contradiction is removed from Technical Approach.
- **Confidence**: high.
- **Impact**: confirms the dashboard-layer warning chip is the right surface; also drives the explicit two-branch render spec (failed vs completed-but-non-clean) added to the Solution to remove the latent ambiguity the Consistency critic raised.

## Data Flow

**Current (hollow) flow:**

1. **Entry point**: Worker executes a granite session → `BridgeAdapter.run()` → `Container.run()` in `asyncio.to_thread`.
2. **Container loop**: spawns 2 `claude` TUIs over PTYs; each TUI writes a complete JSONL transcript under `~/.claude/projects/{cwd-slug}/{uuid}.jsonl`. The loop accumulates `TurnRecord`s and byte counts into an in-memory `ContainerResult`.
3. **Exit**: `BridgeAdapter._publish_exit_summary` writes one `exit_summary` dict (with `exit_reason`, `turns`, `compliance_misses`) into `session_events`. `_maybe_publish_exit_anomaly` skips `exception`. `run()` returns `""`.
4. **Status**: executor sets `status = "completed"` (no `task.error`). `ContainerResult` is **discarded** — turn/tool/token counts, PIDs, current activity never reach the `AgentSession` record.
5. **Output**: `ui/app.py::_session_to_json` reads the `AgentSession`'s default-`0`/`null` fields → hollow card.

**Target (instrumented) flow:**

1. **Entry point** (unchanged): `BridgeAdapter.run()` → `Container.run()`.
2. **Container loop**: the container generates a `uuid4()` per PTY and passes it via `claude --session-id <uuid>` at spawn, so both transcript paths (`~/.claude/projects/{slug(cwd)}/{uuid}.jsonl`) are **deterministically known at spawn time** and exposed on the result alongside the two child PIDs. A lightweight **transcript tailer** (a periodic async task on the worker's event loop) reads the PM + Dev JSONL files **incrementally by byte offset** — each tick `seek(byte_offset)`, reads to EOF, folds deltas into `turn_count`, `tool_call_count`, token totals, `current_tool_name`, `last_tool_use_at`, `recent_thinking_excerpt`, then advances `byte_offset`. It persists via `await asyncio.to_thread(agent_session.save, update_fields=[...])` (never a blocking `.save()` directly in the coroutine), with a strictly disjoint `update_fields` set (excludes `updated_at` and `last_turn_at`).
3. **Exit**: `_publish_exit_summary` now also writes `exit_reason` onto a first-class `AgentSession.exit_reason` field and writes granite identity (PTY slot, both PIDs, both transcript paths) onto the record. `_maybe_publish_exit_anomaly` includes `exception`.
4. **Status / reaction**: a non-clean `exit_reason` renders as a warning state regardless of lifecycle status; the success reaction is gated on `exit_reason` + `user_facing_routed`.
5. **Output**: `_session_to_json` emits the new first-class fields; the card and modal render counters, identity links, the `exit_reason` warning, and a `communicated=False` chip.

## Solution

### Key Elements

- **Transcript tailer** (`agent/granite_container/transcript_tailer.py`): `read_transcript_telemetry(path, prev_state) -> TranscriptTelemetry` — **incremental, byte-offset-stateful**. Tailer state is `(byte_offset, running_totals)`; each call seeks to the prior offset, reads only new bytes to EOF, folds deltas via a separate `fold_events(events, totals)` core (the #1536 seam), and returns updated totals + the new offset. Fields: `turn_count`, `tool_call_count`, `total_input_tokens`, `total_output_tokens`, `total_cache_read_tokens`, `current_tool_name`, `last_tool_use_at`, `recent_thinking_excerpt`, `tailer_last_read_at`. Reads tolerant of partial/in-flight final lines (the file is appended live — a half-written trailing line is normal). MVP fields are `current_tool_name` + `last_tool_use_at` + the counters; `recent_thinking_excerpt` and token totals are stretch (parsing varies by model — see Nits).
- **Tailer driver in `BridgeAdapter`**: a periodic async task (`asyncio.create_task`, started in `run()`, cancel-then-final-read on container exit) that reads the PM + Dev transcripts every ~5s and persists aggregated telemetry. **Persists via `await asyncio.to_thread(agent_session.save, update_fields=[...])`** — never a blocking `.save()` in the coroutine body (the event-loop-safety of `_bump_last_turn_at` comes from its `to_thread` placement, not the call itself). The tailer's `update_fields` set is **strictly disjoint** — it excludes `updated_at` (owned by `_publish_exit_summary`) and `last_turn_at` (owned by `_bump_last_turn_at`) so concurrent per-field HSETs never collide (verified: Popoto `update_fields` does targeted per-field HSET, `popoto/models/base.py:137-147`).
- **Deterministic transcript paths + PID capture**: the container generates a `uuid4()` per PTY and passes it via `claude --session-id <uuid>` at spawn; each transcript path is computed as `~/.claude/projects/{cwd.replace("/", "-")}/{uuid}.jsonl` and is known the instant the PTY spawns (no scrape, no `finally`/`isalive()` gate, no "most-recent file" guess). `PTYDriver` exposes `pid`; `Container` records both transcript paths and both PIDs onto `ContainerResult`. The tailer matches **exactly** the file for its captured UUID; if that file is absent at exit, it logs a WARNING and writes nothing (never falls back to an arbitrary file).
- **Tailer self-observability**: `tailer_last_read_at` is written only on a successful parse yielding ≥1 event (piggybacked on the diff-gated save — no extra round-trip). A stale value on an active session (>2× the poll interval) is the signal the tailer has silently degraded — surfaced for a future health-check/reflection so operators can distinguish "really zero" from "tailer broken."
- **First-class `exit_reason`**: a nullable `AgentSession.exit_reason` field, written by `_publish_exit_summary`. The dashboard renders any non-clean reason as a warning chip even when `status == completed`. **Render rule is two-branch (resolves the latent ambiguity):** `status == "failed"` → failure UI (regardless of `exit_reason`); `status == "completed"` AND `exit_reason` non-clean → warning chip. This is robust no matter how the status mapping behaves.
- **`exception` as anomaly, severity-gated**: add `exception` to the `_maybe_publish_exit_anomaly` path, but **classify before logging** to avoid Sentry alert fatigue. An `exception_is_benign(result)` predicate evaluated BEFORE the log call downgrades soft exits to WARNING (no Sentry) and reserves ERROR+Sentry for hard ones: non-empty `result.turns` + recent activity ⇒ soft (WARNING); crashed before any output ⇒ hard (ERROR+Sentry). The `exit_anomaly` event is appended in both cases (the dashboard always surfaces it); only the log level / Sentry capture is gated.
- **Routing-protocol events**: the container's existing `[/user]`/`[/complete]` callbacks are appended to `session_events` as typed entries with a `text` + `event_type` so `_parse_history` renders them as readable feed rows (instead of opaque dumps). **Scope is narrowed to `[/user]`/`[/complete]` + delivery failures** — the `communicated=False` chip depends only on a routed-to-human boolean, not a classifier schema. Compliance-retry / classifier-decision typed events are **deferred to epic #1536** (its recording layer owns that taxonomy).
- **`communicated=False` completion chip**: the dashboard reads `user_facing_routed` (already on the record post-#1647) and renders a visible "no user message sent" chip on a completed session where it is False. **This chip ships now — it is a non-destructive signal.** The reaction change is gated separately (see reaction-gating below and Open Question #2).

### Flow

Operator opens dashboard → sees a running granite session card → card shows live `turn_count` / `tool_call_count` climbing, `current_tool_name` ("Edit"), token totals, and "PTY slot 1 · PM pid 12345 · Dev pid 12346" with both transcript file paths displayed (the identity link, an accepted requirement — see Success Criteria) → session ends on `exception` → card flips to a warning state ("exit: exception") with a red "no user message" chip (because `user_facing_routed` was False), an `exit_anomaly` event in the feed, and — because `exception` is a non-clean exit_reason — the post-run reaction is `REACTION_ERROR` instead of a green success. (A *clean* completion that simply didn't message the human still gets its normal success reaction plus the visible `communicated=False` chip — the chip informs without crying wolf.)

### Technical Approach

- **Set the transcript UUID ourselves; derive the path deterministically (spike-1 RESOLVED).** Claude Code names transcripts `~/.claude/projects/{cwd.replace("/","-")}/{session-uuid}.jsonl`. The container generates a `uuid4()` per PTY and passes it via `claude --session-id <uuid>` (a real flag that works with the interactive TUI — verified). The path is therefore fully known at spawn time. The old "scrape `last_resume_uuid()` from the dying PM in a `finally`-on-`isalive()`" mechanism is **abandoned** — it was the source of the blocker (dead on clean PM exit, Dev UUID never captured, collision risk on concurrent same-repo sessions). Each tailer is bound to exactly one captured UUID and reads exactly that file; an absent file at exit is a logged WARNING, not a fallback to an arbitrary file.
- **Tail incrementally off the event loop, persist via `to_thread` (spike-2 RESOLVED).** The persistence call mirrors `_bump_last_turn_at`'s `save(update_fields=[...])`, fail-silent — but it is wrapped in `await asyncio.to_thread(...)` because the tailer runs as an async task on the event loop (a direct blocking `.save()` would stall all coroutines every ~5s per session). Cadence default ~5s, hard-bounded by the PTY pool semaphore. Reads are byte-offset incremental (state = `(byte_offset, running_totals)`), so each tick is O(Δbytes), not O(file) — critical for multi-hour sessions. The tailer reads both transcripts each tick, folds counters into the running totals, picks the most recent `current_tool_name` / thinking excerpt across the pair, and writes only when a counter changed (diff-gate).
- **`exit_reason` first-class, status untouched (spike-3 RESOLVED — line-145 contradiction struck).** Verified: `BridgeAdapter.run()` returns `""` and raises no error on `exception` (the exception is caught in `Container.run()`, which returns a normal `ContainerResult`), so `session_executor.py:1860` evaluates `task.error` as falsy and sets `status = "completed"` — the granite `exit_reason` never reaches the mapping. The executor does **not** map granite exceptions to `failed`. We do **not** change the status mapping (Rabbit Hole: it is load-bearing system-wide). Instead, persist `exit_reason` onto the record and render the warning in the *dashboard* via the two-branch rule (status=failed → failure UI; status=completed + non-clean exit_reason → warning chip).
- **Routing events (scope narrowed):** append typed `session_events` from the existing `BridgeAdapter` callback sites (`_on_user`, `_on_complete`, `_record_delivery_failure`), each with `event_type` + a short human `text` so `_parse_history` renders them. Compliance-miss / classifier-decision typed events are **deferred to epic #1536** — the `communicated=False` chip needs only the routed-to-human boolean, which `user_facing_routed` already provides.
- **Reaction gating (Open Q #2 RESOLVED — ship the chip, defer the suppression).** The `communicated=False` chip is the visible, non-destructive signal and ships now. The **success-reaction change is deferred**: this slice does NOT suppress or downgrade the success reaction for clean-but-`communicated=False` completions, because many legitimate autonomous sessions (background/fire-and-forget builds) validly complete silently — flagging them as anomalies is the inverse of the original false-success bug and erodes trust in the reaction signal. The ONLY reaction change in this slice is the unambiguous one: extend the executor branch (`session_executor.py:2001-2014`) so a **non-clean `exit_reason`** (e.g. `exception`) selects `REACTION_ERROR` instead of a success reaction — i.e. escalate on a real failure signal, never on mere silence. The "clean + `communicated=False`" combination keeps its existing reaction unchanged. This keeps the highest-risk edit minimal and well-tested.

## Failure Path Test Strategy

### Exception Handling Coverage
- [ ] The transcript tailer MUST fail-silent on malformed/partial JSONL lines (the file is appended live; a half-written line is normal). Add a test feeding a transcript with a truncated final line and assert the tailer returns telemetry for the complete lines and logs at most a `debug`/`warning` — never raises.
- [ ] The tailer's periodic task MUST never crash the container run. Add a test where the transcript path does not exist (race: tailer ticks before the TUI writes its first line) and assert the tick is a no-op, not an exception.
- [ ] `_maybe_publish_exit_anomaly` and `_publish_exit_summary` already wrap saves in `_append_session_event` (fail-silent); add assertions that a *hard* `exception` exit produces an ERROR log + `exit_anomaly` event, and a *soft* `exception` exit produces a WARNING (no Sentry) + `exit_anomaly` event (`exception_is_benign` severity split).
- [ ] The tailer's byte-offset incremental read MUST not re-count earlier lines: feed a transcript, tick once, append more lines, tick again, and assert the second tick folds only the new lines' deltas (offset advanced, no double-count).

## Rabbit Holes

- **Full token-cost parity with sdk_client.** sdk_client has elaborate per-model cost math. Do not reimplement it. Sum `usage` token fields from the JSONL and reuse any existing cost helper; if no trivial helper exists, populate token counts and leave `total_cost_usd` best-effort. Cost precision is not what the operator needed at 3am.
- **Live PTY-buffer scraping for `current_tool_name`.** Tempting to read the 73KB PTY buffer directly. Don't — the JSONL transcript already carries structured `tool_use` blocks with timestamps. Scraping ANSI-laden PTY text is a parser tar-pit.
- **Reworking the lifecycle status state machine.** The executor's `completed`/`failed` mapping is load-bearing across the whole system. Render the `exit_reason` warning in the dashboard layer; do not retrofit a new terminal status for granite exceptions.
- **A general-purpose transcript-recording subsystem.** That is epic #1536. Build only the dashboard-facing read here; resist designing the durable recording store.
- **Inotify/file-watching for the tailer.** A simple periodic re-read is sufficient for a 6h session at ~5s cadence. File-watch APIs across platforms are a portability rabbit hole.

## Risks

### Risk 1: Claude Code JSONL on-disk format/layout drifts
**Impact:** The tailer silently produces wrong/empty telemetry if Claude Code changes its transcript schema or the `~/.claude/projects/{slug}/` layout.
**Mitigation:** Fail-silent parsing that degrades to "hollow as today" rather than crashing. spike-1 pinned the slug algorithm (`cwd.replace("/","-")`) and the `--session-id` UUID-set mechanism. A unit test fixture captures a real transcript snippet so a JSONL-schema change surfaces as a test failure, not a silent production regression. **`tailer_last_read_at` self-observability** (Solution) makes a slug/layout drift detectable in production: an active session whose `tailer_last_read_at` is stale by >2× the poll interval signals the tailer stopped reading — distinguishing "really zero work" from "tailer broken," which the pre-fix hollow state could not.

### Risk 2: Tailer write volume / Redis pressure
**Impact:** A too-aggressive tick interval multiplies partial saves across every concurrent granite session; an O(n) full re-read per tick compounds it over multi-hour files.
**Mitigation:** spike-2 confirmed a ~5s default and **byte-offset incremental reads** (each tick is O(Δbytes), not O(file)). The save is diff-gated (only when a counter changed) and persisted via `await asyncio.to_thread(...)` so it never blocks the event loop. Concurrency is hard-bounded by the PTY pool semaphore.

### Risk 3: Reaction-gating regression
**Impact:** Over-gating could suppress the legitimate success reaction for healthy autonomous sessions (the inverse of the false-success bug); under-gating leaves the misleading 👏 on real failures.
**Mitigation:** Scope is deliberately narrowed (Open Q #2 resolved): the ONLY reaction change is `non-clean exit_reason → REACTION_ERROR`. The "clean + `communicated=False`" combination keeps its existing reaction — the visible chip carries that signal non-destructively, so silent-but-successful autonomous sessions are never flagged as anomalies. This is still the highest-risk edit and gets explicit table-driven tests for every `(exit_reason, user_facing_routed)` combination (see Test Impact). Keep the change to the existing reaction branch (session_executor.py:2001-2014); do not introduce a parallel path.

## Race Conditions

### Race 1: Tailer ticks before the TUI has written its transcript
**Location:** `agent/granite_container/transcript_tailer.py` (new) + the tailer driver in `bridge_adapter.py::run()`.
**Trigger:** The periodic tailer fires after the PTY spawns but before Claude Code has created/written its `{uuid}.jsonl`.
**Data prerequisite:** The transcript path is known at spawn time (we set the UUID via `--session-id`), but the file itself is created lazily by Claude Code on first write.
**State prerequisite:** The JSONL file must exist and contain at least one complete line.
**Mitigation:** Because the UUID is set by us, there is no "unresolved path" race anymore — only a "file not yet created" race. The tailer treats a missing/empty file as a no-op tick (persists nothing, advances no offset). No exception, no partial-write of bogus zeros over real data — only write when telemetry is non-empty and changed.

### Race 2: Final tick vs. exit-summary write
**Location:** `bridge_adapter.py::run()` — tailer task cancellation vs. `_publish_exit_summary`.
**Trigger:** The container exits while a tailer tick is mid-read; the final telemetry and the exit summary race to save the same record.
**Data prerequisite:** The last telemetry snapshot should reflect the final transcript state.
**State prerequisite:** `exit_reason` and final counters must both land.
**Mitigation:** Cancel-and-await the tailer task before `_publish_exit_summary` runs, then do one final synchronous tailer read (seeking from the last byte offset to catch the tail written after the last async tick) so the final counters are captured deterministically after the loop ends. Partial saves use **strictly disjoint** `update_fields` sets: the tailer writes only telemetry fields and explicitly **excludes** `updated_at` (written by `_publish_exit_summary` as `["user_facing_routed","updated_at"]`) and `last_turn_at` (owned by `_bump_last_turn_at`). Verified safe: Popoto's `update_fields` path filters `encode_popoto_model_obj(self)` to the listed fields and issues a targeted per-field `HSET key field val` (`popoto/models/base.py:137-147`), so disjoint field sets never clobber — the only failure mode would be a shared field name across two writers, which the exclusion prevents.

## Resolved Questions

All three open questions are resolved inline (no remaining human input needed before build):

1. **Tailer cadence vs. push — RESOLVED: periodic ~5s, byte-offset incremental.** The `on_turn` boundary is too coarse (counters would freeze during a multi-minute Dev edit burst, defeating "what is it doing now?"). The ~5s periodic read with byte-offset incremental reads (O(Δbytes)/tick) and a diff-gated `to_thread` save is the chosen design (spike-2).
2. **Reaction gating — RESOLVED: ship the chip, defer the suppression.** A non-clean `exit_reason` (e.g. `exception`) selects `REACTION_ERROR`. A *clean but `communicated=False`* completion keeps its existing reaction — the visible `communicated=False` chip carries that signal non-destructively, so legitimate silent autonomous sessions are never flagged as anomalies (avoiding the inverse-of-false-success trap the User critic raised).
3. **Cost field — RESOLVED: best-effort/omitted is acceptable.** Token counts are populated this slice; `total_cost_usd` is best-effort (reuse any trivial existing cost helper, else leave it to epic #1536). Cost precision was not the operator's 3am need (Rabbit Hole).
