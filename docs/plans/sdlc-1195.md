# Continuation-PM creation crashes silently with session_id=None and working_dir=None

## Problem

The continuation-PM mechanism — the auto-spawned PM session that picks up where a finished dev session left off — has been silently dying on every spawn since at least 2026-04-14. Pipelines that cross any dev-session boundary (Plan→Build, Build→Test, Test→Review, Review→Merge) come to a dead halt unless a human pings the chat to spawn a fresh PM that rediscovers state from scratch.

**Current behavior:**

1. PM session dispatches a dev stage and exits.
2. Dev session completes → `_handle_dev_session_completion` calls `_create_continuation_pm` to spawn the follow-up PM.
3. `agent/session_completion.py:358` calls `_AgentSession.create(...)` **without** `session_id` or `working_dir`. Both fields are `None` on the new session.
4. Worker picks up the continuation PM and crashes immediately:
   - `agent/session_executor.py:634` — `Path(session.working_dir)` → `TypeError: argument should be a str or an os.PathLike object, not 'NoneType'`.
   - The crash-snapshot fallback runs `_session_branch_name(session.session_id).lower()` on a `None` `session_id` → `AttributeError`.
5. Pipeline dies with no Telegram message, no retry. The dispatched-children graph lives off the dead PM, so any subsequent human-triggered PM has to rediscover state via `gh pr list` and looks "confused."

**Desired outcome:**

The continuation PM is created with a valid `session_id` and `working_dir`, runs successfully, and advances the pipeline to the next stage without human intervention. If a future regression ever passes `None` for either field, the worker fails loudly at startup instead of dying silently mid-pipeline.

## Why Previous Fixes Failed

| Prior Fix | What It Did | Why It Failed / Was Incomplete |
|-----------|-------------|-------------------------------|
| PR #940 | Introduced `_create_continuation_pm` with `_AgentSession.create(...)` | Used the raw, permissive `create()` instead of the typed `create_pm` factory. The companion test (`tests/unit/test_continuation_pm.py`) only asserted `session_type`, `status`, `continuation_depth` — never the fields that were actually missing. So the bug shipped with green tests. |
| PR #1123 | Raised the dev-result preview cap from 500 to 10_000 chars | Touched `_create_continuation_pm` for an adjacent concern and did not notice the missing kwargs. Same blind-spot as the original tests. |
| PR #1051 | Refactored `agent_session_queue.py` into modules | Moved the function to a new file but preserved the bug verbatim. The `# noqa: F401` re-export kept the existing tests passing without surfacing the gap. |

**Root cause pattern:** Every previous attempt either (a) used the raw `_AgentSession.create(...)` instead of the typed `create_pm` factory that enforces required fields, or (b) wrote tests that only assert on the fields present in the call site, not on the fields that the executor will need at startup. The fields silently default to `None`, the model permits the save, and the bug only surfaces when the worker tries to use the session. Fixing the call site without also fixing the test gap and adding an executor guard would let the same shape of bug ship again.

## Solution

### Key Elements

- **Typed factory at the spawn site:** Switch `_create_continuation_pm` from raw `_AgentSession.create(...)` to `_AgentSession.create_pm(...)`, which enforces `session_id` and `working_dir` at the type level. Derive `session_id` as `{parent.session_id}_cont{new_depth}` so the chain is debuggable from the id alone. Resolve `working_dir` via the existing `_resolve_working_dir_for_parent(parent)` helper.
- **Executor guard with a clear error:** In `agent/session_executor.py` at line 634, replace `Path(session.working_dir)` with a guarded resolution that raises `ContinuationPMSpawnError` (or equivalent) with the offending session id and field names if either is `None`. The session is then marked `failed` via the existing failure path so it never sits in `pending` forever.
- **Test gap closure:** Update `tests/unit/test_continuation_pm.py` to assert non-`None` `session_id` and `working_dir` on every continuation. Add a test that exercises the executor guard with a synthetic `None` field. Add an integration test that runs Build→Test handoff end-to-end through the queue and asserts the continuation PM actually executes.

### Flow

Dev session completes → `_handle_dev_session_completion` decides the parent is terminal → `_create_continuation_pm(parent, ...)`:
- Resolves `working_dir = _resolve_working_dir_for_parent(parent)` (existing helper, falls back to `os.getcwd()` if no project config).
- Derives `new_session_id = f"{parent.session_id}_cont{new_depth}"`.
- Calls `_AgentSession.create_pm(session_id=new_session_id, working_dir=working_dir, project_key=parent.project_key, chat_id=parent.chat_id, telegram_message_id=0, message_text=message, ...)`.
- The created session has `status="pending"` and is picked up by the worker on the next queue tick. Executor reads `session.working_dir`, finds a real path, and the pipeline advances.

### Technical Approach

- **Spawn site (`agent/session_completion.py:358`):** Replace the raw `_AgentSession.create(...)` block with `_AgentSession.create_pm(...)` plus a `_resolve_working_dir_for_parent(parent)` lookup. The `create_pm` signature requires `session_id`, `project_key`, `working_dir`, `chat_id`, `telegram_message_id`, and `message_text`. The current call already has all of them in scope except `session_id` and `working_dir`. `telegram_message_id` is `0` for non-Telegram-originated continuations (the message_text carries the routing context).
- **Session-id format:** `{parent.session_id}_cont{new_depth}` (e.g. `0_1777387025168_cont1`). Underscores are safe under `sanitize_branch_name`; the chain depth is encoded in the id so debugging via `valor-session status --id ...` is one lookup.
- **Slug propagation:** **Do not inherit the parent's `slug`.** PM sessions don't take a worktree (the worker only resolves a worktree for the dev stage via `resolve_branch_for_stage`). If the continuation PM dispatches another dev stage, that dev session will create / reuse the worktree itself based on the issue context in `message_text` (the existing `(issues/N)` pattern). Reconciles Open Question 1.
- **Defensive validation in `_AgentSession.create()`:** **Skip.** The right defensive layer is the existing `create_pm` / `create_dev` / `create_child` factories, which already enforce required fields by Python signature. We will NOT add a runtime validator inside the raw `create()` because (a) it would create a second source of truth for "what's required for which session_type", (b) the executor guard already covers the runtime failure mode, and (c) `_create_continuation_pm` is the only call site in `agent/`, `bridge/`, and `worker/` that uses raw `_AgentSession.create(...)` to spawn a session — fixing this call site eliminates the pattern. Reconciles Open Question 2.
- **Executor guard (`agent/session_executor.py:634`):** Before `Path(session.working_dir)`, check `if session.working_dir is None or session.session_id is None:` → log a structured error including `session.id`, `session.session_type`, and `session.parent_agent_session_id`, then mark the session `failed` with a clear `failure_reason` like `"missing_working_dir_or_session_id"` and return. This converts the silent `TypeError` mid-startup into an observable failure that the existing reflection / dashboard surfaces will catch.

## Rabbit Holes

- **Rewriting `_AgentSession.create()` to validate required fields by `session_type`.** Tempting because it would prevent every future `None`-field bug at the data layer, but the typed `create_pm` / `create_dev` / `create_child` factories already cover this and adding a runtime validator in the base `create()` creates a second, easier-to-drift source of truth. Out of scope.
- **Migrating every `AgentSession.create(...)` call site to a typed factory.** There are non-spawn callers (e.g. test fixtures) that legitimately need the permissive form. A blanket migration is a separate refactor.
- **Adding a watchdog that resurrects continuation PMs that crashed before the executor guard ships.** Tempting because it would un-stick existing pipelines, but the right path is to ship the fix and let the next dev-session-completion event spawn a healthy continuation. Old broken sessions can be killed manually with `valor-session kill --all`.
- **Adding a "self-healing" path that retries the continuation with a default `working_dir`.** Defaults hide bugs. The executor guard fails the session loudly; the next continuation is healthy. Don't paper over the field-missing case.

## Risks

### Risk 1: The `create_pm` signature change blocks the fix from rolling out
**Impact:** If the continuation PM message lacks fields that `create_pm` requires (e.g. `telegram_message_id`), the spawn raises `TypeError` at runtime — same shape of failure as today, just at a different line.
**Mitigation:** `create_pm` accepts `telegram_message_id: int` (not Optional) but a value of `0` is fine for non-Telegram-originated continuations. Verified by reading `models/agent_session.py:1223-1252`. Test `TestCreateContinuationPM::test_creates_pm_session_with_correct_fields` exercises this path with `telegram_message_id=0`.

### Risk 2: The fix can't ship without itself working
**Impact:** This bug is meta — the very pipeline executing this PR depends on the continuation-PM mechanism that's broken. If the fix is shipped via the SDLC pipeline and the dev-stage completion can't hand off to a Test-stage continuation PM, the PR sits unreviewed (exactly what happened with PR #1194).
**Mitigation:** The fix is small enough to ship via a single dev session that completes both code + tests in one turn, then a human invokes `/do-merge` directly. The plan deliberately does NOT depend on cross-stage handoff. Acceptance criterion: "verified end-to-end on a fresh issue, not just unit-tested" is the post-merge validation, not a pre-merge gate.

### Risk 3: The executor guard regresses sessions that legitimately have `None` `working_dir`
**Impact:** If some code path creates a non-PM session with `working_dir=None` on purpose, the new guard would mark it `failed` instead of letting it proceed.
**Mitigation:** Grep confirms zero such call sites today. The typed factories all require `working_dir`. The guard is gating against a state that should be unreachable; if it ever fires, the structured error tells you exactly which spawn site to fix.

## Race Conditions

### Race 1: Existing dedup race (already mitigated, called out for completeness)
**Location:** `agent/session_completion.py:323-338` (Redis SETNX dedup on `continuation-pm:{parent_id}`)
**Trigger:** Two dev sessions complete simultaneously for the same parent.
**Data prerequisite:** Both invocations of `_handle_dev_session_completion` race to call `_create_continuation_pm`.
**State prerequisite:** Parent PM in terminal status.
**Mitigation:** Existing Redis SETNX with 300s TTL. Unchanged by this fix.

### Race 2: New continuation runs before parent is fully written to Redis
**Location:** `agent/session_completion.py:358` (the `create_pm` call) → worker queue tick.
**Trigger:** The continuation is created and enqueued; worker picks it up before `_resolve_working_dir_for_parent` can read `parent.project_config` because the parent's `project_config` save hasn't flushed yet.
**Data prerequisite:** `parent.project_config` must be populated when the continuation runs.
**State prerequisite:** Parent must be in terminal status (already enforced).
**Mitigation:** `_resolve_working_dir_for_parent` is called synchronously inside `_create_continuation_pm` (before the new session is saved), so the parent's `project_config` is read into a local variable first. The worker only sees the new session after `create_pm` returns. No async gap.

## No-Gos (Out of Scope)

- Adding `_AgentSession.create()` runtime validation for required fields (covered by typed factories already; see Rabbit Holes).
- Migrating other `_AgentSession.create(...)` call sites (none in `agent/`, `bridge/`, `worker/` for spawn purposes — verified).
- Resurrecting historical broken continuation sessions (kill with `valor-session kill --all`; the next live dev-completion will spawn a healthy continuation).
- Inheriting `slug` from parent on the continuation PM (PM sessions don't need worktree isolation; reconciled in Solution).

## Open Questions

Both open questions surfaced in the issue body have been reconciled in the Solution section:

1. **Should the continuation PM inherit the parent's `slug`?** — No. PM sessions don't take a worktree; the dev session that the continuation PM dispatches will resolve its own worktree from the issue context. Inheriting `slug` would couple the PM to a worktree it never enters.
2. **Should `_AgentSession.create()` validate required fields defensively?** — No. The typed `create_pm` / `create_dev` / `create_child` factories already enforce required fields by Python signature. Adding a runtime validator inside the raw `create()` would create a second source of truth for "what's required for which session_type" and drift over time. The executor guard covers the runtime failure mode for any future regression of the same shape.
