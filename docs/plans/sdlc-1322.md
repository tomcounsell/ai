# Size-Aware Media Download Timeout with Retry

## Problem

PR #1316 (issue #1297) introduced bridge-side media download with a hard-coded 10-second timeout. The cutoff is blind to file size and connection speed: a 20MB PDF on a 2 Mbps link needs ~80s and is killed at 10s; a 100KB voice note finishes in <1s but still carries the same overhead. Users get no retry — the worker receives a message with `media_download_error="timeout after 10s"` and no media description.

**Current behavior:**
At `bridge/telegram_bridge.py:1155-1158`:
```python
_local_path = await asyncio.wait_for(
    download_media(client, message, prefix=_media_type_for_download),
    timeout=10.0,
)
```
Any file that takes >10s to download fails permanently. No retry. Error string is fixed at `"timeout after 10s"`.

**Desired outcome:**
- Timeout scales with declared file size (small files stay fast, large files get proportional time, capped at 120s).
- One retry on timeout with 2x leash before giving up.
- Distinct error string after retry so downstream can tell "we tried and it's too big" from "first attempt timed out".
- Telemetry log line capturing size, computed timeout, duration, and retry count.

## Data Flow

1. **Entry point**: Telegram message arrives at the bridge handler in `bridge/telegram_bridge.py`.
2. **Intake**: Around line 1145, the bridge calls `get_media_type(message)` and inspects `message.media`.
3. **Size probe (new)**: Before calling `download_media`, read `message.file.size` via Telethon's `Message.file` accessor. Compute `timeout_for(size)`.
4. **Download attempt 1**: `asyncio.wait_for(download_media(...), timeout=computed)`. On `TimeoutError`, fall through to retry.
5. **Download attempt 2 (new)**: `asyncio.wait_for(download_media(...), timeout=computed * 2)`. Cap absolute total at 240s (120s * 2 worst case).
6. **Persist**: `TelegramMessage.media_local_path` (success) or `TelegramMessage.media_download_error` (terminal failure, distinct string for retried-and-still-failed). Telemetry log line emitted regardless.
7. **Output**: Worker reads the record, runs AI enrichment (or surfaces the error).

## Solution

### Key Elements

- **`compute_download_timeout(size_bytes: int | None) -> float`**: Pure helper in `bridge/media.py`. Returns `max(10.0, min(120.0, 5.0 + size_bytes / (1024*1024)))` when size is known, else 10.0.
- **Retry wrapper inline in `bridge/telegram_bridge.py`**: Two-attempt loop, second attempt at 2x timeout. Distinct error strings. Logs structured telemetry.
- **Distinct error strings**: `"timeout after Xs"` (single failure not retried — only used if retry is itself disabled) vs. `"timeout after Xs (retried)"` (terminal after second attempt). Per issue spec.

### Flow

Bridge intake (with media) -> probe `message.file.size` -> compute timeout -> attempt 1 (timeout T) -> on success, persist path. On TimeoutError, attempt 2 (timeout 2T) -> on success, persist path with retry telemetry. On second TimeoutError, persist `"timeout after Xs (retried)"` error.

### Technical Approach

- **Helper placement**: `compute_download_timeout` lives in `bridge/media.py` next to `download_media`. Pure function, no I/O — trivially unit-testable.
- **Size source**: `message.file.size` from Telethon. Telethon exposes `.file` on `Message` for both `MessageMediaPhoto` and `MessageMediaDocument`. For photos, `.file.size` may be None for some thumbnails — fall back to baseline 10s.
- **Retry placement**: Keep retry in `bridge/telegram_bridge.py` (the call site), not in `download_media` itself, because the retry decision is intake-policy and `download_media` is also called from `process_incoming_media` (test/ad-hoc paths) where retry behavior isn't needed.
- **Telemetry**: One `logger.info` line per download attempt with `chat_id`, `msg_id`, `media_type`, `size_bytes`, `computed_timeout_s`, `duration_ms`, `attempt` (1 or 2), `outcome` (success | timeout | error).
- **Cap**: Hard cap of 120s on the per-attempt timeout (issue spec). Worst-case wall time is 360s (120s + 240s if we doubled — but we cap second attempt at 240s explicitly, so worst is 360s). Actually, simpler: cap each attempt at 120s; second attempt timeout = `min(120.0, 2 * computed)`. Worst case: 240s wall time.

## Rabbit Holes

- **Progress-based stall detection (Option B in the issue)**: Wrapping Telethon's progress callback to cancel on N seconds of no bytes. Worth doing eventually, but doubles the surface area: requires a custom watcher coroutine and reasoning about Telethon's progress semantics across photo/document/voice types. Defer.
- **Connection-speed estimation**: Adapting timeout based on observed historical throughput. Premium accuracy, premium complexity. The issue's `5s + 1s/MB` formula is a deliberate punt on this — it's good enough.
- **Per-media-type timeouts**: Different formulas for voice vs. PDF vs. video. The size-based formula already handles this implicitly (voice is small, PDFs are big).
- **Configurable formula via env var**: Tunable knobs sound nice; in practice nobody tunes them. Hard-code the formula; if production data later shows the formula is wrong, change the constants in code.

## Risks

### Risk 1: `message.file.size` is None or wrong

**Impact:** Falls back to 10s baseline. Same behavior as today for that file. No regression.
**Mitigation:** The fallback is the existing behavior. Telemetry distinguishes `size_bytes=null` from `size_bytes=N` so we can spot-check production frequency.

### Risk 2: Retry doubles user wait time on truly dead connections

**Impact:** A user on a fully broken network waits up to 240s before the message reaches the agent.
**Mitigation:** The bridge intake path is async and doesn't block the user from sending the next message. The wait only delays *that message's* media enrichment. Telemetry will show retry rates; if >5% of downloads retry, we know the formula is too aggressive and tune the constants.

### Risk 3: Telethon's `download_media` doesn't actually honor `wait_for` cancellation cleanly

**Impact:** A cancelled download could leave a partial file in `data/media/`. The next attempt uses a different filename (timestamp+id), so no collision, but disk leaks accumulate.
**Mitigation:** Add a `try/finally` cleanup that unlinks the partial file if `_local_path` was assigned but the file is incomplete. Telethon already handles partial cleanup for many cases — this is belt-and-suspenders. Validate with the integration test.

## Race Conditions

No race conditions identified — all operations in the modified block are sequential within a single async task. The retry happens entirely inside one `await` chain. `TelegramMessage.save()` after the retry resolves is the single write point; no concurrent writers to the same record at that line.
