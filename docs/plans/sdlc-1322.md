# Size-aware media download timeout with retry

## Problem

PR #1316 (issue #1297) introduced bridge-side media download with a hard-coded 10-second timeout. The 10s cutoff is blind to file size, so larger files fail silently on slower connections.

**Current behavior:**
- A 20MB PDF on a slow connection needs far more than 10s; the download is killed at 10s and `media_download_error="timeout after 10s"` is recorded.
- A small voice message finishes well under 10s but still carries the same overhead.
- The user gets no retry — the message reaches the agent with no media description.

**Desired outcome:**
- Timeout scales with the expected file size, capped at a sensible ceiling.
- A progress watcher resets the deadline while bytes are still flowing, so slow-but-progressing downloads complete.
- On timeout, the bridge retries once with a longer leash before giving up.
- Telemetry captures size, computed timeout, actual duration, and retry count so the formula can be tuned from production data.

**Revised:**
- The issue's Option A formula (`5.0 + size_bytes / (1024*1024)`) assumes ~1 MB/s. We will use a more defensive baseline: `max(15.0, min(120.0, 10.0 + size_bytes / (512*1024)))` — 10s baseline + 1s per 512KB, min 15s, max 120s.
- We adopt a **hybrid** approach: size-aware timeout *plus* progress-based stall watcher (not "either/or"). Size sets the absolute deadline; a stall watcher (no bytes for 15s) cancels stuck-but-not-timed-out transfers.

## Data Flow

1. **Entry point**: Telegram message with media arrives at `bridge/telegram_bridge.py` event handler.
2. **Pre-download**: bridge reads `message.file.size` (when available). Computes a size-aware timeout via `compute_timeout(size_bytes)`.
3. **Download attempt 1**: `download_media(client, message, prefix, timeout=computed, stall_timeout=15.0, progress_telemetry=...)`. Internally, Telethon's `download_media` is wrapped with `asyncio.wait_for(timeout=computed)` AND a stall watcher that records last-progress-time and cancels the task if no progress for `stall_timeout`.
4. **On TimeoutError or stall**: log timeout details (size, computed timeout, elapsed). Retry once with `timeout=2 * computed` (still capped at 120s).
5. **On second failure**: write `media_download_error="timeout after Xs (retried)"` to `TelegramMessage`.
6. **On success**: write `media_local_path` to absolute path; emit a structured log with `file_size_bytes`, `computed_timeout`, `actual_duration_ms`, `retry_count=N`.
7. **Output**: worker downstream sees either `media_local_path` (success) or `media_download_error` (final failure) on the `TelegramMessage` record, exactly as today.

## Solution

### Key Elements

- **`compute_media_timeout(size_bytes: int | None) -> float`** (new helper, `bridge/media.py`): pure function returning `max(15.0, min(120.0, 10.0 + size_bytes / (512 * 1024)))` for known sizes, fallback `30.0` when `size_bytes is None`.
- **`download_media(...)` retry wrapper** (modified, `bridge/media.py`): accepts `timeout`, `stall_timeout`, and `progress_callback`. Runs Telethon's `client.download_media(..., progress_callback=...)` under `asyncio.wait_for(..., timeout=timeout)` with a stall watcher coroutine running concurrently.
- **Bridge call site** (modified, `bridge/telegram_bridge.py:1145-1184`): reads `message.file.size`, calls `compute_media_timeout`, attempts download, on timeout retries once with `2 * timeout` (capped at 120s), persists either `media_local_path` or a `"timeout after Xs (retried)"` error.
- **Structured telemetry**: a single `logger.info(...)` line per download with `file_size_bytes`, `computed_timeout`, `actual_duration_ms`, `retry_count`, `success: bool`, parseable from logs for tuning.

### Flow

Telegram media message arrives → bridge inspects `message.file.size` → computes timeout → attempts download with progress watcher → on success persist path; on timeout retry once with double the budget → on second timeout persist final error → telemetry logged in either case.

### Technical Approach

- Keep `compute_media_timeout` a pure function so it can be unit-tested without Telethon mocks.
- Use a small async helper `_download_with_stall_watch(client, message, filepath, timeout, stall_timeout)` inside `bridge/media.py` that owns the stall-detection logic; the existing `download_media()` becomes a thin wrapper that resolves filename + extension and delegates.
- Avoid changing the `TelegramMessage` schema. Retry count and computed timeout go into logs, not the database — they are operational telemetry, not user-facing state.
- Retry strictly **once**. No exponential backoff loop. Two attempts cap total wall time at ~360s worst case (120s + 120s overhead), which is acceptable given the bridge already runs ingestion off the main reply path.

## Failure Path Test Strategy

- [ ] The existing `except Exception as e:` block at `bridge/telegram_bridge.py:1165` swallows all download errors and writes `media_download_error`. ## Rabbit Holes

- **Per-media-type tuning** (different timeouts for photos vs. videos vs. documents). Tempting but premature. Ship the size-aware formula, gather telemetry, revisit with data.
- **Persisting `file_size_bytes` and `download_attempt_count` on `TelegramMessage`**. Adds schema surface for marginal benefit. Logs are sufficient for tuning.
- **Exponential backoff with N retries**. Two attempts is enough; more retries amplify worst-case latency without meaningfully improving success rate on flaky links.
- **Resumable downloads** (HTTP range requests). Telethon doesn't expose this cleanly; not worth the complexity for first-pass timeout fix.

## Risks

### Risk 1: `message.file.size` unavailable on some media types
**Impact:** `compute_media_timeout(None)` falls back to 30s. Slightly more generous than today's 10s, so worst case we are *less* aggressive about timing out. Acceptable.
**Mitigation:** unit test the `None` path; log when size is missing so we can audit frequency.

### Risk 2: Long-tail very-large files (>60MB) still time out
**Impact:** A 100MB video on a 1 MB/s link needs ~100s; computed timeout is capped at 120s, which covers it. A 200MB file would not. We accept that as out of scope for this iteration — the user message reaches the agent with `media_download_error`, which is no worse than today.
**Mitigation:** telemetry will surface the frequency of `>120s` cases; if non-trivial, raise the cap or add a separate "huge file" path in a follow-up.

### Risk 3: Stall watcher false-positives on legitimately slow links
**Impact:** A genuinely slow but progressing download could hit the 15s no-progress window during a brief pause and get cancelled.
**Mitigation:** stall watcher resets on every `progress_callback` invocation; Telethon fires this frequently enough that 15s without any progress is a reasonable signal of a stuck transfer.

## Race Conditions

### Race 1: Stall watcher and `wait_for` racing on cancellation
**Location:** `bridge/media.py` `_download_with_stall_watch` (new helper)
**Trigger:** Both the outer `asyncio.wait_for(timeout=computed)` and the inner stall watcher could attempt to cancel the download task simultaneously.
**Data prerequisite:** download task and stall watcher task share a `last_progress_at` timestamp.
**State prerequisite:** at most one cancellation should fire; the second should be a no-op.
**Mitigation:** wrap the download in a single `asyncio.Task`; the stall watcher cancels via `task.cancel()` and `wait_for` cancels via timeout — `Task.cancel()` is idempotent in asyncio, so racing cancels are safe. The helper catches `CancelledError` and re-raises as `TimeoutError` so the caller sees a uniform exception type.
