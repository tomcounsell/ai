# Infrastructure Tooling: Dynamic MCP Server Loading and Multi-Machine PM-Overlay Drift Check

Closes #1396 (consolidated from #974 and #1337).

## Problem

**Static MCP loading (#974):**
`config/mcp_library.json` lists 11 available MCP servers. All of them load at every session start via `.claude/settings.json`, consuming context window tokens for tools the session will never use (Sentry in a planning session, Notion in a hotfix session). As the MCP catalog grows, this becomes an increasingly expensive tax on every session context window.

**PM-overlay drift (#1337):**
The `/update` skill (`scripts/remote-update.sh`, `.claude/skills/update/`) propagates code and config to bridge machines. One step — diffing the in-repo PM persona template against the private per-machine overlay at `~/Desktop/Valor/personas/project-manager.md` — was deferred as an "operator manual step" during the pm-workflow-announcement plan (#1325 audit). Today, drift accumulates silently across machines: a persona improvement committed to the repo may never reach the overlay, or a local overlay edit may silently diverge from the template.

**Desired outcome:**
1. Sessions load only the MCP servers relevant to their task context (always-on core set + session-type/stage additions).
2. `/update` emits a visible drift warning when the in-repo PM persona template and the private overlay have diverged, so operators know without having to remember to check.

## Research

**How MCP servers are currently activated:**
Claude Code reads MCP server configuration from `.claude/settings.json` (`mcpServers` key) and from the user-level `~/.claude/settings.json`. The `config/mcp_library.json` is a catalog/registry that the agent can query but is NOT automatically read by Claude Code to activate servers. MCP servers are activated by entries in `settings.json`. This means dynamic MCP loading requires either: (a) generating a per-session `.mcp.json` or `settings.json` fragment before session start, or (b) passing MCP server configs as CLI flags to `claude`. The SDK client (`agent/sdk_client.py`) is the session launch point and can inject these.

**Persona overlay location:**
`~/Desktop/Valor/personas/project-manager.md` (iCloud-synced, private vault). The in-repo template lives at `config/personas/segments/`. The diff is between the assembled in-repo template and the private overlay. An exact diff comparison is appropriate; section-level diff is more useful than a raw character diff.

## Data Flow

### Phase 1 — Dynamic MCP Loading

1. **Session creation**: `agent/sdk_client.py` determines `(session_type, current_stage, project_key)`.
2. **Classification**: new `_classify_mcp_categories(session_type, stage)` function maps to a set of MCP categories: `core` (always), `planning` (PLAN/CRITIQUE stages), `comms` (PM/TEAMMATE sessions), `observability` (BUILD/TEST/REVIEW stages), `browser` (BUILD/TEST/REVIEW stages).
3. **Server selection**: filter `config/mcp_library.json` servers by matching category. Always include `core` category.
4. **Config injection**: pass selected MCP server configs to the Claude Code harness invocation. Exact mechanism depends on how the harness accepts per-session MCP overrides (CLI flag or temp config file — to be determined during implementation spike).
5. **Override**: `--mcp-categories` flag on `valor-session create` allows manual category override.

### Phase 2 — PM-Overlay Drift Check

1. **Trigger**: `/update` skill step in `scripts/update/run.py`, after iCloud sync wait.
2. **Load in-repo template**: read `config/personas/segments/` (assembled template).
3. **Load private overlay**: read `~/Desktop/Valor/personas/project-manager.md`.
4. **Diff**: compute section-level diff between the two files.
5. **Output**: if diff is non-empty, append to `result.warnings` (same pattern as lines 922–966) and emit a colored CLI warning block listing diverged sections.
6. **No auto-merge**: drift may be intentional. Surface only — do not modify either file.

## Architectural Impact

**Phase 1:**
- **New function**: `_classify_mcp_categories(session_type, stage) -> set[str]` in `agent/sdk_client.py`.
- **Modified launch path**: harness invocation in `sdk_client.py` injects per-session MCP server subset.
- **New flag**: `--mcp-categories` on `valor-session create` CLI (`tools/valor_session/__main__.py`).
- **Config read**: `config/mcp_library.json` read at session start (currently only read by agents manually; now also read programmatically at session launch).
- **Coupling**: `sdk_client.py` gains a dependency on `config/mcp_library.json` format. If the library format changes, the classifier needs updating.

**Phase 2:**
- **Modified step**: `scripts/update/run.py` gains a new drift-check step following the existing MCP drift pattern at lines 922–966.
- **No new dependencies**: uses `difflib` (stdlib) or simple line comparison.
- **Idempotent**: diff of identical files produces empty output, no warning emitted.

**Reversibility:**
- Phase 1: remove the classifier and revert to loading all servers. No data migration.
- Phase 2: remove the drift check step from `run.py`. No state.

## Failure Path Test Strategy

### Phase 1 Exception Handling
- `config/mcp_library.json` missing or malformed: fall back to loading all servers (fail-open, same behavior as today). Log WARNING.
- Unknown `session_type` or `stage`: default to `core` only (conservative, no crash).
- MCP server injection fails at harness level: log WARNING, proceed without that server — do NOT crash the session start.

### Phase 2 Exception Handling
- Private overlay file absent (`~/Desktop/Valor/personas/project-manager.md` doesn't exist): skip check silently — this is normal for a fresh machine before the overlay has been configured.
- Template directory absent (`config/personas/segments/`): log WARNING, skip check.
- Diff computation raises: catch, log WARNING, treat as no drift (fail-safe).
- All failures are non-fatal to the `/update` flow — drift check is informational only.

### Empty/Invalid Input Handling
- Phase 1: empty category set after filtering → load `core` only.
- Phase 2: both files identical → no warning (test this explicitly to confirm idempotency).

## Rabbit Holes

- **Don't build a per-message MCP selector**: message content analysis to decide which MCPs to load adds latency and unpredictability. Session-type + stage is sufficient signal and is known at session start.
- **Don't auto-sync the PM overlay**: the overlay is intentionally private. Surfacing drift is the right boundary; merging crosses it.
- **Don't redesign `mcp_library.json` format**: the existing schema works. Phase 1 reads it; don't restructure it.
- **Don't add a MCP management UI to the dashboard**: out of scope. The YAML/JSON config is the management surface.
- **Don't block session start on drift**: Phase 1 MCP injection failures must be non-fatal. A session with all servers is better than no session.
