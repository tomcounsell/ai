# PM routes 'complete' after intermediate pipeline stages — ledger-aware completion guard

## Problem

The headless PM (project-manager) session repeatedly finalizes the SDLC run as
`completed` while the pipeline is still mid-flight. Five documented instances
across three sessions (issue #2158 + comments):

1. #2147, 10:06 UTC — routed `complete` two minutes after a **NEEDS REVISION**
   critique verdict, plan doc left uncommitted, ledger stuck at `CRITIQUE`.
2. #2147, 10:25 UTC — after the operator resume it revised the plan and recorded
   `CRITIQUE: completed`, then routed `complete` again with `BUILD: ready`, no
   branch commits, no PR.
3. #2147, 15:48 UTC — resumed with an explicit "do not route complete before
   MERGE" instruction, still routed `complete` with `TEST: ready`, no PR.
4. #2146 (BRAND-NEW session, fresh transcript, dispatch message explicitly
   citing this bug) — routed `complete` after only the PLAN stage, 7 stages
   remaining, `CRITIQUE: ready`.

Datapoint 4 is decisive: the fresh-transcript hypothesis is refuted. The defect
is systemic — the PM reads "the stage I just executed is finished" as "the
session is finished," regardless of ledger state, explicit steering, or
transcript history.

## Root Cause

There are **two distinct completion decision paths**, and the deterministic
router is NOT the one that fails:

- The deterministic router `agent/sdlc_router.py::decide_next_dispatch()`
  (lines 1593-1665) **never emits a "complete" route** — its terminal outcome is
  a `Dispatch(/do-merge)` (row 10, `_rule_ready_to_merge`, lines 1359-1374). It
  already maps **NEEDS REVISION → `/do-plan`** (G1 `guard_g1_critique_loop`
  lines 273-305; row 3 `_rule_critique_needs_revision` lines 786-804; the
  verdict-kind staleness gate `_critique_verdict_is_stale` lines 973-1069) and
  never lets a revision-requiring verdict reach a terminal dispatch.

- The PM's literal `route: "complete"` bypasses the router entirely. It is a
  schema/prompt decision: `PM_TURN_JSON_SCHEMA` route enum
  (`agent/session_runner/router.py:61-69`), classified verbatim by
  `validate_structured_route()` (lines 127-164), and **honored with no ledger
  check** at `agent/session_runner/runner.py::_route_turn` (lines 1334-1340):

  ```python
  if classification.destination == "complete":
      payload = classification.payload or ""
      if payload:
          self._adapter.on_complete_payload(payload, classification.file_paths)
      return _RouteDecision(should_break=True, exit_reason=ExitReason.PM_COMPLETE, ...)
  ```

  The prompt guidance at `.claude/commands/roles/prime-pm-role.md:33` tells the
  PM to route `complete` "when the task is finished" — with no mention of the
  SDLC ledger's terminal state. The `_run_wrapup_guard` path
  (`runner.py:1378-1382`) has the same unguarded `complete` acceptance.

**The gap:** `_route_turn` honors a PM `complete` route without consulting the
per-issue SDLC ledger. An already-correct, ledger-terminal-gated predicate
already exists for the bridge path — `agent/pipeline_complete.py::is_pipeline_complete`
(lines 35-86) — but the headless runner does not consult it.

## Prior Art

- `agent/pipeline_complete.py::is_pipeline_complete` — the pure, ledger-terminal
  predicate already used by the bridge completion path
  (`agent/session_completion.py`). Terminal iff `MERGE == "completed"`, or
  `DOCS == "completed"` with no open PR. Reused here so the headless runner and
  the bridge agree on "terminal."
- `agent/sdlc_router.py::decide_next_dispatch` — the router already refuses to
  advance a NEEDS REVISION verdict past `/do-plan`. The reroute nudge quotes its
  answer so criterion (a) and (b) share one source of truth.
- `tools/sdlc_stage_query.py::query_enriched(issue_number=…)` — returns
  `{stages, _meta}` (pr_number, latest_critique_verdict, …). The guard's ledger
  read.
- The G-guard bounded-escalation pattern in `agent/sdlc_router.py` (G2/G4/G7):
  refuse a bounded number of times, then escalate to a human rather than loop
  forever. The completion guard mirrors this.

## Solution

### Key Elements

1. **New pure predicate module `agent/session_runner/completion_guard.py`.**
   A stateless function plus a small result dataclass:

   ```python
   @dataclass(frozen=True)
   class CompletionDecision:
       allow: bool
       reason: str                 # machine-readable: not_sdlc | terminal | blocked_reason_given | refused_non_terminal | escalate_exhausted | query_failed_fail_open
       reroute_message: str | None # corrective nudge fed back as `continue`, when refusing
       escalate_to_user: bool      # refusals exhausted → surface to the human

   def evaluate_completion(
       *, session_type: str | None, issue_number: int | None,
       stage_states: dict, meta: dict, blocked_reason: str | None,
       refusal_count: int, next_skill: str | None,
   ) -> CompletionDecision: ...
   ```

   Decision logic (in order):
   - `session_type != "eng"` or `issue_number is None` → **ALLOW** (`not_sdlc`).
     Teammate and ad-hoc conversational sessions are unaffected.
   - Ledger terminal (`is_pipeline_complete(stage_states, "success")` returns
     `(True, …)`, i.e. `MERGE == "completed"` or docs-only terminal) → **ALLOW**
     (`terminal`).
   - `blocked_reason` present and non-empty → **ALLOW** (`blocked_reason_given`);
     the guard records the blocked reason for telemetry / exit message.
   - Non-terminal, no blocked reason, `refusal_count < MAX_COMPLETION_REFUSALS`
     → **REFUSE**: `allow=False`, `reroute_message` = a corrective nudge that
     names the router's next dispatch (`next_skill`, e.g. `/do-plan` for a NEEDS
     REVISION verdict) and instructs the PM to continue the pipeline or supply a
     `blocked_reason`.
   - Non-terminal, no blocked reason, refusals exhausted → `allow=False`,
     `escalate_to_user=True` (`escalate_exhausted`): the runner surfaces the
     incomplete state to the human (never silently completes, never loops to
     max_turns).

2. **Optional `blocked_reason` field on the PM turn schema**
   (`agent/session_runner/router.py`). Additive: add
   `"blocked_reason": {"type": "string"}` to `PM_TURN_JSON_SCHEMA["properties"]`
   (NOT to `required`), add `blocked_reason: str | None = None` to
   `ClassificationResult`, and populate it in `validate_structured_route`
   (`classify_pm_prefix` fallback leaves it `None`). This gives the PM a
   structured, non-keyword escape hatch to complete a genuinely abandoned /
   superseded pipeline — no keyword matching (per CLAUDE.md principle 3).

3. **Runner wiring (`agent/session_runner/runner.py`).**
   - A private helper `_load_ledger(issue_number) -> (stage_states, meta,
     next_skill)` that lazily imports `tools.sdlc_stage_query.query_enriched`
     and `agent.sdlc_router.decide_next_dispatch`, and **fail-opens** (any
     exception → return a sentinel that makes the guard ALLOW, reason
     `query_failed_fail_open`). A broken ledger query must never wedge a session.
   - In `_route_turn`, before honoring `destination == "complete"`, call
     `evaluate_completion(...)`. On REFUSE → return
     `_RouteDecision(should_break=False, next_message=reroute_message)` (continue
     the loop). On escalate → deliver to the human
     (`ExitReason.PM_NEEDS_HUMAN`). On ALLOW → the existing complete path.
   - Track `self._completion_refusal_count` (incremented on each refusal). Emit
     a `completion_refused` telemetry event (observable, never silent) via the
     existing `_record_telemetry`.
   - Mirror the guard in `_run_wrapup_guard`'s `destination == "complete"`
     branch so the wrapup path cannot silently complete a non-terminal ledger
     either.

4. **PM prompt (`.claude/commands/roles/prime-pm-role.md`).** Tighten the
   `route: "complete"` bullet: for an SDLC pipeline, `complete` is valid only
   when the pipeline is terminal (PR merged / `MERGE` recorded) OR when supplying
   a `blocked_reason`. Document the new optional `blocked_reason` schema field.

### Flow

```
PM turn → validate_structured_route → ClassificationResult(destination="complete", blocked_reason?)
   → _route_turn: if destination == "complete":
        stage_states, meta, next_skill = _load_ledger(issue_number)   # fail-open
        decision = evaluate_completion(session_type, issue_number, stage_states, meta,
                                       blocked_reason, refusal_count, next_skill)
        if decision.allow:            → existing complete path (PM_COMPLETE)
        elif decision.escalate_to_user: → deliver to human (PM_NEEDS_HUMAN)
        else:                          → refusal_count += 1; continue with reroute_message
```

### Technical Approach

- The guard is a **pure function** (all I/O in `_load_ledger`), so criterion (c)
  tests exercise decision logic with hand-built `stage_states`/`meta` dicts — no
  Redis, no subprocess.
- Reuse `is_pipeline_complete` for the terminal check so the headless runner and
  the bridge share one terminal definition (no divergence).
- `MAX_COMPLETION_REFUSALS` (default 3) is a module constant next to the guard,
  mirroring `MAX_SAME_STAGE_DISPATCHES` in the router.
- The reroute nudge text derives its next-stage name from
  `decide_next_dispatch` (`next_skill`), so a NEEDS REVISION verdict yields a
  nudge that literally says "dispatch `/do-plan`" — binding criterion (a) to the
  reroute path.

## No-Gos (Out of Scope)

- Rewriting or re-tuning `agent/sdlc_router.py`'s dispatch guards. The router
  already maps NEEDS REVISION → `/do-plan` correctly; this plan only adds a
  regression lock and a runner-side completion guard that consults it.
- Changing the bridge completion path (`agent/output_router.py`,
  `agent/session_completion.py`). It already uses `is_pipeline_complete`; this
  plan brings the headless runner to parity, not the reverse.
- Keyword-matching the PM's free-text `message` to infer "blocked." The escape
  hatch is the structured `blocked_reason` schema field only.
- Removing or altering the `route: "continue"`/`user` semantics. Only the
  `complete` route is gated.

non-terminal ledger and assert it continues rather than completing.

## Failure Path Test Strategy

### Exception Handling Coverage
- `_load_ledger` raising (Redis down, `query_enriched` throws, subprocess in
  `is_pipeline_complete`'s `_check_pr_open` times out) → guard returns ALLOW
  (`query_failed_fail_open`); assert the session is NOT wedged (complete is
  honored). Test injects a `query_enriched` that raises.

### Empty/Invalid Input Handling
- `issue_number is None` (unslugged eng or ad-hoc session) → ALLOW (`not_sdlc`).
- Empty/whitespace `blocked_reason` → treated as absent (does NOT unlock a
  non-terminal completion).
- Malformed `stage_states` (missing keys) → `is_pipeline_complete` returns
  non-terminal; guard refuses (fail-safe toward not-completing, which the
  bounded refusal ladder then escalates rather than looping forever).

### Error State Rendering
- Refusals-exhausted path (`escalate_to_user=True`) → the human receives the
  PM's incomplete-state message via `PM_NEEDS_HUMAN`, never a silent completion
  and never a max_turns hang.
- Every refusal emits a `completion_refused` telemetry event; assert it is
  recorded.

## Rabbit Holes

- **Do NOT reimplement terminality inside the runner.** Reuse
  `agent/pipeline_complete.py::is_pipeline_complete` verbatim so the headless and
  bridge paths cannot diverge.
- **Do NOT make the completion guard consult the LLM or parse the PM's prose.**
  It is a pure decision over ledger state + the structured `blocked_reason`
  field. No keyword matching.
- **Do NOT let a ledger-query failure block completion.** Fail-open is mandatory:
  a Redis/subprocess hiccup must never trap a session in a refuse-loop.
- **Do NOT add an unbounded refuse loop.** The `MAX_COMPLETION_REFUSALS` ladder
  must escalate to the human; a PM that keeps insisting on `complete` on a
  non-terminal ledger surfaces to a person, it does not spin to max_turns.
- **Do NOT touch the router's guard set (G1–G8) or the dispatch table.** They are
  correct for this bug; the fix is the missing runner-side consumer, not the
  producer.

## Success Criteria

- A headless eng PM session that routes `complete` with `stage_states` showing
  any of `{CRITIQUE, BUILD, TEST, REVIEW, DOCS}` not completed and no merged PR
  is refused and continued (or escalated after the bounded ladder), never
  finalized `completed`.
- A NEEDS REVISION critique verdict produces a reroute nudge that names
  `/do-plan`; the router's own decision for that state is `/do-plan` (regression
  locked).
- A genuinely terminal ledger (`MERGE == completed`) still completes on the
  first `complete` route.
- A non-eng / unslugged session's `complete` route is unaffected.
- `python -m ruff check` and the targeted unit tests pass under `-n0`.
