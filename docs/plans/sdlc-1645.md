# AgentSession.updated_at stamped in the future (naive local datetime via popoto auto_now)

## Problem

On any machine whose local timezone is not UTC, every `AgentSession` save stamps
`updated_at` with **naive local wall time** that downstream code interprets as UTC.
On a UTC+7 host this puts `updated_at` ~7 hours in the future of the real moment.

**Current behavior (observed live, 2026-06-12, UTC+7 host):** session
`ec1e7c6ede0b4cc491247067dea676f6` had `created_at` epoch `1781261380` (10:49:40 UTC,
correct — set explicitly tz-aware) but `updated_at` epoch `1781286582` (17:49:42 UTC),
7h in the future. `valor-session status` then printed `Updated: 2026-06-12 17:49:42 UTC`
— local time mislabeled as UTC. A 3-minute-old running session looked 7h stale.

**Root cause (confirmed by source inspection):**
1. `models/agent_session.py:153` — `updated_at = DatetimeField(auto_now=True, null=True)`.
2. popoto's `DatetimeField.format_value_pre_save` (`.venv/.../popoto/fields/datetime_field.py:116,118`)
   returns bare `datetime.now()` — **naive local wall time** — for both `auto_now` and `auto_now_add`.
3. popoto's encoder (`.venv/.../popoto/models/encoding.py:87-96`) serializes datetimes via
   `obj.strftime("%Y%m%dT%H:%M:%S.%f")`, which **strips tzinfo entirely**. So the wall-clock
   *value* is what gets persisted; whatever tzinfo was attached is discarded.
4. Downstream defensive code (`bridge/utc.py::to_unix_ts`, `ui/data/sdlc.py::_safe_float`,
   `monitoring/session_watchdog.py::_to_timestamp`) re-attaches UTC to naive datetimes. That
   re-attachment is **correct only when the stored wall-clock value is UTC**. Because the
   producer wrote local wall time, re-attaching UTC bakes in the offset rather than fixing it.

So the defensive consumer layer is sound; the defect is purely at the **producer**: the value
written must be UTC wall-clock. This is the same bug class as closed #777, which fixed one
*consumer* (`_to_timestamp`) but never the *producer* that mints the bad value.

**Desired outcome:** `updated_at` always reflects true UTC wall-clock time, so every
age/staleness/sort consumer — and the human-facing `valor-session status` line — is truthful
on every machine regardless of local timezone. Existing future-dated records are healed once.

## Why Previous Fixes Failed

| Prior Fix | What It Did | Why It Failed / Was Incomplete |
|-----------|-------------|-------------------------------|
| PR #787 (#777) | Made the watchdog `_to_timestamp` treat naive datetimes as UTC | Patched one consumer. The producer still mints naive *local* time, so the value re-interpreted as UTC is wrong by the offset — every other consumer (and any new one) inherits the skew. |

**Root cause pattern:** Fixes were applied at the *consumer/display* layer, treating the symptom (a naive datetime arriving) rather than the *producer* layer (the naive-local value being minted). Defensive UTC-reattachment is only correct if the stored wall-clock is already UTC — which `auto_now` violates.

## Data Flow

1. **Entry point**: any code path that calls `AgentSession.save()` (queue pickup, status transitions, completion, heartbeat — e.g. `models/agent_session.py:1301/1392/1440/1573`).
2. **popoto save pipeline**: `DatetimeField.format_value_pre_save` sees `auto_now=True`, returns naive `datetime.now()` (local wall time) — *overwriting* whatever `updated_at` held.
3. **popoto encoder**: serializes via `strftime` → tzinfo stripped → naive local wall-clock string persisted to Redis.
4. **Read-back**: `_normalize_kwargs` / `__setattr__` re-attach `tz=UTC` to the naive value (treating local wall-clock as if UTC) → in-memory datetime is "UTC" but offset-wrong.
5. **Consumers**: `to_unix_ts` / `_safe_float` / `_to_timestamp` convert to epoch; `valor-session status` formats and labels "UTC". All inherit the offset.
6. **Output**: ages/staleness skewed by the host UTC offset; `valor-session status` prints a future "UTC" time.

**The fix moves the correction to step 2/3**: stamp UTC wall-clock at write time so steps 4-6 (which already treat naive-as-UTC) become correct.

## Architectural Impact

- **New dependencies**: none. Uses existing `bridge/utc.py::utc_now()`.
- **Interface changes**: `updated_at` stops being popoto-managed (`auto_now` dropped); it becomes an explicitly-stamped field, set in a single choke point so every `save()` still updates it.

- **Reversibility**: high. The change is localized to `models/agent_session.py` plus a one-shot heal. Revert = re-add `auto_now=True` and drop the explicit stamp.

## Solution

### Key Elements

- **Explicit UTC stamp at the save choke point**: drop `auto_now=True` from `updated_at`; stamp `updated_at = utc_now()` in a single override of `AgentSession.save()` so every save path updates it with a correct UTC wall-clock value. The encoder strips tzinfo, so what persists is the *correct UTC wall-clock* — and the existing read-back re-attachment (`tz=UTC`) is then truthful.
- **One-shot heal for existing future-dated records**: a `_heal_future_updated_at` classmethod that, for each AgentSession whose `updated_at` is in the future, clamps it to `max(created_at, started_at_or_created, now)` (never below `created_at`), saving with `skip_auto_now=True`-equivalent semantics (i.e. without re-triggering the producer bug). Invoked once at worker startup, idempotent (records already sane are skipped).
- **Truthful `valor-session status` display**: `_format_ts` already re-attaches UTC to naive datetimes and labels "UTC" — once the stored value is correct UTC wall-clock, the label becomes truthful with no display change required. Verify and add a regression test rather than re-implement.
- **auto_now audit**: confirmed via `grep` that `updated_at` is the **only** `auto_now`/`auto_now_add` usage in the repo's own models (`grep -rn auto_now models/ tools/ agent/ bridge/ worker/ reflections/ monitoring/ ui/` returns only `agent_session.py:153`). No other producer to fix locally. (The two test-file hits are assertions, not field declarations.)

### Flow

`AgentSession.save()` called → `save()` override sets `self.updated_at = utc_now()` → popoto persists UTC wall-clock (tzinfo stripped) → read-back re-attaches `tz=UTC` (now correct) → `valor-session status` / dashboard / watchdog compute truthful ages.

Worker startup → `_heal_future_updated_at()` → scans AgentSessions → any `updated_at > now` clamped to `max(created_at, now)` → saved → future-dated legacy records corrected once.

### Technical Approach

- **Primary fix is local (option 2), not upstream (option 1).** Rationale: the installed popoto is **PyPI 1.6.1**, pinned by hash in `uv.lock`. An upstream fix requires editing `~/src/popoto` (1.7.0, `src/` layout) → version bump → build → publish to PyPI → bump the pin in `pyproject.toml` + `uv.lock` → `/update` every machine. That is a multi-repo, multi-machine, human-gated release with real latency, for a one-field bug we can fix self-contained in this repo today. The local fix conforms to the existing `utc_now()` convention already used for `created_at`/`started_at`.
- **Drop `auto_now=True`** at `models/agent_session.py:153` → `updated_at = DatetimeField(null=True)`.
- **Override `save()`** (or extend the existing save choke point if one exists) to stamp `self.updated_at = utc_now()` before delegating to `super().save(...)`. Must cover the bulk/`update_fields`-style save paths too — verify `updated_at` is included when partial saves occur, or stamp it unconditionally in the override. (popoto 1.6.1 supports `skip_auto_now`/`update_fields`; since we no longer use `auto_now`, the override is the single source of truth.)
- **Heal**: model the heal on the existing convention referenced in `models/agent_session.py:282-283` (the `_heal_descriptor_pollution` comment) and the existing `cleanup_expired`/`repair_indexes` classmethods. Clamp formula: `updated_at = max(created_at, now)` (and never less than `started_at`). Save via a path that does NOT re-mint a bad value. Idempotent — second run is a no-op.
- **Heal invocation**: hook into worker startup alongside the existing index rebuild / corrupted+orphan cleanup (see `worker/__main__.py` startup sequence). One-shot, guarded so it runs without crashing the worker if Redis hiccups (memory-system-style fail-soft).
- **Upstream popoto fix is deferred to a separate slug** as defense-in-depth (so any *future* `auto_now` field on any model is correct by default) — not required to close this issue.

## Risks

### Risk 1: A save path bypasses the `save()` override and stops updating `updated_at`
**Impact:** `updated_at` goes stale (never advances) — worse than skewed in some triage views.
**Mitigation:** Stamp unconditionally in the override and audit every `.save(` call site on AgentSession (only a handful: `models/agent_session.py:1301/1392/1440/1573`). Add a unit test that a status-transition save advances `updated_at`.

### Risk 2: The one-shot heal clamps a record that was legitimately ahead, or runs every startup and churns Redis
**Impact:** Data churn or masking a real future-dating bug elsewhere.
**Mitigation:** Idempotent + guarded: only touch records strictly in the future; clamp to `max(created_at, now)`; skip already-sane records so a second run is a no-op. Log a count of healed records once.

### Risk 3: Dropping `auto_now` interacts with popoto `update_fields`/partial-save semantics
**Impact:** Partial saves that previously got a free `auto_now` stamp now don't.
**Mitigation:** The `save()` override stamps `updated_at` regardless of `update_fields`, making it the single source of truth. Verify popoto 1.6.1 honors the explicitly-set value (it does — `format_value_pre_save` returns `field_value` unchanged when `auto_now` is False).

## Race Conditions

### Race 1: Heal runs concurrently with live saves at worker startup
**Location:** `worker/__main__.py` startup sequence; `AgentSession._heal_future_updated_at`.
**Trigger:** A session saves (advancing `updated_at` to a correct value) while the heal is mid-scan.
**Data prerequisite:** The heal reads each record's current `updated_at` at clamp time.
**State prerequisite:** Heal must not resurrect a future value over a freshly-correct one.
**Mitigation:** Heal runs in the worker's single-threaded startup sequence *before* the worker begins picking up sessions (same slot as the existing index rebuild / corrupted cleanup), so no live saves race it. The heal is also idempotent, so even a late-arriving correct save is never clobbered (heal only ever moves future→now, never now→future).
