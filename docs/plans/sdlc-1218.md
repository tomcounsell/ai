# session-liveness-check: reap orphan SDK subprocesses when DB row is already terminal

## Problem

A `claude -p` SDK subprocess can survive indefinitely after its owning `AgentSession` row has already reached a terminal state (`completed`, `failed`, `killed`, etc.). The `session-liveness-check` reflection only scans for sessions whose **Redis row** says `status="running"` — it never walks the OS-side `_active_sessions` map and asks the inverse question: "for each subprocess this worker is still tracking, is the corresponding `AgentSession.status` still non-terminal?"

**Current behavior:**

When `_agent_session_health_check()` encounters a session whose running-index entry is stale (row says `completed` but it still appears in the `running` index), it logs a warning and `continue`s at `session_health.py:631-638`. It does **not** reap the OS subprocess. The `_active_sessions` map is never iterated by the health check.

Concrete impact: A subprocess that has been running for 15+ hours with its owning row at `status=completed` keeps a `worker_key` "occupied" — since PM execution is single-threaded per `worker_key`, every subsequent session for that project queues behind the orphan forever, until a human manually runs `kill <pid>`.

**Desired outcome:**

After each health tick, any entry in `_active_sessions` whose corresponding `AgentSession.status` is in `TERMINAL_STATUSES` is SIGTERMd (SIGKILL after next tick if still alive) and removed from `_active_sessions`. The PM queue is no longer blocked by zombie subprocesses.

## Data Flow

1. **Worker spawns a session** — `_execute_agent_session()` in `session_executor.py` registers `SessionHandle(task=task, pid=None)` into `_active_sessions[agent_session_id]`.
2. **Subprocess starts** — `_on_sdk_started(pid)` callback fires, sets `handle.pid = pid` on the handle.
3. **Session completes normally** — `finalize_session("completed")` writes `status=completed` to Redis. The `_execute_agent_session` `finally` block pops the handle from `_active_sessions`. Subprocess exits cleanly.
4. **Anomaly: session completes but subprocess never exits** — `finalize_session("completed")` writes `status=completed`. The subprocess keeps running (heartbeat loop or blocked I/O). The handle is NOT popped because the `finally` block in `_execute_agent_session` may not fire if the asyncio task itself hangs or if the session was finalized externally (e.g., via `valor-session kill`).
5. **Health tick fires (every 300s)** — `_agent_session_health_check()` iterates `AgentSession.query.filter(status="running")`. The orphan's row says `completed`; it either doesn't appear in the running index at all, or it appears and gets skipped by the terminal-status guard at line 631. Either way, `_active_sessions` is never consulted.
6. **Result** — The orphan subprocess runs indefinitely, holding the `worker_key` busy and blocking all downstream sessions for that project.
7. **Fix entry point** — At the END of `_agent_session_health_check()`, after both running and pending scans, add a third pass that iterates `_active_sessions` and reaps any entry whose DB row is terminal.

## Solution

### Key Elements

- **Orphan reap pass**: A new loop at the end of `_agent_session_health_check()` that iterates `list(_active_sessions.items())` and, for each entry whose `AgentSession.status` is in `TERMINAL_STATUSES`, sends SIGTERM to the PID and pops the handle.
- **Grace window**: A 60-second grace period (`entry.updated_at` within 60s of `now`) prevents SIGTERMing a subprocess that JUST transitioned to terminal and is in its normal teardown path. Under healthy conditions the handle is popped by the `_execute_agent_session` `finally` block before the health tick fires, so the grace window is purely defensive.
- **SIGTERM now, SIGKILL on next tick**: Send SIGTERM and pop the handle immediately. On the NEXT tick (5 min later), if the PID still exists (via `os.kill(pid, 0)`), escalate to SIGKILL. This avoids blocking the async health loop on `proc.wait()`.
- **Redis counter**: `session-health:orphan_subprocess_reaped:{project_key}` incremented per reap for observability.
- **No kill-switch by default**: The orphan reap path is always enabled. A `DISABLE_ORPHAN_REAP=1` env flag is documented but not implemented unless the planner/critique deems it prudent — the existing code pattern in `_cleanup_orphaned_claude_processes` has no kill-switch and that precedent is appropriate here.

### Flow

Health tick fires → running scan (unchanged) → pending scan (unchanged) → **orphan reap pass** → for each handle in `_active_sessions`: check DB status → if terminal AND older than grace window → SIGTERM(pid) → pop handle → increment Redis counter → log INFO → end tick

### Technical Approach

- Add the orphan reap pass at the **end** of `_agent_session_health_check()`, after the pending scan's `if checked > 0: logger.info(...)` block and before the function returns.
- Use `list(_active_sessions.items())` to snapshot the dict before iterating (avoids mutation-during-iteration errors).
- For each `(session_id, handle)`:
  1. Look up `AgentSession` by `session_id` (use `AgentSession.query.filter(agent_session_id=session_id)` or `get()`).
  2. Check `actual_status = getattr(entry, "status", None)` for the phantom-guard pattern (same as the terminal-status guard already uses).
  3. If `actual_status in TERMINAL_STATUSES`:
     - Check grace window: skip if `entry.updated_at` is within 60s of `now` (subprocess may still be in natural teardown).
     - If PID is set (`handle.pid is not None`):
       - Try `os.kill(handle.pid, signal.SIGTERM)`. Catch `ProcessLookupError` (already dead — just stale entry), `PermissionError` (log WARNING), other exceptions (log DEBUG).
     - Pop `_active_sessions.pop(session_id, None)`.
     - Increment Redis counter `session-health:orphan_subprocess_reaped:{project_key}` (non-fatal on failure).
     - Log INFO: `[session-health] Orphan subprocess reaped: session=%s pid=%s status=%s`.
  4. If `entry` is None (session row deleted from Redis entirely): also pop the handle (stale entry with no DB record). Log DEBUG.
- SIGKILL escalation: On a subsequent tick, if `handle.pid` is not None and the PID still exists (`os.kill(pid, 0)` succeeds), escalate to `SIGKILL`. **Implementation note:** Since the handle is popped on SIGTERM, the escalation path requires a separate lightweight mechanism. Options:
  - Store pending-SIGKILL PIDs in a module-level set `_pending_sigkill: set[int]`. At the START of each health tick, iterate this set and `SIGKILL` any PIDs that still exist. Pop the PID from the set regardless. This is the simplest approach — no new data structure type, no new imports.
  - Alternatively, skip SIGKILL escalation entirely and rely on `_cleanup_orphaned_claude_processes()` at next worker restart. Given the 5-min tick cadence and the SIGTERM normally being sufficient, this is an acceptable simplification for the Small appetite.
- **Decision**: Implement `_pending_sigkill` set for SIGKILL escalation. It's 5 lines and completes the termination contract without relying on worker restart.

## Failure Path Test Strategy

### Exception Handling Coverage

- `os.kill(pid, signal.SIGTERM)` — catches `ProcessLookupError` (already dead: pop handle, no counter increment), `PermissionError` (log WARNING, pop handle anyway — don't leave a stale entry), other exceptions (log DEBUG).
- Redis counter increment failure — already non-fatal per established pattern in `_agent_session_health_check()`.
- `AgentSession` lookup failure — if `AgentSession.query` raises, catch and log WARNING; continue to next handle (defensive: never crash the health loop).
- Grace window `updated_at` is None or malformed — treat as "no grace" (proceed to reap). The `_ts()` helper already handles this pattern.

### Empty/Invalid Input Handling

- `_active_sessions` is empty — the `list(_active_sessions.items())` loop iterates zero times; no-op.
- `handle.pid` is None (subprocess spawned but `on_sdk_started` callback hasn't fired yet) — skip SIGTERM, still pop the handle (the task will never be able to run since the session is terminal).
- `entry` is None (session row deleted) — pop handle, do not increment orphan counter (no `project_key` to key the counter on), log DEBUG.

### Error State Rendering

The orphan reap pass operates inside the health check loop which runs as a background asyncio task. Errors surface via:
- `logger.warning` for SIGTERM permission errors (operator-visible in `logs/worker.log`).
- `logger.info` for successful reaps (auditable).
- Redis counter `session-health:orphan_subprocess_reaped:{project_key}` visible in dashboard.

## Rabbit Holes

- **Generalizing subprocess termination across all kill flows**: The issue notes this is the "cleaner architectural fix" but explicitly defers it. This plan stays additive — no changes to the `finalize_session` / `transition_status` / `task.cancel()` kill pipeline.
- **Cross-machine subprocess reconciliation**: Each worker only knows about its own `_active_sessions`. Multi-worker coordination is a separate, much larger problem. Out of scope.
- **Investigating why sessions reach terminal state without subprocess exit**: That root cause analysis belongs in a separate issue. This plan just cleans up the symptom.
- **SIGTERM latency and subprocess cleanup ordering**: The 60s grace window and next-tick SIGKILL escalation are sufficient. Do not attempt finer-grained coordination with the subprocess's internal shutdown sequence.
- **Blocking `proc.wait()` inside the async health loop**: Never do this. Fire-and-forget SIGTERM, escalate on next tick via `_pending_sigkill`.

## Risks

### Risk 1: Grace window mis-fires on a freshly-completed session
**Impact:** A session that just finalized (subprocess in its 1-2s natural teardown) gets SIGTERMd unnecessarily, causing a `ProcessLookupError` (harmless) or an abrupt exit (benign — it was about to die anyway).
**Mitigation:** 60s grace window (10x the normal subprocess teardown time). Under healthy conditions the `_execute_agent_session` `finally` block pops the handle from `_active_sessions` before the health tick fires, so the grace window is never reached.

### Risk 2: `_pending_sigkill` leaks if the process exits between SIGTERM and next tick
**Impact:** `_pending_sigkill` contains a PID that no longer exists. SIGKILL attempt hits `ProcessLookupError`.
**Mitigation:** `os.kill(pid, signal.SIGKILL)` catches `ProcessLookupError` and silently removes the PID from `_pending_sigkill`. No leak.

### Risk 3: `_active_sessions` mutation during iteration
**Impact:** `RuntimeError: dictionary changed size during iteration` if another coroutine pops a handle mid-loop.
**Mitigation:** Snapshot with `list(_active_sessions.items())` at the top of the orphan-reap pass. The health check already uses this pattern in `_cleanup_orphaned_claude_processes`.

## Race Conditions

### Race 1: Session transitions to terminal BETWEEN `updated_at` check and SIGTERM
**Location:** `agent/session_health.py` — new orphan reap block
**Trigger:** Health tick reads `status=completed, updated_at=61s ago`. Between the grace check and `os.kill()`, the subprocess naturally exits. SIGTERM fires on a dead PID.
**Data prerequisite:** `handle.pid` must be non-None and the PID must still exist.
**Mitigation:** `ProcessLookupError` is already caught and treated as "already dead — pop handle". No correctness impact. This is the expected no-harm case.

### Race 2: New session spawned with same `agent_session_id` as a just-reaped terminal session
**Location:** `_active_sessions` dict, health tick boundary
**Trigger:** Session `X` completes, its handle is popped by orphan-reap pass. The worker immediately re-queues a new session that, by coincidence, gets the same `agent_session_id` (vanishingly unlikely — IDs are timestamp-based). Health tick would not see it since it was registered after the snapshot.
**Mitigation:** IDs include microsecond timestamps and random components; collision is negligible. Additionally, the snapshot (`list(_active_sessions.items())`) is taken before the loop, so a new registration after the snapshot would not be visible to this tick.

### Race 3: `_active_sessions.pop()` races with `_execute_agent_session` `finally` block
**Location:** `_active_sessions` dict
**Trigger:** The `_execute_agent_session` `finally` block pops the same handle the orphan-reap pass just reaped.
**Mitigation:** `_active_sessions.pop(session_id, None)` is idempotent — if the handle is already gone, it returns `None` without error. Both code paths use `.pop(session_id, None)`. The asyncio event loop is single-threaded, so the two pops cannot interleave within a single tick.
