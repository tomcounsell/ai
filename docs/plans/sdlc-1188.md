# Daily Log Review — Local Reflection with Telegram Output (#1188)

## Problem

A remote CCR routine called `daily-log-review` was created on claude.ai/code/routines to review previous-day logs per project. It got stuck on every run because it tried to read `logs/*.log`, which is gitignored and absent in the remote git checkout. The platform detected the session exceeded 2× its scheduled interval and force-reset it, ultimately auto-clearing it from the routines list.

**Current behavior:**

- A local `daily-log-review` reflection **already exists** in `config/reflections.yaml` (lines 196-202) as `execution_type: function` calling `reflections.auditing.run_log_review`. It scans `logs/*.log` per project, extracts structured errors, counts warnings, and returns a `{"status": "ok", "findings": [...], "summary": "..."}` dict.
- The findings are written to the reflection scheduler's logs only (via `logger.info`). **Nothing reaches Telegram.** A human has to tail `logs/worker.log` (or read the per-reflection log lines) to see the daily summary.
- No remote routine for this exists anymore (it was force-cleared by the CCR platform). Good — that is the correct posture.

**Desired outcome:**

- `daily-log-review` runs daily on the local machine and sends a concise summary (5-15 lines) to the `Dev: Valor` Telegram chat, scoped per project. Subject line: `Daily Log Review — YYYY-MM-DD`.
- The summary includes file sizes, error counts, regression markers (already covered today by `run_log_review`), plus optional Sentry counts when `SENTRY_DSN` is configured for a project.
- Zero dependence on `logs/` being in git. Zero dependence on the CCR sandbox.

## Data Flow

1. **Entry point**: `ReflectionScheduler.tick()` (asyncio loop, 60s cadence) detects `daily-log-review` is due (last_run + 86400s < now).
2. **Dispatch**: `execute_function_reflection(entry)` calls `reflections.auditing.run_log_review` via `loop.run_in_executor(None, func)` (sync function, runs in thread pool — preserves the PR #1056 contract).
3. **Per-project scan**: For each local project, `run_log_review` walks `logs/*.log`, extracts structured errors, counts warnings, scans for regression markers. Bounded reads via `_read_log_tail_lines(file, n=1000)` and `_read_log_text_bounded`.
4. **(NEW) Sentry enrichment**: If `SENTRY_DSN` is in the project's `.env`, run `sentry-cli issues list --status unresolved --json` (timeout 10s) and append a one-line count per project. Skip silently if `sentry-cli` is missing or the call times out — Sentry data is additive, never blocking.
5. **(NEW) Telegram delivery**: After the existing findings list is built, if `len(findings) > 0` (or always, depending on Open Question 1), format a 5-15 line summary and `subprocess.run(["valor-telegram", "send", "--chat", "Dev: Valor", message], timeout=10)`. Catch `FileNotFoundError`, `subprocess.CalledProcessError`, `subprocess.TimeoutExpired` — all swallow with `logger.warning`.
6. **Output**: Same `{"status": "ok", "findings": [...], "summary": "..."}` dict the function returns today, plus a side-effect Telegram message in the `Dev: Valor` chat.

## Solution

### Key Elements

- **Existing function reflection unchanged in shape**: keep `daily-log-review` as `execution_type: function` calling `reflections.auditing.run_log_review`. Do NOT flip to `execution_type: agent` — the function form is already wired, already bounded-read-safe (PR #1056), and avoids the cost of spawning a full Claude session for a daily digest.
- **Add Telegram notification at end of `run_log_review`**: format the findings into a concise 5-15 line summary; send via `subprocess.run(["valor-telegram", "send", ...])` with a 10s timeout and broad exception suppression.
- **Add optional Sentry enrichment**: per-project, if `SENTRY_DSN` is set, run `sentry-cli issues list --status unresolved --json` (10s timeout) and append a one-line count to the findings. Best-effort; never raise.
- **No new YAML entry**: the `daily-log-review` entry already exists at `config/reflections.yaml:196-202`. Only its `description` field needs a small clarification ("...and sends summary to Telegram").

### Flow

Worker startup → `ReflectionScheduler` ticks → `daily-log-review` due → `execute_function_reflection` runs `run_log_review` in executor → per-project scan + Sentry enrichment + Telegram send → findings dict returned, scheduler logs duration → next tick in 24h.

### Technical Approach

- **Implementation site:** `reflections/auditing.py::run_log_review` — append a Telegram-send block after the existing `findings` list is built (before the `return` statement).
- **Sentry enrichment:** add a helper `_collect_sentry_counts(project) -> str | None` near the top of the function. Returns a one-line summary string per project, or `None` on any error/missing CLI/missing DSN. Wrap the `sentry-cli` invocation in `subprocess.run(..., timeout=10, capture_output=True, text=True)` and catch `FileNotFoundError`, `CalledProcessError`, `TimeoutExpired`, `JSONDecodeError`. Each per-project call is independent — one project's Sentry timeout cannot block another's scan.
- **Telegram-send block:** build the message with the existing summary line as the header, then up to 12 most-recent findings, then a footer "(N more findings — see worker.log)" if `len(findings) > 12`. Subject is the first line: `Daily Log Review — YYYY-MM-DD`. Send with `subprocess.run(["valor-telegram", "send", "--chat", "Dev: Valor", message], timeout=10, check=False)`. Catch all `subprocess` exceptions and `FileNotFoundError`; log a warning, never raise.
- **YAML touch (description only):** update line 197 from `"Review previous day's logs per project: file sizes, error counts, regression markers"` to `"Review previous day's logs per project and send summary to 'Dev: Valor' Telegram"`. Keep all other fields unchanged.
- **No conversion to `execution_type: agent`** — the issue's solution sketch suggests this, but it is the wrong shape. Agent reflections (a) cost a full Claude session per day, (b) are currently disabled across the board for this exact reason (`sentry-issue-triage`, `system-health-digest`, `daily-report-and-notify` are all `enabled: false`), and (c) the existing function form already does the local-filesystem work the issue motivates. Adding 10 lines of Telegram-send is strictly cheaper and equally effective.

## Risks

### Risk 1: `valor-telegram send` blocks when the bridge is down

**Impact:** The reflection function blocks for the subprocess timeout (10s) instead of returning quickly. With `daily-log-review` running once per day, this is one 10s blip per outage day — acceptable.

**Mitigation:** Pass `timeout=10` to `subprocess.run`. The reflection scheduler's per-reflection timeout (`DEFAULT_FUNCTION_TIMEOUT = 1800`) is a much larger backstop. Test that `TimeoutExpired` is caught and logged.

### Risk 2: Sentry CLI scan times out per-project, multiplying delay

**Impact:** With ~12 projects, a hung `sentry-cli` could add 12 × 10s = 120s if all projects had Sentry. In practice, only a few do.

**Mitigation:** Per-project 10s timeout. Each call is independent. The aggregate worst case (120s) is well within the scheduler's 1800s timeout. Skip the Sentry block entirely if `sentry-cli` is not on PATH (one cheap `shutil.which` check at function entry).

### Risk 3: Telegram message exceeds reasonable length

**Impact:** A truly bad day could produce 100+ findings. A multi-thousand-line Telegram message is unhelpful and may be rate-limited.

**Mitigation:** Cap displayed findings at 12; append `(N more — see worker.log)` if more exist. The full findings stay in the return dict, which the scheduler logs in full.

## Open Questions

1. **Resolved:** Send a one-line heartbeat on empty findings. The format is `Daily Log Review — YYYY-MM-DD: 0 findings across N projects` (single Telegram message, no per-project breakdown). Rationale: existing reflection-Telegram precedents (`daily_report._post_to_telegram`, `memory_consolidation._flag_contradiction`) stay silent on empty, but `daily-log-review` is itself the canary for log-scanning health — silence is ambiguous (clean day vs. scheduler-dead vs. bridge-down). A heartbeat resolves the ambiguity at near-zero noise cost (one message/day worst case).
2. **Resolved:** Ship the Sentry helper in this PR. `sentry-cli` is already installed on every machine by `scripts/update/sentry_cli.py` (verified), so the only runtime guards needed are `shutil.which("sentry-cli")` (defensive) and per-project `SENTRY_DSN` presence. Helper is ~15 LOC with a 10s subprocess timeout and broad exception suppression — same shape as the Telegram-send block. No new architectural surface; the precedent for subprocess-with-swallow already exists in `memory_consolidation.py:327`.
