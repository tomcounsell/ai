## Prior Blockers — Verified Resolution

### BLOCKER 1 (Skeptic, Consistency): Unresolved load-bearing spikes + line-24-vs-line-145 status-mapping contradiction — RESOLVED

The revised plan adds a `## Spike Results` section (plan lines 77–101) with spike-1, spike-2, and spike-3 all marked RESOLVED with embedded findings and confidence levels.

Verified against HEAD:
- `BridgeAdapter.run()` returns `""` and raises no error on any `exit_reason`, including `exception` (the exception is caught inside `Container.run()`, which sets `result.exit_reason="exception"` and returns a normal `ContainerResult` rather than re-raising).
- Therefore in `session_executor.py`, `task.error` is falsy and `final_status = "completed" if not task.error else "failed"` yields `completed` — the granite `exit_reason` never reaches the mapping.

The Problem section (plan line 25) now states this verified behavior. The old, contradictory "correctly maps non-clean exits to `failed`" claim (former line 145) is explicitly struck and replaced with the verified two-branch dashboard-render approach (plan lines 99, 175). Problem § and Technical Approach § are now internally consistent.

### BLOCKER 2 (Adversary, Skeptic): Transcript-path identity must select the JSONL by the session's own captured UUID — RESOLVED

The revised plan abandons the dead `last_resume_uuid()`/`finally`-on-`isalive()` scrape (which captured only the PM UUID on a live PM and never captured the Dev UUID).

Verified against HEAD that the replacement mechanism is real and sound:
- `claude --session-id <uuid>` is a genuine flag ("Use a specific session ID for the conversation (must be a valid UUID)") confirmed to work with the interactive TUI.
- The plan generates two `uuid4()` values (one per PTY), pins each at spawn via `--session-id`, and threads them `PairSpawnSpec.pm_session_id`/`dev_session_id` → `PTYDriver.spawn()` (plan lines 86, 126, 160, 356–357).
- The transcript path is computed deterministically as `~/.claude/projects/{cwd.replace("/","-")}/{uuid}.jsonl`, known at spawn time — no discovery scan, no "most-recent file" guess, no collision risk on concurrent same-repo sessions.
- The tailer binds to exactly one captured UUID; an absent file at exit is a logged WARNING, never a fallback to an arbitrary file (plan lines 160, 173).

### BLOCKER 3 (User): Reaction-gating Open Q #2 unresolved while shipping a user-facing behavior change — RESOLVED

Open Q #2 is resolved in the plan (lines 165, 177, 470) per the User critic's recommendation:
- The `communicated=False` chip ships now as the visible, non-destructive signal (reads `user_facing_routed`, already on the record post-#1647).
- Success-reaction suppression for clean-but-silent completions is deferred — legitimate background/fire-and-forget sessions validly complete silently, and flagging them would invert the original false-success bug.
- The only reaction change in this slice is the unambiguous one: a non-clean `exit_reason` (e.g. `exception`) selects `REACTION_ERROR` instead of a success reaction (executor branch, plan line 177). The "clean + `communicated=False`" combination keeps its existing reaction, with table-driven tests asserting every `(exit_reason, user_facing_routed)` combination (plan line 398).
