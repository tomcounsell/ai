### Blockers (Round 1)

#### B1. `agent.sdk_client.get_anthropic_client()` does not exist
- **Critics:** Skeptic, Consistency Auditor
- **Location:** Solution → Resolved Open Questions item 5; Step 2
- **Finding:** Plan instructs builder to call `agent.sdk_client.get_anthropic_client()`. That function does not exist. Repo has `_get_anthropic_client()` (private) in `bridge/routing.py:689`, and an `_anthropic_circuit` (CircuitBreaker, not a client) in `agent/sdk_client.py:101`. A literal builder will fail import.
- **Suggestion:** Use the pattern in `reflections/utils.py:run_llm_reflection` (line 187): instantiate `anthropic.Anthropic()` directly.
- **Implementation Note:** `import anthropic; client = anthropic.Anthropic(); response = client.messages.create(model=HAIKU, max_tokens=2048, messages=[...])`. Wrap in `try/except ImportError`. Pull HAIKU from `config.models`. Do NOT invent a new module path.

#### B2. Per-project Reflection records will not appear on the dashboard
- **Critics:** Operator, User, Consistency Auditor
- **Location:** Success Criteria checkbox 6; Solution "One Reflection record per project"
- **Finding:** `ui/data/reflections.py:get_all_reflections()` only iterates the YAML registry. Per-project records `pm-audio-briefing-{project_key}` are written to Redis but never registered in `reflections.yaml`, so never rendered. Success Criterion is unachievable as written.
- **Suggestion:** (a) extend `ui/data/reflections.py` to merge Redis records matching a registered prefix; (b) drop per-project records and store state inside the top-level record's `run_history` keyed by `project_key`; (c) drop the success criterion.
- **Implementation Note:** Option (a) is cleanest: in `_get_registry_map()`, scan `Reflection.get_all_states()` for records starting with a known prefix; registry entry could declare `expand_per_project: true`. Add `pm-audio-briefing-*` to `REFLECTION_GROUPS` at `ui/data/reflections.py:34`.

#### B3. `load_local_projects()` does NOT filter by machine ownership
- **Critics:** Skeptic, Adversary, Consistency Auditor
- **Location:** Freshness Check; Architectural Impact "Coupling"; Resolved Open Questions item 7; Risk 5
- **Finding:** Plan claims four times that `load_local_projects()` filters by `scutil --get ComputerName`. Actual implementation (`reflections/utils.py:37-65`) only filters by `wd.exists()`. On multi-machine setups with shared checkouts, both machines fire briefings — duplicate voice notes.
- **Suggestion:** (a) add explicit machine-name filter inside the new callable, or (b) fix `load_local_projects()` (separate hotfix) and depend on it.
- **Implementation Note:** Add at the top of the per-project loop in `reflections/pm_audio_briefing/__init__.py::run()`: `import subprocess; this_machine = subprocess.check_output(["scutil", "--get", "ComputerName"], text=True).strip(); if (project.get("machine") or "").strip() != this_machine: continue`. Log skipped at DEBUG, accepted at INFO once per startup.

#### B4. Top-level scheduler `running` lock starves all per-project work behind one slow LLM call
- **Critics:** Adversary, Operator
- **Location:** Solution → Per-project failure isolation
- **Finding:** `agent/reflection_scheduler.py:236-245` defines `is_reflection_running(state)` as `state.last_status == "running"`. With `interval: 60` and per-project Anthropic + TTS + Telegram work, total runtime for 3+ projects easily exceeds 60s. Subsequent ticks see `running` and skip. If runtime exceeds 120s, scheduler marks `error` for "appeared stuck".
- **Suggestion:** (a) bump `interval: 300`; (b) split into per-project registry entries; (c) bypass `mark_started` at top level.
- **Implementation Note:** Pick (a) — set `interval: 300` in the registry entry. The 5-min schedule-slot widen window already covers this. Update Risk 3 mitigation accordingly. With `interval: 300`, the entry fires at most every 5 minutes, giving the per-project fanout up to 5 minutes to complete — orders of magnitude more than the worst case.

### Concerns (Round 1)

#### C1. Test files referenced in the Test Impact section don't exist
- **Critics:** Skeptic, Consistency Auditor
- **Implementation Note:** Real existing tests: `tests/unit/test_reflection_scheduler.py`, `tests/unit/test_reflection_model.py`, `tests/unit/test_reflections_package.py`, `tests/unit/test_ui_reflections_data.py` (this is the dashboard-data test). For YAML loading regression, update `tests/unit/test_reflection_scheduler.py::test_load_registry_*` cases. Replace named non-existent files with these.

#### C2. Per-project idempotency record locks the date even after failure
- **Critics:** Adversary, Operator
- **Implementation Note:** Cheap solution: skip only if `ran_at_date == today AND last_status == "success"`. If `last_status == "error"`, allow the run to proceed. Zero schema cost. Add a comment explaining the design choice in the callable's idempotency block.

#### C3. Race 1 mitigation isn't atomic; needs Redis CAS
- **Critics:** Adversary
- **Implementation Note:** `lock_key = f"pm-briefing-lock:{project_key}:{today_in_tz.isoformat()}"; if not redis.set(lock_key, str(time.time()), nx=True, ex=90000): continue`. TTL 90000s = 25h so stale locks expire. In addition to the Reflection record idempotency, not replacing it.

#### C4. The "no numbers" regex blocks legitimate spoken content
- **Critics:** User, Adversary
- **Implementation Note:** Tighten to `re.search(r"\b(?:issue|pr|#)\s*\d{2,}\b", transcript, re.IGNORECASE)` — catches "issue 1197", "#1197", "PR 1197" but allows "$500", "250 users", and product version numbers. Add unit tests for each.

#### C5. Auto-confirm removes the safety mechanism the source-of-truth skill considers essential
- **Critics:** User, Skeptic
- **Implementation Note:** Add to Pass A LLM system prompt: "Do not invent decisions, commitments, or future actions. Only narrate work that has already shipped or is queued. Never use 'we will', 'I'll push', 'pushing to', 'unless you want it sooner'." Add a unit test asserting the prompt forbids these phrases. This downgrades Pass A from full /do-debrief style to a narration-only variant.

#### C6. Top-level Reflection record floods run_history every 60s
- **Critics:** Operator, Simplifier
- **Implementation Note:** Coupled with B4 fix (`interval: 300`), tolerate the resulting 288 ticks/day. Add a description note clarifying high run_count is expected. v2: extend `ReflectionEntry` with cron-style schedule field.

#### C7. Subprocess shell-out adds 100-300ms × 2-3 calls per project per day for no clear gain
- **Critics:** Simplifier
- **Implementation Note:** Look at `tools/valor_telegram.py:780-816`: builds payload dict and `r.rpush(queue_key, json.dumps(payload))`. Replicate in `delivery.py` directly. Use `from tools.tts import synthesize` for audio. Subprocess only `gh` (no Python wrapper).

#### C8. The 4-builder team / 4-module split is over-decomposed for ~200 lines
- **Critics:** Simplifier
- **Implementation Note:** Replace Tasks 1-3 with a single Task: "Build pm_audio_briefing package (collector, builder, delivery, __init__) with unit tests for each layer". Make Task 4 (registry + schema) depend on it. Reduces the dependency graph to 4 nodes from 7.

#### C9. v1 ships 7 collectible categories when issue requires only 2
- **Critics:** Simplifier, User
- **Implementation Note:** In `collector.py`, define `_COLLECTORS = {"merges": _collect_merges, "open-bugs": _collect_open_bugs, "upvote-queue": _collect_upvote_queue}`. Unknown categories: log warning and skip. Trim example schema to the 2-3 used categories. New categories are then a single function + test addition.

### Nits (Round 1)

- **N1.** `pm_briefing.groups` (list of strings) collides with `projects.<key>.telegram.groups` (dict). Rename to `target_groups` or accept both shapes.
- **N2.** Add a one-liner clarifying that registry-level `interval: N` means "this entry is eligible to fire on every scheduler tick when ≥ N seconds since last run" — the per-project schedule-slot detection is the real gate.
- **N3.** Step 5 manual run will deliver real audio to real groups. Add a `DRY_RUN=1` env var that the callable honors.
- **N4.** Risk 4 rate-limit math should state "20 gh calls per machine per morning, regardless of repo overlap."
- **N5.** Add to builder Pass B: "verify the first sentence is a decision/heads-up, not setup; structural shape: decision → heads-up → batched FYIs → close." From `.claude/skills/do-debrief/SKILL.md:80-83`.

### Verdict

**NEEDS REVISION** — 4 blockers must be resolved before build.

The blockers are factual errors and architectural collisions that will fail at build time:
1. The Anthropic client path doesn't exist (B1) — builder will hit ImportError immediately.
2. Per-project records won't appear on dashboard (B2) — Success Criterion is unachievable as written.
3. `load_local_projects()` doesn't enforce machine ownership (B3) — risks duplicate voice-note delivery on multi-machine deployments, contradicting the single-machine ownership invariant.
4. The 60s top-level scheduler lock starves multi-project fanout under realistic LLM/TTS latency (B4) — the daily brief can fail to fire on busy mornings.

Concerns C1-C5 are also high-impact: bad test paths (C1), idempotency-after-failure (C2), real Race 1 race (C3), over-blocking regex (C4), and the auto-confirm safety gap (C5) the source-of-truth skill calls out as cardinal anti-pattern.

The plan is well-scoped and the resolved open questions are sound. Once the four blockers are corrected and the high-priority concerns embed Implementation Notes into the plan text, this is buildable.
