### Blockers (Round 1)

#### B1. `agent.sdk_client.get_anthropic_client()` does not exist
- **Critics:** Skeptic, Consistency Auditor
- **Location:** Solution → Resolved Open Questions item 5; Step 2
- **Finding:** Plan instructs builder to call `agent.sdk_client.get_anthropic_client()`. That function does not exist. Repo has `_get_anthropic_client()` (private) in `bridge/routing.py:689`, and an `_anthropic_circuit` (CircuitBreaker, not a client) in `agent/sdk_client.py:101`. A literal builder will fail import.
- **Suggestion:** Use the pattern in `reflections/utils.py:run_llm_reflection` (line 187): instantiate `anthropic.Anthropic()` directly.
- **Implementation Note:** `import anthropic; client = anthropic.Anthropic(); response = client.messages.create(model=HAIKU, max_tokens=2048, messages=[...])`. Wrap in `try/except ImportError`. Pull HAIKU from `config.models`. Do NOT invent a new module path.

#### B2. Per-project Reflection records will not appear on the dashboard
- **Critics:** Operator, User, Consistency Auditor
- **Location:** Success Criteria checkbox 6; Solution "One Reflection record per project"
- **Finding:** `ui/data/reflections.py:get_all_reflections()` only iterates the YAML registry. Per-project records `pm-audio-briefing-{project_key}` are written to Redis but never registered in `reflections.yaml`, so never rendered. Success Criterion is unachievable as written.
- **Suggestion:** (a) extend `ui/data/reflections.py` to merge Redis records matching a registered prefix; (b) drop per-project records and store state inside the top-level record's `run_history` keyed by `project_key`; (c) drop the success criterion.
- **Implementation Note:** Option (a) is cleanest: in `_get_registry_map()`, scan `Reflection.get_all_states()` for records starting with a known prefix; registry entry could declare `expand_per_project: true`. Add `pm-audio-briefing-*` to `REFLECTION_GROUPS` at `ui/data/reflections.py:34`.

#### B3. `load_local_projects()` does NOT filter by machine ownership
- **Critics:** Skeptic, Adversary, Consistency Auditor
- **Location:** Freshness Check; Architectural Impact "Coupling"; Resolved Open Questions item 7; Risk 5
- **Finding:** Plan claims four times that `load_local_projects()` filters by `scutil --get ComputerName`. Actual implementation (`reflections/utils.py:37-65`) only filters by `wd.exists()`. On multi-machine setups with shared checkouts, both machines fire briefings — duplicate voice notes.
- **Suggestion:** (a) add explicit machine-name filter inside the new callable, or (b) fix `load_local_projects()` (separate hotfix) and depend on it.
- **Implementation Note:** Add at the top of the per-project loop in `reflections/pm_audio_briefing/__init__.py::run()`: `import subprocess; this_machine = subprocess.check_output(["scutil", "--get", "ComputerName"], text=True).strip(); if (project.get("machine") or "").strip() != this_machine: continue`. Log skipped at DEBUG, accepted at INFO once per startup.

#### B4. Top-level scheduler `running` lock starves all per-project work behind one slow LLM call
- **Critics:** Adversary, Operator
- **Location:** Solution → Per-project failure isolation
- **Finding:** `agent/reflection_scheduler.py:236-245` defines `is_reflection_running(state)` as `state.last_status == "running"`. With `interval: 60` and per-project Anthropic + TTS + Telegram work, total runtime for 3+ projects easily exceeds 60s. Subsequent ticks see `running` and skip. If runtime exceeds 120s, scheduler marks `error` for "appeared stuck".
- **Suggestion:** (a) bump `interval: 300`; (b) split into per-project registry entries; (c) bypass `mark_started` at top level.
- **Implementation Note:** Pick (a) — set `interval: 300` in the registry entry. The 5-min schedule-slot widen window already covers this. Update Risk 3 mitigation accordingly. With `interval: 300`, the entry fires at most every 5 minutes, giving the per-project fanout up to 5 minutes to complete — orders of magnitude more than the worst case.

### Blockers (Round 2)

#### B1 (Round 2). Failure-mode contract for TTS failure contradicts itself
- **Critics:** Operator, User, Consistency Auditor
- **Location:** `Failure Path Test Strategy → Error State Rendering` (line 201) vs. `delivery.py` Key Element (line 109) and Step 1 spec (line 369)
- **Finding:** Two sections disagree on what happens when TTS fails:
  - "Error State Rendering" says: post a "Daily briefing failed: TTS unavailable" text message to the PM group.
  - delivery.py Key Element + Step 1 spec say: do not enqueue the follow-up; raise to caller.
- **Suggestion:** Pick one and propagate to all three sections. Recommended: enqueue a single text-only failure notice per group AND re-raise so the per-project Reflection record still records `error`. Update delivery.py Key Element, Step 1 builder spec, Failure Path Test Strategy, and the test plan to match.
- **Implementation Note:** In `delivery.send()`: wrap the `tools.tts.synthesize()` call in `try/except`. On exception, for each group in `target_groups`, push a payload `{"chat_id": ..., "text": "Daily briefing failed: TTS unavailable", ...}` to the outbox; then re-raise. Test: monkeypatch `synthesize` to raise; assert `r.rpush` called once per group with the failure-notice text; assert `delivery.send()` re-raises so the caller marks `last_status = "error"`.

### Concerns (Round 2)

#### C1 (Round 2). `expand_per_project: true` registry field is decorative
- **Critics:** Skeptic, Consistency Auditor
- **Location:** Step 2 YAML block (line 386); `ui/data/reflections.py` extension
- **Finding:** The YAML loader at `agent/reflection_scheduler.py:152-172` builds `ReflectionEntry` from a fixed set of fields and silently ignores `expand_per_project`. Adding it to the YAML doesn't make it readable from `_load_registry()`. The dashboard helper would still need to know which entries are prefix-expanded.
- **Suggestion:** Two options:
  - (a) Wire `expand_per_project` through `ReflectionEntry` (`agent/reflection_scheduler.py:80-100`) so `_load_registry()` exposes it; then `ui/data/reflections.py` can read it from the dict.
  - (b) Drop the YAML field entirely and use a hard-coded `_PREFIX_EXPANDED_REFLECTIONS = ("pm-audio-briefing",)` tuple in `ui/data/reflections.py`.
- **Implementation Note:** Pick (b) for v1. One name, one place to change. If a second prefix-expanded reflection ships, promote to YAML at that time. Update Step 2 to remove `expand_per_project: true` from the YAML and instead define `_PREFIX_EXPANDED_REFLECTIONS = ("pm-audio-briefing",)` at module-load time in `ui/data/reflections.py`. Document this in the Architectural Impact section.

#### C2 (Round 2). `interval: 300` collides with the 2× stuck-reset at `agent/reflection_scheduler.py:464-474`
- **Critics:** Adversary, Operator
- **Location:** `agent/reflection_scheduler.py:464-474` resets a reflection to `error` if it runs longer than `2 × interval`. With `interval: 300` that's 600s. A 5-project morning fanout doing collect (+gh) + Anthropic Pass A + TTS + outbox per project can plausibly exceed 600s, especially if Kokoro falls back to OpenAI cloud.
- **Suggestion:** Set the per-entry `timeout` field (already supported by `ReflectionEntry`) to a value larger than the worst-case fanout — recommended `1500` (25 minutes). The 5-min `interval` stays as the eligibility gate; the `timeout` controls the stuck-reset.
- **Implementation Note:** Update Step 2 YAML block to add `timeout: 1500` below `interval: 300`. Update Risk 3 mitigation to mention the timeout. Verify `ReflectionEntry` supports the field (search for `timeout` in `agent/reflection_scheduler.py`); if it doesn't, fall back to widening `interval` to 600 (still well below the 2× threshold).

#### C3 (Round 2). SETNX lock + Reflection idempotency don't pin to project tz
- **Critics:** Adversary
- **Location:** Flow section (line 119); Race 1 (line 254)
- **Finding:** The Flow uses `today.isoformat()` for the SETNX lock key but `today_in_tz` for the Reflection check. If those diverge across a midnight boundary (server in UTC, project in PST → 16-hour drift possible at certain times), the same calendar day has two different lock keys and two different idempotency anchors. A late-evening PST tick could see the next day in UTC and re-fire.
- **Suggestion:** Resolve `today_in_project_tz = date.fromtimestamp(time.time(), tz=ZoneInfo(project["pm_briefing"]["timezone"]))` once per project iteration and use the SAME value for both the SETNX lock key and the Reflection date check.
- **Implementation Note:** In Step 1's `__init__.py` spec, change "Acquire Redis SETNX lock keyed by `(project_key, today.isoformat())`" to `(project_key, today_in_project_tz.isoformat())`. Update Race 1 mitigation accordingly. Add a unit test: project tz=PST, server tz=UTC, current time in the boundary window — assert both anchors resolve to the same calendar date.

#### C4 (Round 2). Slot-widen arithmetic breaks for `schedule_minute >= 56`
- **Critics:** Adversary
- **Location:** Implementation Approach: "Schedule slot detection" (line 148)
- **Finding:** `now.minute >= schedule.minute and now.minute < schedule.minute + 5` fails when `schedule_minute = 58` because `schedule_minute + 5 = 63` which `now.minute` can never reach (max is 59). The check accidentally collapses to `now.minute >= 58 and now.minute < 63`, so 58 and 59 match — but the project never fires at minute 0, 1, or 2 of the next hour, which is the actual slack the widen window is supposed to provide.
- **Suggestion:** Use absolute-minute arithmetic: `slot_start = schedule_hour * 60 + schedule_minute; now_abs = now.hour * 60 + now.minute; if not (slot_start <= now_abs < slot_start + 5): continue`. This handles hour rollover correctly.
- **Implementation Note:** Update the Implementation Approach bullet and the Race 1 trigger description with the absolute-minute formula. Add a unit test parametrized over `(schedule="00:58", current="01:02")` asserting the slot matches.

#### C5 (Round 2). Pass B "first sentence is a decision" check has no implementation
- **Critics:** Simplifier, User
- **Location:** Builder Key Element (line 108); Step 1 builder spec (line 367); Test Impact (line 209)
- **Finding:** The plan adds a "first-sentence-is-decision check" to Pass B's deterministic structural cut. But "is this sentence a decision/heads-up rather than setup" is a semantic judgment, not a structural property. Pass B is described elsewhere as deterministic word-count enforcement. Hiding an LLM call (or a pile of regexes) inside Pass B contradicts that contract.
- **Suggestion:** Move the "first sentence must be a decision/heads-up" rule into the Pass A LLM system prompt (it joins "no numbers" and "no forward-looking commitments"). Remove the structural check from Pass B and from Test Impact.
- **Implementation Note:** Update Builder Key Element to remove "+ first-sentence-is-decision check" from Pass B. Add to the Pass A prompt list of constraints: "The first sentence must be a decision or heads-up (the most important shipped or queued item), not setup or context." Update Test Impact to assert the prompt-string contains this constraint instead of asserting Pass B enforces it programmatically.
