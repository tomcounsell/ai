### Concerns (Round 2)

#### C1. `should_respond_async` does NOT guarantee `project != None` at `:1003` for empty-DM-whitelist deployments

- **Severity**: CONCERN
- **Critics**: Skeptic, Adversary
- **Location**: Solution L114, Risk 3 mitigation L189, Task 2
- **Finding**: The plan asserts "when execution reaches `:1003` today, `project` is already non-None in every reachable case." Tracing `should_respond_async` → `should_respond_sync` (`bridge/routing.py:967-1004`): for DMs, the sync gate returns `True` when `RESPOND_TO_DMS=True` and `DM_WHITELIST` is empty (no whitelist gate fires). `RESPOND_TO_DMS` defaults to `True` for any active project (`bridge/telegram_bridge.py:585-588`). In a deployment where any active project sets `respond_to_dms: true` (the default) and `dms.whitelist` is unpopulated, a DM with `project=None` reaches `:1003` today and currently lands as a `dm`-keyed session. The new `if not project: return` silently drops it — a behavior change the plan classifies as "dead code elimination."
- **Suggestion**: Either (a) verify by code-read at PR review that all production deployments have `DM_WHITELIST` populated and document this as the assumed invariant, or (b) demote the early-return to `logger.info("Dropping DM from non-whitelisted sender or unowned chat") ; return` so the silent drop is at least observable.
- **Implementation Note**: Add a one-line `logger.info` immediately before `return` at the new `:1003` guard: `logger.info(f"Dropping message from {sender_name} ({sender_id}) in {chat_title or 'DM'} — no project resolved")`. This makes the behavior change observable in production without requiring an additional commit if the assumption proves wrong. If a deployment has `DM_WHITELIST=[]` and `RESPOND_TO_DMS=True`, this log will fire on every unowned DM, surfacing the regression within minutes instead of the 24h soak window.

#### C2. `_warn_if_legacy_namespace` runs BEFORE `m.save()`, producing false-positive warnings on Redis-down incidents

- **Severity**: CONCERN
- **Critics**: Adversary
- **Location**: Task 3, Solution L131
- **Finding**: Task 3 says "Call the helper from `Memory.safe_save(...)` before the `m.save()` line." If `m.save()` raises (Redis outage, connection refused), the WARNING with stack trace has already fired even though no record was written. During a Redis incident with concurrent dm-tagged saves, operators see a flood of "dm namespace write detected" warnings that misrepresent what happened — the writes never persisted.
- **Suggestion**: Move the `_warn_if_legacy_namespace` call to AFTER successful `save()` returns a non-False result (i.e., inside the success branch at `models/memory.py:146-149`). The helper should observe actual writes, not attempted writes.
- **Implementation Note**: Place the call after `result = m.save()` and inside `if result is not False:` so it only logs on records that actually persisted. Wrap in its own `try/except` so logging failure doesn't propagate. This also aligns the helper's semantics with its name — it warns about records IN the partition, not records attempting to enter it.

#### C3. Stack-trace logging via `traceback.format_stack()` produces oversized log lines

- **Severity**: CONCERN
- **Critics**: Adversary, Operator
- **Location**: Task 3, Risk 2 L181-185
- **Finding**: `traceback.format_stack()` returns a list of multi-line strings; interpolating into `f"...{traceback.format_stack()}"` yields a single log message with embedded newlines easily exceeding 10KB. Many log aggregators truncate or split such lines, fragmenting the stack across multiple records and breaking searchability. A noisy regression could exhaust per-line budgets.
- **Suggestion**: Use `logger.warning("dm namespace write detected", stack_info=True)` (Python logging's native stack-info support) OR `''.join(traceback.format_stack()[-10:])` to bound the depth. Same for the DEBUG path on `"default"`.
- **Implementation Note**: Replace `traceback.format_stack()` with `stack_info=True` argument: `logger.warning("Memory write to legacy namespace project_key=%s; see stack_info", project_key, stack_info=True)`. This produces a single structured log entry the aggregator can parse, with the stack as a single trailing block. For DEBUG, same pattern: `logger.debug("...", stack_info=True)`. The `stack_info=True` argument is universally supported in stdlib logging since Python 3.2.

#### C4. Bridge startup invariant is true by construction; `assert` is stripped under `python -O`

- **Severity**: CONCERN
- **Critics**: Operator, Simplifier
- **Location**: Task 3.5, Risk 3 mitigation L189
- **Finding**: `DM_WHITELIST` and `DM_USER_TO_PROJECT` are populated in the SAME loop body at `bridge/telegram_bridge.py:599-608`. Both `DM_WHITELIST.add(int(_entry["id"]))` (`:605`) and `DM_USER_TO_PROJECT[int(_entry["id"])] = _proj_cfg` (`:608`) are unconditional after the same gate (`:603`). The two structures cannot decouple in the existing loop. Furthermore, `assert` statements are stripped from bytecode under `python -O` — common in optimized production deployments — so the "hard-enforcement" claim is conditional on the harness running in non-optimized mode.
- **Suggestion**: Either (a) replace `assert` with `if set(...) != set(...): raise RuntimeError(...)` so it survives `-O`, or (b) drop Task 3.5 entirely and add a code comment at `:599` warning "DM_WHITELIST and DM_USER_TO_PROJECT must be populated as a unit; do not split this loop." Per Simplifier, option (b) is the Small-appetite path.
- **Implementation Note**: If keeping the runtime check, write it as `if set(DM_WHITELIST) != set(DM_USER_TO_PROJECT.keys()): raise RuntimeError(f"projects.json drift: ...")`. The launchd runner does not pass `-O`, but the worker's `python -m worker` invocations may; standalone worker is unrelated to bridge startup but the principle applies — never rely on `assert` for production invariants. Verify launchd plist by `grep -i 'optimize\|PYTHON_OPTIMIZE' /Library/LaunchDaemons/com.valor.bridge.plist 2>/dev/null` before relying on `assert`.

#### C5. Test Impact misses `tests/integration/test_message_routing.py:185-204` which encodes the obsolete `dm`-fallback session_id pattern

- **Severity**: CONCERN
- **Critics**: Consistency Auditor
- **Location**: Test Impact L156-160
- **Finding**: `tests/integration/test_message_routing.py` contains two tests that mirror the very pattern this plan eliminates: `test_session_id_format` (line 192) uses `project.get("_key", "dm")`, and `test_dm_session_id_uses_dm_key` (line 198-204) explicitly asserts `session_id == "tg_dm_987654321"`. The Test Impact bullet says "REVIEW: this file imports `find_project_for_chat` for routing tests. Verify no test asserts that an unresolved project still produces a memory record." — but the issue isn't memory records, it's session_id formation. After the `:1040` change to `project_key = project["_key"]`, these tests still pass (they exercise the helper API directly, not the bridge's runtime path), but they encode an obsolete pattern the rest of the codebase no longer uses. Future readers will be confused.
- **Suggestion**: Update the Test Impact bullet for `tests/integration/test_message_routing.py` to: "UPDATE: `test_session_id_format` (L192) and `test_dm_session_id_uses_dm_key` (L198-204) reference the `dm`-fallback pattern. Either delete `test_dm_session_id_uses_dm_key` (now-obsolete behavior) or rewrite it to assert the bridge early-returns on `project=None` instead of forming a `tg_dm_*` session_id."
- **Implementation Note**: `test_dm_session_id_uses_dm_key` should be deleted — the new bridge contract (`:1003 if not project: return`) means no `tg_dm_*` session_id is ever formed at runtime, so the test asserts a counterfactual. `test_session_id_format` (L185-196) doesn't depend on the `dm` fallback (its project is always resolved), so it can stay as-is; only update if the codebase wants to remove the misleading `"dm"` literal in the helper-test-style code. Decision applies to Task 4's "Where any existing test asserts `dm`-fallback behavior, update or delete it" — make these the explicit targets.
