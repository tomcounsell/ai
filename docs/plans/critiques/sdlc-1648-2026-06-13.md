## Blockers

### Spikes 1 & 3 are unresolved but load-bearing
- **Severity**: BLOCKER
- **Critics**: Skeptic, Consistency Auditor (independently flagged the same root issue)
- **Location**: Improvement #1 / spike-1; Technical Approach line 145 vs Problem line 24 / spike-3
- **Finding**: The plan references three spikes but contains no spike-result sections — they are unresolved. Two are load-bearing. Spike-1 (the cwd-slug algorithm) decides whether the tailer ever finds the JSONL at all; if the derived path is wrong, the plan's fail-silent contract yields permanently-zero counters with no signal the tailer is broken. Spike-3 decides whether the executor maps a granite `exception` exit to `completed` (asserted in Problem line 24) or `failed` (asserted in Technical Approach line 145) — the plan asserts BOTH in different sections, reconciled only by the unverified hypothesis "the adapter swallows exit_reason before it reaches that code."
- **Suggestion**: Resolve spike-1 and spike-3 and embed the results in the plan before build. Reword line 145 to match the verified path rather than asserting a behavior the spike was meant to confirm.
- **Implementation Note**: Verified against `main`: `BridgeAdapter.run()` returns `""` (no raised error), so `task.error` is falsy and `final_status = "completed" if not task.error` makes status `completed` on exception. Line 145's claim that the executor "correctly maps non-clean granite exits to `failed`" is contradicted by the verified code path. Spike-3 must trace the actual status flow and the plan must reconcile A/B (Problem vs Technical Approach) before any code touches `_publish_exit_summary` or the warning-chip logic.

### Transcript path must be selected by captured UUID; Dev UUID is never captured
- **Severity**: BLOCKER
- **Critics**: Adversary, Skeptic
- **Location**: Improvement #1 (path resolution) / Improvement #2 (identity capture) / container.py:834
- **Finding**: Two concurrent real sessions on the same repo share one `~/.claude/projects/{cwd-slug}/` directory holding two distinct `{uuid}.jsonl` files. The tailer must select by the session's own captured UUID, never "any/most-recent file in the directory," or it will sum the wrong session's telemetry (or double-count onto both records). But today only the PM `resume_uuid` is captured (`container.py:834`), inside a `finally` block gated on `pm_pty.isalive()`, and the Dev UUID is never captured at all. On a clean PM exit the PM is already dead, so `isalive()` is False and the path never resolves — permanently hollow. (Sandbox cwd is unique per-run via `run-{uuid4hex8}`, so the collision is specific to real-repo concurrent sessions.)
- **Suggestion**: Capture both PM and Dev transcript UUIDs at spawn / first-write (not in the dying `finally`), thread them as closed-over constants into the tailer, and match exactly the file whose UUID was captured — asserting a single match and aborting (logged WARNING) rather than falling back to an arbitrary file.
- **Implementation Note**: The capture site cannot be the `finally`-on-`isalive()` path; it must run while the PTY is live and known. If Claude Code generates the UUID only after first write, a brief startup discovery scan is acceptable but must assert exactly one matching file or abort the tailer.

### Reaction-gating behavior change ships on an unresolved Open Question
- **Severity**: BLOCKER
- **Critics**: User
- **Location**: Improvement #6 / Open Question #2 / Success Criteria
- **Finding**: Suppressing or changing the success reaction for `communicated=False` completions is a user-facing behavior change, but the governing decision (Open Q #2) is unresolved. Many legitimate autonomous sessions (background tasks, fire-and-forget builds) validly complete without messaging the operator; flagging them as anomalies is the inverse of the original false-success problem and erodes trust in the reaction signal.
- **Suggestion**: Ship the `communicated=False` chip now (non-destructive signal); gate or defer the reaction change. Either enumerate the session types where silent completion is expected and exclude them, or resolve Open Q #2 provisionally before build.
- **Implementation Note**: The chip already surfaces the information; the reaction change is a second, noisier signal. Provisional default: neutral reaction, escalate to `REACTION_ERROR` only when `exit_reason` is also non-clean. Decide before touching `session_executor.py:2001-2014` (the highest-risk edit per the plan's own Risk 3).

## Concerns

### Tailer re-reads whole JSONL every tick (O(n), unbounded growth)
- **Severity**: CONCERN
- **Critics**: Adversary
- **Location**: Improvement #1 — tailer read loop
- **Finding**: Re-reading from byte 0 every ~5s is O(n) per tick over an unbounded, growing file across a multi-hour session, multiplied by concurrent sessions; ticks can bunch and emit stale telemetry.
- **Suggestion**: Track byte offset + running totals in tailer state; `seek(offset)` and parse only new lines each tick.
- **Implementation Note**: State = `(byte_offset, running_totals)`. Per tick: seek to `byte_offset`, read to EOF, add deltas, `byte_offset = file.tell()`. Final synchronous pre-exit read seeks from the last offset to catch the tail.

### "Disjoint update_fields" holds only if the tailer excludes `updated_at`
- **Severity**: CONCERN
- **Critics**: Adversary (claim verified during aggregation — downgraded from BLOCKER)
- **Location**: Race 2 mitigation
- **Finding**: The Adversary raised a concurrent-save read-modify-write clobber risk between `_bump_last_turn_at` (worker thread) and the tailer (event loop). Verified against `popoto/models/base.py:137-147`: the `update_fields` path encodes the in-memory object, filters to only the listed fields, and issues a targeted `HSET key field val` for each — Redis HSET sets only those hash fields and does not touch others. So disjoint `update_fields` sets are genuinely safe at the Redis layer, which downgrades the BLOCKER to a CONCERN. The residual trap: the tailer must NOT include `updated_at` (written by `_publish_exit_summary`'s `["user_facing_routed","updated_at"]`) or `last_turn_at` (owned by `_bump_last_turn_at`) in its own set.
- **Suggestion**: Specify the tailer's `update_fields` as a strictly disjoint set excluding `updated_at` and `last_turn_at`.
- **Implementation Note**: Popoto filters `encode_popoto_model_obj(self)` to the listed fields, so non-listed fields are never written — the only failure mode is a shared field name across two writers.

### `exception`-as-anomaly risks Sentry alert fatigue
- **Severity**: CONCERN
- **Critics**: Operator
- **Location**: Improvement #4
- **Finding**: Adding `exception` unconditionally to the ERROR+Sentry path fires on every exception exit, including benign ones (network blips, upstream timeouts), causing alert fatigue that defeats the signal.
- **Suggestion**: Downgrade benign exceptions to WARNING (no Sentry) via a predicate; reserve ERROR+Sentry for exceptions with no prior turn activity / known-bad patterns.
- **Implementation Note**: `exception_is_benign(result)` evaluated BEFORE `logger.error` so the level can be downgraded, not suppressed: non-empty `result.turns` + recent activity ⇒ soft (WARNING); crashed before any output ⇒ hard (ERROR+Sentry).

### No signal confirms the tailer is actually reading
- **Severity**: CONCERN
- **Critics**: Operator
- **Location**: Improvement #1 — tailer self-observability
- **Finding**: Silent degradation to hollow reads is indistinguishable from the pre-fix hollow state, but operators will now TRUST non-zero cards and misread zero cards as "tailer broken." Nothing confirms the tailer is running.
- **Suggestion**: Emit `tailer_last_read_at` on every successful parse (≥1 event); a health check / reflection alerts if an active session's value is stale by >2× the poll interval.
- **Implementation Note**: Write the timestamp only on a successful parse yielding ≥1 event (not every tick) so staleness is meaningful; piggyback on the diff-gated partial save — no extra round-trip.

### Tailer on the event loop with a blocking Redis save stalls coroutines
- **Severity**: CONCERN
- **Critics**: Skeptic
- **Location**: Improvement #1 — tailer placement
- **Finding**: The plan runs the tailer on the worker event loop and persists "mirroring `_bump_last_turn_at`," but `_bump_last_turn_at`'s blocking `.save()` is safe precisely because it runs on the `to_thread` worker thread. A blocking `.save()` directly in the event-loop coroutine stalls all coroutines every ~5s per session.
- **Suggestion**: `await asyncio.to_thread(agent_session.save, update_fields=[...])` or use an async Redis client; never call blocking `.save()` directly in the coroutine body.
- **Implementation Note**: The pattern to mirror is the SAVE CALL, not the placement — its event-loop safety comes from the thread. Replicate via `await asyncio.to_thread(...)`.
