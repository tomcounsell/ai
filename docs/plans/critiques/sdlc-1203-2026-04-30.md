### Blockers (Round 1)

#### B1: `AgentSession.query.get(session_id)` will raise — wrong API + wrong field

- **Severity**: BLOCKER
- **Critics**: Archaeologist, Adversary
- **Location**: `## Solution` → "Technical Approach" item 4 (line 168-173); `## Step by Step Tasks` → Task 1 ("session lookup (best-effort)")
- **Finding**: The plan's session reconstruction snippet `AgentSession.query.get(session_id) if session_id else None` has two compounding defects:
  1. **Popoto's `query.get` is not Django-style.** Per `models/agent_session.py:781-815` and the canonical `get_by_id` helper, Popoto rejects positional string lookups; the helper exists precisely because callers must use `query.filter(id=...)[0]` or `query.filter(session_id=...)[0]`. A naked `query.get(session_id)` will raise `AttributeError` or `TypeError`.
  2. **`VALOR_SESSION_ID` is the bridge `session_id` field, not the `id` PK.** Per `agent/sdk_client.py:1362` and the existing `cmd_send` usage at `tools/valor_telegram.py:763`, `VALOR_SESSION_ID` carries the `AgentSession.session_id` (e.g. `"telegram-thread-...-...."`), not the Popoto primary key. Even if the API were `query.get(...)`, it would miss every record.
- **Impact**: The plan's outer try/except will swallow the exception and fall through with `session=None`. That keeps RTR functionally correct (verdict computes), but it **silently disables `session_events` for every Path B agent send** — exactly the observability story the plan sells in Risk #2 mitigation ("`rtr.bypassed`/`rtr.suppressed`/`rtr.trimmed` event counts on Path B will tell us whether Path B's value justifies the cost"). Without those events, the rollout decision in 30 days will have no Path B data at all.
- **Suggestion**: Replace the snippet in Technical Approach (4) and the matching Task 1 instruction with:
  ```python
  from models.agent_session import AgentSession
  session = None
  if session_id:
      try:
          matches = AgentSession.query.filter(session_id=session_id)
          session = matches[0] if matches else None
      except Exception as e:
          logger.warning("Path B RTR: session lookup failed for %s: %s", session_id, e)
          session = None
  ```
  Add an explicit Test Impact entry: `test_rtr_session_events_recorded_for_agent_caller` — assert that after a `trim` verdict for an agent-invoked send, `session.session_events` contains an `rtr.trimmed` entry. This is the regression guard that catches a silent fallback to `session=None`.
- **Implementation Note**: The lookup must not log on the empty-result case (i.e. `[]`) — that's the normal "session deleted between fork and rpush" path and shouldn't pollute warn logs. Only log on actual exceptions. Builder should mirror the lookup pattern used in `agent/sdk_client.py` for `_extract_sdlc_env_vars` callers.

#### C2: argparse mutex-group construct under-specified

- **Severity**: CONCERN
- **Critics**: Skeptic, Simplifier
- **Location**: Technical Approach item 2 (line 160-164), Step by Step Tasks Task 1 (line 359)
- **Finding**: The plan says "Add to a mutually-exclusive group inside argparse" but doesn't specify the group construction. Two flag-paired call sites (`--read-the-room` / `--no-read-the-room` as separate `add_argument` calls, then no group binding) is a common copy-paste defect that would let both flags pass simultaneously. The mutex test (`test_rtr_mutex_flags_error`) would then fail mysteriously.
- **Suggestion**: Specify the explicit construction in Task 1:
  ```python
  rtr_group = send_parser.add_mutually_exclusive_group()
  rtr_group.add_argument("--read-the-room", action="store_true", ...)
  rtr_group.add_argument("--no-read-the-room", action="store_true", ...)
  ```
- **Implementation Note**: The test should assert `argparse.ArgumentError` or non-zero exit (depending on argparse's failure mode) — verify the existing test pattern in `tests/unit/test_valor_telegram.py` for argparse failures.

#### C3: `asyncio.run` fail-open mitigation overstates safety

- **Severity**: CONCERN
- **Critics**: Adversary, Skeptic
- **Location**: Technical Approach item 3 (line 166-167), Risk 3 (line 235)
- **Finding**: The plan's mitigation for "future caller embeds `cmd_send` in an async context" claims `asyncio.run()` will raise `RuntimeError` and the outer try/except will catch it. That's true today on CPython 3.11. But: `asyncio.run` on Python 3.12+ also raises `RuntimeError` for *nested* event loops, *and* for the case where there's already a running loop in the same thread. Worse — if a future caller uses `nest_asyncio.apply()` (common in Jupyter contexts), `asyncio.run` may not raise at all but instead leak the loop or hang. The "fail-open is graceful" framing assumes the only failure mode is a clean exception.
- **Suggestion**: Tighten the mitigation to be specific: "If a future caller embeds `cmd_send` in an async context, `asyncio.run` will raise `RuntimeError: asyncio.run() cannot be called from a running event loop` on CPython 3.11+ without `nest_asyncio`. The outer try/except catches `Exception` so this is graceful for the supported call paths. **If a future use-case requires async-from-async invocation, refactor the RTR call into a `_run_rtr_sync(...)` helper that detects the running loop via `asyncio.get_event_loop().is_running()` and routes accordingly.**" Add a No-Gos entry: "Don't add `nest_asyncio` to support async-embedded `cmd_send` — the right fix is the helper refactor."
- **Implementation Note**: No code change in this PR; just sharpen the plan language so the next person doesn't think `asyncio.run` is robust to all async-from-async cases.

#### C4: `VALOR_SESSION_ID` env inheritance through nested shells

- **Severity**: CONCERN
- **Critics**: Adversary, Operator
- **Location**: Risk 1 mitigation (line 227-228), Technical Approach item 1 (line 158)
- **Finding**: The plan dismisses Risk 1 ("Mistaken caller-type detection") with: "Humans typing `valor-telegram send` in a terminal will never have it set unless they manually export it." But: the agent often uses Bash with shell substitutions, command chains (`&&`, `;`), or sub-shells (`bash -c "..."`) that inherit env vars. If a human launches a terminal *from inside* a Claude Code session (e.g. `tmux new-session` inside Bash, or `claude --resume` from within a session), `VALOR_SESSION_ID` propagates. False-positive RTR runs on what looks like a "manual" send. The plan's escape hatch (`--no-read-the-room`) works, but the human has to know they're in that situation.
- **Suggestion**: Acknowledge the false-positive vector explicitly in Risk 1. Document the diagnostic: `valor-telegram send` should print "(RTR active — running pre-send pass)" to stderr when RTR is about to run, so a human accidentally inheriting the env var sees the activation and can `Ctrl-C` and retry with `--no-read-the-room`. The Path A pre-send pass doesn't need this signal because Path A is always agent-driven; Path B straddles both populations and benefits from the announcement.
- **Implementation Note**: Two-line addition to Task 1 just before the `asyncio.run(read_the_room(...))` call: `print("(RTR active — running pre-send pass)", file=sys.stderr)`. Add a test asserting the stderr line is present when RTR runs and absent when it short-circuits.
