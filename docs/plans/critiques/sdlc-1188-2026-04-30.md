### Concerns (Round 1)

#### "10 lines of Telegram-send" understates real LOC

- **Severity**: CONCERN
- **Critics**: Skeptic, Simplifier
- **Location**: Solution / Technical Approach (line 119) — "Adding 10 lines of Telegram-send is strictly cheaper and equally effective."
- **Finding**: The plan claims the change is "10 lines" of Telegram-send, but the actual delta as scoped is closer to 50-70 LOC: `_collect_sentry_counts` helper (~15 LOC per OQ2 resolution), summary formatter with 12-finding cap and overflow footer (~10 LOC), Telegram-send block with four exception handlers (~10 LOC), the Sentry integration loop inside `run_log_review` (~5 LOC), plus the test file (~80-120 LOC of new code). The "10 lines" framing in the rabbit-holes argument-against-agent-conversion is rhetorical, not a forecast of code size. This understatement could mislead the builder into skipping a careful review pass when the change crosses 50 lines.
- **Suggestion**: Replace "10 lines of Telegram-send" with a more accurate forecast: "~50-70 LOC of additive subprocess-call blocks, fully bounded by existing patterns in `scripts/memory_consolidation.py:327` and `scripts/update/sentry_cli.py`." The argument that agent-reflection conversion is the wrong shape stands on its own without the LOC understatement.
- **Implementation Note**: When `build-log-review-builder` finishes Task 1 and runs `git diff --stat reflections/auditing.py`, expect 50-70 lines added (not 10). Ruff format may reflow some lines. Code review should treat this as a meaningful addition, not a one-liner — specifically verify each `subprocess.run(..., timeout=10, check=False)` call has a matching `try/except (FileNotFoundError, subprocess.TimeoutExpired, subprocess.CalledProcessError, Exception)` ladder, and that each exception branch logs via `logger.warning` (not `logger.info` or `logger.debug`).

#### Heartbeat ambiguity when `valor-telegram` itself is broken

- **Severity**: CONCERN
- **Critics**: Operator, Adversary
- **Location**: Open Questions resolution (OQ1 heartbeat) interacting with Failure Path Test Strategy
- **Finding**: OQ1 resolved to send a heartbeat on empty findings to confirm liveness. But the heartbeat is itself sent via `valor-telegram send`, which the plan instructs to swallow `FileNotFoundError`/`TimeoutExpired`/`CalledProcessError` with `logger.warning`. So if `valor-telegram` is broken (binary missing, bridge dead and Redis backlogged, etc.), the operator sees neither a heartbeat NOR a finding message — exactly the silent-failure mode the heartbeat was added to detect. The heartbeat resolves "scheduler dead vs. clean day" but does NOT resolve "Telegram path dead vs. clean day."
- **Suggestion**: Document the residual ambiguity explicitly in the plan. The heartbeat detects scheduler-dead and `run_log_review`-crash, NOT `valor-telegram`-broken. The detection signal for the latter is `logs/worker.log` containing the warning lines. Optional hardening (defer to follow-up): on a streak of N=3 consecutive empty-heartbeat-suppressed outcomes (warning logged but no exception raised), bump the next outcome to a stderr-only emergency message via a different channel. Out of scope for this PR; just document.
- **Implementation Note**: In the docstring update for `run_log_review` (Task 1), add a "Liveness signal" paragraph: "Sends a daily Telegram heartbeat even on empty findings to confirm the reflection executed. If the heartbeat is suppressed by a `valor-telegram` failure, the failure is logged via `logger.warning` to `logs/worker.log` — operators should `grep -E 'log-review|valor-telegram' logs/worker.log` if no Telegram message arrives for >24h." No code change beyond the docstring; the warning logging is already specified in the plan's failure-path test strategy.

The plan correctly reframes the issue from "create new agent reflection" to "extend existing function reflection," which is the right architectural call: the function form is already wired, already bounded-read-safe (PR #1056), and avoids the cost of spawning a Claude session for a daily summary. - OQ2 (ship Sentry helper): `sentry-cli` is already deployed by `scripts/update/sentry_cli.py` to every machine, so the runtime gate is purely defensive. Helper is small, gated, and matches the established subprocess-with-swallow precedent.
