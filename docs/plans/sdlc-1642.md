# sdlc stage-query: derive `--repo` for PR-state gh calls + distinguishable UNKNOWN merge state

## Problem

The SDLC pipeline runs from "local PM sessions" whose cwd is usually `~/src/ai`, while the
work being shepherded lives in another repo (e.g. `tomcounsell/popoto`). `sdlc-tool next-skill`
reads live PR merge state via `tools/sdlc_stage_query.py` to decide whether the terminal
merge guard **G6** fires and dispatches `/do-merge`.

**Current behavior:**
When `GH_REPO` is unset (the normal local-session case), `_fetch_pr_merge_state` runs
`gh pr view <n>` with **no `--repo` flag**, so `gh` resolves the PR number against the wrong
repo (`tomcounsell/ai`). This is a silent **cross-repo PR-number collision**: for popoto PR #419
the call resolves AI #419 — a real, MERGED PR — and returns `{"mergeStateStatus":"UNKNOWN",...}`
with exit 0. `pr_merge_state` becomes `"UNKNOWN"`, G6 sees `!= "CLEAN"` and does not fire, and the
router returns `Blocked(reason="no matching dispatch rule")`. The Blocked reason carries no
merge-state context, so a wrong-repo UNKNOWN is **indistinguishable** from a genuinely-unmergeable
PR. In the field (popoto#408 / PR popoto#419, 2026-06-12) this cost a manual diagnosis cycle;
re-running with `GH_REPO=tomcounsell/popoto` exported made G6 fire correctly.

**Desired outcome:**
1. PR-state `gh` calls resolve against the correct target repo even when `GH_REPO` is unset, using
   the same resolution ladder PR #1545 established for `find_plan_path` (`SDLC_TARGET_REPO` →
   git-toplevel → fallback).
2. When merge state genuinely cannot be resolved (`UNKNOWN`/`None`) for a known PR, the router's
   blocked output names that explicitly instead of the generic "no matching dispatch rule" — so the
   next failure is self-diagnosing.

## Why Previous Fixes Failed

| Prior Fix | What It Did | Why It Was Incomplete |
|-----------|-------------|-----------------------|
| PR #1545 (D1) | Made `find_plan_path` derive the plans directory from `SDLC_TARGET_REPO`/git-toplevel instead of assuming cwd | Scoped to the **plan-path** resolution only. **Root cause pattern:** SDLC tooling repeatedly assumes the process is running *inside* the target
repo (cwd or `GH_REPO`). When invoked from a local `~/src/ai` PM session shepherding another repo,
any path that doesn't explicitly resolve the target repo silently operates on the wrong repo. The
durable fix is to route every repo-dependent operation through one shared resolver.

## Data Flow

1. **Entry point**: PM session runs `sdlc-tool next-skill --issue-number N` (cwd `~/src/ai`, `GH_REPO` unset).
2. **`query_enriched` → `_compute_meta`** (`sdlc_stage_query.py:270`): resolves `pr_number` (via session attr / `_pr_number` meta / `_lookup_pr_number` → `_gh_pr_list`).
3. **`_fetch_pr_merge_state(pr_number)`** (`:128`): `gh pr view <n>` **with no `--repo`** → resolves against `tomcounsell/ai` → returns `("UNKNOWN", ...)` for a colliding PR number. `_meta["pr_merge_state"] = "UNKNOWN"`.
4. **Router `decide()`** (`agent/sdlc_router.py`): `guard_g6_*` sees `pr_merge_state != "CLEAN"` → returns None → no dispatch rule matches → `Blocked(reason="no matching dispatch rule", guard_id=None)`.
5. **Output**: PM sees a generic block with no merge-state hint → manual diagnosis.

The fix inserts a **repo resolver** between steps 2/3 and the `gh` subprocess, and enriches the
Blocked reason at step 4 when `pr_number` is set but merge state is unresolved.

## Solution

### Key Elements

- **`_resolve_target_repo()` helper**: returns `owner/name` (str) or None, using the same ladder as `find_plan_path`: `SDLC_TARGET_REPO` env → derive from git-toplevel via `gh repo view --json nameWithOwner` (or parse the remote) → None. Falls through on every failure so a missing env degrades to current behavior, not a crash.
- **`_fetch_pr_merge_state` gains `--repo`**: when the resolver returns a repo, the `gh pr view` call includes `--repo <owner/name>`. When it returns None, behavior is unchanged (preserves the in-repo case where cwd already is the target).
- **`_gh_pr_list` gains `--repo`**: same treatment so PR-number lookup also targets the right repo.
- **Distinguishable UNKNOWN**: the router emits a merge-state-aware Blocked reason when `pr_number` is set but `pr_merge_state` is `UNKNOWN`/`None`. This is the minimum-bar guarantee independent of whether repo derivation succeeds.

### Flow

PM runs `sdlc-tool next-skill` → `_compute_meta` resolves target repo → `gh pr view --repo <target> <n>` returns true merge state → G6 sees `CLEAN` → dispatch `/do-merge`.

Failure flow: repo unresolved AND `gh` returns UNKNOWN → router Blocked reason names "PR #N merge state UNKNOWN — could not resolve mergeability (check GH_REPO / target repo)" → operator fixes env instead of guessing.

### Technical Approach

- Reuse `_git_toplevel()` from `tools/_sdlc_utils.py`. Add `_resolve_target_repo()` — prefer placing it in `tools/_sdlc_utils.py` next to `find_plan_path` so both env-gap fixes share one resolver, and import it into `sdlc_stage_query.py`.
- Derivation from git-toplevel: run `gh repo view --json nameWithOwner -q .nameWithOwner` with `cwd=<toplevel>` (or `SDLC_TARGET_REPO`). This is robust to non-`origin` remote names. Cache within a single `_compute_meta` call to avoid double subprocess.
- Thread the resolved repo into both `gh` invocations as `--repo <repo>` **only when non-None**.
- Delete/replace the misleading comment at `sdlc_stage_query.py:185` — it asserts a contract that local sessions break.
- Router change: in `agent/sdlc_router.py`, where the final `Blocked("no matching dispatch rule")` is constructed, branch on `meta.get("pr_number")` being set while `meta.get("pr_merge_state")` is not `"CLEAN"` and is in `{None, "UNKNOWN"}` — emit a specific reason. Do NOT change G6's firing conditions (G6 must still require CLEAN).

## Rabbit Holes

- **Do not** rework `agent/sdk_client.py`'s `GH_REPO`/`SDLC_TARGET_REPO` injection — that path is correct for bridge-spawned sessions; the bug is strictly the local-session path where neither env is set. Touching sdk_client widens blast radius for no gain.
- **Do not** parse git remotes by hand (`git remote get-url` + regex) — `gh repo view --json nameWithOwner` already canonicalizes owner/name and handles SSH/HTTPS/forks. Hand-parsing is the tempting-but-wrong avenue.
- **Do not** make `gh pr view` failures loud/raise — silent-None-degrade is the existing, correct contract; only the *router's report* needs to get louder, and only when a PR number is known.

## Risks

### Risk 1: `gh repo view` adds a subprocess per `next-skill` call
**Impact:** Marginal latency on the hot router path.
**Mitigation:** Resolve once per `_compute_meta` and pass the result into both `gh` calls; `SDLC_TARGET_REPO` (set for bridge sessions) short-circuits before any subprocess.

### Risk 2: In-repo (cwd already is target) case regresses if resolver guesses wrong
**Impact:** A correct in-repo run starts passing `--repo` pointing elsewhere.
**Mitigation:** Resolver derives from the *current* git-toplevel, which IS the target in the in-repo case, so `--repo` matches the implicit default — no behavior change. Test covers it.
