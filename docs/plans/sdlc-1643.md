# Granite PTY Container: Purge Legacy PoC Framing

## Problem

The granite PTY container (`agent/granite_container/`) drives a PM + Dev pair of
real interactive `claude` TUIs over pseudo-terminals. It began as a
proof-of-concept (#1546) and was cut over to production in #1572 / #1612: it is
now the execution path for bridge-originated sessions under the standalone
worker. Per Development Principle #1 (NO LEGACY CODE TOLERANCE), PoC framing must
not survive a cutover.

**Current behavior:**

The production code, skill namespace, and persona prompts still describe the
system as a PoC / "kernel validation." The worst instance is functional, not
cosmetic: the prime personas in `.claude/commands/granite-poc/prime-{pm,dev}-role.md`
tell live sessions "You are the developer (Dev) persona for the granite-operator
interactive TUI PoC (issue #1546)... The PoC is a standalone kernel validation;
the bridge and worker are not in scope." This text is delivered verbatim into
real bridge-originated Telegram sessions running under the production worker,
in a worktree, doing real feature work. The persona actively lies to the session
about its own runtime context — a Dev could wrongly conclude bridge/worker code
is out of scope when it is exactly the deployment target.

**Desired outcome:**

All production code, the prime-command namespace, persona prompts, and feature
docs describe the container as what it now is: the production interactive-TUI
session runner. Persona prompts state the real runtime context (production
worker, bridge-originated session, worktree isolation, PM routing layer).
**Zero behavior change** — every edit is a string, path, comment, or prompt-text
change. No control flow is touched.

## Solution

### Key Elements

- **Persona prompts (priority):** Rewrite `prime-pm-role.md` and
  `prime-dev-role.md` to describe the production runtime truthfully — the
  container is the production session runner; the session is bridge-originated;
  it runs under the standalone worker in worktree isolation; the PM is a routing
  layer, not a standalone-PoC coordinator. Remove "standalone kernel validation"
  and "bridge and worker are not in scope."
- **Command namespace:** Rename `.claude/commands/granite-poc/` →
  `.claude/commands/granite/`; update the two `*_PRIME_SLASH_CMD` constants and
  every reference to `/granite-poc:prime-...`.
- **Backing package:** Rename `tools/granite_interactive_tui_poc/` →
  `tools/granite_loop/`; update the `pyproject.toml [project.scripts]` import
  path, `config/enums.py` docstring, and all importers/tests/docs.
- **Sandbox dir:** Rename `{tmp}/granite-poc` → `{tmp}/granite` in `container.py`.
- **Docstring/comment sweep:** Strip "PoC", "proof of concept", "kernel
  validation", "the production cutover can adopt" (cutover already happened) from
  the 5 source files.
- **Misleading log:** Fix the `container.py:820-823` log — it claims it is
  "nudging PM" but no nudge happens. Reword to describe what actually occurs
  (the stale-buffer guard detected unchanged PM output and falls through to a
  fresh idle read on the next cycle).
- **Feature docs:** Reframe `granite-interactive-tui.md`, `granite-agent-loop.md`,
  and the residual PoC phrasings in `granite-pty-production.md` to describe the
  production runner. Leave `docs/plans/completed/granite-*-poc*.md` as historical.

- Use `git mv` for the two directory renames (`.claude/commands/granite-poc/` and

  Summary.
- The CLI command name `valor-granite-loop` stays; only its `[project.scripts]`
  target import path changes. This avoids touching the many docs that reference
  the command name.
- The sandbox-dir rename is a one-line edit; verify no test asserts the literal

  `test_persona_priming.py`, `test_granite_classifier.py` reference it; update them).
- Persona-prompt rewrite is prose; keep the structural sections ("What you are
  NOT" / "What you DO" / "Operating scope") and correct only the framing claims.
