# Plan: Wedge-reap SIGKILL failure escalates to per-PID subtree kill + durable next-boot kill-list

- **Issue:** #2146
- **Slug:** `sdlc-2146`
- **Branch:** `session/sdlc-2146`
- **Worktree:** `.worktrees/sdlc-2146`

## Problem

When the session-health loop wedge-kills a timed-out tool call, the session
runner reaps the turn's `claude -p` process **group** via `killpg(pgid,
SIGKILL)` in `HeadlessSessionRunner._reap_turn_group` (issue #1938,
`agent/session_runner/runner.py:894`). Observed live 2026-07-17 14:09:17:

```
[runner] reap SIGKILL failed pgid=61593: [Errno 1] Operation not permitted
```

The "killed" process tree survived. The survivor was a **full integration test
suite** (`scripts/pytest-clean.sh tests/`) which — critically — had `setsid`'d
into its **own** process group (observed as `pgid 64644`, distinct from the
harness `pgid 61593`). A single `killpg(61593, …)` therefore never targeted the
pytest subtree at all, and the EPERM meant the harness group itself was not
positively reaped either. The orphan suite then ran unattended against the live
machine for 25+ minutes, signalling the production worker with SIGTERM restarts
and publishing fixture session-notify events to the live `valor:sessions:new`
channel until manually killed.

Two independent gaps combine here:

1. **`killpg` is process-group-scoped and misses `setsid` children.** A tool
   subprocess (a shell script wrapper like `pytest-clean.sh`, or `pytest -n
   auto` xdist workers) that starts a new session/group escapes `killpg` of the
   harness group. The reap must instead enumerate the **parentage** subtree
   (which `setsid` preserves) and kill per-PID.

2. **A reap that cannot confirm death is logged and forgotten.** Today
   `_reap_turn_group` returns `(False, pgid)` and `_record_reap_failed` emits a
   `runner_reap_failed` session-event marker (consumed by `session_executor.py`
   only to skip worktree deletion). There is no per-PID escalation, no verify,
   and no durable kill-list for the next boot's orphan-reap to finish the job.
   The boot-time cross-process reaper (`_reap_orphan_session_processes`, issue
   #1271) only recognises `claude`/MCP-named processes — a bash/pytest subtree
   is invisible to its name net.

**Desired outcome (from the issue):** reap failures escalate until verified dead
(snapshot the recorded harness subtree, per-PID SIGKILL sweep, then verify;
persist any survivors as a durable kill-list marker for the next boot), and the
boot-time orphan-reap actually cleans those persisted survivors. **Hard
constraint (Downstream):** kill logic must never target a process outside the
recorded session subtree — no new name-pattern sweeps beyond the existing
orphan-reap rules; PID-recycle safety via `create_time` guards throughout.

## Root-Cause Analysis of the EPERM (Acceptance Criterion #3)

`killpg(pgid, SIGKILL)` raising `EPERM` (`PermissionError`, errno 1) means the
kernel refused delivery to **at least one** member of the target group. The
plausible causes, in order of likelihood for the observed incident:

1. **Recycled / foreign pgid leader (most likely).** The harness group leader
   (`claude -p`, pid==pgid 61593) exited when the wedge-kill first hit it; macOS
   later recycled pgid 61593's *number* to an unrelated process owned by a
   different context, so a follow-up `killpg(61593)` targeted a group we no
   longer own → EPERM. This is consistent with the survivor carrying a
   **different** pgid (64644): the dangerous child had already `setsid`'d away
   and was never in 61593 to begin with.
2. **Different-euid member.** A test that shells out under `sudo` (or any member
   whose real/effective UID differs) makes the *whole* `killpg` fail with EPERM
   even for same-uid members.
3. **Zombie/reparented member race.** A member mid-reparent to `launchd`
   (ppid==1) during the kill.

**Conclusion:** regardless of which cause fired, group-scoped killing is the
wrong primitive for a subtree that can `setsid`. The fix is parentage-based
per-PID escalation with `create_time` recycle guards, which is correct under all
three causes. The postmortem is captured in the docs deliverable; no live repro
of the EPERM is required (the survivor tree was already manually cleaned on
2026-07-17), and manufacturing a cross-euid group in CI is a No-Go (see below).

## Acceptance Criteria

- [ ] **AC1 — Verified subtree death + per-PID escalation.** On a reap where the
  group SIGKILL raises EPERM/failure OR the post-kill confirm poll times out,
  the runner enumerates the recorded harness subtree (snapshotted while the
  harness was still alive) and issues a per-PID SIGKILL sweep, then re-verifies
  each PID. Each surviving PID is logged loudly (WARNING, one line per survivor).
- [ ] **AC2 — Durable next-boot kill-list, actually drained.** PIDs still alive
  after the per-PID sweep are persisted (with `create_time`, pgid, session ref)
  to a durable kill-list. At the next worker boot the cross-process orphan-reap
  drains the list, `create_time`-verifies each entry, SIGKILLs survivors, and
  removes drained/dead entries. Draining is scoped strictly to recorded PIDs
  (no name-pattern widening).
- [ ] **AC3 — EPERM root cause documented.** The root-cause analysis above is
  captured in a feature doc (postmortem), with the reasoning for the
  parentage-based fix.
- [ ] **AC4 — Unit test simulates pgid-kill failure and asserts escalation.**
  A unit test injects a `killpg` fake that raises `PermissionError` on the group
  kill, provides a fake subtree enumerator, and asserts (a) the per-PID sweep
  ran over exactly the snapshotted PIDs, (b) survivors were persisted to the
  kill-list, and (c) a matching boot-drain test kills + clears them under a
  `create_time` match and skips them on a `create_time` mismatch (recycle).

## Design

### Component map

| Concern | Location | Change |
|---------|----------|--------|
| Wedge/teardown reap | `agent/session_runner/runner.py::_reap_turn_group` | Snapshot subtree pre-kill; add per-PID escalation on failure; return survivors |
| Reap-failed marker | `agent/session_runner/runner.py::_record_reap_failed` | Extend payload with `survivor_pids`; persist to durable kill-list |
| Durable kill-list | **new** `agent/reap_killlist.py` | `add(entries)` / `drain_and_kill()` helpers over a plain Redis key (non-Popoto infra key), `create_time`-guarded |
| Boot orphan-reap | `agent/session_health.py::_reap_orphan_session_processes` (#1271) | Call `reap_killlist.drain_and_kill()` as a first, recorded-PID-only pass |
| Worker startup | `worker/__main__.py` Step 4 (orphan reap) | Surfaced via the reaper it already calls — no new step needed |

### Reap escalation flow (`_reap_turn_group`)

1. **Snapshot before killing (the load-bearing change).** While the harness pid
   is still alive, capture the descendant set:
   `psutil.Process(handle.pid).children(recursive=True)` → list of
   `(pid, create_time)`. `setsid` changes pgid but preserves ppid, so this walk
   catches the pytest/xdist subtree that `killpg` misses. Snapshot is best-effort
   and fail-silent (psutil `NoSuchProcess`/`AccessDenied`/`ImportError` → empty
   snapshot; the existing group kill still runs).
2. `killpg(pgid, SIGKILL)` and confirm-poll exactly as today (unchanged happy
   path — near-instant death, no escalation, no snapshot cost beyond the one
   `children()` call).
3. **On EPERM or unconfirmed group death:** iterate the snapshot; for each
   `(pid, create_time)` still alive (re-checked via psutil with a `create_time`
   match to guard against PID recycle), `os.kill(pid, SIGKILL)` through an
   injected `self._kill` seam. Re-verify; collect stragglers.
4. Return `(confirmed_dead, pgid, survivors)` where `survivors` is the list of
   `(pid, create_time, pgid)` still alive after the per-PID sweep (empty on the
   happy path). Signature extended; the single caller updated.

All new work stays synchronous / `await`-free to preserve the #1938
cancellation-proof guarantee (the recovery path double-cancels this coroutine).
Every signal goes through injected `self._killpg` / `self._kill` seams and a
`self._enum_subtree` seam so the unit test drives outcomes with fakes and never
touches real processes.

### Durable kill-list (`agent/reap_killlist.py`)

- Backed by a plain Redis key `valor:reap:killlist` (a hash `pid -> json({pid,
  create_time, pgid, session_ref, ts})`). This is worker infrastructure state,
  **not** a Popoto-managed model key, so it follows the existing
  `worker:registered_pid:*` / heartbeat-key precedent and does not trip the
  raw-Redis validator (which guards Popoto model keys only). Access via the
  shared Redis client used elsewhere in `agent/`.
- `add(entries)` — best-effort persist of survivors (fail-silent; a reap must
  never crash on persistence).
- `drain_and_kill() -> int` — read all entries, for each: reconstruct
  `psutil.Process(pid)`, verify `create_time()` matches the stored value
  (recycle guard, mirroring `_pending_sigkill_orphans` semantics at
  `session_health.py:273`); if match and alive → `os.kill(pid, SIGKILL)`; then
  remove the entry unconditionally (one-shot drain — a mismatched/dead entry is
  simply discarded). Returns count killed. Fail-silent per-entry.
- TTL on the key (e.g. 24h) so a machine that never reboots does not accumulate
  stale entries indefinitely.

### Boot integration

`_reap_orphan_session_processes` (issue #1271, `session_health.py:5093`) gains a
first pass that calls `reap_killlist.drain_and_kill()` and folds its count into
the returned orphan total. This runs at worker startup Step 4 (which already
calls the reaper via `_cleanup_orphaned_claude_processes`) — no new startup step,
no new call site in `worker/__main__.py`. The drain is recorded-PID-only and
`create_time`-guarded, satisfying the "no name-pattern sweeps" constraint.

## Failure Path Test Strategy

- **EPERM on group kill → per-PID escalation (AC1/AC4).** Unit-test
  `_reap_turn_group` with a fake `killpg` that raises `PermissionError` on the
  SIGKILL, a fake `_enum_subtree` returning a known `[(pid, ctime), …]`
  snapshot, and a fake `_kill`/`pid_alive` pair. Assert the per-PID sweep
  targeted exactly the snapshot PIDs and that unkilled PIDs are returned as
  survivors + persisted.
- **Group SIGKILL succeeds → no escalation (regression guard).** Fake `killpg`
  succeeds and confirm-poll reports dead on first probe; assert `_enum_subtree`
  is invoked at most once (the pre-kill snapshot) and the per-PID sweep and
  `add()` are never called — the happy path is untouched.
- **Boot drain kills under create_time match (AC2).** Unit-test
  `drain_and_kill` with a fake psutil whose `create_time()` matches the stored
  entry → assert `os.kill` called and entry removed.
- **Boot drain skips on create_time mismatch (recycle safety).** Stored
  `create_time` differs from live process → assert `os.kill` NOT called and the
  stale entry removed (no collateral kill of a recycled PID). Directly exercises
  the Downstream "never target outside the recorded subtree" constraint.
- **All-fail-silent.** psutil `ImportError`/`AccessDenied` during snapshot,
  Redis unavailable during `add`/`drain` → the reap/boot path completes without
  raising (assert no exception propagates).

## Rabbit Holes

- **Do NOT rewrite `killpg` group-kill out.** The group kill stays as the fast
  happy path; escalation is strictly additive on failure. Ripping it out would
  regress the common near-instant case and re-litigate #1938.
- **Do NOT try to live-reproduce the EPERM in CI.** Manufacturing a cross-euid
  or recycled-pgid group is flaky and root-privilege-adjacent. The root cause is
  established by analysis (see above) and the fix is correct under all branches;
  tests exercise the escalation via injected fakes, not real EPERM.
- **Do NOT widen the boot orphan-reap's name net.** The issue's constraint is
  explicit: only recorded PIDs (via the durable kill-list) get the widened
  treatment; the claude/MCP name regexes stay exactly as they are.
- **Do NOT block turn teardown on a long per-PID verify.** Keep the escalation
  verify bounded (reuse the existing `REAP_CONFIRM_TIMEOUT_S` budget); anything
  still alive after the bounded sweep goes to the durable kill-list — that is the
  designed hand-off, not a bug to poll around.

## No-Gos

- No `await` inside `_reap_turn_group` or the escalation path (breaks the #1938
  cancellation-proof invariant).
- No raw Redis mutation of Popoto-managed keys; the kill-list is a dedicated
  non-model infra key accessed via the shared client.
- No SIGKILL of any PID not present in the snapshotted harness subtree or the
  durable kill-list, and never without a `create_time` recycle guard.
- No CI test that spawns real long-running subprocesses or requires elevated
  privileges.
- No changes to the wedge-kill *decision* (tool-timeout logic in
  `session_health.py`) — this issue is strictly about the *reap* after the
  decision is made. The declared-timeout decision bug is a filed sibling.

## Update System

No update-system changes required. The change is internal worker/runner code
plus one new intra-repo module (`agent/reap_killlist.py`) and a new Redis key
that is created lazily at runtime. No new dependency (psutil and the Redis
client are already in use across `agent/`), no new config file, no new plist,
and nothing for `/update` or `scripts/remote-update.sh` to propagate beyond the
normal code sync. The kill-list key is self-managing (TTL + one-shot drain), so
existing installations need no migration step.

## Agent Integration

No agent integration required. This is a bridge/worker-internal change on the
reap path; there is no new user-facing capability, no CLI entry point in
`pyproject.toml [project.scripts]`, and the bridge does not import the new code.
The runner already invokes `_reap_turn_group` internally on every turn teardown.
Verification is via the unit tests above (injected-fake escalation + boot-drain),
which is the correct surface for a reap-internal behavior — an integration test
would require spawning real detached subprocesses, which the No-Gos forbid.
