# Prefetch memories on UserPromptSubmit instead of waiting for tool cycle 3

## Problem

A fresh Claude Code session must execute three tool calls before any memory recall happens. Recall logic in `.claude/hooks/hook_utils/memory_bridge.py:172-323` buffers tool calls in a sliding window, fires recall every `WINDOW_SIZE=3` tool calls, extracts keywords from accumulated tool **names + tool inputs** (not from the user's prompt), and surfaces up to `MAX_THOUGHTS=3` memories via `additionalContext`.

**Current behavior:**
- A short session (1–2 tool calls) gets zero memory context.
- A long session operates blind for the first ~3 operations — exactly when prior context would be most useful for orienting.
- Relevance signal is "what tools the agent recently ran." Tool choices are an *output* of the agent's reasoning, not the *input*. A user prompt like "fix the auth flow that broke after PR #800" surfaces nothing on turn one — it must wait for the agent to run tools whose names happen to overlap.

**Desired outcome:**
The agent receives up to 3 relevant memory thoughts **on the first turn**, scored against the user's actual prompt, before it ever runs a tool. Subsequent PostToolUse recall sweeps continue as today, but skip memory IDs already prefetched on this prompt.

## Data Flow

Two parallel data flows, one per call site:

**Flow A — Hooks-side (.claude/hooks/user_prompt_submit.py path):**
1. **Entry:** Claude Code fires `UserPromptSubmit` hook with `{prompt, cwd, session_id}` on stdin.
2. `user_prompt_submit.py::main()` reads input, calls `ingest(prompt, cwd)` (existing).
3. **NEW:** After ingest, calls `prefetch(session_id, prompt, cwd)` (new export from `memory_bridge`).
4. `prefetch()` loads sidecar at `data/sessions/{session_id}/memory_buffer.json`, applies `MIN_PROMPT_LENGTH` and `TRIVIAL_PATTERNS` gates, calls `_recall_with_query(query=prompt, project_key=..., exclude_ids=set())`.
5. `_recall_with_query()` runs the existing pipeline: bloom pre-check → `retrieve_memories(query_text=prompt, ...)` → category-weighted re-rank.
6. Top `MAX_THOUGHTS=3` records become `<thought>` blocks. Their `memory_id` values are appended to `state["injected"]` and the sidecar is saved.
7. `prefetch()` returns the joined string. `user_prompt_submit.py` prints `{"hookSpecificOutput": {"hookEventName": "UserPromptSubmit", "additionalContext": "..."}}` on stdout (if non-empty).
8. **Output:** Claude Code prepends the thought blocks to the agent's first system message, before any tool runs.
9. Later: PostToolUse fires every 3 tool calls, calls `recall(...)` (existing), which now reads the **same sidecar** and skips IDs in `injected[]` from prefetch — no double-surfacing.

**Flow B — SDK-side (agent/memory_hook.py path, worker-spawned subprocesses):**
1. **Entry:** Worker calls `agent/sdk_client.py` to spawn the SDK subprocess with `setting_sources=["user", "local", "project"]` (which loads `.claude/settings.json`).
2. The subprocess fires `UserPromptSubmit` on the initial prompt → Flow A runs inside the subprocess and writes prefetch IDs to the sidecar.
3. Tool calls begin. The SDK harness invokes `agent/health_check.py::watchdog_hook` (PostToolUse).
4. `watchdog_hook` resolves `session_id = AGENT_SESSION_ID env or claude_uuid`, then calls `agent/memory_hook.check_and_inject(session_id, tool_name, tool_input, project_key)`.
5. `check_and_inject()` increments process-local counter; on `count % WINDOW_SIZE == 0`, runs bloom + RRF.
6. **NEW:** Before formatting `<thought>` blocks, `check_and_inject()` reads the hooks-side sidecar (`data/sessions/{claude_uuid}/memory_buffer.json`) and excludes IDs in its `injected[]` list. The sidecar is keyed by `claude_uuid` (input_data.session_id), distinct from the SDK's `agent_session_id` — `check_and_inject()` therefore needs both keys passed in (or a sidecar-key parameter).
7. **Output:** SDK-side `<thought>` blocks via `additionalContext`, with no overlap against prefetch.

**Why two flows, not one:** `.claude/hooks/post_tool_use.py` and `agent/health_check.py::watchdog_hook` *both* fire on PostToolUse for SDK-spawned subprocesses (the former from `.claude/settings.json`, the latter from the SDK's `hooks=build_hooks_config()`). Both run their own bloom+RRF lookup. This pre-existing duplication is out of scope; the plan handles it by having both sides honor the same sidecar `injected[]` list as the source of truth for "already shown."

## Architectural Impact

- **New dependencies:** None. All pieces (`retrieve_memories`, bloom field, sidecar I/O, category weights) already exist.
- **Interface changes:**
  - `memory_bridge.recall(session_id, tool_name, tool_input, cwd)` keeps its signature (PostToolUse caller is unchanged).
  - **New export:** `memory_bridge.prefetch(session_id, prompt, cwd) -> str | None`.
  - **New private helpers:** `memory_bridge._recall_with_query(query, project_key, exclude_ids, *, bloom_check_emit_dejavu, ...) -> list[Memory] | str`, `memory_bridge._format_thought_blocks(records) -> str`, `memory_bridge._strip_pm_boilerplate(prompt) -> str`. Both `recall()` and `prefetch()` wrap them.
  - **Modified:** `agent/memory_hook.check_and_inject(session_id, tool_name, tool_input, project_key=None, claude_uuid=None)` — new optional `claude_uuid` param so it can read the hooks-side sidecar at `data/sessions/{claude_uuid}/`. Read is gated on `claude_uuid` being non-empty and not equal to `"unknown"`. No SDK-side prefetch export added — `.claude/hooks/user_prompt_submit.py` already covers worker-spawned subprocesses (which inherit `.claude/settings.json`).
  - **New constant:** `config/memory_defaults.PREFETCH_LATENCY_WARN_MS = 200`.
- **Coupling:** Hooks-side and SDK-side now coordinate via the JSON sidecar. The sidecar becomes the single source of truth for "already shown memory IDs" within a session. This is a small increase in coupling (SDK side reads a hooks-side artifact) but well-contained: if the sidecar is unreadable, both sides fail silent and the worst case is duplicate thoughts (harmless degradation).
- **Data ownership:** Sidecar `injected[]` was previously written only by hooks-side `recall()`. After this change, `prefetch()` also writes to it, and SDK-side `check_and_inject()` reads from it. Sidecar shape is unchanged.
- **Reversibility:** High. Revert is one commit — restore the old `recall()` signature, remove `prefetch*()` exports, remove the sidecar-aware exclude in `check_and_inject()`. No data migration; sidecar is per-session ephemeral.

## Solution

### Key Elements

- **`_recall_with_query` helper**: Pure retrieval — takes an explicit query string, project key, and exclude-ID set; returns a list of `Memory` records ranked by bloom + BM25 + RRF + category weights. Both `recall()` (today's tool-buffer caller) and `prefetch()` (new prompt caller) wrap it. Accepts a `bloom_check_emit_dejavu: bool = True` flag so prefetch can opt out of the noisy novel-territory thought.
- **Hooks-side `prefetch()`**: New export in `memory_bridge`. Called from `user_prompt_submit.py` after `ingest()`. Applies `MIN_PROMPT_LENGTH`/`TRIVIAL_PATTERNS` gates, **strips PM-style `FROM:`/`SCOPE:`/`MESSAGE:` boilerplate** when present, runs `_recall_with_query(prompt, ..., bloom_check_emit_dejavu=False)`, formats `<thought>` blocks, **load-modify-saves** sidecar `injected[]` (preserves `count`/`buffer` verbatim), logs a warning if elapsed exceeds `PREFETCH_LATENCY_WARN_MS`, returns the joined string.
- **Sidecar-aware exclude in PostToolUse `recall()`**: When formatting thoughts, skip records whose `memory_id` is in `state["injected"]`. Today, `injected[]` is write-only — promoting it to read+write is the de-dup mechanism.
- **Sidecar-aware exclude in SDK `check_and_inject()`**: When a non-empty `claude_uuid` (other than the literal sentinel `"unknown"`) is passed, also load `data/sessions/{claude_uuid}/memory_buffer.json:injected[]` and merge into the SDK-side exclusion set. Fail-silent on read errors. No new SDK-side prefetch export — the `.claude/hooks/user_prompt_submit.py` path already covers worker-spawned subprocesses (which inherit `.claude/settings.json`); de-dup coordination is the only SDK-side change required.
- **No retrieval-pipeline changes**: bloom check, BM25, RRF, category weights are unchanged. The novel-territory deja vu signal stays in `recall()` (tool-buffer caller); it is **suppressed on the prefetch path** to avoid surfacing noisy "this is new territory" thoughts on the user-visible first turn (per Issue #627 lesson).

### Flow

User submits prompt → UserPromptSubmit hook fires → `ingest()` saves prompt as Memory → `prefetch()` runs `_recall_with_query(prompt)` → up to 3 `<thought>` blocks injected via `additionalContext` (with their IDs written to sidecar `injected[]`) → Agent processes prompt with that context → Agent runs tool 1, tool 2, tool 3 → PostToolUse fires → `recall()` runs tool-buffer query → filters out memory IDs already in sidecar `injected[]` → injects up to 3 fresh thoughts (or fewer if the buffer query yields overlap) → cycle continues.

### Technical Approach

1. **Refactor first, add second.** Split `recall()` into `_recall_with_query()` + thin wrapper before adding either new call site. This keeps the diff reviewable: refactor commit (no behavior change) → prefetch commit → SDK-side exclusion commit. (See `Step by Step Tasks` for ordering.)

2. **`_recall_with_query()` signature:**
   ```python
   def _recall_with_query(
       query: str,
       project_key: str,
       exclude_ids: set[str] | None = None,
       *,
       bloom_check: bool = True,
       bloom_check_emit_dejavu: bool = True,
       max_results: int = MAX_THOUGHTS,
   ) -> list[Memory] | str:
       """Pure retrieval. No sidecar I/O, no <thought> formatting.

       When bloom_hits == 0:
         - if bloom_check_emit_dejavu and ≥7 unique tokens: returns the deja vu string
         - otherwise: returns []
       """
   ```
   Returns hydrated, RRF-fused, category-weighted, exclude-filtered Memory records. `recall()` (tool-buffer caller) passes `bloom_check_emit_dejavu=True` — preserves today's behavior. `prefetch()` passes `bloom_check_emit_dejavu=False` — suppresses the noisy fallback on the user-visible first turn.

3. **Prompt-as-query is single-shot, not multi-cluster.** The existing tool-buffer recall clusters keywords (multi-query decomposition). For the prompt path, the prompt itself is a coherent query — we run `retrieve_memories(query_text=prompt, ...)` once, no clustering. Clustering exists to cope with the *noise* in tool-call keywords (path segments, etc.), which prompts don't have.

4. **Bloom gate on prefetch:** keep it for parity — extract a few coarse tokens from the prompt (split on whitespace, drop noise words, take the top N). If zero bloom hits, `prefetch()` returns `None` immediately. **No deja vu fallback from the prefetch path** — Issue #627 confirmed that "this is new territory" thoughts on the user-visible turn are pure noise. (See No-Gos.)

5. **PM-prompt boilerplate stripping.** Worker-spawned PM/Teammate sessions receive prompts wrapped with `FROM: ...\nSCOPE: ...\nMESSAGE: ...` boilerplate. Before passing the prompt to `_recall_with_query`, `prefetch()` checks for the multiline pattern `^FROM:.*\nSCOPE:.*\nMESSAGE:\s*` and, when present, uses only the substring after `MESSAGE:` for the BM25 query. This avoids the dilution risk where short user messages get out-ranked by a long boilerplate template.

6. **`additionalContext` shape on UserPromptSubmit.** Per Claude Code hook docs, the response shape is `{"hookSpecificOutput": {"hookEventName": "UserPromptSubmit", "additionalContext": "..."}}` (matches the form already used in `agent/health_check.py:594-599`). The existing `.claude/hooks/post_tool_use.py:436` uses the older `{"additionalContext": "..."}` form, which still works but is less explicit. Use the explicit `hookSpecificOutput` form for the new prefetch caller. Confirmed by **spike-1**.

7. **Sidecar load-modify-save discipline.** `prefetch()` and `recall()` both write to the same sidecar file at `data/sessions/{session_id}/memory_buffer.json` with shape `{count, buffer, injected}`. `recall()` owns `count` and `buffer`. `prefetch()` only mutates `injected`. Both perform a load-modify-save: `state = _load_sidecar(...); state["injected"] = state.get("injected", []) + new_entries; _save_sidecar(...)`. Atomic rename protects against partial writes; last-writer-wins on `injected[]` is acceptable (worst case: one cycle of duplicate thoughts).

8. **`claude_uuid != "unknown"` guard.** `agent/health_check.py:535` defaults `claude_uuid` to the literal string `"unknown"` when `input_data["session_id"]` is absent. To prevent every malformed-payload session from sharing `data/sessions/unknown/memory_buffer.json` (cross-session contamination), `check_and_inject()` skips the sidecar read whenever `claude_uuid` is empty or equals `"unknown"`. Fail-silent: missing sidecar is degraded (no de-dup) but correct.

9. **Latency budget.** `prefetch()` measures wall-clock around the `_recall_with_query` call. If elapsed exceeds `PREFETCH_LATENCY_WARN_MS = 200` (new constant in `config/memory_defaults.py`), emits `logger.warning(...)`. This mirrors the existing 15ms warning in `memory_bridge.py:277-281` for the multi-cluster recall path; 200ms is a budget appropriate to a single-call user-facing query. Operator can grep logs for the warning to detect silent prod regressions.

10. **Soft-dep on #1178:** prefetch emits full bodies (`<thought>{content}</thought>`), same as `recall()` today. When #1178 lands, it sweeps `recall()` and `prefetch()` together via the shared `_format_thought_blocks` helper. We don't fork the format.

11. **Failure mode:** silent. Every new code path is wrapped in try/except → return None. Memory failures must never block prompt submission.

## Spike Results

### spike-1: UserPromptSubmit additionalContext shape
- **Assumption:** "UserPromptSubmit hook can return `additionalContext` via `{\"hookSpecificOutput\": {\"hookEventName\": \"UserPromptSubmit\", \"additionalContext\": \"...\"}}`."
- **Method:** code-read (search Claude Code SDK + hook docs in repo + agent/health_check.py for existing patterns)
- **Finding:** Confirmed. The shape used in `agent/health_check.py:594-599` for PostToolUse — `{"hookSpecificOutput": {"hookEventName": "PostToolUse", "additionalContext": memory_context}}` — is the documented modern shape. The hookEventName field disambiguates by hook type, so swapping `PostToolUse` → `UserPromptSubmit` is the correct invocation. The legacy bare `{"additionalContext": "..."}` shape used in `.claude/hooks/post_tool_use.py:436` is also accepted but undifferentiated.
- **Confidence:** high
- **Impact on plan:** prefetch emits the explicit `hookSpecificOutput` form. No new dependency on the legacy bare shape.

### spike-2: hooks-side and SDK-side session_id keys
- **Assumption:** "The hooks-side sidecar at `data/sessions/{X}/memory_buffer.json` and the SDK-side `_injected_thoughts[Y]` use the same key `X == Y`, so the SDK can read the sidecar by its own session_id."
- **Method:** code-read (`agent/health_check.py:535-537` + `.claude/hooks/hook_utils/memory_bridge.py:_get_sidecar_dir` + `.claude/hooks/user_prompt_submit.py:48`)
- **Finding:** **DISPROVEN.** The hooks-side sidecar is keyed by Claude Code's session UUID (`hook_input["session_id"]`). The SDK-side `_injected_thoughts` is keyed by `AGENT_SESSION_ID env var or claude_uuid` — the SDK strongly prefers `AGENT_SESSION_ID`. They are different. To make the SDK side honor the sidecar, `check_and_inject()` must accept an optional `claude_uuid` parameter and the watchdog hook must pass `input_data["session_id"]` as that argument.
- **Confidence:** high
- **Impact on plan:** added a `claude_uuid` parameter to `check_and_inject()` and `prefetch_for_prompt()`. Without this, the SDK side cannot find the sidecar for de-dup.

### spike-3: existing buffer-clustering vs prompt single-query trade-off
- **Assumption:** "A user prompt is high enough signal to skip multi-query keyword clustering."
- **Method:** code-read (`utils.keyword_extraction._cluster_keywords` + `agent/memory_retrieval.retrieve_memories`)
- **Finding:** Clustering only matters when extracted keywords are noisy (file path segments, tool names). User prompts already form a coherent query string. `retrieve_memories(query_text=prompt, ...)` runs BM25 over the full prompt directly — no clustering needed. Confirmed by examining how the BM25Field tokenizer in `models.memory` indexes content (it tokenizes on whitespace and punctuation, so a multi-word prompt is a perfectly valid query).
- **Confidence:** high
- **Impact on plan:** prefetch is single-call, not multi-cluster. Lower latency, simpler code.

- **Preempting #1178's stub format.** Don't speculate. Emit full bodies; #1178 sweeps both call sites later.
- **Adding analytics for prefetch hit rate.** Tempting but adds scope. The existing `analytics.collector.record_metric("memory.recall_attempt", ...)` already fires inside `retrieve_memories()` for every call; that's good enough to measure prefetch via `query_text` correlation if needed.

## Risks

### Risk 1: Prefetch latency on the user-facing path
**Impact:** UserPromptSubmit blocks the agent's first response. Adding a 100–500ms BM25+RRF+embedding query before *any* agent reply degrades perceived responsiveness. Issue #627 documented a 344ms import tax on the hooks-side; prefetch hits the same import path.
**Mitigation:** (a) The prefetch runs *after* `ingest()` already imports `models.memory`, so the import tax is paid once per hook invocation regardless. (b) Existing `retrieve_memories()` budget is 15ms warning threshold (`memory_bridge.py:277`); we keep that. (c) Trivial / short-prompt gates short-circuit before any retrieval. (d) If observed first-turn latency exceeds 500ms in practice, fall back to async fire-and-forget with the prefetch result delivered on the *second* turn — but only if measurement shows we need it. The `recall()` path already runs synchronously every 3 tool calls without complaint, so the budget is reasonable.

### Risk 2: De-dup race between prefetch and PostToolUse cycle 1
**Impact:** UserPromptSubmit prefetch writes to sidecar; PostToolUse's first sweep at tool call 3 reads the sidecar. If the agent runs three tools fast enough, both could be in flight. Sidecar atomic write (`tmp + rename`) protects the file, but the read inside `recall()` still races with the prefetch write if the timing is unlucky.
**Mitigation:** (a) The atomic rename pattern in `_save_sidecar` (`memory_bridge.py:116-132`) means readers see either the pre-prefetch or post-prefetch state — never a partial write. (b) Worst-case staleness is one sweep cycle: PostToolUse cycle 1 might re-surface up to 3 IDs that prefetch already showed. The session-level `injected[]` list converges by cycle 2. (c) The duplicate-thought failure mode is harmless (waste of context tokens, not a correctness bug). Acceptable.

### Risk 3: SDK-side `claude_uuid` plumbing breaks under direct CLI
**Impact:** Direct CLI invocations (no worker, no `AGENT_SESSION_ID`) already use `claude_uuid` as their `session_id`. Passing `claude_uuid` again as a separate arg risks key confusion if call sites mix them up.
**Mitigation:** Make `claude_uuid` parameter purely optional and *strictly* used for sidecar reads (never for `_injected_thoughts` keying). When absent, the sidecar exclude is skipped — same behavior as today. Add a unit test asserting that `check_and_inject(session_id, ..., claude_uuid=None)` behaves identically to today.

### Risk 4: Soft-dep #1178 lands first and changes the format mid-build
**Impact:** If #1178 ships before this plan finishes building, the hook-side stub format would diverge from the prefetch output, causing format mismatch in test snapshots.
**Mitigation:** Both call sites use the same `_format_thought_blocks(records)` helper (introduced in step 2 of the build). When #1178 modifies that helper, both prefetch and recall format identically. Coordinate with #1178 owner: if #1178 lands during the build window, rebase and adopt the new helper signature.

## Race Conditions

### Race 1: Prefetch writes sidecar, PostToolUse reads sidecar (within same session)
**Location:** `.claude/hooks/hook_utils/memory_bridge.py:_save_sidecar` (line 116) and `_load_sidecar` (line 91). Triggered when UserPromptSubmit and PostToolUse hooks run in close succession.
**Trigger:** UserPromptSubmit hook process A calls `prefetch()` → calls `_save_sidecar()`. Concurrently or shortly after, the agent fires three tool calls; PostToolUse hook process B (third call) reads the sidecar via `_load_sidecar()`.
**Data prerequisite:** Process A's `_save_sidecar` must complete its `tmp + rename` before process B opens the sidecar for read. Hooks are subprocesses with no shared memory — the only synchronization is the atomic filesystem rename.
**State prerequisite:** Sidecar contains `injected: [...]` reflecting the prefetched IDs.
**Mitigation:** `_save_sidecar` already uses tmp + rename (atomic on POSIX) — readers see the old or new file, never a partial. Worst-case staleness: PostToolUse reads pre-prefetch sidecar and re-surfaces up to 3 IDs already in cycle-1 prefetch. This converges by cycle 2 because PostToolUse *also* writes to `injected[]`. Acceptable.

### Race 2: SDK `check_and_inject()` reads sidecar while hooks-side `recall()` is mid-write
**Location:** `agent/memory_hook.py::check_and_inject` (new sidecar-read code) vs. `.claude/hooks/hook_utils/memory_bridge.py:_save_sidecar`.
**Trigger:** SDK PostToolUse fires while hooks-side PostToolUse is in flight. (Both fire on every tool call — they are separate processes.)
**Data prerequisite:** `data/sessions/{claude_uuid}/memory_buffer.json` exists and is currently being rewritten.
**State prerequisite:** N/A — sidecar reads are best-effort.
**Mitigation:** Same atomic rename pattern. If the rename is mid-flight, the SDK side reads the pre-rewrite version, which is still a valid superset (or subset) of `injected[]` — degraded but correct. SDK-side read is wrapped in try/except → fall back to empty exclude set.

### Race 3: Two hook subprocesses both try to write the sidecar concurrently
**Location:** Same file write contention.
**Trigger:** User submits a prompt while a PostToolUse from a previous turn is still finalizing its sidecar write.
**Data prerequisite:** Two writers active at once.
**State prerequisite:** N/A.
**Mitigation:** Both writers use the same `tmp + rename` pattern. Last-rename-wins. The lost write loses up to 3 IDs from `injected[]` for one cycle, causing at most one duplicate-thought event. Atomic rename means we never see a corrupted file. Acceptable: the cost of file-locking infrastructure is far higher than the cost of an occasional duplicate thought.

## No-Gos (Out of Scope)

- Adding a literal `SessionStart` hook. Recon confirmed SessionStart has no relevance signal — UserPromptSubmit is the correct trigger.
- Changing the relevance scoring algorithm (bloom + BM25 + RRF + category weights stays identical; we just feed it a different query).
- Changing PostToolUse cadence (`WINDOW_SIZE=3`, `BUFFER_SIZE=9`, `MAX_THOUGHTS=3` all unchanged).
- Unifying the parallel hooks-side and SDK-side PostToolUse paths into one code path (separate concern; coexistence is fine).
- Implementing #1178's stub format. Prefetch emits full bodies; #1178 sweeps both call sites later.
- Adding cross-session memory caching, prefetch analytics, or new MCP tools.
- Modifying the bridge or worker code paths around AgentSession spawn — the worker-spawned subprocess already inherits `.claude/settings.json` and fires `UserPromptSubmit` on its initial prompt. SDK-side coverage is purely the de-dup change in `check_and_inject()`.

It remains in `recall()` only, where it's tied to tool-buffer context. Issue #627's lesson: noisy fallbacks on the user-visible path are pure noise.
