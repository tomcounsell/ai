# Patch the Bridge `dm` Memory-Writer Leak

## Problem

Four Memory records survived the #1171 migration with `project_key="dm"` despite the `dm` namespace having been retired in #811 (PR #820, 2026-04-07):

```
id=e3594f40b5bd4d798e4dad0cc2a40a2b agent_id=Tom source=human
id=7b8584531c9d46c9882e61b88c0c7c9d agent_id=Tom source=human
id=e92624c0e0af4ee989b6910e0a07a1ad agent_id=Tom source=human
id=6e8120c83d7f48c8a86f340b58289751 agent_id=Tom source=human
```

The migration in PR #1174 re-keyed them to `valor` correctly, but the writer that produced them was never patched. So long as the writer remains, future `dm`-tagged records will continue to drip in.

**Current behavior:**
`bridge/telegram_bridge.py:917` does `_early_project_key = project.get("_key", "dm") if project else "dm"`, then writes a Memory record at `:956` using that key, with `agent_id=sender_name` (e.g. "Tom") and `source="human"`. When `find_project_for_dm(sender_id)` and `find_project_for_chat(chat_title)` both return `None` — i.e. the sender isn't whitelisted and the chat title doesn't match any declared group — the bridge silently writes the message into the retired `dm` partition. The session-routing branch at `:1040` has the identical fallback shape and feeds the same key into downstream session_id derivation; the edit handler at `:2031` is already guarded by `if not project: return`, so it does not leak.

**Desired outcome:**
A message that doesn't belong to any declared project is not written into any memory partition. The `dm` literal disappears from `bridge/telegram_bridge.py`. A guard at `Memory.safe_save` callsites logs a stack-traced warning if a write is ever attempted with `project_key in ("dm", "default")`, to catch any future regression at the source.

## Solution

### Key Elements

- **Guard the memory write only**: replace the `dm`-fallback in `bridge/telegram_bridge.py:917` with `_early_project_key = project.get("_key") if project else None`. Pass that value (which may be `None`) through to `store_message` and `register_chat` — both already accept `project_key: str | None = None` and remain useful for conversation history regardless of project resolution. **Only** the `Memory.safe_save(...)` block at `:949-964` is wrapped in `if _early_project_key:`. Conversation history (the data feeding `valor-telegram read`) is preserved for unowned chats; only the canonical Memory partition write is skipped, which is the actual leak surface.
- **Move the existing project guard earlier**: `bridge/telegram_bridge.py:998` (`if not should_reply: ... return`) already early-returns for unresolved projects (since `should_respond_async` requires a resolved project for non-whitelisted senders). Promote that guarantee into a hard check: at `:1003` change the conditional `project_name = project.get("name", "DM") if project else "DM"` to be preceded by `if not project: return`. Then at `:1040`, simplify to `project_key = project["_key"]` — guaranteed non-None because of the new guard. This eliminates the `dm` fallback at `:1040` AND avoids the NameError class problem at `:1117, :1136, :1248, :1462, :1730, :1965` that a bare early-return at `:1040` would create.
- **Patch the third site**: `bridge/telegram_bridge.py:2031` also contains `project.get("_key", "dm")`. The function early-returns at `:2028` (`if not project: return`), so the fallback is dead code, but Success Criterion L216 (`grep '"dm"'` returns no fallback hits) requires it to go. Replace with `project_key = project["_key"]` (subscript safe — guarded by `:2028`).
- **Creation-site prevention, not tripwire**: this plan's stance per `feedback_prevention_over_cleanup.md` is that the **bridge-side patches at `:917` and `:1003-1040`** ARE the prevention. They guarantee at the creation site that no Memory record can be born with `project_key="dm"`. The `_warn_if_legacy_namespace` helper inside `Memory.safe_save` is **not** a tripwire-instead-of-fix — it is a regression detector covering the other 5+ writer callsites we are not editing in this PR (extraction, hooks, indexer). It exists so a future PR introducing a new writer with a hardcoded fallback fires a warning on first invocation, not after a migration months later. Without it, the prevention is bridge-local; with it, the prevention extends to any writer that touches `safe_save`.

### Flow

Telegram message arrives → bridge.handle_incoming → resolve project (`find_project_for_dm` or `find_project_for_chat`) → conversation history is stored regardless (`store_message`/`register_chat` accept `project_key=None`) → if project is None: skip Memory partition write only; if project resolved: write Memory under the project's `_key` → `should_respond_async` decides whether to reply → if no project: hard early-return at `:1003` before any session-routing.

The flow change has two decision points:
1. **Memory partition write** (`:917-964`): "no project resolved" routes to "skip Memory partition write" instead of "write under `dm`." Conversation history continues to flow.
2. **Session routing** (`:1003+`): "no project resolved" routes to "early return" instead of "fall through to `dm`-prefixed session_id." This was already the de facto behavior (`should_respond_async` filtered these out) — the patch makes it explicit and removes the latent NameError surface.

### Technical Approach

- Edit `bridge/telegram_bridge.py:917` from `_early_project_key = project.get("_key", "dm") if project else "dm"` to `_early_project_key = project.get("_key") if project else None`. **Keep** the `store_message` (`:920-931`) and `register_chat` (`:938-943`) calls running with `project_key=_early_project_key` (which may be `None` — both functions already accept `str | None`). **Only** wrap the `Memory.safe_save(...)` block at `:949-964` in `if _early_project_key:`. This preserves conversation history for unowned chats while preventing memory-partition pollution.
- Edit `bridge/telegram_bridge.py:1003` to insert `if not project: return` immediately before the `project_name = project.get("name", "DM") if project else "DM"` line. The right-hand `else "DM"` becomes dead code after the guard but harmless to leave (or simplify to `project["name"]` if every resolved project has a `name` field — verify during build). At `:1040`, replace `project_key = project.get("_key", "dm") if project else "dm"` with `project_key = project["_key"]` (guaranteed non-None by the `:1003` guard). This single guard placement also resolves the NameError concern for the six downstream `project_key` references at `:1117, :1136, :1248, :1462, :1730, :1965`.
- Edit `bridge/telegram_bridge.py:2031` from `project_key = project.get("_key", "dm")` to `project_key = project["_key"]`. The early-return at `:2028` (`if not project: return`) already guarantees `project` is truthy, so the subscript is safe and the `"dm"` fallback is dead code being eliminated for Success Criterion compliance.
- Add a small helper `_warn_if_legacy_namespace(project_key)` in `models/memory.py` that the `Memory.safe_save` classmethod calls before persisting. **Differentiated severity**: `"dm"` logs at `WARNING` with `traceback.format_stack()` (the retired namespace — should never appear in production); `"default"` logs at `DEBUG` with stack trace (still appears legitimately in test fixtures, single-machine bootstrap, and migration paths — DEBUG keeps the audit trail without log spam). The helper wraps its own emission in `try/except` so it can never raise. This addresses Risk 2 directly: the original WARN-on-`default` design would have swamped logs.
- **Defensive startup invariant**: at bridge startup (after `bridge/telegram_bridge.py:608`), `assert set(DM_WHITELIST) == set(DM_USER_TO_PROJECT.keys())`. Risk 3's mitigation cited the equality but did not enforce it; this prevents the stale-config trap if the two structures ever decouple in `projects.json`.
- Add a unit test on `_warn_if_legacy_namespace` covering `"dm"`, `"default"`, `"valor"`, `None`, `""` (Simplifier finding: the integration test on `handle_incoming` is over-spec for Small appetite — verify the bridge guard via code-read during PR review and via the `grep` Verification rows).

## Rabbit Holes

It's the right long-term cleanup but out of scope for this fix — each callsite has its own context (cwd, env var, project lookup) and a one-size-fits-all helper would need careful design. **Filed as follow-up issue #1232** (referencing #1173 and `feedback_prevention_over_cleanup.md`). The deferral is justified because (a) this PR is bug-scoped Small, (b) the bridge patches at `:917`, `:1003-1040`, and `:2031` ARE the creation-site prevention for the `dm` leak, (c) the `_warn_if_legacy_namespace` guard is a regression detector (not a tripwire-instead-of-fix), and (d) a unified helper requires per-callsite context analysis that exceeds Small appetite.
- **Adding stack-trace logging to every `Memory.safe_save` callsite individually.** The single-point guard inside `safe_save` is strictly better — one change, catches any future writer regardless of where it lives.
- **Backfilling/sweeping `dm`-tagged records that landed since PR #1174.** Out of scope unless the post-deploy verification (Success Criteria) shows leakage; if it does, file a one-line follow-up to re-run the migration with a narrower filter.

They are NOT the `dm`-leak source. The DEBUG-level guard inside `safe_save` covers them as a regression detector for the future unified-helper PR (above).

## Risks

### Risk 1: Skipping Memory partition writes for unowned messages drops legitimate signal
**Impact:** A message from a sender who SHOULD be on the whitelist but was missed in `projects.json` is no longer recorded in any Memory partition.
**Mitigation:** Two-layer mitigation now baked into the plan:
1. **Conversation history is preserved** — the revised approach keeps `store_message` and `register_chat` running with `project_key=None`, so `valor-telegram read` continues to work for unowned chats. Only the canonical Memory partition write (the `safe_save` block at `:949-964`) is skipped. This was the BLOCKER finding from the prior critique.
2. The bridge already does NOT respond to unowned-chat messages (`should_respond_async` requires a resolved project for non-whitelisted senders). The `_warn_if_legacy_namespace` guard inside `Memory.safe_save` catches any other code path that might still try to write `dm`/`default`, so the audit trail stays — `dm` at WARNING, `default` at DEBUG (revised severity per Risk 2).

### Risk 2: `Memory.safe_save` warning fires noisily on legitimate `default`-fallback callers
**Impact:** Log spam if any writer legitimately uses `project_key="default"` (e.g., during single-machine bootstrapping or test fixtures). The original plan framed the warning as "for `dm` AND `default`" because both should be retired, but `default` still appears in test and migration paths.
**Mitigation:** Differentiated log severity in `_warn_if_legacy_namespace`:
- `"dm"` → `logger.warning(...)` with stack trace (retired namespace, should never appear in production)
- `"default"` → `logger.debug(...)` with stack trace (still legitimate during bootstrap; surfaces only when `LOG_LEVEL=DEBUG` is set for investigation)
This makes the audit trail searchable without polluting normal logs.

### Risk 3: Hard early-return at `:1003` breaks a code path that depended on the project fallback
**Impact:** A message that previously fell through to `project_name="DM"` and `project_key="dm"` is now early-returned at `:1003`, and downstream code that expected to see those messages no longer does.
**Mitigation:** Verified via code-read in Freshness Check: the `should_respond_async` gate at `:987-997` already returns `should_reply=False` for any non-whitelisted DM sender or unrecognized group title. The `:998` block (`if not should_reply: return`) early-returns those messages BEFORE `:1003`. So when execution reaches `:1003` today, `project` is already non-None in every reachable case — the fallback `else "DM"` is dead code. The new `if not project: return` at `:1003` simply makes that invariant explicit and prevents a future refactor of `should_respond_async` from silently re-introducing the leak. Additionally, the **startup invariant** (`assert set(DM_WHITELIST) == set(DM_USER_TO_PROJECT.keys())`) prevents the related stale-config trap where the whitelist and project mapping decouple in `projects.json`.

## Race Conditions

No race conditions identified. The memory-write decision is sequential within `handle_incoming`, and the `Memory.safe_save` classmethod is internally idempotent (popoto handles concurrent saves to the same key via the underlying Redis pipeline).
