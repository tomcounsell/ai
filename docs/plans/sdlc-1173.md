# Patch the Bridge `dm` Memory-Writer Leak

## Problem

Four Memory records survived the #1171 migration with `project_key="dm"` despite the `dm` namespace having been retired in #811 (PR #820, 2026-04-07):

```
id=e3594f40b5bd4d798e4dad0cc2a40a2b agent_id=Tom source=human
id=7b8584531c9d46c9882e61b88c0c7c9d agent_id=Tom source=human
id=e92624c0e0af4ee989b6910e0a07a1ad agent_id=Tom source=human
id=6e8120c83d7f48c8a86f340b58289751 agent_id=Tom source=human
```

The migration in PR #1174 re-keyed them to `valor` correctly, but the writer that produced them was never patched. So long as the writer remains, future `dm`-tagged records will continue to drip in.

**Current behavior:**
`bridge/telegram_bridge.py:917` does `_early_project_key = project.get("_key", "dm") if project else "dm"`, then writes a Memory record at `:956` using that key, with `agent_id=sender_name` (e.g. "Tom") and `source="human"`. When `find_project_for_dm(sender_id)` and `find_project_for_chat(chat_title)` both return `None` — i.e. the sender isn't whitelisted and the chat title doesn't match any declared group — the bridge silently writes the message into the retired `dm` partition. The session-routing branch at `:1040` has the identical fallback shape and feeds the same key into downstream session_id derivation; the edit handler at `:2031` is already guarded by `if not project: return`, so it does not leak.

**Desired outcome:**
A message that doesn't belong to any declared project is not written into any memory partition. The `dm` literal disappears from `bridge/telegram_bridge.py`. A guard at `Memory.safe_save` callsites logs a stack-traced warning if a write is ever attempted with `project_key in ("dm", "default")`, to catch any future regression at the source.

## Solution

### Key Elements

- **Guard the bridge writer**: replace the `dm`-fallback in `bridge/telegram_bridge.py:917` with a strict project resolution. If `project is None`, skip the `Memory.safe_save(...)` call entirely (an unowned message has no canonical home). The `_early_project_key` derivation only feeds memory persistence and chat-mapping registration, both of which are no-ops for unowned chats by their existing semantics.
- **Tighten the second fallback**: at `bridge/telegram_bridge.py:1040`, replace the `"dm"` fallback with an early-return — the surrounding code already requires a resolved project to spawn an AgentSession. Today, an unresolved project would silently form a session_id like `tg_dm_<chat>_<msg>`; the early-return makes that case explicit.
- **Belt-and-braces logger**: at every `Memory.safe_save(...)` callsite (currently 5 in non-worktree code: `bridge/telegram_bridge.py:956`, `agent/memory_extraction.py:435,662`, `tools/memory_search/__init__.py:233`, `.claude/hooks/hook_utils/memory_bridge.py:740`, `tools/knowledge/indexer.py:338,355`), if `project_key in ("dm", "default")`, emit a `logger.warning` with `traceback.format_stack()`. If a future regression introduces another writer fallback, the warning fires immediately rather than waiting for a migration to surface it months later.

### Flow

Telegram message arrives → bridge.handle_incoming → resolve project (`find_project_for_dm` or `find_project_for_chat`) → if None: skip memory write (return early), if Some: write Memory under the project's `_key` → continue with should_respond_async.

The flow change is one decision point: "no project resolved" routes to "no memory write" instead of "memory write under `dm`."

### Technical Approach

- Edit `bridge/telegram_bridge.py:917` from `_early_project_key = project.get("_key", "dm") if project else "dm"` to a guarded form. The cleanest pattern: assign `_early_project_key = project.get("_key") if project else None`, then guard the `store_message`/`register_chat`/`Memory.safe_save` block with `if _early_project_key:`.
- Edit `bridge/telegram_bridge.py:1040` similarly: `project_key = project.get("_key") if project else None`, then `if not project_key: return` before any session_id derivation. (The downstream session-id format `tg_{project_key}_...` is broken if `project_key` is None, so the early-return is also a correctness fix.)
- Add a small helper `_warn_if_legacy_namespace(project_key, callsite)` in `models/memory.py` that the `Memory.safe_save` classmethod calls before persisting. If `project_key in ("dm", "default")`, log a `WARNING` with `traceback.format_stack()`. This is a single-point change rather than touching every callsite, and it doubles as a regression catcher for any future writer that hardcodes a fallback.
- Add an integration test that drives `handle_incoming` with a mock event whose chat_title and sender_id resolve to no project, and asserts that `Memory.query.filter(project_key="dm").count()` does not increase.

## Risks

### Risk 1: Skipping memory writes for unowned messages drops legitimate signal
**Impact:** A message from a sender who SHOULD be on the whitelist but was missed in `projects.json` is now silently un-recorded, instead of being recorded under `dm` and surfacing later in audits.
**Mitigation:** This is a feature, not a bug. The bridge already does NOT respond to such messages (`should_respond_async` requires a resolved project for non-whitelisted senders). Logging silently to `dm` while not responding is the worst of both worlds — it pollutes memory without helping the user. The `_warn_if_legacy_namespace` guard inside `Memory.safe_save` catches any other code path that might still try to write `dm`/`default`, so the audit trail stays.

### Risk 2: `Memory.safe_save` warning fires noisily on legitimate `default`-fallback callers
**Impact:** Log spam if any writer legitimately uses `project_key="default"` (e.g., during single-machine bootstrapping or test fixtures). The warning was framed as "for `dm` AND `default`" because both should be retired, but `default` may still appear in test or migration paths.
**Mitigation:** The warning fires only on `safe_save`, and emits at WARN level (not ERROR). Test fixtures that intentionally use `"default"` can be updated to use a mock or an explicit project key. If the warning fires in production logs after deploy, that's a real signal — investigate, don't suppress.

### Risk 3: Changing `bridge/telegram_bridge.py:1040` to early-return breaks a code path that depended on the `dm` fallback
**Impact:** A message that previously got a session_id like `tg_dm_<chat>_<msg>` no longer gets one, and downstream session-routing or response logic crashes or silently drops the message.
**Mitigation:** Read the code path between `:1040` and the next 200 lines carefully during build. The `should_respond_async` gate already filters un-resolved projects out of the response path (the bridge does NOT reply to chats with no project), so any session created with `project_key="dm"` was already orphaned — never spawned as an AgentSession, never replied to. Confirming this empirically is part of the build's pre-flight; if the read shows otherwise, the fix narrows to leave the session_id fallback alone and only patch `:917`.

## Open Questions

1. **Should `_warn_if_legacy_namespace` block the write or just warn?** Two modes: (a) WARN-and-write (back-compat, observability only) or (b) WARN-and-skip (refuse to persist legacy namespaces). Recommendation: start with (a) for the soak window, then flip to (b) in a follow-up after the soak confirms no legitimate writer remains. This avoids a regression risk if any test fixture or migration path silently writes `"default"`.

2. **Does `Memory.safe_save` already emit any logging on success?** If yes, the new warning slots in cleanly. If no, we may want to gate the warning by an env var (`MEMORY_LEGACY_NAMESPACE_WARN=1`) to keep production logs quiet while letting CI surface regressions. Recommendation: check during build; default to "always warn" unless the existing logging is silent-by-design.

3. **Should we sweep any `dm`-tagged records that landed since PR #1174?** Run `python -m tools.memory_search inspect --stats --project-key dm` after deploy. If count > 0, run a one-line re-key script analogous to PR #1174. Recommendation: check post-deploy as part of Success Criteria; out of scope to plan now.
