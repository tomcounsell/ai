# Local CLI Dev Sessions Survive Worker Restart

## Problem

On a skills-only machine (no Telegram bridge), the PM session spawns dev sessions as local Claude Code processes. Each dev session lives on the AgentSession queue with a `session_id` prefixed `local-`. On any worker restart — scheduled (`com.valor.update` every 30 min before #1091 landed), manual (`valor-service.sh restart`), or crash (watchdog escalation) — startup recovery in `agent/session_health.py::_recover_interrupted_agent_sessions_startup` abandons these local sessions unconditionally. The PM's next turn re-spawns a fresh dev from scratch and the in-flight transcript, tool output cache, and partial reasoning are gone.

**Current behavior:**
- Worker boots; `_recover_interrupted_agent_sessions_startup()` queries all `status="running"` sessions
- For each stale session (older than `AGENT_SESSION_HEALTH_MIN_RUNNING`), the loop checks `entry.session_id.startswith("local")`
- If true, the session is `finalize_session(..., "abandoned", ...)` — regardless of `session_type`
- A mid-build dev session that had already produced commits on its worktree branch is abandoned; its `claude_session_uuid` and transcript are orphaned
- The PM's next turn calls `valor-session create --role dev` (not `resume`) and re-starts the stage from scratch
- Committed work on the branch remains, but transcript context, in-flight tool calls, and reasoning state are lost

**Desired outcome:**
- Local **dev** sessions (session_type = dev) survive worker restart: they get re-queued to `pending` just like bridge-owned dev sessions, and the worker resumes them via `claude --resume <UUID>` on next pickup
- Local **PM** and **Teammate** sessions continue to be abandoned — the original `#986` hijack rationale still holds for those (a live human CLI at the same UUID must not be competed with)
- No new terminal state to recover from; no change to `valor-session resume` CLI
- Existing tests for the local-session guard continue to pass for PM/teammate cases and gain new assertions for the dev case

## Prior Art

- **#986 / PR #989** (CLOSED 2026-04-15): "Startup recovery hijacks live local Claude Code sessions via --resume." **This PR ADDED the current `is_local` guard** at `session_health.py:255`. The motivation was an active human Claude Code CLI at a `local-*` session_id — startup recovery would `claude --resume <UUID>` against that same UUID, producing two concurrent harnesses writing to one transcript. PR #989 correctly abandoned all `local-*` running sessions to let the originating CLI reclaim them. That fix is correct for PM/teammate sessions but **too broad** for worker-owned dev sessions (which have no human CLI writing to them).
- **#944** (CLOSED 2026-04-14): Health check skipped recovery for stuck dev sessions when a shared project-keyed worker was alive — orthogonal to this issue, but confirms that per-`session_type` behavior in the health check is an accepted pattern.
- **#1001** (CLOSED 2026-04-16): `user_prompt_submit` hook created AgentSession for direct CLI sessions with no worker owner — a separate class of local-session bug, now resolved. Confirms the architectural distinction: CLI-owned local sessions (PM/teammate) vs. worker-owned local sessions (dev). The guard must differentiate them.

## Why Previous Fixes Failed

| Prior Fix | What It Did | Why It Failed / Was Incomplete |
|-----------|-------------|-------------------------------|
| PR #989 | Added `is_local` guard to startup recovery — abandons any `session_id.startswith("local")` running session | Didn't distinguish `session_type`. Applied the right remedy (abandon, not re-queue) for PM/teammate sessions (which have a competing human CLI) but over-applied it to worker-owned dev sessions (which have no competing process). Result: mid-build dev sessions get abandoned on worker restart even though nothing is competing for their UUID. |

**Root cause pattern:** The `local-*` session_id prefix was being used as a proxy for "cannot be resumed by worker" when the real predicate is "has a competing human CLI process writing to the same Claude UUID." PM/teammate local sessions have that competitor (the human invoking `claude -p`); dev local sessions do not (they are spawned via `valor-session create --role dev` and fully owned by the worker harness).

## Solution

### Key Elements

- **`session_type`-aware local guard in startup recovery**: the `is_local` branch at `agent/session_health.py:257` splits into two sub-branches based on `entry.session_type`.
- **Dev path (new)**: if `is_local AND session_type == SessionType.DEV`, follow the same recovery path as bridge sessions — call `update_session(new_status="pending", ...)` and log as "Recovered local dev session." No change to `claude_session_uuid`, no finalize, no new terminal state.
- **PM/teammate path (unchanged)**: if `is_local AND session_type in (SessionType.PM, SessionType.TEAMMATE)`, retain the existing abandon path and its rationale comment (human CLI may still be competing).
- **Log differentiation**: add a distinct `[startup-recovery] Recovered local dev session <id>` message so the existing "Abandoned N local session(s)" counter doesn't hide the new behavior.

### Flow

**Worker boots** → Startup recovery queries `status="running"` → Loop each stale session → Check `session_id.startswith("local")`:
- **If local AND dev**: → `update_session(..., new_status="pending", expected_status="running")` → worker picks it up via `claude --resume <UUID>` on next queue scan → PM steering resumes normally
- **If local AND (PM or teammate)**: → `finalize_session(entry, "abandoned", reason="startup recovery: local session cannot be resumed by worker")` (unchanged)
- **If NOT local**: → `update_session(..., new_status="pending", expected_status="running")` (unchanged bridge path)

### Technical Approach

- Import `SessionType` from `models.agent_session` at the top of `agent/session_health.py` (it already imports `AgentSession`; no new module).
- In the `for entry in stale_sessions:` loop at line 253, replace the single `is_local` branch with a three-way split. Use early returns / continues to keep the flow flat.
- Log the recovered count split: `logger.warning("[startup-recovery] Recovered %d bridge session(s), %d local dev session(s), abandoned %d local PM/teammate session(s)", bridge_count, dev_count, abandoned)`.
- The function's return value (`count`) should include recovered local-dev sessions in the same tally as bridge sessions — they are equivalent from the caller's perspective.
- No change to `_handle_dev_session_completion`, `valor-session`, or any downstream code. The resumed dev session hits `_handle_dev_session_completion` exactly as before when it completes.

## Risks

### Risk 1: A local dev session IS actually competing with something
**Impact:** Two `claude -p` processes could attempt to resume the same UUID, producing garbled transcripts.
**Mitigation:** Dev sessions are spawned exclusively via `valor-session create --role dev` by a PM session. The PM is the only entity that knows the UUID. When the worker dies, the PM's child execution ends (no orphaned external writers). When the worker restarts, it's the sole consumer. Verified by code read: no code path outside the worker calls `claude --resume <dev_uuid>`. The human CLI spawns PM/teammate sessions, never dev.

### Risk 2: SessionType enum changes in the future
**Impact:** A new `SessionType` value (e.g., `REFLECTION`, `WORKFLOW`) added to the enum without updating this branch would fall through to the PM path (safer default) and still get abandoned — correct behavior but may surprise future maintainers.
**Mitigation:** Use explicit `if session_type == SessionType.DEV` instead of `if session_type != SessionType.PM`. Log the default-branch path clearly (`"abandoning local session_type=%s as PM/teammate default"`). No enumeration check — the fall-through to abandon is the safe default.

### Risk 3: The timing guard (`AGENT_SESSION_HEALTH_MIN_RUNNING`) already skips young sessions
**Impact:** None — this is an existing behavior preserved by the fix. A dev session that just started (< guard seconds ago) is skipped entirely, not abandoned and not re-queued.
**Mitigation:** No change needed. Note in the fix's logging so future debugging is clear: "Skipped recent session" → unaffected by this change.

## Race Conditions

### Race 1: Worker restart while dev session is mid-completion
**Location:** `agent/session_health.py::_recover_interrupted_agent_sessions_startup` (lines 253-324) and `agent/session_completion.py::_handle_dev_session_completion` (lines 640+)
**Trigger:** Worker 1 has dev session D executing; Worker 1 dies before D reaches `_complete_agent_session`. Worker 2 boots; startup recovery finds D in `status="running"`.
**Data prerequisite:** D's `claude_session_uuid` must be populated (written on SDK authentication, survives the restart via Popoto).
**State prerequisite:** D's status must still be `running` (the IndexedField). The CAS in `update_session(expected_status="running")` prevents a race where another concurrent worker already transitioned D away from running.
**Mitigation:** The existing `update_session(expected_status="running")` call (same pattern as the bridge path) provides CAS. If D's status already transitioned (e.g., a concurrent health-check kill fired), the update fails cleanly and falls through to `delete()` (same behavior as bridge path). No new race introduced.

### Race 2: PM session terminates between worker restart and dev session resume
**Location:** `agent/session_completion.py::_handle_dev_session_completion` (line 693+)
**Trigger:** Dev D's parent PM P is in `status="completed"` when D resumes and eventually calls `_handle_dev_session_completion`. The PM steering call would fail because P is terminal.
**Data prerequisite:** D.parent_agent_session_id points to P.
**State prerequisite:** Existing logic at `session_completion.py` handles this: when steering fails because P is terminal, a continuation-PM is created instead (documented in issue #987 fix). No new race introduced by this plan.
**Mitigation:** This race already existed for bridge-owned dev sessions. The fix here makes local dev sessions subject to the same existing race handling. No new mitigation needed; the continuation-PM path covers it.

1. **Teammate local sessions** — **Resolved: abandon (unchanged).** Teammate sessions are conversational with a human at the CLI and fall under the #986 hijack rationale. Treating "successfully recovered a local dev session" the same as "successfully recovered a bridge session" is the correct mental model — both are sessions that will resume under the worker — so a combined total is clearer. `abandoned` remains reported separately in the summary log line but is NOT included in the return value (matches existing behavior).
