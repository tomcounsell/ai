### Technical Approach

- **Item 1**: `grep "new_session_id=None" logs/bridge.log* | grep -v "old.log" || echo "CLEAR"`. Pre-merge baseline: 15+ recurrences between 2026-04-23 and merge. Pass: zero hits in the 24h post-merge window. If non-zero, immediately re-open #1195 and stop the plan.
- **Item 2**: The deliberately broken session uses `AgentSession.create_pm(...)` with `working_dir=None`. The executor guard predicate at `agent/session_executor.py:641` is strict `is None` (verified at plan revision time): `if session.working_dir is None or session.session_id is None`. Empty string would NOT fire the guard — `None` is mandatory. The `create_pm` factory annotates `working_dir: str` but Python does not enforce typing; Popoto's `Field()` accepts `None`. The script must enqueue via `await asyncio.sleep` is not needed — direct `enqueue()` then poll. Poll for the worker to pick it up and transition to `failed` (max 30s). Capture the three artifacts in order. Cleanup via `AgentSession.query.filter(session_id=test_id).delete()`.
- **Item 3**: The fresh issue must be small (single-file fix or doc update) so the dev session completes in under 5 minutes.
  - **Issue selection (concrete criteria, in priority order):**
    1. Open issue with the `chore` or `bug` label, no `plan` label, opened in the last 14 days, single-file or single-doc scope. Discover via `gh issue list --state open --search 'label:chore,bug -label:plan created:>2026-04-15' --limit 20 --json number,title,labels,createdAt`.
    2. If no organic candidate within 7 days of plan execution, fall back to the substitute path in Risk #3 (open a re-do issue cherry-picking a recently-merged trivial fix).
  - **T0/T1 log line patterns (concrete grep targets):**
    - `T0` = `[harness] _handle_dev_session_completion` line for the dev session — grep `logs/bridge.log* logs/worker.log` for `_handle_dev_session_completion.*<dev_session_id>` and take the first match's timestamp.
    - `T1` = `[continuation-pm-created] parent_id=<pm_session_id>` log line — grep `logs/bridge.log* logs/worker.log` for `\[continuation-pm-created\].*parent_id=<pm_session_id>` and take the timestamp.
  - **Pass condition:** `T1 - T0 < 60s` AND no human `continue` Telegram message landed between T0 and T1 (`grep 'continue' logs/bridge.log` for the chat_id within the window).
  - **Tooling:** Use `python -c "from datetime import datetime; ..."` to compute the delta if needed; otherwise eyeball the timestamps when both are within the same minute.
- **Item 4**:
  - **Code change** at `agent/session_completion.py:437`: extend the log line to include `parent_id={parent.session_id}` so the test can assert on parent context. The current line is:
    ```python
    logger.error(f"[harness] _create_continuation_pm failed: {e}", exc_info=True)
    ```
    Becomes:
    ```python
    logger.error(
        f"[harness] _create_continuation_pm failed for parent_id={getattr(parent, 'session_id', '?')}: {e}",
        exc_info=True,
    )
    ```
    Rationale: the existing log line lost parent context. This makes triage easier and gives the new test a natural assertion target. (`getattr` with default `'?'` because `parent` is the function arg — guaranteed non-None — but `session_id` could theoretically be missing in pathological cases; we already guarded against `parent.session_id is None` at line 327 so by the time we reach line 437 it is set.)
  - **Test** in `tests/unit/test_continuation_pm.py::TestCreateContinuationPM`:
    ```python
    def test_resolve_helper_raise_is_logged_not_swallowed(self, terminal_pm, redis_test_db, caplog):
        """If _resolve_working_dir_for_parent raises, the error is logged
        with the [harness] tag and parent context, and no malformed continuation
        is saved."""
        from agent.agent_session_queue import _create_continuation_pm

        before = len(list(AgentSession.query.all()))
        with patch(
            "agent.session_completion._resolve_working_dir_for_parent",
            side_effect=RuntimeError("simulated config failure"),
        ), caplog.at_level(logging.ERROR):
            _create_continuation_pm(
                parent=terminal_pm,
                agent_session=None,
                issue_number=934,
                stage="BUILD",
                outcome="success",
                result_preview="r",
            )

        after = len(list(AgentSession.query.all()))
        assert after == before, "no malformed continuation session was saved"

        msgs = [r.message for r in caplog.records]
        assert any("[harness] _create_continuation_pm failed" in m for m in msgs), (
            "Expected [harness] tag in error log; got: %r" % msgs
        )
        assert any(terminal_pm.session_id in m for m in msgs), (
            "Expected parent session_id in error log; got: %r" % msgs
        )
        assert any("simulated config failure" in m for m in msgs), (
            "Expected the simulated RuntimeError text in error log; got: %r" % msgs
        )
    ```
  - Re-uses the existing `terminal_pm` fixture and the patch path already used at `tests/unit/test_continuation_pm.py:188`.
  - Imports: `import logging` (top of file already has logging-related imports — check) and `from unittest.mock import patch` (already imported).

## Rabbit Holes

- **Refactoring the broad `except Exception as e:` at line 436 into multiple targeted excepts.** Tempting because broad excepts violate the "exception handling coverage" pattern, but PR #1196 deliberately kept it broad to avoid regressing any of the eight inner code paths under one exception filter. Out of scope.
- **Adding a `failure_reason` column to `AgentSession`.** The issue body explicitly notes "There is no `failure_reason` column on `AgentSession` — the structured log is the durable failure record." Adding one would be a multi-PR migration. Out of scope.
- **Replacing item 2's manual dashboard test with an automated test.** The existing unit and integration tests already cover the `[executor-guard]` failure path under isolation. Item 2's value is verifying the **observer surfaces** (status CLI, dashboard JSON, log surface) honor `failed` status against a real worker — automation would defeat the purpose.
- **Wrapping items 1, 2, 3 into a `python scripts/verify_continuation_pm.py` orchestrator.** Single-shot verification work; the orchestration overhead exceeds the benefit.

## Risks

### Risk 1: Item 1 grep finds non-zero hits

**Impact:** Indicates a regression slipped through PR #1196 — `[continuation-pm-created] new_session_id=None` is the original bug signature.
**Mitigation:** Stop the plan immediately. Re-open #1195 with the grep output as evidence. Do not proceed to items 2–4 until #1195 is re-fixed.

### Risk 2: Item 2's broken-session test triggers the worker but the executor-guard does not fire

**Impact:** Indicates a defense-in-depth gap. The session would crash later with a different signature.
**Mitigation:** Capture the actual failure mode (whatever surface caught it), file a follow-up issue. Do not claim item 2 passed if the guard did not fire — failure to observe the guard firing is a finding, not a pass.

### Risk 3: Item 3 cannot find a small-enough fresh issue

**Impact:** Item 3 is meaningless without a real Build→Test boundary.
**Mitigation:** Acceptable substitute (decided at plan revision): if no organic candidate is found within 7 days of starting Item 3, cherry-pick a recently-merged trivial issue from the last 14 days (`gh issue list --state closed --search "label:bug closed:>2026-04-15" --limit 30 --json number,title,url`), filter to single-file fixes by reading the merged PR's `files`, open a fresh re-do issue with the same fix, and run that. Document the substitute lineage in the Item 3 comment so reviewers can audit.

### Risk 4: Item 4 unit test depends on a specific log message format that future refactors may change

**Impact:** Test breaks under cosmetic log-message edits.
**Mitigation:** Assert on three independent substrings (`[harness] _create_continuation_pm failed`, `parent.session_id`, `simulated config failure`) — each refactor would have to break all three to defeat the test. The `[harness]` tag is also the production grep target for `tools/log_search.py` and is unlikely to change without a deliberate ADR.
