# Fix Tier 1 Liveness Signal: Replace Watchdog-Tick Heartbeat with Per-Turn Progress Timestamp

## Problem

`session-liveness-check` cannot recover a session that is hung from the very first turn
— or that hangs mid-run — because the Tier 1 input signal it reads reflects watchdog
ticks, not actual SDK output.

**Current behavior:**

`BackgroundTask._watchdog` (in `agent/messenger.py:328`) runs every 60s while the SDK
subprocess is alive and unconditionally invokes `messenger.notify_heartbeat_tick()`.
That callback writes `now()` into `AgentSession.last_sdk_heartbeat_at` regardless of
whether the SDK has emitted any structured output (assistant message, tool_use,
tool_result, or result event).

`_has_progress()` in `agent/session_health.py:465–472` treats a fresh
`last_sdk_heartbeat_at` as positive Tier 1 evidence of progress. Any subprocess that
exists keeps the heartbeat fresh and passes Tier 1 forever. A wedged SDK that never
produces output is indistinguishable from a working one.

Meanwhile, two fields that DO capture actual SDK output — `last_tool_use_at` and
`last_turn_at`, written by `agent/hooks/liveness_writers.py` — are used only by the
dashboard. They are never read by `_has_progress`.

The queue-layer heartbeat (`last_heartbeat_at`, written by `_heartbeat_loop` in
`session_executor.py`) has the same defect: it too is refreshed unconditionally every 60s
while the session is alive, independent of SDK output.

**Observed instance (2026-04-30):**

Session `0_1777532128936` (PM, slug `sdlc-1219`) ran for ≈3 hours with
`SDK heartbeat: running Ns, communicated=False` logged every 60s from 60s through
10380s+. `session-liveness-check` fired ~30+ times and recovered the session zero times.
Resolution required manual `valor-session kill`.

**Desired outcome:**

A session that authenticates, spawns a subprocess, but never produces any structured SDK
output is recovered within a freshness window — not after manual intervention. A session
that produces output once and then hangs mid-run is also recovered within the same window,
with Tier 2 reprieves still applying.

## Data Flow

**Current (broken) path for a hung session:**

1. `BackgroundTask._watchdog` (every 60s while subprocess alive) →
2. `messenger.notify_heartbeat_tick()` →
3. `_on_heartbeat_tick()` in `session_executor.py` →
4. `session.last_sdk_heartbeat_at = now()` written to Redis →
5. `_has_progress()` reads `last_sdk_heartbeat_at` < 90s → returns True →
6. Health check skips recovery → hung session survives forever.

**Correct path for a session with actual SDK output:**

1. SDK subprocess emits a `result` event or tool boundary →
2. `liveness_writers.record_turn_boundary()` or `record_tool_boundary()` is called from `sdk_client.py` or hooks →
3. `session.last_turn_at` or `session.last_tool_use_at` written to Redis →
4. `_has_progress()` reads freshest of `{last_tool_use_at, last_turn_at}` < SDK_PROGRESS_FRESHNESS_WINDOW → returns True →
5. Health check skips recovery → healthy session continues.

**Correct path for a hung session (after fix):**

1. SDK subprocess emits nothing → `last_tool_use_at` and `last_turn_at` both stale →
2. `_watchdog` still ticks `last_sdk_heartbeat_at` (watchdog-alive signal only) →
3. `_has_progress()` reads `last_tool_use_at`/`last_turn_at` both stale → falls through →
4. Own-progress fields (`turn_count`, `log_path`, `claude_session_uuid`) all sticky-but-stale →
5. `_has_progress()` returns False →
6. `_tier2_reprieve_signal()` evaluated → "alive" gate: subprocess exists but produces no children and is not compacting →
7. Eventually all Tier 2 gates fail → session recovered.

Note: Step 6 means Tier 2's "alive" gate (process status not zombie/dead/stopped) will
reprieve the session on every tick until the SDK_PROGRESS_FRESHNESS_WINDOW approach is
adopted. See the own-progress field discussion in Solution below.

## Why Previous Fixes Failed

| Prior Fix | What It Did | Why It Failed / Was Incomplete |
|-----------|-------------|-------------------------------|
| PR #1039 (#1036) | Introduced `last_sdk_heartbeat_at` as "messenger-sourced heartbeat" | Wired the field to `BackgroundTask._watchdog`, which fires on subprocess existence, not SDK output. Signal meaning ("messenger heartbeat") and writer semantics ("subprocess alive") diverged from day 1. |
| PR #1177 (#1172) | Added `last_tool_use_at` / `last_turn_at` for dashboard Pillar A | Explicitly kept these out of `_has_progress` (out of scope for #1172). Created the right fields but didn't connect them to the health check. |

**Root cause pattern:** Both fixes were applied at the wrong layer. The messenger/watchdog
layer cannot observe SDK output (it only observes subprocess existence). The correct layer
is the sdk_client + hooks layer, which already observes `result` events and tool boundaries.
The fix must promote the sdk_client-layer fields into `_has_progress`.

## Solution

### Key Elements

- **Promote `last_tool_use_at` and `last_turn_at` into Tier 1**: `_has_progress` checks
  these fields against a `SDK_PROGRESS_FRESHNESS_WINDOW` constant. Any field fresher than
  the window is evidence of actual SDK output → return True.

- **Demote `last_sdk_heartbeat_at` from Tier 1**: Remove it from the dual-heartbeat OR
  check. The field keeps being written by the watchdog (useful for dashboard `last_evidence_at`)
  but no longer gates the health check.

- **Address own-progress field leaks**: `turn_count > 0`, `log_path` non-empty, and
  `claude_session_uuid` non-empty are all sticky once set. They detect "ever started" but
  cannot detect mid-run hangs. These must be tightened: they should only pass Tier 1
  when paired with a fresh per-turn timestamp, OR be demoted to standalone signals only
  for the never-started case (when `last_tool_use_at` and `last_turn_at` are both None).

- **Keep `last_heartbeat_at` for "executor running" signal**: The queue-layer heartbeat
  (`last_heartbeat_at`, written by `_heartbeat_loop`) proves the executor is running. This
  is meaningfully different from "SDK is making progress" but is still a useful signal to
  keep. Under the new design, a fresh `last_heartbeat_at` with a stale
  `last_tool_use_at`/`last_turn_at` means the executor is alive but the SDK is hung.

- **Set `SDK_PROGRESS_FRESHNESS_WINDOW`**: The longest legitimate single-turn duration
  without any structured SDK output. Tool use produces events at tool boundaries; a
  long-thinking turn produces a `result` event at completion. The window must accommodate
  long-thinking turns (potentially 15–30 minutes on large prompts). Starting value:
  1800s (30 minutes), env-tunable via `SDK_PROGRESS_FRESHNESS_WINDOW_SECS`. This is
  intentionally generous — the Tier 2 "alive" gate continues to reprieve during this
  window on sessions that are genuinely working.

### Flow

**Session hangs from first turn:**

Health tick →  `_has_progress()` checks `last_tool_use_at` (None) and `last_turn_at` (None) → stale → checks own-progress fields (sticky, set at start) → new guard: only passes if `last_tool_use_at` or `last_turn_at` is set → falls through → `_has_progress` returns False → Tier 2 evaluated → "alive" passes → session reprieved until window expires → after `SDK_PROGRESS_FRESHNESS_WINDOW` of no tool/turn events, even "alive" can no longer reprieve indefinitely (Tier 2 is session-level, not time-based, but each reprieve increments `reprieve_count`; no automatic escalation at this stage).

**Tier 2 "alive" gate and the recovery guarantee:**

After Tier 1 is fixed, a hung session will fail Tier 1 on every health tick. However the
Tier 2 "alive" gate reprieves it every tick as long as the subprocess exists and is not
a zombie. Without further action, recovery would still require manual intervention — the
original 3-hour hang scenario persists.

This plan addresses this by adding a reprieve-count escalation path in
`_tier2_reprieve_signal`: when `sdk_ever_output is False` (session never produced a
tool or turn event) AND `reprieve_count >= MAX_NO_OUTPUT_REPRIEVES`, suppress the "alive"
reprieve and allow recovery to proceed. `MAX_NO_OUTPUT_REPRIEVES` defaults to
`SDK_PROGRESS_FRESHNESS_WINDOW // HEARTBEAT_FRESHNESS_WINDOW` (1800 // 90 = 20 ticks,
i.e., 30 minutes of reprieves before escalating). This keeps the Tier 2 "alive" gate
working correctly for all other sessions while ensuring sessions that have never produced
output are eventually recovered.

Sessions that have produced output (sdk_ever_output is True) are NOT subject to this
escalation — their Tier 2 "alive" gate operates normally, and recovery depends on
`last_tool_use_at` / `last_turn_at` becoming stale beyond `SDK_PROGRESS_FRESHNESS_WINDOW`
in Tier 1.

### Technical Approach

**Consistent terminology:** Tier 1 (per-turn SDK progress) vs Tier 2 (reprieve gates) is
the two-level structure from #1036. Within Tier 1, the new code uses two sub-checks:
(A) per-turn SDK activity fields (`last_tool_use_at`, `last_turn_at`) and (B) startup-window
executor-alive fallback (`last_heartbeat_at`, only when sdk_ever_output is False).
The documentation tasks use this same A/B naming, not 1a/1b/1c.

**Changes to `agent/session_health.py`:**

1. Add constant:
   ```python
   # Freshness window for per-turn SDK progress signals (last_tool_use_at, last_turn_at).
   # 1800s (30 min) accommodates the longest observed extended-thinking turns.
   # Env-tunable for operators who observe edge cases in production.
   SDK_PROGRESS_FRESHNESS_WINDOW = int(os.environ.get("SDK_PROGRESS_FRESHNESS_WINDOW_SECS", 1800))

   # Max reprieve-count for sessions that have never produced SDK output.
   # After this many consecutive Tier 2 reprieves with sdk_ever_output=False,
   # the "alive" reprieve is suppressed and recovery proceeds.
   MAX_NO_OUTPUT_REPRIEVES = SDK_PROGRESS_FRESHNESS_WINDOW // HEARTBEAT_FRESHNESS_WINDOW  # 20
   ```

2. Rewrite `_has_progress` Tier 1 block (using A/B sub-check naming to match docstring):
   ```python
   # Compute sdk_ever_output once — used in both sub-checks and own-progress guard.
   sdk_ever_output = bool(
       getattr(entry, "last_tool_use_at", None)
       or getattr(entry, "last_turn_at", None)
   )

   # Sub-check A: per-turn SDK progress (#1226).
   # last_tool_use_at (written by PreToolUse/PostToolUse hooks) and last_turn_at
   # (written by sdk_client on result event) are evidence of actual SDK output.
   # Fresh within SDK_PROGRESS_FRESHNESS_WINDOW → real progress.
   for progress_attr in ("last_tool_use_at", "last_turn_at"):
       ts = getattr(entry, progress_attr, None)
       if isinstance(ts, datetime):
           ts_aware = ts if ts.tzinfo else ts.replace(tzinfo=UTC)
           if (now_utc - ts_aware).total_seconds() < SDK_PROGRESS_FRESHNESS_WINDOW:
               return True

   # Sub-check B: startup-window executor-alive fallback (#1036 retained, narrowed).
   # Use last_heartbeat_at as a standalone progress signal ONLY before the SDK has
   # produced any tool or turn output. Once sdk_ever_output is True, per-turn
   # freshness (sub-check A) is the only Tier 1 signal.
   # Backward-compatible: sessions from before PR #1177 (no tool/turn fields) fall
   # here and behave identically to the pre-fix behavior.
   if not sdk_ever_output:
       hb = getattr(entry, "last_heartbeat_at", None)
       if isinstance(hb, datetime):
           hb_aware = hb if hb.tzinfo else hb.replace(tzinfo=UTC)
           if (now_utc - hb_aware).total_seconds() < HEARTBEAT_FRESHNESS_WINDOW:
               return True
   ```

3. Remove `last_sdk_heartbeat_at` from the Tier 1 check entirely. It is a watchdog-alive
   signal, not a progress signal.

4. Tighten own-progress field guard (gate on `not sdk_ever_output`):
   ```python
   # Own-progress fields: only pass Tier 1 when the SDK has never produced tool/turn output.
   # Once sdk_ever_output is True, per-turn freshness (sub-check A) is authoritative.
   # These fields remain valid as "session ever started" signals for pre-output sessions.
   if not sdk_ever_output:
       if (entry.turn_count or 0) > 0:
           return True
       if bool((entry.log_path or "").strip()):
           return True
       if bool(entry.claude_session_uuid):
           return True
   ```

5. Add reprieve escalation to `_tier2_reprieve_signal` for never-output sessions:
   ```python
   # Escalation guard for sessions that have never produced SDK output (#1226).
   # After MAX_NO_OUTPUT_REPRIEVES consecutive reprieves, suppress "alive" and
   # allow recovery to proceed. Only applies when sdk_ever_output is False —
   # sessions that have produced output are never subject to this cap.
   sdk_ever_output = bool(
       getattr(entry, "last_tool_use_at", None)
       or getattr(entry, "last_turn_at", None)
   )
   reprieve_count = getattr(entry, "reprieve_count", 0) or 0
   if not sdk_ever_output and reprieve_count >= MAX_NO_OUTPUT_REPRIEVES:
       return None  # escalate: suppress all Tier 2 reprieves
   ```
   This block is added before the "compacting", "children", "alive" gate checks in
   `_tier2_reprieve_signal`.

**No changes to:**
- `BackgroundTask._watchdog` — continues to write `last_sdk_heartbeat_at` (watchdog-alive signal, retained for dashboard `last_evidence_at`)
- `_on_heartbeat_tick` in `session_executor.py` — unchanged
- `_heartbeat_loop` — unchanged
- `liveness_writers.py` — unchanged
- `models/agent_session.py` — no new fields needed (`reprieve_count` already exists)

## Risks

### Risk 1: False kills on legitimately long-thinking turns

A session in a 30-minute extended-thinking turn produces no `result` event and no tool
boundaries for the full duration. If `SDK_PROGRESS_FRESHNESS_WINDOW` is set below this
duration, it would be incorrectly killed.

**Impact:** Lost work on long-thinking turns.
**Mitigation:** Default `SDK_PROGRESS_FRESHNESS_WINDOW = 1800s` (30 minutes) is
intentionally generous. Extended-thinking turns (even large ones) complete within 30 min
in practice. Env-tunable for operators who observe edge cases. Tier 2 "alive" gate
continues to reprieve if the process is running children (tool execution).

### Risk 2: Startup window false kill before first tool/turn

A session that authenticates and begins inference but hasn't produced any tool or turn
event yet. `last_tool_use_at` and `last_turn_at` are both None. The old `last_sdk_heartbeat_at`
check is removed.

**Impact:** Premature kill of a session in early inference.
**Mitigation:** The Tier 1c guard retains `last_heartbeat_at` as a progress signal
specifically for the `not sdk_ever_output` case. A fresh `last_heartbeat_at` with
`last_tool_use_at is None` and `last_turn_at is None` still passes Tier 1 — the health
check does not kill the session during the startup window.

### Risk 3: Pre-existing sessions with no `last_tool_use_at` / `last_turn_at`

Sessions started before PR #1177 (merged 2026-04-27) have `None` on both fields because
the hooks weren't writing them yet. If the worker restarts after this fix lands with
pre-#1177 sessions still running, those sessions might appear hung.

**Impact:** Possible false kill of sessions that are actually working.
**Mitigation:** The `not sdk_ever_output` branch (sub-check B) handles this: if both
per-turn fields are None, fall back to `last_heartbeat_at` — exactly preserving the
pre-fix behavior for sessions that predate the Pillar A writes. No migration needed.

### Risk 4: `last_tool_use_at` / `last_turn_at` silently dropped on session recovery/continuation

`_AGENT_SESSION_FIELDS` (if it exists) or equivalent field-copy logic in `_recover_interrupted_agent_sessions_startup`, `_create_continuation_pm`, and any other code that creates new `AgentSession` rows by copying fields from an existing session may not include `last_tool_use_at` or `last_turn_at`. If dropped, a recovered session starts with `sdk_ever_output = False` and falls through to the startup-window behavior rather than retaining the prior progress evidence.

**Impact:** A recovered session's `_has_progress` treats it as never-started (sdk_ever_output False), meaning the startup-window fallback applies. This is conservative (less likely to false-kill) but means the new Tier 1 sub-check A is dormant until the session produces fresh tool/turn events post-recovery.
**Mitigation:** This behavior is safe — the session is in a real startup-equivalent state after recovery. However, the builder must verify that `reprieve_count` is reset to 0 on recovery (to avoid inheriting a high reprieve_count from the pre-recovery hang and immediately triggering the escalation guard). Confirm `reprieve_count` reset is part of the recovery path or explicitly add it.

## Race Conditions

The `_has_progress` function reads ORM fields that are written from two separate
processes: the worker's executor (via `liveness_writers` hooks) and the health check
itself (read-only). Popoto's `DatetimeField` reads are atomic at the Redis layer.
There is no shared mutable in-process state between the writer and the health check —
they run in the same worker event loop but on separate asyncio tasks (the health loop
and the executor). No race conditions identified in the new code path.

The `AGENT_SESSION_HEALTH_MIN_RUNNING = 300s` guard already prevents false kills on
sessions younger than 5 minutes, which covers the startup window independently of the
Tier 1c guard.

## Open Questions

1. **`SDK_PROGRESS_FRESHNESS_WINDOW` value**: Is 30 minutes (1800s) the right default? The issue says "likely 15–30 minutes". We can start at 30 minutes (conservative to avoid false kills) and tighten after observing production data. Env-tunable so no deploy needed to adjust.

2. **Queue-layer `last_heartbeat_at` disposition**: The queue-layer heartbeat is also written unconditionally by `_heartbeat_loop`. Under the new design, a session with stale per-turn fields but fresh `last_heartbeat_at` (and `sdk_ever_output = True`) will fail Tier 1. This is intentional — the executor is alive but the SDK is hung. Confirm this is the correct interpretation before shipping.
