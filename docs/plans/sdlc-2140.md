# valor-session create sets classification_type / issue_url for SDLC sessions

## Problem

SDLC pipelines can be launched two ways: via the Telegram bridge (which classifies the
message before enqueue) or via the CLI — `python -m tools.valor_session create`. The CLI
path never sets `classification_type` on the sessions it enqueues (`tools/valor_session.py`
contains zero classification references). Everything downstream that keys off
`ClassificationType.SDLC` silently degrades.

**Current behavior:**
- Enqueue-time `stage_states` initialization is gated on
  `classification_type == ClassificationType.SDLC` (`agent/agent_session_queue.py:360`). CLI-created
  SDLC sessions skip it, so `AgentSession.stage_states` stays `{}` and the dashboard
  (`ui/app.py`) renders `current_stage: None, stages: []` for the entire run — even while
  `sdlc-tool stage-query --issue-number N` shows `ISSUE/PLAN/CRITIQUE completed` from the
  issue-keyed `PipelineLedger`. Observed live on session `0_1784286820434` (issue #2133).
- The output router's auto-continue rule matches
  `session_type == "eng" and classification_type == "sdlc"` → `nudge_continue`
  (`agent/output_router.py:158`). With `classification_type=None`, a turn-end status update
  falls through to `deliver`, so on a bridge machine the pipeline may pause as if awaiting a
  human instead of auto-continuing.
- `issue_url` is never populated by the CLI path even when the message contains the issue
  URL or an `issue #N` reference, so the dashboard cannot link the session to its issue or
  its ledger-side stage state.

**Desired outcome:**
`valor-session create` with an SDLC-shaped message (contains `issue #N` or a GitHub issue URL)
sets `classification_type=sdlc`, initializes `stage_states` at enqueue (dashboard shows stage
progression from the start), and populates `issue_url` when derivable from the message. CLI-created
SDLC sessions then behave identically to bridge-classified SDLC sessions.

## Solution

### Key Elements

- **`_derive_sdlc_metadata(message, project_config)`** (new helper in `tools/valor_session.py`): returns `(classification_type, issue_url)`. Reuses the same issue-detection philosophy as `_derive_slug_from_message`. Precedence:
  1. If the message contains a full GitHub issue URL → `("sdlc", <that URL>)`.
  2. Else if the message contains `issue #N` (existing `_ISSUE_REF_RE`) → `("sdlc", <URL built from project_config github org/repo>)` (or `("sdlc", None)` if org/repo unavailable).
  3. Else → `(None, None)`. Non-SDLC / conversational messages leave metadata unset.
- **`cmd_create` wiring**: call `_derive_sdlc_metadata` after `project_config` is resolved; pass `classification_type` + `issue_url` into `_push_agent_session`.
- **`issue_url` threading**: add an `issue_url: str | None = None` param to `_push_agent_session` (thread to `async_create(issue_url=...)`) — this is the REQUIRED path, since `cmd_create` calls `_push_agent_session` directly. Also add it to the public `enqueue_agent_session` wrapper (thread to `_push_agent_session`) to keep the wrapper's signature symmetric with the existing `classification_type` param (already present on both), matching the adopted #2152 diff. (Critique concern C2: the wrapper param is not exercised by the CLI path today; we keep it for API symmetry with `classification_type` and to avoid a future re-thread when a bridge/email caller populates `issue_url`. This is a deliberate, minimal 2-line symmetry choice, not scope creep.)
- **No output-router change** (constraint from #1058): the existing rule at `output_router.py:158` already fires once `classification_type=sdlc` is set at creation. We add a test asserting the rule fires for CLI-shaped field values; we do NOT add content inspection to the router.

### Ledger ↔ session-store divergence — recorded decision (Acceptance Criterion 5)

**Decision: Fix at creation time only. Keep the two stores independent; do NOT add a runtime ledger→`stage_states` sync, and do NOT change the dashboard to read the ledger.**

Rationale:
- The divergence in this bug is caused entirely by `classification_type` being unset on the CLI path, which suppresses the enqueue-time `stage_states` init. Setting it at creation restores parity with the bridge path — the minimal, root-cause fix.
- A runtime sync (ledger→session `stage_states`) would introduce new cross-store coupling, a new write path, and periodic reconciliation cost, for a divergence that only exists because of the missing classification. That is over-engineering relative to the appetite.
- Making the dashboard read the ledger directly (via `issue_url`) is a plausible *future* simplification but is a larger architectural change (dashboard currently reads session-keyed `stage_states`, and non-SDLC sessions have no ledger). It is explicitly out of scope here.
- Honors #1058: classification is set at creation time, never inferred at routing/render time.

This decision is durable evidence for the reviewer; it satisfies acceptance criterion 5.

### Flow

CLI invocation with `issue #N` → `cmd_create` derives `slug=sdlc-N`, `classification_type=sdlc`, `issue_url=https://github.com/{org}/{repo}/issues/N` → `_push_agent_session` stores classification in `extra_context`, sets `issue_url`, initializes `stage_states` → dashboard shows `current_stage: ISSUE, stages: [...]` from enqueue → output router nudge-continues turn-end status updates.

### Technical Approach

- Add `_GITHUB_ISSUE_URL_RE` and `_derive_sdlc_metadata()` near `_ISSUE_REF_RE`/`_derive_slug_from_message` in `tools/valor_session.py`.
- `_derive_sdlc_metadata` reads GitHub org/repo from `project_config["github"]` (`org`, `repo`) to construct the URL for a bare `issue #N`; a full URL in the message wins outright.
- Thread `issue_url` through `_push_agent_session` → `async_create` and through `enqueue_agent_session` → `_push_agent_session`. `async_create` already accepts arbitrary model fields; `issue_url` is a real Field.
- No change to `agent/output_router.py`, `ui/app.py`, `agent/pipeline_state.py`, or `models/agent_session.py`.
- No Popoto schema change — `issue_url` and the `classification_type` property already exist; we only start populating them from the CLI path. Therefore no migration in `scripts/update/migrations.py`.

## Rabbit Holes

- **Runtime ledger↔stage_states sync**: tempting "make the stores never diverge" refactor. Out of scope — see the recorded decision. Do not build a reconciliation loop.
- **Dashboard reading the ledger via issue_url**: larger architectural change; non-SDLC sessions have no ledger. Not this plan.
- **Over-broad SDLC detection**: matching bare `pr #N` / `pull request #N` as SDLC could misclassify conversational messages. Keep detection anchored to issue references (`issue #N` or a GitHub issue URL) — the same signal `_derive_slug_from_message` already uses. Do not expand the trigger surface beyond what reliably indicates an SDLC pipeline run.
- **Refactoring the classifier**: do not try to unify the bridge's message classifier with the CLI path. The CLI only needs the narrow issue-reference detection.

## Risks

### Risk 1: Over-classification of non-SDLC CLI sessions
**Impact:** A conversational/teammate session whose message happens to contain `issue #N` gets `classification_type=sdlc`, changing router behavior to nudge-continue.
**Mitigation:** This mirrors the existing `_derive_slug_from_message` behavior (which already auto-provisions an SDLC worktree from `issue #N`), so the classification is consistent with slug derivation. Detection is anchored to explicit issue references, not loose keywords. A teammate session with no issue reference is unaffected (test covers this).

### Risk 2: issue_url built from wrong repo
**Impact:** For a bare `issue #N`, the URL is constructed from `project_config` github org/repo; if the message refers to a different repo's issue, the URL could point to the wrong repo.
**Mitigation:** A full GitHub issue URL in the message always wins (precedence rule 1). For bare `issue #N`, the project's own repo is the correct default — the same assumption the slug/worktree derivation makes. Edge case documented; acceptable for the CLI's single-project invocation model.

## Race Conditions

No race conditions identified. `_derive_sdlc_metadata` is pure and synchronous. The enqueue path already exists and is single-writer per `create` invocation; we add two scalar values (`classification_type`, `issue_url`) to the same `async_create` call that already runs before the `stage_states` init reads back the just-created record. The init block already uses `AgentSession.query.filter(session_id=..., status="pending")` after the create completes (existing, unchanged), so there is no new read-before-write ordering introduced.
