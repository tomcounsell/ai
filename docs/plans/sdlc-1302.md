# SDLC State Machine: Build Must Block on In-Flight Plan Critique

## Problem

The SDLC pipeline state machine treats `/do-plan-critique` and `/do-build` as sequential stages, but the router has no exclusive lock between them. Critique can keep editing the plan after build has already consumed it, with no signal that build's input has gone stale.

**Current behavior:**

- `/do-plan-critique` writes the CRITIQUE verdict via `sdlc-tool verdict record` and emits stage marker `CRITIQUE=completed`. Once that marker lands, the router (Row 4a/4c in `agent/sdlc_router.py`) immediately accepts a `/do-build` dispatch.
- Nothing prevents a *second or third* `/do-plan-critique` round from being dispatched after build has started. The router has no "PLAN/CRITIQUE in revision" lock that blocks BUILD.
- `/do-build` reads the plan once at the start (`Read PLAN_PATH` at WORKFLOW.md Step 1) and never re-reads it. There is no plan-mtime or plan-hash check anywhere in the build skill.
- The `_handle_dev_session_completion()` callback in the worker has no awareness that "PLAN may still be revising" — it simply marks BUILD complete when the dev session emits OUTCOME success.

The cuttlefish #350 incident captured the failure verbatim: round-2 critique flagged a doubly-nested-HTML defect (B-NEW-1) and revised the plan on `main` *after* `/do-build` had already opened PR yudame/cuttlefish#354 and shipped the buggy version. The bug was rediscovered downstream as a review blocker, forcing a `/do-patch` round to apply exactly the fix the critique had prescribed hours earlier.

**Desired outcome:**

- "PLAN is being revised" is an explicit, queryable state on the PM session.
- `/do-build` cannot be dispatched while that state is set; if dispatched and the plan moves under it, build aborts on its own at a checkpoint (defense-in-depth).
- The state has a deterministic clear signal — the plan revision pass commit (or a frontmatter marker) — so the router doesn't deadlock waiting for human input.
- Plan-critique findings reliably block build BEFORE build emits code, eliminating the rediscover-downstream loop.

## Data Flow

The end-to-end flow this plan touches:

1. **Entry point**: PM session invokes `/sdlc 1302` → `sdlc-tool next-skill` → router decides `/do-plan-critique`
2. **Critique runs**: `/do-plan-critique` spawns critics, writes verdict via `sdlc-tool verdict record --stage CRITIQUE`, marks `CRITIQUE=completed`, **and emits the new `plan_revising` lock if the verdict is `NEEDS REVISION` or `READY TO BUILD (with concerns)` (revision pass pending)**
3. **Plan revision** (NEEDS REVISION path): `/do-plan` revises the plan, commits it to `main`, and clears the `plan_revising` lock by setting `revision_applied: true` in the plan frontmatter (existing mechanism, used by Row 4c)
4. **Router re-evaluates**: `sdlc-tool next-skill` checks the `plan_revising` lock as a new G7 guard. If set, dispatching `/do-build` is rejected — router routes to `/do-plan` (or escalates if critique is mid-cycle and no plan dispatch is pending)
5. **Build starts**: Once `plan_revising` is clear AND CRITIQUE is completed AND (verdict is `READY TO BUILD` with no concerns OR `revision_applied: true`), router dispatches `/do-build`. Build resolves `PLAN_REPO` (which may differ from `TARGET_REPO` in cross-repo builds), `git fetch`'s the plan repo, and records `git log -1 --format=%H origin/main -- {PLAN_PATH}` into `_plan_hash_at_build_start`.
6. **Defense-in-depth check inside build**: At Instructions section Step 21 (immediately before the push+PR-create at Step 22), the dev session re-fetches the plan repo and re-reads the plan hash. If it differs from the stored hash, abort the build, mark BUILD `failed`, and let the router re-route to `/do-plan` for a fresh round. This is a single check at PR-creation time, not a per-task-iteration check (per Rabbit Hole below).
7. **Output**: A pipeline that cannot ship code based on a plan whose critique is still settling.

## Architectural Impact

- **Storage model**: `_meta` is computed on-demand by `_compute_meta()` in `tools/sdlc_stage_query.py` from `AgentSession.stage_states`. Persistence happens at the top level of `stage_states` under leading-underscore keys (the existing pattern: `_verdicts`, `_patch_cycle_count`, `_critique_cycle_count`, `_dispatch_history`). New persisted keys: `stage_states["_plan_revising"]` (bool) and `stage_states["_plan_hash_at_build_start"]` (str|None). `_compute_meta()` is extended to surface them as `meta["plan_revising"]` and `meta["plan_hash_at_build_start"]`. No Popoto schema change, no new Redis key.
- **New guard G7**: `guard_g7_plan_revising` in `agent/sdlc_router.py`, evaluated alongside G1-G6.
- **Interface changes**: None to `decide_next_dispatch()` signature; the guard reads `meta` like all others. `record_dispatch()` unchanged.
- **Coupling**: Slightly increases coupling between `/do-plan-critique`, `/do-plan`, and the router — these three skills now collaborate via the `_plan_revising` storage key. This is acceptable: they are already a cohesive plan-stage cluster.
- **Data ownership**: PM session owns `stage_states`. No change in ownership, only two new top-level keys.
- **Reversibility**: Trivial. Remove G7, the meta-set CLI subcommand, and the `_compute_meta()` extension. Stale `_plan_revising` keys in old sessions become inert (no reader consults them).

## Solution

### Key Elements

- **G7 guard in `agent/sdlc_router.py`**: New guard, evaluated before the dispatch table, that blocks BUILD when `meta["plan_revising"]` is truthy AND `meta["revision_applied"]` is falsy. Guard returns `Dispatch(/do-plan)` if a plan revision is the obvious next step, or `Blocked` if the critique appears stuck without follow-up. The `revision_applied` conjunction is critical: if a stale `_plan_revising` flag is left set but the plan frontmatter already shows `revision_applied: true`, the lock self-heals and the guard falls through.
- **Critique sets the lock**: `/do-plan-critique` writes `_plan_revising=true` to `stage_states` whenever its verdict is `NEEDS REVISION`, `MAJOR REWORK`, or `READY TO BUILD (with concerns)`. The write goes through a new `sdlc-tool meta-set --key plan_revising --value true` invocation right after `verdict record`.
- **Plan clears the lock**: `/do-plan` writes `_plan_revising=false` after committing the revised plan to `main`, in the same step that sets `revision_applied: true` in the plan frontmatter (so the two signals stay coupled).
- **Defensive plan-hash check inside `/do-build`**: At BUILD start (Instructions section Step 7, after worktree creation), the build skill resolves the plan's owning repo separately from the build's target repo (the cross-repo case: plan in orchestrator repo, build in target repo), runs `git fetch origin main` against the **plan repo**, then records `git log -1 --format=%H origin/main -- {PLAN_PATH}` into `_plan_hash_at_build_start`. The dev session's pre-PR step (Instructions section Step 21, just before `gh pr create` at Step 22) re-fetches and re-reads the hash from the plan repo. If it changed, build aborts and the router re-dispatches `/do-plan`. Cross-repo resolution is required because `git -C $TARGET_REPO log -- $PLAN_PATH` returns empty when the plan and the build live in different repos (the cuttlefish #350 case this issue addresses).
- **Deadlock backstop**: If `_plan_revising=true` is set but no `/do-plan` is dispatched within `MAX_PLAN_REVISING_DISPATCHES` (default 2) router turns, G7 escalates to `Blocked` so a human resolves it. Default `2` allows one self-healing turn before escalating — aggressive enough to prevent silent deadlocks, permissive enough to absorb the "critique decided round 1 was sufficient and exited without explicit signal" timing edge.

### Flow

**Plan critique completes** → critique sets `plan_revising=True` (if revisions needed) → router sees lock → blocks `/do-build` → routes to `/do-plan` → `/do-plan` revises and clears the lock → router re-evaluates → dispatches `/do-build`

If critique decides round 1 was sufficient with no concerns → critique does NOT set `plan_revising` → router proceeds to `/do-build` as today (no behavior change on the happy path).

### Technical Approach

- **Storage**: persist two new top-level keys on `AgentSession.stage_states`: `_plan_revising` (bool) and `_plan_hash_at_build_start` (str|None). This matches the existing leading-underscore convention used by `_verdicts`, `_patch_cycle_count`, `_critique_cycle_count`, and `_dispatch_history`.
- **Compute layer**: extend `_compute_meta()` in `tools/sdlc_stage_query.py` to surface the two raw keys as `meta["plan_revising"]` (default `False`) and `meta["plan_hash_at_build_start"]` (default `None`). Update `_default_meta()` with the same defaults.
- **CLI**: add `tools/sdlc_meta_set.py` exposing `python -m tools.sdlc_meta_set --key <whitelisted_key> --value <value> --issue-number <N>`. Whitelist keys to `{plan_revising, plan_hash_at_build_start}`. Use `tools/stage_states_helpers.py::update_stage_states()` for the write (existing helper that wraps Popoto save with optimistic concurrency). Fail-soft (exit 0 + `{}` on error) like `sdlc_stage_marker.py`. Append `meta-set` to `ALLOWED_SUBCOMMANDS` in the bash wrapper at `~/.local/bin/sdlc-tool` (the wrapper maps kebab-case `meta-set` → `tools.sdlc_meta_set`). Note: there is no `tools/sdlc_cli.py`; `sdlc-tool` is a shell wrapper that maps each subcommand to a separate `tools/sdlc_<sub>.py` module. The wrapper script is provisioned by `scripts/install_sdlc_tool.sh` (installed to each machine via `/update`); the install script must also be updated.
- **Router guard**: `agent/sdlc_router.py::guard_g7_plan_revising` runs BEFORE Row 4a/4c/5/6/7/8/8b/9/10 (plan revision must beat any BUILD/post-BUILD route). Order: G1, G2, G3, G4, G5, G6, **G7**, then dispatch table. G7 is gated on `meta.get("pr_number") is None` so an already-shipped PR isn't blocked.
- **Critique skill**: `/do-plan-critique` SKILL.md Step 5.5 gains a sibling step that calls `sdlc-tool meta-set --key plan_revising --value true --issue-number {N}` when verdict is NEEDS_REVISION / MAJOR_REWORK / READY-TO-BUILD-WITH-CONCERNS-and-revision-not-yet-applied.
- **Plan skill**: `/do-plan` SKILL.md gains a Phase 4.5 step (after revision commit, before reporting) that calls `sdlc-tool meta-set --key plan_revising --value false --issue-number {N}`. Coupled with the existing `revision_applied: true` write so the two signals never diverge.
- **Build skill (record)**: `.claude/skills-global/do-build/SKILL.md` Instructions section Step 7 (after worktree creation, before agent deployment). The plan repo is *not necessarily* the target repo (cross-repo case). Resolve and fetch the plan repo:
  ```bash
  PLAN_REPO=$(git -C "$(dirname "$PLAN_PATH")" rev-parse --show-toplevel)
  git -C "$PLAN_REPO" fetch origin main 2>/dev/null || true
  PLAN_REL=$(realpath --relative-to="$PLAN_REPO" "$PLAN_PATH")
  PLAN_HASH=$(git -C "$PLAN_REPO" log -1 --format=%H origin/main -- "$PLAN_REL")
  sdlc-tool meta-set --key plan_hash_at_build_start --value "$PLAN_HASH" --issue-number {N}
  ```
- **Build skill (verify)**: Instructions section Step 21 (currently "Verify commits exist before PR" — runs immediately before Step 22's `git push` + `gh pr create`). Add a sibling check after the existing commit-count guard:
  ```bash
  git -C "$PLAN_REPO" fetch origin main 2>/dev/null || true
  CURRENT_HASH=$(git -C "$PLAN_REPO" log -1 --format=%H origin/main -- "$PLAN_REL")
  STORED_HASH=$(sdlc-tool stage-query --issue-number {N} | python -c "import sys,json; print(json.load(sys.stdin)['_meta'].get('plan_hash_at_build_start') or '')")
  if [ "$CURRENT_HASH" != "$STORED_HASH" ]; then
    echo "BUILD ABORT: plan revised mid-build (was=$STORED_HASH, now=$CURRENT_HASH, path=$PLAN_PATH)"
    sdlc-tool stage-marker --stage BUILD --status failed --issue-number {N}
    exit 1
  fi
  ```
  The router will then re-route per existing rules (G7 will catch the next turn if `_plan_revising` is set, or fall through to `/do-plan` via the dispatch table).

## Rabbit Holes

- **Building a full DAG dispatcher (#1307)**: Tempting because it generalizes mutex semantics, but it's an order of magnitude more work and introduces parallel-stage concerns (multi-dev fan-out, partial completion). #1302 is the prerequisite mutex primitive; ship it standalone.
- **Per-plan-doc file locks on the filesystem**: Solves a different problem (concurrent writers to the same `.md` file). Use the in-memory PM-session lock instead — it scopes to one issue's pipeline.
- **Re-architecting `/do-build` to re-read the plan at every task boundary**: Tempting "fully reactive" approach. The cost is high (every task iteration re-parses the plan), the benefit duplicates G7. Single hash check at PR-creation time is enough.
- **Adding a new pipeline stage `PLAN_REVISING` to `PIPELINE_EDGES`**: Sounds clean, breaks `DISPLAY_STAGES` invariants, every parity test, and every dashboard. Use `_meta` flag instead — same effect, zero schema impact.

## Risks

### Risk 1: Lock leaks if critique crashes mid-step
**Impact:** `plan_revising=True` is set but no `/do-plan` is ever dispatched, pipeline deadlocks.
**Mitigation:** G7 escalates to `Blocked` after `MAX_PLAN_REVISING_DISPATCHES` (default 1) router turns where the lock is set but no plan-stage skill is in the recent dispatch history. Operator unblocks manually via `sdlc-tool meta-set --key plan_revising --value false`.

### Risk 2: Build skill races to record `plan_hash_at_build_start` after critique already started revising
**Impact:** Build's recorded hash is the *post-revision* hash, the defensive check at Step 21 sees no diff, and the same race re-occurs.
**Mitigation:** G7 fires *before* the router would dispatch `/do-build` — if critique set `plan_revising=True` after build dispatched, G7 still triggers on the next router turn and the next `/sdlc` invocation routes back to `/do-plan`. Combined with the build-side hash check (which catches plan revisions that committed *during* the build's execution), this is two independent safeguards.

### Risk 3: G7 ordering vs G6 (terminal merge fast-path)
**Impact:** If G7 runs before G6, a stuck `plan_revising` lock could prevent a legitimate merge of an already-shipped PR.
**Mitigation:** G7 is gated on `pr_number is None` — if a PR exists, G7 returns `None` and lets G6/G3 run. Same pattern Row 4 family uses.

## Race Conditions

### Race 1: Critique writes lock concurrently with router reading meta
**Location:** `tools/sdlc_meta_set.py` (new) write vs `tools/sdlc_stage_query.py` read
**Trigger:** Critique writes `plan_revising=True` mid-router-evaluation; the router's snapshot is from before the write.
**Data prerequisite:** `_meta` dict must be the latest committed Popoto state when the router reads it.
**State prerequisite:** `AgentSession.stage_states["_meta"]` is consistent (single Popoto instance per session).
**Mitigation:** Use the existing `tools/stage_states_helpers.py::update_stage_states()` write helper (already wraps Popoto save with optimistic concurrency) for the meta-set write. Reads use `Model.query.filter()` which gets the latest committed state. Worst case, the router runs once with stale state and re-evaluates on the next `/sdlc` turn — eventually consistent, no permanent corruption.

### Race 2: Build records hash before critique commits revision
**Location:** `/do-build` Step 7 vs `/do-plan` revision commit
**Trigger:** Build computes `plan_hash_at_build_start = SHA1`, then critique-driven revision commits `SHA2`, then build's Step 21 sees mismatch.
**Data prerequisite:** Plan file's git history must reflect any revision before Step 21 reads it.
**State prerequisite:** `git log -1 -- {PLAN_PATH}` is read against the worktree (which sees `main` commits as `git fetch` runs on session start).
**Mitigation:** This is the *intended* path — Step 21 mismatch triggers build abort. The defensive check is the second layer of the lock.

## No-Gos (Out of Scope)

- DAG dispatcher / parallel-stage execution (#1307) — separate plan, prerequisite is this mutex.
- Multi-dev session fan-out (#1308) — separate plan.
- Per-file lock infrastructure on `docs/plans/*.md` — wrong scope; the mutex is per-pipeline, not per-file.
- Locking REVIEW or DOCS stages — same race could exist, but is not in scope and has not surfaced as a real-world failure case. Add later if observed.
- New CRITIQUE-after-build path (re-running critique after review surfaces something new) — this stays one-directional per the issue's explicit constraint: "preserve the ability to dispatch additional critique rounds AFTER build, if review surfaces something the round-2 critique missed". The G7 guard is gated on `pr_number is None` for exactly this reason.
