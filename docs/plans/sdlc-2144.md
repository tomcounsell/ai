# Self-Healing SDLC Run Identity on Resumed Pipeline Turns

## Problem

SDLC pipeline state (stage markers, verdicts, dispatch records, `_meta`
keys) is written via `sdlc-tool` state-mutating subcommands that each require
a **run identity** — a `run_id` minted exactly once by `sdlc-tool
session-ensure` (`tools/sdlc_session_ensure.py`), persisted to `.sdlc-run` in
the worktree, mirrored to `AgentSession.active_run_id`, and pinned on a
TTL-bounded Redis issue lock (`session:issuelock:{issue_number}`). Every
state-mutating call must pass that `run_id` back via `--run-id`. A missing
flag is a hard, named refusal `RUN_ID_REQUIRED` (`tools/sdlc_dispatch.py:338-345`,
mirrored in `tools/sdlc_stage_marker.py`, `tools/sdlc_verdict.py`,
`tools/sdlc_meta_set.py`); an expired/foreign lease surfaces `LEASE_ABSENT`
or `ISSUE_LOCKED` via `tools/_sdlc_utils.py::resolve_ledger_lease`.

By skill convention every best-effort marker write is wrapped
`2>/dev/null || true` (see `docs/sdlc/do-plan.md:12`, `docs/sdlc/do-build.md:85`,
`docs/sdlc/do-pr-review.md:43`, etc.). So when a refusal happens it is
**doubly silent**: stderr is discarded by `2>/dev/null`, and the non-zero
exit is swallowed by `|| true`.

**The failure (observed live 2026-07-17, issue #2133, session `0_1784286820434`):**
The `run_id` lives only in the killed turn's conversation context. When a PM
turn is killed mid-pipeline (worker restart — #2141/#2143) and the session
resumes from transcript (`agent/session_runner/runner.py::_resume_active`,
line 448-464), the resumed turn continues pipeline work **without
re-establishing run identity**. It no longer carries the `run_id`, so every
subsequent `stage-marker`/`verdict`/`meta-set`/`dispatch` write refuses with
`RUN_ID_REQUIRED` (no `--run-id` in context) or `LEASE_ABSENT` (a stale
`run_id` whose lock TTL lapsed), and the ledger **freezes while real work
proceeds**. In the live case the ledger stayed `BUILD: ready` / `TEST:
pending` for ~2 hours of wall-clock while the plan was revised, the fix
committed, and PR #2142 opened with CI green. The confirmed comment on the
issue narrows the root cause precisely: turn 4 re-ran `session-ensure` and
markers immediately resumed flowing — so **marker writes work fine once run
identity is re-established; the silent-failure window is exactly the resumed
turns that continue pipeline work without re-ensuring.**

This is the state-tracking twin of #2138: there the *agent's* claims diverge
from reality; here the *system's own record* diverges. `stage-query`, the G4
oscillation guard, `/sdlc` routing, and any operator reading pipeline state
all see a frozen pipeline while the PR advances.

## Freshness Check

- `tools/sdlc_dispatch.py:336-345` — the `RUN_ID_REQUIRED` hard-exit gate in
  `main()` (identical shape in `sdlc_stage_marker.py:355-361`,
  `sdlc_verdict.py`, `sdlc_meta_set.py`). Confirmed present and unconditional.
- `tools/_sdlc_utils.py::resolve_ledger_lease` (line 568) returns
  `{"reason": "LEASE_ABSENT"}` when the lock is unheld — the second refusal
  mode a resumed turn hits when it carries a *stale* `run_id`.
- `tools/sdlc_session_ensure.py::ensure_session` (line 362) +
  `_acquire_run_lock_and_bind` (line 129): `--reuse-run-id` is honored only
  when `_validated_reuse_candidate` (line 82) proves continuity (live-lock
  owner match, or free lock + record-mirror match). This is the exact,
  already-built re-establishment primitive the fix leans on.
- `agent/supervised_run.py` — the supervised-run signal (Redis key
  `session:supervisedrun:{issue}` + worktree `.sdlc-run` file) makes a bare
  `session-ensure` under a live lease return `SUPERVISED_RUN_ACTIVE` carrying
  the owner `run_id` to inherit. Liveness is anchored on the issue lock.
- `agent/session_runner/runner.py:448-464` — `_resume_active` seeds
  `--resume` and reintroduces the dev agent, but does **nothing** about SDLC
  run identity. No re-ensure, no `.sdlc-run` read.
- The live-ops working pattern all week has been
  `sdlc-tool session-ensure --reuse-run-id <id>` after lease expiry — i.e.
  humans are manually doing exactly what the tool should self-heal.

## Prior Art / Context

- **#2003** introduced run identity + the issue lock (the exclusive minting
  site + `--reuse-run-id` verified-echo path).
- **#2012** moved the ledger onto the durable issue-keyed `PipelineLedger`
  (`agent/pipeline_ledger.py`) that the markers write into; refusals are
  intentionally LOUD at the tool boundary (there is no session to fall back
  to).
- **#2026 (WS1)** added the single-owner supervised-run signal so stage forks
  *inherit* the supervisor's `run_id` instead of re-minting — the mechanism a
  resumed turn can ride to re-establish identity when the lease is still live.
- **#2141 / #2143** are the restart sources that create the mid-turn
  interruptions triggering this bug.
- **#2136 / #2138** are companion resume-context losses (goal, dev_agent_id,
  completion evidence). `run_id` is another piece of state that lived only in
  the killed turn's context — this plan closes that specific gap.

## Solution

The principal's live-ops guidance is weighted heavily and decides the shape:
**make marker writes self-heal (re-ensure + retry) rather than silently
refusing. Visibility, not fatality — markers stay best-effort and never block
pipeline work.** The fix lives in the **tool**, not in skill prose, so it is
deterministic and cannot be forgotten by a resumed LLM turn that doesn't know
it was resumed.

### Component 1 — Shared self-heal helper (`tools/_sdlc_run_identity.py`, new)

A single function `reestablish_run_id(issue_number, prior_run_id, working_dir)
-> str | None` that all four state-mutating CLIs call when they would
otherwise refuse for a run-identity reason (`RUN_ID_REQUIRED`, `LEASE_ABSENT`).

**CRITICAL correctness fix (CRITIQUE BLOCKER, flagged independently by two
panels).** The naive design "always call `ensure_session(reuse_run_id=
candidate)` and read a `SUPERVISED_RUN_ACTIVE` return" is WRONG: that inherit
branch in `_acquire_run_lock_and_bind` is guarded by `if not reuse_run_id:`
(`tools/sdlc_session_ensure.py:199`), so passing ANY `reuse_run_id` makes the
supervised-inherit path **structurally unreachable**. Worse, a resumed turn
carrying its own *stale* `run_id` as the candidate then fails
`_validated_reuse_candidate` against the live supervisor lock (owner != stale
claim) → fresh-mint contest → `ISSUE_LOCKED` on a live foreign lease, or a
*competing* `run_id` on a free lock. So the algorithm must consult the
supervised signal directly and never route a supervisor's id through
`reuse_run_id`. Corrected algorithm:

1. **Supervised-inherit first (direct, not via `reuse_run_id`).** Call
   `agent.supervised_run.supervised_run_status(issue_number, working_dir)`.
   If `.live`, **return `status.run_id` directly** — this reaches the #2026
   inherit mechanism without touching the reuse gate.
2. **Else resolve an env-corroborated reuse candidate**, precedence corrected
   so a corroborated environment signal outranks a stale caller-supplied id:
   worktree `.sdlc-run` / `read_supervised_run_signal(...)["run_id"]` →
   session record's `active_run_id` → `prior_run_id` (last, since a resumed
   turn's own `--run-id` is the value most likely to be stale).
   **Non-supervised note (CRITIQUE CONCERN, History panel):** for a
   bridge-originated PM pipeline (the exact #2133 case) there is NO supervised
   signal and NO `.sdlc-run` file — those are written only for `/do-sdlc`
   supervised runs (`agent/supervised_run.py`). In that scenario healing rests
   **solely** on `AgentSession.active_run_id` surviving the resume (the record
   mirror `_resume_active` never touches — `runner.py:448-464` seeds resume
   scalars only). This dependency is now stated explicitly and covered by a
   dedicated non-supervised test.
3. Call `ensure_session(issue_number, reuse_run_id=candidate)`. On a
   free/expired lock this re-acquires under the caller's own identity (via
   `_validated_reuse_candidate`'s free-lock + record-mirror branch) or mints
   fresh.
   **Terminal guard (CRITIQUE CONCERN, Scope panel):** before returning a
   *freshly minted* `run_id` (the reuse candidate was NOT corroborated and a
   new uuid was acquired on a lapsed lock), peek the issue's `PipelineLedger`
   (`PipelineStateMachine.for_issue(target_repo, issue)`); if `MERGE ==
   completed` (terminal), **return `None`** — decline to heal rather than
   resurrect a finished run's lease and show a "live" pipeline the operator
   wanted dead. The supervised-inherit and corroborated-reuse paths are exempt
   (they re-establish the *same* run, never a new lease on a closed pipeline).
4. A foreign live holder (`ISSUE_LOCKED`) → return `None` (cannot heal).

The helper is fail-open (any exception → `None`) and performs **no writes of
its own** beyond what `ensure_session` already does under its lock contract.

### Component 2 — Wire self-heal into the four state-mutating CLIs

In `sdlc_dispatch.py`, `sdlc_stage_marker.py`, `sdlc_verdict.py`,
`sdlc_meta_set.py`: replace the unconditional `RUN_ID_REQUIRED` hard-exit and
the `LEASE_ABSENT` refusal with a single self-heal attempt. On refusal for a
run-identity reason **and** a resolvable `--issue-number`, call
`reestablish_run_id`, and if it returns a `run_id`, retry the write once under
it. Only a genuinely unhealable state (foreign `ISSUE_LOCKED`, or no
issue-number to key on) still refuses — and that refusal is now *recorded*
(Component 3), not swallowed. Exit code stays 0 on a healed write so `|| true`
is a no-op; the write actually lands.

Guardrails preserved: `session-ensure` remains the exclusive minting site;
the no-adopt invariant holds (self-heal only ever *reuses/echoes* a candidate
the environment corroborates, or inherits a live supervisor's id — never
adopts a foreign live lease); a foreign live holder still hard-refuses.

### Component 3 — Make refusals visible (acceptance criterion 2)

Since both stderr and exit code are swallowed by the skill convention,
visibility must land in a durable sink the operator/session can read.

**Single sink: an operator-readable JSON-lines log file (CRITIQUE CONCERN,
Scope + Operator panels — dropped the redundant Redis key).** The plan
originally proposed *two* sinks (a log file AND a raw-Redis counter), but the
Redis key had no defined consumer (its only proposed reader — the dashboard —
is explicitly a follow-up), and a counter named `refusals` incremented on
*every* self-heal attempt would conflate recoveries with genuine unhealable
refusals, reading alarmingly high on a healthy pipeline that self-heals on the
resume edge. So we ship the log file alone and drop the Redis key until a
concrete reader lands.

Emit one JSON line per self-heal attempt (healed or not) to
`logs/sdlc_run_identity.log`:
`{ts, issue, subcommand, reason, healed: bool, old_run_id, new_run_id}`. The
`healed: bool` field is the ground truth that separates a recovery from an
unhealable refusal (what a Redis counter could not do without splitting into
two keys). Append is best-effort, fail-open on `OSError`.

**Path resolution (CRITIQUE CONCERN, Operator panel):** the log path MUST
resolve against the repo / git-common-dir root, NOT a cwd-relative `logs/` — a
worktree-isolated run (`.worktrees/{slug}/`) writing to a cwd-relative `logs/`
would be invisible to an operator tailing the canonical repo's
`logs/sdlc_run_identity.log`. Resolve via the git common dir (the same
approach the full-suite pytest lock uses) so all worktrees converge on one
file.

### Component 4 — Frozen-ledger-repair decision (acceptance criterion 3)

**Decision: do NOT add an artifact-inference `--reconcile`.** The SDLC
SKILL.md invariant is explicit and load-bearing — "Never infer stage
completion from artifacts (plan files, PR existence, docs/ files);
completion is exclusively determined by stored state" (`.claude/skills/sdlc/SKILL.md`).
A `--reconcile` that recomputes stage states from a plan doc / PR / CI would
directly violate that invariant and re-introduce the exact
artifact-vs-record ambiguity #2012 removed. Instead:

- **Forward repair is automatic:** with self-heal, the *next* state-mutating
  write after a resume re-establishes identity and lands correctly, so the
  ledger self-repairs going forward without inference.
- **Back-fill of the freeze window is a bounded manual operation** using the
  existing, now-self-healing `stage-marker` / `verdict record` writes — an
  operator (or the resumed pipeline itself) issues the missed completion
  markers explicitly. This keeps every ledger mutation grounded in an
  explicit, authorized `run_id` write, never in inference.

This decision is recorded here and mirrored into the SDLC docs (see
Documentation).

### Component 5 — Live cleanup of #2133 (acceptance criterion 4)

**This is a post-merge OPERATIONAL checklist item, not a build-blocking code
component (CRITIQUE NIT, Scope panel)** — the code PR's Definition-of-Done is
not gated on a hand-run reconcile against live state. Track it separately and
run it once the code lands.

**Do NOT depend on fresh-mint self-heal for the back-fill (CRITIQUE CONCERN,
Adversary panel).** For a done/stale issue whose lock long expired, each
bare `stage-marker` would fresh-mint a *new* `run_id` and `SET NX` a fresh
live lease on #2133 — not safely repeatable (each of the four marker writes
could land under a different id if the lease lapses between commands), and it
risks racing session `0_1784286820434` if that session is still resumable.
Instead, the reconcile procedure is:

1. Precondition: confirm `AgentSession` `0_1784286820434` (and any other
   #2133 session) is in `TERMINAL_STATUSES` — not resumable.
2. Recover the authoritative `run_id` from the issue's `active_run_id` mirror
   / `.sdlc-run`; if none survives, mint one explicit `session-ensure` and
   reuse its id for all writes.
3. Issue all four `stage-marker --status completed` (BUILD/TEST/REVIEW/MERGE)
   + the REVIEW `verdict record` under that **one** explicit `--run-id` inside
   a single lease window (re-touch between commands if needed).
4. Verify with `sdlc-tool stage-query --issue-number 2133`; the reconcile
   events also land in `logs/sdlc_run_identity.log` so a competing mint is
   detectable after the fact.

Manual reconcile is explicitly acceptable per the issue's acceptance criteria.

## Data Flow (post-fix)

```
resumed PM turn → sdlc-tool stage-marker --stage BUILD ... [no/stale --run-id]
  → resolve_ledger_lease → LEASE_ABSENT / gate → RUN_ID_REQUIRED
  → reestablish_run_id(issue, prior, working_dir):
        1. supervised_run_status(issue).live ? return status.run_id (inherit, direct)
        2. candidate = .sdlc-run | active_run_id | prior   (env-corroborated first)
        3. ensure_session(reuse_run_id=candidate) → re-acquire(same id) | mint(fresh)
             fresh + MERGE==completed ? return None (terminal guard)
        4. foreign live lease → ISSUE_LOCKED → return None
  → healed run_id ? retry write under it (exit 0, ledger advances)
                  : append refusal line to logs/sdlc_run_identity.log (exit non-zero, still || true)
```

## Why Previous Fixes Failed

Nothing previously attempted this — #2003/#2012/#2026 all *tightened*
run-identity enforcement (LOUD refusals, no-adopt) to stop competing writers,
which is correct for the concurrency threat but produced the exact silent
freeze here when the *legitimate same-run* caller lost its context on resume.
The self-heal path threads that needle: it re-establishes the *same* run's
identity (via the already-built verified-reuse / supervised-inherit paths)
without ever adopting a *foreign* live lease.

## Appetite

Small-to-medium. The re-establishment primitive (`ensure_session` +
`--reuse-run-id` + supervised-run inherit) already exists; the work is a
shared helper, four ~10-line call-site edits, a visibility sink, docs, and
the one-time #2133 cleanup. No new locking semantics, no schema changes.

## Failure Path Test Strategy

- **Self-heal on `RUN_ID_REQUIRED` (no `--run-id`), lease free:** seed a
  `.sdlc-run` / `active_run_id`, expire the lock, call `stage-marker` with no
  `--run-id`; assert `reestablish_run_id` re-acquires and the write lands
  (ledger shows the stage), exit 0.
- **Supervised run live (direct inherit, CRITIQUE-corrected):** a LIVE
  supervised signal for the issue; call `reestablish_run_id` with a stale
  `prior_run_id`; assert it returns the *supervisor's* `run_id` via the
  `supervised_run_status().live` branch — NOT routed through
  `ensure_session(reuse_run_id=...)` (which would refuse `ISSUE_LOCKED`) — and
  the write lands under the supervisor's id.
- **Non-supervised bridge PM heal (CRITIQUE-added):** `read_supervised_run_
  signal` returns `None` and only `active_run_id` is set (the #2133 shape);
  expire the lock; assert `ensure_session(reuse_run_id=active_run_id)`
  re-acquires via the free-lock + record-mirror branch and the write lands.
- **Terminal guard (CRITIQUE-added):** free/expired lock, reuse candidate NOT
  corroborated, but the issue's ledger has `MERGE == completed`; assert
  `reestablish_run_id` returns `None` (declines to resurrect a closed run)
  rather than fresh-minting a new lease.
- **Unhealable foreign `ISSUE_LOCKED`:** a foreign live holder; assert
  self-heal returns `None`, the write is refused, the refusal is recorded to
  `logs/sdlc_run_identity.log` (visibility criterion), and pipeline work is
  never blocked (exit non-zero tolerated by `|| true`).
- **End-to-end resume (acceptance criterion 1):** simulate kill-mid-BUILD +
  resume (via the resume harness, see Test Impact) and assert the next
  `stage-marker` write succeeds without a manual `session-ensure`.
- **Fail-open:** force an exception inside `reestablish_run_id`; assert it
  returns `None` and never raises into the calling tool.

## Rabbit Holes

- **Do not** touch the locking/TTL semantics or `touch_issue_lock` — self-heal
  rides the existing contract; changing it risks re-opening the concurrency
  bugs #2003/#2012 closed.
- **Do not** build `--reconcile`/artifact inference (Component 4 decision).
- **Do not** try to re-inject `run_id` into the resumed LLM's conversation
  context — the fix is intentionally in the tool, context-independent.
- **Do not** move visibility into the Popoto-managed `PipelineLedger`
  stage_states (that write itself needs identity — chicken-and-egg); the log
  file is deliberately outside the gated path.
- **Do not** re-add the raw-Redis observability counter — dropped per CRITIQUE
  (no consumer; conflates healed vs refused). Revisit only if the dashboard
  follow-up gives it a concrete reader.
- **Do not** widen self-heal to read-only subcommands — they take no `run_id`
  and never refuse for identity.

## Risks

- **Self-heal minting a competitor `run_id`** — mitigated two ways: a live
  supervised run is inherited *directly* via `supervised_run_status()` (never
  routed through `reuse_run_id`, which would bypass the inherit gate — the
  CRITIQUE BLOCKER); a fresh mint happens only on a genuinely free/expired
  lock, and even then the terminal guard declines to mint on a `MERGE ==
  completed` pipeline so a closed run is never resurrected.
- **Retry loop / thrash** — mitigated: self-heal is attempted **once** per
  call; a `None` result refuses immediately (no loop), preserving best-effort
  semantics.
- **Resurrecting a terminally-done pipeline** — mitigated by the terminal
  guard on the fresh-mint branch (CRITIQUE CONCERN); inherit and
  corroborated-reuse re-establish the *same* run only.
- **Worktree-invisible log** — mitigated: the log path resolves against the
  git-common-dir root, so all worktrees write one operator-tailable file
  (CRITIQUE CONCERN).
- **Log-file growth** — mitigated: one compact JSON line per event,
  low-frequency (only on the resume edge); no rotation needed at this volume,
  documented as an operator-readable diagnostic.

## No-Gos

- No changes to issue-lock TTL, `touch_issue_lock`, or `release_issue_lock`.
- No artifact-inference reconcile.
- No new Popoto model or schema change.
- No mutation of a foreign live lease under any circumstance (no-adopt
  invariant preserved).
- No skill-prose-only fix — the healing must be structural in the tool.

## Success Criteria

- A resumed pipeline turn re-establishes run identity before its first
  successful state-mutating write, with **no** manual `session-ensure`
  (verified by the kill-mid-BUILD → resume → marker-succeeds test).
- `RUN_ID_REQUIRED` / `LEASE_ABSENT` refusals (and heals) are recorded to
  `logs/sdlc_run_identity.log` with a `healed: bool` field, at a
  git-root-resolved path visible across worktrees (operator-visible).
- Self-heal never resurrects a terminally-done pipeline: a fresh mint on a
  `MERGE == completed` issue returns `None` (terminal guard), verifiable via
  `stage-query` staying terminal after a stray marker attempt.
- The frozen-ledger-repair decision (no artifact reconcile; forward-heal +
  bounded manual back-fill under one explicit `--run-id`) is recorded in the
  plan and the feature doc.
- #2133's ledger reflects reality after the post-merge operational reconcile
  (`stage-query --issue-number 2133`).
- Markers remain best-effort: no self-heal path ever blocks or crashes
  pipeline work; a foreign live lease still hard-refuses.
