# Memory Health Audit: 3-Layer Always-Apply Reflection

## Problem

The `memory-quality-audit` reflection currently flags zero-access and low-confidence records but does nothing about pre-fix junk that PR #1217 left behind, and it cannot detect new misconfiguration symptoms when the extractor breaks again. Two pain points:

1. **Pre-fix junk is still polluting the dashboard.** On this machine, 1,515 of 1,875 non-superseded `extraction-*` records (81%) match the `_looks_like_refusal` predicate — JSON shrapnel like `"category": "decision",` and refusal prose like `"There is no agent session response to analyze..."`. PR #1217 shipped `scripts/cleanup_memory_extraction_junk.py` but it requires a human to remember to invoke `--apply` on each machine.
2. **No continuous detection of new memory-integration anomalies.** If the extractor regresses (different escape bug, new refusal phrase Haiku adopts, an extractor stuck producing duplicates from a single session, importance/category distribution skew), nothing surfaces it. We notice only via manual dashboard inspection.

**Current behavior:** `run_memory_quality_audit` walks `Memory.query.all()`, flags zero-access + low-confidence as findings, and logs them. No supersedes, no anomaly detection, no GitHub issues.

**Desired outcome:** `run_memory_quality_audit` becomes a 3-layer audit that runs on its existing daily schedule and (a) auto-supersedes known-bad records using the same predicate that gates new writes, (b) detects misconfiguration symptoms the regex doesn't catch and surfaces them as GitHub issues with evidence, and (c) self-quiesces — once the corpus is clean and the extractor is healthy, reports `0 superseded, 0 anomalies, 0 issues filed` and stays quiet until something breaks again.

## Data Flow

End-to-end, per audit invocation:

1. **Trigger:** Reflection scheduler (`agent/reflection_scheduler.py`) wakes the `memory-quality-audit` job per its 86400s interval (`config/reflections.yaml:298`).
2. **Entry point:** `reflections/memory_management.py::run_memory_quality_audit()` is called with no args, returns `{status, findings, summary}`.
3. **Layer 0 — Legacy zero-access + low-confidence flagging (preserved).**
   - Iterate `Memory.query.all()` (the same load shared with Layer 1; do not re-query).
   - Flag records with `access_count == 0 AND age > 30d` and records with `confidence < 0.2`.
   - Append the existing finding strings (zero-access / low-confidence summaries) — read-only, no supersede, no issue.
4. **Layer 1 — Deterministic supersede.**
   - Load `Memory.query.all()`.
   - Filter to `agent_id.startswith("extraction-")` AND `superseded_by == ""` AND `_looks_like_refusal(content)`.
   - For each (capped at `MAX_PRUNE_PER_RUN` = 50): set `superseded_by = "cleanup-junk-extraction"` and `superseded_by_rationale = "auto-cleanup: refusal/json-shrapnel from issue #1212"`. `record.save()`.
   - Append finding `"Layer 1: superseded N of M candidates (cap=50)"`.
5. **Layer 2 — Heuristic anomaly detection.**
   - Re-load (or filter from prior load) the **post-supersede** non-superseded `extraction-*` corpus.
   - Compute four signals against last-7d and last-24h windows. Each signal that crosses threshold is appended as an "anomaly candidate" with name + counts + 3-5 sample memory_ids.
6. **Layer 3 — Gemma classification (fail-soft).**
   - Sample up to 20 last-24h non-superseded `extraction-*` records that **passed** Layer 1.
   - Batch through `ollama.chat(model=OLLAMA_LOCAL_MODEL, messages=..., options={"temperature": 0})` with one positive + one negative few-shot.
   - Each `is_junk: true` response with `anomaly_signal != null` is appended as an "anomaly candidate".
   - On any exception (Ollama daemon down, model missing, network error): log debug, skip Layer 3.
7. **Issue surfacing.**
   - For each anomaly candidate (Layers 2 + 3 only — Layer 0 flags and Layer 1 supersedes never produce candidates):
     - Compute issue title: `[memory-audit] {signal_name}: {observed} (threshold {threshold})`.
     - Search for existing open issue with matching title prefix and same `signal_name`. If found: skip (idempotent on re-runs).
     - Else: invoke `gh issue create --label memory --label investigation --title ... --body ...` with the body template defined in **Solution → Issue Body Template**.
8. **Return:** `{"status": "ok", "findings": [...], "summary": "Memory health audit: N superseded, M anomalies, K issues filed"}`.

## Solution

### Key Elements

- **Layer 0 (legacy zero-access + low-confidence flag — preserved):** Retain the existing audit body verbatim — iterate `Memory.query.all()`, flag `access_count == 0 AND age > 30d` and `confidence < 0.2`. **Read-only**, files no GitHub issues, only appends to `findings`. This is the resolution to Open Question Q3 (Critique C4): Layer 0 is preserved, not migrated, because (a) it operates on the entire `Memory` corpus, not just `extraction-*` records, so it provides orthogonal signal coverage; (b) it has no false-positive risk (read-only); (c) deleting it would silently regress observability for non-extraction records (post-merge learnings, human saves, telegram memories) where the new Layers 1-3 are blast-radius-gated to `extraction-*` only.
- **Layer 1 (regex supersede):** Imports `_looks_like_refusal` from `agent.memory_extraction` directly. Operates on `extraction-*` records only. Caps at 50 supersedes/run. Always-apply.
- **Layer 2 (heuristic anomaly detection):** Four pure-Python signal computations against the post-Layer-1 `extraction-*` corpus, each with a documented threshold.
- **Layer 3 (gemma classification):** `ollama.chat` with `gemma4:e2b`, two-example few-shot prompt, `temperature=0`, JSON response shape. Wrapped in try/except for fail-soft. Enforces a 30s wallclock budget (see Layer 3 section).
- **Anomaly issue surfacing:** New helper `_file_anomaly_issue(signal_name, observed, threshold, sample_ids, evidence)` that searches for an open dup before invoking `gh issue create` via subprocess.
- **Cleanup script deletion:** Delete `scripts/cleanup_memory_extraction_junk.py` and its test file `tests/unit/test_cleanup_memory_extraction_junk.py` once Layer 1 is wired in.

### Flow

Reflection scheduler → `run_memory_quality_audit()` → Layer 0 (legacy flag, read-only) → Layer 1 (supersede capped) → Layer 2 (compute 4 signals) → Layer 3 (gemma sample, fail-soft, 30s budget) → For each layer-2/3 candidate: dup-check → file issue → return findings (Layer 0 entries appear in findings list but never surface issues).

### Technical Approach

#### Layer 0 — Legacy Flagging (Preserved In-Place)

The existing `run_memory_quality_audit` body (lines 154-228 of `reflections/memory_management.py`, baseline `bfa0b09c`) is preserved verbatim as the first phase of the new audit. It computes `flagged_zero_access` and `flagged_low_confidence` against the full `Memory.query.all()` corpus and appends summary findings. **No code changes required for Layer 0** — the new layers are appended around it. #### Layer 1 — Deterministic Supersede (Regex)

```python
# reflections/memory_management.py (appended after Layer 0 body in run_memory_quality_audit)
from agent.memory_extraction import _looks_like_refusal

CLEANUP_SUPERSEDED_BY = "cleanup-junk-extraction"
CLEANUP_RATIONALE = "auto-cleanup: refusal/json-shrapnel from issue #1212"
MAX_LAYER1_SUPERSEDES_PER_RUN = MAX_PRUNE_PER_RUN  # 50, reuses module constant

# Inside the audit, after loading all_memories:
extraction_records = [m for m in all_memories
                       if str(m.agent_id or "").startswith("extraction-")]
layer1_candidates = [m for m in extraction_records
                      if not (m.superseded_by or "")
                      and _looks_like_refusal(m.content or "")]

superseded = blocked = 0
for m in layer1_candidates[:MAX_LAYER1_SUPERSEDES_PER_RUN]:
    try:
        m.superseded_by = CLEANUP_SUPERSEDED_BY
        m.superseded_by_rationale = CLEANUP_RATIONALE
        result = m.save()
        if result is False:
            blocked += 1
        else:
            superseded += 1
    except Exception as e:
        logger.warning(f"layer1 save failed for {m.memory_id}: {e}")
```

The same `_iter_candidates` predicate logic from `scripts/cleanup_memory_extraction_junk.py:76-94` is reused via the imported `_looks_like_refusal` predicate. The blast-radius check (`agent_id.startswith("extraction-")`) is restated explicitly, matching the script convention.

#### Layer 2 — Heuristic Anomaly Detection (No Model)

Compute against the **post-Layer-1** non-superseded `extraction-*` corpus (i.e., refresh the `extraction_records` list and exclude any record we just superseded in this run, plus all already-superseded). Four signals:

| Signal name (used in issue title + dup detection) | Computation | Threshold | Rationale |
|---|---|---|---|
| `category-default-skew` | Of last-7d non-superseded `extraction-*` records, fraction where `_has_no_category(metadata)` (see canonical predicate below) | **> 0.70** | Healthy extractor produces categorized observations. On this machine, the post-Layer-1 surviving corpus is 98.3% no-category — broken state. 70% buffers the noise floor (pattern + surprise records can legitimately use uncategorized fallback). |
| `importance-1.0-skew` | Of last-7d non-superseded `extraction-*` records, fraction with `importance == 1.0` | **> 0.85** | `pattern` and `surprise` legitimately get 1.0; `correction` and `decision` should be 4.0. On healthy data we expect 60-80% at 1.0. 85% catches the case where corrections/decisions are never produced. |
| `agent-id-cluster` | Any single `agent_id` produced **> 10** Layer-1-junk records **whose `created_at` is in the last 24h AND that were superseded in *this* run** | **> 10** | One stuck session producing dozens of shrapnel rows is a strong signal the extractor looped on a malformed response. This machine has a 273-record cluster from a single session. **Idempotency (C5):** the count is restricted to records superseded *in the current run*, not the cumulative `agent_id`-grouped backlog. Once an issue is filed for a stuck `agent_id`, subsequent runs only re-fire if that same `agent_id` produces fresh shrapnel — backlog records already superseded by prior runs do not contribute. The dup-check on issue title (signal name includes the truncated `agent_id` suffix, e.g. `agent-id-cluster-tg_valor_179144806_9161_edit`) provides belt-and-suspenders. |
| `html-escape-rate` | Of last-7d non-superseded `extraction-*` records, fraction with `&#\d+;` or `&amp;`/`&lt;`/`&gt;` literal in content | **> 0.10** AND week-over-week ratio jumped > 2x | The two-condition guard prevents false positives at low volume (the WoW jump alone fires when baseline is 0). |

**Canonical predicate for "no category" (resolves B1):**

`agent/memory_extraction.py` writes `metadata` via two paths and they disagree on shape:

- **JSON path** (`memory_extraction.py:527-531`): writes `metadata = {"category": <str>, "file_paths": [...], "tags": [...]}`. The `<str>` can be empty (`item.get("category", "")`) or any of the four CATEGORY_IMPORTANCE keys. Legacy records may carry `"default"` as the literal string.
- **Line-based fallback** (`memory_extraction.py:543-573`): writes `metadata={}` for every record. The key is absent, not empty.

The Layer 2 `category-default-skew` signal MUST treat both shapes as equivalent or it will produce false positives on whichever shape is excluded. Pin a single canonical predicate, used in Layer 2 and in tests:

```python
def _has_no_category(metadata: dict | None) -> bool:
    """Canonical 'no category' predicate. True when:
    - metadata is None or empty dict (line-based fallback path), OR
    - metadata['category'] is missing, empty string, or the legacy 'default' literal.

    Resolves B1: JSON path writes metadata={"category": <str>}, fallback writes metadata={}.
    Both shapes must count as 'no category' for the skew detector to be meaningful.
    """
    if not metadata:
        return True
    cat = metadata.get("category")
    return not cat or cat == "default"
```

This predicate is the **only** place the Layer 2 detector reads category state; do not inline the check in the signal computation. Tested separately with both shapes (see `TestMemoryHealthAuditLayer2::test_has_no_category_predicate_handles_both_shapes`).

Window definitions:

- **Last 24h / 7d** = `to_unix_ts(record.created_at) >= time.time() - N*86400`. Skip records where `created_at` is None or unparseable.

Sample memory_ids: when a signal crosses threshold, pick 3-5 representative records (sorted by `created_at` desc, take head) for the issue body.

#### Layer 3 — Gemma Classification (`gemma4:e2b`, Fail-Soft)

```python
GEMMA_AUDIT_PROMPT = """You are auditing memory records produced by an automated extraction pipeline. Each record is supposed to be a one-sentence observation about an agent session. Some are valid; some are extractor failures (raw JSON output, refusal prose, error text).

CRITICAL: code snippets, HTML examples, JSON config, and shell commands are LEGITIMATE memory content. Do not flag them as junk merely because they contain `<`, `&`, `{`, or quotes. Junk looks like extractor JSON output stored as the observation OR refusal/error text.

Examples:

Record: "Telegram bridge handles SIGTERM via async signal handler at bridge/telegram_bridge.py:842, calling client.disconnect() before exit."
Verdict: {"is_junk": false, "anomaly_signal": null, "why": "valid observation about a code location and behavior"}

Record: "\\"observation\\": \\"Update orchestrator automatically bumps critical dependencies\\""
Verdict: {"is_junk": true, "anomaly_signal": "json-key-as-content", "why": "raw extractor JSON output stored as the observation field"}

Now audit this record. Respond with ONLY a JSON object: {"is_junk": bool, "anomaly_signal": str | null, "why": str}.

Record: {content}
Verdict:"""

def _gemma_classify(content: str) -> dict | None:
    try:
        import ollama
        response = ollama.chat(
            model=OLLAMA_LOCAL_MODEL,
            messages=[{"role": "user", "content": GEMMA_AUDIT_PROMPT.format(content=content[:1000])}],
            options={"temperature": 0},
        )
        raw = response["message"]["content"].strip()
        # Tolerant JSON parse (handles fenced output)
        from agent.memory_extraction import _extract_json_payload
        payload = _extract_json_payload(raw) or raw
        return json.loads(payload)
    except Exception as e:
        logger.debug(f"layer3 gemma classify failed (non-fatal): {e}")
        return None
```

**Justification for `_extract_json_payload` reuse (resolves C6):**

`_extract_json_payload` is the parser used by `agent/memory_extraction.py:510` to handle the same model-output shape: a small `gemma`-class model emitting JSON that may be wrapped in markdown fences (` ```json ... ``` `), preceded by preamble prose, or followed by trailing commentary. The audit's Layer 3 model (`gemma4:e2b`) is a sibling of the extractor's prompt-shape population — both produce single-object JSON with possible fence/preamble/trailing-text contamination. Three concrete reasons to reuse rather than write a new parser:

1. **Empirical fit, not aesthetic reuse:** the same failure modes (fenced output, trailing newline noise, occasional preamble) that motivated `_extract_json_payload` (issue #1212) appear in `gemma4:e2b` outputs at temperature=0. Writing a separate parser invites divergent tolerance behavior across two parsers that exist to solve the same problem.
2. **Single-source-of-truth principle (matches Layer 1's `_looks_like_refusal` reuse):** the entire reflection's design pivots on importing predicates from `agent.memory_extraction` rather than duplicating them. `_extract_json_payload` is private (leading underscore) but the alternative — a stdlib-only parser — is materially weaker (does not handle fences) and a dedicated copy is just rename-drift waiting to happen.
3. **Failure path is symmetric:** when `_extract_json_payload` returns `None` (no JSON-shaped substring detected), we fall through to `raw` as the payload, then the outer `json.loads` raises, then the outer `try/except` returns `None`. This is exactly the existing extractor behavior — no new failure mode introduced.

If `_extract_json_payload` ever evolves to be extractor-specific (e.g., gains a `expects_observation_field` flag), Layer 3 will pin the older signature behavior and either copy a frozen helper at that point or coordinate the extension. Tracked as a follow-up note in `reflections/memory_management.py`'s module docstring.

Layer 3 control flow:

- Sample up to **20** last-24h non-superseded `extraction-*` records that survived Layer 1.
- For each, call `_gemma_classify`. On `None` return: skip record (Ollama unavailable) — counts toward "Layer 3 skipped" finding if every call returns `None`.
- Group records by `anomaly_signal` value. Each signal with **≥ 3 records** in the sample becomes a Layer 3 anomaly candidate.
- Issue dup-check uses `signal_name = f"gemma-{anomaly_signal}"` so Layer 3 anomalies don't collide with Layer 2 names.
- The whole Layer 3 block is wrapped in a top-level `try: ... except Exception: pass` so a Layer 3 failure cannot break the audit's return contract.

Async-safety: `ollama.chat` is synchronous. Wrap each call in `await asyncio.get_event_loop().run_in_executor(executor, _gemma_classify, content)` to keep the worker event loop unblocked, where `executor` is a **dedicated** `ThreadPoolExecutor(max_workers=1, thread_name_prefix="memory-audit-l3")` constructed per-invocation and explicitly shut down in a `finally` block. Sharing the default executor with other reflections risks unbounded queueing if Ollama is slow.

**Wallclock budget (resolves C1):** the ~20 × ~500ms ≈ 10s estimate assumes a warm, healthy `gemma4:e2b`. In practice cold-start can add seconds and the model can stall. Enforce an explicit cap:

```python
LAYER3_WALLCLOCK_BUDGET_S = 30  # hard cap; abort remaining records past this
GEMMA_CALL_TIMEOUT_SEC = 10     # per-record asyncio.wait_for timeout (5s was too tight on cold-start)

deadline = time.monotonic() + LAYER3_WALLCLOCK_BUDGET_S
processed = 0
for record in sample:
    if time.monotonic() >= deadline:
        findings.append(
            f"layer-3: wallclock budget {LAYER3_WALLCLOCK_BUDGET_S}s exceeded "
            f"after {processed}/{len(sample)} records — skipping rest"
        )
        break
    try:
        verdict = await asyncio.wait_for(
            asyncio.get_event_loop().run_in_executor(executor, _gemma_classify, record.content),
            timeout=GEMMA_CALL_TIMEOUT_SEC,
        )
    except asyncio.TimeoutError:
        verdict = None  # treated identically to Ollama-unavailable; counts toward "skipped"
    processed += 1
    ...
```

When the budget is exceeded mid-sample: stop processing further records, append a finding noting how many were processed, and continue to issue surfacing using whatever verdicts were collected. The audit return contract is preserved (`status: "ok"`); Layer 3 simply contributes partial data. Tested in `TestMemoryHealthAuditLayer3::test_wallclock_budget_exceeded_aborts_remaining` and `test_per_call_timeout_treated_as_unavailable`.

#### Duplicate-Issue Detection

```python
def _find_open_audit_issue(signal_name: str) -> int | None:
    """Return the issue number of an open audit issue for this signal, or None.

    Resolves C3: `gh issue list` with multiple `--label` flags performs OR-matching
    (any label matches), not AND. To require BOTH `memory` AND `investigation`,
    use `--search "label:memory label:investigation"` (space-separated terms in
    the search string are AND-combined by GitHub's search syntax). The `--label`
    flag is dropped entirely; all label filtering moves into `--search`.
    """
    title_prefix = f"[memory-audit] {signal_name}:"
    # Build search query: AND-combined labels + title substring match.
    # GitHub search treats space-separated terms as AND when not OR'd.
    search_query = (
        f'label:memory label:investigation in:title "{title_prefix}"'
    )
    cmd = ["gh", "issue", "list",
            "--state", "open",
            "--search", search_query,
            "--json", "number,title",
            "--limit", "20"]
    try:
        result = subprocess.run(cmd, check=True, capture_output=True, text=True, timeout=15)
        issues = json.loads(result.stdout)
        for issue in issues:
            if issue.get("title", "").startswith(title_prefix):
                return issue["number"]
        return None
    except Exception as e:
        # On gh failure, prefer not to spam — return a sentinel that suppresses
        # filing for this run. Re-tries next run.
        logger.warning(f"dup-check failed for {signal_name}, suppressing file: {e}")
        return -1  # sentinel: treated as "duplicate exists" by callers
```

Returning `-1` on `gh` failure means "act like a dup exists" — better to skip a real anomaly for one run than to spam duplicates if the search is broken. The duplicate detection runs once per signal per audit invocation.

## Rabbit Holes

- **Don't refactor `reflections/memory_management.py` to one-file-per-reflection.** That's #1028's job. Stay in-file.
- **Don't add new `_REFUSAL_PATTERNS` entries inside this audit.** New refusal phrasings get appended to the upstream tuple in `agent/memory_extraction.py`. The audit imports the predicate; adding patterns here would re-introduce the drift the import was meant to prevent.

- **Don't widen `_looks_like_refusal` to catch the patterns Layer 3 detects.** The whole point of Layer 3 is to catch what regex can't. New regex patterns belong upstream when (and only when) they're high-confidence full-phrase matches.

- **Don't run Layer 3 on every record.** The 20-record sample bounds gemma cost; expanding the sample is a separate tuning decision.
- **Don't try to make Layer 2 thresholds adaptive.** Static thresholds + sample memory_ids in the issue body let a human investigate; adaptive thresholds defer the human's work and risk drifting.

## Risks

### Risk 1: False positives on Layer 1 (legitimate observations marked superseded)

**Impact:** A real memory disappears from recall. Reversible, but invisible until a human notices.

**Mitigation:**

- Layer 1 reuses `_looks_like_refusal` directly — the same predicate already gates new writes upstream. If it had a meaningful FP rate, legitimate observations would be blocked at write time, and they aren't (PR #1217's 1,821-candidate dry-run on this machine showed only refusal/shrapnel, no false positives).
- `agent_id.startswith("extraction-")` blast-radius gate keeps human saves, post-merge learnings, and Telegram memories untouched.
- Reversibility: clearing `superseded_by` restores the record. No deletion.
- 50-supersede cap per run prevents runaway in any pathological scenario.

### Risk 2: Layer 2 issue spam (same signal re-triggers daily)

**Impact:** GitHub issue queue floods, audit becomes noise.

**Mitigation:**

- Duplicate-issue detection: `_find_open_audit_issue` searches for an open issue with matching `[memory-audit] {signal}:` title prefix before filing. Idempotent on re-runs.
- On `gh` search failure, prefer to skip filing (`return -1` sentinel) rather than spam. Re-attempt next run.
- Once a human closes the issue (with or without a fix), the signal can re-trigger on a fresh run — but only if the underlying condition is still present. After the fix lands and the threshold drops, the audit self-quiesces.

### Risk 3: Gemma model unavailable on some machines

**Impact:** Layer 3 skipped, audit reports only Layers 1+2 findings.

**Mitigation:**

- Layer 3 is fully fail-soft (top-level `try: ... except: pass`). Layers 1 and 2 are deterministic and don't depend on Ollama.
- Summary string explicitly states "layer-3 skipped" so operators can distinguish "no anomalies" from "couldn't check".
- On the populated machine where this matters most (1,875 candidates), Layer 1 alone will dominate the cleanup work for the first ~38 runs (1,875 / 50 ≈ 38 days at daily cadence). Layer 3 absence during that window is acceptable.

### Risk 4: `gh issue create` rate limit / auth failure

**Impact:** Issue not filed, but audit continues.

**Mitigation:**

- Wrapped in try/except. Failure logged as a finding; next run retries (after dup-check confirms no issue exists).
- 30s timeout on subprocess prevents hangs.

### Risk 5: Layer 2 threshold tuning drift over time

**Impact:** As the corpus shape changes, thresholds become miscalibrated.

**Mitigation:**

- Each issue body includes `Observed: X (threshold Y)` so humans can see the absolute numbers and judge whether a threshold revision is needed.
- Thresholds are constants at module top; revising them is a one-line PR.
- Memory archives the tuning rationale (this plan) so future readers know why the values are what they are.

## Race Conditions

### Race 1: Layer 1 supersede racing with concurrent extractor writes

**Location:** `reflections/memory_management.py::run_memory_quality_audit` Layer 1 loop, racing with `agent/memory_extraction.py::extract_observations_async` save.

**Trigger:** Audit loads `Memory.query.all()` and starts iterating Layer 1 candidates. Concurrently, the worker writes a fresh extraction record matching `_looks_like_refusal` (which can happen if the upstream guards somehow miss a new refusal phrasing).

**Data prerequisite:** None — Layer 1 only writes `superseded_by` on records that were already in the snapshot.

**State prerequisite:** Records loaded by `Memory.query.all()` are a snapshot at load time; new records arriving after the snapshot are not in the candidate list.

**Mitigation:** No real race — new records arriving during the audit are simply not in this run's snapshot. They'll be candidates next run. The `MAX_LAYER1_SUPERSEDES_PER_RUN = 50` cap further bounds any potential thrash. The audit makes no assumption that the corpus is frozen.

### Race 2: Two concurrent audit invocations

**Location:** Reflection scheduler, in theory.

**Trigger:** Scheduler bug or manual invocation produces two simultaneous audits.

**Data prerequisite:** None.

**State prerequisite:** None.

**Mitigation:** Idempotency by design — `superseded_by` is only set when empty, so a concurrent supersede on the same record is a no-op. Layer 2/3 issue dup-check uses `gh issue list` which is read-after-write consistent within seconds; worst case two audits file the same issue at the exact same moment, in which case a human closes one. Acceptable.

### Race 3: Concurrent supersede write from `memory-dedup`

**Location:** `m.save()` in Layer 1 loop, racing with `scripts.memory_consolidation.run_consolidation` (the `memory-dedup` reflection registered at `config/reflections.yaml:117`) writing `superseded_by` on the same record.

**Trigger:** Both reflections run daily at the same 86400s cadence. The scheduler does not serialize per-class-of-reflection. If they happen to overlap, both may attempt to write `superseded_by` on the same Memory.

**Data prerequisite:** Record exists in Redis and has `superseded_by == ""` at the moment Layer 1 reads it but is overwritten by `memory-dedup` before Layer 1's `.save()`.

**State prerequisite:** Layer 1 candidate filter passed at snapshot time; `memory-dedup` independently selected the same record.

**Mitigation (resolves C2):**

- **Detection:** Before each `m.save()`, re-read `record.superseded_by` via the live model (Popoto re-hydrate, not the snapshot). If non-empty, skip the write — `memory-dedup` already claimed it. Log `superseded_by_already_set` finding (info, not warning).
- **Ordering:** No explicit lock. `memory-dedup`'s consolidation logic also short-circuits when `superseded_by` is already populated (verified in `scripts/memory_consolidation.py`). Both reflections are write-once-on-empty-only by design. The detect-and-skip preserves whichever predicate ran first; for our purposes the rationale string differs (`cleanup-junk-extraction` vs. dedup's UUID) but both are valid terminal states.
- **Acceptable race window:** TOCTOU between the snapshot read and the live re-read is microseconds. If a write lands inside that window, the per-record `try/except` catches the resulting Popoto error and continues.

**NOT applicable:** `memory-decay-prune` (lines 76-77 of `reflections/memory_management.py`) explicitly skips superseded memories before deleting, so it cannot race with our supersede write. The original critique conflated dedup and decay; the actual race is with dedup.
