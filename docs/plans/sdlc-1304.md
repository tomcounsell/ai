# Plan-doc commits on main: route plans through the feature PR

## Problem

Today the `/do-plan` and `/do-plan-critique` skills commit plan documents directly to `main` with no PR review. This was observed during cuttlefish issue #350's pipeline run, but the same convention is encoded in this repo's `.claude/skills-global/do-plan/SKILL.md` (lines 332-340 and 462-464) and is exercised continuously here too — the recent main log shows `Plan: skills-audit issue filing...`, `Plan(#1272): migrate ...`, `Plan(#1288): pin last_comment_id ...` all landing as direct commits.

Three distinct harms compound:

1. **Unilateral edits to canonical documents.** Plan docs at `docs/plans/*.md` record architectural rationale across multiple critique rounds. The agent has more write authority than any human contributor — substantive plan rewrites land on `main` with zero peer review (e.g., cuttlefish `565e4c8` rewrote a plan body during round-2 critique with no review).
2. **Rebase pressure on open PR branches.** When `/do-plan-critique` revises a plan mid-build, the commit lands on `main`. If a `session/{slug}` PR branch is open, GitHub flips its mergeable state to `CONFLICTING/DIRTY`, which (a) wastes a `/do-pr-review` cycle on a dirty tree and (b) forces an in-pipeline rebase. Cuttlefish PR #354 burned ~30 minutes on exactly this race.
3. **Post-merge plan deletion destroys forensic context.** Two deletion sites compound the loss:
   - `scripts/migrate_completed_plan.py:186` calls `plan_path.unlink()` on the path it was given. Today `do-merge.md:539` hands it a temp copy (`/tmp/plan_${SLUG}.md`), so the script ends up only deleting the temp — the script never moves the working-tree plan into `docs/plans/completed/`.
   - `do-merge.md:542` then `rm -f`s the real working-tree plan unconditionally, regardless of whether the migration script succeeded.
   The net effect: today's "migration" is actually two independent deletions plus a validation pass, with no rename of the canonical file. If the migration validator returns nonzero, the plan is still gone from the working tree. Cuttlefish has the worse variant: outright deletion with no validation.

**Current behavior:**
- `/do-plan` Step 2.5 explicitly switches to `main` before committing the plan: `git stash; git checkout main; git stash pop; git add docs/plans/{slug}.md && git commit; git push`. This is mandated at SKILL.md:332-340 and reaffirmed at 462-464.
- `/do-plan-critique` does not commit directly, but its NEEDS REVISION verdict re-invokes `/do-plan`, which re-commits to `main`.
- `/do-build` reads the plan from `origin/main:docs/plans/{slug}.md` (do-merge.md:538), so the build chain assumes plans live on main.
- `/do-merge` Step 4 (do-merge.md:530-549) reads the plan from `origin/main:` into a temp file, hands the temp to `migrate_completed_plan.py` (which `unlink()`s the temp), then `rm -f`s the real working-tree plan. The migration script never sees the canonical path, so the rename-into-`completed/` step the issue's solution sketch (3) describes does not actually happen today — only validation + deletion does.
- The `git show origin/main:${PLAN_PATH}` read at do-merge.md:538 happens immediately after `gh pr merge --squash` in Step 3 with no intervening `git fetch`. Local `origin/main` is therefore one commit behind the squash. Today this works because the plan was committed to main long before the merge; under the new convention (plan only on `session/{slug}` and lands on main *via* the squash) this read returns empty for every cutover-era merge unless do-merge explicitly fetches first.

**Desired outcome:**
- Plans live on the feature branch (`session/{slug}`) from the start. They ship as part of the same PR that ships the feature.
- Plan revisions during critique amend the plan-on-branch. They do not touch `main` and do not invalidate the open PR's mergeable state.
- Post-merge, the canonical plan file is **renamed** (single atomic `git mv`-shaped operation) into `docs/plans/completed/{slug}.md` by the migration script. There is no separate deletion step. Failure leaves the plan in `docs/plans/{slug}.md` as a visible warning.
- `do-merge` Step 4 reads the plan from the synced main working tree (after explicit `git fetch && git checkout main && git pull`), not from `origin/main:` against a stale local ref.
- A guard hook prevents any future agent commit named `Plan: ...` or `Plan(#N): ...` from landing on `main`. The hook is a `PreToolUse:Bash` matcher (the only push-time surface the harness exposes today, mirroring `validate_no_raw_redis_delete.py`) — there is no `PrePush` event in `.claude/settings.json`.

## Data Flow

Trace of a plan document's lifecycle today vs. proposed:

**Today:**
1. **Entry:** `/do-plan` is invoked with `{slug}` and an issue number.
2. **Plan write:** Plan template is filled in at `docs/plans/{slug}.md` on the current branch (whichever it is — could be main, could be a session branch, could be detached).
3. **Branch switch:** `git stash; git checkout main; git stash pop` — forced switch to `main` regardless of current branch.
4. **Commit + push:** `git add docs/plans/{slug}.md && git commit -m "Plan: ..."; git push`.
5. **Tracking issue update:** Issue body is prepended with a `**Plan:**` link pointing to `main` blob.
6. **Critique loop:** `/do-plan-critique` runs. NEEDS REVISION → re-invoke `/do-plan` Phase 4 → step 4 re-commits the revised plan to `main`.
7. **Build:** `/do-build` resolves `PLAN_PATH` and creates `session/{slug}` worktree from the current main (which now includes the plan). Build edits land on the session branch only.
8. **PR opens:** PR #N references plan via the issue link. Plan file is on main.
9. **Race window:** If `/do-plan-critique` runs again during build (say, a code review insight prompts a plan revision), the new plan commit lands on main. Open PR's mergeable state flips to CONFLICTING.
10. **Merge:** `/do-merge` migrates plan to `docs/plans/completed/{slug}.md` AND `rm -f`s `docs/plans/{slug}.md` regardless of migration success.

**Proposed:**
1. **Entry:** `/do-plan` invoked with `{slug}` and issue number.
2. **Branch ensure:** Switch to or create `session/{slug}` branch off current main. The session branch is the home of the plan doc from the first commit.
3. **Plan write:** Plan template filled at `docs/plans/{slug}.md` on `session/{slug}`.
4. **Commit + push:** `git add docs/plans/{slug}.md && git commit -m "plan(#N): {title}"; git push -u origin session/{slug}`.
5. **Tracking issue update:** Issue body links to plan blob on `session/{slug}` branch (not main). When the build PR merges via squash, the plan is preserved in the squash commit and migrated to `docs/plans/completed/`.
6. **Critique loop:** `/do-plan-critique` NEEDS REVISION → re-invoke `/do-plan` → revised plan amends or appends to the session branch. No main commit. Open PR (if exists) is unaffected.
7. **Build:** `/do-build` already runs in `.worktrees/{slug}/` checked out to `session/{slug}` — the plan is already there. (Today's build reads from `origin/main:docs/plans/{slug}.md` because the plan was on main; that read switches to `origin/session/{slug}:docs/plans/{slug}.md` or just the local working tree of the worktree.)
8. **PR opens:** PR includes the plan doc as one of its files.
9. **No race window:** Plan revisions during build land on the session branch — same place build commits go. No CONFLICTING flip.
10. **Merge:** `/do-merge` calls `migrate_completed_plan.py`. On success, the plan moves to `docs/plans/completed/{slug}.md` as part of the post-merge cleanup commit on main. On failure, the plan stays at `docs/plans/{slug}.md` as a visible warning — no silent loss.

## Solution

### Key Elements

- **`/do-plan` rewrite (Phase 2.5):** Drop `git checkout main`. Replace with "ensure on `session/{slug}`" — switch to the branch if it exists, create it from the merge-base of main otherwise. Commit and push the plan there. Use `git switch` + explicit "create if missing" rather than `git checkout -B`, since `-B` resets an existing session branch's HEAD and loses uncommitted build work (see Implementation Note 3).
- **`/do-plan` rewrite (Branch Workflow section):** Replace the "MANDATORY: Plans are committed on `main`" mandate with the inverted rule: plans live on the session branch.
- **`migrate_completed_plan.py` rewrite:** Change `delete_plan()` (currently `plan_path.unlink()` at line 186) to `move_plan(src, dst)` that does `src.rename(dst)` after validation passes. Caller passes the canonical path (`docs/plans/{slug}.md`), not a temp copy. The script becomes the *single* mover; there is no separate deletion step anywhere.
- **`/do-merge` Step 4 rewrite:** Two changes. (a) Before reading the plan, explicitly `git fetch origin main && git checkout main && git pull --ff-only` so the squash-merge is in the working tree. (b) Drop the temp-file dance and the `rm -f`. Hand the migration script the working-tree plan path directly. The script renames it; the rename is staged and committed as part of the post-merge cleanup commit (or amended into a follow-up commit).
- **Push-time hook:** A `PreToolUse:Bash` matcher (named `validate_no_plan_commits_on_main.py`, mirroring `validate_no_raw_redis_delete.py`) inspects any `git push` command. If the current branch is `main` and any commit in `origin/main..HEAD` has a message matching `^(P|p)lan(\(#\d+\))?:` **and** touches only `docs/plans/*.md`, the hook rejects the push. The harness has no PrePush event today, so this is the only available surface; a real `.git/hooks/pre-push` is rejected as a backup because update-skill propagation across machines is brittle for git hooks (see Implementation Note 1).
- **Migration of in-flight plans:** Forward-only. Plans currently in `docs/plans/` stay on main as historical artifacts. The hook does NOT retroactively reject historical plan commits — its `origin/main..HEAD` scan only fires on new pushes, and the `Plan: ...` regex matches the agent's commit format going forward (see Implementation Note 4).

### Technical Approach

- **Skill edit (do-plan):** Lines 332-340 replace the `git stash; git checkout main; git stash pop; git add ...; git commit; git push` block with a `session/{slug}` ensure block:
  ```bash
  if git show-ref --verify --quiet refs/heads/session/${SLUG}; then
    git switch session/${SLUG}
  else
    git switch -c session/${SLUG} origin/main  # create from main; do NOT use -B (would reset existing branch)
  fi
  git add docs/plans/${SLUG}.md && git commit -m "plan(#${ISSUE}): ${TITLE}"
  git push -u origin session/${SLUG}
  ```
  Lines 462-464 ("Branch Workflow") rewrite to state plans live on the session branch.
- **Script rewrite (`scripts/migrate_completed_plan.py`):** `delete_plan()` becomes `move_plan(src: Path, dst: Path)`. Body: validate `dst.parent.exists()` (`docs/plans/completed/`), then `src.rename(dst)`. The caller (do-merge) supplies the canonical path. Update `main()` to expect the canonical plan path, validate, then call `move_plan`. Update test fixtures accordingly.
- **Skill edit (do-merge):** Replace lines 530-549 with:
  ```bash
  BRANCH=$(gh pr view $ARGUMENTS --json headRefName -q .headRefName)
  SLUG=$(echo "$BRANCH" | sed 's|^session/||')
  PLAN_PATH="docs/plans/${SLUG}.md"

  # Ensure local main has the squash-merge before reading the plan
  git fetch origin main
  git checkout main
  git pull --ff-only

  if [ -f "$PLAN_PATH" ]; then
    if python scripts/migrate_completed_plan.py "$PLAN_PATH"; then
      git add "$PLAN_PATH" "docs/plans/completed/${SLUG}.md"
      git commit -m "Migrate plan to completed/: ${SLUG}"
      git push origin main
    else
      echo "ERROR: migrate_completed_plan.py failed for $PLAN_PATH — leaving plan in place" >&2
      exit 1
    fi
  else
    echo "No plan at $PLAN_PATH — skipping migration"
  fi
  ```
  Note the `exit 1` on migration failure: prior behavior was to log and continue, hiding failures. New behavior surfaces them.
- **Skill edit (do-plan-critique):** No git commits there today, but any references to "plan on main" in `ROUND_VERDICT.md` or revision instructions are updated to "plan on session branch."
- **Hook (`PreToolUse:Bash` matcher):** Path: `.claude/hooks/validators/validate_no_plan_commits_on_main.py`. Hook receives the Bash command via stdin (per the existing matcher convention — see `validate_no_raw_redis_delete.py:1-30`). Logic:
  1. Parse the command. If it does not match `git\s+push(\s|$)`, exit 0 (allow).
  2. Determine target ref. If the push is to `refs/heads/main` (explicit) or current branch is `main` (implicit `git push`), continue; else exit 0.
  3. For each commit in `git log origin/main..HEAD --format='%H %s'`, check if subject matches `^(P|p)lan(\(#\d+\))?:` AND `git show --name-only --format= <sha>` lists only paths under `docs/plans/` (excluding `docs/plans/completed/`). If any such commit is found, print a remediation message naming the offending SHA and subject, exit 2 (block).
  4. Otherwise exit 0.
  Register in `.claude/settings.json` under the existing `PreToolUse:Bash` array (lines 38-58 currently host four sibling validators — append a fifth).
- **Tracking-issue link generator:** The "Plan link" in the issue body changes from `https://github.com/{owner}/{repo}/blob/main/docs/plans/{slug}.md` to `https://github.com/{owner}/{repo}/blob/session/{slug}/docs/plans/{slug}.md`. After the build PR squash-merges, do-merge Step 4 rewrites the issue's plan-link line to point at `docs/plans/completed/{slug}.md` on main. The session-branch URL stops resolving once the branch is deleted by the merge; the rewrite to the `completed/` blob restores forensics.

### Implementation Notes (from Critique Round 1)

1. **Hook bypass scope is intentional.** A human can `git push --no-verify` to override. That is the correct escape valve for emergency plan touchups on main (typo fixes via `chore(docs):` prefix already won't match the regex). The hook is a behavioral nudge for the agent, not a security boundary.
2. **Squash-merge collapses critique-round commits into one.** The plan file's *content* survives because the file lives in the squash; the *history* of "Plan revision 1, Plan revision 2" is lost. This is the same harm the issue cited about hidden plan rewrites — the mitigation is that `docs/plans/completed/{slug}.md` becomes the canonical artifact post-merge, and the multi-round evolution is recoverable from the closed PR's commit history (PR view, not main's `git log`). Acceptable trade.
3. **`git checkout -B` is destructive on existing session branches.** If a session branch already exists with uncommitted build work, `-B` resets HEAD and loses the work. Solution uses `git switch` (no force flag) plus explicit existence check. If the branch exists, switch; if not, create from `origin/main`.
4. **No transitional fallback for in-flight main-resident plans.** Plans currently sitting at `docs/plans/{slug}.md` on main with active session branches will continue to live on main until their PR merges and the do-merge migration moves them to `completed/`. The new convention applies to plans created *after* this PR ships. Existing plans-on-main don't get retroactively cherry-picked. This was Risk 3's call; we accept the cost (no in-flight migration tool).
5. **Test impact on `test_migrate_completed_plan.py`.** The current test (if it exists; not yet audited at plan time) likely asserts `unlink()` was called on the plan path. After this change it must assert `rename()` into `docs/plans/completed/`. A grep during build confirms the test's existence and current shape; the build agent updates assertions to match the new contract.

## Risks

### Risk 1: Build skill assumes plan on main
**Impact:** `/do-build` reads from `origin/main:docs/plans/{slug}.md` in some paths. If the plan is only on `session/{slug}`, the read fails and build aborts.
**Mitigation:** `/do-build` already runs inside `.worktrees/{slug}/` checked out to `session/{slug}` — the plan is in the working tree. Audit all `origin/main:` references to plans in skill files (`grep -rn "origin/main:.*docs/plans" .claude/`) and replace with working-tree reads. The do-merge.md:538 read at merge time still needs the plan from somewhere — but at merge time the squash-merge has already landed it on main, so reading from main there is fine.

### Risk 2: Hook false positives on legitimate plan-related commits
**Impact:** A human PM commits a plan touch-up directly on main as a deliberate exception, hook rejects.
**Mitigation:** Hook scope is narrow: matches commits whose message starts with `Plan:` or `Plan(#N):` AND touch ONLY `docs/plans/*.md`. Mixed commits (plan + other files) are exempt. A human override is `git push --no-verify`, which is the correct escape valve.

### Risk 3: Migration of in-flight plans is bumpy
**Impact:** Currently 30+ plans live on main with active session branches. Cherry-picking each onto its branch is tedious.
**Mitigation:** Don't migrate. The convention applies to NEW plans starting from this PR's merge. In-flight plans stay on main as historical artifacts — no harm. The hook only rejects commits with `Plan: ...` message format from this point forward; existing plan files on main are not touched.

### Risk 4: Squash-merge swallows the plan body in commit history
**Impact:** When the build PR squash-merges, the plan commit's body is folded into the squash commit message. Future archaeology via `git log -- docs/plans/` would show one squash commit, not the multi-round critique evolution.
**Mitigation:** The `docs/plans/completed/{slug}.md` file IS the forensic artifact — the post-merge migration commits it on main. Critique rounds amend the same file across multiple commits on the session branch; the squash-merge collapses those into one commit, but the file's content reflects the final state, which is what matters for archaeology. This is acceptable because the plan was always meant to be a living document, not a commit-history-driven log.

## Race Conditions

### Race 1: Concurrent plan revisions during build
**Location:** `/do-plan-critique` re-invocation during `/do-build` execution
**Trigger:** Build is mid-flight on `session/{slug}`, reviewer triggers a critique that produces NEEDS REVISION, `/do-plan` is re-dispatched.
**Data prerequisite:** The plan file at `docs/plans/{slug}.md` on the session branch must be writable.
**State prerequisite:** Build's worktree at `.worktrees/{slug}/` is on `session/{slug}` and may have uncommitted changes.
**Mitigation:** This plan does NOT solve the build/critique race — that's Issue 3. What this plan DOES do is move the race off main, so it no longer flips PR mergeable state. Concurrent edits to the plan file on the session branch are still possible; coordination is Issue 3's job. Today the issue is worse (race spans main + session); tomorrow it's contained to one branch.

### Race 2: Hook race during rebases
**Location:** `.claude/hooks/validators/no_plan_commits_on_main.py` (pre-push)
**Trigger:** Developer rebases a long-lived branch onto main, the rebase replays a plan commit that originally landed on a session branch, hook fires.
**Data prerequisite:** The plan commit's parent during replay is on main.
**Mitigation:** Hook checks the commit's REF (where it's being pushed to), not its parent. Pre-push only fires if the destination ref is `refs/heads/main`. Rebases that target session branches don't trigger.
