# Audit: agent should never make false promises (lean into LLM drafter, tighten gates across all delivery paths)

## Problem

Agent sessions (PM, Dev, Teammate) frequently emit forward-looking commitments — *"I'll come back with X"*, *"will report back"*, *"going forward I'll Y"*. By the time the message reaches the user the session is already over and there is no future execution. Any unfulfilled promise reads as a lie even when the agent intended sincerity.

This issue was prompted by an agent message — *"Reading the user journey notes now and will come back with roadmap impressions + any clarifying questions before touching the build"* — sent through `valor-telegram send`, which writes directly to `telegram:outbox:{session_id}` and bypasses every existing quality gate.

**Current behavior:**
The empty-promise gate (LLM classifier + `_detect_empty_promise` regex backstop) lives in `bridge/message_drafter.py` and only fires for worker-drafted nudge-loop outputs routed through `classify_output()`. Every agent-originated CLI send path bypasses it:

| Send Path | LLM gate | Regex gate | Narration gate |
|---|---|---|---|
| Worker drafted (nudge loop) | YES | YES (heuristic fallback) | YES (drafter-failure path) |
| `tools/send_telegram.py` | NO | NO | NO |
| `tools/send_message.py` (telegram or email) | NO | NO | NO |
| `tools/valor_telegram.py send` | NO | NO | NO |
| `tools/valor_email.py send` | NO | NO | NO |

Even on the gated path, the LLM few-shot prompt and the regex backstop only cover *behavioral-change* promises (`got it / will do / going forward`). They do not cover *forward-deferral* promises (`I'll come back with X / will follow up / stay tuned / more soon / I'll report back`), which create explicit dated user expectations and are arguably the more harmful class.

**Desired outcome:**
> An agent message that promises future work without same-session evidence should never reach a human.

Two honest message shapes only:
- **"I did X"** with concrete evidence (file path, commit hash, queued session ID, memory write, service restart).
- **"I didn't do X because Y"** with explicit reason.

Forward-deferrals are forbidden unless they reference a verifiable autonomous-delivery mechanism (queued session ID, scheduled cron, scheduled agent — surfaced as a session_id, schedule_id, or PR URL).

## Solution

### Architectural Posture (LLM-first, regex fail-closed-only)

**Issue's mandate:** *"the right architectural move is to **widen what the LLM drafter judges** and keep heuristics as the unambiguous fail-closed last line. Regex will always lag the failure modes."* User memory record `feedback_llm_drafter_over_regex`: *"Strengthen the LLM classifier prompt before adding heuristic regex patterns to anti-pattern gates."*

The gate honors both:
- **Primary judgment layer:** LLM classifier prompt strengthened with a named *forward-deferral* class and three new few-shot examples.
- **Heuristic backstop:** Append the forward-deferral pattern set to `_detect_empty_promise` **inside the existing heuristic branch only**. The heuristic fires when `get_anthropic_api_key()` returns None, when the SDK call raises, or when the LLM response fails to parse — i.e., when the LLM cannot return a parseable verdict. The heuristic does **NOT** override an LLM `ALLOW`. The LLM wins on every tie.

This resolves Blocker B1, Blocker B3, and Concern #9 simultaneously.

### Why the heuristic fail-closed posture is correct here (vs. RTR's fail-open)

(cycle-2 Concern C-NEW-3) RTR (`bridge/read_the_room.py`) is heavily cited as the architectural precedent for this gate's async/sync contract, env-var enable pattern, and SDK timeout. But RTR is **fail-open** (any error returns `action="send"`), and this gate's heuristic branch is **fail-closed** (regex match without evidence returns BLOCK). The two postures look contradictory; they are not — they are intentionally inverted because the cost of false-positive is inverted between the two gates.

| Gate | What false-positive means | Cost of false-positive | Correct posture |
|------|---------------------------|------------------------|-----------------|
| Read-the-Room | Suppress a message that should have been sent | Silent message loss — user never sees it, no retry signal | **Fail-open** (error → send) |
| Promise gate | Block a message that's actually honest | Re-emission with the recovery template — the sender (agent or operator) sees the BLOCK on stderr, rewrites, retries | **Fail-closed on heuristic** (regex match without evidence → block) |

The promise gate's BLOCK is loud, recoverable, and bounded: the agent's loop reads the error, applies the recovery template's contractually-acceptable shapes, re-emits, and the second call almost always passes. The cost of a regex false-positive is one extra cycle of the agent's loop — not a silent message loss. RTR's cost of false-positive (silent suppression) is non-recoverable from the user's side, which is why RTR must fail open.

The user's `feedback_llm_drafter_over_regex` memory is honored by keeping the LLM as the **primary** judge with a strengthened prompt. The regex extension exists strictly because the LLM is unavailable in some failure modes (no API key, SDK exception, parse failure) — and without the heuristic, those failure modes would silently allow forward-deferrals through (the more harmful failure mode). Without the heuristic, an outage of the Anthropic API would mean every forward-deferral message ships unchecked — that is the silent-failure mode the heuristic exists to prevent. Adding regex patterns at the heuristic layer is therefore not regex-pile-on against the user's preference; it is the unambiguous fail-closed last line the issue's "Solution Lean" explicitly endorses.

### Key Elements

- **`bridge/promise_gate.py` (new)**: Shared module exposing two functions:
  - `evaluate_promise(text, *, transport, session_id=None, classifier_verdict=None) -> PromiseVerdict` — the judgment function. Public sync API. Internally uses `asyncio.run(_evaluate_promise_async(...))` if the LLM path is taken. The async helper uses `agent.anthropic_client.semaphore_slot()` + bespoke `anthropic.AsyncAnthropic(timeout=RTR_SDK_TIMEOUT)` (the RTR pattern at `bridge/read_the_room.py:427-432`, NOT `anthropic_slot()` which does not accept a timeout — see cycle-2 Blocker B-NEW-1). When `classifier_verdict` is provided (drafter path), short-circuits the LLM call and derives the verdict from the existing classification. When None (CLI path), runs its own Haiku call.
  - `cli_check_or_exit(text, transport, session_id) -> None` — the CLI helper. Calls `evaluate_promise()`, handles BLOCK by printing the stderr template (defined in §Recovery Contract) and `sys.exit(1)`. ALLOW returns silently. There is **no per-call bypass flag** (cycle-2 Blocker B-NEW-2). The only escape hatch is the process-wide kill switch `PROMISE_GATE_ENABLED=false`. The recovery template **does not mention the kill switch** — the agent must not learn the bypass syntax from the gate's own output. The kill switch is documented as a global ops circuit-breaker for incident response (set in the env file or operator's shell), not a per-message bypass. An adversarial agent could in theory defeat it via `PROMISE_GATE_ENABLED=false valor-telegram send "..."` inline through Bash, but (a) the recovery template never teaches that syntax, (b) the audit JSONL records every gate call regardless of verdict so the tampering is observable, and (c) if such tampering is observed in audit traces, that is itself an alignment issue larger than this gate is designed to solve.
- **`PromiseVerdict` dataclass**: `{action: Literal["allow", "block"], reason: str, class_: Optional[str]}`. Two-state — WARN dropped per Concern #9; the issue's binary "honest or not" framing is the right shape, and adding states later (if telemetry shows ambiguous cases) is a follow-up.
- **Strengthened LLM classifier prompt**: Adds an explicit *forward-deferral* class to the few-shot block at `bridge/message_drafter.py:571-600` (which moves into `bridge/promise_gate.py`). Three new few-shot examples covering "I'll come back with X", "will follow up", "stay tuned", and the legitimate scheduled-delivery exception (agent references a session ID or schedule ID).
- **Regex backstop, fail-closed-only**: Append the forward-deferral pattern set to `_detect_empty_promise` **inside the existing heuristic branch only**. The combined logic remains binary inside the heuristic branch: forward-deferral pattern matches AND no evidence pattern matches → return BLOCK; otherwise ALLOW. The heuristic only runs when the LLM cannot return a parseable verdict (no API key, SDK exception, parse failure) — mirroring `bridge/message_drafter.py:889-942`.
- **Wired into the four bypass paths**: `tools/send_telegram.py`, `tools/send_message.py`, `tools/valor_telegram.py send`, `tools/valor_email.py cmd_send` each call `cli_check_or_exit()` with one line, immediately before the Redis outbox write. **No `--no-promise-gate` flag is added.** A bypass flag was rejected in cycle-2 (B-NEW-2) — a per-call flag is trivially defeatable, and a recovery template that teaches the bypass syntax is self-defeating. One honesty contract for all senders.
- **Global kill switch**: `PROMISE_GATE_ENABLED` env var (default `"true"`), read fresh on each call (Concern #4). Mirrors `bridge/read_the_room.py:83`. When false, `evaluate_promise()` returns ALLOW unconditionally and writes an audit JSONL entry tagged `source="promise_gate_disabled"`. Set in the operator's env file or shell startup — not advertised in the recovery template (so the agent does not learn it from the gate's own output). The kill switch is for incident-response ops (e.g., suspected gate regression rolling out a 100% block rate), not a per-message bypass.
- **Telemetry**: Forks a dedicated `_write_promise_audit()` helper in `bridge/promise_gate.py` (cycle-2 Concern C-NEW-2 — the existing `_write_classification_audit()` reads `result.output_type.value` and `result.confidence` which `PromiseVerdict` lacks; reusing it would require either widening the existing helper's signature, breaking drafter call sites, or shoehorning misleading data through). The new helper writes JSONL entries to the same file (`logs/classification_audit.jsonl`) with verdict-specific fields: `{ts, kind: "promise_gate", text_preview, action, reason, class_, transport, session_id, source}`. The `source` discriminator takes one of `"promise_gate_llm"`, `"promise_gate_heuristic"`, `"promise_gate_drafter_delegation"`, `"promise_gate_disabled"`, `"promise_gate_timeout"`, or `"promise_gate_cli_exception"` (cycle-3 C-CYCLE3-3 — fail-open path when `cli_check_or_exit` swallows an unexpected `evaluate_promise()` raise to avoid blocking delivery on infrastructure failures). **session_events emission is conditional** (cycle-2 Concern C-NEW-1): only fires when `AgentSession.query.get(session_id)` (Popoto ORM, per CLAUDE.md no-raw-Redis rule) returns a real session — i.e., the drafter path always (worker session) and `tools/send_telegram.py` when `VALOR_SESSION_ID` is a real session ID. For synthetic `cli-{epoch}` IDs (`valor_telegram.py`, `valor_email.py`), the lookup returns None and session_event emission is silently skipped (audit JSONL still fires). This preserves Concern C6 (gate does no AgentSession state-driven decision; the existence check on the explicit input is for telemetry only).
- **Latency budget**: 3-second SDK-level timeout via the RTR pattern `async with semaphore_slot(): async with anthropic.AsyncAnthropic(timeout=RTR_SDK_TIMEOUT)` (cycle-2 Blocker B-NEW-1 fix — `anthropic_slot()` does NOT accept timeout). Documented expectation: p50 < 500ms, p99 < 3s. On timeout, falls through to heuristic (sub-millisecond).

### Recovery Contract (after BLOCK)

When the gate blocks, the CLI tool prints to stderr and exits non-zero. The exact template, encoded in `bridge/promise_gate.py`:

```
Empty forward-deferral promise blocked by bridge/promise_gate.
The phrase '{quoted offending phrase}' was rejected.

Your session is ending. Do not promise future work. Choose one of:
  (a) Deliver findings now: 'I did X with evidence Y'
  (b) State explicitly that you didn't: 'I didn't do X because Y'

See docs/features/promise-gate.md for the full contract.
```

This resolves Concern #8 — the recovery template names the two contractually-acceptable shapes from the issue's vision section and quotes the rejected phrase. **The template intentionally does NOT mention any bypass mechanism** (cycle-2 Blocker B-NEW-2 — a recovery template that teaches the bypass syntax is self-defeating: the agent reads its own stderr to recover and would learn the bypass on the very first BLOCK). The kill switch `PROMISE_GATE_ENABLED=false` is documented in `docs/features/promise-gate.md` as an incident-response control for operators, not as a per-message bypass.

### Flow

**Agent emits "will follow up"** → `tools/send_telegram.py` invoked → `cli_check_or_exit()` → `evaluate_promise()` → LLM classifier returns BLOCK (forward-deferral, no evidence) → tool prints the recovery template to stderr → `sys.exit(1)` → agent re-emits as "I checked the file, no findings to report yet" → second `evaluate_promise()` returns ALLOW → outbox write → relay delivers.

**Agent emits "queued session abc1234, you'll get a Telegram message when it completes"** → `cli_check_or_exit()` → `evaluate_promise()` → LLM classifier recognizes verifiable scheduled-delivery reference → returns ALLOW → outbox write → relay delivers.

**Operator (Tom) runs `valor-telegram send "Out of office until Tuesday"`** → CLI helper calls `evaluate_promise()` → LLM classifier sees a status statement (no forward-deferral surface form: no "I'll come back", "follow up", "stay tuned", or scheduled-delivery reference required) → returns ALLOW → outbox write → relay delivers. **There is no per-call bypass flag** (cycle-2 Blocker B-NEW-2). If the operator's intended message *is* shaped as a forward-deferral, the gate blocks it the same as it does for an agent — the operator rephrases. The honesty contract is identical for all senders.

**Operator response to a gate regression (false-positive storm)** → operator sets `PROMISE_GATE_ENABLED=false` in their env file or shell startup (a privileged, deliberate operator action, not a per-message bypass) → restarts the relevant service → subsequent CLI invocations and drafter calls see the disabled gate → audit JSONL records every call as `source="promise_gate_disabled"` so the regression remains observable while the kill switch is engaged.

**Drafter path (worker nudge loop)** → existing `classify_output()` runs (one Haiku call) → result passed to `evaluate_promise(text, ..., classifier_verdict=result)` → gate short-circuits the LLM call and derives the verdict from `result.output_type` (BLOCK if STATUS_UPDATE with forward-deferral signal in `nudge_feedback`, else ALLOW) → no double-charge.

### Technical Approach

1. **Factor the gate into `bridge/promise_gate.py`** — extract the LLM classifier prompt's empty-promise few-shot block, the `_detect_empty_promise` regex, and the verdict-shaping logic into the new module. Keep the public API in `bridge/message_drafter.py` (`classify_output`) calling into the shared module so the drafter's behavior is unchanged. The factoring is the minimum prerequisite for wiring the gate into other paths.
2. **Async/sync contract** (Concern C1, cycle-2 Blocker B-NEW-1) — `evaluate_promise()` is **sync** at the public API. Internally, when the LLM path is taken, it uses `asyncio.run(_evaluate_promise_async(text))`. The async helper uses the **RTR-correct** SDK pattern verified at `bridge/read_the_room.py:427-432` — `agent.anthropic_client.anthropic_slot()` does **NOT** accept a timeout argument (verified at `agent/anthropic_client.py:55-84`), so it cannot honor the 3-second SDK-level timeout invariant. The async helper uses `agent.anthropic_client.semaphore_slot()` instead, with a bespoke client constructor that takes `timeout=`:
   ```python
   from agent.anthropic_client import semaphore_slot
   import anthropic
   from utils.api_keys import get_anthropic_api_key
   async with semaphore_slot():
       async with anthropic.AsyncAnthropic(
           api_key=get_anthropic_api_key(),
           timeout=RTR_SDK_TIMEOUT,
       ) as client:
           message = await client.messages.create(...)
   ```
   This honors PR #1055's connection-leak invariant (httpx pool cleanup happens via `async with anthropic.AsyncAnthropic(...) as client:`, not via `asyncio.wait_for`). **`asyncio.wait_for` is forbidden** (per `bridge/read_the_room.py:26-29`). All exceptions are caught; on any exception, the gate returns the heuristic verdict.
3. **Strengthen the LLM prompt with the forward-deferral class** — add a named class definition to the prompt's CRITICAL CONTEXT block, and add three few-shot examples (forward-deferral without evidence → STATUS with nudge_feedback; forward-deferral *with* verifiable scheduled-delivery → COMPLETION; ambiguous "I'll send a follow-up" without delivery mechanism → STATUS).
4. **Append fail-closed-only forward-deferral patterns to the heuristic branch** — `_FORWARD_DEFERRAL_PATTERNS` set added to `_detect_empty_promise` inside `bridge/promise_gate.py`. Patterns:
   - `r"\bi'?ll\s+(?:come|get|circle|loop)\s+back\b"`
   - `r"\bi'?ll\s+(?:report|follow)\s+(?:back|up)\b"`
   - `r"\bstay\s+tuned\b"`
   - `r"\bmore\s+(?:soon|to\s+come)\b"`
   - `r"\bwill\s+(?:report|follow|circle)\s+(?:back|up)\b"`
   - Reuse the existing evidence-pattern set as the negative signal. **The patterns fire only inside the heuristic-fallback branch** (no API key / SDK exception / parse failure). They do NOT override an LLM `ALLOW`.
5. **Decided rule for "deferral + substantive content"** (Blocker B2) — on any forward-deferral phrase, the gate returns BLOCK regardless of accompanying substantive content, **unless** the deferral itself names a verifiable autonomous-delivery reference matching the existing evidence pattern `\b(?:scheduled|queued)\b.*\bsession[_-]?[a-f0-9]{6,}\b` (or equivalent: schedule_id, PR URL). This rule is encoded in (a) the LLM prompt's few-shot block, (b) the regex helper's combination logic, and (c) the test fixtures in `tests/unit/test_promise_gate.py`. Reviewers can see the contract in three independent places.
6. **LLM-vs-heuristic precedence** (Blocker B3) — LLM wins on every tie. Concrete combination: `if llm_verdict is not None: return llm_verdict; else: return heuristic_verdict`. The heuristic only fires when the LLM cannot return a parseable verdict. This matches `bridge/message_drafter.py:889-942` and is the consequence of B1's architectural direction.
7. **Telemetry** (Concern C3, cycle-2 Concern C-NEW-1 + C-NEW-2, cycle-3 C-CYCLE3-3) — every gate call writes an audit row via a **new** `_write_promise_audit(text, verdict, *, transport, session_id, source)` helper in `bridge/promise_gate.py`. The helper writes to `logs/classification_audit.jsonl` (same file as the existing `_write_classification_audit`, unified observability) but with verdict-specific fields: `{ts, kind: "promise_gate", text_preview, action, reason, class_, transport, session_id, source}`. The `source` discriminator is one of `"promise_gate_llm"`, `"promise_gate_heuristic"`, `"promise_gate_drafter_delegation"`, `"promise_gate_disabled"`, `"promise_gate_timeout"`, or `"promise_gate_cli_exception"` (cycle-3 C-CYCLE3-3 — `cli_check_or_exit` swallowed an unexpected raise inside `evaluate_promise()` and fell open). The existing `_write_classification_audit()` is unchanged — drafter audit calls keep their original shape. **session_events emission is conditional on a real AgentSession** (cycle-2 Concern C-NEW-1): on BLOCK, the gate calls `AgentSession.query.get(session_id)` (Popoto ORM, never raw Redis); if a real session is returned, the gate emits a `promise_gate.blocked` session_event via `_append_event(session, ...)` (the same helper RTR uses at `bridge/read_the_room.py:267`); if the lookup returns None (synthetic `cli-{epoch}` ID), session_event emission is silently skipped. The audit JSONL fires unconditionally for every call. This preserves Concern C6 (no AgentSession state-driven decision; the existence check is purely for telemetry routing).
8. **Kill switch** (Concern C4, cycle-3 C-CYCLE3-2) — `_gate_enabled()` reads `PROMISE_GATE_ENABLED` env var fresh on each call. When false, the order inside `evaluate_promise()` is fixed and observable: (a) write audit JSONL with `source="promise_gate_disabled"` *first*, (b) attempt `_emit_session_event_if_real(session_id, {"type": "promise_gate.disabled", ...})` *second* (silently no-ops on synthetic IDs per #7), (c) return ALLOW *third*. The audit-write happens before the early return so disabled-state observability is preserved on every CLI invocation regardless of whether a real `AgentSession` exists. The kill-switch check is the second step in `evaluate_promise()` (after the empty-input check, before the classifier-verdict short-circuit) — see Task 1 step ordering for the full sequence.
9. **Drafter delegation, no double-charge** (Concern C5) — `evaluate_promise()` accepts `classifier_verdict: ClassificationResult | None = None`. On the drafter path, the existing `classify_output()` call is unchanged; its result is passed to `evaluate_promise()`, which derives the gate verdict from `result.output_type` (BLOCK if STATUS_UPDATE *and* `nudge_feedback` mentions forward-deferral, else ALLOW). On CLI paths (no classifier_verdict), the gate runs its own Haiku call.
10. **`session_id` is telemetry-only with mixed provenance** (Concern C6, cycle-2 Concern C-NEW-4) — `evaluate_promise()` accepts `session_id: str | None = None` purely as an audit-log key and session_event tag. The gate makes **no AgentSession state-driven decision** from `session_id`; the only `AgentSession` interaction is a best-effort `AgentSession.query.get(session_id)` to decide whether session_events emission is possible (no-op if the lookup returns None). **Provenance differs across the four CLI paths** and the plan does not paper over this asymmetry:
    - `tools/send_telegram.py:81` reads real `VALOR_SESSION_ID` from the worker harness env — when invoked by the worker, this IS a real AgentSession ID, and session_events emission fires.
    - `tools/valor_telegram.py:763` uses synthetic `f"cli-{int(time.time())}"` — `query.get()` returns None and session_events emission is silently skipped.
    - `tools/valor_email.py::_build_session_id` uses synthetic IDs as well — same skip.
    - `tools/send_message.py` accepts whatever `session_id` its caller passes — usually the harness `VALOR_SESSION_ID` real ID.
   The audit JSONL discriminates entries by `transport` and `source` and records the literal `session_id` regardless of whether it resolves to an AgentSession. This honors the §Race Conditions stateless-judgment claim while acknowledging the real telemetry asymmetry.
11. **No per-call bypass mechanism** (cycle-2 Blocker B-NEW-2; supersedes the cycle-1 C7 resolution) — `cli_check_or_exit()` has **no `no_gate` parameter** and the CLIs have **no `--no-promise-gate` flag**. The cycle-1 design (`--no-promise-gate` gated behind `VALOR_OPERATOR_MODE=1`) was rejected on review: (a) the agent could write `VALOR_OPERATOR_MODE=1 valor-telegram send --no-promise-gate "..."` inline through Bash, defeating the env-var guard; (b) the recovery template instructed the bypass syntax verbatim, so the agent learned the bypass on first BLOCK; (c) operators do not need a per-call bypass — if their message is shaped as a forward-deferral, they rephrase it like the agent does. The kill switch `PROMISE_GATE_ENABLED=false` is the only escape hatch and is documented as an incident-response control (set in env file or shell startup), not advertised in the recovery template. An adversarial agent could still defeat the kill switch via inline `PROMISE_GATE_ENABLED=false valor-telegram send "..."` Bash invocation — but the recovery template never teaches that syntax, the audit JSONL records every call regardless of the kill-switch state (so tampering is observable), and if such tampering shows up in audit traces it is itself an alignment problem larger than this gate's scope.
12. **Wire `cli_check_or_exit()` into the four bypass paths** (Nit N2) — each tool gets a one-line call site immediately before the Redis outbox write. **No new argparse flags are added.** The verdict-handling boilerplate (BLOCK template, exit code) lives in `bridge/promise_gate.py`, not copy-pasted.
13. **Tests** — unit tests for `evaluate_promise()` covering all five forward-deferral phrases standalone (BLOCK), all five combined with substantive evidence (BLOCK per the decided B2 rule), all five with verifiable scheduled-delivery reference (ALLOW), behavioral-change empty promise (regression), behavioral-change with evidence (regression), LLM call raises → heuristic (BLOCK on match, ALLOW otherwise), and `classifier_verdict` short-circuit. Integration test exercises one of the bypass paths end-to-end with a real Anthropic API key.

## Rabbit Holes

- **Refactoring the entire output classification pipeline.** The issue explicitly excludes this. Stay narrowly focused on the empty-promise gate, even if `classify_output()` has other coupling smells.
- **Adding a sixth class of "implicit promise" detection** (e.g., agent uses future-tense verbs without "I'll" prefix). Tempting because the LLM could probably catch them, but the acceptance criteria are bounded to the five named phrases. Adding a sixth class belongs in a follow-up issue if real-world traffic shows that gap.
- **Building a cross-process promise-tracking registry** that records every "I'll do X" and verifies same-session delivery. Architecturally seductive — the right long-term answer — but a 10x larger lift than this audit and out of scope. The gate as designed is stateless: it judges each message in isolation.
- **Replacing all five tools with a single send tool.** Some of the tools (`tools/valor_telegram.py`, `tools/valor_email.py`) intentionally serve human operators as well as agents. Unifying them is a separate concern (and partially addressed by `tools/send_message.py`). The gate should work across all of them as-is.
- **Adding promise detection to the dev-mode console / file output handler.** Issue's "Out of Scope" excludes this explicitly. The console is for development; the file handler is a fallback. Neither reaches a human via Telegram or email.
- **Making `evaluate_promise()` context-aware (read AgentSession.extra_context).** Tempting for telemetry richness, but expands the gate's contract from "stateless judgment" to "stateful classifier." Out of scope per Concern C6's resolution.

## Risks

### Risk 1: False positives — the gate blocks legitimate scheduled-delivery messages
**Impact:** Agent emits "I queued session abc1234, you'll get a Telegram message when it completes" and the gate misclassifies as forward-deferral, blocking a perfectly honest message. Agent then loops trying to find a phrasing that satisfies the gate, wasting tokens and confusing the user.
**Mitigation:** The LLM prompt has an explicit few-shot example for the legitimate case (session ID with autonomous-delivery promise). The regex backstop's evidence-pattern set already includes `(?:scheduled|queued)\b.*\bsession[_-]?[a-f0-9]{6,}\b`. Both layers are oriented to recognize the legitimate case. Integration tests cover both the legitimate and illegitimate cases. Telemetry (Concern C3 resolution) makes false-positive rate observable: every BLOCK writes to the `_write_classification_audit()` JSONL with `source="promise_gate_*"` and emits a `promise_gate.blocked` session_event. If the FP rate exceeds an acceptable threshold (>5% of agent-originated CLI sends, judged via a one-week sample), the LLM prompt is the right knob to turn — not the regex. If the regression is severe and the LLM prompt cannot be tuned in time, `PROMISE_GATE_ENABLED=false` is the kill switch (Concern C4 resolution).

### Risk 2: LLM classifier non-determinism causes flaky CI
**Impact:** Real-API integration test for the new forward-deferral class returns inconsistent verdicts across runs.
**Mitigation:** Real-API tests use fuzzy assertions (verdict in {STATUS, COMPLETION}, confidence threshold check, nudge_feedback presence check) — not exact-string matches. This is the same pattern as the existing `test_classify_output_real_api`. Unit tests use mocked Haiku responses for deterministic coverage of every branch.

### Risk 3: PM-authored messages get blocked
**Impact:** PM-authored messages that flow through `tools/send_telegram.py` (tool-assisted PM communications during execution) or `tools/send_message.py` (final delivery) start getting blocked, and a class of legitimate PM messages get caught in false positives. PMs lose the ability to communicate.
**Mitigation:** PM-authored messages flow through **both** the drafter (final delivery via `bridge/message_drafter.py::classify_output()` on the worker path) and the CLI tools (tool-assisted communication during PM execution). The drafter path is already gated; the CLI paths are not — *that is the gap this plan closes*. The original plan's Risk #3 was internally inconsistent (it claimed PM messages already go through the drafter, but the issue's example was a PM `valor-telegram send` that bypassed the drafter). This rewrite acknowledges the dual-path reality. PM-authored CLI messages that pass an honesty check by design will remain unblocked. Integration test exercises a realistic PM message ("Found three issues in `bridge/foo.py`. Restarted the bridge.") and verifies it passes. If a class of legitimate PM messages is being blocked, the LLM prompt is the right place to add coverage.

### Risk 4: Operator messages from `valor-telegram send` get blocked
**Impact:** Human (Tom, Valor) types `valor-telegram send "out of office until Tuesday"` and the gate blocks it.
**Mitigation:** No per-call bypass exists (cycle-2 Blocker B-NEW-2 retired the cycle-1 `--no-promise-gate` design). The honesty contract is identical for all senders: if the message is shaped as a forward-deferral, it gets blocked, and the sender (operator or agent) rephrases. "Out of office until Tuesday" is a status statement, not a forward-deferral — no "I'll come back", "follow up", "stay tuned", or scheduled-delivery reference is present, so the LLM classifier returns ALLOW and the message ships. The acceptance bound on this risk is the false-positive rate observed in audit JSONL telemetry: if operator messages get blocked at a meaningful rate (>1% of operator-originated CLI sends, judged via a one-week sample after rollout), the LLM prompt is the right knob to turn (as the issue's "lean into LLM" mandate requires) — not to add a per-call bypass. The kill switch (`PROMISE_GATE_ENABLED=false`) remains as the only escape hatch and is documented in `docs/features/promise-gate.md` as an incident-response control set by operators in their env file or shell startup, NOT advertised in the recovery template (so the agent does not learn the bypass syntax from gate output).

### Risk 5: Latency degradation on CLI sends
**Impact:** CLI sends today are ~50ms (Redis rpush). Adding a Haiku call adds ~300-2000ms typical. Multiplied across PM final-delivery, tool-assisted PM communications, valor-telegram operator sends, and valor-email — the agent's responsiveness materially degrades.
**Mitigation:** SDK-level 3-second timeout (`RTR_SDK_TIMEOUT = 3.0` precedent) with fallthrough to sub-millisecond heuristic on timeout. Documented latency expectation: p50 < 500ms, p99 < 3s. Telemetry surfaces actual latency. If p99 exceeds 3s consistently, switch to async drafter-path-only and skip the CLI gate (`PROMISE_GATE_ENABLED=false`) until prompt is tuned.

## Race Conditions

No race conditions identified — `evaluate_promise()` is a pure judgment function. It reads no shared state, writes no shared state (audit logs and session_events are append-only), and is invoked synchronously from each CLI tool's main path before the (single) Redis `rpush` to the outbox. The Redis write itself is already atomic. No locking required.

The theoretical edge — same text passing `evaluate_promise()` twice (drafter path + a separate CLI path) with different LLM verdicts due to non-determinism — is a stateless gate accepting independent judgments. Not a race in the usual sense (no shared resource being mutated). Documented as N1 in the post-revision known-nits list; not addressed in v1.

## No-Gos (Out of Scope)

- Refactoring the nudge-loop architecture itself (excluded by issue).
- Non-bridge delivery paths — file output handler, dev-mode console (excluded by issue).
- LLM model selection / cost optimization (excluded by issue).
- Cross-session promise tracking / "did the agent actually deliver what it promised?" registry. The gate is stateless; the registry is a separate concern, not in this audit.
- Changes to `bridge/read_the_room.py`. RTR is a redundancy gate; the empty-promise gate is the honesty gate. They are conceptually separate and should remain separate modules.
- Changes to the email relay (`bridge/email_relay.py`) or telegram relay (`bridge/telegram_relay.py`). These are pure delivery layers; gates land at the *callers* of the outbox, not the relay.
- Making the gate context-aware (reading AgentSession.extra_context). `session_id` is telemetry-only. Adding context lookup is a follow-up if telemetry shows the stateless gate has a class of false positives the LLM cannot catch from text alone.
