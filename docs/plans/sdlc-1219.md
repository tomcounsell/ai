# Audit: agent should never make false promises (lean into LLM drafter, tighten gates across all delivery paths)

## Problem

Agent sessions (PM, Dev, Teammate) frequently emit forward-looking commitments — *"I'll come back with X"*, *"will report back"*, *"going forward I'll Y"*. By the time the message reaches the user the session is already over and there is no future execution. Any unfulfilled promise reads as a lie even when the agent intended sincerity.

This issue was prompted by an agent message — *"Reading the user journey notes now and will come back with roadmap impressions + any clarifying questions before touching the build"* — sent through `valor-telegram send`, which writes directly to `telegram:outbox:{session_id}` and bypasses every existing quality gate.

**Current behavior:**
The empty-promise gate (LLM classifier + `_detect_empty_promise` regex backstop) lives in `bridge/message_drafter.py` and only fires for worker-drafted nudge-loop outputs routed through `classify_output()`. Every agent-originated CLI send path bypasses it:

| Send Path | LLM gate | Regex gate | Narration gate |
|---|---|---|---|
| Worker drafted (nudge loop) | YES | YES (heuristic fallback) | YES (drafter-failure path) |
| `tools/send_telegram.py` | NO | NO | NO |
| `tools/send_message.py` (telegram or email) | NO | NO | NO |
| `tools/valor_telegram.py send` | NO | NO | NO |
| `tools/valor_email.py send` | NO | NO | NO |

Even on the gated path, the LLM few-shot prompt and the regex backstop only cover *behavioral-change* promises (`got it / will do / going forward`). They do not cover *forward-deferral* promises (`I'll come back with X / will follow up / stay tuned / more soon / I'll report back`), which create explicit dated user expectations and are arguably the more harmful class.

**Desired outcome:**
> An agent message that promises future work without same-session evidence should never reach a human.

Two honest message shapes only:
- **"I did X"** with concrete evidence (file path, commit hash, queued session ID, memory write, service restart).
- **"I didn't do X because Y"** with explicit reason.

Forward-deferrals are forbidden unless they reference a verifiable autonomous-delivery mechanism (queued session ID, scheduled cron, scheduled agent — surfaced as a session_id, schedule_id, or PR URL).

## Why Previous Fixes Failed

| Prior Fix | What It Did | Why It Was Incomplete |
|-----------|-------------|----------------------|
| PR #408 | Added observer-level handling of false-promise halts. | Operated on the observer's classification result, not on outbound text. The drafter-level few-shot examples added later (in PR #1072 and follow-ups) covered behavioral-change language; forward-deferrals were never enumerated. |
| PR #1072 | Established `tools/send_message.py` polymorphic send tool. | Designed as a transport router, not as a quality gate. Routed through `linkify_references` only — promise detection was never wired in because at the time the assumption was that the drafter (which calls send) is the gate. The tool is now invoked *directly by the agent*, bypassing the drafter. |
| `_detect_empty_promise` initial design | Added regex backstop for behavioral-change empty promises. | Pattern set targets "got it / will do / going forward" acknowledgments. Forward-deferral language has different surface form ("I'll come back with X", "stay tuned") and was never enumerated. The decision to add patterns one class at a time created an open-ended whack-a-mole. |

**Root cause pattern:** Quality gates were added at the drafter layer on the assumption that all agent-to-user delivery flows through the drafter. That assumption was true at the time of the gates' design. PR #1072 then created a direct-send path (`tools/send_message.py`) that bypasses the drafter, and `valor-telegram send` / `valor-email send` were similarly built as direct-outbox writers. The gates were never extended to the new paths because they were architecturally siloed in `bridge/message_drafter.py`.

The fix must (a) factor the gate out of the drafter into a shared module that all five send paths call, and (b) extend the LLM classifier prompt and regex backstop to cover the forward-deferral class.

## Architectural Impact

- **New dependencies:** none. The shared promise-detection module reuses the existing Haiku LLM call (already wired) and the existing regex helpers.
- **Interface changes:** A new public function `bridge.promise_gate.evaluate_promise(text, *, context) -> PromiseVerdict` exposed to the four bypass paths (`tools/send_telegram.py`, `tools/send_message.py`, `tools/valor_telegram.py`, `tools/valor_email.py`). The drafter's existing call site delegates to the same shared function.
- **Coupling:** Increases coupling between `tools/*` send paths and `bridge/promise_gate`. This is intentional — the gate is the architectural boundary the issue mandates. The existing import of `linkify_references` from `bridge/message_drafter.py` already establishes the pattern.
- **Data ownership:** No change. The gate is a pure judgment function (text in, verdict out).
- **Reversibility:** High. The gate is a single import + call per send path; reverting is mechanical. The LLM prompt strengthening is a string change. The regex pattern additions are append-only.

## Solution

### Key Elements

- **`bridge/promise_gate.py` (new)**: Shared module exposing `evaluate_promise(text, *, transport, session_id) -> PromiseVerdict`. Wraps the existing Haiku classifier prompt + regex backstop into a single entry point. The existing `_detect_empty_promise` heuristic and the LLM classification logic are factored out of `bridge/message_drafter.py` and re-exported (no behavior change for the drafter path; new available behavior for direct-send paths).
- **Strengthened LLM classifier prompt**: Adds an explicit *forward-deferral* class to the few-shot block at `bridge/message_drafter.py:571-600` (which moves into `bridge/promise_gate.py`). Three new few-shot examples covering "I'll come back with X", "will follow up", "stay tuned", and the legitimate scheduled-delivery exception (agent references a session ID or schedule ID).
- **Tightened regex backstop**: New `_FORWARD_DEFERRAL_PATTERNS` set added to `_detect_empty_promise`. Patterns cover "I'll come back / get back / report back / circle back / follow up / loop back / stay tuned / more soon" combined with absence of scheduled-delivery evidence (session ID, schedule ID, PR URL).
- **Wired into the four bypass paths**: `tools/send_telegram.py`, `tools/send_message.py`, `tools/valor_telegram.py send`, `tools/valor_email.py cmd_send` each call `evaluate_promise()` before queueing. On verdict `BLOCK`, the tool exits non-zero with a stderr message explaining the violation; the agent sees the error and must re-emit honestly. On verdict `WARN`, the tool queues the message and emits a stderr warning. On verdict `ALLOW`, the tool queues the message normally.
- **Bypass mechanism for human/operator sends**: A `--no-promise-gate` flag on the CLI tools, used when a human operator (not an agent) is invoking the tool directly. Justified, documented, and emit-a-stderr-warning so it is auditable.

### Flow

**Agent emits "will follow up"** → `tools/send_telegram.py` invoked → `evaluate_promise()` → LLM classifier returns `BLOCK` (forward-deferral, no evidence) → tool exits non-zero with stderr `Empty forward-deferral promise — your session is ending; either deliver findings now or state explicitly that you didn't.` → agent re-emits honest message → second `evaluate_promise()` returns `ALLOW` → outbox write → relay delivers.

**Agent emits "queued session abc1234, you'll get a Telegram message when it completes"** → `tools/send_telegram.py` invoked → `evaluate_promise()` → LLM classifier recognizes verifiable scheduled-delivery reference (matches existing `(?:scheduled|queued)\b.*\bsession[_-]?[a-f0-9]{6,}\b` evidence pattern, or new equivalent) → returns `ALLOW` → outbox write → relay delivers.

**Operator runs `valor-telegram send --no-promise-gate "Out of office until Tuesday"`** → tool emits `WARNING: --no-promise-gate bypasses honesty checks; only use for operator messages` to stderr → outbox write → relay delivers.

### Technical Approach

1. **Factor the gate into `bridge/promise_gate.py`** — extract the LLM classifier prompt, the `_detect_empty_promise` regex, and the verdict-shaping logic. Keep the public API in `bridge/message_drafter.py` (`classify_output`) calling into the shared module so the drafter's behavior is unchanged. The factoring is the minimum prerequisite for wiring the gate into other paths.
2. **Strengthen the LLM prompt with the forward-deferral class** — add a named class definition to the prompt's CRITICAL CONTEXT block, and add three few-shot examples (forward-deferral without evidence → STATUS with nudge_feedback; forward-deferral *with* verifiable scheduled-delivery → COMPLETION; ambiguous "I'll send a follow-up" without delivery mechanism → STATUS).
3. **Tighten the regex backstop** — append the forward-deferral pattern set to `_detect_empty_promise`. Existing evidence-pattern set (commit hashes, file paths, scheduled session IDs) already serves as the negative signal; reuse it.
4. **Wire `evaluate_promise()` into the four bypass paths** — each tool calls the gate just before queueing the outbox payload. The gate is *fail-closed* in the strict sense: when the LLM call fails AND the regex matches a deferral pattern, the gate blocks. When the LLM call fails AND the regex does not match, the gate allows (falls through). This is the same fail-open posture as `read_the_room` for unrelated errors, but fails closed when the heuristic detects a violation.
5. **CLI bypass flag** — `--no-promise-gate` on all four tools. Documented in `--help` and in `docs/features/promise-gate.md`. Emits a stderr warning on use.
6. **Tests** — unit tests for `evaluate_promise()` covering all five forward-deferral phrases from the issue's acceptance criteria, both standalone and combined with substantive content. Integration test that exercises one of the bypass paths end-to-end with a real Anthropic API key (parallel to `test_message_drafter_integration.py::test_classify_output_real_api`).

## Failure Path Test Strategy

### Exception Handling Coverage
- [ ] `bridge/promise_gate.py::evaluate_promise()` — when the Anthropic call raises (timeout, auth failure, malformed response), the gate logs `logger.warning` and falls through to the heuristic. Test asserts the warning fires and the heuristic verdict is returned.
- [ ] CLI tools (`tools/send_telegram.py`, `tools/send_message.py`, etc.) — when `evaluate_promise()` itself raises (Redis down, import error), the tool falls through to the unwrapped behavior and emits a stderr warning. Test asserts the warning is emitted and the message is queued (don't break delivery on infrastructure failure).
- [ ] Existing `bridge/message_drafter.py::classify_output()` exception handlers are NOT modified by this work; their tests in `tests/unit/test_message_drafter.py` still pass.

### Empty/Invalid Input Handling
- [ ] `evaluate_promise("")` returns `ALLOW` (empty input is not a promise).
- [ ] `evaluate_promise(None)` returns `ALLOW` (None is not a promise; defensive).
- [ ] `evaluate_promise("   \n\t  ")` returns `ALLOW` (whitespace-only is not a promise).
- [ ] CLI tools that receive empty stdin or empty `--message` argparse already exit before reaching the gate; this remains true after the wire-in.

### Error State Rendering
- [ ] When the gate blocks, the CLI tool's stderr message names the *class* of violation ("forward-deferral promise without evidence") and quotes the offending phrase. Test asserts the stderr contains both.
- [ ] When the gate is bypassed via `--no-promise-gate`, the warning is visible on stderr (not swallowed). Test asserts the warning is emitted.

## Rabbit Holes

- **Refactoring the entire output classification pipeline.** The issue explicitly excludes this. Stay narrowly focused on the empty-promise gate, even if `classify_output()` has other coupling smells.
- **Adding a sixth class of "implicit promise" detection** (e.g., agent uses future-tense verbs without "I'll" prefix). Tempting because the LLM could probably catch them, but the acceptance criteria are bounded to the five named phrases. Adding a sixth class belongs in a follow-up issue if real-world traffic shows that gap.
- **Building a cross-process promise-tracking registry** that records every "I'll do X" and verifies same-session delivery. Architecturally seductive — the right long-term answer — but a 10x larger lift than this audit and out of scope. The gate as designed is stateless: it judges each message in isolation.
- **Replacing all five tools with a single send tool.** Some of the tools (`tools/valor_telegram.py`, `tools/valor_email.py`) intentionally serve human operators as well as agents. Unifying them is a separate concern (and partially addressed by `tools/send_message.py`). The gate should work across all of them as-is.
- **Adding promise detection to the dev-mode console / file output handler.** Issue's "Out of Scope" excludes this explicitly. The console is for development; the file handler is a fallback. Neither reaches a human via Telegram or email.

## Risks

### Risk 1: False positives — the gate blocks legitimate scheduled-delivery messages
**Impact:** Agent emits "I queued session abc1234, you'll get a Telegram message when it completes" and the gate misclassifies as forward-deferral, blocking a perfectly honest message. Agent then loops trying to find a phrasing that satisfies the gate, wasting tokens and confusing the user.
**Mitigation:** The LLM prompt has an explicit few-shot example for the legitimate case (session ID with autonomous-delivery promise). The regex backstop's evidence-pattern set already includes `(?:scheduled|queued)\b.*\bsession[_-]?[a-f0-9]{6,}\b`. Both layers are oriented to recognize the legitimate case. Integration tests cover both the legitimate and illegitimate cases. If false-positive rate exceeds an acceptable threshold (TBD with PM), the LLM prompt is the right knob to turn — not the regex.

### Risk 2: LLM classifier non-determinism causes flaky CI
**Impact:** Real-API integration test for the new forward-deferral class returns inconsistent verdicts across runs.
**Mitigation:** Real-API tests use fuzzy assertions (verdict in {STATUS, COMPLETION}, confidence threshold check, nudge_feedback presence check) — not exact-string matches. This is the same pattern as the existing `test_classify_output_real_api`. Unit tests use mocked Haiku responses for deterministic coverage of every branch.

### Risk 3: Wiring the gate into `tools/send_telegram.py` breaks the PM session's outbox writes
**Impact:** PM-authored messages from `tools/send_telegram.py` start getting blocked, and a class of legitimate PM messages (status updates, "queued session abc1234") get caught in false positives. PMs lose the ability to communicate.
**Mitigation:** PM-authored messages already go through the drafter on the worker path. The direct `tools/send_telegram.py` path is used in two cases: (a) self-summarized PM final delivery, (b) tool-assisted communication during PM execution. Both should pass an honesty gate by design. Integration test exercises a realistic PM message and verifies it passes. If a class of legitimate PM messages is being blocked, the LLM prompt is the right place to add coverage.

### Risk 4: Operator messages from `valor-telegram send` get blocked
**Impact:** Human (Tom, Valor) types `valor-telegram send "out of office until Tuesday"` and the gate blocks it.
**Mitigation:** The CLI tools accept a `--no-promise-gate` flag for operator use. Documented in `--help` and the feature doc. Emits a stderr warning so it remains auditable. This is an explicit, justified bypass — the kind the issue's acceptance criterion calls for.

## Race Conditions

No race conditions identified — `evaluate_promise()` is a pure judgment function. It reads no shared state, writes no shared state, and is invoked synchronously from each CLI tool's main path before the (single) Redis `rpush` to the outbox. The Redis write itself is already atomic. No locking required.

## Open Questions

1. **CLI exit behavior on `BLOCK` — non-zero exit vs. silent re-prompt?** The plan currently has the tool exit non-zero with stderr. The agent will see this in its Bash tool output and naturally re-emit. Alternative: write a "blocked draft" record to Redis with the gate's verdict + reason, and have the agent's loop detect it and re-emit without surfacing the error to the human. Non-zero exit is simpler and matches Unix conventions; the alternative is less noisy but adds a new Redis key shape. **Recommendation: non-zero exit. Confirm?**

2. **`--no-promise-gate` access control.** The flag is documented and emits a stderr warning, but there's no programmatic way to prevent the agent from passing it. The agent could theoretically bypass the gate by always passing the flag. Two options: (a) trust the agent (the warning is a tripwire that surfaces in agent output and humans will catch it); (b) gate the flag behind an env var (`VALOR_OPERATOR_MODE=1`) that the agent's session context never sets. **Recommendation: option (a) — the warning + stderr surfacing is sufficient for an audit-grade tripwire, and option (b) adds operational complexity.**

3. **Verdict precedence when LLM and heuristic disagree.** When the LLM returns `ALLOW` but the regex matches a forward-deferral pattern (no evidence), which wins? The LLM has more context but is non-deterministic; the regex is rigid but lower-false-negative for the named phrases. **Recommendation: regex wins on `BLOCK` (fail-closed posture). LLM wins on everything else.** Confirm?

4. **Integration test budget for real-API calls.** Adding a real-API integration test for the new class costs ~$0.001 per run and ~2-5 seconds. The existing `test_classify_output_real_api` is the only real-API test in this area today. The plan adds two more (forward-deferral case in the existing file + dedicated `test_promise_gate_real_api.py`). **Confirm this aligns with the project's real-API test budget?**
